## 版本目标

**一句话目标**: 在 `v3.14.16` 中完成广告聚合 MCP 的统一需求收敛，将有效需求全部纳入版本，并按“直接支持、待开发评估工期、需补原型”三种状态推进。

### 本版执行口径
- **本版支持**: 需求有效，边界足够清晰，可直接进入开发排期。
- **待开发评估工期**: 需求有效并确认要做，但实现成本、依赖或工期待开发评估后补充。
- **需补原型**: 需求有效并确认要做，但交互、页面或流程需要先出原型收敛。

### 本版不采用的口径
- 不使用“待定”“后续讨论”“先放着”这类模糊状态。
- 不因为需要开发评估或原型而将需求移出 `v3.14.16`。

## 用户反馈需求清单

以下内容从两份广告业务分析文档和会议纪要中提炼，作为“用户反馈”口径统一收敛，避免只保留原文描述而没有产品化拆解。

| 反馈编号 | 用户反馈/原始诉求 | 归纳后的需求 | 来源 | 当前问题 |
|----------|-------------------|--------------|------|----------|
| U1 | 用户满意度不高，分析需求很多 | 需要低门槛的自助分析入口 | 分析文档 | 当前取数和分析路径过重 |
| U2 | 不让下载到本地，只能导入内部工具文档，操作有限制 | 需要数据不落地的在线分析方式 | 分析文档 | 本地 Excel 路径不合规，内部工具又不够灵活 |
| U3 | 新增指标需求很多 | 需要支持组合指标即时查询 | 分析文档 + 广州二部需求 | 现状依赖数分或开发加字段 |
| U4 | 聚合对比分析场景很多 | 需要支持多维聚合、对比、透视 | 分析文档 + `04-16` 会议纪要 | 现状要么手工 SQL，要么导出后人工处理 |
| U5 | 数分手动 SQL 配报表 | 需要把高频 SQL 模板沉淀为可复用能力 | 分析文档 | 数分成为取数瓶颈 |
| U6 | 内部工具导入后仍受限，跨表分析能力差 | 需要支持跨表关联或结果卡片化复用 | `04-16` 会议纪要 + 分析文档 | 用户拿到数据后仍难继续分析 |
| U7 | 想要灵活分层、灵活时间范围、灵活筛选 | 需要对话式或多维分析承接动态分析需求 | `04-16` 会议纪要 | 传统固定报表无法覆盖 |
| U8 | 想看新增指标、再加一个指标、只看某几个媒体并排序 | 需要结果二次加工能力 | 分析文档 | 当前每次变更都要重新改 SQL |

### 用户反馈拆解结果

| 收敛方向 | 包含的反馈 | 产品结论 | 本版归类 |
|----------|------------|----------|----------|
| 对话式分析入口 | U1、U2、U4、U8 | 保留为本版核心能力，不再砍掉分析诉求 | 本版支持 |
| 指标即时组合查询 | U3、U8 | 通过问数/MCP 模板承接，不再默认走 BI 报表排期 | 本版支持 |
| 多维聚合与动态分层 | U4、U7 | 由多维分析能力承接，先试点验证 | 待开发评估工期 |
| 高频 SQL 模板沉淀 | U5 | 形成 MCP 分析模板库，替代重复手工 SQL | 本版支持 |
| 跨表分析与结果复用 | U6 | 作为协同与配套能力推进，不单列重 BI 建设 | 待开发评估工期 |
| 复杂分析交互 | U2、U7、U8 | 需要原型定义问数、结果追加、过滤排序、分享等流程 | 需补原型 |

### 用户反馈收敛原则

1. 不把“用户想分析”简单理解为“要重做一个 BI 平台”。
2. 优先用问数、MCP 模板、结果卡片化满足高频分析反馈。
3. 动态分群、复杂聚合、跨表编排统一收口到多维分析试点和协同评估，不在本版承诺大而全落地。
4. 所有“用户反馈”必须转换成明确需求项或任务项，不能只保留原话。

## 功能要求

### 1. 标签与基础查询能力
- 维护广告业务基础标签体系，覆盖时间、活动、项目、包体、媒体、账户、团队等核心维度。
- 输出标签字典文档，供发行、数分、系统和 AI 共用。
- 提供基础实体查询能力，支撑自然语言问数前的实体解析与候选回填。
- 增加三大媒体字段映射说明，明确 Excel 中相同业务含义在不同媒体下的字段名、展示名和口径差异。

### 2. 报表 MCP 能力
- 覆盖小时、日周月、ROI、留存等高频广告报表能力。
- 输入参数与业务口径保持一致，避免问数结果与 BI 口径偏离。
- 在本版内优先保证核心报表链路可用，再逐步补充更多低频报表。
- 日周月报表中的有效用户分析体系保留在版本内，但不作为当前主收口项。
- 素材报表、监测活动报表、ROI 分析页优先补齐七日付费指标组、LTV 增长系数与分成后指标。
- 素材报表优先补齐图片/视频筛选，并继续收口媒体、应用类型、优化目标、营销场景、投放模式、状态等高频筛选项。
- 报表汇总行需支持按渠道拆分展示，不再仅保留日期级汇总。

### 3. 多维分析能力
- 删除聚合分析引擎 MCP 的独立需求，不再建设单独的聚合分析引擎 MCP。
- 按时间、团队、优化师等维度的分析与聚合，统一放到多维分析能力中实现。
- 金额类、数值类、成本类、比率类指标遵循统一口径。
- 周/月聚合是否支持、长线指标如何计算，统一归入开发评估项。
- 灵活付费分层、留存、分群等高变动需求优先通过多维分析试点承接，不固化为传统 BI 报表。
- 问数与 Chart 是本版系统性解决路径，用于承接聚合、细分、对比、透视等分析场景。

### 4. 数据推送能力
- 预留广告数据整理后推送到 Hive 或下游系统的方案。
- 明确目标表、调度频率、失败重试和消费方。
- 在需求有效已确认前提下，工期待开发评估后回填。
- 与小闪团队的跨表分析能力优化，作为下游消费链路配套协同事项跟进。

### 5. 跨媒体字段映射说明
- 需要从 Excel 中整理三大媒体的字段名、中文名称和业务口径，作为问数解析、筛选映射和报表展示的统一基线。
- 重点覆盖同一业务概念在不同媒体中的异名字段，例如“投放模式”在不同媒体下的字段名和展示名称不一致。

## 关键约束与决策

1. **生产环境导向**: 本次以生产可用方案为目标，不保留数据库连接演示类描述。
2. **标签表结构由开发定**: 宽表还是多表由开发结合多维分析方案评估确定。
3. **聚合分析并入多维能力**: 不再单列聚合分析引擎 MCP，相关实现全部归入多维分析能力。
4. **元数据与字段定义持续补充**: 不阻塞本次版本需求收敛，但需作为实施输出的一部分。
5. **广州二部需求和需求池新增需求不单独拆版**: 默认进入 `v3.14.16`，仅按执行前置条件分类。
6. **跨媒体字段必须统一命名**: Excel 中三大媒体相同业务概念的异名字段需先映射，再进入问数、报表和配置实现。
7. **分成后指标以真实结算数据为准**: 会议纪要已明确优先推动财务接口修复，暂不采用手动固定比例估算方案。
8. **RY 共性指标统一收口**: RY 分析中的注册成本、付费成本、付费率、UP 值等，优先归入日周月报表统一承接。
