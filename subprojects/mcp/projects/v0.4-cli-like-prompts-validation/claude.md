# MCP与CLI-like格式效果验证Demo 产品需求&开发全文档
**文档版本**：V1.0
**生成日期**：2026-04-08
**适用场景**：Claude Code本地开发、Cherry Chat接入、BI测试环境对接、大模型调用效果对比测试
**核心目标**：从0到1搭建可直接运行的双模式MCP服务，严格控制变量，验证「原生MCP结构化调用」vs「CLI-like命令行格式调用」在内部BI数据查询场景的效果差异

---

## 一、产品概述
本项目为**最小可用验证型Demo**，核心是搭建两个完全隔离、底层逻辑100%复用的MCP标准HTTP服务：
1. 「原生MCP服务」：将现有BI接口封装为多工具结构化MCP服务，大模型通过标准ToolCall方式调用
2. 「CLI-like MCP服务」：将现有BI接口封装为单工具CLI格式MCP服务，大模型仅需输出命令行字符串，后端解析后复用同一套BI接口逻辑

**核心设计原则**：唯一变量为「大模型调用格式」，其余环节（参数校验、BI接口调用、数据返回、日志记录）完全一致，确保测试结果的有效性、可对比性、可复现性。
**无额外开发负担**：完全复用现有BI测试环境接口，不做SQL生成、不碰数据库、不新增业务逻辑，仅做协议封装与格式适配。

## 二、核心验证目标
### 2.1 主验证目标（量化可统计）
通过同一测试集、同一大模型、同一业务接口，对比两种模式的5项核心指标，验证CLI-like格式是否具备显著优势：
| 指标名称 | 指标定义 | 计算公式 |
|----------|----------|----------|
| 参数完整率 | 大模型调用时，必填参数无遗漏的占比 | （参数齐全的有效调用次数/总调用次数）×100% |
| 参数正确率 | 大模型填充的参数值符合业务规则的占比 | （参数值正确的调用次数/总调用次数）×100% |
| 格式合规率 | 大模型输出完全符合对应格式要求的占比 | （格式合规的调用次数/总调用次数）×100% |
| 工具匹配准确率 | 大模型选择正确工具/正确命令的占比 | （工具匹配正确的调用次数/总调用次数）×100% |
| 幻觉/拒绝率 | 大模型拒绝调用、编造工具/参数、无意义输出的占比 | （幻觉/拒绝的调用次数/总调用次数）×100% |

### 2.2 次验证目标
1. 验证「一键发布」能力：通过配置文件快速将BI接口封装为两种格式的MCP服务，无需修改核心代码
2. 验证兼容性：双模式服务均符合MCP标准协议，可直接在Cherry Chat客户端绑定使用
3. 验证可扩展性：新增BI接口时，仅需修改配置文件，无需重构核心代码

### 2.3 边界与约束
1. 底层完全复用现有BI测试环境接口，不做任何SQL编写、模板配置、数据库直连操作
2. 双模式服务必须严格遵循MCP官方协议标准，确保Cherry Chat可正常识别、绑定、调用
3. 除大模型调用格式外，双模式的参数校验、超时控制、接口转发、日志记录逻辑必须100%一致
4. 全程本地可运行，无需服务器部署、无需数据库依赖，开箱即用

## 三、验证方法与测试设计
### 3.1 测试环境
- 开发环境：本地Python环境 + Claude Code开发工具
- 客户端：Cherry Chat（支持MCP协议绑定）
- 底层依赖：公司BI测试环境真实业务接口（带鉴权Token）
- 大模型：与生产环境一致的大模型API（确保测试结果可复现）
- 测试工具：现有自动化测试工具（可批量调用大模型API、自动统计指标、生成测试报告）

### 3.2 测试变量控制
- **唯一变量**：MCP服务的调用格式（原生MCP结构化ToolCall / CLI-like命令行格式）
- **固定不变量**：BI接口、业务数据、大模型版本、测试用例、参数校验规则、超时时间、鉴权逻辑

### 3.3 测试用例设计
覆盖3个难度梯度，总计≥30条用例，同一条用例分别在双模式下各运行≥3次，取平均值：
1. **简单意图（10条）**：单指标、单维度、无歧义查询，示例：
   - 查1001游戏2026-04-01的日活数据
   - 查1002游戏近7天的总付费金额
2. **中等意图（10条）**：多指标、多维度、带筛选条件查询，示例：
   - 按渠道拆分1001游戏2026年3月的付费人数和ARPU值
   - 对比1001和1002游戏近30天的DAU和7日留存率
3. **复杂/歧义意图（10条）**：模糊需求、多条件组合、趋势分析类查询，示例：
   - 看一下2026年3月1001游戏表现最好的3个渠道
   - 把安卓和iOS分开，看1001游戏近14天的新用户次留趋势

### 3.4 测试执行流程
1. 本地启动双模式MCP服务，获取两个独立的HTTP端点
2. 在Cherry Chat中分别绑定「原生MCP服务」和「CLI-like MCP服务」
3. 用同一套测试用例，分别在两个服务中批量运行
4. 自动化测试工具自动抓取调用日志、统计5项核心指标
5. 生成对比测试报告，得出最终验证结论

## 四、详细开发需求文档
### 4.1 整体架构设计
↑（统一请求转发，100% 复用）
[统一适配层：参数校验 + 日志记录 + BI 接口调用]
↑
[双模式 MCP 服务]
├── 原生 MCP 模块 → 暴露 /mcp 端点 → 输出多工具结构化清单 → 大模型 ToolCall 调用
└── CLI-like 模块 → 暴露 /cli 端点 → 输出单工具 CLI 命令清单 → 大模型输出命令字符串调用

### 4.2 功能需求明细
#### 模块1：BI接口统一适配层
| 需求ID | 需求描述 | 验收标准 |
|--------|----------|----------|
| F001 | 对接BI测试环境接口，支持配置接口地址、请求方法、鉴权Token、超时时间 | 1. 可通过配置文件修改接口信息，无需改核心代码；2. 支持GET/POST请求；3. 自动携带鉴权Header |
| F002 | 统一参数接收与转发，将前端传入的参数映射为BI接口的入参 | 1. 支持参数名映射配置；2. 自动过滤非必填参数；3. 转发参数与BI接口要求100%匹配 |
| F003 | 统一接口返回处理，将BI接口的返回结果格式化为MCP标准返回格式 | 1. 成功时返回结构化数据；2. 失败时返回明确的错误信息；3. 格式兼容大模型解析要求 |

#### 模块2：原生MCP服务模块
| 需求ID | 需求描述 | 验收标准 |
|--------|----------|----------|
| F004 | 遵循MCP官方协议标准，暴露/mcp HTTP端点 | 1. 支持Cherry Chat直接绑定；2. 可正常返回工具清单、处理工具调用请求 |
| F005 | 支持通过配置文件，将多个BI接口自动封装为多个独立的MCP工具 | 1. 每个BI接口对应一个MCP工具；2. 工具名称、描述、参数配置可通过配置文件修改；3. 新增工具无需改核心代码 |
| F006 | 接收大模型的ToolCall结构化请求，完成参数校验后转发至BI适配层 | 1. 严格校验必填参数；2. 校验参数类型与格式；3. 校验不通过时返回明确的错误提示 |

#### 模块3：CLI-like MCP服务模块
| 需求ID | 需求描述 | 验收标准 |
|--------|----------|----------|
| F007 | 遵循MCP官方协议标准，暴露/cli HTTP端点，与/mcp端点完全隔离 | 1. 支持Cherry Chat直接绑定；2. 与原生MCP服务端口一致，路径不同，互不干扰 |
| F008 | 固定仅暴露1个名为`cli`的MCP工具，仅接收1个`command`字符串类型参数 | 1. 工具描述中明确命令行格式规范与示例；2. 仅接收command参数，无其他入参；3. Cherry Chat可正常识别 |
| F009 | 内置CLI命令解析器，可将`命令名 --参数名 参数值`格式的字符串，解析为结构化参数 | 1. 支持长参数`--key value`格式；2. 自动提取命令名、参数键值对；3. 支持带空格的参数值（引号包裹）；4. 解析失败时返回明确错误 |
| F010 | 解析完成后，将参数转发至统一BI适配层，与原生MCP复用完全一致的逻辑 | 1. 参数校验规则、接口转发逻辑、返回格式与原生MCP100%一致；2. 仅解析环节不同，其余环节无差异 |

#### 模块4：一键发布配置模块
| 需求ID | 需求描述 | 验收标准 |
|--------|----------|----------|
| F011 | 支持通过单一配置文件，完成双模式服务的全量配置 | 1. 配置文件包含BI接口信息、工具配置、命令配置、鉴权信息；2. 所有配置集中管理，无需分散修改 |
| F012 | 一键启动双模式服务，自动生成工具清单、完成服务注册 | 1. 执行单条命令即可启动服务；2. 启动时自动校验配置合法性；3. 启动成功后打印两个端点的访问地址 |

#### 模块5：日志与辅助模块
| 需求ID | 需求描述 | 验收标准 |
|--------|----------|----------|
| F013 | 全链路日志记录，自动记录每次调用的入参、解析结果、BI接口返回、耗时、错误信息 | 1. 区分原生MCP/CLI-like调用日志；2. 日志格式清晰，可直接用于自动化测试统计指标；3. 支持控制台输出与文件存储 |
| F014 | 统一异常处理，所有异常均返回标准化错误信息，不暴露服务内部细节 | 1. 参数错误、解析错误、BI接口调用错误均有明确的错误提示；2. 异常不会导致服务崩溃 |

### 4.3 非功能需求
1. **轻量性**：最小依赖，仅使用Python通用第三方库，无重框架依赖
2. **兼容性**：严格遵循MCP 2.0标准协议，兼容Cherry Chat、Claude Code等所有支持MCP的客户端
3. **可维护性**：模块化开发，配置与代码分离，核心逻辑与业务配置分离
4. **稳定性**：单服务并发支持≥10QPS，满足测试场景需求，无内存泄漏，异常不崩溃


## 五、核心功能交付
双模 MCP 服务：
原生 MCP（/mcp）：将 BI 报表接口（如广告日报、小时报表）自动映射为结构化 Tool，支持大模型原生调用。
类似 CLI 的/cli）：暴露统一的 cli 工具，支持大模型通过命令行字符串（如 ads_report_day --appId 10100042）进行查询。
统一适配层：底层复用 bi_client.py，自动处理 Token 鉴权、参数映射及报表字段包装。
自动化评估引擎：内置 测试引擎.py，基于您提供的测试集，利用 Gemini API 自动对比两种模式的准确率并生成 Markdown 报告。
可视化管理后台：提供 React 界面，支持交互式调试及一键全量评估。
快速开始
交互测试：在当前系统的预览窗口中，您可以切换 原生 MCP 或 类似命令行界面 模式，输入查询意图进行实时测试。
全量评估：点击右上角 “执行全量评估”，系统将自动运行 30+ 条测试用例，并在右侧面板生成详细的对比报告。
外部接入：
Native MCP 地址：https://<当前AppURL>/mcp
CLI-like 地址：https://<当前AppURL>/cli
目录结构说明
config.yaml: 核心配置文件，可在此快速增减 BI 接口定义。
server.py: FastAPI 后端，处理 MCP 协议转换。
bi_client.py: BI 接口通讯模块。
cli_parser.py: 命令行字符串解析器。
测试引擎.py: 自动化测试与报告生成逻辑。
src/App.tsx: 可视化评估与调试面板。
您可以直接开始测试，或根据需要修改 config.yaml 来接入更多报表接口。
