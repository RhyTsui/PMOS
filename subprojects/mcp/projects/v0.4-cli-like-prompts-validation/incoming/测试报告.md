## 一、产品概述
本项目为**最小可用验证型Demo**，核心是搭建两个完全隔离、底层逻辑100%复用的MCP标准HTTP服务：
1. 「原生MCP服务」：将现有BI接口封装为多工具结构化MCP服务，大模型通过标准ToolCall方式调用
2. 「CLI-like MCP服务」：将现有BI接口封装为单工具CLI格式MCP服务，大模型仅需输出命令行字符串，后端解析后复用同一套BI接口逻辑

**核心设计原则**：唯一变量为「大模型调用格式」，其余环节（参数校验、BI接口调用、数据返回、日志记录）完全一致，确保测试结果的有效性、可对比性、可复现性。
**无额外开发负担**：完全复用现有BI测试环境接口，不做SQL生成、不碰数据库、不新增业务逻辑，仅做协议封装与格式适配。

## 二、核心验证目标
### 2.1 主验证目标（量化可统计）
通过同一测试集、同一大模型、同一业务接口，对比两种模式的5项核心指标，验证CLI-like格式是否具备显著优势：
| 指标名称 | 指标定义 | 计算公式 |
|----------|----------|----------|
| 参数完整率 | 大模型调用时，必填参数无遗漏的占比 | （参数齐全的有效调用次数/总调用次数）×100% |
| 参数正确率 | 大模型填充的参数值符合业务规则的占比 | （参数值正确的调用次数/总调用次数）×100% |
| 格式合规率 | 大模型输出完全符合对应格式要求的占比 | （格式合规的调用次数/总调用次数）×100% |
| 工具匹配准确率 | 大模型选择正确工具/正确命令的占比 | （工具匹配正确的调用次数/总调用次数）×100% |
| 幻觉/拒绝率 | 大模型拒绝调用、编造工具/参数、无意义输出的占比 | （幻觉/拒绝的调用次数/总调用次数）×100% |

### 2.2 次验证目标
1. 验证「一键发布」能力：通过配置文件快速将BI接口封装为两种格式的MCP服务，无需修改核心代码
2. 验证兼容性：双模式服务均符合MCP标准协议，可直接在Cherry Chat客户端绑定使用
3. 验证可扩展性：新增BI接口时，仅需修改配置文件，无需重构核心代码

### 2.3 边界与约束
1. 底层完全复用现有BI测试环境接口，不做任何SQL编写、模板配置、数据库直连操作
2. 双模式服务必须严格遵循MCP官方协议标准，确保Cherry Chat可正常识别、绑定、调用
3. 除大模型调用格式外，双模式的参数校验、超时控制、接口转发、日志记录逻辑必须100%一致
4. 全程本地可运行，无需服务器部署、无需数据库依赖，开箱即用

## 三、验证方法与测试设计
### 3.1 测试环境
- 开发环境：本地Python环境 + Claude Code开发工具
- 客户端：Cherry Chat（支持MCP协议绑定）
- 底层依赖：公司BI测试环境真实业务接口（带鉴权Token）
- 大模型：与生产环境一致的大模型API（确保测试结果可复现）
- 测试工具：现有自动化测试工具（可批量调用大模型API、自动统计指标、生成测试报告）

### 3.2 测试变量控制
- **唯一变量**：MCP服务的调用格式（原生MCP结构化ToolCall / CLI-like命令行格式）
- **固定不变量**：BI接口、业务数据、大模型版本、测试用例、参数校验规则、超时时间、鉴权逻辑

### 3.3 测试用例设计
覆盖3个难度梯度，总计≥30条用例，同一条用例分别在双模式下各运行≥3次，取平均值：
1. **简单意图（10条）**：单指标、单维度、无歧义查询，示例：
   - 查1001游戏2026-04-01的日活数据
   - 查1002游戏近7天的总付费金额
2. **中等意图（10条）**：多指标、多维度、带筛选条件查询，示例：
   - 按渠道拆分1001游戏2026年3月的付费人数和ARPU值
   - 对比1001和1002游戏近30天的DAU和7日留存率
3. **复杂/歧义意图（10条）**：模糊需求、多条件组合、趋势分析类查询，示例：
   - 看一下2026年3月1001游戏表现最好的3个渠道
   - 把安卓和iOS分开，看1001游戏近14天的新用户次留趋势

### 3.4 测试执行流程
1. 本地启动双模式MCP服务，获取两个独立的HTTP端点
2. 在Cherry Chat中分别绑定「原生MCP服务」和「CLI-like MCP服务」
3. 用同一套测试用例，分别在两个服务中批量运行
4. 自动化测试工具自动抓取调用日志、统计5项核心指标
5. 生成对比测试报告，得出最终验证结论
