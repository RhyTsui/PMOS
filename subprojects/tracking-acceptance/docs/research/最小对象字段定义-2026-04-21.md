# 最小对象字段定义

日期：2026-04-21

目的：把前面研究里提到的核心对象压成最小可执行字段集。  
不是数据库最终设计，而是第一阶段可落地的数据结构草案。

## 1. BusinessIntent

表示：业务方真正想解决什么问题。

### 最小字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `intent_id` | string | 唯一 ID |
| `title` | string | 需求标题 |
| `business_goal` | text | 业务目标 |
| `request_type` | enum | 新报表 / 新分析 / 新埋点 / 变更 / 验收 |
| `source` | enum | 会议纪要 / 群聊 / 文档 / 口头整理 |
| `requester_role` | enum | 产品 / 运营 / 投放 / 分析 / 研发 / 其他 |
| `consumer_roles` | array | 最终谁要用结果 |
| `decision_to_support` | text | 这个需求最终支持什么判断 |
| `urgency` | enum | P0 / P1 / P2 |
| `current_unknowns` | array | 当前未讲清楚的问题 |
| `status` | enum | draft / clarifying / ready_for_spec / closed |

### 第一阶段 AI 动作

- 从会议纪要或自然语言中自动抽出 `title / business_goal / request_type`
- 自动补 `current_unknowns`
- 自动判断是否能进入 `ready_for_spec`

## 2. BusinessFlow

表示：需求落在哪条业务链上。

### 最小字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `flow_id` | string | 唯一 ID |
| `intent_id` | string | 对应需求 |
| `flow_type` | enum | 归因 / 报表 / 漏斗 / 付费 / 透传 / 联调 |
| `upstream_nodes` | array | 上游节点 |
| `downstream_nodes` | array | 下游节点 |
| `critical_points` | array | 关键风险点 |
| `expected_consumers` | array | 谁消费这个链路结果 |

### 第一阶段 AI 动作

- 自动挂链路类型
- 自动给出典型风险点

## 3. TrackingSpec

表示：需求被翻译成的标准埋点规格。

### 最小字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `spec_id` | string | 唯一 ID |
| `intent_id` | string | 来源需求 |
| `version` | string | 当前规格版本 |
| `coverage_status` | enum | covered / partial / missing |
| `events` | array | 事件定义 |
| `fields` | array | 字段定义 |
| `entities` | array | 复用实体 |
| `client_scope` | array | 客户端负责内容 |
| `server_scope` | array | 服务端负责内容 |
| `transmission_requirements` | array | 必须透传的字段 |
| `business_usage` | array | 对应业务用途 |
| `known_gaps` | array | 已知缺口 |
| `status` | enum | draft / reviewing / approved / deprecated |

### events 子结构

| 字段 | 类型 | 说明 |
|---|---|---|
| `event_name` | string | 事件名 |
| `trigger_condition` | text | 触发条件 |
| `business_purpose` | text | 事件服务的业务用途 |
| `affected_flows_if_missing` | array | 缺失会伤到哪些链路 |

### fields 子结构

| 字段 | 类型 | 说明 |
|---|---|---|
| `field_name` | string | 字段名 |
| `field_type` | string | 字段类型 |
| `required_level` | enum | required / optional / conditional |
| `meaning` | text | 字段语义 |
| `source_side` | enum | client / server / sdk / attribution |
| `usage` | text | 被哪个分析 / 归因 / 报表使用 |
| `impact_if_missing` | text | 缺失后果 |

### 第一阶段 AI 动作

- 自动判断现有覆盖情况
- 自动生成事件和字段初稿
- 自动标记 `known_gaps`

## 4. AcceptanceCase

表示：如何判断 TrackingSpec 是否被正确实现。

### 最小字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `acceptance_id` | string | 唯一 ID |
| `spec_id` | string | 对应规格 |
| `validation_scope` | enum | debug / log / data_compare / report / callback |
| `rule_name` | string | 规则名 |
| `rule_description` | text | 规则说明 |
| `validation_method` | text | 校验方式 |
| `time_window` | text | 时间窗口 |
| `expected_result` | text | 预期结果 |
| `failure_impact` | text | 失败后果 |
| `manual_confirmation_required` | boolean | 是否要人工确认 |
| `status` | enum | draft / active / deprecated |

### 第一阶段 AI 动作

- 基于规格自动生成规则初稿
- 自动判断哪些规则可全自动判定

## 5. ExecutionResult

表示：一次执行或验收的结果。

### 最小字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `result_id` | string | 唯一 ID |
| `acceptance_id` | string | 对应验收对象 |
| `execution_context` | text | 执行环境 |
| `observed_result` | text | 实际观察到的结果 |
| `status` | enum | pass / fail / partial / unknown |
| `failed_rules` | array | 失败规则 |
| `needs_human_input` | boolean | 是否需要人工补充 |
| `one_line_summary` | text | 一句话总结 |
| `business_impact_summary` | text | 一句话业务影响 |

### 第一阶段 AI 动作

- 自动生成 `one_line_summary`
- 自动把失败规则翻译成 `business_impact_summary`

## 6. AmbiguityRisk

表示：当前会导致 delay 的灰区和歧义。

### 最小字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `risk_id` | string | 唯一 ID |
| `linked_object_type` | enum | intent / spec / acceptance / result / change |
| `linked_object_id` | string | 对应对象 |
| `risk_type` | enum | ambiguity / ownership / dependency / exception / version_gap |
| `description` | text | 风险说明 |
| `blocking_level` | enum | blocker / warning / watch |
| `owner_to_confirm` | string | 需要谁拍板 |
| `suggested_question` | text | 系统建议追问的问题 |
| `status` | enum | open / resolved / ignored |

### 第一阶段 AI 动作

- 自动产生 `AmbiguityRisk`
- 只把 blocker 级别抛给人

## 7. ChangeRecord

表示：版本变化及其后续动作。

### 最小字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `change_id` | string | 唯一 ID |
| `target_type` | enum | spec / acceptance / report |
| `target_id` | string | 变更对象 |
| `change_type` | enum | add / remove / modify / deprecate |
| `before_snapshot` | text | 变更前摘要 |
| `after_snapshot` | text | 变更后摘要 |
| `reason` | text | 变更原因 |
| `affected_roles` | array | 影响哪些角色 |
| `revalidation_actions` | array | 需要补验什么 |
| `status` | enum | draft / confirmed / synced |

### 第一阶段 AI 动作

- 自动生成变更摘要
- 自动生成补验动作

## 8. BusinessImpact

表示：一个异常或变更对业务的影响。

### 最小字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `impact_id` | string | 唯一 ID |
| `linked_result_id` | string | 来源结果 |
| `impact_type` | enum | attribution / report / funnel / payment / callback / trust |
| `affected_roles` | array | 影响哪些角色 |
| `severity` | enum | high / medium / low |
| `explanation` | text | 业务解释 |
| `decision_risk` | text | 对业务判断的风险 |
| `blocking_release` | boolean | 是否阻断发布 |

### 第一阶段 AI 动作

- 自动从结果生成影响解释
- 自动标记严重级别草案

## 9. 第一阶段最小对象关系

主链：

`BusinessIntent -> TrackingSpec -> AcceptanceCase -> ExecutionResult -> BusinessImpact`

辅助：

- `BusinessIntent -> BusinessFlow`
- `任何对象 -> AmbiguityRisk`
- `TrackingSpec / AcceptanceCase / ExecutionResult -> ChangeRecord`

## 10. 第一阶段为什么只定义这些字段

因为第一阶段目标不是建完整平台，而是先验证 3 件事：

1. AI 能不能把自然输入变成可执行对象
2. AI 能不能把规格变成规则和结果
3. AI 能不能把结果翻译成业务影响与后续动作

如果这 3 件事没成立，字段再多也没价值。
