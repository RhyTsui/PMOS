# 真实样例推演：新增分析需求

日期：2026-04-21

目的：用一个贴近真实业务的样例，跑一遍最小 AI-native 闭环。

## 1. 原始输入

假设业务方在群里提了这样一句需求：

> 想看巨量安卓渠道里，不同分包的新用户在新手引导第 1、2、3 步的流失情况，并且想知道这些新用户后续 7 日付费有没有差异。

## 2. AI 抽出的 BusinessIntent

### 2.1 需求标题

`巨量安卓分包新用户新手引导漏斗与 7 日付费差异分析`

### 2.2 业务目标

- 评估不同渠道分包的新用户质量。
- 判断新手引导流失是否和后续 7 日付费表现有关。
- 支持买量与产品优化决策。

### 2.3 需求类型

`新分析需求`

### 2.4 消费角色

- 投放 / 运营
- 数据分析师
- 产品

### 2.5 决策用途

- 哪些分包需要优化买量策略
- 哪些新手引导步骤需要重点优化

### 2.6 AI 自动识别的未知项

- “新用户”的定义是什么：首次启动、首次注册，还是首次进入引导？
- 新手引导第 1、2、3 步当前是否已有稳定埋点？
- 7 日付费是按自然日、注册后 7 日，还是激活后 7 日？
- 分包 ID 是否能稳定透传到后续付费链路？

## 3. AI 识别的 BusinessFlow

### Flow 1：新手引导漏斗链路

- 上游：启动 / 注册 / 新手引导步骤事件
- 下游：漏斗分析、流失分析

### Flow 2：付费归因链路

- 上游：激活 / 注册 / 分包标识 / 付费事件
- 下游：7 日付费对比

### Flow 3：报表消费链路

- 上游：埋点与透传
- 下游：BI 查询、渠道对比、版本对比

## 4. 现有规格覆盖判断

AI 针对现有规格做判断，得到：

### 已覆盖

- 启动事件
- 注册事件
- 付费事件

### 部分覆盖

- 新手引导步骤事件：已有事件，但步骤枚举和版本兼容性不明确

### 未覆盖 / 高风险

- 付费事件是否保留了激活时的分包 ID：未知
- 新手引导步骤事件是否统一定义“步骤完成”：未知
- 7 日付费时间窗口口径：未知

结论：

`部分覆盖，不能直接做可信分析`

## 5. AI 生成的 TrackingSpec 初稿

### 5.1 新增 / 修正规格目标

为了支持“分包新用户引导漏斗 + 7 日付费差异分析”，需要保证：

1. 新手引导步骤事件定义稳定。
2. 分包 ID 能从激活链路关联到后续付费。
3. 7 日付费统计口径明确。

### 5.2 事件规格初稿

#### 事件：`tutorial_step_complete`

- 触发条件：用户完成新手引导某一步骤时触发
- 业务用途：用于漏斗分析、流失定位、不同分包新用户质量分析
- 缺失后果：无法判断引导步骤流失差异，产品优化失去依据

#### 事件：`payment_success`

- 触发条件：支付成功并完成服务端确认后触发
- 业务用途：用于 7 日付费差异分析
- 高风险：若无法关联分包 ID，则无法做“分包新用户后续付费”分析

### 5.3 字段规格初稿

#### 字段：`package_id`

- 含义：买量分包标识
- 来源：激活 / 注册链路
- 用途：分包分析、渠道效果评估
- 风险：若后续付费链路丢失，付费归因和分包效果分析失真

#### 字段：`tutorial_step_index`

- 含义：新手引导步骤序号
- 用途：漏斗步骤切分
- 风险：若枚举不统一，不同步骤无法比较

#### 字段：`new_user_anchor_time`

- 含义：定义 7 日付费窗口的起始时间
- 用途：明确 7 日付费统计口径
- 风险：口径不清会导致报表不可复现

### 5.4 已知灰区

- “新用户”定义待产品拍板
- 引导步骤是否跨版本可比待确认
- 分包 ID 是否已在服务端付费事件中可追溯待确认

## 6. AI 生成的 AcceptanceCase 初稿

### 规则 1：新手引导步骤事件完整性

- 校验方式：debug + 日志抽样
- 预期结果：步骤 1 / 2 / 3 的事件都能稳定上报
- 失败后果：漏斗分析失真

### 规则 2：步骤枚举一致性

- 校验方式：日志比对
- 预期结果：同一版本内步骤定义一致，跨版本变更有 change record
- 失败后果：不同版本数据不可比

### 规则 3：分包 ID 透传完整性

- 校验方式：激活 -> 注册 -> 付费链路联查
- 预期结果：付费事件可关联回激活时的分包 ID
- 失败后果：7 日付费按分包分析失真

### 规则 4：7 日付费口径一致性

- 校验方式：规则定义比对 + 报表核验
- 预期结果：窗口口径明确且和报表一致
- 失败后果：业务结论不可复现

## 7. 假设执行结果

### 自动检查结果

- 规则 1：通过
- 规则 2：部分通过
  - 发现旧版本和新版本 `tutorial_step_index` 枚举逻辑不同
- 规则 3：失败
  - `payment_success` 无法稳定关联激活时的 `package_id`
- 规则 4：待人工确认
  - 7 日付费统计口径未在原需求中写清

## 8. AI 自动生成的结果解释

### 一句话总结

当前无法直接输出可信的“分包新用户引导漏斗与 7 日付费差异分析”报表。

### 业务影响解释

1. 因为付费事件无法稳定关联激活分包 ID，按分包比较 7 日付费会失真。  
2. 因为引导步骤枚举存在版本差异，跨版本对比漏斗会失真。  
3. 因为 7 日付费窗口口径未明确，后续结果无法被复现和确认。

### 对不同角色的解释

- 对运营 / 投放：当前看出来的“分包付费差异”可能是假的，不能直接拿来调预算。
- 对产品：当前漏斗结果不能直接用于判断哪个新手步骤有问题。
- 对研发：需要补分包 ID 关联链路和步骤枚举统一。
- 对数开 / 分析：当前报表口径不足以上线。

## 9. AI 生成的 AmbiguityRisk

### 风险 1：新用户定义歧义

- 类型：ambiguity
- owner：产品
- blocking_level：warning
- 建议问题：新用户是按首次启动、首次注册，还是首次进入引导定义？

### 风险 2：7 日付费窗口口径未拍板

- 类型：ambiguity
- owner：分析 / 产品
- blocking_level：blocker
- 建议问题：7 日付费窗口以哪个锚点时间计算？

### 风险 3：分包 ID 透传责任不清

- 类型：ownership
- owner：客户端 + 服务端
- blocking_level：blocker
- 建议问题：由哪一层保证付费链路可回溯分包 ID？

## 10. AI 生成的后续动作

### 给产品

- 确认新用户定义
- 确认 7 日付费窗口定义

### 给客户端 / 服务端

- 补齐 `package_id` 从激活到付费链路的可追溯性
- 统一 `tutorial_step_index` 枚举定义

### 给数开 / 分析

- 在需求放行前补充 7 日付费口径说明
- 在报表上线前重新校验规则 3 和规则 4

## 11. 这个样例说明了什么

这个样例里，系统做的不是：

- 把一句话改写成更漂亮的文档

而是：

- 自动发现分析需求背后的埋点缺口
- 自动指出哪些地方会让报表失真
- 自动把问题翻译给不同角色
- 自动生成下一步动作

这才是第一阶段真正值得验证的能力。
