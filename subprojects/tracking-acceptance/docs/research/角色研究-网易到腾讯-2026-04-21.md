# 角色研究：网易到腾讯

日期：2026-04-21

说明：本文件聚焦“真实工作模式、真实痛点、真实诉求”。  
其中：

- 来自本地业务文档和公开岗位描述的内容，按事实记录；
- 对角色日常协作模式的拼接与归纳，属于基于资料的推断。

## 1. 网易角色研究

### 1.1 网易数据分析师

#### 工作模式

- 负责 BI 看板、日常分析报告、行为数据整合、专题分析。
- 深入游戏玩法、商业化、运营活动，对留存、付费、转化、渠道质量做分析。
- 常常要和产品、运营、研发一起确认指标和口径。

资料依据：

- 公开岗位描述提到 `负责BI看板开发、分析报告、行为数据整合、为产品研发及发行提供数据支持`。
- 本地广告业务文档强调分析师经常被临时 SQL 配报表淹没。

#### 最烦什么

- 需求是“想看个报表”，但没人说清楚最终要支持什么决策。
- 埋点文档和报表口径脱节，导致分析做着做着发现数据不能用。
- 版本改了、字段变了、事件触发条件变了，没有同步通知。

#### 真正想要什么

- 新分析需求能快速判断“现有埋点够不够”。
- 如果不够，系统能自动指出缺什么事件、字段、透传信息。
- 异常结果能直接解释为业务影响，而不是给一堆字段错误。

#### 最容易 delay 的地方

- 需求说的是“分析目标”，文档给的是“埋点明细”，中间缺翻译层。
- 口径冲突没人拍板。
- 依赖研发和数开补字段，但影响范围不透明。

---

### 1.2 网易 BI / 数据产品

#### 工作模式

- 规划数据平台、埋点平台、数据地图、数仓与分析工具。
- 既要考虑平台能力，又要推动游戏项目接入与落地。
- 常常处在“中台规范”和“项目组灵活诉求”的冲突中。

资料依据：

- 公开岗位提到 `负责数据平台产品规划、设计与项目推进实施，如埋点平台、数据地图；负责埋点验证、数仓建设`。

#### 最烦什么

- 每个项目组都有自己的说法，规范难统一。
- 文档有人写，但没人持续维护。
- 平台提供能力了，项目组依然靠群聊和 Excel 推进。

#### 真正想要什么

- 一套统一对象模型，把需求、规格、验收、变更、影响串起来。
- 不是靠培训推动执行，而是系统自动收敛状态。
- 能快速发现复用机会，而不是每个项目从头来一遍。

#### 最容易 delay 的地方

- 平台和项目组之间的“最后一公里”。
- 规范设计出来了，但没人能低成本执行。
- 版本变化后补验动作没人主动维护。

---

### 1.3 网易游戏客户端

#### 工作模式

- 根据埋点需求实现客户端事件和属性上报。
- 参与联调、debug、SDK 集成、触发时机实现。
- 经常是第一个被问“为什么数据不对”的角色。

资料依据：

- 本地广告文档强调自动联调、SDK 日志检查、客户端事件上报。

#### 最烦什么

- 需求文档只有业务描述，没有可实现规格。
- 事件名、字段名、触发条件、边界场景不稳定。
- 联调阶段才被告知缺字段或口径不对。

#### 真正想要什么

- 一份可以直接映射到实现的规格。
- 能在开发 / debug 阶段尽早看到错误样例和缺口。
- 哪些必须客户端负责，哪些是服务端透传，写清楚。

#### 最容易 delay 的地方

- 需求灰区：到底什么时候触发算“注册成功”“新手引导完成”。
- 客户端已经发了，但服务端或 SDK 没串起来。
- 需求中没有明确“错了会影响什么”，优先级难判断。

---

### 1.4 网易游戏服务端

#### 工作模式

- 承接服务端事件、业务日志、透传信息、订单与回传逻辑。
- 负责把客户端、归因链路、支付 / 业务系统串起来。

#### 最烦什么

- 客户端和服务端对同一字段的语义理解不一致。
- 客户端没透传关键字段，服务端也没被提前告知后果。
- 出了问题所有人都说“数据不对”，但没人指出是断在哪一段。

#### 真正想要什么

- 明确的透传规格。
- 明确的链路责任图。
- 一旦透传缺失，系统能直接指出“会影响哪个下游”。

#### 最容易 delay 的地方

- 透传职责没有结构化记录。
- 一条字段看似不重要，但其实会伤到付费归因或分包分析。
- 问题往往到 BI 层才暴露，回头修链路成本很高。

---

### 1.5 网易 SDK / 数据中台

#### 工作模式

- 提供 SDK、埋点接入、数据入库、debug、实时错误、数据治理能力。
- 支撑多个项目组反复接入和验收。

资料依据：

- 本地广告文档和三方平台资料都指向：SDK / 中台是埋点接入和验收的承载层。

#### 最烦什么

- 每个项目组都重复问类似问题。
- 没有统一规格时，debug 也只能帮忙看表面错误。
- 中台承担了重复解释劳动，却很难沉淀成系统资产。

#### 真正想要什么

- 把项目组的“口头经验”转成复用规则和模板。
- 把 failed events、debug 样例、验收报告沉淀成可搜索资产。
- 项目状态和风险不是靠中台同学人肉追。

#### 最容易 delay 的地方

- 项目方以为“接了 SDK 就完了”。
- 数据错误来自规格问题，但最后又压回到中台排查。

---

### 1.6 网易游戏运营 / 投放

#### 工作模式

- 看渠道效果、活动效果、素材表现、用户转化、付费结果。
- 提需求时更关注“业务结果能不能看”，而不是技术实现细节。

#### 最烦什么

- 报表有数，但不知道能不能信。
- 一旦异常，排查时间长，且听不懂技术术语。
- 想提一个新分析需求，却没人能快速回答现有埋点是否支持。

#### 真正想要什么

- 结果可信。
- 异常有一句话业务解释。
- 新需求能迅速知道“可做 / 不可做 / 缺什么”。

#### 最容易 delay 的地方

- 业务提问很自然，但系统无法自动翻译成埋点 / 验收需求。
- 需求被转手多次后失真。

---

### 1.7 网易项目推进者 / 项目管理

#### 工作模式

- 在多个需求方、多个研发团队、多个数据角色之间推进。
- 需要知道当前卡点、依赖、确认项、版本影响。

#### 最烦什么

- 状态散落在群聊、纪要、文档、工单、口头同步里。
- 需要专人维护项目状态，维护成本高。
- 最大问题不是没人做，而是没人知道“现在到底卡在哪”。

#### 真正想要什么

- 不额外新增维护成本的状态面板。
- 只把真正需要拍板的问题推给人。
- 一眼看到灰区、歧义、依赖、变更影响。

#### 最容易 delay 的地方

- 责任未明确。
- 歧义未暴露。
- 变更后无人补验。

## 2. 腾讯对照

### 2.1 腾讯数据分析 / 商业分析角色

资料显示，腾讯相关岗位普遍强调：

- 全生命周期数据跟踪与分析；
- 漏斗、留存、收入、付费转化分析；
- 买量渠道质量与 ROI 路径优化；
- 埋点、采集、清洗、可视化全流程；
- 与产品、运营、技术多部门协作。

#### 说明

这说明腾讯系角色和网易系角色在问题结构上高度一致：

- 数据分析不是只看结果，而是参与埋点、指标和验证；
- 角色跨部门协作是常态，而不是例外；
- 数据问题天然就是“业务+技术+协作”的复合问题。

### 2.2 网易与腾讯的共性

| 维度 | 共性 |
|---|---|
| 数据分析角色 | 不只是出报表，而是深入玩法、商业化、活动与埋点 |
| 数据产品角色 | 不只是建平台，还要推进项目接入与治理落地 |
| 技术角色 | 经常被需求歧义和联调延迟拖累 |
| 业务角色 | 关注的是结果是否可信，而不是字段本身 |
| 项目管理角色 | 关注的是灰区、依赖、责任、变更，而不是仅任务状态 |

### 2.3 对产品设计的直接启发

如果两家大厂的角色结构都类似，那么最优解大概率也不该偏某一方。  
真正要做的是一层“多角色公共语义层”，把：

- 业务需求
- 埋点规格
- 验收规则
- 结果解释
- 版本影响

统一起来。

## 3. 当前结论

从网易和腾讯对照看，这类问题不是个别公司的流程问题，而是游戏 / 广告 / 数据协作场景下的结构性问题。

所以系统必须优先服务这些跨角色共性诉求：

1. 需求讲清楚
2. 规格能实现
3. 验收能自动
4. 结果能解释
5. 变更能追踪
6. 灰区能提前暴露
