# AI-native 解决路径研究

日期：2026-04-21

## 1. 研究边界

本轮明确排除：

- Jira / 禅道 / Confluence 一类传统项目管理与文档系统作为主解
- “只是把现有流程搬到新系统”的轻量改造
- 依赖专职 PM 长期手工维护状态、字段和流程的方案

本轮只看：

- AI 如何自动结构化需求与规格
- AI 如何自动形成项目状态与控制面
- AI 如何自动发现灰区、歧义、依赖、变更影响
- AI 如何自动生成补验、通知、追踪动作

## 2. 这类问题为什么必须 AI-native

用户给出的核心约束是：

1. 不能再引入一套重流程
2. 不能新增太多维护动作
3. 新问题超过 3 个，方案判失败

因此可行解必须同时满足：

- 从现有工作产物中自动长出结构化状态
- 只向人索取必须拍板的信息
- 让灰区和歧义显式化，而不是隐藏在群聊与口头沟通里
- 能持续跟随需求演进，而不是一次性文档整理

## 3. AI-native 方案空间

### 3.1 AI-native 项目控制面

代表参考：

- Plane AI  
  官方定位是 AI-native project management，把项目、文档、AI workflow 放在一个 workspace 里，并强调：
  - AI 读取完整工作区上下文
  - agent 可被分配到 work item
  - 自动 triage、owner 分配、blocker 跟踪、状态汇总
  - 能从 Slack / Teams 等对话入口接工作

对本项目的启发：

- 项目状态不应依赖手工录入，而应从现有 docs、threads、work items 自动汇总
- AI 可以不只是“写总结”，而是真正承担 triage、blocker 提示、状态更新
- 但 Plane 仍以 work item 为中心，业务语义层不够深，不能直接满足“埋点规格 / 验收 / 影响”这类对象建模需求

### 3.2 AI-native 规格驱动开发

代表参考：

- Spec Native  
  强调 specification-driven software development，即：
  - 先有 living specification
  - 规格同时作为文档、验证层和生成层
  - AI 始终基于规格生成架构、代码、测试

- SENAR  
  强调 AI 开发时代需要“监督式工程标准”，防止 vague prompt -> plausible wrong output 的路径，要求有 gate、规则、记录和监督

对本项目的启发：

- 埋点需求不应继续是“说明文档”，而应升级为 living specification
- 文档、验收规则、测试、报告都应从同一规格长出来
- 只要规格不结构化，AI 自动化一定会漂

### 3.3 AI-native 需求生成 / 歧义消解

代表参考：

- IdeaForge  
  核心思路是先 interview，再生成工程规格，而不是直接 prompt AI

- resetDocs / Requirements AI / EltegraAI  
  共同点是从 Slack 线程、会议纪要、对话记录自动抽出 requirements / PRD / acceptance criteria

对本项目的启发：

- 这类需求不是靠一次 prompt 写出来，而是需要 AI 追问、补问、显式标出 gaps
- “把原始需求直接整理成规格”还不够，必须内置 ambiguity detection
- 对埋点场景尤其重要，因为灰区和歧义就是 delay 的来源

### 3.4 AI-native 控制面 / 可观测运行面

代表参考：

- MUTX AI Agent Control Plane  
  核心强调：
  - control plane is the product
  - runtime visibility
  - agent lifecycle durable record
  - operator-facing surface

- Upbound AI-native control plane  
  关注点是：
  - control plane model
  - IDE / project tooling
  - resource schemas
  - AI-powered authoring

对本项目的启发：

- 你这个产品不应该只是“生成内容”，而应该把控制面本身当成产品
- 需要 durable record：谁改了、谁确认了、哪个版本在用、哪个工具链/规则版本生效
- 需要 operator surface：出了问题后，人能从控制面倒推出哪里断了

### 3.5 AI-native 数据平台 / 埋点治理

代表参考：

- 数数科技 Thinking Analytics / Gaia
  官方已经把这些能力做出来：
  - 元数据管理
  - 数据集成
  - 埋点方案与数据验收
  - 实时上报明细与错误原因
  - Debug 模式
  - 数据治理与追溯
  - 投放数据深度回传

对本项目的启发：

- “埋点方案 + 数据验收 + 实时错误 + 追溯”是行业验证过的能力底座
- 但这类方案更像数据平台，并不天然等于“AI 语义型项目控制面”
- 可作为底座参考，不应作为最终范式照搬

## 4. 对本项目最重要的设计推导

### 4.1 主体不是“项目管理系统”

更准确的产品定义是：

`AI 驱动的语义型项目控制面`

它不是围绕 issue / status / assignee 建模，而是围绕：

1. 业务需求对象
2. 埋点规格对象
3. 验收对象
4. 变更对象
5. 影响对象

### 4.2 AI 的职责不是替人填表，而是替人维护语义一致性

AI 应该负责：

- 从会议纪要、群聊、需求文档、版本差异中自动抽取对象
- 自动识别冲突、缺失、灰区
- 自动生成埋点规格初稿
- 自动生成验收规则初稿
- 自动把执行结果翻译成业务影响
- 自动形成补验、通知、追踪任务

人工只负责：

- 业务拍板
- 灰区裁决
- 最终发布与例外确认

### 4.3 第一阶段最值得做的是“嵌入式 AI”

不能要求团队切换到一套新的重平台。  
第一阶段更现实的方式是：

- 保留现有工作入口：会议、文档、群聊、联调、平台记录
- AI 在背后自动结构化
- 只把必须确认的问题推给人

这比“定义新流程、强推新系统”更可能成功

## 5. AI-native 最优解应该长什么样

从目前研究看，最优解不应是单个产品，而应是一个组合范式：

### 层 1：语义采集层

- 输入：会议纪要、埋点文档、群聊、需求变更、联调记录、执行结果
- AI 做：抽取需求、规格、依赖、风险、待确认项

### 层 2：对象模型层

- 统一对象：业务需求、埋点规格、验收规则、变更、影响、结论
- 这一层是 single source of truth

### 层 3：控制面层

- 自动显示当前状态
- 自动显示 blocker、灰区、责任人、影响范围
- 自动汇总给不同角色

### 层 4：执行层

- 自动生成文档
- 自动生成规则
- 自动触发检查
- 自动生成报告
- 自动通知到研发本地 / 数据开发平台 / 机器人

### 层 5：演化层

- 新报表 / 新分析需求能反推埋点需求初稿
- 新的异常和人工裁决能回流优化规则与模板

## 6. 这轮研究后的判断

### 应该直接 pass 的方向

- 传统 PM 系统做主控
- 传统 wiki 做主文档
- 靠专职 PM 手工维护所有状态
- 靠人工补全多份文档与多份表单

### 应该继续深挖的方向

1. AI 需求面谈与歧义消解
2. living specification 驱动的埋点需求体系
3. AI 项目控制面与 operator surface
4. AI 驱动的任务 triage / blocker 暴露 / 状态汇总
5. 数据平台能力与项目控制面的耦合方式

## 7. 下一轮研究建议

1. 继续横向对比 AI-native 产品与方法论
2. 深挖“需求对话 -> 规格 -> 验收 -> 报告 -> 项目状态”这条链
3. 提炼一份 `AI 语义型项目控制面` 的对象模型
4. 设计“如何不新增流程，只嵌入现有流程”的最小落地方案
