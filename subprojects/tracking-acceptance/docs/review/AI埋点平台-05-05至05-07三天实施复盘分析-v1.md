## 1. 三天实施的真实结果

这三天在 `tracking-acceptance` 项目中，真实完成的不是单个页面，而是一条从 0 到 1 的完整链路：

1. 从原始业务想法收口为正式需求
2. 从需求继续拆到原型、交互、UI、前端交付
3. 从单页尝试扩展到整套页面
4. 从页面实现反推会议纪要、需求池、主真源、交互文档、设计规范和整改清单
5. 从项目内试错抽出了平台级可复用经验

因此，这次项目的价值不只是“把页面做出来”，而是验证了：

- AI 产品交付链是否能被跑通
- 哪些环节必须显式化
- 哪些流程必须平台化
- 哪些工具值得保留、哪些不应神化

## 2. 这次实施真正跑通了哪些链路

### 2.1 需求治理链跑通了

这次不再只是“有会议纪要、有聊天、有页面修改”，而是逐步建立了：

- 需求池
- 主真源
- 交互承接
- 设计规范
- 页面整改清单
- 任务表 / 研发任务单

最关键的变化是：

- 反馈不再直接跳到实现
- 反馈先进入需求池，再决定是否进入真源、页面和任务单

这是从“对话驱动修改”升级到“有治理的产品迭代”的关键一步。

### 2.2 设计确认链跑通了

这次项目在设计层最大的进步，是把原来混杂的“设计稿”拆成了多层：

- 原型设计稿
- 交互设计稿
- CodexUI设计初稿
- 高保真视觉稿
- 前端交付页

这解决了一个长期问题：

- 以前所有设计产物都容易被混叫为“设计稿”
- 导致无法判断哪份是结构、哪份是行为、哪份是视觉、哪份是交付

现在这条链虽然还没有在所有项目里平台化，但至少在本项目里已经被验证可用。

### 2.3 前端交付链跑通了

这次明确验证了：

- 最终交付物不应是示意稿
- 也不应只是静态 HTML
- 而应是“带真实前端交互、未接后端”的最终交付前端页

这意味着：

- mock 数据是允许的
- 后端接口可以后接
- 但前端结构、组件语义、交互行为都应尽量真实

这是从“设计演示”跨到“工程交付”的关键分水岭。

### 2.4 页面整改机制跑通了

这次还验证了一个很重要的方法：

`页面整改不应直接 patch 页面，而应先诊断差异，再整改。`

也就是：

1. 看现状
2. 对照设计规范与需求
3. 明确差异
4. 再改页

这件事后面已经被写进：

- 主真源
- 前端交互文档
- 通用设计规范
- 页面整改清单

## 3. 本次实施暴露出的关键问题

### 3.1 没有需求池时，反馈会散掉

三天里非常明确地暴露出：

- 有会议纪要
- 有用户反馈
- 有 UI 反馈
- 有页面修改

但如果没有项目级常设需求池，这些内容会：

- 散在聊天里
- 散在页面里
- 散在单次会议纪要里
- 最后无法判断哪些已经升级成正式需求

这就是为什么后来必须补：

- 项目常设需求池总表
- 会议反馈转需求池
- UI 反馈转需求汇总

### 3.2 设计工具不是流程本身

Figma 和 Pixso 都被尝试过，但结果很清楚：

- 工具可以辅助
- 工具不是流程
- 工具更不能替代真源

这次的实际经验是：

- Figma 更适合视觉稿与设计协作
- Pixso 在当前环境下更适合辅助与局部同步
- 两者都不能天然成为产品真源
- 真正稳的是：
  - 文档定义
  - 工具辅助
  - 页面交付

### 3.3 如果不明确“最终交付物”，页面会一直漂

三天里有过多次来回：

- 示意稿
- 静态稿
- HTML 确认稿
- 前端实现

最终才收口成：

`最终交付产品前端页`

这说明：

- 如果不先定义“最后交给后端的是什么”
- 页面就会一直在设计稿和实现稿之间摇摆

### 3.4 组件库与设计规范必须先确认

还有一条经验很明确：

- 如果组件基线没定
- 如果设计规范没定
- 过早宣布页面“定稿”会造成大量返工

这次项目后半段之所以能逐步稳定，是因为后面补齐了：

- YokaUI / 内部组件语义
- Ant Design 兜底边界
- 通用设计规范
- 页面整改清单

### 3.5 inbox 作为原始输入区必须隔离

中途出现过一次对 `inbox` 文件处理不当的问题。  
这件事虽然小，但很重要，因为它暴露出：

- 原始输入区如果被污染
- 后续所有转换、模板、页面引用都会失去基线

因此这次项目明确沉淀出：

- `docs/sources/inbox/` 只做原始输入区
- 一切处理、转换、导入、前端引用都基于副本

## 4. 这次实施中最有效的方法

### 4.1 先做对象和链路，再做页面

有效原因：

- 页面不会反复失焦
- 页面修改始终挂在产品对象上

### 4.2 逐页确认，而不是试图一次性收所有页面

有效原因：

- 能先用概览页试出方法
- 方法跑通后再扩到其他页

### 4.3 文档和页面同步推进

有效原因：

- 页面不是黑箱实现
- 文档不是空转

### 4.4 会议纪要必须转需求、再转任务

有效原因：

- 会议结论不再只停在会议
- 能变成实际推进主线

### 4.5 UI 反馈必须进入正式文档，而不只是设计附件

有效原因：

- 后续页面整改有共同依据
- 不再只有“设计师说过”

## 5. 这次实施中无效或高返工的方法

### 5.1 直接把用户一句话想法翻成实现

问题：

- 容易跳过边界澄清
- 后期返工大

### 5.2 把所有稿件都混叫“设计稿”

问题：

- 无法判断责任边界
- 无法判断是否已确认

### 5.3 还没接组件库就宣布最终稿

问题：

- 页面和后端可还原标准不一致

### 5.4 把工具当成真源

问题：

- Figma/Pixso 一旦漂，产品真源就会失控

### 5.5 试图用更多说明文案补救结构问题

问题：

- 页面会越来越重
- 不会真正解决结构问题

## 6. 这次项目沉淀出来的可平台化经验

### 6.1 高优先级平台化经验

1. 每个活跃项目必须维护常设需求池
2. 会议纪要/反馈必须先入需求池，再进真源与任务链
3. 设计确认链必须分层
4. 最终交付前端页的标准必须写清
5. inbox 原始输入必须与工作副本隔离
6. 页面整改默认先差异诊断再修改
7. 页面整改清单应作为长期对象存在

### 6.2 中优先级平台化经验

1. 设计工具引入规则
2. 组件库优先级与兜底边界
3. UI 反馈如何转为正式产品规则

### 6.3 暂不宜直接平台化的部分

1. Figma 与 Pixso 的具体选型
2. 每页是否必须有高保真设计稿
3. 某些页面的视觉装饰程度
