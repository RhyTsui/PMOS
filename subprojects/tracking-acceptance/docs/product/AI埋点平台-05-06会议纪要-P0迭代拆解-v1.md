## 1. 文档目标

本文档只做一件事：

- 把 05-06 会议纪要里已经确认需要进入当前开发主线的 `P0` 事项，拆成可直接排期、可直接映射页面和对象、可直接验收的最小迭代项

本文档不替代主真源，职责是：

1. 明确 P0 分母
2. 明确每一项的对象边界
3. 明确页面落点
4. 明确前后依赖
5. 明确最小验收点

## 2. P0 总体判断

当前最需要先完成的，不是“把所有页面都补更多细节”，而是先把以下 6 条能力主线闭合：

1. `验收版本成为一级治理对象`
2. `概览页只做同步，不做明细堆砌`
3. `报告必须区分简报与详细报告`
4. `简报必须暴露核心未通过事件`
5. `模板输入正式统一为 xlsx`
6. `失败批次支持一键重试`

这 6 项都是 `P0`，因为它们共同决定了系统能不能从“能看页面”进入“能真正组织验收工作”。

## 3. P0 迭代拆解

### P0-1 验收版本成为一级治理对象

#### 要解决的用户问题

- 用户无法明确区分当前在看哪个验收范围
- 需求、规则、报告、渠道包容易混在一起
- 多轮验收无法稳定归档和回查

#### 产品对象

- `验收版本`
- `验收周期`
- `版本目标`
- `验收范围`
- `版本状态`

#### 页面落点

- `概览`
- `埋点需求 -> 验收版本`
- `数据字典`
- `校验规则`
- `实时查询`
- `验收报告`

#### 页面要求

- 所有核心页面必须能明确知道当前挂在哪个验收版本下
- 验收版本必须作为主入口对象，而不是辅助筛选条件
- 版本切换后，范围、规则、报告、查询上下文必须同步切换

#### 依赖关系

- 先于模板导入细化
- 先于报告归档细化
- 先于规则实例绑定细化

#### 最小验收点

- 能新建/编辑/删除验收版本
- 每个版本有独立的周期、目标、范围
- 同一项目下至少支持 2 个版本并存且互不串数据
- 页面切换版本后，需求/规则/报告视图随之变化

### P0-2 概览页只做同步，不做明细堆砌

#### 要解决的用户问题

- 当前用户进入首页后，无法快速知道当前版本状态和卡点
- 首页如果堆过多明细，会稀释同步效率

#### 产品对象

- `当前验收版本`
- `版本目标`
- `验收范围`
- `关键进度指标`
- `更新动态`
- `趋势`
- `关键风险`

#### 页面落点

- `概览`

#### 页面要求

- 首页必须首先回答“当前版本状态如何”
- 只展示有限的高优信息和 Top 问题
- 详细问题通过跳转页承接，不在首页平铺

#### 依赖关系

- 依赖 P0-1 中的版本对象稳定

#### 最小验收点

- 首页能明确显示当前验收版本、周期、范围
- 首页能显示有限的关键统计
- 首页只显示有限的问题摘要，不展开大量明细
- 首页的“更多”链路能跳转到下游模块

### P0-3 报告必须区分简报与详细报告

#### 要解决的用户问题

- 同步群里需要快速结论，但当前详细报告太重
- 深入排查又不能只看简报

#### 产品对象

- `验收简报`
- `详细报告`
- `报告配置`

#### 页面落点

- `验收报告`

#### 页面要求

- 报告列表中每个批次都能查看简报/详细报告
- AI 报告配置至少分成两部分提示词：
  - `简报提示词`
  - `详细报告提示词`

#### 依赖关系

- 依赖 P0-1 中的版本归档
- 依赖报告明细数据可追溯

#### 最小验收点

- 同一批次有“简报”和“详细报告”两种表达
- 简报适合同步和推送
- 详细报告适合查看完整明细
- 分享链路可区分简报与详细报告

### P0-4 简报必须暴露核心未通过事件

#### 要解决的用户问题

- 只看通过率和总数，无法快速知道真正卡住了什么

#### 产品对象

- `核心未通过事件`
- `未上报明细`
- `未通过明细`
- `未通过规则`

#### 页面落点

- `概览`
- `验收报告`

#### 页面要求

- 简报里至少列出本轮最关键的未通过事件
- 概览页的风险摘要与报告简报口径一致

#### 依赖关系

- 依赖规则结果可归因到事件
- 依赖报告对象区分“未上报 / 未通过 / 已通过”

#### 最小验收点

- 简报不是只有比例和数量
- 能看到核心未通过事件列表
- 至少能覆盖高优问题

### P0-5 模板输入正式统一为 xlsx

#### 要解决的用户问题

- 导入模板格式不统一，容易在 csv / xlsx 之间反复漂移
- sheet 结构不稳定，导致导入校验不可信

#### 产品对象

- `埋点需求模板`
- `数据字典模板`
- `sheet 结构`
- `上传覆盖更新任务`

#### 页面落点

- `埋点需求`
- `数据字典`

#### 页面要求

- 所有正式模板下载统一为 xlsx
- 上传校验默认按 xlsx 结构校验
- sheet 结构必须稳定可解释

#### 依赖关系

- 依赖对象结构冻结：渠道包、事件、步骤、参数、数据字典

#### 最小验收点

- 页面下载的是 xlsx，不再默认 csv
- 上传入口按 xlsx 工作
- 模板至少能承载：渠道包、事件、步骤、参数、数据字典

### P0-6 失败批次支持一键重试

#### 要解决的用户问题

- 失败后重复操作成本高
- 用户需要重新手动筛选失败对象，效率低

#### 产品对象

- `验收批次`
- `失败对象集合`
- `重试任务`

#### 页面落点

- `验收报告`

#### 页面要求

- 历史失败批次支持直接复用失败对象集合
- 用户只需要补充必要条件后即可重新发起

#### 依赖关系

- 依赖报告明细可定位失败对象
- 依赖发起验收入口支持从批次继承对象范围

#### 最小验收点

- 失败批次存在 `一键重试`
- 不需要重新逐个勾选失败事件
- 可修改账号、设备、时间范围后再发起

## 4. P0 页面映射

| 页面 | 本轮 P0 必须承接 |
| --- | --- |
| 概览 | 当前验收版本、版本目标、验收范围、关键指标、更新动态、趋势、核心风险 |
| 埋点需求 | 验收版本对象管理、xlsx 模板导入承接 |
| 数据字典 | xlsx 模板导入承接、版本维度下的数据字典查看 |
| 校验规则 | 规则继续按版本绑定，服务于后续报告与失败归因 |
| 实时查询 | 继续消费版本维度与规则结果，不作为本轮 P0 主改动中心 |
| 验收报告 | 发起验收、简报/详细报告、一键重试、失败对象归因 |
| AI 助手 | 暂不作为本轮 P0 主改动中心，但后续要消费报告与规则能力 |

## 5. P0 依赖顺序

建议开发顺序：

1. `验收版本对象稳定`
2. `埋点需求 / 数据字典 xlsx 承接统一`
3. `报告结构切成简报 + 详细报告`
4. `核心未通过事件进入简报`
5. `失败批次一键重试`

原因：

- 没有版本对象，后面的规则、报告、模板都会继续混
- 没有模板统一，结构化输入就不稳定
- 没有报告分层，就无法高效同步
- 没有失败重试，验收闭环就不完整

## 6. P0 完成判定

本轮 P0 只有在以下条件同时满足时，才算完成：

- 验收版本是一级治理对象，并贯穿页面与数据对象
- 首页只承担同步职责，不回到明细堆砌
- 报告明确区分简报与详细报告
- 简报能列出核心未通过事件
- 正式模板输入统一为 xlsx
- 失败批次支持一键重试

若只是“页面已有入口”，但以上能力没有形成稳定对象和闭环，则判定为：`partial`
