## 12.13 系统承载方式纠正

对 05-06 会议后的产品判断作如下正式纠正：

- 当前并未确认在旧系统或现有平台内直接开发
- 当前默认口径是：`按新系统规划设计`
- 是否复用旧系统的登录、权限、用户体系或底层能力，后续在实施方案阶段再单独评估

因此，从现在起：

- 不再使用“优先在现有平台内开发”作为当前生效判断
- 不再使用“默认嵌入旧系统”作为页面、权限、导航或系统边界前提
- 所有页面规划、对象建模、规则设计、报告设计，统一按新系统形态继续推进

## 12.14 05-07 UI设计反馈承接补充

本节用于正式承接 2026-05-07 的 UI 设计反馈。该批反馈不是独立的新业务需求，而是：

- 对既有页面进行设计规范补强
- 对组件复用边界进行收口
- 对概览页、实时查询、验收报告、数据字典等页面给出更稳的视觉与信息层级要求

因此，本节生效后，后续页面整改与前端实现默认遵守以下口径。

### 12.14.1 全局设计规范正式成为整改依据

当前项目页面整改正式以 UI 设计规范为依据推进，默认遵守：

- 延续 B 端企业后台风格，不重造新视觉语言
- 优先复用内部组件库与 YokaUI 语义
- 页眉、筛选区、表格区、操作区、卡片区保持统一规范
- 不再通过增加解释性文案来弥补信息层级问题

这意味着：

- 页面优化优先是规范化，而不是自由发挥
- 后续产品、设计、前端、AI 改页都应先参考设计规范，再进入实现

### 12.14.2 页面整改默认先做差异诊断

后续页面调整默认采用：

`现状诊断 -> 与设计规范比对 -> 明确差异 -> 再修改页面`

不再默认直接改页面样式。

该规则用于避免：

- 局部越改越散
- 页面之间风格继续漂移
- AI/Codex 在没有规范边界时自由创造一套新风格

### 12.14.3 概览页继续保持同步页定位

UI 反馈未改变概览页定位，正式确认：

- 概览页仍然是同步页
- 不是明细堆砌页
- 不是实时监控页

因此概览页继续只承载：

- 版本目标
- 核心指标
- 最近更新
- 验收趋势
- 不通过埋点
- 未上报埋点

不回退到大面积说明文案或明细平铺。

### 12.14.4 实时查询正式固定为三段式

实时查询页正式固定为：

1. 筛选条件
2. 分布区域
3. 日志明细

产品定位继续维持：

- 这是查询执行页
- 不是实时监控大盘
- 分布图只服务于查询证据，不升级为监控首页

### 12.14.5 验收报告正式固定为简报与详细报告双层

UI 反馈与会议纪要口径一致，正式确认：

- 列表页默认承载简报卡片
- 详情页承载详细报告
- 简报用于同步
- 详细报告用于查阅与追责

因此后续报告整改与实现继续遵守：

- 简报卡片优先暴露核心结论
- 详细报告优先暴露结构化明细
- 不再把报告页重新拉回普通列表页

### 12.14.6 数据字典继续按参数对象化管理

UI 反馈确认当前方向正确，数据字典继续按以下结构管理：

- 参数主表
- 枚举值展开子表
- 参数详情抽屉

并继续坚持：

- 页面内以查阅、联动、上传覆盖更新为主
- 不做在线 Excel 式逐条编辑器

### 12.14.7 当前不建议继续修改的方向

从产品层正式明确，以下方向当前不建议继续扩散：

- 为了“更好看”重造新的视觉语言
- 再向概览页和报告页补说明性废话
- 把实时查询升级成实时监控平台
- 把数据字典和规则页做成在线 Excel 编辑器
- 牺牲后台可读性去追求装饰性卡片效果

### 12.14.8 当前判断

- UI 设计反馈已进入正式产品承接：`solved`
- UI 设计规范已成为页面整改依据：`solved`
- 所有页面已完全冻结为最终设计稿：`partial`

原因：

- 设计规范边界已经进入产品真源
- 页面整改方向已经收口
- 但逐页最终设计冻结仍需继续通过页面与交互承接文档落地
