## 1. 这批反馈的性质判断

这批输入不应被当成：

- 已冻结的完整视觉定稿
- 可以逐条机械照搬的实现清单

更准确的性质是：

`UI设计师对当前系统的规范补充 + 组件使用约束 + 典型页面优化建议 + 概览页示意参考`

因此它的产品化处理方式应为：

1. 提取正式设计规范
2. 提取必须进入需求池的页面与交互要求
3. 区分：
   - 已确认应吸收
   - 建议吸收
   - 待确认
   - 不建议继续修改

## 2. 这批输入真正解决的产品问题

从资深产品经理视角，这批反馈真正要解决的是 4 类问题：

1. 当前页面虽然功能在落地，但视觉与组件规范没有统一
2. 同类页面缺少一致的信息层级、表格规范、筛选规范和页眉规范
3. AI/Codex 改页面时没有足够强的设计边界，容易自由发挥
4. 当前系统还没有把“UI 设计规范”正式转成产品需求与设计承接规则

## 3. 已确认应吸收的设计与产品要求

以下内容应视为正式吸收：

### 3.1 全局设计语言

1. 继续沿用当前 B 端后台蓝白体系，不重造视觉风格
2. 使用既有规范中的：
   - 主色
   - 状态色
   - 中性色
   - 表格背景与 hover 色
3. 页面名称、辅助说明、模块标题、正文、表格、弱提示维持统一层级
4. 后续 AI 改页、前端改页、设计复核都应优先引用这套规范

### 3.2 组件复用要求

1. 优先复用内部组件库与 YokaUI 语义
2. 表格、筛选、日期、表单、分页、卡片、按钮、提示、空状态，禁止再随意新造样式
3. 同类组件在不同页面中必须统一
4. 页面优化不得改变现有业务逻辑、字段含义和模块关系

### 3.3 页面布局与层级要求

1. 页眉高度、标题区、筛选区、卡片区、表格区之间间距要统一
2. 页面结构优先按：
   - 标题/页眉
   - 筛选/操作
   - 主体内容
   这条链组织
3. 概览页、实时查询、验收报告等页面不得继续堆解释性文案

### 3.4 筛选区规范

1. 顶部筛选应优先做成紧凑、统一的一组过滤工具条
2. 时间筛选、事件筛选、字段筛选、日志接收等筛选项应同高同层级
3. 查询、重置、导出等动作要统一位置
4. 支持模糊匹配、关键词查询、必选校验等交互

### 3.5 表格规范

1. 表格作为系统主载体，必须统一：
   - 表头
   - 行高
   - 分页
   - 操作列
   - 文本截断
   - 局部滚动
2. 不允许再把横向滚动条挂到浏览器全局
3. 操作型表格的操作区和开关列要固定策略

### 3.6 概览页示意图吸收结果

`1.png` 传达的重点应吸收为：

1. 顶部深蓝一体化导航
2. 左侧菜单 + 中间内容区的后台式布局
3. 概览页首屏用：
   - 版本目标
   - 核心指标
   - 最近更新
   - 验收趋势
   - 不通过埋点
   - 未上报埋点
4. 概览页仍然是同步页，不是明细堆砌页

## 4. 建议吸收，但应按产品边界实现的要求

### 4.1 数据字典管理方式

UI 视角建议的数据字典更像：

- 参数主表
- 枚举值子层
- 详情抽屉

产品判断：

- 这条已被吸收，而且方向正确
- 但不建议做成在线逐条编辑器
- 保持“查阅 + 展开 + 详情 + 上传覆盖更新”更稳

### 4.2 验收报告的简报视图

UI 反馈倾向于用卡片化简报承载报告列表。

产品判断：

- 这条应继续吸收
- 卡片展示的应该是“简报”
- 详细报告仍应在详情页承接

### 4.3 实时查询的布局和图表呈现

UI 反馈强调：

- 筛选条件
- 分布区
- 明细区

应显式分层。

产品判断：

- 这条应吸收
- 但系统定位仍不是“实时监控”
- 因此分布图只服务查询证据，不应升级成监控大盘

## 5. 待确认事项

以下内容当前不宜直接视为已冻结，需要继续按页面确认：

1. 所有页面是否都要完全按 Pixso 资源库一比一还原
2. 某些业务页是否要继续增强阴影、卡片装饰和视觉层次
3. 概览页以外的页面是否需要更多“卡片式切换”而不是纯列表结构
4. 某些筛选/表单交互到底优先内部业务组件，还是允许 Ant Design 兜底

## 6. 不建议继续改的地方

从资深产品经理角度，这些方向不建议继续扩散：

1. 继续增加说明性废话文案
2. 继续为局部页面自由创造一套新风格
3. 只追求视觉效果而牺牲后台信息密度和读数效率
4. 把数据字典、规则管理做成在线 Excel 编辑器
5. 让概览页和报告页重新回到监控大盘路线

## 7. 当前真正要改的地方

基于这批 UI 反馈，当前应该正式进入改动的地方是：

1. 全局设计规范正式作为页面整改依据
2. 筛选区、表格区、操作区、页眉区统一规范化
3. 概览页继续保持“同步页”定位，并按示意图收视觉层级
4. 数据字典继续按“参数主表 + 枚举值展开 + 详情抽屉”收口
5. 实时查询继续按“三段式”结构收紧，不再扩散为监控页
6. 验收报告继续按“简报卡片 + 详细报告详情”收口
7. AI/Codex 改页前默认先做“差异诊断 -> 再修改”

## 8. 依赖事项

这些要求要真正落地，依赖如下：

1. `UI设计规范` 成为正式设计规则来源
2. `组件使用说明` 成为产品/前端统一调用基线
3. `YokaUI / Storybook` 继续作为实现参考
4. 页面层继续维护：
   - 顶部导航规范
   - 筛选区规范
   - 表格规范
   - 报告卡片规范
   - 数据字典层级规范

## 9. 当前判断

- `UI设计师反馈 -> 设计规范沉淀`：`solved`
- `UI设计师反馈 -> 正式产品改动点`：`solved`
- `UI设计师反馈 -> 已全部冻结为最终设计稿`：`partial`

原因：

- 现在已经能明确哪些是正式吸收的要求
- 也能明确哪些只是建议、哪些仍待页面确认
- 但要真正成为全项目统一标准，仍需继续回写到需求池、主真源和页面主线
