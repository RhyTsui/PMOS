## 1. 文档目标

本文档用于把 05-06 会议纪要里已经确认、但优先级晚于 P0 的事项，拆成可直接进入下一轮规划的 `P1` 需求清单。

本文档的职责是：

1. 明确哪些事项属于当前阶段必须继续收口，但可以晚于 P0
2. 明确对象边界、页面落点、依赖关系与最小验收点
3. 防止 P1 被继续混在“讨论意见”里无法进入排期

## 2. P1 总体判断

当前最适合纳入 `P1` 的，不是“边角优化”，而是以下 4 类能力：

1. `双端排期与时间线可视化`
2. `渠道包特殊规则前置录入`
3. `基于验收标准的动态规则生成`
4. `数据字典服务于规则与查询`

这些能力之所以是 P1，而不是 P0，是因为：

- 没有它们，系统仍可先跑起来
- 但缺了它们，后续版本化治理和规则自动化会很快进入重人工状态

## 3. P1 迭代拆解

### P1-1 双端排期与时间线可视化

#### 要解决的用户问题

- 当前概览页能看版本状态，但不能清楚判断客户端与服务端的关键里程碑是否一致
- 验收节奏与开发节奏容易脱节

#### 产品对象

- `客户端里程碑`
- `服务端里程碑`
- `提测时间`
- `完成时间`
- `风险提示`

#### 页面落点

- `概览`
- `埋点需求 -> 验收版本`

#### 页面要求

- 能输入并展示客户端/服务端两条时间线
- 能在同一版本下看到双端时间节点差异
- 不做重项目管理系统，但要能直观暴露排期风险

#### 依赖关系

- 依赖 P0 中验收版本对象稳定

#### 最小验收点

- 至少支持双端时间节点录入
- 概览页能看到关键排期状态
- 版本页能维护这些节点

### P1-2 渠道包特殊规则前置录入

#### 要解决的用户问题

- 渠道包现在不仅是范围对象，还承载特殊参数和特殊验收标准
- 如果不前置结构化录入，后续规则生成会继续人工判断

#### 产品对象

- `渠道包`
- `渠道号`
- `渠道名称`
- `终端平台`
- `特殊参数要求`
- `渠道专属验收标准`

#### 页面落点

- `埋点需求 -> 渠道包`
- `校验规则`

#### 页面要求

- 渠道包列表不仅展示数量和状态，还要能承接特殊规则前置定义
- 后续规则生成可引用渠道包级配置

#### 依赖关系

- 依赖 P0 中版本化治理与模板标准化

#### 最小验收点

- 可录入渠道包级特殊说明
- 规则页能看见渠道包维度筛选与关系
- 渠道包特殊规则能形成生成输入

### P1-3 基于验收标准的动态规则生成

#### 要解决的用户问题

- 当前规则如果只靠静态模板，会在终端、渠道、条件分支下迅速膨胀
- 验收标准如果不能转成条件规则，就无法真正自动化

#### 产品对象

- `验收标准`
- `条件规则`
- `动态生成规则`
- `规则实例`

#### 页面落点

- `校验规则`
- `埋点需求 -> 事件 / 参数 / 步骤`

#### 页面要求

- AI 能根据验收标准生成带条件的规则
- 规则不是只看“必填/非必填”，而是支持条件分支
- 生成后仍保留人工确认与启停能力

#### 依赖关系

- 依赖数据字典、渠道包、事件、参数结构化稳定

#### 最小验收点

- 支持条件规则表达
- 支持 AI 生成后人工修改
- 至少存在 1 类动态规则示例，如终端/渠道条件

### P1-4 数据字典服务于规则与查询

#### 要解决的用户问题

- 数据字典若只是查阅页，无法真正支撑规则生成和实时查询
- 字段来源、枚举值、类型、说明应该成为规则和查询的基础源

#### 产品对象

- `参数字典`
- `枚举值字典`
- `字段类型`
- `查询字段来源`
- `规则字段来源`

#### 页面落点

- `数据字典`
- `校验规则`
- `实时查询`

#### 页面要求

- 数据字典中的字段定义可被规则页消费
- 数据字典中的字段定义可被实时查询消费
- 枚举值与字段类型不是展示文本，而是结构化配置源

#### 依赖关系

- 依赖 P0 中模板和版本结构稳定

#### 最小验收点

- 规则页可引用字典字段
- 查询页的字段筛选可从字典来源解释
- 数据字典存在枚举值子表与参数详情分层

## 4. P1 页面映射

| 页面 | 本轮 P1 必须承接 |
| --- | --- |
| 概览 | 双端排期、时间线补强 |
| 埋点需求 | 渠道包特殊规则、版本时间线、事件/参数/步骤标准承接 |
| 数据字典 | 结构化字段定义、枚举值、规则/查询来源 |
| 校验规则 | 动态规则生成、渠道包/事件/步骤/参数筛选联动 |
| 实时查询 | 字段来源消费、字典联动说明 |
| 验收报告 | 本轮不作为 P1 主改动中心 |
| AI助手 | 后续作为规则生成与知识库消费端，不作为本轮 P1 主改动中心 |

## 5. 推荐开发顺序

建议顺序：

1. `P1-4 数据字典服务于规则与查询`
2. `P1-2 渠道包特殊规则前置录入`
3. `P1-3 基于验收标准的动态规则生成`
4. `P1-1 双端排期与时间线可视化`

原因：

- 数据字典是规则和查询的上游
- 渠道包特殊规则先前置，动态规则生成才有稳定输入
- 动态规则生成依赖上游结构化完成
- 双端排期虽重要，但对自动化主链依赖更少，可后置
