# AI埋点平台查询方案与成熟看板 v1

日期：2026-05-05

## 1. 目的

本文件把日志查询页和概览页中原本只存在于操作细节里的“查询方案”和“成熟看板”升级为正式产品对象。

它要解决的问题是：

- 查询结果只能临时筛选，无法沉淀、共享、复用
- 批次之间缺少稳定可复用的看板视角
- 环境、包体、接入单元等上下文不能稳定地贯穿查询与验收

## 2. 用户场景

- 数据分析师保存一套常用查询，直接复用于不同验收批次
- QA 把某次验收的关键筛选方案共享给研发复查
- 项目经理在概览页看到成熟看板，而不是零散日志结果
- 研发按角色默认进入自己最相关的查询方案

## 3. 与 7 个正式模块的关系

- `日志查询` 是查询方案的主创建和主使用入口
- `概览` 承接成熟看板摘要和跨批次对比
- `验收报告` 可引用某个查询方案或某个成熟看板视角
- `AI助手` 可基于当前查询方案做解释和建议
- 其他模块只消费查询方案上下文，不重复创建新对象

## 4. 页面与入口归属

- `日志查询` 增加“保存查询方案”“覆盖保存”“共享方案”“从方案打开”入口
- `概览` 增加“成熟看板”区块
- `概览` 增加“按批次 / 环境 / 包体 / 接入单元对比”入口
- `验收报告` 增加“从报告跳到关联查询方案”入口

## 5. 关键业务对象

- `scheme_name`

- `role_scope`
- `project_id`

## 6. 状态流 / 生命周期

1. 用户在日志查询页筛选
2. 保存为查询方案
3. 方案可选择私有或共享
4. 概览页引用若干方案聚合为成熟看板
5. 看板可按批次和上下文维度对比
6. 报告或 AI 助手引用这些对象继续解释

## 7. 角色作用域与权限边界

- 数据分析师可创建和共享全链路查询方案，并维护成熟看板
- 研发默认看到与自身端侧相关的方案，可共享到本角色组
- QA 可保存与验收复查相关的方案，并将报告关联到方案
- 项目经理 / 运营重点查看成熟看板和推荐方案，不直接维护复杂筛选

## 8. 与主真源的衔接点

本文件不改变“日志查询是证据层、概览是摘要层”的主真源定义，而是把两者之间缺失的中间沉淀对象补齐。

## 9. 明确不做什么

- 不做通用 BI 平台式图表工厂
- 不做任意拖拽式自定义看板编辑器
- 不做跨项目无限共享所有查询条件
- 不替代日志原始证据层

## 10. 本版结论

从现在起，`查询方案` 和 `成熟看板` 视为 AI 埋点平台正式产品对象，后续设计、工程、测试都必须承接它们，而不是只保留“可筛选日志”这种弱合同。
