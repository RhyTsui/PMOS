# AI埋点平台需求文档 v1.1

日期：2026-05-09
状态：生效中
适用范围：本版仅新增 `v1.1` 需求口径，不覆盖既有 `v1.0`

## 1. 文档目的

本文件用于承接 2026-05-09 最新评审反馈，对 AI 埋点平台的一期产品形态进行重新收束。

`v1.1` 的目标不是在 `v1.0` 页面基础上继续堆功能，而是把产品主链改写为：

`版本管理 -> 通用需求装配 -> 补充个性需求 -> 查询验证 -> 验收执行 -> 报告生成 -> 推送同步 -> 版本沉淀`

本版从产品层正式回答三个问题：

1. 产品最重要的核心目标是什么
2. 哪些能力必须围绕同一条主链重排
3. 如何在不增加数分工作量的前提下完成验收闭环

## 2. 触发背景

本次升级来自最新评审反馈，核心输入如下：

1. 三个核心诉求：`验收报告`、`查询体验`、`版本管理`
2. 产品约束：`不要增加数分工作量`
3. 能力方向：系统内置各类通用埋点需求，非固定项改为按次勾选
4. 外部依赖：数分下周提供结构化需求表格
5. 新增结果分发要求：验收报告不仅站内可看，还要支持推送到 `小闪`
6. 产品判断修正：当前需要调整整个产品形态，而不是只对现有页面做局部补丁

## 3. v1.1 产品定位

### 3.1 这是什么

AI 埋点平台 `v1.1` 是一个面向数分、研发、QA、项目管理协作的：

`版本化埋点需求装配与验收分发系统`

它不是单纯的埋点台账，也不是单纯的查询页集合，更不是只在站内查看结果的验收页面。

### 3.2 它解决什么问题

它重点解决以下三个高频问题：

1. 数分每次重复整理通用埋点需求，工作量高且容易漏
2. 查询、验收、报告、版本之间彼此割裂，无法形成闭环
3. 验收结果无法快速同步给外部协作对象，结论传播效率低

### 3.3 成功标准

当 `v1.1` 落地后，应至少满足：

1. 数分发起一个新版本时，不需要从零重填全部通用埋点需求
2. 查询页是为定位证据和回看问题服务，而不是孤立的日志工具
3. 验收报告既能站内查看，也能一键推送简报到 `小闪`
4. 任一验收结果都必须能回挂到具体版本
5. 页面重做后，用户面对的是一条清晰链路，而不是一组散页

## 4. v1.1 设计原则

### 4.1 降低输入成本优先

本版第一优先级不是“功能更全”，而是“少填、少配、少重复劳动”。

因此默认采用：

1. 平台内置通用埋点需求模板
2. 非固定项通过勾选补充
3. 自定义项只处理差异部分

### 4.2 版本是一等治理对象

`版本` 不再只是筛选条件，而是整条产品链路的一级治理单元。

以下对象都必须绑定版本：

1. 需求装配结果
2. 渠道包范围
3. 查询上下文
4. 验收批次
5. 验收报告
6. 推送记录

### 4.3 报告是结果产品，不是附属页面

`验收报告` 在 `v1.1` 中正式成为结果交付物，至少分为：

1. 站内详细报告
2. 推送到 `小闪` 的同步简报

### 4.4 查询服务于验收闭环

`查询体验` 的定位不是做“实时监控大盘”，而是：

1. 快速筛选问题范围
2. 快速查看证据样本
3. 快速回挂到具体版本、事件、参数、报告

## 5. v1.1 核心业务主链

### 5.1 主链定义

本版统一主链为：

1. 创建或选择验收版本
2. 选择内置通用埋点需求模板
3. 勾选本次需要的可选需求项
4. 补充少量自定义需求
5. 导入或接收结构化需求表格
6. 进入查询与校验执行
7. 生成验收报告
8. 推送简报到 `小闪`
9. 版本归档并供后续继承复用

### 5.2 原则变化

与 `v1.0` 相比，本版最重要的变化是：

1. 从“先做页面集合”改为“先做版本主链”
2. 从“手工录入需求明细”改为“模板内置 + 勾选补充”
3. 从“报告只在系统内消费”改为“报告需要对外同步”

## 6. 信息架构重排

`v1.1` 建议重排为以下 5 个一级能力域：

### 6.1 版本管理

负责：

1. 创建版本
2. 维护版本目标、周期、范围、渠道包
3. 管理版本状态
4. 继承历史版本配置

### 6.2 需求装配

负责：

1. 平台内置通用埋点需求模板
2. 按版本勾选本次适用需求
3. 补充自定义需求
4. 承接数分提供的结构化表格

### 6.3 查询验证

负责：

1. 版本上下文查询
2. 事件、参数、渠道、端侧筛选
3. 证据回看
4. 与验收报告联动跳转

### 6.4 验收报告

负责：

1. 发起验收批次
2. 汇总通过、不通过、未上报
3. 生成站内详细报告
4. 生成推送给 `小闪` 的简报

### 6.5 沉淀复用

负责：

1. 版本归档
2. 模板迭代
3. 常见问题回写
4. 通用埋点需求持续沉淀

## 7. 关键对象模型

### 7.1 通用需求模板

字段至少包括：

1. 模板ID
2. 模板名称
3. 适用端侧
4. 适用业务域
5. 默认是否启用
6. 事件清单
7. 参数清单
8. 验收标准

### 7.2 版本需求装配单

字段至少包括：

1. 版本ID
2. 来源模板集
3. 勾选启用项
4. 自定义补充项
5. 结构化表格导入记录
6. 当前生效范围

### 7.3 查询会话

字段至少包括：

1. 版本ID
2. 查询条件
3. 证据样本
4. 关联问题
5. 关联报告

### 7.4 验收报告

字段至少包括：

1. 报告ID
2. 版本ID
3. 验收批次ID
4. 通过率
5. 未上报数量
6. 不通过数量
7. 详细结论
8. 小闪推送状态
9. 推送时间

## 8. 关键页面要求

### 8.1 全局要求

1. 系统右上角必须支持 `需求版本` 切换，至少可切换 `v1.0 / v1.1`
2. 页面重做阶段默认按当前激活的需求版本显示对应口径
3. `v1.0` 保留不动，`v1.1` 作为新增口径逐步承接页面重做

### 8.2 版本管理页

必须突出：

1. 当前版本目标
2. 当前版本范围
3. 当前版本来源模板
4. 当前版本报告与推送状态

### 8.3 需求装配页

必须从“结构化录入表”升级为：

1. 通用模板区
2. 勾选补充区
3. 自定义差异区
4. 结构化表格导入区

### 8.4 查询页

必须强化：

1. 版本上下文可见
2. 常用筛选组合可快速切换
3. 结果能直接回挂报告与需求对象

### 8.5 验收报告页

必须支持：

1. 简报与详细报告双层结构
2. 一键推送简报到 `小闪`
3. 展示推送状态与推送时间

## 9. 范围与非目标

### 9.1 本版范围

`v1.1` 本轮先明确以下正式范围：

1. 产品形态重排
2. 需求模板内置化
3. 勾选式补充方案
4. 查询体验升级
5. 验收报告推送到 `小闪`
6. 版本管理升为一级能力

### 9.2 本版暂不冻结

以下内容本轮先不冻结为最终实现细节：

1. 数分下周提供的结构化表格最终字段
2. `小闪` 推送接口细节
3. 页面最终视觉稿与交互稿
4. 后端表结构的最终字段定义

## 10. 外部依赖

当前已知依赖：

1. 数分下周提供结构化需求表格
2. 需进一步确认 `小闪` 的推送载体与推送格式

在依赖到位前，本版可以先冻结：

1. 产品主链
2. 页面入口结构
3. 核心对象关系
4. 简报 / 详细报告双层模型

## 11. 与用户诉求映射

### 11.1 用户诉求

用户当前真实诉求是：

1. 验收结果要能形成正式报告
2. 查询过程要更高效
3. 版本管理要成为主链中心
4. 数分不要重复劳动
5. 报告要能推送到 `小闪`

### 11.2 产品承接结果

本版承接判断如下：

1. `验收报告`：`solved`
原因：已经正式定义为站内详细报告 + 小闪简报推送双层结果产品

2. `查询体验`：`partial`
原因：已经明确定位与目标，但仍需页面和交互承接

3. `版本管理`：`solved`
原因：已正式升级为一级治理对象和产品主链起点

4. `不要增加数分工作量`：`partial`
原因：方向已收敛为模板内置 + 勾选补充，但仍依赖结构化表格和页面落地验证

## 12. 本版结论

从 `v1.1` 开始，AI 埋点平台的一期不再按“单页功能补丁”推进，而改按以下口径推进：

`版本管理驱动、模板装配优先、查询验证承接、报告结果分发、对外同步闭环`

后续页面重做、前端交互、后端对象、测试验收，默认都应以本文件作为新增口径的产品真源。
