四、关键 EventSpec

1. EventSpec: 106 / 221 / 205

Fact：三者共同构成支付链。106 前端步骤、221 后端步骤、205 支付成功/发货完成。

Conflict：`channel_order_id`、`order_id`、`pay_channel`、`recharge_channel` 在规范层是关键字段，在步骤层又有 `未获取 / 不发送`。

Assumption：支付链验收应以“字段最终闭环可回查”为目标，而不是要求每个步骤都当前已具备完整字段。

2. EventSpec: 107

Fact：前置事件通过 `GSSdk.analytics.track()`，`type = "1"` 上报，且存在多个 `step_version`。

Conflict：历史表与 RM 中步骤命名、步骤分段存在扩展和迁移。

3. EventSpec: 108 + 20001

Fact：`108` 承担前端引导步骤，`20001` 承担后端引导状态。二者都依赖内容 ID / 枚举解释。

Assumption：两者应被视为同一业务链的不同视角，而不是独立孤立事件。

4. EventSpec: 20001-20027

自定义事件覆盖玩法成长、种植、制作、售卖、订单、场景、剧情、任务、换装、社交、交易、排行、公会、活动、平台特性等对象。 其中 `20011`、`20022`、`20023`、`20026` 明确是客户端/SDK 自定义事件；其余以服务端为主。

Conflict：`20019` 与 `20020` 当前字段设计未完整展开，不能按“完整规范已定”处理。

六、Conflict / Assumption 清单

C-001：前置 / 新手链是必须还是增强项，资料与项目推进现实存在张力。本文默认按关键链处理。

C-002：支付关键字段在不同步骤存在“应有但未有”的实现冲突。本文不消解该冲突，只显式标注。

A-001：`getTrackingInfo` 的定义能力可信，但落地完整性待后续验收确认。

A-002：`20001+` 迁移对旧 `10001+` 已发生，但消费端同步状态未知，按“迁移已存在，消费待核”处理。

C-003：术语层面同时存在 `pay_channel / recharge_channel`、`channel_order_id / app_order_id / paycenter_order_id` 等历史命名差异，需在字段映射层处理，不应直接混用。
