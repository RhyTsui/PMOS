二、埋点接入架构与职责分工

1. 总体架构

客户端 / SDK 层

  • 负责账号类基础事件、前置事件、新手引导、支付前端流程、激励视频前端流程、客户端自定义事件。
  • 通过 GSSDK.getInstance().call("analytics", "track", info) 发起埋点。
  • 通过 getTrackingInfo 读取设备/渠道透传信息。

服务端层

  • 负责账号验签、角色登录、登出、支付成功、资源产销、支付流程后端链、自定义玩法事件。
  • 负责补充客户端无法稳定获得的业务上下文字段。
  • 负责把客户端透传字段接入后端关键链路。

数据 BP / 产品层

  • 定义事件口径、字段意义、步骤命名、验收标准和冲突优先级。
  • 维护步骤版本和内容 ID 引用语义。

测试 / 验收层

  • 不只校验“是否上报”，还要校验时序、字段完整性、字段一致性和链路可回查能力。

三、客户端 SDK 接入规范

1. 通用调用方式

统一调用：GSSDK.getInstance().call("analytics", "track", info)

客户端现有资料显示，项目组不是直接自行决定使用 GET/POST 上报，而是通过 SDK analytics 模块发起事件。也就是说，客户端侧的“接口协议”本质上是 SDK 调用协议。

2. 客户端事件类型

类型 type 场景 必带字段
前置事件1107stepId, stepName
新手引导2108stepId, stepName
自定义事件020011 / 20022 / 20023 / 20026 等event, extra
支付式流程事件9106 / 110event, step_version, step_id, step_name

3. 标准请求体示例

前置事件

{ "type": "1", "extra": { "stepId": "100", "stepName": "开始播放CG", "param1": "aaa" } }

新手引导

{ "type": "2", "extra": { "stepId": "100", "stepName": "开始新手引导", "param1": "aaa" } }

客户端自定义事件

{ "type": "0", "event": "20011", "eventName": "GameStory", "extra": { "story_id": "123", "operation_type": "skip" } }

支付流程事件

{ "type": "9", "event": "106", "extra": { "step_version": "20250724", "step_id": "501", "step_name": "支付成功", "result": "0", "game_order_id": "..." } }

4. 设备透传读取

同步读取：GSSDK.getInstance().callString("analytics", "getTrackingInfo", new HashMap())

异步回调：getTrackingInfoAsync，返回 code / data / message 结构。

期望字段：s_oneid、c_oneid、device_id、cpsid、channel、os_type、app_version 等。

5. 客户端接入要求

  • stepId 必须为数字字符串，且按业务真实流程递增。
  • stepName 必须是可读文本，供验收和问题排查使用。
  • 自定义事件 event 必须是数字字符串，且大于 10000。
  • 支付链和激励视频链必须带步骤版本。
  • 失败步骤不能丢，建议保留失败原因。

四、服务端接入规范

1. 服务端不是补充说明，而是正式链路一部分

当前项目的埋点体系不是“客户端把所有数据都打完”，而是客户端和服务端共同完成闭环。特别是支付链、资源链、角色链、自定义玩法链，大量关键字段必须由服务端产出。

2. 服务端基础事件

事件 职责 关键字段
201 GameLogin账号登录完成event_id, start_time
203 RoleLogin角色登录role_id, user_lev, server, realserver
204 RoleLogout角色登出duration
205 Pay支付成功 / 发货完成order_id, recharge_channel, currency, amount
206 LevelUp角色升级pre_lev, pre_vip_lev
207 CCU在线人数user_count
208 Prop资源产销item_consumetype, scene, item_id, item_num
215 GameLoginSign账号验签result
221 PayFlow支付流程后端步骤step_id, order_id, pay_channel, recharge_channel
223 AdVideo激励视频后端步骤step_id, ad_sid, reward_target

3. 服务端事件产出要求

  • 统一携带公共头：game_uuid、app_id、ip、role_id、server、realserver。
  • 统一使用 13 位毫秒时间戳作为 start_time。
  • 字段定义与实际发送不一致时，必须在事件说明中保留该差异。
  • 支付链必须明确哪些字段来自发货通知，哪些字段来自中台订单，哪些字段来自客户端透传。

关键事实：当前资料没有给出项目组服务端直接调用的 REST/POST 网关。因此本文只定义“服务端事件契约”，不伪造不存在的 HTTP API 细节。

五、关键事件生命周期与时序

1. 登录前置链

  1. SDK 初始化 / 登录开始
  2. 设备信息获取
  3. 请求服务器列表
  4. 加载配置文件
  5. 获取网关地址
  6. 连接网络服务器
  7. 客户端角色登录

该链路受 step_version 影响极大，必须区分 20250122 与 20250724。

2. 支付闭环链

  1. 客户端支付流程事件 `106` 记录用户侧步骤和结果
  2. 服务端支付流程事件 `221` 记录后端关键步骤和补充字段
  3. 支付成功 / 发货事件 `205` 作为最终闭环结果

通过标准：必须能够从订单相关标识把 `106 -> 221 -> 205` 串起来。

3. 新手链

  1. 前端 `108 - TutorialStep` 记录引导步骤级行为
  2. 后端 `20001 - Tutorial` 记录引导状态

七、字段字典与类型要求

1. 类型约定

  • 时间：13 位毫秒时间戳
  • 步骤 ID / 事件 ID：数字字符串或整型，项目内保持一致
  • 金额：必须注明单位；支付链资料中存在“法币单位 ×100”的约定
  • map 类型字段：必须说明 key/value 语义
  • 枚举字段：必须可回溯到枚举字典

2. 高价值字段

字段 用途 问题点
order_id支付中台订单号部分步骤未获取
channel_order_id商户/渠道订单号部分步骤未获取
pay_channel充值渠道定义关键但存在未发送
recharge_channel支付方式定义关键但存在未发送
step_version步骤版本跨版本分析不可缺
s_oneid / c_oneid / device_id / cpsid透传服务端使用完整性待验证

八、校验规则与验收标准

客户端验收

  • SDK 调用参数结构正确
  • stepId / stepName / type / event 合法
  • 失败事件可观测

服务端验收

  • 公共头齐全
  • 关键业务字段不缺
  • 支付链与资源链字段可解释

链路验收

  • 106 -> 221 -> 205 可串查
  • 107 按 step_version 可重建前置漏斗
  • 108 + 20001 可解释新手链

灰区验收

  • 所有“不发送 / 未获取”字段单列缺口
  • 不因 demo 省略冲突说明

九、变更管理与兼容策略

  • 步骤链变更必须更新 step_version，不得静默改 step_id。
  • 自定义事件迁移必须保留新旧事件映射，例如 `20001 -> 原 10001`。
  • 字段重命名或来源变更时，必须同步更新字段映射表和验收规则。
  • 新增事件可以先接 demo 级字段，但必须显式标出“字段待补完”，不能假装完整。

十、常见问题

1. 这是不是 HTTP POST 接口文档？

客户端不是。当前资料只证明客户端通过 SDK analytics 模块调用，不应伪造 POST 地址。

2. 为什么还保留“不发送 / 未获取”？

因为这正是开发和验收最关键的信息。正式接入文档不能把冲突抹掉。

3. 这份文档是生产级还是 demo？

这是 demo，但写法按生产级开发者文档组织。它仍缺少真实网关、环境地址、鉴权策略等生产细节，因为源资料本身没有给出。
