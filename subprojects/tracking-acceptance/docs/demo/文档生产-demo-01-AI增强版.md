# 花花世界埋点需求文档 Demo

版本：AI 增强版  
样例来源：`『花花世界』RM版·数据埋点_V251222.xlsx`、`SDK行为日志埋点表.xlsx`、`数据埋点梳理-20241223.xlsx`

## 1. 文档目的

本文档用于说明《花花世界》当前高价值埋点链路的业务用途、事件定义、字段要求、灰区风险和验收重点，优先覆盖三条链：

- 新手引导漏斗链
- 支付与订单关联链
- 渠道分包与设备透传链

本文档不是全量埋点表，而是面向需求理解、研发接入和验收协同的增强版规格说明。

## 2. 业务背景

当前埋点需求不是单点日志上报，而是服务于以下业务消费场景：

- 新用户引导漏斗分析
- 支付流程漏斗和订单关联
- 渠道分包效果分析
- 设备标识透传与服务端关键行为串联
- 问题排查与版本回溯

现有文档已经能支撑接入，但存在以下问题：

- 文档是过程产物，存在迁移、替换、补充痕迹
- 有些字段定义存在，但实现侧未获取
- 必接与推荐项混在一起，后果说明不充分
- 客户端、SDK、服务端链路分散在不同文档中

## 3. 关键业务链

### 3.1 新手引导漏斗链

业务目的：

- 评估新用户引导流程完成情况
- 支撑新手漏斗、关键流失点和引导优化分析

当前事件：

- `108 / TutorialStep`
  - 前端/SDK 上报
  - 通过 `GSSdk.analytics.track()` 上报
  - `type = "2"`
- `20001 / Tutorial`
  - 后端自定义事件
  - 表示新手引导环节开始与完成
  - 当前文档中已映射到原事件 `10001`

关键字段：

- `stepId`
  - 游戏定义的步骤 ID
  - 必须为数字字符串
  - 要按游戏真实流程递增
- `stepName`
  - 游戏定义的步骤名称
  - 必填

当前风险：

- 新手引导打点在接入说明中被标记为“推荐”，不是“必接”
- `stepId` 需要与数据部门确认，说明口径还依赖人工对齐
- 步骤顺序如果不递增，后续漏斗不可比

业务影响：

- 任一步骤缺失、命名漂移、顺序错误，都会导致引导漏斗失真
- BI 新手转化分析会直接失效或不可比

### 3.2 支付与订单关联链

业务目的：

- 支撑支付漏斗分析
- 串联支付成功、发货通知、订单关联
- 为付费归因和问题回查提供基础链路

当前事件：

- `106 / PayFlow`
  - 前端/SDK 支付流程事件
  - `type = "9"`
- `221 / PayFlow`
  - 后端支付流程事件
  - 与前端 `106` 属于同一业务链
- `205 / Pay`
  - 角色充值发货完成事件
  - 与 `221` 的特定步骤同时上报

关键字段：

- `order_id`
  - 中台充值订单 ID
- `channel_order_id`
  - 商户/渠道订单号
- `pay_channel`
  - 充值渠道
- `recharge_channel`
  - 支付方式

当前风险：

- 多个支付链关键字段在核心事件里直接写明：`不发送，未获取`
- 部分字段又被标记为必填，但特定步骤 `402/501` 不发送
- 发货通知映射关系存在，但链路并不完整闭环

业务影响：

- 订单号缺失会让支付链断裂
- 无法稳定回查支付成功与发货通知
- 影响付费分析、订单串联和异常定位

### 3.3 渠道分包与设备透传链

业务目的：

- 支撑渠道分包效果分析
- 支撑客户端到服务端设备标识透传
- 为服务端关键行为上报提供基础上下文

关键字段：

- `cpsid`
  - 渠道分包 ID
- `device_id`
  - 设备 ID
- `c_oneid`
  - 客户端设备标识
- `s_oneid`
  - 服务端设备标识

链路说明：

- `SDK行为日志埋点表.xlsx` 中把 `cpsid` 定义为上报结构字段
- `数据埋点梳理-20241223.xlsx` 中客户端和服务端都要求 `cpsid` 从 SDK 获取并保持一致
- `花花世界` 的 Android 接入说明里，`getTrackingInfo` 返回 `cpsid/device_id/c_oneid/s_oneid/channel` 等信息

当前风险：

- 虽然文档里已经写出透传方案，但是否所有上报链路都稳定使用还不清楚
- `getTrackingInfo` 已经被定义为新增链路，说明这部分能力仍在演进

业务影响：

- 透传不一致会影响渠道分包分析
- 服务端行为无法稳定关联到客户端上下文
- 付费、归因和链路排障都可能失真

## 4. 文档中已经暴露出的灰区

### 4.1 未获取但业务关键

已明确命中：

- `order_id`
- `channel_order_id`
- `pay_channel`
- `recharge_channel`

这些都不是普通字段，而是支付链核心字段。

### 4.2 推荐项但业务价值高

已明确命中：

- 前置事件打点：推荐
- 新手引导打点：推荐

这两类不是锦上添花，而是直接影响漏斗分析和问题排查。

### 4.3 事件迁移和版本演进

已明确命中：

- `20001 -> 对应原 10001`
- `20002 -> 对应原 10002`
- `20004 -> 对应原 10004`

说明：

- 事件体系已经演进过
- 后续版本继续变更的概率高
- 没有自动迁移映射就很难维护

## 5. 当前待确认项

1. 新手引导和前置事件是否要从“推荐”升级为项目级必接
2. 支付链关键字段未获取的真实原因是什么
3. `getTrackingInfo` 返回的设备字段是否已经稳定透传到服务端关键链路
4. 新旧事件 ID 映射是否已经在下游消费侧同步完成

## 6. 第一阶段验收重点

1. 新手引导链
   - `108 / TutorialStep`
   - `stepId / stepName`
   - 步骤顺序、命名和覆盖率

2. 支付链
   - `106 / 221 / 205`
   - `order_id / channel_order_id / pay_channel / recharge_channel`
   - 发货通知映射是否闭环

3. 透传链
   - `cpsid / device_id / c_oneid / s_oneid`
   - 客户端获取、服务端接收、关键事件使用是否一致

## 7. 结论

现有 RM 文档已经足够支撑接入，但仍然更像“过程协作文档”，不是“最终规格真相源”。

如果第一阶段要快速落地，最合适的文档生产方式不是 AI 从零重写，而是：

`保留人工初稿 -> AI 补业务用途、风险、目录、说明、待确认项`

这能最大程度保留业务语义，同时明显降低理解成本。
