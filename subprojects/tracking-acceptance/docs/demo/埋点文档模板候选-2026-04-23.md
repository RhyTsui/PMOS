## 2. 今天建议只看 4 套模板

不再继续发散到很多风格，今天只保留 4 套可执行模板。

### 模板 A：AI 增强版

适用阶段：

- 第一阶段最稳妥
- 最适合从现有 RM / Excel / 群聊整理出可读首版

结构：

1. 文档目的
2. 业务背景
3. 关键业务链
4. 当前风险 / 灰区
5. 待确认项
6. 第一阶段验收重点
7. 结论

优点：

- 最贴近现有工作方式
- 最容易保留业务语义
- 最适合给人快速阅读
- AI 也能继续提炼风险和待确认项

缺点：

- 结构还不够硬
- 后续自动生成规则时还需要再抽一次结构

适合今天的用途：

- 先把 demo 内容写出来
- 先让你和相关角色快速确认方向

### 模板 B：三段式规格版

适用阶段：

- 第一阶段 demo 可直接用
- 第二阶段可以继续长成标准规格真相源

结构：

1. 业务目标
2. 埋点规格
3. 验收与影响

建议固定子块：

- 业务目标
- 事件
- 字段
- 规则
- 风险
- 验收口径
- 影响范围
- 待确认项

优点：

- 结构稳定
- 人和 AI 都能消费
- 适合后续生成规则、报告、检查项

缺点：

- 业务语义没有 A 自然
- 如果写得太硬，会像规范说明书

适合今天的用途：

- 作为 demo 的主模板
- 方便后续挂到工作台里展示

### 模板 C：验收优先版

适用阶段：

- 面向数据 BP / 验收执行 / 联调阶段
- 不适合直接当需求起草文档

结构：

1. 验收目标
2. 检查对象
3. 检查规则
4. 异常信号
5. 业务影响
6. 回归建议

优点：

- 最适合执行验收
- 一眼能看到校验重点和风险
- 适合接异常报告

缺点：

- 不适合承载完整业务上下文
- 不适合给产品或研发做首次理解

适合今天的用途：

- 作为 A/B 模板生成后的第二视图
- 用于 demo 里的“验收执行”页面

### 模板 D：双视图版

定义：

- 一份“人读版”
- 一份“AI 结构版”

建议拆法：

- 人读版：走 A 模板
- AI 结构版：走 B 或 C 模板

优点：

- 最符合长期路线
- 同时兼顾阅读体验和结构消费
- 后续适合规则生成、版本对比、报告生成

缺点：

- 今天实现成本最高
- 如果没有自动同步机制，容易双份维护

适合今天的用途：

- 只适合作为路线目标
- 不适合作为今天唯一交付物

## 3. 今天建议怎么选

如果目标是今天必须把 demo 搭出来，不建议继续在模板理论上绕。

今天的推荐组合是：

1. 主模板选 `模板 B：三段式规格版`
2. 展示层文案风格借 `模板 A：AI 增强版`
3. 验收页视图借 `模板 C：验收优先版`

也就是：

`底层结构用 B，首屏表达用 A，执行页用 C`

这样做的原因：

- A 单独用，适合看，但不够利于后续系统化
- B 单独用，适合系统，但容易太硬
- C 不能单独承担需求文档
- B + A + C 的组合，最适合今天做 demo

## 4. 我们今天不建议选的方案

### 不建议：纯 AI 全写

原因：

- 风险最大
- 业务灰区容易被 AI 脑补
- 不利于快速对齐真实项目语义

### 不建议：今天直接做 D 双真相源

原因：

- 会把今天 demo 搭建目标拖慢
- 容易继续消耗在结构设计，而不是产出 demo

## 5. 关于“我自己写 / 辅助写 / 辅助改”的判断

### 模式 1：我自己写

适合条件：

- 原始资料足够多
- 今天先要一个能演示的首版
- 允许先出 70 到 80 分版本，再人工确认

当前判断：

- 这个项目我可以自己写首版
- 因为样例、风险链路、关键字段、研究结论都已经有了

更适合写什么：

- demo 用主文档
- 工作台里的示例基线
- 验收页里的规则与问题样例

### 模式 2：辅助写

适合条件：

- 你已经知道业务想表达什么
- 但还没决定如何组织文档

当前判断：

- 这个模式不是今天最高效的
- 因为会把大量时间消耗在来回问答和共同起草

### 模式 3：辅助改

适合条件：

- 你或业务侧已经有 RM 文档 / 规格初稿
- 目标是快速变成可读、可验收、可演示版本

当前判断：

- 这是今天最现实、最快的主模式
- 也是昨天结论最支持的模式

## 6. 今天最优工作方式

今天不建议只选一种模式，而是分两段：

1. 我先自己写 demo 首版
2. 再按 `辅助改` 模式快速收口

原因：

- 这样最快
- 不会卡在空白页
- 也不会把错误一次性固化
