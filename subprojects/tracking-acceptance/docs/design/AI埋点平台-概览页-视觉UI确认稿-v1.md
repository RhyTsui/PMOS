# AI埋点平台-概览页-视觉UI确认稿-v1

日期：2026-05-05

## 1. 文档定位

本文档是概览页的视觉/UI规则稿。

它只负责确认：

- 页面视觉定位
- 信息层级
- 栅格与布局
- 组件表达
- 状态样式
- 内容呈现约束
- 对 Figma、静态 HTML、前端实现的约束

它不负责确认：

- 业务口径
- 页面交互逻辑
- 示例性页面文案
- 真实前端组件实现

## 2. 页面视觉定位

页面视觉定位固定为：

- 企业级数据平台
- 轻量展示台
- 中密信息密度
- 浅底深字
- 弱装饰、强结论

页面视觉气质必须满足：

- 稳定
- 克制
- 可扫读
- 可判断

页面不允许退化为：

- 数据大屏
- 导航页
- 营销官网风
- 半个执行工作台

## 3. 信息层级规则

页面视觉优先级固定为：

1. `版本验收态势`
2. `版本目标与门槛`
3. `当前进度摘要`
4. `更新影响摘要`
5. `进度趋势`
6. `当前主要风险`

强约束：

- 首屏第一视觉层必须是 `版本验收态势`
- 指标卡不得抢占第一视觉层
- `更新影响摘要` 必须低于进度摘要与风险区
- `TOP5` 风险区必须低于态势区与目标区

## 4. 栅格与布局规则

### 4.1 页面级布局

- 桌面端采用 `左侧导航 + 主内容区`
- 主内容区按纵向分段布局
- 页面主宽度优先服务阅读，不优先服务装饰

### 4.2 区块级布局

- `全局上下文区` 单独成段
- `态势区` 与 `目标区` 允许并排，但态势区权重高于目标区
- `指标区` 固定采用 `4 + 2`
- `趋势区` 固定采用 `图表区 + 口径提示区`
- `风险区` 固定采用 `双摘要表并排`

### 4.3 响应式规则

- 中等宽度下允许并排区块改为纵向堆叠
- 移动端允许压缩结构，但不得改变信息层级
- 响应式调整不得导致态势区降权

## 5. 组件表达规则

### 5.1 切换器

- 项目、版本、视角切换器统一为浅底卡片式选择器
- 当前选中态必须同时具备底色变化和边框变化
- 说明文字权重必须低于主值

### 5.2 态势区

- 必须采用高识别度主结论卡或状态条
- 当前态势判断必须是区块主视觉
- 原因与偏差信息必须是次级层
- 态势区不得表现为普通信息卡

### 5.3 目标与门槛区

- 采用规整字段卡结构
- `验收通过门槛` 必须与普通字段区分表达
- 状态标签必须稳定、克制、不抢态势区权重

### 5.4 指标卡

- 主指标 4 张卡必须统一表达
- 风险指标 2 张卡必须显式使用风险语义
- 不允许所有卡片同权同色
- `P0` 相关卡片必须区别于普通统计卡

### 5.5 更新影响摘要

- 使用轻量摘要卡或摘要行
- 信息密度低于指标区与风险区
- 不得设计成通知流主区块

### 5.6 趋势区

- 必须同时显示 `目标线` 与 `实际线`
- `实际线` 视觉权重高于 `目标线`
- 图旁必须常驻口径提示
- 趋势区背景必须干净，不引入额外视觉噪音

### 5.7 风险摘要表

- `不通过事件 TOP5` 与 `待验收埋点 TOP5` 必须采用摘要表
- 不允许扩成完整详情表中心
- 优先级、来源、状态必须一眼可扫
- `P0 / P1` 标签必须稳定可识别

## 6. 状态样式规则

必须形成统一状态体系：

- 空态
- 加载态
- 异常态
- 无权限态
- 风险态
- 已通过态

具体约束：

- 空态使用克制状态块，不用大插画
- 加载态使用局部骨架屏，不允许全页闪烁
- 异常态使用轻量错误提示块，并保留重试入口
- 无权限态必须与“暂无数据”明确区分
- 风险态优先依靠状态语义表达，不依靠装饰堆叠
- 已通过态必须稳定、克制，不做庆祝化视觉

## 7. 内容呈现约束

### 7.1 页面文案约束

- Pixso 视觉稿不得使用大段解释性文案作为页面主体
- 页面主内容应使用产品化短文案，而不是说明稿语言
- 说明性文字只允许作为辅助 caption 或口径提示出现
- 页面示例数据应接近真实业务场景

### 7.2 设计稿表达约束

- 设计稿中的文案应服务页面观感验证，不服务文档解释
- 设计稿不得把“为什么这样设计”直接写进页面内容
- 设计稿不得把规则稿中的原则句原样铺进页面

## 8. 禁止项

- 不把态势区做成普通信息卡
- 不把指标卡做成首屏唯一主视觉
- 不把更新摘要做成通知中心
- 不把 TOP5 做成完整详情页
- 不在概览页加入 AI 助手建议主模块
- 不在概览页加入复杂编辑动作
- 不在视觉稿中加入大段教程式说明文案

## 9. 与 Pixso高保真视觉稿的关系

- `Pixso高保真视觉稿` 是本稿的上游具象化设计载体
- 当前项目的概览页设计责任人为产品本人
- Pixso 必须根据本稿出真实高保真视觉稿，不得只停留在线框或区块骨架
- Pixso 中若发生冻结级视觉调整，必须同步回写本稿
- 未回写本稿的 Pixso 改动，不得直接作为静态 HTML 或最终实现依据

建议在 Pixso 中固定这些元信息：

- 页面名称
- 设计版本
- 当前状态：`设计中 / 待确认 / 已冻结`
- 设计责任人

## 10. 与静态HTML确认稿的关系

- `静态HTML确认稿` 必须服从本稿
- HTML 不得反向定义视觉规则
- 若 HTML 与本稿冲突，以本稿为准
- 若 HTML 继续带有说明稿式长文案，应视为未按本稿输出

## 11. 与其他稿件的关系

- 上游依赖：`原型确认稿`、`交互确认稿`、`Pixso高保真视觉稿`
- 下游承接：`静态HTML确认稿`
