# 会议纪要追溯补充

日期：2026-04-28

本轮补看三份更早纪要：

1. `04-02 内部会议. 数据系统对接与分析需求.docx`
2. `04-03 内部会议. AI驱动数据上报与测试自动化.docx`
3. `纪要_04-14 内部会议_ 事件接入自动化.docx`

结论：这三份纪要没有推翻当前一期方向，但补出了几类之前容易被弱化或误读的点。

## 1. 已被当前方向正确继承的点

### 1.1 标准模板 + 自定义扩展

04-14 很早就强调：

- 需要核心事件表 / 默认模板
- 需要动态扩展
- 需要自动注册与兜底

这与当前的：

- 标准事件
- 自定义事件
- 默认字段
- 项目裁剪

是一致的，说明方向没有错。

### 1.2 研发本地是高频使用场景

04-03 明确说明：

- 研发不希望所有调试都绕到后台页面
- 本地调试、本地看结果、本地结合代码上下文是高频诉求

这与当前强调：

- 结构化导出
- 角色化 Markdown
- 后续 AI coding / vibe coding 适配

是连上的。

### 1.3 真实数据验证优先于 mock

04-03 已明确：

- 不要长期停在 demo 数据和空模板
- 要用真实样例验证平台能力

这与当前改成 `花花世界` 真实数据填充前端完全一致。

## 2. 之前容易遗漏的点

### 2.1 需求对话式导入

04-03 提醒：

- 输入不只是 Excel / Word 上传
- 还会有口头描述、自然语言描述、会议纪要输入

所以后续真源和设计里，除了文档导入，还应保留：

- AI 对话式导入
- 零散输入转结构化基线草稿

当前状态：`部分满足`

### 2.2 临时查询 / 标签 / 聚合需求

04-02 明确说明：

- 有些真实需求不是固定流程
- 而是临时查询、聚合、分群、标签分析

这意味着系统不能只做刚性的流程型验收平台，还要保留：

- 查询方案
- 临时组合查询
- 标签 / 维表 / 外部属性挂接能力

当前状态：`部分满足`

### 2.3 增量优先，不等于存量统一治理

04-14 已经说明：

- 一期重点是新接入和增量治理
- 历史存量只在特殊 bug 情况下人工介入

所以要避免一个错误：

- 把“历史可追溯”误写成“一期统一治理全部历史存量”

正确口径应该是：

- 历史必须可回看、可解释、可比差异
- 但存量修复不是一期主目标

当前状态：`已明确，需继续守边界`

## 3. 之前容易出现的错误抽象

### 3.1 过度平台化

如果直接用研究图或 AI-native 拼装图压一期，很容易把系统讲成：

- 更大的控制面
- 更重的中台
- 更完整的自动化编排平台

但从更早纪要看，原始诉求更务实：

- 先降低低价值重复劳动
- 先把高频协作链和验收链跑通

### 3.2 只从平台页面视角看问题

04-03 很明确地提醒：

- 平台只是中枢
- 研发本地是高频使用端之一

所以后续设计与实现不能只考虑页面操作，也要考虑：

- 导出格式
- 本地消费
- 机器人同步

### 3.3 只承接固定流程，不承接临时分析

04-02 明确表明：

- 临时分析、分群、标签是现实需求

所以不能把系统只设计成一条 rigid workflow。

## 4. 当前应补进真源的点

后续应继续补强这些项：

1. 对话式输入 / 纪要式输入
2. 研发本地高频消费
3. 临时查询 / 标签 / 聚合能力
4. 增量优先、历史可追溯但不做历史存量统一治理
5. 真实数据验证优先于 mock

## 5. 总结

这三份更早纪要的价值不在于改主线，而在于提醒当前方案：

- 不要忘了用户真实输入仍然很散
- 不要弱化研发本地
- 不要把历史追溯误做成历史统一治理
- 不要再长期使用 mock 掩盖承载问题

这些点已经和当前一期方向对齐，但还需要继续落到真源和产品设计中。
