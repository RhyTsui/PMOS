## 一、问题现状

### 1.1 埋点数据上报问题

**问题描述：**
- 游戏事件字段非标准化，各游戏项目上报格式不一致
- 数据上报后存在很多问题，需要人工对接和排查
- 底层数据上报不规范：字段缺失、格式错误、命名不统一

**具体案例 (来自 04-10 周报)：**
```
问题：
1. 巨量引擎 - 广告主问题
2. 用户活跃 - 微小 - 腾讯/讯盟 H5+APP 合并
   - 微小 channel:520214 被识别为 OTHER
   - 原因：修改项目配置时未刷新 redis 的 app_config
3. 注册渠道 channel 和 cpsid 不一致
4. 4.7-4.8 注册状态上报缺失 channel 和 open_id
5. 腾讯注册上报媒体平台问题
6. 安卓数据未采集
```

### 1.2 测试验证效率问题

**当前流程：**
```
人工录入测试数据 → 人工检查报表 → 人工验证结果 → 人工回归测试
```

**痛点：**
- 测试用例需要人工编写，耗时且容易遗漏
- 每次版本更新需要重新验证所有埋点
- 问题修复后需要人工回归验证
- 跨系统排查困难 (SDK → 上报 → 数仓 → 报表)

### 1.3 数据验收流程问题

**现状：**
- 埋点验收状态无法追踪
- 缺乏统一的验收标准
- 问题定位需要跨多个系统排查
- 没有自动化验收工具

## 二、会议决策与需求

### 2.1 04-03 会议 - AI 驱动数据上报与测试自动化

**核心决策：**
1. 利用 AI 自动生成测试用例和测试数据
2. AI 自动识别上报数据与预期格式的差异
3. 建立自动化埋点验收流程

**会议记录摘要：**
> "当前的主要问题是数据上报不规范，需要人工对接和排查。可以通过 AI 来自动生成测试用例，自动识别数据差异，提高验收效率。"

**预期方案：**
- 输入：埋点需求文档/Schema
- AI 处理：自动生成测试用例、测试数据
- 输出：测试报告、问题定位

### 2.2 04-03 会议 - AI 驱动数据对接自动化

**核心决策：**
1. 游戏运营数据通过 MCP 进行对接
2. AI 自动解析数据格式并生成对接文档
3. 5 月份计划完成数据对接自动化

**时间线：**
- 4 月 25 日：需求对齐确认
- 5 月份：计划完成

**会议记录摘要：**
> "游戏运营数据的对接工作量大，可以通过 AI 自动解析数据格式，生成对接文档和代码。"

### 2.3 04-13 会议 - 新数仓数据架构与采集优化

**核心决策：**
1. 统一采集平台，各业务线数据统一接入
2. 解决历史数据缺失和格式不一致问题
3. 建立数据质量监控机制

**技术要点：**
- 每套数据一个 JSON 配置 (url, body, method)
- 统一字段定义
- 支持历史数据回溯

## 三、需求总结

### 3.1 功能需求

| 需求 ID | 需求描述 | 优先级 |
|---------|----------|--------|
| F001 | 埋点 Schema 自动校验 | P0 |
| F002 | 测试用例自动生成 | P0 |
| F003 | 测试数据自动生成 | P0 |
| F004 | 验收报告自动生成 | P0 |
| F005 | 问题自动定位 | P1 |
| F006 | MCP 服务集成 | P1 |
| F007 | 数据质量监控看板 | P2 |
| F008 | 历史数据对比 | P2 |

### 3.2 非功能需求

- **效率**: 验收时间从小时级降低到分钟级
- **准确性**: 自动识别 90%+ 的常见问题
- **可扩展**: 支持新游戏/新埋点的快速接入
- **可追溯**: 完整的验收记录和日志

### 3.3 验收标准

1. **Schema 校验**: 能自动识别字段缺失、类型错误、格式问题
2. **测试生成**: 根据 Schema 自动生成覆盖所有字段的测试用例
3. **问题定位**: 能定位问题在数据链路的哪个环节
4. **报告生成**: 生成包含问题列表、修复建议的验收报告

## 四、技术方案建议

### 4.1 系统架构

```
┌─────────────────────────────────────────────────────────┐
│                    AI 埋点验收系统                        │
├─────────────────────────────────────────────────────────┤
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐   │
│  │ Schema  │  │ 测试    │  │ 验收    │  │ 报告    │   │
│  │ 校验器  │  │ 生成器  │  │ 执行器  │  │ 生成器  │   │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘   │
│                         │                               │
│  ┌──────────────────────┴──────────────────────────┐   │
│  │              AI 引擎 (LLM)                       │   │
│  └─────────────────────────────────────────────────┘   │
│                         │                               │
│  ┌──────────────────────┴──────────────────────────┐   │
│  │              MCP 服务层                          │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
                         │
         ┌───────────────┼───────────────┐
         │               │               │
    ┌────▼────┐    ┌────▼────┐    ┌────▼────┐
    │ SDK     │    │ 数据    │    │ 报表    │
    │ 测试    │    │ 采集    │    │ 系统    │
    └─────────┘    └─────────┘    └─────────┘
```

### 4.2 核心流程

1. **Schema 定义**: 定义埋点数据格式规范
2. **测试生成**: AI 根据 Schema 生成测试用例和数据
3. **执行验收**: 自动执行测试，采集数据
4. **问题定位**: AI 分析问题，定位根因
5. **报告生成**: 生成验收报告和改进建议
