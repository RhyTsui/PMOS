## 2. 设计确认原则

1. 所有改动页先出图，再开发
2. 不只换 Logo，要把 Dataki 的“数据蓝 / 知识紫 / 洞察渐变”语义真正落到产品
3. 保留 WeKnora 的布局骨架，不做大规模页面重构
4. 首版只出与品牌改造、权限改造直接相关的图
5. 所有页面优先对齐现有组件库，不按想象发明组件

## 4. 逐页确认点

### 4.1 登录与注册

确认点：

- 首个用户注册成为超管的入口是否独立
- 普通用户是否只保留小闪扫码登录与小闪手机号登录
- 小闪登录失败、未命中组织、未授权时的错误态

红线：

- 登录页必须只支持小闪扫码登录与小闪手机号登录
- 其他登录方式必须去掉

### 4.2 首页 / 主入口

确认点：

- 在保留 WeKnora 壳体下，品牌资源如何替换
- LOGO、标题、Slogan、主色如何落到现有页面
- 哪些区域只做视觉替换，不改结构

建议：

- 首版优先用现有壳体承接 Dataki 品牌，不另起一套首页结构

### 4.3 超管系统配置

确认点：

- 大模型统一配置区的布局
- MCP 统一配置区的布局
- 哪些配置项只给超管可见
- 普通用户只读状态如何表达

### 4.4 用户管理与组织映射

确认点：

- 用户列表展示哪些字段
- 小组 / 部门 / 公司 / 集团层级如何展示
- 用户管理入口与组织映射入口如何摆放

### 4.5 MCP 与模型

确认点：

- 普通用户只展示统一模型与统一 MCP 状态的前台表达
- MCP 外挂是否放在显式管理页
- 默认内置“数据部 MCP”如何被识别与展示

## 5. 设计产物建议格式

1. 每页高保真图
2. 每页状态图：默认 / 空态 / 加载 / 错误 / 权限不足
3. 涉及弹窗的交互补图
4. 需要开发确认的组件级局部图

## 6. 当前结论

设计输入已经足够开始逐页出图。

当前真正未闭环的不是视觉规范，而是：

1. 小闪联调参数未到位
2. WeKnora 容器环境未跑通

因此建议：

1. 先按本清单出关键页面图
2. 先做 P0 页面的逐页确认
3. 开发顺序严格跟随已确认页面，不反向驱动设计
4. 登录页和登录流必须同步参考 `Dataki-v1.1组件库与小闪登录对齐说明.md`
