## 2. 登录设计红线

当前必须明确一条红线：

- 登录页不能被设计成只支持“扫码”和“手机号”两种方式

原因：

1. 你的业务要求已经明确“其他用户通过小闪登录”
2. 组件库与小闪登录快的目标是复用企业登录入口，而不是只做消费级双态登录
3. 后续还可能接入域账号、企业 SSO、登录快跳转态、未授权态和组织映射态

因此首版登录页必须是“可扩展登录入口容器”，而不是“写死的两种登录卡片”。

## 3. Dataki v1.1 的正确登录结构

### 3.1 首版分流

登录链路应至少分成两类：

1. 首次超管注册入口
2. 普通用户小闪登录入口

### 3.2 普通用户登录区要求

普通用户区至少要满足：

1. 可承接“小闪登录快”主按钮
2. 可承接企业 SSO / 域账号等后续扩展入口
3. 有独立错误态
4. 有未授权态
5. 有组织未匹配态

### 3.3 UI 结构要求

建议结构：

1. 顶部品牌区
2. 登录方式容器区
3. 主推荐入口：小闪登录快
4. 次级扩展区：预留更多企业登录方式
5. 状态提示区：失败 / 未授权 / 映射异常

## 4. 推荐组件映射

基于现有组件库，登录与入口页优先考虑以下组合：

1. `YkContainer`：页面骨架
2. `FlexGrid`：登录方式布局
3. `InputTheme`：输入风格统一
4. `YkTabs` 或分段选择器：切换登录场景，但不能限制为只有两种方式
5. `ButtonWithProgress`：登录中的主按钮
6. `Empty`：无权限、无组织映射等状态页
7. `YkDrawer`：登录帮助、组织说明、异常说明

## 5. 对设计稿的直接约束

后续所有登录页设计稿必须满足：

1. 不只出现扫码与手机号两个入口
2. 必须有“小闪登录快”主入口位
3. 必须有扩展登录方式槽位
4. 必须出失败态、未授权态、组织未命中态

## 6. 对 AI / Agent 的执行约束

后续 agent 在处理 Dataki 登录页时，必须优先参考：

1. `http://10.236.15.36:6006/`
2. `subprojects/knowledge-base/docs/design-review/ykui-index.json`

4. 不预留企业登录扩展位
