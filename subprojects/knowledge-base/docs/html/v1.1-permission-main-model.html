v1.1 权限主模型

当前权限不是单纯的角色设计，而是要先把 归属层、业务域空间层、资产层 和 操作层 四层关系立住。 这样后面的更新、接入、广场、自动标签结构、主题推荐才不会把权限打穿。

模型主结论

资产先有归属，空间只做协作；一个资产可以被多个业务域空间消费，但不会因为进入空间就改变主权。 权限判定要先看归属，再看空间，再看操作。

四层主模型

Layer 1

归属层

回答“这份知识本来归谁负责、谁主维护、谁决定升级/下线/共享”。

个人 部门 组织
Layer 2

协作层

按业务域组织跨部门协作，不直接等于组织架构树，也不改变资产归属。

广告数据分析域 游戏数据分析域 游戏发行运营域 游戏研发域 财务域 产品经理域 自动化测试域 AICoding域
Layer 3

资产层

知识库、数据库、方法论库、项目知识库、SKILL知识库等都属于资产层。

广告知识库 游戏数据分析数据库 SDK知识库 策划案知识库 产品方法论知识库 产品SKILL知识库 活动复盘知识库 埋点知识库
Layer 4

操作层

先用四个动作把权限打粗粒度主框，后续再细化检索、引用、发布等动作。

查看 修改 挂载 管理

关键原则

原则 1：资产先有归属 每个知识资产必须先归到个人、部门或组织之一，先解决主权，再谈共享。
原则 2：空间不改变归属 业务域空间只负责协作和消费，一个资产进入空间后仍然保持原有 owner。
原则 3：一个资产可以进入多个空间 例如 `SDK知识库` 可以同时被游戏研发域和游戏数据分析域消费。
原则 4：挂载权独立于查看权 能看某个知识库，不代表能把它挂到 AI 应用或 Agent 上。
原则 5：自动生成结构必须继承原权限 后面的自动标签、自动聚合、自动主题、自动导航，都不能绕开原资产权限。

判定链

Step 1 先识别用户属于哪个个人 / 部门 / 组织主体。
Step 2 再识别用户当前处在哪些业务域空间中。
Step 3 判断目标资产的归属、分类和被挂入的空间集合。
Step 4 判断用户对该资产是否拥有 `查看 / 修改 / 挂载 / 管理` 中的动作权限。
Step 5 输出可见资产、可修改资产、可挂载资产、可管理资产四个结果面。
