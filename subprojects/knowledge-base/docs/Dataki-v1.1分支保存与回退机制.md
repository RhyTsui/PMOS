# Dataki v1.1 分支保存与回退机制
- 版本：v1.1
- 日期：2026-04-23
- 状态：Draft
- 适用范围：Dataki v1.1 首版研发过程
- 读者对象：产品、前端、后端、测试、AI

## 1. 目标

Dataki v1.1 需要逐功能确认开发，因此必须有显式的分支保存、阶段回退和发布前回查机制。

## 2. 分支建议

建议至少保留三层分支：

1. 主分支：稳定可回看基线
2. 集成分支：Dataki v1.1 当前集成面
3. 功能分支：单功能逐个开发与确认

命名建议：

1. `main`
2. `feat/dataki-v1.1-integration`
3. `feat/dataki-v1.1-<module>`

## 3. 提交原则

1. 一次提交只做一类改动
2. 设计改动、前端改动、后端改动、配置改动不要混成一个不可读提交
3. 每个功能确认后立刻形成可回退节点

## 4. 回退原则

1. 以功能为单位回退，不整包粗暴回退
2. 文档真源先更新，再更新代码
3. 任何高风险改造前先打检查点
4. 容器配置改动与业务代码改动尽量分离

## 5. 推荐检查点

1. Dataki 品牌替换完成
2. 登录链路调整完成
3. 三类知识范围模型完成
4. 主对话框挂载链路完成
5. MCP 管理完成
6. Demo 验收通过

## 6. 需求变更约定

1. 每次砍需求或改范围，先改 `Dataki-v1.1首版定义.md`
2. 页面流、任务表、验收清单同步更新
3. 不允许只改代码不改真源

## 7. 当前执行提醒

这轮最容易失控的不是代码量，而是需求漂移。

因此 Dataki v1.1 的默认顺序应该是：

1. 先确认页面和功能
2. 再进实现
3. 每完成一个功能就形成一个可回退节点
