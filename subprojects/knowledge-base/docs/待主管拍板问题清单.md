## 二、必须拍板的问题

### 1. 知识库、Prompt、Chat、Agent 的上层归属

- 当前问题：
  Dataki v1.1 已明确把首版工作入口收敛为主对话框 + 知识资产，不再把 Prompt 资产单独做成首版一级导航；但从平台长期演进看，知识库、Prompt、Chat、Agent 后续是归到统一平台治理层，还是继续在业务子项目内长期并列，仍需主管明确。
- 为什么必须拍板：
  这会直接影响 v1.1 之后的 IA 延展方式、权限边界、导航结构和资产模型是否继续抽象。
- 不拍板的影响：
  v1.1 虽然能落地，但后续版本容易在“业务工作台”与“平台中台”之间反复改口径，导致页面结构和权限模型返工。
- 候选选项：
  1. `Dataki v1.1 先作为 knowledge-base 子项目工作台，Prompt / Chat / Agent 的平台级归属后移到 vNext 统一治理阶段`
  2. `从 v1.1 开始就预埋统一平台资产中心口径，把知识、Prompt、Chat、Agent 当并列一级资产设计`
- 当前建议：
  建议主管确认选项 1。理由是当前首版目标是把 Dataki 做成可用工作台，先把知识与主对话框闭环做通，避免平台级抽象过早压进 v1.1。

### 2. WeKnora 在整体架构中的层级

- 当前问题：
  这项问题已基本收敛：Dataki v1.1 基本保留 WeKnora 产品壳，只做轻品牌替换与必要范围收敛；当前版本不再按“深接管产品语义”推进。
- 为什么必须拍板：
  当前主要不是方向未定，而是需要主管确认这条收敛结论可作为后续研发默认前提。
- 不拍板的影响：
  团队会反复在“把 WeKnora 当临时产品壳”与“只当纯底座 SDK”之间摇摆，影响页面接管和模块边界。
- 候选选项：
  1. `按当前收敛结论执行：基本保留 WeKnora 产品壳，只做轻品牌替换`
  2. `只保留 WeKnora 底层能力，前后端产品壳另起一套`
- 当前建议：
  已按选项 1 收敛，不再作为 v1.1 待拍板问题。

### 3. 首版做平台层还是业务层应用

- 当前问题：
  Dataki v1.1 已明确不是大而全平台，而是面向公司内部知识与数据协同场景的首版工作台；但是否要在宣传口径和后续排期中把它定义成“平台首版”，还是“业务工作台首版”，需要主管统一口径。
- 为什么必须拍板：
  这会影响外部预期管理、后续需求进入方式和排期承诺。
- 不拍板的影响：
  对外会被误解为完整知识平台首版，研发则按工作台闭环建设，造成承诺与交付错位。
- 候选选项：
  1. `定义为业务工作台首版：先解决内部知识与数据协同场景`
  2. `定义为平台首版：默认承担后续统一平台能力延展预期`
- 当前建议：
  已按选项 1 收敛：v1.1 先解决内部知识与数据协同场景，成功标准是当前场景闭环，不是平台能力求全。
