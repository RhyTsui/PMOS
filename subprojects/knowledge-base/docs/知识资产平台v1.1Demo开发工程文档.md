# 知识资产平台 v1.1 Demo 开发工程文档

- 版本：v1.1 Demo
- 日期：2026-04-20
- 状态：开发规划
- 目标：验证知识资产平台框架成立

## 1. Demo 目标

Demo 的目的不是交付完整生产系统，而是验证 v1.1 框架是否成立。

必须验证的 4 件事：

1. 多源知识能够进入统一资产框架
2. 权限治理能够作用于消费链路
3. 外部/业务 Chat 能调用知识补强
4. 知识广场与会议纪要资产模型能跑通

## 2. Demo 架构

### 2.1 分层结构

建议的 Demo 分层：

1. `知识底座层`
2. `平台适配层`
3. `场景消费层`
4. `展示层`

### 2.2 各层职责

知识底座层：

- 使用 `WeKnora`
- 负责文档解析、索引、检索、引用片段能力

平台适配层：

- 定义统一资产对象
- 定义权限对象
- 定义接入与消费接口
- 屏蔽 WeKnora 的产品壳细节

场景消费层：

- 平台内 Chat
- 数据分析 Chat 样例
- 广告业务 Chat 样例

展示层：

- 接入页
- 资产页
- 广场页
- Chat 演示页

## 3. Demo 必做能力

### 3.1 接入样例

至少实现 4 类接入样例：

- 文本文档
- 会议纪要文本
- 内部文档样例
- 项目系统样例数据

说明：

- 不要求所有来源都是真实连接器
- 允许通过适配后的导入结果模拟连接器输入
- 重点是证明统一接入模型成立

### 3.2 资产样例

至少展示 4 类资产：

- 正式知识
- 会议纪要资产
- 业务规则/口径
- 外部订阅内容

### 3.3 消费样例

至少打通 3 类消费端：

- 平台内 Chat
- 数据分析 Chat 样例
- 广告业务 Chat 样例

### 3.4 广场样例

至少展示：

- 推荐资产
- 热门资产
- 可订阅资产
- 按类型筛选

### 3.5 修正样例

至少支持：

- 修改摘要
- 修改标签
- 修正会议纪要字段
- 标记错误
- 查看引用记录

## 4. Demo 对象与接口

### 4.1 平台统一对象

建议 Demo 层定义的核心对象：

- `asset`
- `asset_type`
- `source`
- `permission_policy`
- `reference`
- `consumer_binding`

### 4.2 知识消费接口

知识消费接口最小能力：

- 按权限检索
- 返回引用片段
- 指定知识范围
- 指定资产类型
- 返回来源信息

### 4.3 会议纪要解析输出

会议纪要解析必须能输出：

- 原文
- 结构化摘要
- 决策项
- 待办项
- 待确认问题

## 6. Demo 验收

### 6.1 产品验收

Demo 需要证明：

- 这不是单纯知识库页面，而是知识资产平台框架
- 同时支撑个人与企业业务场景
- 同时支撑接入、治理、消费、运营四个方向

### 6.2 技术验收

技术验收标准：

1. WeKnora 底座工作正常
2. 至少 4 类来源可进入统一资产模型
3. 至少 3 类消费方可调用
4. 权限控制影响检索与引用结果
5. 会议纪要资产类型可全链路演示

## 7. 测试场景

建议 Demo 必测场景：

1. 导入会议纪要文本并自动解析为会议纪要资产
2. 导入广告业务规则并供广告业务 Chat 调用
3. 导入数据口径文档并供数据分析 Chat 调用
4. 不同权限用户检索同一主题时返回不同结果
5. 从知识广场订阅一项资产后可在个人 Chat 中使用
6. 修正错误摘要后，后续消费端拿到更新结果

## 8. Demo 风险与控制

主要风险：

1. WeKnora 底座范围过重
2. 平台适配层与底座边界不清
3. 连接器深度实现拖慢节奏
4. Demo 变成“功能堆砌”而不是“框架验证”

控制策略：

1. 所有复杂连接器都可先用导入结果模拟
2. 所有演示围绕 3 个主场景展开
3. 只保留平台骨架必需的 UI 与接口
4. 不在 Demo 阶段推进录音转写、Agent 编排、深度洞察

## 9. 结论

v1.1 Demo 的任务不是证明平台已经做完，而是证明：

- 路线是对的
- 框架是能跑的
- 个人知识和企业业务知识可以共用一套平台骨架
- WeKnora 作为中二开底座是可落地的
