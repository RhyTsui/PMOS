- 版本：v0.1

- 目的：基于现有规划稿与新增会议纪要，统一需求池口径，输出 Q2-Q3 规划、1.0/1.1/1.2 月度版本规划，以及 1.0/1.1 交付工作流。

## 一、先统一版本口径

当前知识库相关文档存在两套版本命名：

1. `PRD.md` 使用 `v0.1 / v1.0 / v2.0`
2. `知识库整体规划.md`、`知识库 v1.0 整体设计方案.md` 使用 `v1.0 / v2.0 / v3.0`

为了支撑今天的交付，先统一为以下口径：

| 对外交付口径 | 时间 | 定位 | 对应旧文档口径 |
|---|---|---|---|
| **v1.0** | 4 月 | 首次可上线版本 | 原 `v0.1 MVP` / 部分文档中的 `v1.0 基础上线` |
| **v1.1** | 5 月 | 能力增强版本 | 原 `v2.0 能力建设` |
| **v1.2** | 6 月 | 质量完善版本 | 原 `v3.0 数据质量` |

结论：今天所有新产出都按 `v1.0 / v1.1 / v1.2` 编写，避免继续混用。

### 2.2 会议纪要里还需要补抽的缺失项

以下内容在现有规划稿里不完整，应该优先从会议纪要中补录到需求池：

| 缺失项 | 为什么必须补 |
|---|---|
| **目标用户与角色** | 决定原型页面、权限模型、用户手册结构 |
| **核心业务场景清单** | 决定 v1.0 先支持哪些问法、导入哪些知识 |
| **知识源清单 + 负责人** | 决定导入排期与上线前验收范围 |
| **知识对象分类标准** | 决定知识资产如何分库、分层、打标签 |
| **文档生命周期** | 决定草稿、发布、归档、失效、版本回滚规则 |
| **治理规则** | 决定谁能创建、审核、发布、修改、删除 |
| **更新机制** | 决定手动导入、定时同步、增量更新、失效提醒 |
| **检索效果验收指标** | 决定 v1.0 是否可上线，避免“能搜但不可用” |
| **Agent 接入边界** | 决定是只读检索，还是支持引用、权限透传、上下文注入 |
| **权限并轨边界** | 决定哪些能力放在知识库内，哪些依赖 AI 权限系统 |
| **运营反馈闭环** | 决定是否要记录无结果查询、低质量回答、人工修正 |
| **上线依赖与风险** | 决定今天排期是否真实可落地 |
| **培训/手册对象** | 决定用户手册是管理员版、编辑版还是普通使用版 |
| **原型范围** | 决定 v1.0 原型做后台即可，还是要补前台检索/问答入口 |
| **灰度与上线策略** | 决定内测、试点团队、回滚方案、监控方式 |

### 2.3 需求池建议字段

建议把每条会议纪要拆成一条或多条需求，至少补齐以下字段：

| 字段 | 说明 |
|---|---|
| 需求 ID | 例如 `KB-REQ-001` |
| 需求名称 | 一句话描述 |
| 来源会议 | 会议名称 + 日期 |
| 发起角色 | 谁提出 |
| 业务问题 | 当前痛点 |
| 用户场景 | 谁在什么场景下使用 |
| 需求描述 | 具体要什么 |
| 归属模块 | 资产、治理、导入、检索、Agent、权限、运营等 |
| 优先级 | P0 / P1 / P2 |
| 建议版本 | v1.0 / v1.1 / v1.2 / 待定 |
| 依赖项 | 权限系统、Agent 平台、数据源负责人等 |
| 验收标准 | 可量化 |
| 负责人 | 产品 / 技术 / 业务 |
| 状态 | 待分析 / 已确认 / 已排期 / 已上线 |

## 三、按工作流推进今天的产出

### 阶段 1：更新需求池

目标：把会议纪要里的新增事项先结构化，避免直接散落到 PRD 和规划文档中。

输出物：

1. 需求池总表
2. 新增需求去重结果
3. P0/P1/P2 优先级结果
4. 版本归属建议

### 阶段 2：形成 Q2-Q3 产品规划

目标：从需求池向上抽象成季度目标，不直接堆功能点。

建议口径：

| 季度 | 主题 | 目标 |
|---|---|---|
| **Q2** | 建底座、先上线 | 完成知识资产接入、基础检索、后台管理、v1.0-v1.2 连续交付 |
| **Q3** | 做规模、做效果 | 完成治理闭环、质量评估、跨系统集成、检索效果优化、运营体系建设 |

Q2 建议聚焦：

1. 知识资产可沉淀
2. 知识可被检索
3. Agent 可接入知识库
4. 权限能力完成并轨方案

Q3 建议聚焦：

1. 治理闭环
2. 检索效果可度量、可优化
3. 更多业务线接入
4. 运营与质控体系建立

### 阶段 3：形成 1.0 / 1.1 / 1.2 月度版本规划

| 版本 | 时间 | 版本目标 | 核心范围 |
|---|---|---|---|
| **v1.0** | 4 月 | 首次可上线 | 需求池、知识资产模型、导入能力、基础检索、后台管理、1.0 PRD、1.0 原型、1.0 用户手册、开发上线 |
| **v1.1** | 5 月 | 能力增强 | 知识资产治理、Agent 接入、混合检索/Rerank、1.1 PRD、1.1 原型 |
| **v1.2** | 6 月 | 质量完善 | 检索效果评估、质量监控、反馈闭环、知识更新机制、权限并轨落地 |

版本拆分建议：

#### v1.0

- 完成知识资产分类、标签、分层
- 完成文档导入和基础管理
- 完成基础检索能力
- 完成后台管理原型与 PRD
- 完成用户手册最小版
- 完成开发、测试、上线

#### v1.1

- 完成治理流程：创建、审核、发布、失效
- 完成 Agent 接入方式和权限透传
- 完成检索增强：关键词 + 语义 + Rerank
- 完成 v1.1 PRD 和原型

#### v1.2

- 完成效果评估与质量看板
- 完成低质量答案反馈闭环
- 完成更新机制、失效治理、版本追踪
- 完成权限并轨后的适配

### 阶段 4：拆 1.0 交付包

v1.0 今天要推进的交付不是一个文档，而是一组相互依赖的交付件：

| 交付件 | 目标 | 前置输入 |
|---|---|---|
| **v1.0 PRD** | 定义做什么、不做什么、验收标准 | 需求池、版本范围 |
| **v1.0 原型** | 定义页面、流程、交互 | PRD、角色、场景 |
| **v1.0 用户手册** | 定义谁怎么用 | 原型、功能范围 |
| **v1.0 开发上线计划** | 定义排期、责任、里程碑 | PRD、原型、依赖项 |

建议顺序：

1. 先出需求池
2. 再锁 v1.0 范围
3. 再写 v1.0 PRD
4. 再出 v1.0 原型
5. 再写 v1.0 用户手册
6. 最后补 v1.0 开发上线排期

### 阶段 5：并行准备 1.1

v1.1 不建议今天就写到实现级别，只需要完成：

1. `v1.1 PRD` 的骨架和范围
2. `v1.1 原型` 的页面清单和关键流程
3. 明确它依赖 v1.0 哪些能力先落地

## 七、现阶段明确的风险

| 风险 | 影响 |
|---|---|
| 版本命名口径不一致 | 规划、PRD、研发排期会互相打架 |
| 会议纪要未结构化入池 | 今天会重复改文档且容易遗漏需求 |
| 权限能力并入新项目 | v1.0/v1.1 边界不清，容易写重或漏写 |
| Agent 接入范围未定 | v1.1 PRD 无法锁范围 |
| 用户手册对象未定 | 手册难以一次写对 |
| 上线策略未定 | “开发上线”无法形成真实计划 |
