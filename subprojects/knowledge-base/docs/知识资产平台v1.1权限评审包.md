# 知识资产平台 v1.1 权限评审包

- 版本：v1.1
- 日期：2026-04-21
- 状态：Active

## 1. 需要评审清楚的权限问题

这次权限不是单一角色设计，而是企业 AI 应用与知识资产权限设计。

必须同时覆盖：

1. 谁是权限主体
2. 什么是被管对象
3. 哪些操作需要分级
4. AI 应用如何挂知识库
5. 外部 BI / 诊断 / 联调工具如何授权

## 2. 权限主体

`v1.1` 默认的权限主体包括：

1. `user`
2. `department`
3. `project`
4. `app`

说明：

1. `user` 是最小使用主体。
2. `department` 是企业协作与公共管理主体。
3. `project` 是跨部门的实际维护主体。
4. `app` 是知识消费主体，不等同于用户。

## 3. 知识资产对象

统一用 `knowledge_asset` 表示知识资产。

首版默认覆盖：

1. 个人库
2. 部门库
3. 项目库
4. 公共库
5. 应用内置库
6. 应用外挂库

统一用 `asset_scope` 表达可见范围：

1. `personal`
2. `department`
3. `project`
4. `company_public`
5. `app_private`
6. `app_shared`

## 4. 权限操作模型

首版不再只谈“可见/不可见”，统一按操作建模：

1. `view`
2. `retrieve`
3. `quote`
4. `edit`
5. `publish`
6. `bind`
7. `admin`

说明：

1. `view` 是页面可见。
2. `retrieve` 是检索可命中。
3. `quote` 是可被回答引用。
4. `edit` 是可维护资产内容。
5. `publish` 是可从个人或草稿态提升到更高 scope。
6. `bind` 是可挂到 AI 应用。
7. `admin` 是可改权限和成员配置。

## 5. AI 应用挂载模型

AI 应用必须作为独立消费主体建模。

首版引入：

1. `app_member`
2. `app_role`
3. `app_knowledge_binding`

`app_knowledge_binding` 必须表达：

1. 应用 ID
2. 资产 ID
3. 挂载模式：`built_in` / `external`
4. 是否允许检索
5. 是否允许引用
6. 是否允许在应用内编辑

## 6. 工具与数据源权限

知识权限和工具权限不能混成一套。

首版引入 `connector_policy`，覆盖：

1. 游戏 BI
2. 广告 BI
3. 多维分析引擎
4. 广告全链路诊断
5. 自动联调

工具权限至少区分：

1. 可查看结果
2. 可主动查询
3. 可执行动作
4. 可配置或联调

## 7. 判定链

统一权限中心只负责鉴权后，`v1.1` 的判定链固定为：

1. 鉴权输入
2. 主体关系展开
3. 资产 scope 判定
4. 操作权限判定
5. AI 应用挂载判定
6. 工具与数据源权限判定
7. 返回当前上下文下可用的知识与工具集合

## 8. 关键影响

如果按本方案推进，会带来以下结构变化：

1. WeKnora 原有组织分享模型不再足够，需要平台层补完整权限主模型。
2. AI 应用不再只是页面入口，而是权限计算主体。
3. 知识库不再只分“共享/不共享”，而是要支持显式 scope 与操作级权限。
4. 统一权限中心未来接入时，需要走“鉴权输入 -> 本地权限控制面”的模式，而不是直接替换控制面。

## 9. 需要持续评审的风险

1. scope 定义过多导致首版过重
2. 工具权限与知识权限耦合过深
3. AI 应用挂载模型不清导致后端接口反复修改
4. 本地 Demo 若不覆盖完整闭环，生产交接会失真

## 10. 当前评审结论

当前默认结论：

1. 权限中心只复用鉴权
2. 控制面在本项目
3. v1.1 先完成本地完整 Demo 闭环
4. 生产交接按任务包推进
