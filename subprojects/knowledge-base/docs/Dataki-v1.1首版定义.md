## 0. 真源说明

本文件作为 Dataki v1.1 的首版定义真源，优先级高于当前 `knowledge-base` 下更早期的通用型 v1.1 规划文档。

当前用于日常跟进的最新执行清单见：

- `docs/Dataki-v1.1最新确认需求清单.md`

本次收敛后的核心原则：

1. 当前版本只保留品牌改造与权限改造两条主需求。
2. 基本保留 WeKnora 现有产品壳，在现有壳体上完成改造。
3. 首版只保留被最终确认的需求，其他讨论项不混入当前范围。

## 1. 版本目标

Dataki v1.1 不是“通用知识平台完整版”，而是围绕品牌改造与权限改造落地的第一版产品化定义。

这一版必须同时解决两件事：

1. 在基本保留 WeKnora 产品壳的前提下完成 Dataki 品牌替换。
2. 完成超管、高级设置、统一配置、小闪组织结构接入和登录收敛在内的权限改造。

品牌口径：

- 产品名：`Dataki`
- Slogan：`让知识与数据相遇`
- 对外定位文案：`LLM-wiki-Agent-MCP-Prompts`

品牌与设计输入来源：

- `docs/sources/inbox/Dataki_brand_pack (1).zip`
- `docs/sources/inbox/Dataki_ui_specs.zip`

已确认可用的品牌输入包括：

- `Dataki_品牌与产品规范.md`
- `Dataki_Figma_前端_Tokens_说明.md`
- `Dataki_首页UI结构.md`
- `Dataki_Vue组件规范.md`
- `http://10.236.15.36:6006/` 组件库
- `subprojects/knowledge-base/docs/design-review/ykui-index.json` 本地组件索引

## 2. 首版范围

### 2.1 必做范围

1. Dataki 品牌替换与设计改造
2. 指定超管统一管理系统级配置
3. 其他用户采用统一配置
4. 默认内置数据部大模型服务
5. 默认内置知识库和数据 MCP 服务
6. 用户管理
7. 小闪多级组织结构接入
8. 去掉其他登录方式，收敛为小闪扫码登录与小闪手机号登录

### 2.2 本版不做

1. 项目知识库单独成类
2. 普通用户自定义模型配置
3. 与更多外部身份源并行接入
4. 权限中心与统一平台能力整合
5. 发布、版本、评测能力建设
6. 大规模重新设计 WeKnora 的底层文档解析与检索链路

## 3. 需求清单

### 3.1 品牌与设计

1. 产品名统一改为 `Dataki`
2. Slogan 统一改为 `让知识与数据相遇`
3. LOGO、设计改造以 `Dataki_brand_pack (1).zip` 与 `Dataki_ui_specs.zip` 为准
4. 前端采用统一组件库嵌入
5. 当前版本默认保留 WeKnora 现有产品壳与主要页面结构，不做大规模重设计
6. 所有改动页面都需要出图逐个确认后再开发
7. 不再使用 `内部知识与数据协同工作台` 作为当前版本主文案

### 3.2 权限改造

1. 增加 `超管` 角色
2. 超管是指定管理人，负责统一管理系统配置
3. 超管拥有高级设置权限
4. 大模型配置、MCP 配置、系统级高级设置不对普通用户开放
5. 其他用户默认采用统一配置
6. 默认内置数据部大模型服务
7. 默认内置知识库和数据 MCP 服务
8. 增加用户管理能力
9. 小闪组织结构作为用户组织真源，覆盖：
   - 小组
   - 部门
   - 公司
   - 集团
10. 去掉其他登录方式，收敛为小闪扫码登录与小闪手机号登录

补充口径：

1. 为了保证系统可初始化，仍需保留超管落位能力
2. 上述登录收敛指日常用户登录入口只保留小闪扫码登录与小闪手机号登录，不再保留并行登录体系
3. 普通用户只消费统一下发的模型与 MCP 能力，不参与系统级配置
4. 对外能力描述不再使用 `统一账号接入 / 统一模型配置 / 统一知识协同`
5. 统一改为 `内置数据部大模型、知识库和数据 MCP 服务`

## 4. 小闪对接定义

### 4.1 当前已知输入

- 测试环境地址：`https://testwapi.zhimagame.net`
- 通过第三方账号获取小闪用户详细信息
- 获取 access_token 接口

### 4.2 当前外部依赖

当前缺失：

- `appkey`
- `appsecret`

你已明确说明“开发者密钥明天提供”，因此这不是产品范围未定，而是外部依赖待补。

### 4.3 首版接入要求

1. 服务端缓存 `access_token`
2. 超管之外的用户登录链路默认走小闪
3. 通过第三方域账号拉取完整用户信息
4. 用户首次进入时同步组织树字段
5. 后端必须预留 token 续期与失败兜底处理
6. 前台登录容器必须为企业登录扩展预留槽位

### 4.4 对接字段

Dataki 首版至少要接住以下字段：

- `userid`
- `user_name`
- `real_name`
- `group_id`
- `group_name`
- `company_id`
- `company_name`
- `department_id`
- `department_name`
- `org_id`
- `org_name`
- `phone`

## 5. 与现有 WeKnora / PMAIOS 架构的影响

### 5.1 对 WeKnora 的影响

高影响改造点：

1. 登录注册逻辑
2. 用户组织模型
3. 前台品牌与必要文案
4. 超管与高级设置权限
5. 模型配置可见性
6. MCP 默认挂载

低影响沿用点：

1. 文档导入
2. 解析与切片
3. 检索与引用
4. 会话基础链路
5. 容器化运行底座

### 5.2 对现有 PMAIOS 架构的影响

低影响：

- 当前主要是 `knowledge-base` 子项目内收敛，不直接改动 PMAIOS 主内核。

中影响：

- Dataki 首版把知识库前台壳、权限口径、模型展示口径压实后，会影响后续平台级 Chat / Prompt / Runtime / 治理层的边界定义。

### 5.3 当前确认的冲突

1. WeKnora 原生组织 / 注册逻辑与“小闪登录为主”的要求存在直接冲突。
2. WeKnora 原生多登录方式与“去掉其他登录方式”的要求存在直接冲突。
3. WeKnora 原生模型入口可见性与“由指定超管统一管理、普通用户不参与配置”存在冲突。
4. 小闪开发者密钥未到位前，SSO 联调不能闭环。
