# 游戏数据埋点需求与验收平台 v1.0 设计定位

## 1. 重新定义问题

这次要做的不是一篇埋点文档，也不是一个单纯的埋点需求提交流程。

要做的是一个面向游戏项目、SDK、BI、数据中台、运营、发行、QA、研发共同协作的产品：

`游戏数据埋点需求与验收平台`

它的核心目标不是“把埋点写清楚”，而是让以下链路在同一套标准里闭环：

`业务目标 -> 指标体系 -> 事件模型 -> 字段规范 -> 研发接入 -> 数据验收 -> BI 分析 -> 版本治理`

## 2. 用户需求与产品需求映射

### 用户需求

- 项目方希望在测试前快速搭建一套可执行的埋点方案
- BI 希望指标口径、事件定义、字段口径统一，不再反复对齐
- 研发希望知道接什么、谁接、什么时候接、怎么验收
- QA 和数据平台希望能够自动校验，而不是人工逐条核对
- 运营和发行希望最终能直接拿到可用于渠道评估、留存分析、付费分析、ROI 分析的数据

### 产品需求

- 沉淀统一事件字典
- 沉淀统一字段字典
- 建立指标到事件的映射关系
- 建立验收规则和自动校验体系
- 支持埋点需求版本管理和变更 Diff
- 支持多角色协同和责任归属
- 支持 AI 生成埋点方案、验收规则、报告草稿

### 当前判断

- 用户需求层面：你要解决的是“跨角色协同闭环”，不是“出一份文档”
- 产品需求层面：平台最小闭环必须围绕“四张表 + 一个流程”建设
- 结论：此前单页协同看板只能算 `partial`，只能作为平台首页子视图，不能代表完整方案

## 3. 平台核心骨架

### 四张核心表

#### 3.1 事件字典表

核心字段：

- `event_name`
- `event_desc`
- `business_domain`
- `trigger_condition`
- `report_side`
- `priority`
- `owner`
- `developer`
- `acceptance_required`
- `version`
- `status`

作用：

- 统一“事件是什么”
- 统一“由谁上报”
- 统一“是否需要验收”
- 统一“该事件归属于哪条业务链路”

#### 3.2 字段字典表

核心字段：

- `field_name`
- `field_type`
- `required`
- `example_value`
- `description`
- `enum_scope`
- `reuse_scope`
- `version`

作用：

- 统一字段命名和类型
- 控制字段复用与冗余增长
- 为客户端、服务端、SDK、数据中台提供同一口径

#### 3.3 指标映射表

核心字段：

- `metric_name`
- `metric_desc`
- `dependency_events`
- `calculation_rule`
- `granularity`
- `output_owner`
- `dashboard_target`

作用：

- 让业务目标直接映射到数据设计
- 保证 BI 指标不是事后拼凑
- 避免“有埋点没指标”或“有指标无事件支撑”

#### 3.4 验收规则表

核心字段：

- `rule_name`
- `rule_type`
- `target_event`
- `rule_expression`
- `severity`
- `owner`
- `auto_check_supported`
- `report_template`

作用：

- 把验收从人工经验升级为规则系统
- 支持必填、枚举、类型、时序、数量、金额、去重、异常检测
- 支持自动出验收报告和问题清单

## 4. 平台模块设计

v1.0 建议拆成 8 个模块。

### 4.1 埋点需求中心

负责：

- 新需求创建
- 业务背景录入
- 测试目标录入
- 埋点范围定义
- 责任人与开发归属
- 版本状态管理

### 4.2 事件字典中心

负责：

- 事件清单维护
- 事件按业务域分层
- 触发时机管理
- 上报端管理
- P0 / P1 / P2 优先级管理

### 4.3 字段标准中心

负责：

- 通用字段标准
- 事件级字段扩展
- 枚举值约束
- 字段复用检测
- 字段变更版本记录

### 4.4 BI 指标映射中心

负责：

- 指标树维护
- 指标依赖事件配置
- 指标口径说明
- BI 看板反向追溯到事件定义

### 4.5 埋点版本管理

负责：

- 版本对比
- 变更 Diff
- 字段新增/删除/修改记录
- 版本归档
- 影响面分析

### 4.6 埋点验收中心

负责：

- 验收批次管理
- 自动校验执行
- 失败样本查看
- 问题分派
- 验收报告生成

### 4.7 数据质量报告

负责：

- 每日上报进度
- 燃尽图
- 优先问题清单
- 事件成功率
- 字段完整率
- 异常趋势

### 4.8 AI 埋点助手

负责：

- 从 PRD / 玩法文档生成埋点初稿
- 生成事件字典草案
- 生成字段规范草案
- 生成验收规则草案
- 生成研发接入文档
- 生成验收报告草稿

## 5. 页面信息架构

### 平台首页

不是“一个大列表”，而是治理驾驶舱。

首页应展示：

- 当前项目版本状态
- 本周待上线埋点需求
- 待确认版本 Diff
- 最新验收批次
- 数据质量趋势
- P0 / P1 问题
- 手动推送到群

### 二级页面

- `需求详情页`
- `事件字典页`
- `字段标准页`
- `指标映射页`
- `版本 Diff 页`
- `验收中心页`
- `验收报告页`
- `数据质量页`
- `AI 生成页`

## 6. v1.0 最小可落地闭环

v1.0 不应该一上来做全自动平台，而要先做可跑起来的最小闭环。

### 必做

- 需求提交与版本管理
- 四张核心表在线维护
- 版本 Diff 查看
- 验收批次执行与结果页
- 自动校验规则执行
- 验收报告生成
- 燃尽图与优先问题清单
- 手动推送到小闪群

### 可后置

- AI 自动生成全链路文档
- SDK 自动接入文档导出
- BI 看板自动建模
- 异常检测智能归因
- 自动审批流

## 7. 页面优先级判断

如果现在继续出图，不应该再只画一个协同看板，而应该按下面顺序出：

1. 平台首页总览
2. 需求详情页
3. 事件字典页
4. 验收中心页
5. 版本 Diff 页

原因：

- 首页负责讲清楚平台价值
- 详情页负责承接真实协作
- 事件字典页负责体现平台标准化能力
- 验收中心页负责体现闭环能力
- Diff 页负责体现治理能力

## 8. 对上一版设计图的修正

上一版 `埋点需求协同看板` 不是废掉，而是降级为首页中的一个工作区块。

调整后定位如下：

- 原“需求池与版本记录” -> 首页卡片 + 需求详情页主体
- 原“差异识别与批次验证” -> 版本 Diff 页 + 验收中心页
- 原“实时上报概览” -> 数据质量页
- 原“燃尽与问题优先级” -> 首页治理驾驶舱
- 原“手动推送到小闪群” -> 首页和报告页操作区

## 9. 最终产品判断

最终产品不是：

- SDK 文档工具
- 单次埋点提单系统
- 大模型写文档助手

最终产品应该是：

`游戏数据埋点需求与验收平台`

平台核心价值是：

- 统一事件标准
- 统一字段标准
- 统一指标口径
- 统一验收规则
- 统一版本演进记录
- 统一跨角色协作流程

只有这样，业务目标、研发接入、数据验收、BI 分析和治理沉淀才会真正连起来。
