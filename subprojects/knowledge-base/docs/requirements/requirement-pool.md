# 需求池

## 2026-04-22 企业个人知识库加密

### 来源

- 来源材料：`E:/AI/ai-os/docs/sources/inbox/企业个人知识库加密-豆包.png`
- 来源性质：需求分析过程稿

### 用户需求

- 个人知识库必须默认加密
- 除用户主动分享的部分外，平台超管不能直接查看明文
- 如需查看，必须经过审批与审计
- 产品服务介绍中需要明确表达“加密”

### 提炼后的产品需求

- 所有知识内容默认按“存储加密”处理，不能以明文持久化作为常态
- 解密权限与存储分离，不能因为“可搜索 / 可调试 / 可分享”就默认开放明文访问
- 超管对个人知识库默认无明文读取权限，只能看到必要元数据或审批态信息
- 分享不是“取消加密”，而是“开放解密权限”
- 审批访问必须有完整留痕，至少包含：谁申请、谁批准、何时访问、访问范围、访问原因

### 分层规则

#### 1. 统一安全规则

- 所有知识库统一加密存储
- 解密权限与查看权限分离
- 检索、RAG、调试不能成为绕过权限直接拿明文的后门

#### 2. 三级权限规则

##### 个人知识库

- 查看：仅本人
- 超管：默认不可见
- 例外：审批 + 授权 + 留痕后，才可临时解密查看
- RAG 效果查看：默认仅本人可见
- 分享：被分享对象获得解密权限，但内容仍保持加密存储

##### 部门知识库

- 存储：加密
- 查看：部门内成员可见
- 部门管理员：可管理、可检索、可上架/下架
- 超管：可直接查看与管理，但必须保留留痕
- RAG 调试：部门内部与授权角色可直接查看

##### 公司 / 组织知识库

- 存储：加密
- 查看：按公司范围开放
- 超管 / 产品 / 数分等平台角色：可在合规范围内直接查看与调试
- 所有访问动作保留留痕

### 分享与流转规则

- 个人内容分享到个人 / 同事 / 组内：本质是开放解密权限，不是转成明文
- 个人内容提级到公共知识库：先审批，再脱敏/分类，再进入目标知识库
- 进入部门 / 公司知识库后，仍保持加密存储，只是按目标范围开放查看权限

### RAG 与检索规则

- 存储层：向量库 / 文档库中的知识内容保持加密存储
- 检索层：只有在通过权限校验后，才允许在内存中做临时解密
- 推理层：模型消费的是本次请求授权后的明文片段，不写回明文持久层
- 结束后：内存明文清理，磁盘中仍为密文

### 为什么这条需求重要

- 员工个人知识库会承载策略、投放、数据口径、未上线逻辑、业务敏感内容
- 一旦平台或超管可直接读明文，用户对“个人知识库”的信任就会坍塌
- 同时又不能因为“绝对隔离”彻底牺牲分享、RAG、合规审计与运营能力

### 当前推荐方向

- 对外口径优先写成：`默认加密存储 + 受控访问`
- 架构方向优先采用：`统一加密存储，按知识库层级开放不同解密权限`
- 权限边界优先采用：
  - 个人：隐私级，超管默认不可见
  - 部门：协作级，部门内默认可见
  - 公司：公共级，授权范围内可见

### 待决策项

- “绝对加密”的实际技术定义：
  - 用户主控密钥
  - 平台托管密钥但默认不可见
  - 合规审批后可临时解密
- 分享链接 / 分享对象的权限时效策略
- 超管审批访问的批准人是谁：直属主管、部门负责人、安全负责人还是系统双签
- 审批后的解密窗口时长、可复制性、可导出性
- 服务介绍页面对“加密”的承诺强度，是否可以使用“超管不可见”这类强承诺措辞
