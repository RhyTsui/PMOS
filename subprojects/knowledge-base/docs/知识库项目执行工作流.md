## 二、项目推进顺序

### 阶段 1：问题定义

目标：

明确知识库项目当前要解决的问题，而不是直接讨论产品形态。

输出：

1. 问题定义文档
2. 业务目标
3. 目标用户与角色
4. 项目范围假设
5. 待研究问题单

### 阶段 2：研究分析

目标：

补齐事实依据，形成候选方向，而不是定案。

必须完成的 4 份文档：

1. `WeKnora 适配性评估`
2. `知识库竞品与开源方案对比`
3. `内部能力盘点`
4. `待主管拍板问题清单`

阶段产出只允许写成：

1. 候选方向 A/B/C
2. 利弊分析
3. 风险
4. 待决策问题

### 阶段 3：总体架构确认

目标：

由主管或架构负责人确认知识库在整体产品线里的边界。

必须确认：

1. 知识库、Prompt、Chat、Agent 的关系
2. 权限中心是否复用统一平台能力
3. 发布、版本、评测是否由知识库负责
4. WeKnora 所处层级
5. 首版是平台层还是业务层应用

输出：

1. 总体服务架构图
2. 模块职责表
3. 范围内 / 范围外清单
4. 主管拍板结论

### 阶段 4：产品方案

目标：

在研究和架构确认后，收敛一期方向。

输出：

1. 方案 A/B/C 对比
2. 一期范围
3. 非一期范围
4. 模块设计
5. 版本路径

### 阶段 5：PRD / 原型

目标：

把已确认方案转成可执行产品定义。

输出：

1. PRD
2. 页面清单
3. 用户流
4. 原型
5. 验收标准

### 阶段 6：交付规划

目标：

把产品定义转为排期和协同。

输出：

1. 开发排期
2. 责任分工
3. 测试与验收计划
4. 上线计划

## 三、提交与评审规则

### 3.0 角色责任

当前知识库项目按以下责任分工执行：

1. 真实产品主管 / 上层主管：负责规则、阶段门禁、架构边界、最终拍板
2. 虚拟产品主管 Agent：负责阶段判断、任务分发、评审、打回、升级
3. 虚拟产品经理 Agent：负责研究执行、项目级初稿、需求池整理、问题单整理

明确要求：

1. 当前阶段的研究包由虚拟产品经理 Agent 先产出
2. 虚拟产品主管 Agent 负责审核，不直接代写首稿
3. 只有当首稿长期缺失或质量持续不达标时，主管才接管重做

### 3.1 虚拟产品经理提交物

每次提交必须包含：

1. 结论
2. 依据
3. 不确定性
4. 影响范围
5. 请求上级决策的问题

提交动作还必须包含主动同步：

1. 每完成一份首稿或关键更新，必须主动同步一次当前进度
2. 同步内容必须写清“已产出什么、还缺什么、卡点是什么”
3. 不允许只在本地持续工作而不提交阶段结果
4. 未主动同步的产出，不视为已正式提交

### 3.2 主管评审结论

每次评审只允许使用：

1. `accepted`
2. `accepted-with-constraints`
3. `needs-more-research`
4. `blocked-by-architecture`
5. `rejected`

### 3.3 禁止事项

1. 未完成研究前，禁止写“最稳路线是”
2. 未完成架构确认前，禁止写正式 PRD
3. 未完成 A/B/C 对比前，禁止写定版方案
4. 未给出证据来源前，禁止把猜测写成事实

### 3.4 当前阶段的明确分工

当前这 4 份研究输入，必须由虚拟产品经理 Agent 提交首稿：

1. `WeKnora 适配性评估`
2. `知识库竞品与开源方案对比`
3. `内部能力盘点`
4. `待主管拍板问题清单`

虚拟产品主管 Agent 的动作是：

1. 审核研究是否充分
2. 判断是否越级下结论
3. 判断是否能进入总体架构确认
4. 决定 accepted / needs-more-research / blocked-by-architecture / rejected

### 3.5 当前阶段的同步要求

虚拟产品经理 Agent 在当前阶段必须主动同步：

1. 接到任务后，先同步计划做哪 4 份文档
2. 每完成一份文档首稿，立即同步一次
3. 若 30 分钟内仍未形成可提交首稿，也必须同步当前卡点
4. 不允许只在聊天里口头说“正在做”，必须落到文档和提交状态
