# 知识资产平台 v1.1 权限二开降级方案

- 版本：v1.1
- 日期：2026-04-23
- 状态：Active
- 适用范围：当前权限二开执行基线

## 1. 版本定位

`v1.0` 只负责：

1. 部署
2. Logo 替换
3. 版权处理
4. 可运行底座确认

当前真正要做的业务版本是 `v1.1`，但 `v1.1` 不做完整企业知识资产平台，而是做：

`基于腾讯开源现有权限架构微调的数据部知识服务可用版`

## 2. 当前目标

本轮 `v1.1` 只服务以下需求：

1. 数据部个人知识使用
2. 数据部部门知识共享
3. 数据部部门 AI 应用/官方知识服务入口
4. 数据部对业务提供受控知识服务

一句话：

先把“数据部内部可用 + 数据部对业务可服务”做出来，不把系统做成完整企业权限平台。

## 3. 范围内

本轮范围内只做：

1. 企业统一登录接入
2. 基于现有权限架构做最小权限微调
3. 个人私有知识与部门共享知识的边界梳理
4. 数据部官方知识服务入口的最小挂载与使用边界
5. 后台最小配置能力
6. 文档、知识库、服务入口之间的权限边界说明

## 4. 范围外

当前明确不做：

1. 完整企业级权限平台
2. 项目作为独立权限主体
3. 复杂审批流
4. 组织变更自动回收闭环
5. 多应用独立治理体系
6. 工具权限与知识权限全量统一控制面
7. 自研自动标签能力
8. 为了权限二开重写上传链路

## 5. 上游兼容约束

腾讯开源项目后续已有或计划中的“文档上传后自动标签”能力，默认视为上游能力。

本轮约束：

1. 当前权限二开不改上传后自动标签处理链
2. 标签字段可继续存在，但不新增自研自动标签流水线
3. 后续如接入腾讯上游自动标签能力，应尽量直接复用
4. 当前权限策略不能绕开原资产权限去放大标签带来的可见范围

一句话：

这次先做权限，不碰自动标签，避免下次接上游能力时返工。

## 6. 成功标准

本轮完成后，至少满足：

1. 数据部成员可以使用个人知识
2. 数据部成员可以在部门范围内共享和消费知识
3. 数据部官方知识服务入口可以基于部门知识对业务提供问答
4. 权限边界不要求完整平台化，但必须清楚、可配置、可解释
5. 自动标签相关上游能力未被当前二开破坏

## 7. 当前产品判断

当前版本成功的关键，不是“权限模型多完整”，而是：

1. 范围收得住
2. 二开不打穿上游架构
3. 数据部内部能真实用起来
4. 后续还能继续接上游自动标签与更完整平台能力
