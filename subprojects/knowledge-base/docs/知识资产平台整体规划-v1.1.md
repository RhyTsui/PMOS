# 知识资产平台整体规划 v1.1

- 版本：v1.1
- 日期：2026-04-20
- 状态：规划中
- 适用范围：v1.0 基础搭建完成后的正式框架迭代

## 1. 规划背景

### 1.1 当前状态

- `v1.0` 正在开发中，月底上线。
- `v1.0` 的目标是完成基础搭建，验证知识底座、基础接入和最小问答链路。
- `v1.1` 不再只补单点能力，而是要正式搭起知识资产平台的框架。

### 1.2 规划输入

本次规划基于以下输入收敛：

- `inbox` 中既有知识库定位、分层、权限、版本规划材料
- 新增研究文档 `知识库deep-research-report.md`
- 新增研究文档 `知识库需求爆发研究-gemini.md`
- 今日沟通中的产品边界、版本目标与二开决策
- Bloomfire《The 6 Knowledge Management Trends That Redefine Strategic Intelligence in 2026》

### 1.3 外部趋势判断

结合外部研究与产品调研，知识库正在从“文档存储库”升级为“知识基础设施”：

- 从静态归档转向动态知识供给
- 从知识搜索转向知识消费与知识运营
- 从单应用内能力转向 Chat、MCP、模型和未来 Agent 的统一知识底座
- 从手工维护转向自动沉淀、自动修正、自愈式治理

结论：

知识资产平台的首要目标不是再做一个上传文档的知识库，而是搭建一套可持续接入、可治理、可供给的知识基础设施。

## 2. 平台定位

### 2.1 平台名称

`知识资产平台`

### 2.2 一句话定位

统一管理个人与企业业务知识资产，为 Chat、业务 Chat、MCP 与模型调用提供可权限控制、可检索引用、可持续沉淀和修正的知识供给能力。

### 2.3 平台服务对象

平台同时服务两类场景：

1. 个人知识场景
2. 企业业务知识场景

个人场景重点：

- 个人长期上下文
- 会议纪要、文档、订阅内容沉淀
- 各类 Chat 的个人背景挂载

企业场景重点：

- 数据分析 Chat 的知识补强
- 广告业务 Chat 的知识补强
- 业务规则、口径、案例、流程、FAQ 的统一供给

### 2.4 平台核心价值

平台核心价值不再只是“存和搜”，而是 6 个高价值方向：

1. 统一知识接入
2. 统一知识治理
3. 高质量知识检索与引用
4. 面向 Chat 的知识供给
5. 知识修正与确认
6. 知识洞察与演进预留

## 3. 平台总体结构

### 3.1 两类核心资产

平台内两类并列核心资产：

1. `知识资产`
2. `提示词资产`

说明：

- 平台当前叙事主轴仍以知识资产为主。
- 提示词资产保留平台级治理和后续扩展位，但不作为 v1.1 的主开发重点。

### 3.2 统一治理层

知识资产与提示词资产共享统一治理层，但不共享能力处理链路。

统一治理层负责：

- 资源目录
- 资源类型
- 权限
- 版本
- 发布
- 生命周期
- 引用关系
- 来源追踪
- 责任人
- 质量状态

### 3.3 独立能力链路

知识资产域独立能力链路：

- 导入
- 解析
- 结构化抽取
- 切片
- 语义/逻辑分层
- 索引
- 检索
- 引用片段返回

提示词资产域独立能力链路：

- Prompt 内容管理
- 模板变量
- 约束
- 发布快照
- Agent/Chat/MCP Prompt 分类

### 3.4 消费层

v1.1 明确的消费方：

- 平台内 Chat
- 外部 Chat
- 数据分析 Chat
- 广告业务 Chat
- MCP / 模型调用

当前不把 Agent / Workflow 编排作为平台主线，但后续汇总会自然发生在更高层运行平台。

## 4. 场景与需求框架

### 4.1 v1.1 高优先级场景

P0 场景：

1. 个人知识接入闭环
2. 数据分析 Chat 知识补强
3. 广告业务 Chat 知识补强

P1 场景：

1. 会议纪要资产化
2. 知识广场发现与订阅
3. 内部文档与项目系统接入

P2 场景：

1. 知识洞察
2. 知识质量评测
3. 视频内容接入
4. 提示词资产深化

### 4.2 v1.1 要搭起的四个框架

v1.1 不是做全量能力，而是要把以下四个框架先搭起来：

1. 多源接入框架
2. 知识资产模型框架
3. 统一消费框架
4. 治理与运营框架

### 4.3 知识来源框架

v1.1 需要预留和验证的知识来源：

- 手动文本
- 文档上传
- 会议纪要文本
- 公司文档工具
- 项目系统
- 外部网页/订阅导入结果

首版不要求把所有连接器做深，但必须把接入模型、类型和消费关系搭出来。

## 5. 技术路线与二开策略

### 5.1 技术路线

采用 `WeKnora 中二开`

### 5.2 中二开定义

`WeKnora` 作为知识处理与检索底座，不作为最终产品壳。

WeKnora 负责：

- 文档解析
- 切片与索引
- 检索与引用能力
- 基础 RAG 管线
- 可复用的连接器能力

平台层自建：

- 资产模型
- 权限治理
- 业务知识类型体系
- 消费接口与绑定
- 知识广场
- 修正与运营框架
- 场景化产品层

### 5.3 为什么不选重二开

不采用重二开的原因：

- 会过早把平台主系统绑到 WeKnora 的产品结构上
- 会放大未来升级与维护成本
- 会与后续 Prompt 平台、Chat 平台、Agent 汇总层发生边界冲突

### 5.4 v1.1 不做的方向

- 不深改 WeKnora 主系统
- 不以内置 Agent/workflow 为产品主线
- 不把录音转写做进平台核心
- 不把 AI 测试平台并入知识资产平台
- 不做深度 GraphRAG 生产化

## 6. 版本路线

### 6.1 v1.0

目标：

- 完成基础搭建
- 验证底座能力
- 验证最小知识问答链路

### 6.2 v1.1

目标：

- 完成知识资产平台框架搭建
- 打通接入、治理、消费三层骨架
- 支撑个人知识与企业业务知识双主线
- 支撑数据分析 Chat 与广告业务 Chat 的知识补强

### 6.3 v1.2+

方向：

- 深化连接器与订阅
- 建立知识洞察
- 加入知识质量评测
- 深化提示词资产平台
- 逐步支持视频和多模态内容
- 为 Agent 汇总层预留标准知识接口

## 7. 交付原则

v1.1 的成功不以“功能点数量”衡量，而以“框架是否搭起”衡量。

v1.1 交付判定标准：

- 已具备统一知识资产模型
- 已具备统一权限治理骨架
- 已具备多源接入框架
- 已具备可被 Chat / MCP / 业务 Chat 消费的统一知识供给框架
- 已具备知识广场与会议纪要资产的结构性入口

## 8. 结论

`v1.1` 的任务不是把知识库做厚，而是把知识资产平台做对。

具体来说：

- 用 `WeKnora` 解决底座问题
- 用平台层解决治理与产品问题
- 用 `个人知识 + 企业业务知识` 双场景验证平台价值
- 用 `接入、治理、消费、运营` 四个框架为后续迭代搭底座
