# Dataki v1.1 最新确认需求清单
- 版本：v1.1
- 日期：2026-04-24
- 状态：Current
- 适用范围：Dataki v1.1 当前最终确认口径
- 读者对象：产品、设计、前端、后端、测试、AI

## 1. 当前版本定位

Dataki v1.1 当前只保留两条最终需求：

1. 品牌改造
2. 权限改造

这意味着当前版本不再按“大而全平台”推进，也不再继续扩张页面和能力范围，而是围绕现有 WeKnora 产品壳做最小必要改造。

## 2. 总原则

1. 基本保留 WeKnora 产品壳
2. 前端嵌入统一组件库
3. 只做与品牌改造、权限改造直接相关的内容
4. 其他已讨论但未进入这两条主需求的内容，一律不进入当前版本

## 3. 最终需求清单

### 3.1 品牌改造

目标：

- 把当前产品统一改造成 `Dataki`

明确要求：

1. 产品名改为 `Dataki`
2. Slogan 改为 `让知识与数据相遇`
3. LOGO 替换为 Dataki 品牌资源
4. 设计改造以以下资料为准：
   - `docs/sources/inbox/Dataki_brand_pack (1).zip`
   - `docs/sources/inbox/Dataki_ui_specs.zip`
5. 前端改造需要对齐统一组件库，不单独发明组件体系
6. 当前版本只做品牌替换和必要设计调整，不做大规模重写产品结构
7. 对外定位文案统一为 `LLM-wiki-Agent-MCP-Prompts`
8. 不再使用 `内部知识与数据协同工作台` 作为当前版本主文案

### 3.2 权限改造

目标：

- 在现有 WeKnora 基础上补齐 Dataki 当前业务所需的权限与组织能力，并把系统级配置能力收敛到指定管理员

明确要求：

1. 增加 `超管` 角色
2. 超管是指定管理人，不是开放给普通用户申请或自助获得的角色
3. 超管拥有高级设置权限
4. 大模型配置、MCP 配置、系统级高级设置不对普通用户开放
5. 其他用户默认采用统一配置
6. 系统默认内置数据部大模型服务
7. 系统默认内置知识库和数据 MCP 服务
8. 增加用户管理能力
9. 用户组织结构读取小闪，覆盖：
   - 小组
   - 部门
   - 公司
   - 集团
10. 去掉其他登录方式，日常用户登录只保留小闪扫码登录与小闪手机号登录

补充实现口径：

1. 为了让系统可初始化，仍需保留超管落位能力
2. 上述登录收敛指日常用户登录入口只保留小闪扫码登录与小闪手机号登录，不再保留并行登录体系
3. 普通用户只能使用已下发的统一模型与 MCP 能力，不能自行修改配置
4. 对外能力描述不再使用 `统一账号接入 / 统一模型配置 / 统一知识协同`
5. 统一改为 `内置数据部大模型、知识库和数据 MCP 服务`

## 4. 当前版本不做

1. 提示词资产建设
2. 权限中心与统一平台能力整合
3. 发布、版本、评测能力建设
4. 普通用户自定义模型配置
5. 与更多外部身份源并行接入
6. 大规模重做 WeKnora 前台产品结构
7. 大规模重新设计 WeKnora 底层文档解析与检索链路

## 5. 当前版本影响范围

基于这两条最终需求，当前版本实际影响的区域主要是：

1. 登录页与登录链路
2. 用户、角色、组织模型
3. 高级设置 / 系统配置页
4. 模型服务与 MCP 默认配置展示
5. 品牌资源、LOGO、文案、视觉语义
6. 统一组件库嵌入点

## 6. 执行边界

1. 品牌改造优先解决“看起来是 Dataki”
2. 权限改造优先解决“谁能配、谁能看、谁按什么组织进入系统”
3. 不主动扩展到知识资产、Prompt、版本治理等额外产品线
4. 设计、前端、后端都以本文件为当前最高频执行清单

## 7. 当前状态判定

- 用户需求层：`partial`
  原因：最终需求已收敛，但设计落图、SSO 联调、环境恢复还未闭环

- 产品需求层：`solved`
  原因：当前版本只保留两条最终需求，方向与边界已明确
