- 状态：Draft

- 读者对象：产品、设计、前端、后端、测试

## 零、版本定位

### 0.1 版本目标

`v1.1` 目标不是继续做知识库单点能力，而是基于 `WeKnora` 现有底座，完成知识资产平台的一版产品化改造。

本版本必须同时满足两条主线：

1. 支持个人知识、部门知识、项目知识、公共知识的统一接入与管理
2. 支持主对话框、业务 Chat、MCP/工具调用对知识资产的统一消费

### 0.2 版本定义

`v1.1 = 基于 WeKnora 底座，补齐资产模型、权限模型、接入中心、消费入口四层能力，形成知识资产平台的一版可开发基线。`

### 0.3 不做范围

- 不重写 WeKnora 底层文档解析、切片、索引、检索、RAG 能力
- 不把 Agent / Workflow 编排做成主线产品
- 不深做 GraphRAG 生产化
- 不做完整录音转写平台
- 不做测试评测平台整合
- 不做深连接器重接入，首版允许通过导入结果或适配层验证

## 二、核心对象

### 2.1 统一资产对象

所有知识类内容统一抽象为 `asset`。

```json
{
  "asset_id": "asset-001",
  "asset_type": "knowledge_document",
  "title": "广告投放口径说明",
  "scope_type": "department",
  "scope_id": "dept-data",
  "source_type": "manual_upload",
  "source_id": "source-001",
  "owner_id": "user-001",
  "maintainers": ["user-001", "user-002"],
  "permission_policy_id": "policy-001",
  "status": "published",
  "consumer_bindings": ["chat-main", "chat-ad"],
  "created_at": "2026-04-23T00:00:00Z",
  "updated_at": "2026-04-23T00:00:00Z"
}
```

### 2.2 资产类型

`v1.1` 至少支持以下类型：

1. `knowledge_document` 正式知识文档
2. `meeting_note` 会议纪要资产
3. `business_rule` 业务规则 / 业务口径
4. `external_subscription` 外部订阅内容
5. `prompt_asset` 提示词资产

### 2.3 范围类型

统一支持以下 `scope_type`：

1. `personal`
2. `project`
3. `department`
4. `public`

### 2.4 消费绑定对象

统一使用 `consumer_binding` 表示资产被哪些消费端使用。

```json
{
  "binding_id": "binding-001",
  "consumer_type": "chat",
  "consumer_id": "chat-ad",
  "asset_id": "asset-001",
  "binding_mode": "read_only",
  "enabled": true
}
```

## 三、用户场景

### 场景 A：个人知识接入与使用

员工上传个人会议纪要、文档、网页内容，资产默认进入 `personal` 范围，仅本人可见，并可挂载到主对话框中使用。

### 场景 B：部门知识维护与服务

部门知识管理员创建 `department` 级知识资产，发布后可供部门成员和指定业务 Chat 使用。

### 场景 C：项目知识共建

项目成员创建 `project` 范围资产，项目内协作维护，项目结束后支持转交或回收。

### 场景 D：公共知识服务

公共支持库由指定负责人维护，发布后可供公司内部用户检索和指定 AI 应用挂载。

### 场景 E：主对话框统一消费

用户在主对话框中选择多个知识范围与提示词资产，并调用 MCP / Skill / 外部工具进行联合分析。

### 场景 F：业务 Chat 消费知识

数据分析 Chat、广告业务 Chat 等业务入口只消费已授权绑定的知识资产，不直接访问全部底层知识库。

## 四、核心功能

### 4.1 一级信息架构重构（P0）

**功能描述**：将现有 WeKnora 知识库产品壳改造成平台式信息架构。

**要求**：

1. 登录后默认进入主对话框
2. 一级菜单必须包含六个模块
3. 现有知识库入口并入“知识资产”
4. 模型、MCP、知识挂载等入口在主对话框中统一表达

### 4.2 主对话框（P0）

**功能描述**：主对话框作为统一工作入口。

**必须支持**：

1. 选择知识范围
2. 选择资产类型
3. 挂载提示词资产
4. 调用 MCP / Skill / 外部能力
5. 返回引用片段
6. 返回来源信息
7. 最近会话列表

**默认策略**：

1. 默认关闭高级配置
2. 默认使用系统内置模型
3. 默认使用系统分配的 MCP

### 4.3 知识资产管理（P0）

**功能描述**：统一管理知识资产。

**必须支持**：

1. 资产列表
2. 资产详情
3. 资产创建
4. 资产编辑
5. 资产发布
6. 资产归档
7. 资产按范围筛选
8. 资产按类型筛选
9. 资产责任人和维护人配置

### 4.4 提示词资产管理（P0）

**功能描述**：提示词资产作为一级并列资产进入平台。

**必须支持**：

1. 分类管理
2. 新增 / 编辑
3. 版本管理
4. 与主对话框挂载
5. 与业务 Chat 挂载

### 4.5 接入中心（P0）

**功能描述**：统一管理知识来源接入。

**首版接入类型**：

1. 手工文本
2. 文档上传
3. 会议纪要文本
4. 外部 URL
5. 外部订阅结果

**必须支持**：

1. 创建接入任务
2. 查看接入记录
3. 查看接入结果
4. 关联生成的资产

### 4.6 知识广场（P1）

**功能描述**：提供资产发现和订阅入口。

**必须支持**：

1. 推荐资产
2. 热门资产
3. 按类型筛选
4. 按范围筛选
5. 订阅入口

### 4.7 权限管理（P0）

**功能描述**：在统一鉴权输入基础上，自建平台内知识资产权限控制面。

**权限对象**：

1. 资产可见
2. 资产检索
3. 资产引用
4. 资产编辑
5. 资产发布
6. 资产绑定到 AI 应用
7. MCP / 工具调用

**平台必须支持**：

1. 个人默认私有
2. 项目范围隔离
3. 部门范围隔离
4. 公共范围受控开放
5. AI 应用单独作为绑定主体

### 4.8 统一消费接口（P0）

**功能描述**：提供统一消费壳，不允许业务 Chat 直接绕过平台层访问底座知识。

**最小能力**：

1. 按权限检索
2. 指定资产范围
3. 指定资产类型
4. 返回引用片段
5. 返回来源信息
6. 返回绑定信息

**接口示例**：

```json
{
  "query": "广告异常波动的常见排查方法",
  "consumer_type": "chat",
  "consumer_id": "chat-ad",
  "scopes": ["department", "public"],
  "asset_types": ["knowledge_document", "business_rule", "meeting_note"],
  "limit": 10
}
```

### 4.9 会议纪要资产（P1）

**功能描述**：会议纪要作为独立资产类型进入平台。

**必须输出字段**：

1. 原文
2. 结构化摘要
3. 决策项
4. 待办项
5. 待确认问题

1. 统一资产对象定义
2. 范围对象定义
3. 权限策略对象定义
4. 消费绑定对象定义

7. 会议纪要资产结构化输出接口

### 6.2 与 WeKnora 的边界

平台层负责：

1. 资产模型
2. 权限模型
3. 消费绑定
4. 页面信息架构
5. 业务入口表达

WeKnora 底座负责：

1. 解析
2. 切片
3. 索引
4. 检索
5. 引用片段
6. 基础模型链路

## 七、非功能需求

| 需求 | 目标值 | 说明 |
|------|--------|------|
| 权限控制 | 100% 按策略过滤 | 包含检索、引用、绑定 |
| 来源追踪 | 100% 返回来源信息 | 主对话框、业务 Chat 均适用 |
| 可扩展性 | 支持新增资产类型与消费端 | 不改底层主结构 |
| 本地可跑通 | 本地 Demo 可联通 | 作为 v1.1 开发基线 |
| 架构边界 | 平台层与底座层职责明确 | 不允许职责混杂 |
