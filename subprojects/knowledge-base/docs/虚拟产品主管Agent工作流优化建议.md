# 虚拟产品主管 Agent 工作流优化建议

- 版本：v0.2
- 日期：2026-04-19
- 状态：Draft
- 适用对象：真实产品主管、虚拟产品主管 Agent、虚拟产品经理 Agent、项目产品经理
- 目标：把“先研究、再架构、再方案、再 PRD”的顺序固化为强制规则，避免虚拟产品经理越级拍板。

---

## 一、为什么要改

本次知识库项目已经暴露出 4 个共性问题：

1. `WeKnora`、竞品、开源底座调研未完成时，已经开始讨论定版方案。
2. 整体产品线服务架构未确认时，子项目已经在锁页面和模块边界。
3. 虚拟产品经理直接给出“最稳路线”，实际越过了主管和架构确认职责。
4. 缺少研究类中间产物，导致复盘时只有方案，没有依据。

因此，虚拟产品主管 Agent 不能再只是“汇总和输出”，而必须承担流程治理职责。

---

## 二、升级后的标准流程

统一改为 6 个阶段：

1. 问题定义
2. 研究分析
3. 总体架构确认
4. 产品方案
5. PRD / 原型
6. 交付规划

核心原则只有一句：

**凡涉及底座选型、竞品路线、平台边界、公共能力归属的项目，必须先完成研究分析与总体架构确认，再进入产品方案与 PRD 阶段。**

---

## 三、角色分工

### 3.1 真实产品主管

负责：

1. 定义规则和阶段门禁
2. 审批总体架构边界
3. 对候选方向做最终拍板
4. 决定是否允许进入 PRD

### 3.2 虚拟产品主管 Agent

负责：

1. 判断当前项目所处阶段
2. 检查输入是否齐全
3. 给虚拟产品经理分发正确任务
4. 对下级提交物做结构化评审
5. 发现缺口时中止推进并上提问题

### 3.3 虚拟产品经理 Agent

负责：

1. 执行研究任务
2. 产出候选方案和分析文档
3. 整理需求池、问题单、依赖清单
4. 不直接拍板总体路线

---

## 四、强制闸门

### 4.1 研究闸门

出现以下任一情况，必须先做研究分析，不得直接出产品方案：

1. 涉及开源底座选型
2. 涉及竞品路线判断
3. 涉及多个系统的职责拆分
4. 涉及平台级公共能力复用
5. 涉及权限、发布、评测、运行时等基础设施能力

### 4.2 架构闸门

出现以下任一情况，必须先由主管或架构负责人确认总体架构边界：

1. 子项目依赖上层产品线能力
2. 子项目模块边界受平台架构影响
3. 需要复用统一权限、模型网关、MCP、评测、Prompt、知识库等公共能力
4. 需要定义一级菜单或平台模块归属

### 4.3 降级规则

如果研究未完成或架构未确认，虚拟产品经理 Agent 的输出必须自动降级为：

1. 候选方向
2. 风险清单
3. 待确认问题
4. 下一步研究建议

不得写成：

1. 正式方案
2. 定版信息架构
3. 确定版本承诺
4. 默认已拍板的模块拆分

---

## 五、标准交互协议

主管与产品经理不靠口头感觉协作，只靠提交物和评审结论协作。

### 5.1 虚拟产品经理提交格式

每次提交只能是以下类型之一：

1. `problem-definition`
2. `research-note`
3. `option-compare`
4. `architecture-question`
5. `prd-draft`

每次提交必须包含：

1. 标题
2. 当前阶段
3. 核心结论
4. 证据来源
5. 不确定项
6. 影响范围
7. 请求上级决策的问题

### 5.2 虚拟产品主管评审格式

每次评审只能给出以下结论之一：

1. `accepted`
2. `accepted-with-constraints`
3. `needs-more-research`
4. `blocked-by-architecture`
5. `rejected`

并必须写清：

1. 为什么
2. 缺什么
3. 谁来补
4. 补完回到哪个阶段

---

## 六、阶段输出与准入条件

### 阶段 1：问题定义

标准输出：

1. 问题定义文档
2. 目标用户与业务目标
3. 初步范围
4. 初步风险
5. 待研究问题列表

准入条件：

1. 问题不是解法，已经被定义清楚
2. 已明确下一步要研究什么

### 阶段 2：研究分析

标准输出：

1. 开源底座调研报告
2. 竞品对比报告
3. 内部能力盘点
4. 依赖系统列表
5. 候选方向 A/B/C

准入条件：

1. 至少完成一轮外部调研
2. 至少完成一轮内部能力盘点
3. 待拍板问题已整理完毕

### 阶段 3：总体架构确认

标准输出：

1. 总体服务架构图
2. 控制面 / 数据面边界
3. 模块职责表
4. 范围内 / 范围外清单
5. 主管拍板结论

准入条件：

1. 有书面架构边界
2. 子项目归属清晰
3. 未决策问题已压缩到可执行范围

### 阶段 4：产品方案

标准输出：

1. 产品定位
2. 信息架构
3. 模块方案
4. 版本路径
5. 风险与依赖

准入条件：

1. 研究完成
2. 架构确认
3. 不再依赖口头假设

### 阶段 5：PRD / 原型

标准输出：

1. PRD
2. 页面清单
3. 用户流
4. 原型
5. 验收标准

### 阶段 6：交付规划

标准输出：

1. 开发排期
2. 依赖清单
3. 验收计划
4. 上线计划
5. 风险预案

---

## 七、知识库项目的立即适用规则

知识库项目从现在起按以下顺序推进：

1. 完成 `WeKnora` 调研报告
2. 完成知识库竞品对比报告
3. 完成内部能力盘点
4. 形成待主管拍板问题清单
5. 等主管给出总体服务架构图或边界说明
6. 再进入产品方案
7. 最后才进入 PRD / 原型 / 排期

当前必须先确认的事项：

1. 知识库、Prompt、Chat、Agent 的上层归属
2. 权限中心是否复用统一平台能力
3. 发布、版本、评测是否属于本项目范围
4. WeKnora 是底座还是直接产品外壳
5. 首版做平台层还是业务层应用

---

## 八、对现有知识库文档的处理原则

从本规则生效起：

1. 现有 `PRD.md` 视为早期草稿，不视为当前定版基线
2. 现有 `ARCHITECTURE.md` 视为子项目候选架构，不视为上层已拍板架构
3. 后续所有业务需求必须先进入研究池和待拍板问题清单
4. 未过研究与架构闸门，不再新增正式 PRD 内容

---

## 九、直接行动项

1. 将本规则作为知识库项目的主管治理基线
2. 将知识库项目回退到“研究分析阶段”
3. 先补研究包，再补架构确认，再恢复业务方案设计
4. 现有 PRD 和架构文档统一标记为“候选稿/草稿”
