# 需求池模板增强建议

- 版本：v0.1
- 状态：建议稿
- 日期：2026-05-08
- 适用范围：小乔智投及后续 Agent 类项目

---

## 0. 当前模板的问题

现有 `需求池.xlsx` 已有这些优点：

1. 有模块、子模块、需求简称、描述、解决方案
2. 有优先级、状态、收益价值、开发量、性价比
3. 有需求来源、当前版本、上线时间

但它更像：

- 需求管理表
- 排期评估表

还不够像：

- 可直接进入产品分析与验收的需求对象

当前主要缺口有四个：

1. 没把“具体问题”单独拉出来
2. 没把“用户场景”单独拉出来
3. 没把“验收标准”单独拉出来
4. 没把“不做什么 / 边界”单独拉出来

这会导致一个常见问题：

`记录了很多需求，但无法判断是否真的解决了问题。`

---

## 1. 模板增强原则

增强后的需求池模板应该满足：

1. 每条需求只解决一个明确问题
2. 每条需求能映射到具体用户场景
3. 每条需求有可验收标准
4. 每条需求有明确解决方案与边界
5. 每条需求能继续进入功能、原型、交互、前端、后端

## 3. 建议新增的关键字段

### 3.1 问题定义层

建议新增：

1. `具体问题(*)`
   - 不是写方向，而是写当前真实问题
   - 例：广告支持问题高度依赖少数专家，常见排查响应慢且重复

2. `目标用户(*)`
   - 例：投放优化师、广告运营、数分、广告支持

3. `核心场景(*)`
   - 例：用户直接提问“为什么这个计划激活比 BI 少了 28%”

### 3.2 验收层

建议新增：

4. `验收标准(*)`
   - 必须是可以核验的
   - 例：输入广告异常问题后，系统能生成任务、输出路由结果、生成结构化排查结果

5. `衡量指标`
   - 例：误路由率、任务创建成功率、首轮可回答率、追问成功率

6. `验收方式`
   - 例：接口验证 / 前端联调 / 人工验收 / 数据回放

### 3.3 方案层

建议新增：

7. `方案边界(*)`
   - 说明这条需求做什么、不做什么
   - 例：只做自由问答入口，不做多菜单后台导航

8. `依赖项`
   - 例：Dataki、需求池 MCP、媒体文档、运维 Agent

9. `结果物`
   - 例：任务、需求单、排查报告、联调报告、Case

### 3.4 阶段层

建议新增：

10. `阶段归属(*)`
   - 例：P0、P1、P2 或 M1、M2

11. `是否主链路`
   - 例：是 / 否

12. `是否必须本期交付`
   - 例：是 / 否

## 5. 最小增强版

如果你不想一下子把 Excel 扩太宽，最少也建议只新增这 8 个字段：

1. `具体问题(*)`
2. `目标用户(*)`
3. `核心场景(*)`
4. `验收标准(*)`
5. `方案边界(*)`
6. `依赖项`
7. `结果物`
8. `阶段归属(*)`

这 8 个字段一补，需求池就会从“记录表”更接近“需求对象表”。

---

## 6. 小乔项目为什么特别需要这些字段

小乔不是普通页面需求，它是：

- 数字员工
- 多 Agent 体系
- 工作流型产品

这种项目如果没有：

1. `具体问题`
2. `核心场景`
3. `验收标准`
4. `方案边界`

就会非常容易出现：

- 只有规划，没有需求
- 只有功能，没有问题定义
- 只有页面，没有验收标准
- 只有想法，没有阶段边界

所以小乔这类项目对模板要求比普通后台页更高。

## 7. 小乔项目的推荐填法示例

以“自由问答统一入口”为例：

| 字段 | 示例 |
| --- | --- |
| 模块 | 小乔智投 |
| 子模块 | 对话入口 |
| 需求简称 | 自由问答统一入口 |
| 具体问题 | 用户进入系统后需要先理解模块结构，无法像通用大模型一样直接提问 |
| 目标用户 | 投放优化师 / 广告运营 / 数分 |
| 核心场景 | 用户打开小乔后直接输入问题，而不是先点菜单 |
| 描述 | 小乔需要提供统一自由问答入口 |
| 解决方案 | 建设单对话入口，后台隐式做路由判断与工作流分发 |
| 方案边界 | 只做一个统一入口，不做多菜单功能导航作为主交互 |
| 依赖项 | 路由服务、任务服务、Dataki |
| 结果物 | Conversation / Message / RoutingDecision |
| 验收标准 | 用户发起输入后，系统能返回自然回复或生成业务任务，不要求先选模块 |
| 阶段归属 | P0 |

## 8. 建议执行方式

正确顺序建议改成：

1. 先保留原始 `需求池.xlsx`
2. 新增一版“增强字段模板”
3. 小乔先按增强模板起草
4. 再决定是否把增强字段正式并入全局需求池

这样不会一下子打断现有流程，但能让 Agent 类项目先用更对的模板。
