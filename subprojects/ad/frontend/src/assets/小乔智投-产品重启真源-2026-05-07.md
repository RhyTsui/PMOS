## 2. 用户需求层

### 2.1 原始问题

广告投放相关的需求沟通、短链追踪、素材评估、广告分析、数据 gap 异常排查、归因口径解释等工作，当前长期依赖极少数核心人员承接，已经出现明显的人力瓶颈。

### 2.2 用户为什么现在必须解决它

因为业务支持面持续扩大，但承接人力没有同步扩充，且组织存在持续流动：

- 多个投放团队
- 多个设计团队
- 多个市场推广团队
- 30 多个游戏项目组成员需要看推广效果
- 团队每年都有离职与新人进入

当前状态下，广告支持、需求沟通、问题排查、口径解释高度依赖单点经验人力，已经不可持续。

### 2.3 用户当前如何凑合完成

当前主要依赖人工沟通和人工判断完成：

- 由单点负责人对接多个团队
- 通过反复解释归因口径和逻辑完成认知补齐
- 通过人工排查广告数据 gap 异常完成问题定位
- 通过人工沟通推进需求标准化、开发协作和质量监控

### 2.4 用户最痛的损失

当前主要损失包括：

- 人效低
- 判断慢
- 风险高
- 新老成员都需要反复接受同类解释
- 低效沟通持续占用关键支持人力

### 2.5 用户需求闭环判定

- 状态：unsolved
- 说明：尚未完成问题定义，无法判断是否真正解决用户场景。

## 3. 产品需求层

### 3.1 项目愿景

小乔智投的项目定义需要明确区分三个层次：

- 战略层：为什么做这件事
- 产品层：这个产品本质上是什么
- 能力机制层：AI 与 agent / workflow 在里面如何发挥作用

当前已确认的战略层表述为：

利用大模型能力和 agent / workflow 的产品搭建经验，锁定核心业务场景，以全流程闭环方式提升业务支持效率，并通过持续蒸馏个人与团队经验，逐步解放团队去做更有价值的事情。

当前已确认的能力机制层表述为：

让 AI 深度参与判断和执行，强调 agent 化。

当前产品层表述仍待确认。

但当前已收敛出的产品形态方向为：

小乔智投不是单点问答 Agent，也不是传统广告后台，而是一个以对话入口驱动、围绕广告支持闭环逐步扩展到分析、执行、创意生产的广告 Agent 工作台。

### 3.1.1 战略层解读

这一定义意味着：

- 目标不是单点做一个 AI 功能，而是做可持续复用的业务支持提效体系
- 目标不是简单替人，而是把高频、重复、依赖专家经验的支持工作沉淀下来
- 重点不是局部提效，而是围绕核心业务场景形成端到端闭环
- 最终价值不是“多一个系统”，而是让团队把时间转移到更高价值工作

### 3.1.2 能力机制层解读

这一定义意味着：

其核心特征包括：

- AI 不只负责回答问题，还要参与判断
- AI 不只提供建议，还要逐步参与执行
- 产品中心不是页面容器，而是 agent 化的任务闭环
- 目标不是一次性全自动，而是形成可控、可追踪、可逐步放权的代理体系

### 3.2 项目背景

小乔智投的启动背景不是单纯的产品创新，而是广告投放支持体系已经出现明显的人力与协作失衡。

当前业务环境具有以下特征：

- 超过 20 个优化师
- 30 个设计师
- 10 个市场运营
- 30 多个游戏项目组成员需要查看推广效果
- 涉及短链追踪、素材评估、广告分析等多类高频需求

与此同时，实际承接侧已经从“1 人主对接 + 2 产品 + 2 测试协助沟通”收缩到接近单点承接，导致以下问题持续放大：

- 广告数据 gap 异常排查成本高
- 归因口径逻辑需要对新人和老人反复解释
- 需求沟通低效且依赖个人经验
- 标准化需求开发效率不足
- 问题排查效率不足
- 质量监控效率不足

因此，这个项目的真实起点是：用 agent 化方式把原本依赖单点专家人力的重复判断、重复解释、重复协同、重复排查工作沉淀成可复用能力。

### 3.3 业务目标

当前可确认的业务目标分三层：

- 近期目标：先替代广告支持体系中高频、重复、强依赖专家经验的沟通、排查、联调工作
- 中期目标：把已存在的广告 BI 数据分析能力通过 MCP 接入统一工作台，让分析能力成为同一入口内的任务态
- 远期目标：把广告执行与创意生产逐步纳入同一 Agent 工作台，形成覆盖支持、分析、执行、创意的广告业务代理体系

### 3.4 核心角色

当前已可识别的核心角色包括：

- 投放优化师
- 数据支持 / 广告产品 / 分析人员
- 开发与测试协作角色
- 市场、设计、项目组等需要获得支持或查看结果的业务角色

### 3.5 核心用户故事

当前已收敛出的核心用户故事主线为：

用户进入同一个对话工作台，不再先判断自己该去哪个系统，而是直接描述目标或问题；系统先识别意图，再在同一任务上下文里完成使用帮助、需求标准化、问题排查、广告联调，并把结果沉淀为后续可复用资产。

### 3.6 第一阶段产品范围

第一阶段不是纯单页聊天产品，也不是传统多菜单后台，当前范围应拆为两层：

#### 3.6.1 主工作台

以 `单对话主工作台` 为统一入口，承载四类高频主任务：

- 使用帮助
- 需求沟通
- 问题排查
- 广告联调

#### 3.6.2 专项能力页

第一阶段同时纳入三类专项页，但它们不是主入口首屏：

- 监控大屏页（P1）
- 素材分析页（P2）
- 广告预测页（P2）

#### 3.6.3 明确已知的后续接入节奏

- 2026 年 6 月：接入广告 BI 数据分析 MCP
- 2026 年 8 月：推进广告自动搭建、自动暂停等广告执行 Agent
- 2026 年 9 月：推进基于创意脚本衍生视频的素材生产能力

#### 3.6.4 当前范围边界判断

- 当前阶段要为后续分析、执行、创意能力预留产品结构
- 当前阶段不应把 6 月、8 月、9 月的能力伪装成“已经完成的本期交付”
- 当前阶段前端原型需要体现整体定位，但交付优先级仍需区分主入口与专项页
- 当前阶段同时要为应用支撑层预留产品结构，避免业务页和运行底座脱节
- 当前阶段的流程编排与任务状态应优先落在自有 runtime，而不是外部低代码编排平台
