## 1. 需求对象：自由问答统一入口

### 基础信息

- 模块：小乔智投
- 子模块：对话入口
- 需求简称：自由问答统一入口
- 需求分类：产品需求
- 优先级：P0
- 阶段归属：M1 / P0
- 是否主链路：是
- 是否必须本期交付：是

### 具体问题

当前广告技术服务请求需要用户先理解系统结构、找人、找群、找页面，进入成本高，且不符合数字员工形态。

如果入口仍然是菜单或功能页，用户会把小乔理解成后台工具，而不是统一服务窗口。

### 目标用户

1. 投放优化师
2. 广告运营
3. 数分
4. 与广告支持协作的其他业务同学

### 核心场景

用户打开小乔后，直接输入：

- “为什么这个计划激活比 BI 少了 28%”
- “这个指标是什么意思”
- “帮我开始联调”
- “我有一个新媒体回传需求”

而不是先选：

- 帮助页
- 排查页
- 联调页

### 用户预期

像在和一个懂广告技术服务的人直接对话，而不是先学习一套产品导航。

### 解决方案

建设一个统一对话入口：

1. 首屏以输入框为主
2. 支持自然语言直接提问
3. 后台隐式路由到不同工作流
4. 结果通过对话区和动态结果面板展开

### 方案边界

做：

1. 单对话入口
2. 输入即开始
3. 对话驱动任务流

不做：

1. 多菜单作为主交互
2. 多入口并列作为首页主形态
3. 先填表再开始的强制流程

### 依赖项

1. 路由判断服务
2. 会话服务
3. 任务服务
4. 前端单工作台页面

### 结果物

1. Conversation
2. Message
3. 首页工作台视图

### 验收标准

1. 用户进入小乔后无需先选模块即可发送问题
2. 首条消息发送后系统必须返回自然回复
3. 命中业务工作流时可继续生成任务，不要求用户先切页面
4. 前端主视图以对话区为中心，而不是菜单页

### 衡量指标

1. 首条提问发送成功率
2. 从进入页面到首次提问的中位时长
3. 首轮有回复率

### 验收方式

1. 前端交互验收
2. 接口联调验收
3. 走查首页主路径

## 2. 需求对象：工作/非工作隐式判断

### 基础信息

- 模块：小乔智投
- 子模块：路由判断
- 需求简称：工作/非工作隐式判断
- 需求分类：产品需求
- 优先级：P0
- 阶段归属：M1 / P0
- 是否主链路：是
- 是否必须本期交付：是

### 具体问题

如果所有输入都强行业务化，小乔会变得僵硬且高误判；如果所有输入都按普通 Chat 回答，小乔又失去业务工作流价值。

需要先判断这条输入是否属于广告技术服务相关工作。

### 目标用户

1. 所有进入小乔的使用者

### 核心场景

- “今天天气真好” 应按普通对话回答
- “这个广告归因指标是什么意思” 应进入广告工作流
- “为什么这个计划激活比 BI 少了 28%” 应进入重工作流

### 用户预期

不相关内容不会被硬拉进业务流程；相关内容能被正确接住。

### 解决方案

在对话入口后增加第一层隐式判断：

1. 非工作：按普通大模型 Chat 回答
2. 工作：继续进入业务域判断和工作流分发
3. 判断不清：允许追问

### 方案边界

做：

1. 工作/非工作判断
2. 追问补齐
3. 路由记录

不做：

1. 把分类过程强暴露给用户
2. 因为系统暂时没能力就直接把业务问题打成非工作

### 依赖项

1. routing_service
2. clarification_service
3. RoutingDecision 存储

### 结果物

1. RoutingDecision
2. clarification 消息

### 验收标准

1. 非工作输入不创建业务任务
2. 工作输入进入广告域后可继续分发
3. 判断不清时能返回追问而不是拒答或沉默
4. 每次路由都有记录可回查

### 衡量指标

1. 非工作误建任务率
2. 工作误拒答率
3. 追问后成功分流率

### 验收方式

1. 路由样例测试
2. 接口返回检查
3. bad case 回放

## 3. 需求对象：使用帮助轻工作流

### 基础信息

- 模块：小乔智投
- 子模块：轻工作流
- 需求简称：使用帮助工作流
- 需求分类：业务需求
- 优先级：P0
- 阶段归属：M1 / P0
- 是否主链路：是
- 是否必须本期交付：是

### 具体问题

广告技术服务中大量问题属于高频重复解释：

1. 指标解释
2. 逻辑说明
3. 系统去哪看
4. 一般知识问答

这类问题消耗支持人力，但又不应升级为重工作流。

### 目标用户

1. 投放优化师
2. 广告运营
3. 数分

### 核心场景

- “首日 ROI 倒推是什么意思”
- “这个指标口径怎么看”
- “去哪个系统看回传结果”

### 用户预期

快速得到清楚解释、引用来源和系统路径，不必先找真人支持。

### 解决方案

建设帮助型轻工作流：

1. 接 Dataki 知识
2. 支持示例回答
3. 支持系统路径与跳转说明
4. 保留对话上下文
5. 必要时升级为重工作流

### 方案边界

做：

1. 指标解释
2. 逻辑说明
3. 系统路径说明
4. 一般知识问答

不做：

1. 直接输出排查结论
2. 直接输出联调报告
3. 把真实异常伪装成帮助问题继续轻处理

### 依赖项

1. Dataki
2. 帮助工作流服务
3. 系统路径配置

### 结果物

1. help_answer
2. 引用来源
3. 推荐下一步

### 验收标准

1. 输入帮助类问题后，系统进入轻工作流
2. 返回结果至少包含解释或路径说明
3. 若识别出其实是异常排查诉求，可升级到重工作流

### 衡量指标

1. 首轮回答率
2. 追问升级率
3. 人工转接率

### 验收方式

1. 帮助类样例回放
2. 前端结果面板验收

## 4. 需求对象：问题排查重工作流

### 基础信息

- 模块：小乔智投
- 子模块：重工作流
- 需求简称：问题排查工作流
- 需求分类：业务需求
- 优先级：P0
- 阶段归属：M1 / P0
- 是否主链路：是
- 是否必须本期交付：是

### 具体问题

广告异常排查是当前最核心、最痛的支持工作：

1. 数据 gap
2. 归因不一致
3. 回推异常
4. 媒体数据异常
5. 客户端上报异常

当前高度依赖少数专家人工排查，响应慢，且同类问题重复劳动严重。

### 目标用户

1. 投放优化师
2. 广告运营
3. 广告支持
4. 数分

### 核心场景

- “为什么这个计划激活比 BI 少了 28%”
- “这个媒体回推是不是出问题了”
- “归因结果和报表为什么对不上”

### 用户预期

小乔能先接住问题，给出像人工支持一样的初步结论、证据链和下一步动作。

### 解决方案

建设排查重工作流：

1. 创建排查任务
2. 写入业务上下文
3. 串联上报、预处理、归因、回推、媒体数据、客户端记录
4. 输出结构化排查结果
5. 支持后续追问

### 方案边界

做：

1. 初步排查结论
2. 证据链组织
3. 风险和下一步建议

不做：

1. 没有证据时强行确定根因
2. 第一阶段自动执行高风险修复动作

### 依赖项

1. diagnosis_workflow
2. 数据源适配器
3. evidence_service

### 结果物

1. diagnosis_report
2. EvidenceItem
3. Case 候选

### 验收标准

1. 输入异常问题后，系统创建 diagnosis 任务
2. 返回结果包含结论、证据链、置信度、下一步建议中的至少三项
3. 结果可在任务详情中回看

### 衡量指标

1. 排查任务创建成功率
2. 首轮可用结论率
3. 误路由率

### 验收方式

1. 接口样例验证
2. 结构化结果验收
3. 典型异常回放

- 优先级：P0
- 阶段归属：M1 / P0
- 是否主链路：是
- 是否必须本期交付：是

### 具体问题

广告支持需求目前口头化严重：

1. 诉求模糊
2. 缺字段
3. 标准化不足
4. 后续对接成本高

### 目标用户

1. 投放优化师
2. 广告运营
3. 需要发起对接需求的业务同学

### 核心场景

- “我要接一个新的媒体回传需求”
- “帮我收一下这个广告对接需求”
- “这个媒体配置需求怎么提”

### 用户预期

模糊诉求被自动收口成结构化需求，而不是来回补材料。

### 解决方案

建设需求沟通重工作流：

1. 对话中追问缺失字段
2. 调 Dataki / 媒体文档
3. 调需求池 MCP
4. 生成结构化需求对象
5. 预留写回需求池入口

### 方案边界

做：

1. 需求收口
2. 缺字段追问
3. 标准化输出

不做：

1. 第一阶段直接替代完整项目管理系统
2. 第一阶段承诺自动排期和自动开发

### 依赖项

1. demand_workflow
2. 需求池 MCP
3. 标准需求模板
4. 媒体文档能力

### 结果物

1. demand_form
2. 缺失字段列表
3. 依赖项

### 验收标准

1. 输入需求类问题后创建 demand 任务
2. 系统能追问缺失字段
3. 最终生成结构化需求结果

### 衡量指标

1. 需求收口完整率
2. 平均追问轮数
3. 需求池写入前补充率

### 验收方式

1. 对话样例验收
2. 需求结果结构验收

## 6. 需求对象：广告联调重工作流

### 基础信息

- 模块：小乔智投
- 子模块：重工作流
- 需求简称：广告联调工作流
- 需求分类：业务需求
- 优先级：P0
- 阶段归属：M1 / P0
- 是否主链路：是
- 是否必须本期交付：是

### 具体问题

广告联调现在依赖人工串行推进：

1. 账号分散
2. 设备分散
3. 自动化状态不可见
4. 日志和结果不集中

### 目标用户

1. 广告支持
2. 投放运营
3. 联调相关协作人

### 核心场景

- “帮我开始联调”
- “这次巨量联调先带默认账号和设备”
- “现在联调卡在哪一步”

### 用户预期

一轮联调能在一个窗口里看到准备项、执行状态、日志和结果。

### 解决方案

建设联调重工作流：

1. 创建 debugging 任务
2. 带入默认账号
3. 带入白名单设备
4. 展示自动化执行状态
5. 生成联调报告

### 方案边界

做：

1. 联调任务入口
2. 统一状态展示
3. 联调日志与结果沉淀

不做：

1. 第一阶段全自动无监督执行所有高风险动作
2. 第一阶段覆盖所有媒体和所有设备类型

### 依赖项

1. debugging_workflow
2. 浏览器自动化
3. App 自动化
4. 账号/设备配置

### 结果物

1. debugging_report
2. 执行日志
3. 失败节点

### 验收标准

1. 输入联调请求后创建 debugging 任务
2. 返回当前执行状态
3. 结果中可看到至少一个执行阶段和下一步动作

### 衡量指标

1. 联调任务创建成功率
2. 可视化日志覆盖率
3. 联调结果生成率

### 验收方式

1. 接口联调
2. 前端工作台验收
3. 日志回放检查

## 7. 需求对象：会话与任务沉淀

### 基础信息

- 模块：小乔智投
- 子模块：运行时
- 需求简称：会话与任务沉淀
- 需求分类：技术需求
- 优先级：P0
- 阶段归属：M1 / P0
- 是否主链路：是
- 是否必须本期交付：是

### 具体问题

如果只有聊天回复，没有任务、上下文、结果沉淀，小乔就不是数字员工，而只是一次性聊天工具。

### 目标用户

1. 所有使用小乔的业务同学
2. 广告支持维护人员

### 核心场景

用户今天问过的问题、系统给出的结论、当前任务状态、后续追问和历史记录，都需要可追踪。

### 用户预期

小乔记得之前说过什么，也能回看之前做过什么。

### 解决方案

建设小乔自有业务运行时：

1. Conversation
2. Message
3. Task
4. TaskContext
5. WorkflowResult
6. RoutingDecision

### 方案边界

做：

1. 小乔自己的业务上下文
2. 会话与任务沉淀
3. 结果可回看

不做：

1. 用连弩承接业务会话历史
2. 用 Dataki 直接替代业务运行时

### 依赖项

1. 后端运行时
2. 数据表
3. 任务服务

### 结果物

1. 会话记录
2. 任务记录
3. 结果记录
4. 后续 Case 候选

### 验收标准

1. 一条业务消息后至少产生 Message、RoutingDecision、Task、WorkflowResult
2. 刷新后仍可读回会话和任务
3. 任务详情可读取上下文和结果

### 衡量指标

1. 任务沉淀成功率
2. 历史回读成功率
3. 结果读取成功率

### 验收方式

1. 接口验证
2. 数据结构核验
3. 历史会话回读验收

- 是否主链路：是
- 是否必须本期交付：是

### 具体问题

当前页面形态偏后台功能台，既不符合数字员工形态，也不满足原型确认标准。

### 目标用户

1. 业务使用者
2. 产品确认者
3. 前后端协作开发者

### 核心场景

用户进入页面后，像在 Codex 一样：

1. 先看到小乔
2. 直接提问
3. 在对话中推进任务
4. 在右侧看结构化结果

### 用户预期

页面看起来就是一个真实可用的数字员工工作台，而不是功能展板。

### 解决方案

重做为单对话工作台：

1. 顶部身份与状态区
2. 中央对话区

做：

1. 首屏是否可直接进入对话
2. 任务路径是否一眼可懂
3. 原型评审通过率

2. 原型走查
3. 前端确认稿验收
