## 0. 文档目的

本文档用于把小乔智投第一阶段的后端实现边界正式收清。

目标不是先定具体技术细节，而是先回答这几个问题：

1. 小乔自己的业务运行时由哪些模块组成
2. 哪些模块属于统一对话入口
3. 哪些模块属于多 Agent 调度与工作流承接
4. 哪些模块负责结果沉淀、Case 沉淀和历史回看
5. Dataki、连弩、运维 Agent 与小乔后端的边界在哪里

---

## 1. 后端结构设计原则

### 1.1 小乔必须有自己的业务运行时

小乔后端不是：

1. PMAIOS runtime
2. n8n 流程拼装
3. Dataki 知识库后端
4. 连弩测试平台后端

小乔后端是自己的业务运行时，负责：

1. 对话接入
2. 路由判断
3. 工作流建任务
4. 上下文维护
5. 结果沉淀
6. 多 Agent 调度与回写

### 1.2 前台单人格，后台多模块

用户只和`小乔`对话。

后端内部则拆为：

1. 入口层
2. 路由层
3. 任务层
4. 工作流层
5. 结果层
6. 外部能力接入层

不要把多 Agent 路由过程直接暴露成前台主要概念。

### 1.3 轻工作流和重工作流共用同一运行框架

使用帮助、指标解释、系统导航等轻工作流，与排查、需求、联调等重工作流，不应建设两套完全割裂的后端体系。

统一点包括：

1. Conversation
2. Message
3. Task
4. TaskContext
5. WorkflowResult
6. RoutingDecision

### 1.4 高风险环节优先确定性

第一阶段后端要允许自然对话，但不能把重任务完全交给自由生成。

因此：

1. 自由对话用于接待、追问、解释、总结
2. 路由判断要可记录
3. 排查、需求、联调要进入确定性工作流服务
4. 结果物必须结构化落库

---

## 2. 小乔第一阶段后端总分层

建议第一阶段后端拆成六层：

1. `API 接入层`
2. `会话与路由层`
3. `任务与工作流层`
4. `结果与沉淀层`
5. `Agent / 外部能力适配层`
6. `基础存储与治理层`

---

## 3. API 接入层

### 3.1 角色

负责承接前端请求，不承载复杂业务判断。

### 3.2 建议职责

1. 参数接收与校验
2. 调用应用服务
3. 返回统一响应体
4. 错误码映射

### 3.3 建议模块

1. `src/ad/api/routes/xiaoqiao_conversations.py`
2. `src/ad/api/routes/xiaoqiao_messages.py`
3. `src/ad/api/routes/xiaoqiao_tasks.py`
4. `src/ad/api/routes/xiaoqiao_results.py`
5. `src/ad/api/routes/xiaoqiao_cases.py`
6. `src/ad/api/routes/xiaoqiao_workspace.py`

### 3.4 当前判断

当前代码里的：

1. `integration.py`
2. `flow.py`
3. `full_design.py`

属于旧 demo 路由，不是小乔正式业务运行时结构。

## 4. 会话与路由层

### 4.1 角色

这是第一阶段最关键的后端层。

它负责把：

`用户自由输入 -> 路由判断 -> 必要追问 -> 进入对应工作流`

真正执行起来。

### 4.2 子模块

建议拆成四个服务：

1. `conversation_service`
2. `message_service`
3. `routing_service`
4. `clarification_service`

### 4.3 各服务职责

#### conversation_service

负责：

1. 创建会话
2. 读取会话
3. 维护会话状态
4. 维护最近任务摘要

#### message_service

负责：

1. 写入用户消息
2. 写入助手回复
3. 关联消息与任务
4. 读取会话消息流

#### routing_service

负责：

1. 判断是否业务相关
2. 判断业务域
3. 判断广告域诉求类型
4. 判断轻工作流 / 重工作流
5. 记录路由结论

当前第一阶段至少支持输出：

1. `is_business_related`
2. `business_domain`
3. `intent_type`
4. `workflow_level`
5. `clarification_needed`
6. `decision_reason`

#### clarification_service

负责：

1. 缺字段判断
2. 追问生成
3. 追问后的上下文补写
4. 二次路由

### 4.4 设计判断

第一阶段最关键的不是“模型直接回答得多聪明”，而是：

1. 不要误拒答
2. 不要误建重任务
3. 判断不清时要会追问
4. 路由过程要可追溯

---

## 5. 任务与工作流层

### 5.1 角色

负责把广告域请求正式转成可运行、可回看、可沉淀的任务。

### 5.2 建议子模块

1. `task_service`
2. `task_context_service`
3. `workflow_dispatch_service`
4. `workflow_services/`

### 5.3 task_service

负责：

1. 创建任务
2. 更新任务状态
3. 读取任务详情
4. 维护任务生命周期

### 5.4 task_context_service

负责：

1. 写入业务上下文
2. 写入媒体、应用、计划、设备、时间窗等字段
3. 记录缺失字段
4. 支撑后续追问与回写

### 5.5 workflow_dispatch_service

负责根据路由结论，把任务分发到具体工作流。

第一阶段至少分发到：

1. `help_workflow`
2. `diagnosis_workflow`
3. `demand_workflow`
4. `debugging_workflow`

后续预留：

1. `monitor_workflow`
2. `material_analysis_workflow`
3. `forecast_workflow`

### 5.6 workflow_services

建议按工作流拆文件，不按 Agent 名字先拆。

原因：

第一阶段真正稳定的是“任务流”，不是“人格化 Agent 名册”。

建议目录：

1. `src/ad/services/xiaoqiao/workflows/help_workflow.py`
2. `src/ad/services/xiaoqiao/workflows/diagnosis_workflow.py`
3. `src/ad/services/xiaoqiao/workflows/demand_workflow.py`
4. `src/ad/services/xiaoqiao/workflows/debugging_workflow.py`

每个工作流都要统一提供：

1. `start`
2. `continue_with_followup`
3. `build_result`
4. `recommend_next_action`

## 6. 结果与沉淀层

### 6.1 角色

负责把对话结果从“说过了”变成“留得下、看得见、能复用”。

### 6.2 建议子模块

1. `result_service`
2. `evidence_service`
3. `case_service`
4. `workspace_view_service`

### 6.3 result_service

负责：

1. 生成 WorkflowResult
2. 统一不同工作流的结果体
3. 维护结果摘要
4. 支撑前端右侧结构化面板

### 6.4 evidence_service

负责：

1. 写入排查证据
2. 写入联调日志摘要
3. 写入外部数据源引用摘要
4. 生成可回放证据链

### 6.5 case_service

负责：

1. 从任务升级为 Case
2. 维护 Case 状态
3. 输出可复用要点
4. 为后续知识沉淀提供入口

### 6.6 workspace_view_service

负责：

1. 聚合工作台首页视图
2. 聚合最近任务
3. 聚合状态摘要
4. 聚合专项页所需视图数据

这样前端不必自己拼十几个独立请求。

## 7. Agent / 外部能力适配层

### 7.1 角色

这一层不存业务真相，只负责连接外部能力。

### 7.2 原则

外部系统不要直接侵入核心任务模型。

小乔内部应保持：

1. 自己的 Task
2. 自己的 TaskContext
3. 自己的 WorkflowResult
4. 自己的 CaseRecord

外部能力通过适配器挂进来。

### 7.3 第一阶段建议适配器

1. `knowledge_adapter`
   - Dataki 知识库
2. `evaluation_adapter`
   - 连弩测试平台
3. `ops_agent_adapter`
   - 运维 Agent
4. `notification_adapter`
   - 小闪通知
5. `media_doc_adapter`
   - 媒体文档 / 爬虫
6. `mcp_adapter`
   - 跨部门 MCP
7. `upload_adapter`
   - 图片 / 文档上传

### 7.4 第一阶段不要做的事

不要让：

1. Dataki 直接成为小乔主存储
2. 连弩直接承接业务会话历史
3. 运维 Agent 直接改写小乔任务表
4. 外部服务直接定义小乔业务状态机

## 8. 基础存储与治理层

### 8.1 角色

负责支撑运行时的持久化、配置和可观测性。

### 8.2 建议子模块

1. `repositories/`
2. `models/`
3. `config/`
4. `telemetry/`

### 8.3 repositories

按数据表真源对应：

1. `conversation_repository`
2. `message_repository`
3. `task_repository`
4. `task_context_repository`
5. `workflow_result_repository`
6. `evidence_repository`
7. `case_repository`
8. `routing_decision_repository`

### 8.4 telemetry

第一阶段至少记录：

1. 路由结果
2. 追问次数
3. 任务创建次数
4. 工作流完成状态
5. 错误与降级原因

这一层是后续接连弩和持续优化机制的前置基础。

## 10. 第一阶段最小后端闭环

如果只做最小可运行版本，建议先落这条闭环：

1. 创建 Conversation
2. 写入用户 Message

4. 必要时追问
5. 创建 Task
6. 写入 TaskContext
7. 生成 WorkflowResult
8. 聚合 Workspace 视图

最小服务集合：

1. `conversation_service`
2. `message_service`
3. `routing_service`
4. `task_service`
5. `task_context_service`
6. `workflow_dispatch_service`
7. `help_workflow`
8. `diagnosis_workflow`
9. `result_service`
10. `workspace_view_service`

这样先能打通：

1. 自由问答入口

3. 核心重工作流排查

再继续补：

1. `demand_workflow`
2. `debugging_workflow`
3. `evidence_service`
4. `case_service`

## 11. 与用户需求的回映

### 11.1 用户需求层

用户真正要的是：

1. 像 Codex / DeepSeek 一样自由问答
2. 不要老拒答、老沉默
3. 小乔像一个统一窗口先接住问题
4. 背后可以调多 Agent，不要求用户先理解系统结构
5. 广告技术服务相关任务真的能推进，不只是聊天

### 11.2 当前后端结构如何支撑

1. `会话与路由层`支撑自由问答和隐式判断
2. `任务与工作流层`支撑真的把事推进下去
3. `结果与沉淀层`支撑不是只说不留
4. `适配层`支撑多 Agent 与外部能力接入

### 11.3 当前闭环判断

- 后端结构层状态：`solved`

说明：

后端结构边界已经收清，足以作为下一步后端代码切分与接口实现的正式依据。

但对原始用户场景“数字员工可稳定对外服务”来说，当前整体仍是：

- 用户需求层：`partial`

原因：

1. 结构已定
2. 接口已定
3. 业务运行时代码尚未按该结构落地
