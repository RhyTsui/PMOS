## 0. 本轮更新说明

本版不再沿用“先定义 4.0 再往下拆”的口径，而是按“重新起一套小乔智投系统”重写。

与上一版相比，本版新增了三层落地约束：

1. 明确小乔不是从 0 发明广告业务，而是重新组织现有广告数仓、日志、媒体后台、联调流程、知识资产与协作链路。
2. 明确第一阶段四条业务流都必须跑通，不接受 demo / 示意版交付。
3. 明确后续需求与功能文档必须围绕真实业务对象、真实字段包、真实结果物来写，而不是只写抽象页面模块。

## 1. 产品定义

### 1.1 一句话定义

小乔智投是一套面向广告支持、投放协同、数据排查与联调执行的新系统：  
以前台单对话工作台为统一入口，以任务运行时为骨架，以现行广告能力封装为可调用能力层，承接使用帮助、需求沟通、问题排查、广告联调四条核心业务流。

### 1.2 它不是什么

小乔智投不是：

1. 旧广告平台页面的简单换皮。
2. 只会聊天的通用问答机器人。
3. 只做指标解释的知识库门户。
4. 只做 BI 展示的看板系统。
5. 只做一条演示链路的 AI demo。

### 1.3 它是什么

小乔智投是：

1. 一个重新组织广告业务入口的新系统。
2. 一个把现行线上线下流程重新编排为任务流的新工作台。
3. 一个把现有数仓、日志、媒体、联调、知识能力重新封装为可调用能力层的 AI 应用。
4. 一个要求结果物可沉淀、可回看、可继续协同，而不是只返回一段对话文本的企业系统。

---

## 2. 为什么要重新起这套系统

### 2.1 现状不是“缺一个页面”，而是承接方式断裂

现行广告业务已经有很多年沉淀，底层并不空白：

1. 有广告数仓。
2. 有 BI / 报表体系。
3. 有归因、回传、上报、媒体后台等业务链路。
4. 有支持、联调、排查、需求沟通等长期运行的人工作业流程。
5. 有知识、经验、规则、排查套路，但大量隐含在人和分散系统里。

真正缺的是统一承接方式：

1. 用户不知道从哪里进。
2. 问题进入后没有稳定任务化。
3. 结果出来后没有结构化沉淀。
4. 不同系统能力彼此孤立，无法被 agent 连起来。

### 2.2 重新起系统的必要性

如果继续在旧系统上零散加入口，会继续遇到同样的问题：

1. 用户入口仍然分散。
2. 帮助、需求、排查、联调仍然各自为政。
3. 结果仍然只停留在人肉沟通和临时处理。
4. 现有能力即使强，也无法成为可编排的系统能力。

所以本项目不是“补一个 AI 模块”，而是“重建一套统一承接层”。

## 3. 当前业务全貌判断

### 3.1 第一阶段必须承接的四条业务主线

第一阶段全部必须跑通：

1. 使用帮助
   - 指标口径解释
   - 系统路径导航
   - 广告规则说明
   - 常见技术问题帮助

2. 需求沟通
   - 媒体回传接入
   - 事件映射
   - 埋点 / 归因 / 配置类需求
   - 协作信息补齐与结构化收口

3. 问题排查
   - 激活 / 付费 / 回传 / 归因 / BI 不一致等异常
   - 证据链收集
   - 初步结论与下一步建议

4. 广告联调
   - 联调准备项检查
   - 账号 / 应用 / 包体 / 设备 / 白名单 / 环境确认
   - 执行状态与日志回看
   - 联调结果报告

### 3.2 本系统承接的不是“所有广告业务”

第一阶段边界明确如下：

做：

1. 统一入口。
2. 四条业务流任务化。
3. 结果结构化沉淀。
4. 现有能力封装接入位。
5. 前后端可联调的正式工作台。

暂不做成第一阶段主目标：

1. 创意生产主系统。
2. 自动投放策略主系统。
3. ROI/LTV 预测主系统。
4. 全量媒体规则监控中心。
5. 大而全后台替换工程。
6. 知识库平台重做。
7. 评测平台重做。

这些能力可以保留接口位或专项页入口，但不作为第一阶段验收主目标。

## 4. 目标用户与价值

### 4.1 第一阶段主用户

第一阶段不是单一角色产品，而是围绕同一协作链路的多角色工作台：

1. 广告技术支持
2. 投放 / 运营同学
3. 数据分析 / 排查同学
4. 联调协作角色

### 4.2 给用户带来的直接变化

1. 不再先找系统，再找入口，再找人。
2. 模糊问题可以先被接住，再被系统追问收口。
3. 需求、排查、联调结果不再只散落在聊天和人工流程里。
4. 同一会话里能持续推进，而不是每个环节重开一次。

---

## 5. 新系统的产品形态

### 5.1 前台形态

前台主形态：

`单对话工作台 + 结构化结果区 + 历史任务区 + 系统管理页入口 + 少量专项页`

原因：

1. 用户问题天然先以自然语言进入。
2. 四条业务流在入口层不应该先分系统。
3. 结构化结果、历史任务、专项页都应该围绕同一会话上下文服务。
4. 管理能力属于同一套系统内页面跳转，不做第二套登录系统。

### 5.2 管理页面形态

管理页面不是第二套后台，而是同一登录态下的系统管理页。

第一阶段至少包含：

1. 提示词管理
2. 路由规则管理入口
3. MCP / skill / tool 绑定管理入口
4. 结果模板管理入口
5. 通知与开关配置入口

明确不自建：

1. 知识库管理后台
2. 评测平台管理后台

### 5.3 后台形态

后台不是单一 chatbot，而是任务运行时：

1. Conversation
2. Message
3. RoutingDecision
4. Task
5. TaskContext
6. WorkflowResult

前台只有一个“小乔”，后台允许根据任务类型调用不同能力与流程。

### 5.4 能力封装位

本系统必须为下列能力保留正式封装位：

1. 数仓查询能力
2. BI / 指标读取能力
3. 日志与链路排查能力
4. 归因 / 回传 / 媒体配置能力
5. 自动化联调能力
6. 知识与规则引用能力
7. 通知与协同能力

这里的知识能力是外部知识库接入后的引用能力，不代表小乔自建知识库后台。

实现形式可以是：

1. MCP
2. skill
3. server
4. 自研工具服务

本规划文档不先绑定某一种实现，但明确要求系统能力必须可被统一调用。

---

### 5.5 当前阶段的数据接入策略

当前阶段已经确认：

1. 智投预发布数据库不是当前本地开发可直接连通的数据源。
2. 小乔第一阶段不能以数据库打通作为页面、功能和联调的前置条件。
3. 当前最优解是先把前端、后端、任务结果和管理页都压到正式 mock contract 上。

这里的 mock 必须满足三条要求：

1. mock 对象要对应真实业务对象，而不是随意拼装卡片数据。
2. mock 结构要对应后续真实接口结构，而不是一次性静态页面数据。
3. mock 内容要能支撑四条业务流和管理页的正式验收，而不是只够演示。

因此第一阶段的交付策略是：

`先以本地仓库后端 + 正式 mock contract 建立稳定交付链，再在后续迭代中切换测试 / 生产数据源。`

## 6. 第一阶段交付标准

### 6.1 必须成立的产品标准

1. 四条业务流全部可进入、可追问、可沉淀、可回看。
2. 结果区展示的是正式结果结构，而不是示意卡片。
3. 历史任务区能够承接真实任务，而不是静态 mock 列表。
4. 前端接本地仓库内真实后端骨架，不再只靠纯静态页面。
5. 后端 contract 必须为后续接测试 / 生产数据预留升级位。
6. 管理页面至少提供提示词管理能力，并与主工作台共用同一登录系统。
7. 当前联调以正式 mock contract 完成，不把本地直连预发库作为验收前提。

### 6.2 不允许的交付方式

1. 只做演示稿。
2. 只做浅层页面 without 真实对象。
3. 只做概念化需求 without 可实现字段包。
4. 只做接口名 without 明确结果结构。

## 7. 当前系统架构方向

### 7.1 当前最稳的实现方向

1. 前端继续采用 React + Vite。
2. 企业系统基础组件默认使用 Ant Design。
3. 会话式交互组件默认使用 Ant Design X。
4. 后端继续以当前本地小乔 runtime 为第一承接层。

### 7.2 当前最关键的产品化缺口

1. 四条业务流的结果结构仍需进一步业务化。
2. 需求与功能的拆解还需要更贴近真实对象与依赖。
3. 前端当前还是骨架页面，尚未体现正式运行时联动。
4. 后端目前已有统一骨架，但各业务流 payload 仍偏轻。

---

## 8. 用户需求与产品需求回查

### 8.1 用户需求层

原始用户要求：

1. 逐步完成正式交付物链路。
2. 不做原型汇报思路，直接走交付。
3. 需求拆到功能层，功能拆到接口层。
4. 评审要更真实，贴近业务和开发现状。
5. 重新吸收历史文档和行业背景，让产出更落地。
6. 不要遗漏管理页面，尤其是提示词管理。

### 8.2 当前产品规划是否解决

- 统一产品方向：`solved`
- 明确重新起系统而非旧版本续写：`solved`
- 明确四条业务流都是第一阶段必须项：`solved`
- 明确依托现行系统能力封装而非空造：`solved`
- 明确知识库、评测只接入不自建：`solved`
- 管理页面与提示词管理纳入正式范围：`solved`
- 管理页与主工作台同一登录系统：`solved`
- 当前阶段以正式 mock contract 承接真实业务对象：`solved`
- 对真实字段包、真实能力映射的深拆：`partial`

原因：

产品规划层已经把方向压实，但真正闭环还依赖需求文档和功能文档继续下钻。

## 补充：会话支撑能力域

为避免小乔退化成“只有对话壳的企业聊天页”，第一阶段正式补充一条公共能力域：

`会话支撑能力域`

这条能力域不独立于主工作台之外，而是服务全部业务流与管理页承接，至少覆盖：

1. 会话上下文维护
2. 任务上下文维护
3. 缺失字段追问与补录
4. 文件上传
5. 图片上传
6. 粘贴与拖拽上传
7. 附件预览、删除、重试
8. 附件解析与摘要
9. 图片识别与截图理解
10. 对话与附件绑定
11. 结果区证据回显
12. 上下文继承、切换、重置

对应专项文档：

- `docs/小乔智投-会话支撑能力设计-2026-05-09.md`

这条补充的产品要求意味着：

1. 前端不能只做消息与结果展示。
2. 后端 contract 不能只保留 message 文本。
3. 需求、功能、设计文档都要把会话支撑能力作为正式承接对象。
