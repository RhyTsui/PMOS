---

## 0. 规划定位

小乔智投不是旧广告后台的换皮页，也不是纯聊天问答产品。

它是一个面向广告支持、需求沟通、问题排查、广告联调的统一工作台系统：

- 以前台单会话工作台为统一入口
- 以后台任务运行时为骨架
- 以结构化结果沉淀为交付对象
- 以企业级管理页承接 prompt、路由、能力绑定与运行控制

---

## 1. 当前要解决的问题

当前广告支持场景的主要问题不是“少一个页面”，而是：

1. 用户入口分散
2. 问题进入后没有稳定任务化
3. 结果没有结构化沉淀
4. 数据、日志、知识、联调能力彼此孤立
5. 协作结果散落在聊天、口头、临时文档中

小乔智投当前阶段要做的是重建承接层，而不是重做所有底层平台。

---

## 2. 当前版本范围

第一阶段必须跑通四类业务流：

1. 使用帮助
2. 需求沟通
3. 问题排查
4. 广告联调

第一阶段明确不作为主目标的内容：

1. 创意生产主系统
2. 自动投放策略主系统
3. ROI/LTV 预测主系统
4. 知识库平台重做
5. 评测平台重做

---

## 3. 当前产品形态

### 3.1 前台

前台默认形态：

`单会话工作台 + 结构化结果区 + 历史任务区 + 管理页入口 + 少量专项页`

补充治理约束：

1. `统一入口` 不等于把会话、任务、结果、运行验证、治理信息平铺在同一首页
2. 默认首屏只服务“发起问题并推进当前任务”这一主操作
3. 历史任务、持续执行验证、运行证据、治理控制默认作为二级视图、抽屉或管理页承接，不与首屏主操作抢层级
4. 专项页可以存在，但不能反过来要求主工作台首屏同时解释所有专项能力

### 3.2 管理页

管理页与主工作台同属一套系统，不做第二套登录。

当前至少包括：

1. Prompt 管理
2. 路由规则管理
3. MCP / skill / tool 绑定入口
4. 结果模板管理
5. 通知与开关配置

### 3.3 后台

后台不是单一 chatbot，而是任务运行时：

1. Conversation
2. Message
3. RoutingDecision
4. Task
5. TaskContext
6. WorkflowResult

## 4. 当前交付链要求

当前唯一有效交付链：

`调研文档 -> 规划文档 -> 需求文档 -> 功能文档 -> 设计文档 -> 前端页面 -> 数据表 -> 后端接口 -> 联调与验收`

补充硬约束：

1. `功能文档` 必须先定义核心对象模型
2. `设计文档` 必须暴露关键数据状态和交互状态
3. `数据表` 是正式定表，不是第一次想数据
4. `后端接口` 是正式定契约，不是第一次想交互响应
5. `前端页面` 必须是交付级、动态的、面向用户的真实页面

---

## 5. 前端交付方向

小乔智投前端不能继续落回“AI 默认工作台壳”。

必须满足：

1. 面向真实用户任务
2. 结构化结果与证据并存
3. 支持真实状态切换
4. 不接受说明文档式页面
5. 企业系统基础组件优先使用 `Ant Design`
6. 会话组件优先使用 `Ant Design X`
7. 前端开源基座组合与分层以 `docs/小乔智投-前端分层与开源基座方案-2026-05-09.md` 为准：`Ant Design + Ant Design X` 负责工作台与会话表现层，`CopilotKit` 负责 agent 驱动交互编排，`Zustand` 负责本地 UI 状态，`React Query` 负责服务端数据同步
8. Ant Design X 官方体系（RICH / Components / X SDK / X Markdown / Ultramodern / X Skill）作为默认主设计规范与主开发基线，见 `docs/小乔智投-Ant-Design-X默认规范-2026-05-09.md`

额外强制约束：

1. 主工作台首页只能有一个主操作焦点，不能同时把“会话区、任务区、结果区、运行验证区、治理说明区”平级摊开
2. `结果区` 是当前任务的检查器，不是说明文案容器
3. `持续执行验证` 属于运行证据视图，不属于默认业务首屏
4. `管理页入口` 只保留进入能力，不在业务首页展开管理细节

禁用模式：

1. hero 首屏
2. summary-card 工作台
3. explanation-first
4. 假玻璃感、运营卡片壳

## 6. 当前数据与接入策略

当前阶段明确采用：

`本地仓内后端 + 正式 mock contract`

原因：

1. 当前预发布数据库不作为本地交付前提
2. 页面、接口、联调不能被真实数据源阻塞
3. mock 必须对齐真实业务对象和未来接口结构
