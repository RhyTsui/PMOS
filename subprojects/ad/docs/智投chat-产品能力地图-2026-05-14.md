# 智投chat 产品能力地图

更新时间：2026-05-14

## 1. 产品定位

智投chat 是面向游戏发行团队的广告业务 AI 服务入口。

它不是一个普通问答机器人，也不是后台菜单搜索框，而是在广告 BI、监测管理、媒体配置、报表系统、内部知识库、MCP 工具和 Skill 工作流之上，提供“对话发起、系统判断、工具执行、组件交付、结果沉淀”的端到端服务。

产品目标：

- 让发行、投放、数据、运营同学用自然语言发起广告业务服务。
- 系统自动识别意图，判断是解答、排查、对接、联调、报表、分析、监控还是预测。
- 能自动获取的条件不再反复追问用户。
- 需要用户确认的条件用结构化组件一次性确认。
- 最终交付可查看、可追踪、可复用、可评测。

核心链路：

```text
用户输入
→ 意图识别
→ 上下文准备
→ 能力路由
→ 条件补齐
→ MCP / Skill / 知识库调用
→ 动态组件交付
→ 结果沉淀
→ 后续追踪或自动任务
```

## 2. 服务能力总览

| 服务 | 用户目标 | 服务边界 | 首选交付形态 | 当前优先级 |
| --- | --- | --- | --- | --- |
| 解答 | 解释口径、规则、路径、概念 | 不做复杂排查，不直接创建任务 | 指标解释器、知识卡、来源 | P0 |
| 排查 | 定位数据不一致、异常、链路问题 | 有具体异常对象或系统巡检发现异常后进入 | 排查链路、证据、结论、修复动作 | P0 |
| 对接 | 把新媒体、新事件、新渠道需求收成结构化需求 | 不直接替代开发排期和审批 | 需求表单、依赖校验、需求池记录 | P1 |
| 联调 | 发起并观测媒体/事件/回传自动联调 | 不让用户填写后台已有配置 | 联调工作台、步骤、截图、结果报告 | P0 |
| 报表 | 查数、拼表、日报、定时发送 | 不把监控任务误判成报表，反之亦然 | 表格模板、数据预览、定时任务 | P0 |
| 分析 | 解释波动、拆解贡献、输出洞察 | 不承诺开放式经营结论完全自动化 | 分析卡、图表、结论、下钻建议 | P1 |
| 监控 | 创建巡检、阈值告警、定时检查 | 监控发现异常后再进入排查 | 监控任务、规则、告警、巡检结果 | P0 |
| 预测 | ROI、消耗、回收、趋势预估 | 依赖模型和历史数据成熟度，当前规划中 | 预测卡、置信区间、风险说明 | P2 |

## 3. 通用用户旅程

### 3.1 新会话发起

用户可能从三类入口进入：

- 直接输入问题。
- 点击首页核心功能。
- 从资产、报表、联调记录、排查记录等历史对象继续。

系统动作：

1. 读取当前用户、项目、历史记忆、会话上下文。
2. 识别服务意图。
3. 判断是否已有足够条件。
4. 如果足够，直接进入服务流程。
5. 如果不足，生成结构化追问组件。

验收标准：

- 新会话不会无意义重复创建。
- 多轮对话能继承上文。
- 跨会话能命中用户级历史记忆。
- 用户不需要重复输入系统已经知道的信息。

### 3.2 服务执行

系统不应直接输出大段建议文本，而应按服务步骤流式推进：

1. 思维链展示服务过程。
2. 调用来源展示 MCP、Skill、知识库、网页或内部系统。
3. 会话正文展示用户需要看的结论和组件。
4. 右侧展开区展示当前选中的来源、工具详情、执行步骤或记录详情。

验收标准：

- 每一步有序号、状态、耗时。
- 子步骤可展开查看请求和返回。
- 来源和调用能力可追溯。
- 页面刷新后过程仍可查。

### 3.3 结果沉淀

每次服务结束后沉淀为以下对象之一：

- 解答：知识引用、指标解释、历史问答。
- 排查：排查记录、证据链、结论、修复动作。
- 对接：需求单、依赖、验收标准。
- 联调：联调任务、步骤截图、结果报告。
- 报表：报表模板、数据结果、定时任务。
- 分析：分析报告、图表、下钻路径。
- 监控：监控规则、告警记录、触发后的排查任务。

## 4. 解答服务

### 4.1 服务边界

适合：

- 指标定义。
- 统计口径。
- 媒体规则。
- 系统入口。
- 操作路径。
- 常见问题。

不适合：

- 已有具体异常，需要进入排查。
- 需要拉真实数据，需要进入报表或分析。
- 需要创建任务，需要进入报表、监控、对接或联调。

### 4.2 用户旅程

```text
用户提问
→ 判断为解答
→ 检索知识库和指标库
→ 判断相关性
→ 低相关内容丢弃
→ 生成指标解释器或知识卡
→ 展示来源
→ 支持继续追问
```

### 4.3 输入与输出

输入：

- 问题文本。
- 当前项目上下文。
- 用户历史偏好。
- 可选：指标名、系统名、媒体名。

输出：

- 简明结论。
- 口径说明。
- 计算过程。
- 易混淆点。
- 适用系统。
- 来源。

### 4.4 组件

- 指标解释器。
- 知识卡。
- 来源列表。
- Markdown 结构化正文。

### 4.5 验收问题

```text
帮我解释一下激活数、注册数和付费数的统计口径有什么区别
```

预期：

- 意图：解答。
- 不进入排查。
- 展示指标解释组件。
- 展示埋点位置、上报事件、字段、上报方、计算过程、歧义说明。
- 如果知识库缺少内容，要明确缺少，不用 mock 填充。

## 5. 排查服务

### 5.1 服务边界

适合：

- 数据不一致。
- 指标异常。
- 报表缺数。
- 激活、注册、付费、回传差异。
- 消耗采集异常。

不适合：

- 宽泛“昨天投放有没有异常”，应先进入监控巡检。
- 只是问口径，应进入解答。
- 要创建长期规则，应进入监控。

### 5.2 排查分类

| 类型 | 首要检查 | 不应误查 |
| --- | --- | --- |
| 消耗异常 | 媒体采集、报表调度、入库、权限 | 不默认查回传、包名、渠道号 |
| 激活/注册/付费不一致 | 媒体侧、BI侧、归因、上报、回传 | 不把投放计划状态当首要条件 |
| 报表无数据 | 调度、入库、权限、筛选条件 | 不先要求用户提供实际值 |
| 回传失败 | 事件、归因、回传配置、媒体返回 | 不混入消耗采集问题 |

### 5.3 用户旅程

```text
用户描述异常
→ 判断是否是具体排查
→ 提取项目、媒体、时间、指标
→ 查询已有上下文和报表 MCP
→ 判断缺失条件
→ 缺失时一次性表单追问
→ 条件齐备后调用排查 Skill
→ 输出证据链和结论
→ 必要时创建排查记录
```

### 5.4 追问规则

必须追问：

- 对比数据源不明确。
- 项目或媒体无法从上下文确定。
- 时间范围不明确。
- 指标无法识别。

不应追问：

- 系统可通过 MCP 查询的实际值。
- 后台配置里已有的账户、包名、设备、查看位置。
- 和当前指标无关的信息。

### 5.5 组件

- 结构化补充条件表单。
- 排查路径。
- 证据链。
- 差异比对表。
- 指标解释器。
- 修复动作卡。

### 5.6 验收问题

```text
APPID: 10100011
媒体ID：10007
账户ID：4175800
日期：2026-05-14
指标：消耗
预期值：1,002.23
排查一下
```

预期：

- 意图：数据排查。
- 识别为消耗排查。
- 调报表 MCP 获取系统实际值。
- 只追问对比数据源。
- 不追问包名、渠道号、投放状态。
- 不误判 2026-05-14 为未来日期。

## 6. 对接服务

### 6.1 服务边界

适合：

- 新媒体接入。
- 新渠道接入。
- 新事件回传。
- 新报表字段。
- 媒体 API 数据接入。

不适合：

- 已经完成接入后要验证，应进入联调。
- 单纯解释媒体规则，应进入解答。

### 6.2 用户旅程

```text
用户提出对接需求
→ 判断是否新增媒体或已有媒体
→ 检查是否已有接入能力
→ 生成结构化对接表单
→ 校验依赖信息
→ 通过则提示可去后台创建监测链接
→ 特殊处理则生成需求并记录需求池
```

### 6.3 必要信息

- 媒体名称。
- 接入类型。
- 对接文档。
- 数据字段。
- 账号权限。
- 验收标准。
- 期望时间。

### 6.4 组件

- 对接需求表单。
- 依赖校验列表。
- 需求池记录。
- 后续进度卡。

### 6.5 验收问题

```text
AppLovin 要接 API 花费
```

预期：

- 意图：对接。
- 先判断是否已支持。
- 缺文档时生成结构化表单，不只给建议。
- 可记录到需求池。

## 7. 联调服务

### 7.1 服务边界

适合：

- 媒体回传联调。
- 事件联调。
- 扫码联调。
- 自动化执行和观测。

不适合：

- 新媒体尚未完成接入，应进入对接。
- 单纯查历史联调结果，应进入联调记录。

### 7.2 用户旅程

```text
用户提出联调
→ 识别媒体和项目
→ 查询后台联调配置
→ 系统校验应用共享、验收状态、上报记录、默认事件
→ 满足条件则展示自动联调组件
→ 用户点击发起联调
→ 右侧展示联调步骤和截图
→ 会话区输出最终结论
```

### 7.3 用户侧输入

用户侧只需要提供：

- 项目。
- 媒体。
- 应用。
- 联调目标。

系统侧自动读取：

- 媒体账号。
- 密码。
- 事件资产地址。
- 回传查看位置。
- 设备。
- 包名。
- deeplink。
- 关键字。
- 游戏登录账号。

### 7.4 默认联调目标

- 激活。
- 注册。
- 付费。
- 关键行为。

如果游戏没有真实付费，也按模拟付费流程联调。

### 7.5 组件

- 自动联调启动卡。
- 联调百叶窗步骤。
- 截图观测区。
- 错误日志。
- 结果报告。

### 7.6 验收问题

```text
巨量这边要立即联调，应用已经共享到 wuyanlan@dobest.com
```

预期：

- 意图：联调。
- 不调用无关知识库。
- 不展示新媒体对接提示。
- 系统自动校验巨量应用共享和数据上报。
- 会话区展示联调功能组件。
- 右侧只在发起后展示执行步骤。

## 8. 报表服务

### 8.1 服务边界

适合：

- 查数。
- 拼表。
- 上传 Excel 模板。
- 截图提取模板。
- 定时日报。
- 生成报表数据。

不适合：

- 阈值告警，应进入监控。
- 异常原因定位，应进入排查或分析。

### 8.2 用户旅程

```text
用户提出报表需求
→ 意图识别为报表
→ 提取指标、维度、时间、频率
→ 生成报表模板
→ 校验指标
→ 会话区表格确认模板
→ 用户确认
→ 调用 MCP 查询数据
→ 展示数据预览
→ 用户确认是否定时发送
→ 创建报表任务
```

### 8.3 四种模板来源

- 截图提取报表模板。
- 上传 Excel 模板。
- 手动输入模板。
- 指定系统页面并描述需求。

### 8.4 组件

- 报表模板表格。
- 指标校验结果。
- 数据预览表。
- 定时任务确认卡。

### 8.5 验收问题

```text
帮我创建一个每天上午 10 点发送的广告消耗日报
```

预期：

- 意图：报表，不是监控。
- 模板确认前有思维链。
- 只展示有问题的指标。
- 模板确认后流式查询数据。
- 数据确认后再问是否定时发送。

## 9. 分析服务

### 9.1 服务边界

适合：

- ROI 波动分析。
- Campaign 贡献拆解。
- 渠道效果分析。
- 素材表现洞察。
- 国家、账户、计划等维度下钻。

不适合：

- 只查一个数，应进入报表。
- 已明确异常定位，应进入排查。
- 长期规则，应进入监控。

### 9.2 用户旅程

```text
用户提出分析目标
→ 判断分析框架
→ 拉取多源数据
→ 拆解指标贡献
→ 识别主要变化因素
→ 生成图表和结论
→ 给出下钻建议
```

### 9.3 组件

- 趋势图。
- 贡献拆解图。
- 维度排行表。
- 结论卡。
- 下钻问题建议。

### 9.4 验收问题

```text
帮我分析昨天巨量 ROI 为什么下降
```

预期：

- 先查 ROI、消耗、回收、激活、付费。
- 自动拆解主要贡献维度。
- 如果缺项目上下文，优先使用顶部项目选择。

## 10. 监控服务

### 10.1 服务边界

适合：

- 创建长期巡检。
- 阈值告警。
- 每日异常检查。
- 回传延迟监控。
- 投放异常巡检。

不适合：

- 已有具体异常并要求定位，应进入排查。
- 定时生成固定表，应进入报表。

### 10.2 用户旅程

```text
用户提出监控需求
→ 判断监控对象
→ 提取项目、媒体、指标、阈值、频率
→ 使用内置规则补充默认基准
→ 创建监控任务
→ 监控触发后自动进入排查
```

### 10.3 投放异常巡检规则

巡检应先在思维链中执行：

- 检查消耗突增、突降、预算撞线、账户授权可见性。
- 检查激活、注册、付费、回传延迟、回传成功率。
- 检查昨日报表是否完成调度和入库。

发现异常：

- 携带项目、时间、指标、来源进入排查。

未发现异常：

- 只返回监控结果，不要求用户补充对比数据源。

### 10.4 组件

- 监控任务确认卡。
- 规则配置表单。
- 巡检结果卡。
- 告警记录。
- 异常转排查入口。

### 10.5 验收问题

```text
帮我监控巨量回传延迟，超过 30 分钟就告警
```

预期：

- 意图：监控，不是排查。
- 创建回传延迟监控任务。
- 告警触发后自动进入排查。

```text
我想看看昨天投放有什么异常
```

预期：

- 意图：监控巡检。
- 不要求填写对比数据源。
- 先执行全链路巡检。
- 发现异常后再进入排查。

## 11. 预测服务

### 11.1 服务边界

当前规划中，不作为 P0 主交付。

适合未来：

- ROI 预测。
- 消耗预测。
- 回收预测。
- 趋势预判。

当前策略：

- 首页入口可暂时隐藏或置灰。
- 不在用户流程中承诺可用。

## 12. Agent / Skill / MCP 分工

| 层级 | 责任 |
| --- | --- |
| Agent | 理解用户意图、处理多轮上下文、判断服务路由、决定是否追问 |
| Skill | 执行稳定、可复用、可评测的业务工作流 |
| MCP | 调用报表、配置、监测、归因、回传、任务、知识库等系统能力 |
| Prompt | 约束 Agent 的判断、追问、输出结构和失败策略 |
| 动态组件 | 承载表单、表格、图表、联调过程、排查证据和结果 |
| 记忆 | 保存用户偏好、最近上下文、历史处理经验 |
| 评测 | 验证意图、追问、工具调用、来源、结果和组件是否符合预期 |

## 13. 能力路由规则

同一个问题可能包含多个意图，路由优先级如下：

1. 如果用户要求创建长期规则、定时检查、告警：优先监控。
2. 如果用户要求生成、拼接、定时发送表：优先报表。
3. 如果用户要求立即验证媒体事件链路：优先联调。
4. 如果用户提出新接入、新媒体、新字段：优先对接。
5. 如果用户有具体异常对象：优先排查。
6. 如果用户只是问概念和口径：优先解答。
7. 如果用户要求波动原因但没有具体异常：优先分析。

冲突处理：

- 宽泛异常先监控，监控发现异常再排查。
- 报表任务不是监控任务。
- 消耗排查不默认进入回传链路。
- 联调需求不默认检索知识库。

## 14. 当前缺口

P0 缺口：

- 部分路由仍在前端关键字判断，需要迁移到后端 Agent runtime。
- MCP 调用还未全部变成真实调用。
- 指标解释器仍有 mock 兜底，需要接真实指标知识结构。
- 联调服务还需要把后台配置、用户输入、执行观测完整串成端到端流程。
- 监控和排查之间的自动转接还不稳定。

P1 缺口：

- Skill 管理后台需要支持 contract、输入输出、绑定 MCP、评测用例。
- Prompt 管理需要按服务能力绑定，不是只存文本。
- 评测集需要覆盖每类服务的典型问题和预期组件。

## 15. 下一步落地顺序

1. 固化 P0 服务路由：解答、排查、联调、报表、监控。

3. 把前端关键字拦截迁移到后端 Agent runtime。
4. 建立 Skill contract 管理。
5. 接真实 MCP，并删除会冒充真实能力的 mock。
6. 把思维链、来源、右侧展开区统一到 `AgentProcessEvent`。
7. 用连弩评测持续回归。
