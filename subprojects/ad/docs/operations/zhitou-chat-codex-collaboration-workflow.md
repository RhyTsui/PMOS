# 智投chat 与 Codex 协作工作流

更新时间：2026-05-14

## 目标

把“连续对话式改产品”升级为“可验收的产品交付协作”。

这套工作流解决四类问题：

- 用户需求、产品需求、技术实现混在同一轮对话里，容易浅层实现。
- UI 调整、Agent 设计、Prompt、后端接口、测试验收同时发生，优先级失焦。
- Codex 容易按局部关键词执行，缺少阶段边界和验收口径。
- 每天大量修改没有稳定沉淀，经验没有转成 guardrail、文档、测试集和组件契约。

## 基本原则

1. 每轮只确认一个主目标
   - 可以记录旁支问题，但不能让旁支抢走主线。
   - 如果用户连续补充多个问题，Codex 必须先归类，再按优先级处理。

2. 需求必须拆成四层
   - 用户场景：用户为什么需要这个能力。
   - 产品机制：系统应该用什么交互、流程、规则解决。
   - 技术改动：前端、后端、Prompt、MCP、Skill、数据、配置分别改什么。
   - 验收口径：用户如何判断完成。

3. 不把“做了页面”当成“解决需求”
   - 每项改动必须回到原始用户问题，标记 `solved`、`partial`、`unsolved`。
   - 产品机制存在但用户场景未闭环，只能算 `partial`。

4. 先设计工作流，再改 UI
   - UI 只是服务流程的载体。
   - 对 Agent、Skill、MCP、表单、动态组件、右侧展开区，必须先明确流程位置。

5. 每次实现后必须验证
   - 前端：类型检查、构建、8002 可访问。
   - 后端接口：至少验证写入、读取、异常状态。
   - Agent/Prompt：给出测试问题、预期意图、预期组件、预期结果。

## 需求入口格式

以后建议用户按这个格式提复杂需求：

```text
目标：
我希望用户能完成什么。

场景：
用户在什么情况下会用。

当前问题：
现在哪里不对，最好附上问题输入、实际输出、截图或接口返回。

期望结果：
用户看到什么、能点击什么、系统自动做什么。

验收方式：
我会用哪几个问题或操作来验收。
```

如果用户没有按格式提供，Codex 要自动补齐为：

```text
我理解的目标：
影响范围：
缺失信息：
建议优先级：
本轮先处理：
暂存问题：
```

## 任务分层

每批需求进入处理前，Codex 先归入下面类别。

| 类型 | 处理方式 | 验收物 |
| --- | --- | --- |
| UI 细节 | 直接改，保持设计规范 | 页面效果、构建通过 |
| 交互流程 | 先写状态流和组件契约，再实现 | 可操作闭环 |
| Agent 路由 | 先写意图、条件、追问策略 | 测试问题与预期路由 |
| Prompt | 先写输入输出契约，再调提示词 | 返回结构和失败策略 |
| MCP/Tool | 先确认真实接口、参数、权限 | 请求、返回、来源展示 |
| Skill | 只做高频稳定工作流 | skill schema、步骤、结果 |
| 后台配置 | 先定义配置项归属和默认值 | 保存后刷新不丢 |
| 数据记忆 | 先定义用户级、会话级、任务级边界 | 跨会话命中验证 |
| 测试评测 | 先写测试集和预期结果 | 用例清单、评测接口 |

## 阶段拆解

### 阶段 0：需求收敛

输出：

- 本轮主目标
- 非目标
- 依赖信息
- 风险点
- 验收问题

完成标准：

- 用户和 Codex 对“先做什么、不做什么”一致。

### 阶段 1：产品工作流设计

输出：

- 用户路径
- Agent/Skill/MCP 运行路径
- 需要追问的字段
- 哪些字段系统自动获取
- 哪些内容进入思维链
- 哪些内容进入会话正文
- 哪些内容进入右侧展开区

完成标准：

- 不再用大段文本建议替代组件化交付。

### 阶段 2：组件与接口契约

输出：

- 前端组件结构
- 后端接口形态
- 数据字段
- 状态枚举
- 错误状态
- 来源与 trace 展示规则

完成标准：

- 前端、后端、Prompt 之间不会互相猜字段。

### 阶段 3：实现

执行顺序：

1. 后端数据和接口
2. 前端组件和状态
3. Agent/Prompt/Skill 接入
4. 来源、思维链、右侧展开区
5. 样式规范和响应式检查

完成标准：

- 不用 mock 替代真实能力，除非明确标注。
- 用户能按真实路径完成操作。

### 阶段 4：验收

每项需求至少给出：

- 验收问题或操作
- 预期路由
- 预期组件
- 预期数据来源
- 预期结果
- 未完成项

状态标记：

- `solved`：用户场景已闭环。
- `partial`：机制已有，但缺数据、接口、真实工具或稳定性。
- `unsolved`：只做了计划或 UI 片段。

### 阶段 5：沉淀

每次重要迭代结束后，必须沉淀到至少一个位置：

- `AGENTS.md`：长期执行规则。
- `docs/operations/*`：协作流程、开发流程、验收流程。
- `docs/review/*`：复盘、问题清单、测试结论。
- `docs/memory/*`：项目背景、产品判断、长期记忆。
- 后台 Prompt/Skill 配置：可运行的 Agent 能力。
- 评测用例：可重复验证的问题与预期结果。

## 智投chat 专用协作节奏

### UI 调整类

适合直接对话快速改：

- 图标替换
- 圆角、间距、颜色
- 宽度、居中、滚动条
- 文案微调
- 移入、点击、展开动画

要求：

- 每轮最多一组相关 UI 问题。
- 改完必须构建并保持 8002 可访问。

### Agent 产品能力类

不能直接边说边改，必须先收敛：

- 意图路由
- 排查流程
- 自动联调
- 报表拼表
- 监控任务
- 历史记忆
- Skill 管理
- MCP 调用

要求：

- 先给工作流图。
- 再给输入输出契约。
- 再给测试集。
- 最后实现。

### Prompt 调优类

不能只改一句提示词。

必须同时定义：

- 适用意图
- 必填字段
- 可自动获取字段
- 追问策略
- 禁止输出
- 输出组件
- 失败策略
- 来源披露

## 当前项目建议优先级

P0：

- 多轮上下文与用户记忆接后端用户表。
- 思维链步骤、子步骤、调用来源稳定展示。
- 自动联调从文本建议改为端到端组件闭环。
- 监控、排查、报表三类意图的路由和追问策略分清。

P1：

- Skill 管理后台与 Agent 调用 Skill 的 Prompt 契约。
- 指标解释器接真实知识结构，禁止 mock 平铺。
- 连弩评测接口和测试集。

P2：

- UI 细节持续打磨。
- Figma 同步。
- 日报蒸馏自动化。

## 关键约束

- 复杂产品能力不直接进入实现，先有工作流。
- UI 调整可以快，但不能破坏主工作流。
- 每个 Agent 能力必须有测试问题和预期结果。
- 每个接口能力必须能证明真实调用，不得用 mock 假装完成。
- 每天必须把新增经验沉淀为规则、文档、Prompt、Skill 或测试集。
