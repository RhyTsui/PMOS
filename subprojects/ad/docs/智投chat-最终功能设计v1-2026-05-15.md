## 0. 已确认设计决策

1. v1 只聚焦 P0 + P1，不提前做洞察、预测、决策主流程。
2. `AgentProcessEvent` 作为前端、后端、后台、评测的唯一过程真源。
3. P0 先打透监控巡检、指标差异排查、自动联调、投放前质量保障。
4. P1 先打透疑问解答、指标解释器、监测链接交付、自动报表。
5. 对接作为投放前质量保障和自动联调前置，不单独做成大模块。
6. 后台 v1 只做配置和追踪；评测由连弩负责，不先做复杂审批流。
7. 连弩评测作为 Skill 上线准入。
8. 监控巡检 7 层对象需要新增采集 MCP，覆盖媒体报表、点击日志、自归因、调度、上报报告、数据质量等链路。
9. 自动联调 v1 先打透巨量。
10. 监测链接交付 v1 支持查询已有链接，也支持创建新链接。
11. 指标解释资料以后台结构化录入为主，知识库作为补充来源。
12. 自动报表 v1 先支持文本模板和 Excel 模板，截图识别后置。
13. 采集 MCP 采用统一采集网关 + 内部分 adapter，避免 Agent 和前端直接面对多个零散采集工具。
14. 巨量自动联调 v1 按最小后台配置打透，用户侧只指定项目、媒体和联调目标。
15. 监测链接创建 v1 采用有权限用户直接创建，不先做审批流。
16. 指标解释首批覆盖核心投放、转化、回传、BI 差异指标。
17. 自动报表 v1 支持文本模板和标准二维 Excel 模板，复杂 Excel 和截图识别后置。

## 1. 设计目标

智投chat 的功能设计目标不是继续补单点页面，而是建立一个可持续交付广告业务服务的 Agent 产品系统。

当前 v1 的目标：

1. 给用户提供没有问题的包、没有问题的归因回推、没有问题的报表数据，并持续提供监控保障。
2. 直接给到用户监测链接、疑问解答、数据排查和自动报表。
3. 在前两层稳定后，再寻求洞察、预测和决策能力。

对应优先级：

- P0：业务质量保障。
- P1：高频服务直接交付。
- P2：洞察、预测、决策。
- P3：自动投放。

当前实现聚焦 P0 + P1。

## 2. 总体设计原则

### 2.1 端到端直接交付

系统响应用户需求时，不以多轮聊天和长文本解释为目标，而以完成用户业务结果为目标。

默认流程：

用户输入  
→ 意图推理  
→ 上下文准备  
→ 能力可用性检查  
→ 必要条件确认  
→ Skill / MCP / 知识库调用  
→ 动态组件交付  
→ 来源与证据披露  
→ 结果沉淀和可评测事件

### 2.2 系统能获取的信息不追问

系统应自动获取：

- 用户身份与权限。
- 当前项目与历史项目偏好。
- appid / appname / 媒体 / 账户 / 包 / 分包。
- 报表实际值与更新时间。
- 媒体应用状态和过审状态。
- 监测链接、归因配置、回传状态。
- 后台配置的媒体账号、设备、回传查看位置。
- 历史会话和跨会话记忆。

只有系统无法获取且业务必须依赖时，才用结构化确认表单一次性收齐。

### 2.3 所有服务统一运行时协议

不再为单个需求临时增加字段和 UI。

前端、后端、后台和连弩评测统一消费 `AgentProcessEvent`：

- 思维链展示来自事件。
- Skill 执行过程来自事件。
- MCP 请求返回来自事件。
- 来源披露来自事件。
- 确认表单来自事件。
- 动态组件来自事件。
- 评测判断来自事件。

### 2.4 不用 mock 伪装真实能力

如果 MCP、Skill、知识库或后台配置不可用，必须明确说明缺口。

允许演示数据存在，但必须标记为演示，不得进入真实业务结论。

## 3. 用户与场景

### 3.1 用户

- 发行人员。
- 投放人员。
- 非投放管理职能人员。
- 内部产品、技术、数据支持人员。

### 3.2 核心场景

P0 场景：

- 投放前质量检查。
- 包可用性和监测链接交付。
- 自动联调。
- 投放异常监控。
- 指标差异排查。
- 系统健康巡检。

P1 场景：

- 疑问解答。
- 指标解释。
- 数据查询。
- 自动报表。
- 业务规则和归因逻辑解释。

P2/P3 场景暂不作为 v1 主交付：

- 洞察。
- 预测。
- 决策辅助。
- 自动投放。

## 5. 运行时协议设计

### 5.1 事件类型

v1 必须覆盖以下事件：

| 事件 | 用途 | 前端表现 | 评测用途 |
| --- | --- | --- | --- |
| `intent.detected` | 意图识别 | 思维链步骤 | 判断路由是否正确 |
| `context.prepared` | 准备项目、用户、时间、指标等上下文 | 思维链步骤 | 判断是否自动补齐上下文 |
| `capability.checked` | 检查 Skill/MCP/知识库是否可用 | 思维链步骤 | 判断是否误用 mock |
| `clarify.requested` | 缺条件时生成确认表单 | 表单组件 | 判断追问是否结构化 |
| `clarify.submitted` | 用户提交确认信息 | 思维链步骤 | 判断是否自动继续 |
| `skill.selected` | 选择 Skill | 调用与来源区 | 判断 Skill 是否正确 |
| `skill.started` | Skill 开始执行 | 思维链步骤 | 判断流程是否开始 |
| `skill.step` | Skill 子步骤 | 子步骤折叠 | 判断过程是否完整 |
| `skill.finished` | Skill 完成 | 思维链步骤 | 判断闭环是否完成 |
| `skill.failed` | Skill 失败 | 错误说明 | 判断失败是否可解释 |
| `mcp.tool_call` | MCP 请求 | 调用详情 | 判断参数是否正确 |
| `mcp.tool_result` | MCP 返回 | 调用详情/来源 | 判断结果是否真实 |
| `mcp.tool_error` | MCP 失败 | 错误说明 | 判断缺口是否披露 |
| `knowledge.search` | 知识库检索 | 调用详情 | 判断是否检索 |
| `knowledge.result` | 知识库结果 | 来源 | 判断来源质量 |
| `knowledge.rejected` | 低相关知识丢弃 | 思维链步骤 | 判断是否误采纳 |
| `source.attached` | 来源挂载 | 来源列表 | 判断来源披露 |
| `ui.component_rendered` | 动态组件渲染 | 表单/表格/联调/解释器 | 判断是否组件交付 |
| `answer.delta` | 流式输出 | 打字机效果 | 判断是否流式 |
| `answer.final` | 最终结论 | 结果消息 | 判断结果是否完整 |

### 5.2 事件持久化

每条助手消息必须保存完整 `process_events`。

刷新页面后：

- 思维链不能消失。
- 来源不能消失。
- 调用详情不能消失。
- 已提交确认表单不能回退为未提交。
- 联调步骤和报表任务状态可恢复。

### 5.3 用户可见思维链边界

展示的是“服务过程”和“证据链”，不是模型隐私推理。

可展示：

- 已识别意图。
- 已准备上下文。
- 已检查能力。
- 已调用的数据源。
- 已执行的检查项。
- 工具调用状态。
- 用时。
- 失败或跳过原因。
- 来源。

不展示：

- 模型内部 token 级推理。
- 未脱敏的敏感参数。
- 密码、token、账号密钥。
- 无业务意义的内部代码名。

## 6. Agent 路由设计

### 6.1 路由顺序

Agent 不只做关键词判断，而按以下顺序决策：

1. 识别用户目标。
2. 判断是否属于 P0 质量保障。
3. 判断是否属于 P1 高频直接交付。
4. 准备上下文。
5. 检查可用 Skill/MCP。
6. 判断是否需要确认表单。
7. 选择 Skill。
8. 生成过程事件和交付组件。

### 6.2 路由规则

| 用户表达 | 路由 | 注意 |
| --- | --- | --- |
| 能不能投、投放前、包、过审、应用共享 | 投放前质量保障 | 对接是联调前置，不单独做大模块 |
| 立即联调、联调记录、巨量联调 | 自动联调 | 用户不填设备和回传查看位置 |
| 昨天有什么异常、帮我监控、超过阈值告警 | 监控巡检 | 宽泛异常先监控，异常后转排查 |
| A 和 B 不一致、媒体数和智投数差很多 | 指标差异排查 | 先拿真实数据，再解释差异 |
| 为什么、区别、口径、什么意思 | 疑问解答 / 指标解释 | 先查知识库和指标资料 |
| 给我监测链接 | 监测链接交付 | 返回可用链接和校验结果 |
| 创建日报、周报、模板、拼表 | 自动报表 | 不误判为监控 |
| 洞察、预测、决策 | P2 记录 | 当前不承诺完整闭环 |

## 7. 确认表单设计

确认表单用于“系统无法自动获得且必须依赖”的信息。

规则：

- 多个缺失条件合并为一个表单。
- 表单提交后自动继续执行，不要求用户再点发送。
- 表单只展示输入项和必要 tips，不展示长篇说明。
- 可选项不得伪装成必填。
- 已经从上下文得到的信息不再次追问。

典型表单：

| 服务 | 表单字段 |
| --- | --- |
| 指标差异排查 | 对比数据源、时间范围、指标、媒体侧值或截图 |
| 自动联调 | 项目、媒体、联调目标 |
| 自动报表 | 报表模板、频率、发送时间 |
| 监测链接交付 | 项目、媒体、包或分包 |

## 8. P0 服务闭环

### 8.1 投放前质量保障

触发：

- 新包检测。
- 包更新。
- 媒体过审。
- 对接方式变化。
- 用户主动问“能不能投”。

流程：

1. 识别项目、媒体、包。
2. 检查包下载地址和版本。
3. 检查媒体应用状态。
4. 检查监测链接和归因配置。
5. 检查报表数据链路。
6. 检查是否需要自动联调。
7. 输出是否可以投放。

交付：

- 可以投放。
- 不可以投放。
- 阻塞项。
- 责任侧。
- 证据来源。
- 可复测入口。

确认问题：

- 是否所有媒体都先统一走这套质量保障流程，媒体差异只体现在 Skill 配置里？

### 8.2 自动联调

用户侧只需指定：

- 项目。
- 媒体。
- 联调目标。

后台配置：

- 媒体账号。
- 回传查看位置。
- 测试设备。
- 渠道配置。
- 游戏账号。
- 自动化运行环境。

默认联调目标：

- 激活。
- 注册。
- 付费。
- 关键行为。

如果游戏无付费，也模拟付费联调。

流程：

1. 确认项目和媒体。
2. 检查后台配置。
3. 检查媒体应用和事件资产。
4. 检查是否已有数据上报。
5. 发起自动联调。
6. 流式展示步骤和截图。
7. 轮询回传结果。
8. 输出通过/不通过原因。

确认问题：

- 巨量是否作为 v1 第一个打透媒体，其他媒体先只保留配置和流程框架？

已确认：

- 自动联调 v1 先打透巨量。
- 巨量自动联调 v1 后台最小配置包括媒体账号、事件资产地址、回传查看位置、应用共享默认账号、渠道包配置、抖音端配置、自动化关键词、游戏包配置、游戏登录配置、设备配置。
- 用户侧只填写项目、媒体、联调目标。

巨量最小后台配置字段：

| 配置组 | 字段 |
| --- | --- |
| 媒体账号 | `username`、`password` |
| 事件资产 | `event_asset_url`、回传查看位置 |
| 应用共享 | 默认账号 `wuyanlan@dobest.com` |
| 渠道包 | `aadvid`、`target_channel` |
| 抖音端 | `app_package`、`app_activity`、`deeplink` |
| 自动化关键词 | `auth_keyword`、`feed_keyword`、`action_keyword` |
| 游戏包 | `game_package`、`package_name` |
| 游戏登录 | `login_type`、`account`、`password` |
| 设备 | `device_id` |

### 8.3 监控巡检

触发：

- 用户主动：“我想看看昨天投放有什么异常”。
- 用户创建监控任务。
- 系统定时巡检。

默认检查：

- 消耗突增、突降。
- 预算撞线。
- 账户授权可见性。
- 报表调度和入库。
- 激活、注册、付费、关键行为。
- 回传延迟和成功率。

系统监控不能只看最终报表指标，必须覆盖广告数据从外部采集到平台入库、归因、回传、质量校验的完整链路。

监控对象分为 7 层：

| 层级 | 监控对象 | 监控点 | 用途 |
| --- | --- | --- | --- |
| L1 媒体报表指标资产 | 媒体后台报表、指标资产、字段映射、口径上下文 | 指标是否可取、字段是否变更、媒体口径是否变更、账号权限是否可见 | 判断消耗、激活、注册、付费等媒体侧基础数据是否可信 |
| L2 媒体报表采集 | 媒体 API 拉取、采集任务、采集结果 | 采集是否成功、延迟、空数据、字段缺失、异常响应、授权失效 | 判断媒体报表数据是否正常进入智投 |
| L3 媒体点击日志采集 | 点击日志、落地页参数、监测链接参数 | 点击是否进入、参数是否完整、渠道包/分包是否匹配、点击量是否异常 | 判断归因前置链路是否正常 |
| L4 自归因结果采集 | 自归因结果、匹配结果、归因失败原因 | 是否归因成功、匹配率、失败类型、延迟、重复归因 | 判断激活/注册/付费差异是否来自归因链路 |
| L5 平台数据上报报告 | 客户端/服务端事件上报、上报报告 | 事件是否上报、字段是否完整、事件量是否异常、版本/包维度是否异常 | 判断数据源头是否正常 |
| L6 数据质量监控 | 数据质量规则、阈值、异常检测 | 是否为 0、突增突降、延迟、重复、缺维度、口径漂移 | 判断数据是否达到可用标准 |
| L7 运维调度系统 | 采集任务、清洗任务、聚合任务、报表任务、告警任务 | 调度是否成功、耗时、失败重试、入库状态、最后更新时间 | 判断报表和监控结果是否受到任务异常影响 |

结果：

- 无异常：返回覆盖范围和巡检结论。
- 有异常：携带项目、时间、指标、来源进入排查。

确认问题：

- v1 监控阈值是否先采用系统默认规则，后台允许覆盖，但不先做复杂规则编排？
- 以上 7 层监控里，v1 是否先以“能接到真实数据源的层级”为准，未接入的层级在思维链中明确标记为未接入，而不是用 mock 结果替代？

已确认：

- 监控巡检 7 层对象需要新增采集 MCP，优先补齐媒体报表指标资产、媒体报表采集、媒体点击日志、自归因结果、运维调度、平台数据上报报告、数据质量监控等链路观测能力。
- 采集 MCP 采用统一采集网关 + 内部分 adapter。Agent / Skill 调用统一网关，后台和日志按 adapter 展示具体数据源。

采集网关建议能力：

| 能力 | 用途 |
| --- | --- |
| `collect.media_report_status` | 检查媒体报表采集状态 |
| `collect.media_metric_asset_status` | 检查媒体指标资产和口径上下文 |
| `collect.click_log_status` | 检查媒体点击日志采集状态 |
| `collect.self_attribution_status` | 检查自归因结果采集状态 |
| `collect.platform_event_report_status` | 检查平台事件上报报告 |
| `collect.data_quality_status` | 检查数据质量监控结果 |
| `collect.scheduler_status` | 检查运维调度系统状态 |

### 8.4 指标差异排查

消耗差异优先检查：

- 媒体采集。
- 报表调度。
- 入库状态。
- 用户权限。
- 账户可见性。
- 媒体账单源。

激活、注册、付费、关键行为差异检查：

- 指标口径。
- 媒体侧值。
- 智投 BI 值。
- 事件上报。
- 归因。
- 回传。
- 入库。
- 报表聚合。

确认问题：

- 差异排查是否统一先调报表 MCP 获取系统实际值，再决定是否追问媒体侧对比源？

## 9. P1 服务闭环

### 9.1 疑问解答

流程：

1. 判断是否为概念、流程、规范、口径问题。
2. 检索知识库。
3. 判断相关性。
4. 低相关内容丢弃。
5. 生成结构化答案。
6. 附来源。

确认问题：

- 知识库相关性不足时，是否默认回答“当前知识库缺少资料”，而不是联网搜索补齐内部业务口径？

### 9.2 指标解释器

展示模块：

- 指标定义。
- 计算口径。
- 埋点位置。
- 上报事件。
- 关键字段。
- 上报方。
- 计算过程。
- 报表差异。
- 常见歧义。
- 来源。

交互：

- 默认只展开核心定义和口径。
- 其他模块折叠。
- 不全部平铺。

确认问题：

- 指标解释资料是否以后台模块化录入为主，知识库只作为补充来源？

已确认：

- 指标解释资料以后台结构化录入为主，知识库作为补充来源。
- 首批指标覆盖消耗、展示、点击、激活、注册、付费、ROI、自然量、媒体回传和 BI 差异指标。

首批指标范围：

- 消耗
- 展示
- 点击
- 激活数
- 注册数
- 注册设备数
- 注册账号数
- 付费数
- 付费金额
- 首日 ROI
- 首周 ROI
- 自然量
- 媒体回传激活数
- BI 激活数
- 回传注册数
- BI 注册数
- 回传付费数
- BI 付费数

每个指标结构化字段：

- 指标定义
- 计算公式
- 数据来源
- 上报事件
- 关键字段
- 统计时间
- 去重规则
- 常见差异
- 涉及报表
- 适用场景
- 不适用场景
- 来源资料

### 9.3 监测链接交付

流程：

1. 识别项目、媒体、包或分包。
2. 检查是否已有可用链接。
3. 无链接时检查是否能创建。
4. 生成或返回监测链接。
5. 校验归因参数和包关系。
6. 输出链接和可投放状态。

确认问题：

- 监测链接创建是否纳入 v1，还是 v1 只做已有链接查询和可用性校验？

已确认：

- 监测链接交付 v1 既支持查询已有链接，也支持创建新链接。
- 创建新链接采用有权限用户直接创建，不先做审批流。必须包含权限校验、创建前预检、创建参数确认、创建日志、来源记录和失败原因。

### 9.4 自动报表

流程：

1. 识别报表任务。
2. 提炼模板。
3. 校验指标。
4. 用类 Excel 组件确认模板。
5. 调用 MCP 拉取数据。
6. 展示数据预览。
7. 用户确认是否定时。
8. 保存报表任务。

确认问题：

- v1 是否先支持文本模板和 Excel 模板，截图识别模板放到下一阶段？

已确认：

- 自动报表 v1 先支持文本模板和 Excel 模板，截图识别后置。
- 文本模板支持用户描述报表名称、指标、维度、筛选条件、时间范围、发送频率。
- Excel 模板 v1 只支持标准二维表，读取表头、sheet、固定字段、指标列、维度列。
- 复杂合并单元格、多区域透视表、跨 sheet 公式和截图识别后置。

## 10. 后台设计

后台是运行时配置中心，不是传统后台功能堆叠。

需要模块：

| 模块 | 用途 |
| --- | --- |
| Skill 管理 | 启停、版本、输入输出、依赖 MCP、评测用例 |
| Prompt 管理 | 路由、调用 Skill、结果排版、追问策略 |
| MCP 管理 | 工具列表、权限、参数 schema、调用日志 |
| 联调配置 | 媒体账号、事件资产、回传位置、设备、游戏账号 |
| 指标解释资料 | 指标定义、字段、事件、口径、差异说明 |
| 监控规则 | 默认阈值、项目覆盖、告警策略 |
| 记忆管理 | 用户偏好、项目上下文、历史问题 |
| 评测对接 | 连弩评测准入状态、最近评测结果摘要、失败原因链接 |

已确认：

- 后台 v1 只做配置和追踪，不先做复杂审批流。
- 评测执行由连弩负责，后台只展示准入状态和追踪入口。

## 13. 开发任务拆解

### T1 运行时协议补齐

- 核查 `AgentProcessEvent` 类型是否覆盖 v1 事件。
- 后端所有服务输出 `process_event`。
- `done.metadata.process_events` 完整持久化。
- 前端刷新后恢复思维链、来源、组件状态。
- 连弩接口返回同一事件链。

验收：同一条消息在页面、刷新后、评测接口中看到一致事件链。

### T2 Skill Contract 落地

- 建立 Skill contract 文件或数据表。
- 后台支持查看、启停、编辑、保存。
- 每个 Skill 绑定输入、输出、MCP、组件、评测用例。

验收：Skill 状态刷新不丢，配置能被 Agent runtime 读取。

### T3 MCP 真实调用治理

- 梳理现有 MCP。
- 删除冒充真实能力的 mock。
- 请求和返回参数按真实工具展示。
- 工具失败返回缺口。
- 来源挂到 `source.attached`。

验收：查不到数据时不生成假结论。

### T4 监控巡检闭环

- 实现 `ads_health_monitor_skill`。
- 支持宽泛异常先巡检。
- 异常后自动转排查。
- 输出巡检结果卡。
- 覆盖媒体报表指标资产、媒体报表采集、媒体点击日志、自归因结果、平台数据上报报告、数据质量监控、运维调度系统 7 层监控对象。
- 对未接入的数据源输出“未接入/不可用”状态，不生成假巡检结论。

验收：“我想看看昨天投放有什么异常”不再要求用户先填对比数据源。

### T5 指标差异排查闭环

- 实现 `metric_diff_diagnosis_skill`。
- 消耗、激活、注册、付费分路径。
- 先取系统实际值。
- 需要媒体侧来源时再表单确认。

验收：激活差异、消耗差异不走错排查链路。

### T6 自动联调闭环

- 实现 `auto_debug_skill`。
- 后台联调配置可维护。
- 会话中展示联调发起和步骤。
- 右侧栏展示截图、日志、执行状态。

验收：巨量联调可以从对话发起到给出通过/不通过结论。

### T7 投放前质量保障闭环

- 实现 `preflight_quality_check_skill`。
- 包、媒体应用、对接、归因、报表、联调状态合并判断。
- 输出可投放结论。

验收：用户问“这个包能不能投”时给出明确结论和证据。

### T8 指标解释器闭环

- 实现 `metric_explainer_skill`。
- 后台录入指标解释资料。
- 前端动态组件展示。
- 来源可追踪。

验收：注册数、注册设备数、付费数解释不再是纯文本。

### T9 自动报表闭环

- 实现报表模板和定时报表 Skill。
- 支持模板确认、指标校验、数据预览、定时任务。

验收：每天 10 点广告消耗日报能从对话创建为任务。

### T10 评测集与准入

- 建立 P0/P1 用例集。
- 每个 Skill 至少 5 条核心用例。
- 接入连弩执行。
- 输出失败原因。

验收：每次发布前能跑路由、Skill、MCP、来源、组件闭环评测。
