## 0. 这份文档解决什么问题

这份文档不再讨论“要不要做自动联调”，而是直接回答：

1. 小乔第一阶段自动联调具体做什么
2. 哪些现有研究成果可以直接吸收
3. 要拆成哪些后端对象、接口、页面承接和执行任务
4. 哪些点先做成专项模板，哪些点暂不纳入第一批实施

## 1. 当前实施结论

当前最合理的落地方式不是把交接包原样塞进小乔，而是把它改造成：

`小乔联调流中的第一批自动联调专项能力`

第一批专项范围先明确为：

1. 媒体侧：巨量 / 抖音链路
2. 终端侧：Android
3. 执行目标：扫码联调、刷到广告、点击广告、拉起应用、等待联调成功标记
4. 运行方式：小乔发起任务，执行器半自动执行，过程中保留人工接管点

当前不把它写成“通用全媒体全终端自动联调引擎”，否则范围会虚。

---

## 2. 可以直接吸收的实施骨架

来自交接包、可直接转成小乔能力骨架的部分：

1. 三段执行链路
   - Web 准备阶段
   - Mobile 执行阶段
   - Success 轮询阶段

2. 五个核心能力块
   - `WebSOPExecutor`
   - `MobileSOPExecutor`
   - `MiniMaxVision`
   - `ADBTools`
   - `SuccessPoller`

3. 执行留痕机制
   - `logs/steps`
   - `logs/errors`
   - `logs/artifacts`

4. 典型失败点
   - 登录协议勾选失败
   - 弹窗阻塞
   - 二维码抓取失败
   - ADB 连接失败
   - 广告识别失败
   - 坐标点击偏移
   - 成功标记轮询超时

这些内容适合直接进入小乔的任务模型、结果模型和异常模型。

## 3. 不原样吸收的内容

以下内容不能直接写进小乔主干设计：

1. 固定媒体文案与关键词
   - 如“转化联调”“授权测试”“打开”

2. 固定技术实现细节
   - `向左偏移 25px 点击 checkbox`
   - `最多滑动 8 次`

3. 固定环境假设
   - 固定 scrcpy 标题
   - 固定设备分辨率
   - 固定 URL

4. 把 MiniMax 直接写成唯一视觉模型

这些都应降级为：

- 专项配置
- 执行模板参数
- 可替换能力实现

## 4. 小乔内的正式实施形态

### 4.1 产品形态

在小乔里，自动联调不是一个独立系统，而是：

`广告联调流 -> 自动联调专项任务`

用户路径：

1. 用户在主工作台发起联调问题
2. 小乔识别为“可进入自动联调专项”
3. 小乔生成联调任务
4. 用户确认关键输入
5. 小乔触发自动联调执行
6. 页面实时展示执行阶段、截图、日志、状态
7. 失败时允许人工接管
8. 最终沉淀联调结果报告

### 4.2 第一批只做半自动执行

第一批建议定位为：

`协同工作台 + 半自动执行`

原因：

1. 当前执行链路对环境依赖重
2. 视觉识别和移动端操作不够稳定
3. 广告是否刷到、点击是否生效、联调成功文案是否出现，都存在波动

因此第一批必须保留：

1. 执行前确认点
2. 执行中暂停 / 继续
3. 失败后人工接管
4. 留痕与回看

## 6. 接口拆解

第一批建议新增一组自动联调专项接口。

### 6.1 任务创建与执行

1. `POST /api/v1/xiaoqiao/debug-automation/tasks`
   - 创建自动联调任务

2. `POST /api/v1/xiaoqiao/debug-automation/tasks/{task_id}/start`
   - 启动执行

3. `POST /api/v1/xiaoqiao/debug-automation/tasks/{task_id}/pause`
   - 暂停执行

4. `POST /api/v1/xiaoqiao/debug-automation/tasks/{task_id}/resume`
   - 恢复执行

5. `POST /api/v1/xiaoqiao/debug-automation/tasks/{task_id}/takeover`
   - 标记人工接管

### 6.2 任务查询

1. `GET /api/v1/xiaoqiao/debug-automation/tasks`
2. `GET /api/v1/xiaoqiao/debug-automation/tasks/{task_id}`
3. `GET /api/v1/xiaoqiao/debug-automation/tasks/{task_id}/steps`
4. `GET /api/v1/xiaoqiao/debug-automation/tasks/{task_id}/result`

### 6.3 配置管理

1. `GET /api/v1/xiaoqiao/admin/debug-automation/configs`
2. `GET /api/v1/xiaoqiao/admin/debug-automation/configs/{config_id}`
3. `POST /api/v1/xiaoqiao/admin/debug-automation/configs`
4. `PUT /api/v1/xiaoqiao/admin/debug-automation/configs/{config_id}`

## 7. 页面承接拆解

### 7.1 主工作台

主工作台的联调结果区新增：

1. 当前阶段
2. 当前步骤
3. 最近截图
4. 最近一条日志
5. 成功判断状态
6. 失败原因
7. 人工接管按钮
8. 最终报告入口

### 7.2 管理页面

管理页面新增一个专项入口：

`自动联调配置`

页面最少包含：

1. 模板列表
2. 媒体 / 终端 / 环境过滤
3. 关键词配置
4. 超时配置
5. 启停状态

## 9. 第一批开发任务

### 9.1 后端

1. 建立 `DebugAutomationTask / Config / Step / Result` mock schema
2. 提供任务创建、启动、查询、步骤列表、结果查询接口
3. 先不接真实执行器，先用 mock 执行状态流转打通

### 9.2 前端

1. 在联调结果区增加自动联调专项面板
2. 展示状态机、步骤流、截图位、日志位、结果报告位
3. 增加人工接管与重试按钮

### 9.3 管理页

1. 增加自动联调配置入口
2. 增加关键词和超时配置的列表 / 编辑承接

### 9.4 第二阶段再接执行器

第二阶段再考虑：

1. 把现有 `executor.py` 拆成服务能力
2. 用小乔任务调度去触发执行器
3. 将截图、日志、产物回写到小乔对象
