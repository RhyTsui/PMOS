# 小乔智投-业务与系统全貌

- 版本：v0.1
- 状态：本轮重构基线
- 日期：2026-05-08
- 负责人：Codex + 用户
- 用途：为重新起一套小乔智投系统提供业务和系统全貌基线，作为新版规划、需求、功能文档的共同上游。

---

## 0. 这份文档解决什么问题

本轮之前的文档更多回答了“小乔要做什么”，但对“当前广告业务和现行系统到底是什么样、为什么新系统要这样承接”写得不够具体。

这份文档补的是中间层：

1. 现行广告业务链条有哪些固定工作。
2. 现行系统能力大致分布在哪里。
3. 新系统应该承接什么，不应该承接什么。
4. 哪些能力未来要被封装为 agent 可调用能力。

---

## 1. 业务全貌判断

### 1.1 当前业务不是空白场景

现状前提：

1. 广告数仓已经运行多年。
2. 线上与线下工作流已经长期存在。
3. 数据查询、联调、配置、排查、知识解释等能力分布在现行系统与人工流程中。
4. 现有问题不在于“完全没能力”，而在于“没有被统一编排成新系统能力”。

### 1.2 新系统的真实目标

重新起一套小乔智投系统，不是替代所有旧系统，而是新增一层统一承接层：

1. 把自然语言问题收进来。
2. 把问题判断成哪类业务流。
3. 把现有系统能力重新封装成可调用能力。
4. 把结果沉淀成可回看、可继续协同的结构化对象。

## 2. 现行业务链条

### 2.1 使用帮助链

当前人工工作内容通常包括：

1. 解释指标口径。
2. 解释系统路径和入口。
3. 解释媒体规则、归因逻辑、回传规则。
4. 回答常见广告技术问题。

这类问题本质上依赖：

1. 指标定义。
2. 路径说明。
3. 知识文档。
4. 历史处理经验。

### 2.2 需求沟通链

当前人工工作内容通常包括：

1. 接媒体回传 / 接入类需求。
2. 收集应用、媒体、事件、回传目标、验收方式等信息。
3. 反复追问缺失字段。
4. 在多个角色间推动需求收口。

这类问题的典型痛点：

1. 需求口头化。
2. 字段缺失。
3. 责任边界不清。
4. 后续落地难以追踪。

### 2.3 问题排查链

当前人工工作内容通常包括：

1. 根据异常描述确定属于哪种问题。
2. 补齐媒体、应用、计划、时间窗等最小上下文。
3. 在多个系统中找证据。
4. 给出初步结论、置信度和下一步建议。

典型证据源包括：

1. 数仓明细或聚合表。
2. BI / 看板指标。
3. SDK 上报链路。
4. 回传日志。
5. 归因结果。
6. 媒体后台状态。

### 2.4 广告联调链

当前人工工作内容通常包括：

1. 确认账号、应用、包体、环境和测试设备。
2. 检查白名单、回传配置、入口方式等准备项。
3. 拉起联调流程。
4. 跟踪执行状态。
5. 回收日志、截图、结果报告。

这类问题的痛点：

1. 准备项散。
2. 依赖多。
3. 过程不透明。
4. 日志与结论难回看。

## 3. 现行系统能力地图

### 3.1 数据与分析层

至少存在如下能力域：

1. 广告数仓
2. BI / 报表系统
3. 指标口径体系
4. 分析 SQL / 模板能力

### 3.2 业务执行层

至少存在如下能力域：

1. 媒体后台
2. 归因 / 回传能力
3. 配置与接入能力
4. 联调与验证能力

### 3.3 观测与排查层

至少存在如下能力域：

1. SDK / 上报日志
2. 链路状态观测
3. 回传日志
4. 异常案例经验

### 3.4 协同与知识层

至少存在如下能力域：

1. 需求沟通与工单式协作
2. 文档 / wiki / 知识资产
3. 历史问题与解决方案
4. 规则与流程约束
5. 已有知识库平台
6. 已有评测平台

---

## 4. 新系统应该如何重编这些能力

### 4.1 重编原则

1. 不替代已有成熟底层能力。
2. 不让用户直接面对分散系统。
3. 先用统一工作台承接，再路由到能力层。
4. 所有关键结果都回写到任务与结果对象。

补充治理说明：

1. 这里的 `统一工作台` 指统一入口和统一任务承接，不代表首页同时展示所有能力层
2. 运行治理、专项能力、历史沉淀默认作为次级承接面，不应反推首页平铺结构

### 4.2 目标形态

旧世界：

`人 -> 多系统切换 -> 人工串联 -> 结果散落`

新世界：

`人 -> 小乔统一工作台 -> 任务运行时 -> 能力层调用 -> 结构化结果沉淀`

页面解释约束：

1. 这条链描述的是系统运行关系，不是首页布局草图

### 4.3 能力封装位

未来应被封装为 agent 可调用能力的对象，至少包括：

1. 数仓查询能力
2. BI 结果拉取能力
3. 指标口径与路径引用能力
4. 日志检索与链路比对能力
5. 回传 / 归因检查能力
6. 联调准备项检查能力
7. 自动化执行能力
8. 通知与协同推送能力

同时明确两条边界：

1. 知识库平台已有，小乔只接入知识能力，不自建知识库平台。
2. 评测平台已有，小乔只引用评测结果或跳转评测平台，不自建评测平台。

---

### 4.4 当前阶段的数据接入现实

当前现状已经明确：

1. 智投预发布数据库存在，但运维已确认本地开发环境禁止直连。
2. 因此小乔第一阶段不能把“本地连通预发库”当成前置条件。
3. 当前更合理的实现路径不是等数据库权限，而是先把业务对象、任务结构、结果结构和接口 contract 稳定下来。

这意味着第一阶段的数据承接要分成两层：

1. 正式 mock contract 层
   - 服务当前返回什么对象
   - 页面当前消费什么结构
   - 四条业务流当前分别沉淀什么结果物
2. 后续真实数据切换层
   - 真实数仓 / 业务库 / 外部平台能力接入
   - mock 字段与真实字段映射
   - mock 内容与真实结果替换

这里的 mock 不是示意数据，而是正式交付期的稳定承接层。

## 5. 第一阶段结果物判断

第一阶段的正式结果物不应该再是聊天文本，而应分别形成：

1. 使用帮助
   - 指标解释
   - 系统路径
   - 引用来源
   - 下一步建议

2. 需求沟通
   - 结构化需求单
   - 缺失字段清单
   - 依赖项清单
   - 下一步协作动作

3. 问题排查
   - 一句话结论
   - 证据链
   - 置信度
   - 下一步建议

4. 广告联调
   - 联调准备项
   - 执行状态
   - 执行日志 / 摘要
   - 联调结果报告

---

## 6. 当前明确的系统边界

### 6.1 第一阶段一定要做

1. 统一入口。
2. 四条业务流全部打通。
3. 任务运行时沉淀。
4. 结果区正式结构。
5. 本地仓库真实前后端对接。
6. 同一登录系统下的管理页面至少补齐提示词管理。
7. 以 mock contract 形式完成可持续联调，不等待本地直连预发库。

补充约束：

1. 这里没有要求把运行验证、管理页、专项页全部做成首页常驻内容

### 6.2 第一阶段先不扩成主目标

1. 素材生产系统。
2. 预测主系统。
3. 自动投放主系统。
4. 大屏主系统。
5. 全量规则监控主系统。
6. 知识库平台重做。
7. 评测平台重做。

这些能力如果在历史稿中出现，本轮只作为后续扩展候选，不直接成为第一阶段主交付。
