## 1. 核查结论

当前系统已经有运行时协议的雏形，但还没有形成完整的“协议驱动服务闭环”。

已落地：

- 前端工程已有 `AgentProcessEvent` 类型。
- `/api/chat` 已支持 SSE 输出 `process_event`。
- `done.metadata.process_events` 已有兼容逻辑。
- 前端 `useConversation` 和 `ChatContainer` 已能保存并从 `process_events` 恢复思维链和调用记录。
- MCP 管理已有 server 存储、发现、测试和工具 schema 展示基础。
- Skill 管理已有 MCP Skill 列表启停基础。
- 连弩评测接口已有 `/api/xiaoqiao/evaluation/run`，能返回 `process_events`、`sources`、`scores`。

主要缺口：

- `AgentProcessEvent` 类型缺少 v1 设计里的 `capability.checked`、`skill.failed`。
- `/api/chat` 内仍有本地 `buildServicePlan` 计划和固定流程，不是真正的 Skill Contract Runner。
- Skill 管理当前存的是 `McpSkill`，不是业务 `SkillContract`。
- MCP 调用虽然会选择真实工具，但没有统一采集网关 MCP，也没有 P0/P1 固定工具契约。
- 自动联调配置和任务接口仍从 `demo-data` 返回，刷新后不具备正式持久化。
- 指标解释器仍引用 `features/metric-explainer/mock/*`，尚未接后台结构化指标资料。
- 连弩评测是确定性 adapter，不是真实 Agent runtime 执行结果。
- 来源有 `source_refs`，但 `source.attached` 事件还没有作为独立事件稳定输出。
- 确认表单尚未统一由 `clarify.requested` + `ui.component_rendered` 驱动。

## 5. 当前不直接改的原因

这轮先做代码核查和任务拆解，不直接进入 patch，原因是：

- 运行时协议、SkillContract、MCP 和评测耦合较高，直接改单点会继续补丁化。
- 需要先确定文件级任务边界，避免前端、后端、评测再次各自实现一套字段。
- 后续实现应按任务顺序推进，每完成一项都跑浏览器和连弩评测。
