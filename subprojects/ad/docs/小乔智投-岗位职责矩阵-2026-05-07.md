## 0. 文档目的

本文档用于把“小乔作为广告技术服务统一对外数字员工”在第一阶段具体承担哪些职责、承担到什么深度、哪些暂不承担，正式拆成岗位职责矩阵。

它承接：

1. 产品重启真源
2. 第一阶段需求真源
3. 数字员工方法论与第一阶段设计原则

---

## 1. 当前角色定义

小乔第一阶段是广告技术服务的统一对外数字员工。

它对外代表：

1. 广告技术服务窗口
2. 当前真人窗口的数字化延伸

它对内依赖：

1. 多 Agent 体系
2. 内部知识库
3. 确定性工作流
4. 业务运行时

---

## 2. 岗位职责总览

第一阶段岗位职责分为七类：

1. 接待职责
2. 判断职责
3. 解释职责
4. 排查职责
5. 需求收口职责
6. 联调推进职责
7. 记录沉淀职责

---

## 3. 职责矩阵

| 职责 | 第一阶段是否承担 | 承担深度 | 当前目标 | 暂不承担 |
| --- | --- | --- | --- | --- |
| 接待 | 是 | 高 | 接住进入广告技术服务窗口的请求 | 不做人为前置分流 |
| 判断 | 是 | 高 | 判断业务/非业务、广告域、轻重工作流 | 不做高风险最终拍板 |
| 解释 | 是 | 高 | 指标解释、逻辑说明、系统导航、知识问答 | 不替代所有复杂分析 |
| 排查 | 是 | 高 | 广告异常、数据 gap、归因、回推问题排查 | 不在证据不足时强行定责 |
| 需求收口 | 是 | 中 | 模糊需求转结构化需求 | 不代替所有需求优先级决策 |
| 联调推进 | 是 | 中 | 广告联调流程推进与状态记录 | 不默认自动执行高风险动作 |
| 记录沉淀 | 是 | 高 | 会话、任务、结果、Case 沉淀 | 不把沉淀完全外包给外部平台 |

---

## 4. 各职责详细说明

## 4.1 接待职责

### 4.1.1 要做什么

1. 作为统一入口接住请求
2. 不要求用户先学习系统结构
3. 不要求用户先判断该找谁

### 4.1.2 做到什么程度

第一阶段要实现：

1. 自由问答入口
2. 连续追问
3. 接住广告技术服务相关请求

### 4.1.3 暂不承担

1. 替代组织级人工客服体系
2. 承担所有非广告技术服务域问题

## 4.2 判断职责

### 4.2.1 要做什么

1. 判断是否是业务问题
2. 判断是否属于广告域
3. 判断是轻工作流还是重工作流
4. 判断属于哪个诉求类型
5. 不清楚时主动追问

### 4.2.2 做到什么程度

第一阶段要实现：

1. 业务 / 非业务判断
2. 广告域判断
3. 轻 / 重工作流判断
4. 典型诉求类型判断

### 4.2.3 暂不承担

1. 高风险最终业务裁决
2. 缺证据时替真人强拍板

## 4.3 解释职责

### 4.3.1 要做什么

1. 指标解释
2. 逻辑说明
3. 系统路径说明
4. 一般知识问答
5. 使用引导

### 4.3.2 做到什么程度

第一阶段要实现：

1. 走轻工作流
2. 保留上下文
3. 可引用知识和示例
4. 必要时升级到重工作流

### 4.3.3 暂不承担

1. 所有复杂策略分析
2. 所有跨域综合决策

## 4.4 排查职责

### 4.4.1 要做什么

1. 对广告异常进入排查链路
2. 输出结论、证据链、置信度和下一步建议
3. 支撑多轮追问

### 4.4.2 做到什么程度

第一阶段要实现：

1. 广告数据 gap 排查
2. 归因问题排查
3. 回推问题排查
4. 报表结果异常排查

### 4.4.3 暂不承担

1. 对所有复杂未知问题一次性给完美判断
2. 在无充分依据时给绝对结论

## 4.5 需求收口职责

### 4.5.1 要做什么

1. 把模糊需求转成结构化需求
2. 补齐缺失字段
3. 明确依赖项
4. 给出下一步动作

### 4.5.2 做到什么程度

第一阶段要实现：

1. 广告相关需求标准化
2. 对接类需求收口
3. 基础配置类需求收口

### 4.5.3 暂不承担

1. 所有需求的最终优先级决策
2. 所有排期与资源拍板

## 4.6 联调推进职责

### 4.6.1 要做什么

1. 发起联调
2. 展示联调准备项
3. 推进联调链路
4. 记录联调状态
5. 输出联调结果

### 4.6.2 做到什么程度

第一阶段要实现：

1. 联调流程可跟踪
2. 联调状态可见
3. 联调结果可沉淀

### 4.6.3 暂不承担

1. 默认放权执行所有高风险自动化动作
2. 替代真人做所有联调后置判断

## 4.7 记录沉淀职责

### 4.7.1 要做什么

1. 记录会话
2. 记录任务
3. 记录结果
4. 沉淀历史
5. 沉淀 Case

### 4.7.2 做到什么程度

第一阶段要实现：

1. 轻工作流有记录
2. 重工作流有结构化结果
3. 历史任务可回看
4. Case 可复用

### 4.7.3 暂不承担

1. 全量替代知识库平台
2. 全量替代评测平台

---

## 5. 第一阶段优先级排序

按当前需求判断，第一阶段职责优先级为：

1. 接待
2. 判断
3. 排查
4. 解释
5. 记录沉淀
6. 需求收口
7. 联调推进

说明：

- 接待与判断是统一入口前提
- 排查是当前最痛的业务主问题
- 解释是高频轻工作流
- 记录沉淀是数字员工成立的必要条件
- 需求与联调当前也重要，但优先级稍后于排查主链路
