# 智投chat 端到端直接交付设计原则

更新时间：2026-05-14

## 1. 原则定义

智投chat 的每个需求响应目标都应该是“端到端直接交付结果”。

系统不是为了延长对话，也不是为了生成长文本说明，而是基于预设流程、已有上下文、后台配置、MCP、Skill、知识库和动态组件，尽快把用户目标推进到可用结果。

一句话原则：

```text
能直接交付，就不要追问；
能自动获取，就不要让用户填写；
能组件承载，就不要长文本平铺；
能创建结果对象，就不要只给下一步建议。
```

## 2. 为什么这是设计原则

智投chat 面向的是广告业务服务，不是开放式聊天。

用户真正要的不是“聊明白”，而是：

- 查到数据。
- 定位问题。
- 创建报表。
- 发起联调。
- 生成需求单。
- 创建监控任务。
- 得到可复查的结论。
- 形成可继续追踪的记录。

因此，多轮对话不是初衷，长文本也不是初衷。

多轮对话只应该作为兜底：

- 条件缺失。
- 用户主动探索。
- 高风险操作确认。
- 系统能力不足，需要说明缺口。

长文本只应该作为辅助：

- 解释结论。
- 补充风险。
- 说明限制。
- 给出必要背景。

## 3. 默认响应目标

每个 Agent、Skill、Prompt、前端组件都必须围绕下面的目标设计：

1. 识别用户真实目标。
2. 自动准备上下文。
3. 判断是否能直接执行。
4. 自动调用已有能力。
5. 用动态组件交付过程和结果。
6. 把结果沉淀成可追踪对象。
7. 只在必要时一次性向用户确认。

标准链路：

```text
用户输入
→ 意图识别
→ 上下文准备
→ 能力可用性判断
→ 自动获取条件
→ 必要时一次性确认
→ 执行 Skill / MCP / Tool
→ 组件化展示结果
→ 记录沉淀
→ 后续可追踪
```

## 4. 不允许的默认行为

以下行为默认视为产品设计错误：

- 把用户可以直接完成的目标变成建议列表。
- 把系统可以查询的字段让用户手动填写。
- 把后台已经配置的字段暴露给用户确认。
- 用“下一步建议”替代实际执行。
- 用长文本替代表单、表格、图表、联调步骤、排查证据。
- 没有真实工具调用时用 mock 结果冒充已完成。
- 每次都要求用户提供项目、应用、媒体等可从上下文获取的信息。
- 宽泛问题不先巡检，直接让用户补大量排查条件。
- 联调需求不执行校验，只返回联调清单。
- 报表需求不生成模板和数据，只返回制作建议。

## 5. 必须优先自动获取的信息

系统应优先自动获取：

- 当前用户。
- 当前项目或关注项目。
- 用户历史记忆。
- 当前会话上下文。
- 最近一次任务上下文。
- 后台联调配置。
- 媒体账号配置。
- 回传查看位置。
- 测试设备。
- 项目、APPID、应用名称。
- 报表实际值。
- 数据更新时间。
- 已有监控规则。
- 已有需求或联调记录。

只有这些信息不存在、权限不足或存在歧义时，才追问用户。

## 6. 一次性确认规则

确实需要用户补充时，不能拆成多轮零散追问。

应该生成一次性结构化表单：

- 把所有缺失字段放在一个表单里。
- 必填和选填要区分。
- 提示信息放在 tips 中，不写成大段说明。
- 用户确认后自动继续流程，不再要求用户点击发送。
- 已经从上下文获得的字段默认填入，可编辑。

## 7. 各服务的直接交付要求

### 解答

直接交付：

- 指标解释器。
- 口径说明。
- 计算过程。
- 易混淆点。
- 来源。

不要只输出长段百科解释。

### 排查

直接交付：

- 排查路径。
- MCP 查询结果。
- 证据链。
- 差异比对。
- 根因或缺失项。
- 修复动作。

不要把缺少条件放在“下一步建议”里。

### 对接

直接交付：

- 结构化需求表单。
- 依赖校验。
- 是否已有接入能力。
- 是否进入需求池。
- 是否可先创建监测链接。

不要只给“建议准备文档”。

### 联调

直接交付：

- 联调前置校验结果。
- 自动联调启动组件。
- 联调步骤和截图。
- 错误日志。
- 最终报告。

不要让用户确认后台已有配置。

### 报表

直接交付：

- 报表模板。
- 指标校验。
- 数据预览。
- 定时任务。

不要把报表任务误判成监控，也不要只输出报表制作思路。

### 分析

直接交付：

- 数据图表。
- 贡献拆解。
- 关键结论。
- 下钻入口。

不要只给泛泛分析框架。

### 监控

直接交付：

- 监控规则。
- 巡检结果。
- 告警任务。
- 异常后自动转排查。

宽泛“看看昨天有什么异常”应先巡检，不先追问对比数据源。

## 8. Agent 执行约束

Agent 每次响应前必须做内部判断：

```text
用户目标是什么？
是否已有预设服务流程？
哪些字段可以自动获取？
哪些字段必须用户确认？
是否有真实 MCP / Skill / Tool 可用？
最终交付物是什么？
如果无法交付，第一句话应该说明缺什么？
```

Agent 输出时必须遵守：

- 能流式步骤输出，就不要一次性返回大文档。
- 能组件化确认，就不要文本追问。
- 能调用真实工具，就不要用 mock 替代。
- 能沉淀结果，就不要只停留在聊天消息。
- 能直接执行，就不要写“你可以去做”。

## 9. 前端组件约束

前端不是展示长文本的容器，而是交付过程和结果的载体。

必须优先使用：

- 表单组件。
- 表格组件。
- 指标解释器。
- 思维链。
- 调用与来源。
- 右侧详情区。
- 联调步骤组件。
- 监控任务组件。
- 报表模板组件。
- 排查证据组件。

消息正文只承载：

- 用户需要看到的结论。
- 必要解释。
- 当前可操作组件。
- 最终结果。

## 10. 验收标准

一个需求只有满足以下条件，才算真正完成：

- 用户目标被识别正确。
- 系统自动获取了可获取的信息。
- 缺失信息用一次性组件补齐。
- 调用了真实能力或明确说明缺失能力。
- 页面展示了可操作或可复查的交付物。
- 结果沉淀为记录、任务、报表、需求、联调或排查对象。
- 用户可以用一个明确操作完成下一步，而不是读一堆建议。

状态判断：

- `solved`：已直接交付结果。
- `partial`：流程或组件已有，但缺真实工具、数据或沉淀。
- `unsolved`：只有文本建议、计划或浅层 UI。

## 11. 与协作方式的关系

后续我和 Codex 协作时，复杂需求不应混成一条长对话直接开改。

每个需求先按直接交付原则拆解：

```text
直接交付路径：
自动获取信息：
需要确认信息：
执行能力：
交付组件：
结果沉淀：
验收问题：
```

如果这个拆解无法完成，说明需求还没有达到可实现状态，应先补流程或能力，而不是继续堆 UI 和 Prompt。
