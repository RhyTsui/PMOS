## 0. 文档目的

本文档用于把小乔第一阶段最核心的数据对象正式收敛出来，作为后续：

1. 数据表设计
2. 接口定义

## 1. 对象设计原则

### 1.1 对话与任务分离

小乔不是单纯聊天产品，所以必须区分：

1. 会话对象
2. 任务对象

一段会话里可以没有正式任务，也可以产生一个或多个任务。

### 1.2 轻重工作流统一建模

轻工作流和重工作流都属于工作，只是状态推进复杂度不同。

因此对象层不应割裂成两套完全不同模型，而应共享主干结构。

### 1.3 结果物独立建模

需求单、排查结论、联调报告、监控事件、Case 不应只留在 message 文本里，应以独立结果对象存在。

### 1.4 上下文作为一等对象

广告技术服务数字人能否像真人窗口一样工作，关键就在于上下文不能丢。

因此上下文必须有明确对象承接，而不是散落在 prompt 或 message 文本里。

## 2. 核心对象总览

第一阶段建议至少定义八类对象：

1. 会话对象 `Conversation`
2. 消息对象 `Message`
3. 任务对象 `Task`
4. 任务上下文对象 `TaskContext`
5. 工作流结果对象 `WorkflowResult`
6. 证据对象 `EvidenceItem`
7. Case 对象 `CaseRecord`
8. 能力路由对象 `RoutingDecision`

---

## 3. 会话对象 Conversation

## 3.1 作用

承接用户与小乔的一段连续对话。

## 3.2 核心字段

1. `conversation_id`
2. `user_id`
3. `title`
4. `status`
5. `started_at`
6. `updated_at`
7. `last_message_at`
8. `current_mode`

## 3.3 说明

`current_mode` 不是暴露给用户的 Agent 名称，而是内部当前主态，例如：

1. natural-chat
2. light-workflow
3. heavy-workflow

---

## 4. 消息对象 Message

## 4.1 作用

承接会话中的单条输入或输出。

## 4.2 核心字段

1. `message_id`
2. `conversation_id`
3. `role`
4. `content`
5. `message_type`
6. `created_at`
7. `related_task_id`

## 4.3 message_type 建议

1. user_input
2. assistant_reply
3. clarification
4. system_notice
5. workflow_summary

## 4.4 说明

不是所有消息都会产生任务，但任务消息要能反查回 message。

## 10.2 核心字段

1. `routing_id`
2. `conversation_id`
3. `source_message_id`
4. `is_business_related`
5. `business_domain`
6. `intent_type`
7. `workflow_level`
8. `decision_reason`

## 10.3 说明

---

## 11. 对象关系

建议关系如下：

1. 一个 `Conversation` 包含多条 `Message`
2. 一个 `Conversation` 可产生多个 `Task`
3. 一个 `Task` 对应一个 `TaskContext`
4. 一个 `Task` 可产生一个或多个 `WorkflowResult`
5. 一个 `Task` 可关联多个 `EvidenceItem`
6. 一个 `Task` 可升级为一个 `CaseRecord`
7. 一个 `Message` 可触发一个 `RoutingDecision`

---

## 12. 第一阶段最关键对象优先级

第一阶段最先要稳定的对象是：

1. `Conversation`
2. `Task`
3. `TaskContext`
4. `WorkflowResult`
5. `RoutingDecision`

理由：

- 没有会话和任务，就没有数字员工运行时
- 没有上下文，就无法像真人窗口一样连续处理
- 没有结果物，就只能停留在聊天
- 没有路由对象，就无法调试误判与误分发
