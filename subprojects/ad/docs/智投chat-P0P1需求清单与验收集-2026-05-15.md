# 智投chat P0/P1需求清单与验收集

版本日期：2026-05-15  
适用范围：智投chat 产品设计、Agent/Skill/MCP 编排、前后端开发、测试验收  
当前阶段：聚焦 P0 + P1，P2/P3 只保留路线，不进入本轮交付

## 1. 产品定位

智投chat 是面向游戏发行团队的广告业务 AI 服务入口。

它不是通用聊天工具，也不是后台功能问答入口，而是基于广告 BI、监测管理、媒体配置、报表数据、知识库、MCP 和自动化 Skill，帮助用户通过对话完成广告业务需求的端到端直接交付。

核心描述：

> 广告业务需求端到端直接交付。

这里的“直接交付”指：

- 用户只需要说明目标范围和业务诉求。
- 系统自动识别意图、补齐上下文、调用知识库/MCP/Skill、完成校验、生成结论或任务。
- 能直接执行的，不返回“建议用户去做”；不能执行的，第一时间说明缺什么、为什么缺、下一步谁来补齐。
- 依赖用户确认的信息，一次性用结构化表单收齐。
- 输出以结论、证据、结果组件、下一步动作呈现，不以长篇文档式回复为默认形态。

## 2. 目标用户

主要用户：

- 游戏发行人员
- 广告投放人员
- 非投放类管理职能人员
- 内部产品、技术、数据支持人员

用户共同诉求：

- 少解释，直接告诉我能不能投、哪里有问题、怎么处理。
- 系统能拿到的数据不要再让我填写。
- 问题排查要排除系统问题，减少反复拉群、截图、人工查表。
- 联调、监控、报表这类高频流程要自动化、可追踪、可复用。
- 业务口径、归因逻辑、指标差异要能解释清楚，降低沟通成本。

## 3. 三层价值目标

智投chat 的能力建设按三层递进，不能跳过前两层直接追求“智能洞察”。

### 第一层：业务质量保障

目标是持续给用户提供可信的投放基础设施：

- 没有问题的包。
- 没有问题的归因回推。
- 没有问题的报表数据。
- 持续运行的监控保障。

这一层是生命线，优先级最高。它解决的是“能不能安全投放”“系统有没有问题”“业务事故能不能提前发现”。

### 第二层：高频服务直接交付

目标是把用户最常用、最耗人工的广告技术服务直接交付出来：

- 直接给到用户监测链接。
- 直接完成疑问解答。
- 直接完成数据排查。
- 直接生成自动报表。

这一层是效率入口，要求少追问、少长文、少建议，优先用结构化组件和真实工具完成闭环。

### 第三层：洞察、预测与决策

目标是在前两层稳定后，再进一步提供：

- 经营洞察。
- 效果预测。
- 投放决策辅助。
- 后续自动投放能力。

这一层不能脱离可信数据、可信归因、可信报表和稳定监控，否则容易变成不可靠的分析文本。

## 5. 总体服务边界

### P0-1 投放前质量保障

目标：确认投放包没有问题、媒体对接功能没有问题、系统运行没有问题，最终回答“能不能投放”和“不能投放的原因”。

包含能力：

- 对接：新媒体、新对接方式、新应用或新事件接入，属于联调前置场景。
- 包检查：识别新包、包更新、媒体过审、不需要过审媒体的出包状态。
- 自动联调：检查媒体后台、事件资产、渠道包、二维码、扫码、激活、注册、付费、关键行为、回传轮询。
- 投放可行性结论：给出通过、不通过、阻塞项、需人工处理项。
- 联调记录：支持查看历史联调记录、失败原因、截图、日志、复测入口。

触发方式：

- 用户主动对话触发，例如“巨量这边要立即联调，应用已经共享到 wuyanlan@dobest.com”。
- 系统事件触发，例如检测到新包、包更新、媒体后台过审、对接方式变化。
- 后台配置触发，例如媒体账号、回传查看位置、设备环境、游戏账号配置完成后自动巡检。

用户需要提供：

- 项目范围：appid / appname / 项目选择。
- 媒体范围：例如巨量、Applovin。
- 联调目标：默认激活、注册、付费、关键行为；如果游戏无付费，也模拟付费联调。

系统应自动获取：

- 应用包下载地址。
- 媒体后台是否有应用。
- 媒体后台应用是否过审。
- 包版本、分包、渠道包、监测链接。
- 媒体账号、回传查看位置、测试设备。
- 数据是否已上报，数据上报 MCP 是否通过。
- 自动联调步骤、截图、日志、轮询结果。

交付结果：

- 联调是否通过。
- 是否可以立即投放。
- 如果不通过，明确阻塞原因和责任侧。
- 如果需要用户操作，只输出具体操作指导，例如“请先完成应用共享”。
- 不输出泛泛的“下一步建议”。

### P0-2 数据排查与系统健康保障

目标：帮助用户判断数据是否正常，若异常，定位是系统、采集、调度、权限、上报、媒体侧还是业务操作导致。

包含能力：

- 宽泛异常巡检：例如“我想看看昨天投放有什么异常”。
- 指标差异排查：例如媒体付费数 100，智投是 3。
- ROI 下降排查：例如最近首日 ROI 下降了一倍。
- 自然量异常排查：例如 H5 游戏没开搜索但自然量 100 多。
- 回传延迟监控：例如巨量回传延迟超过 30 分钟告警。
- 系统健康巡检：调度、入库、报表、账户授权、可见性、MCP 权限。

路由原则：

- 宽泛异常请求优先进入监控/巡检，不提前要求用户填写对比数据源。
- 巡检发现异常后，携带项目、时间、指标和来源自动进入对应排查。
- 用户明确给出差异对象时，直接进入排查。
- 消耗类问题不默认检查监测回传，应优先检查采集、报表调度、入库、权限、媒体账单源。
- 激活、注册、付费、关键行为差异才进入归因、上报、回传链路检查。

监控巡检默认检查项：

- 消耗突增、突降。
- 预算撞线。
- 账户授权与数据可见性。
- 激活、注册、付费、关键行为。
- 回传延迟与回传成功率。
- 昨日报表是否完成调度和入库。

交付结果：

- 未发现异常：返回监控结果和覆盖范围。
- 发现异常：进入对应排查，输出异常对象、差异数据、原因判断、证据来源和处理动作。
- 信息不足：用一次结构化表单收齐最小必要条件，而不是分多次追问。

### P1-1 业务理解与规范解释

目标：帮助用户理解广告业务流程规范、归因回推逻辑、报表指标口径和不同系统口径差异。

典型问题：

- 日报、周报、月报和 ROI 报表里的注册数为什么不同？
- 注册设备数和注册数有什么区别？
- 激活数、注册数、付费数的统计口径有什么区别？
- 某个指标未知，帮我解释这个指标是什么意思。

交付方式：

- 使用指标解释器动态组件，不把所有内容一次性平铺。
- 展示定义、计算口径、埋点位置、上报事件、关键字段、上报方、计算过程、常见歧义、报表差异。
- 知识库无对应内容时明确提示缺少指标解释资料，不使用 mock 数据替代。
- 来源需要标明来自知识库、报表 MCP 或其他可追踪来源。

### P1-2 报表与聚合分析提效

目标：通过聚合分析 MCP 和拼表任务，为用户生成自定义格式的日周月报、维度对比数据和分析结论。

典型问题：

- 帮我创建一个每天上午 10 点发送的广告消耗日报。
- 看一下昨天巨量按项目维度的消耗和 ROI。
- 上传 Excel 模板，按这个格式生成周报。

报表前置方式：

- 截图提取报表模板。
- 上传 Excel 模板。
- 手动输入报表模板。
- 用户指定系统页面并描述报表需求。

处理流程：

1. 识别这是报表任务，不误判为监控任务。
2. 提炼最终报表模板。
3. 校验指标是否存在，只展示有问题的指标；无问题则提示校验无误。
4. 用类 Excel 表格组件让用户确认模板。
5. 确认后立即调用 MCP 获取数据，流式展示数据预览。
6. 用户确认数据无误后，询问是否设置定时任务。
7. 定时任务只处理频率和时间，不重复展示消息操作栏已有的保存、分享能力。

交付结果：

- 可查看的报表模板。
- 可预览的数据结果。
- 可保存的定时任务。
- 历史 20 个报表和定时任务记录。

## 6. Agent 路由规则

| 用户表达 | 优先路由 | 说明 |
| --- | --- | --- |
| 投放前、联调、应用共享、能不能投 | 投放前质量保障 | 包含对接、包检查、自动联调 |
| 新媒体、新渠道、新对接方式 | 对接 | 对接完成后触发联调 |
| 看昨天有什么异常 | 监控巡检 | 宽泛异常先巡检，异常后转排查 |
| 某指标 A 和 B 不一致 | 数据排查 | 先调报表/配置/日志，再解释口径 |
| 为什么、区别、口径、什么意思 | 解答/指标解释 | 走知识库和指标解释器 |
| 创建日报、周报、拼表、模板 | 报表任务 | 不是监控任务 |
| 预测、创意、行为洞察 | P2 能力 | 本轮仅记录，不承诺交付 |
| 自动调预算、自动投放 | P3 能力 | 当前不进入执行 |

路由要求：

- 不允许只靠关键词直接进入下一步。
- 需要先完成意图推理、上下文准备、已有工具能力检查、传参完整性判断。
- 系统能获取的数据，不追问用户。
- 若当前能力缺失，第一句话就说明不能直接交付的原因和缺口。

| Skill | 服务方向 | 说明 |
| --- | --- | --- |

| `package_readiness_check_skill` | 包检查 | 检查包下载、版本、分包、过审状态 |
| `ads_health_monitor_skill` | 监控巡检 | 检查消耗、预算、授权、调度、入库、回传 |
| `metric_diff_diagnosis_skill` | 指标差异排查 | 差异数据、口径、链路定位 |

P1 应 Skill 化：

| Skill | 服务方向 | 说明 |
| --- | --- | --- |
| `metric_explainer_skill` | 指标解释 | 结构化解释指标定义、字段、链路、差异 |
| `report_template_builder_skill` | 报表模板 | 从截图、Excel、文本、页面需求提炼模板 |
| `scheduled_report_skill` | 定时报表 | 生成并维护日报、周报、月报任务 |
| `knowledge_answer_skill` | 规范解答 | 基于知识库、来源和系统上下文回答 |

暂不 Skill 化：

- 开放式经营策略分析。
- 高层复合经营判断。
- 动态探索式深度分析。
- 自动投放执行。

## 8. MCP 与后端能力需求

已知需要补齐：

- 分包功能改造。
- 获取包下载地址的 MCP。
- 媒体后台应用状态 MCP。
- 媒体过审状态 MCP。
- 数据上报检测 MCP。
- 报表明细与聚合查询 MCP。
- 数仓多指标、多维度查询 MCP。
- 回传延迟和成功率 MCP。
- 项目、用户、权限、历史记忆持久化。
- Agent 路由和提示词持久化管理。
- Skill 管理、启停、版本、输入输出 schema、调用日志。

数据原则：

- 能调真实 MCP 就不使用 mock。
- 查不到数据时说明“未取到数据”和缺失来源。
- 来源必须告诉用户取自哪张报表、哪个知识库、哪个工具或哪个网页。
- MCP 请求参数和返回参数必须与真实工具一致。

## 9. 前端组件需求

P0/P1 需要的关键组件：

- 项目选择组件：支持单项目、全部项目、跨项目分析；当前阶段不要多选交互误导。
- 思维链组件：展示意图识别、上下文准备、MCP/Skill/知识库调用、结果生成；支持展开收起、子步骤折叠、序号、用时、来源。
- 结构化追问表单：一次性收齐必要条件，确认后自动继续。
- 自动联调组件：会话中展示联调发起、步骤、截图、日志、轮询状态、结论；右侧栏展示执行详情。
- 指标解释器组件：动态交互，不全部平铺。
- 报表模板组件：类 Excel 表格确认模板。
- 数据预览组件：流式展示查询和汇总结果。
- 来源与调用组件：默认收起，展开展示知识库、MCP、Skill、网页来源。
- 右侧详情栏：展示来源、调用参数、联调步骤、报表任务详情，宽度不足时允许横向滚动，不遮挡内容。

## 10. 验收用例

### 解答类

| 用例 | 输入 | 预期 |
| --- | --- | --- |
| 指标差异解释 | 日周月报和 ROI 报表的注册数为什么有差异 | 路由到解答/指标解释，展示口径、计算方式、报表差异、来源 |
| 指标概念解释 | 注册设备数和注册数有什么区别 | 展示结构化指标解释器，不全部平铺 |
| 未知指标 | 这个指标我不知道是什么 | 若知识库有资料则解释；无资料则提示缺少指标解释资料 |

### 排查类

| 用例 | 输入 | 预期 |
| --- | --- | --- |
| ROI 下降 | 最近首日 ROI 下降了一倍 | 先拉取项目和时间范围内 ROI、消耗、回收、注册、付费数据，再定位主要贡献因子 |
| 付费数差异 | 媒体付费数 100 而智投是 3，是有问题吧 | 先比对媒体和智投付费口径与数据，再检查归因、上报、回传链路 |
| 自然量异常 | 自然量有 100 多但是 H5 游戏没开搜索，不应该有这么多自然量 | 检查自然量来源、归因规则、渠道配置、H5 入口和数据分流 |
| 宽泛异常 | 我想看看昨天投放有什么异常 | 路由到监控巡检，不先要求对比数据源；异常后自动转排查 |

### 联调类

| 用例 | 输入 | 预期 |
| --- | --- | --- |
| 巨量联调 | 巨量这边要立即联调，应用已经共享到 wuyanlan@dobest.com | 系统确认项目和媒体，检查条件，通过后展示自动联调组件 |
| 自动触发联调 | 新包检测到更新 | 系统自动检查包、过审、对接配置，触发联调并生成结论 |
| 查看联调记录 | 查看昨天怒焰三国杀巨量联调记录 | 展示联调步骤、截图、日志、成功/失败原因、复测入口 |

### 报表类

| 用例 | 输入 | 预期 |
| --- | --- | --- |
| 定时报表 | 帮我创建一个每天上午 10 点发送的广告消耗日报 | 路由到报表任务，确认模板、拉取预览、再确认定时任务 |
| 维度对比 | 看一下昨天巨量按项目维度的消耗和 ROI | 调用报表 MCP，返回表格和简短分析结论 |
| 模板报表 | 上传 Excel 模板，按这个格式生成周报 | 解析模板，校验指标，确认后拉取数据并生成报表 |

### 监控类

| 用例 | 输入 | 预期 |
| --- | --- | --- |
| 回传延迟监控 | 帮我监控巨量回传延迟，超过 30 分钟就告警 | 路由到监控任务，不误判为排查；配置项目、媒体、阈值、通知 |
| 投放异常巡检 | 昨天投放有什么异常 | 检查消耗、预算、授权、调度、入库、激活、注册、付费、回传 |

## 11. 当前缺口与实施顺序

### 立即优先

1. 固化 Agent 路由提示词和路由评测集。
2. 建立 P0 质量保障 Skill：包检查、自动联调、健康巡检、指标差异排查。
3. 接真实 MCP：项目、报表、包、媒体状态、回传、上报、权限。
4. 去掉 mock 替代真实结果的链路。
5. 完成思维链、来源、调用、结构化追问、联调组件的稳定展示。

### 第二优先

1. 建立指标解释器资料录入和动态组件。
2. 建立报表模板、数据预览、定时报表 Skill。
3. 完成历史记忆、用户上下文、会话间记忆。
4. 完成后台 Skill 管理、Prompt 管理、功能开关持久化。

### 暂缓

1. 新增指标和口径体系建设。
2. 预测、创意分析、行为洞察。
3. 自动投放。

## 12. 交付验收原则

每个需求验收时必须回答：

1. 是否解决了原始用户问题？
2. 是否直接交付了结果，而不是只给建议？
3. 系统能获取的信息是否已经自动获取？
4. 是否存在不必要的多轮追问？
5. 是否调用了真实 MCP/Skill/知识库？
6. 是否清楚展示了来源和证据？
7. 是否没有使用 mock 数据伪装真实结果？
8. 是否形成可复用 Skill 或可沉淀流程？

验收状态定义：

- `solved`：用户目标已闭环，结果可直接使用。
- `partial`：主要链路可用，但仍依赖人工补充或部分工具缺失。
- `unsolved`：只完成页面、文案或建议，未解决用户真实场景。
