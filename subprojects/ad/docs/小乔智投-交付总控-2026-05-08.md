## 0. 文档目的

这份文档只做三件事：

1. 固定小乔智投当前唯一有效的交付链
2. 说明哪些文档和页面可以作为 `aiocoding` 输入真源
3. 约束后续前端、后端、评审、联调必须遵守的交付口径

它不是补丁说明，也不是阶段汇报。

---

## 1. 当前唯一有效交付链

`调研文档 -> 规划文档 -> 需求文档 -> 功能文档 -> 设计文档 -> 前端页面 -> 数据表 -> 后端接口 -> 联调与验收`

明确废止：

- `前端页面（简单版）`
- `前端页面（UI/UX版）`
- 任何 `示意页 / 解释页 / 静态文档页` 口径

原因：

1. 两阶段前端口径会持续诱导产出平铺页和假页面
2. `aiocoding` 需要的不是页面草图，而是可交付、可验证、可联调的动态页面
3. 前端必须直接面向真实用户、真实任务、真实状态和真实结果结构

---

## 2. 前端页面交付标准

`前端页面` 阶段必须直接满足以下标准：

1. 布局正确
2. 功能模块划分合理
3. 用户主流程可走通
4. 页面具备真实动态交互
5. 页面基于正式业务对象和正式 contract，而不是静态说明文案
6. 页面能被设计评审、开发评审、联调验收直接消费

明确禁止：

1. `hero 大标题首屏`
2. `summary-card 四宫格工作台`
3. `explanation-first` 文案说明区主导页面
4. 玻璃拟态、运营卡片化、Claude 风格默认壳
5. 用平铺模块代替真实交互流
6. 把会话、任务、结果、运行验证、治理信息做成同层平铺首页
7. 把持续执行验证直接做成业务首页常驻主区

## 4. 当前产品与工程边界

### 4.1 当前用户问题

小乔智投要解决的是广告支持场景中的四类核心任务：

1. 使用帮助
2. 需求沟通
3. 问题排查
4. 广告联调

### 4.2 当前工程边界

当前阶段：

1. 用本地仓内 runtime 承接正式页面
2. 用正式 mock contract 承接真实业务对象
3. 不等待预发布数据库直连才开始交付
4. 先把页面、contract、联调闭环跑通，再切真实数据源

---

## 5. 当前必须强制生效的评审规则

所有关键阶段默认触发多角色评审，最少包括：

1. `Solution-Optimality Review`
2. `Development Review`
3. `Design Review`
4. `Research Review`
5. `Hermes Governance`
6. `Human Final Approval`

其中：

- 产品评审必须判断当前解法是不是当前阶段更优解
- 开发评审必须结合当前系统现状审开发 task
- 设计评审必须判断页面是否真的面向用户使用
- Hermes 必须给出 `keep / revise / block / promote`

---

## 6. 当前 `aiocoding` 前最关键的现实约束

进入 `aiocoding` 前，必须接受三个现实：

1. 当前最容易漂移的是前端页面，不是文档
2. 当前最容易返工的是结果区 schema，不是基础布局
3. 当前最该先做的是“低返工主链 + 高约束 contract”，不是一次性堆满所有细节

因此执行顺序是：

1. 先保证页面主工作台真的活起来
2. 再细化四类业务流结果 contract
3. 再补联调与验收

治理修正：

1. `页面主工作台真的活起来` 的含义是“用户进来先做事”，不是“信息模块全部可见”
2. 运行时验证、评审记录、压测证据属于二级承接面，不得反客为主决定首页结构
