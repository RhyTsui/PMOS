数据排查类 Agent 兜底规则设计（可直接落地进 PRD）
核心原则：只兜底、不硬扛；只推单、不转人；可界定、不泛滥。
兜底不是 “AI 不会就兜底”，而是明确哪些场景系统绝不处理、必须交给产品 / 数据同学。
一、兜底总规则（4 条铁律）
系统只负责可结构化、可复现、可口径化的排查
AI 不做跨领域推理、不做主观推断、不做未定义归因
兜底≠转接人工对话，只自动生成 Case 推产品
同类问题一旦被产品解决，就从兜底移除，进入规则库
二、明确可自动处理范围（反向定义兜底）
系统可以自己处理，不触发兜底的场景：
单指标 / 单计划 / 常规时间波动
可通过版位、素材、人群、时段下钻定位
有明显维度贡献（某素材 / 某版位占比突降）
有明确操作记录（启停、改价、换素材）
波动在阈值范围内，可正常解释
数据可在单张宽表内完成计算
不在上述范围内 → 进入兜底
三、标准兜底触发条件（必写死）
1. 数据层兜底（最常见）
数据口径冲突、指标定义不一致
数据缺失、延迟、明显异常但无法解释
多表关联、跨模块、跨数据源复杂计算
数据与业务认知严重不符，无法简单归因
2. 归因层兜底
多维交叉影响，无法定位主因
无任何维度下钻出现显著贡献
受大盘 / 政策 / 外部因素影响，系统无信息
根因涉及策略模型、算法逻辑、底层机制
3. 需求层兜底
需求模糊到无法结构化（“效果不好” 但无指标）
需求超出广告数据范畴
需求涉及历史久远数据 / 离线加工数据
需求需要人工经验判断、行业判断
4. 风险层兜底
连续多次追问同一问题，表达质疑
涉及大额消耗、线上事故、敏感问题
可能引发业务误判的高风险结论
5. 能力边界兜底
规则库无匹配场景
意图识别失败超过 2 次
系统无法获取必要查询权限
四、绝对不兜底的场景（避免滥用）
以下情况禁止兜底，必须系统自行给出结论：
信息完整、维度清晰的常规波动
有明显操作导致的异常
简单数据核对、昨日 / 今日对比
计划未开启、预算耗尽等明显问题
五、兜底执行流程（标准闭环）
满足任一兜底条件 → 触发兜底
停止继续分析，不再尝试解释
自动打包：
业务原始问题
已填字段（指标、时间、计划）
系统已做动作
无法处理原因
对话摘要
生成《数据排查 Case 通知单》推产品
对业务返回统一话术，不暴露内部逻辑
六、统一对外话术（固定不变）
您这个问题涉及较复杂的数据逻辑，我已把完整背景和排查情况整理成专项问题同步给产品同学进一步分析，有明确结论后会第一时间同步您。
七、兜底后的进化规则（关键）
产品处理完成后，必须回填结论
系统将该 Case 转为新规则 / 新样本
同类问题下次不再兜底，由 Agent 自动处理
每周统计兜底率，目标持续下降
八、极简版（可直接写进 PRD）
口径冲突、数据异常→兜底
无法下钻、无主因→兜底
多表复杂、跨模块→兜底
模糊需求、超范围→兜底
高风险、强质疑→兜底
兜底只推 Case 不转人
处理后回填进化规则
