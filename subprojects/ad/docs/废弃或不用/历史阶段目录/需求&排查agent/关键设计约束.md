数据排查 Agent 关键设计约束（PRD 专用）
一、交互设计约束
交互范式约束：必须采用「自然对话为主、固定菜单为辅」的混合范式，禁止纯对话或纯菜单设计；自然对话需支持业务模糊提问解析，固定菜单需包含「排查指标、时间范围、计划 / 版位、异常类型」四大核心快捷入口，不可缺失。
提问收口约束：无论业务提问方式如何，Agent 必须引导收集完整四大必填信息（排查指标、时间范围、关联计划 / 版位、异常表现），缺失任一信息不得执行排查，禁止跳过信息校验直接输出结果。
回复规范约束：所有排查结果必须以「一句话结论 + 波动数据 + 核心原因 + 优化建议」为固定结构，禁止仅输出数据、不做归因，禁止使用 “数据正常”“波动不大” 等甩锅式表述。
话术约束：统一使用预设柔性话术库，禁止 AI 自由生成话术；拒绝 / 兜底场景必须附带替代方案或安抚说明，禁止生硬拒绝、模糊答复。
二、数据能力约束
排查逻辑约束：固定排查链路为「整体波动判断→多维度下钻（版位→素材→人群→时段→大盘）→操作记录校验→根因归因」，禁止 AI 自由调整排查顺序，确保结果可复现。
数据口径约束：曝光、点击、转化、消耗、成本等核心指标的统计口径需提前固化至知识库，Agent 仅可按预设口径解释数据，禁止自创口径、混淆口径；出现口径疑问时，直接触发兜底流程，不得随意解读。
异常阈值约束：异常波动阈值需支持配置（默认阈值：波动＞20% 为异常，10%-20% 为轻微波动，＜10% 为正常），阈值调整需同步更新至规则引擎，禁止 Agent 无基准判断异常。
自动下钻约束：核心指标异常时，必须自动完成全维度下钻分析，识别贡献度最高的异常维度，禁止仅呈现整体数据、不做拆解。
三、决策与兜底约束
AI 权限约束：AI 仅负责自然语言理解、字段抽取、结果呈现与话术输出，核心决策（异常判定、归因逻辑、兜底触发）必须由规则引擎执行，禁止 AI 自主决策、自由联想、跨领域推理。
兜底规则约束：兜底仅执行「自动生成 Case 通知单→推送产品端→告知业务等待结果」流程，绝对禁止实时转接人工对话；兜底触发范围严格遵循预设规则（数据口径异常、多表关联复杂问题、无法定位根因等 5 类场景），禁止滥触发或遗漏。
Case 通知单约束：兜底 Case 通知单需包含「提交人、业务诉求、已处理动作、无法处理原因、对话摘要」五大核心内容，自动推送至产品指定端口，禁止信息缺失、延迟推送。
四、闭环与进化约束
输出规范约束：每一次排查完成（无论正常处理还是兜底），必须自动生成标准化《数据排查结论单》，内容固定、格式统一，禁止无单据输出、单据信息不全。
自进化约束：成功 Case（意图识别正确、结果准确、业务无异议、未触发兜底）必须自动沉淀为规则库、样本库，无需人工干预；产品处理后的兜底 Case，需支持结果回填，自动转化为新规则，用于系统迭代。
归档约束：所有对话记录、排查结果、Case 通知单、成功 Case 样本需结构化归档，支持按提交人、时间、排查指标检索，禁止数据丢失、无法追溯。
五、稳定性与上线约束
上线范围约束：MVP 版本仅覆盖高频排查场景（曝光突降、转化下滑、无消耗 / 不起量、成本异常），禁止初始上线全场景，降低风险。
回归测试约束：每次系统迭代后，必须执行回归测试，复用预设测试 Case 集，确保原有排查逻辑、规则不被破坏，老问题不复现。
自由度约束：严格限制 AI 回答范围，仅允许围绕「广告数据排查」相关内容响应，超出范围（如非广告数据、无关业务问题）直接触发兜底，禁止 AI 跨范围应答。
性能约束：常规排查场景（单计划、单指标、常规时间范围）需在 3 秒内输出结果，兜底 Case 通知单需在 10 秒内生成并推送，禁止延迟卡顿。
