# 广告项目月底 Demo 重规划

更新日期：2026-04-20

## 1. 重规划结论

广告项目是 2026 Q2 重点，但当前仓库实现仍停留在 `Ads Flow Insight` 单页漏斗原型，和现有 PRD、Demo 设计文档中的目标范围不一致。  
因此从 2026-04-20 起，项目不再按“大而全 Ad OS”推进，而是切换为 **Demo First、需求拆解同步收口** 的执行模式。

月底目标固定为：

1. 在 **2026-04-30** 前完成一套可演示、可讲故事、可验证方向的 Demo。
2. 在 **2026-04-30** 前完成 Demo 对应的需求任务拆解，形成后续 MVP 排期输入。

## 2. 当前判断

### 2.1 已有资产

现阶段仓库内已经具备以下输入：

1. 长期产品蓝图与 PRD。
2. 一份更贴近 Phase 1 的 Demo 文档。
3. FastAPI + React 的前后端原型工程。
4. 漏斗流转、用户列表、用户路径追踪等基础可视化能力。

### 2.2 当前缺口

距离月底 Demo 验证仍有三个核心缺口：

1. **目标不收敛**：长期 PRD覆盖对接、诊断、分析、素材、知识库等多个大模块，但月底无法全部验证。
2. **故事不闭环**：现有代码能看数据流，但还不能支撑“业务发问 -> Agent 执行 -> 结论输出”的演示路径。
3. **任务未落盘**：没有围绕 2026-04-30 反推的需求拆解、负责人视角任务包和验收口径。

## 3. 月底 Demo 目标

## 3.1 Demo 核心目标

月底 Demo 只验证三件事：

1. 用户是否接受“对话式广告问题处理台”这个产品方向。
2. 团队是否能在 Mock 数据下跑通一个最小业务闭环。
3. 后续 MVP 应该优先投在哪些能力，不再继续做无效铺量。

## 3.2 Demo 成功标准

到 **2026-04-30**，演示时必须能稳定完成以下闭环：

1. 用户进入首页，看到清晰的产品定位与可选操作入口。
2. 用户发起一个典型问题，系统能进入对应的处理流程。
3. 系统展示过程状态，而不是只返回静态结果。
4. 系统输出结构化结论、证据或分析卡片。
5. 演示全程基于 Mock 数据即可，不依赖真实生产环境。

## 4. 范围重切

## 4.1 月底必须做

月底只保留 3 个演示模块：

1. **诊断 Agent Demo**
   用户输入一个典型 Gap 问题，系统展示诊断路径、证据链和结论。
2. **对话式分析 Demo**
   用户输入一个对比分析请求，系统返回对比卡片或表格。
3. **总入口工作台**
   用统一首页或对话页把两个 Demo 串起来，形成产品感。

## 4.2 月底可选做

如果主闭环提前完成，再补以下增强项：

1. 对接/联调流程的静态演示版。
2. 更完整的日志流或步骤动画。
3. 更细的 Drill-down 视图。

## 4.3 月底明确不做

以下内容不进入 2026-04-30 Demo 范围：

1. 真实媒体 API 接入。
2. 真正的自动化联调执行。
3. 素材 Agent 全链路。
4. 知识库/规则变更监控系统化建设。
5. 大而全控制台、权限、审计、生产化治理。

## 5. Demo 场景定义

## 5.1 场景 A：异常诊断

建议固定演示话术：

`为什么昨天巨量激活比 BI 少 30%？`

系统应展示：

1. 识别为“激活 Gap 诊断”。
2. 展示 4 到 5 个固定诊断步骤。
3. 每一步返回 Mock 结果。
4. 最终输出结论、证据链、建议动作。

## 5.2 场景 B：对话式分析

建议固定演示话术：

`帮我对比素材 A 和素材 B 最近 7 天的消耗和 ROI。`

系统应展示：

1. 识别为“对比分析请求”。
2. 填充模板参数。
3. 返回对比卡片或表格。
4. 结果支持继续追问一轮，例如追加 CTR。

## 5.3 场景 C：工作台入口

工作台负责：

1. 告诉用户这不是漏斗图工具，而是广告问题处理工作台。
2. 提供诊断、分析两个入口。
3. 承接演示过程中的状态、结果和切换。

## 6. 需求任务拆解

## 6.1 P0：月底必须完成

### 产品与交互

1. 明确月底 Demo 统一故事线与演示脚本。
2. 明确首页信息架构与两个核心入口。
3. 明确诊断结果卡片和分析结果卡片的字段结构。

### 前端

1. 将现有漏斗页升级为 Demo 工作台，而不是仅展示 Sankey。
2. 增加问题输入区、快捷问题入口、模块切换入口。
3. 增加诊断执行过程展示组件。
4. 增加诊断结果卡片组件。
5. 增加分析结果卡片/表格组件。

### 后端

1. 增加统一的 Demo 场景接口，而不是只有 flow/users/trace。
2. 增加诊断场景 Mock API。
3. 增加分析场景 Mock API。
4. 统一响应结构，便于前端按场景渲染。

### 数据与内容

1. 固化诊断场景的 Mock 步骤数据、证据数据、结论数据。
2. 固化分析场景的 Mock 表格数据与追加问答数据。
3. 固化首页文案、模块说明、演示话术。

### 交付

1. 更新 README，写清楚如何启动和演示。
2. 形成一份 Demo 验证清单。
3. 形成一份月底后的 MVP 需求池输入。

## 6.2 P1：月底后进入 MVP 评估

1. 联调流程自动化。
2. 真正的 MCP 工具抽象。
3. 多诊断场景扩展。
4. 多分析模板扩展。
5. 素材 Agent 与知识库底座。

2. 界面是否体现“Agent 工作台”而不是“数据图表页”。
3. 用户是否能看懂系统做了什么、为什么得出这个结论。
4. 是否已经把 Demo 后续工作拆成明确任务，而不是继续停留在概念文档。
