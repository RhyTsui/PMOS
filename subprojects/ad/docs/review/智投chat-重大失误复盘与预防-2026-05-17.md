# 智投chat 重大失误复盘与预防

日期：2026-05-17

## 事件概述

今天早上已经完成过一版联调相关改动，但在后续调整中，我错误理解了“只做前端展示优化”的边界，把本应只改展示层的事情扩展到了数据链路和联调流程，导致联调结果需要重新构建和重新验证。

这不是单点 bug，而是一次**改动边界失控**。

## 直接后果

- 联调展示和联调数据链路出现了偏移。
- 原本已经完成的联调结果不能直接复用。
- 后续不得不重新走一遍联调流程。
- 用户对“改完又回退、回退又重做”的信任感会下降。

## 根因分析

### 1. 没先确认改动边界

我把“展示优化”误扩成了“展示 + 数据 + 默认值 + 流程调整”。

### 2. 没把高风险项单独拎出来

联调、报表、项目筛选、agent 路由这几类能力，本来就属于高风险区域，但我没有在动手前先做边界确认。

### 3. 改动前没有固定回退点

没有先保留旧逻辑作为可切换版本，导致一旦方向不对，回退成本变高。

### 4. 没把“只改前端”当作硬约束

在用户明确说“只改前端展示”的情况下，我仍然动到了数据源和结果组织方式，这属于执行失误。

## 需要固化的原则

### 原则 1：展示层和行为层必须分开

如果用户说的是“优化 UI 视觉”“调整展示格式”“只改前端样式”，默认只能动：

- 文案
- 布局
- 字号
- 颜色
- 排版
- 组件显示/隐藏

不能默认去动：

- 数据来源
- 轮询
- 默认值
- 结果摘要
- 任务状态
- 路由逻辑
- mock / 真源切换

### 原则 2：高风险链路先确认，再落代码

联调、报表、项目筛选、agent / skill 编排、监控、定时任务，这几条链路只要涉及下面任一项，必须先确认：

- 默认值
- 兜底策略
- 数据归属
- 轮询频率
- 结果输出口径
- 是否允许回退

### 原则 3：每次只动一个层

同一次修改里不要混：

- 前端展示
- 服务端路由
- 数据兜底
- 流程状态
- 结果摘要

一旦混在一起，后续很难判断问题到底是 UI 还是链路。

### 原则 4：先列清单，再实施

每次我先输出三块：

1. 这次会改什么
2. 这次不会改什么
3. 发现偏差后怎么回退

你确认后再动。

## 预防措施

### 1. 改动前强制写边界

今后凡是你让我改“联调 / 报表 / 项目筛选 / 排查”相关内容，我必须先明确：

- 是否只改前端
- 是否允许改数据链路
- 是否允许改默认值
- 是否允许改轮询

如果没确认，就不能直接开改。

### 2. 关键链路保留回退版本

对联调、报表、agent 路由这类模块，改动时保留旧版本逻辑，不先删。

建议做法：

- 先复制旧逻辑
- 只切展示分支或开关
- 通过后再清理旧路径

### 3. 先做最小影响改动

能只改一个文件，就不要顺手带上第二个层级。

### 4. 改完做行为检查

不能只看 build 通过，还要检查：

- 联调步骤有没有被压缩
- 思维链有没有被误改
- 正文结果有没有提前输出
- 报表有没有回样例数据
- 项目列表有没有回假兜底

### 5. 用户确认优先级提高

遇到下面这些词时，我必须先停一下：

- “只是优化一下”
- “先别动别的”
- “这次只改展示”
- “不要影响对接”

这些都说明当前请求的风险是“边界误扩”，不是“功能缺失”。

## 建议你以后直接对我说的约束句

为了减少再次失误，你后面可以直接用下面这些句式锁边界：

- “这次只改前端展示，不改数据流。”
- “先列出修改点，我确认后再动。”
- “不要碰默认值、轮询、结果口径。”
- “只动这个组件，别动其他链路。”
- “如果要改数据层，先停下来确认。”

## 结论

今天这次失误的核心不是技术复杂，而是执行纪律问题。

后续对联调、报表、项目筛选、agent 编排这几条主链路，我要按“先边界、后实施、先回退、再清理”的顺序推进，避免再发生“本来只想改展示，结果把链路打散”的情况。
