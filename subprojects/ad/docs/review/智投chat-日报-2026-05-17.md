## 三、今日关键判断

### 1. 不能把“展示优化”误做成“链路改造”

今天最大的共识是：  
很多问题表面看是 UI 或文案问题，实际一旦动到默认值、轮询、结果收敛、任务状态，就会变成链路问题。

所以后续必须先区分：

- 只改展示层
- 只改数据过滤
- 只改接口契约
- 只改提示词编排

不能混着改。

### 2. 产品需要按成功结果倒推

今天把 v0.2 的规划从“功能清单”往回收成了“结果倒推”：

- 先定义用户最终要拿到什么
- 再定义中间层怎么承接
- 再定义模块、接口、workflow
- 最后再落到具体团队分工

这个方法比顺着列任务更适合当前阶段。

### 3. 知识库不是文档仓库

今天对知识库的定义也更明确了：  
它不是资料堆放区，而是要参与：

- 意图识别
- skill 选择
- 过程解释
- 结果校验
- 失败建议

## 四、今日风险与反思

### 1. 高风险误改仍然存在

今天再次确认了一个问题：  
如果没有先确认“这次只动哪一层”，就很容易把项目选择器、联调流程、默认结果、提示词编排混在一起改，导致联调链路重新来一遍。

### 2. 产品与工程边界还在收紧中

目前 chat 的真实问题不是“缺功能”，而是：

- agent、skill、MCP、思维链、正文之间的职责没有彻底固定
- mock、提示词、代码逻辑、展示逻辑之前经常混在一起
- 一个模块的改动容易误伤别的模块

这件事必须继续治理。

### 3. 日报/蒸馏能力还需要产品化

今天的记录主要还是人工整理。  
后续要把“每日蒸馏 / checkpoint / 结果归档”变成固定机制，不然回顾成本还是会很高。
