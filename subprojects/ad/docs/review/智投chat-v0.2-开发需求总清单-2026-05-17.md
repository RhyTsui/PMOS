# 智投chat v0.2 开发需求总清单

## 目标

v0.2 的目标不是继续堆单点功能，而是把智投 chat 从“能对话、能联调”升级为“能进入任务、能解释过程、能交付结果、能持续回看”的完整工作台。

这份文档面向全团队，拆成五个责任面：

1. 后端接口需求清单
2. MCP 需求清单
3. 知识库建设需求清单
4. UI / 产品体验优化需求清单
5. 测试与 agent skill workflow 编排需求清单

## 一、总体规划

### 1. 产品主线

- 入口层：项目筛选、监控、定时任务、报表、联调、数据排查、项目包列表
- 过程层：思维链、任务步骤、子步骤、截图、日志、证据
- 结果层：成功结论、失败定位、可复用证据、下一步动作
- 回看层：历史任务、历史结果、联调记录、包记录、报表请求记录

### 2. 核心原则

- 用户正文只展示结论，不展示过程产物
- 过程信息放进思维链、组件详情、日志、证据里
- 真实数据优先，mock 仅限开发隔离层，不得进入用户结果
- 任何会影响任务状态、轮询、默认值、摘要结论的改动，都必须可回退

### 3. 迭代目标

- 能筛选项目，默认不自动选中
- 能稳定发起联调并持续观测
- 能稳定问报表并展开请求/返回
- 能把数据排查、系统监控、项目监控做成可编排 agent skill workflow
- 能让知识库支撑标准化判断，而不是只放文档

## 二、后端开发需求清单

### 1. Chat 基础接口

#### 必须补齐

- 会话创建、继续会话、会话标题维护
- 消息发送、流式返回、终态收敛
- 思维链步骤流、步骤更新、步骤终态
- 结果摘要返回
- 附件上传、附件解析、附件引用

#### 需要支持的能力

- 运行中状态与终态分离
- 失败时只返回最后失败步骤和原因
- 成功时只返回结论和证据
- 支持按会话承接项目、媒体、终端上下文

### 2. 项目筛选接口

- 支持按项目 ID / 名称关键字搜索
- 支持默认项目为空
- 支持过滤烽火工作室
- 支持过滤非游戏项目
- 支持返回项目类型、状态、图标、别名等用于前端展示的字段
- 不允许用假项目兜底替代真实结果

### 3. 联调接口

- 创建联调任务
- 获取任务详情
- 获取步骤列表
- 获取结果摘要
- 暂停任务
- 恢复任务
- 人工接管
- 删除任务

#### 联调返回要求

- 步骤必须可结构化解析成“步骤 / 子步骤 / 状态 / 截图 / 耗时”
- 不需要把英文辅助描述、传参、绝对时间戳暴露给主视图
- 轮询必须持续，直到步骤终态或任务终态
- 失败时能明确定位到最后一个失败步骤

### 4. 报表接口

- 报表问数时，能返回具体 tool 的请求和返回
- 支持展开 / 收起 tool 请求结果
- 生产报表接口与 mock 数据服务必须彻底分离
- 查询报表时，正文只输出结论，不输出示例数据

### 5. 监控 / 定时任务接口

- 系统监控、项目监控、定时任务必须有独立意图路由
- 不能误判成排查、报表或联调
- 需要返回任务状态、执行时间、结果摘要、异常原因

### 6. 证据与记录接口

- 支持任务证据列表
- 支持截图、日志、报表查询结果的统一引用
- 支持历史回看
- 支持按项目、任务、时间筛选

## 三、MCP 需求清单

### 1. 联调 MCP

#### 要求

- 能创建任务
- 能查询任务详情
- 能查询步骤
- 能查询结果摘要
- 能支持持续轮询

#### 输出约束

- 返回内容必须结构化
- 步骤内容必须可拆分
- 状态字段必须稳定
- 截图字段必须可识别
- 不要把工具内部字段当成用户结果直接输出

### 2. 报表 MCP

#### 要求

- 生产环境直连
- 支持真实 tool 请求与返回
- 取消 mock 数据回填
- 支持问数、维度解释、口径说明、异常定位

#### 输出约束

- 查询到什么就展示什么
- 不允许用假用户、假数据、假结果替代真实返回

### 3. 数据排查 MCP

#### 要求

- 支持数据排查流程中的固定 skill
- 支持结果证据回传
- 支持项目维度约束
- 支持排查结论摘要

### 4. 监控 MCP

- 支持报警识别
- 支持系统监控任务
- 支持项目监控任务
- 支持定时任务与临时任务的区分

### 5. 项目包 / 质量保障 MCP

- 支持项目下包列表查询
- 支持包可用状态查询
- 支持包联调记录查询
- 支持失败定位与状态追踪

## 四、知识库建设需求清单

> 这里面向全团队，不是单独运营。

### 1. 需要沉淀的知识类型

- 项目定义：项目、媒体、终端、包名、默认账号、分包规则
- 联调知识：步骤定义、失败模式、常见阻塞项、结果判定
- 报表知识：指标口径、维度解释、常见误差、查询注意事项
- 排查知识：问题分类、分析路径、证据样例、修复建议
- 监控知识：报警类型、触发规则、处理建议
- 视觉规范：项目筛选、联调组件、结果摘要、日志区、截图区

### 2. 知识库必须补齐的结构

- 场景
- 适用对象
- 输入条件
- 可调用能力
- 预期输出
- 失败分支
- 证据样例
- 常见误区
- 回退方式

### 3. 知识库使用原则

- 知识库不替代真实接口
- 知识库负责解释和规范，不负责造数
- 知识必须能被 agent 读到，也能被前端解释层引用
- 知识库内容要能按项目、媒体、终端、任务类型检索

### 4. 建议团队分工

- 产品：定义场景与优先级
- 开发：把知识映射到接口与 workflow
- 测试：补充真实样例、失败样例、边界样例
- 设计：补充展示规范和交互说明
- 业务/运营/排查使用方：补充场景、术语、典型案例

## 五、UI / 产品体验优化需求清单

### 1. 项目筛选

- 支持关键字搜索项目 ID / 名称
- 默认项目为空
- 项目列表过滤烽火工作室与非游戏项目
- 选择器必须轻量，不要做成传统后台筛选器

### 2. 思维链

- 只展示真实推进步骤
- 不堆英文描述、参数、无意义时间字段
- 需要能看出为什么进入联调、为什么进入排查、为什么进入报表
- 正文不输出过程产物，只输出结果

### 3. 联调组件

- 展示步骤 / 子步骤 / 状态 / 截图 / 耗时
- 截图有图才显示，无图不占位
- 步骤树要持续更新
- 失败时能看到最后失败步骤
- 成功时能看到结论和证据

### 4. 调用与来源 / 日志

- 日志应按时间线展示
- 代码块样式优先
- 日志不应和用户结果混在同一个层级

### 5. 报表体验

- 报表问数时允许展开 tool 请求与返回
- 正文只保留结论
- 去掉 mock 造数带来的信任损失

### 6. 包列表 / 可用状态

- 项目下应该能看包列表
- 能看联调状态
- 能看记录
- 能看可用状态

### 7. 监控 / 定时任务 / 排查

- 入口要分开
- 结果摘要要统一
- 不能让用户在多个入口之间来回猜

## 六、测试与 agent skill workflow 编排需求清单

### 1. 数据排查 agent

#### 需求

- 必须先绑定项目
- 必须固定几个核心 skill
- 思维链展示排查过程
- 排查组件展示分析过程
- 正文只返回结论

#### 需要覆盖的 workflow

- 先识别问题类型
- 再选 project / media / terminal 上下文
- 再调用 skill
- 再做证据汇总
- 最后输出结论

### 2. 系统监控 agent

- 监控类意图必须准确识别
- 不能误入定时任务
- 要能区分报警、巡检、定时任务、项目监控

### 3. 项目监控 agent

- 只允许在项目上下文内运行
- 要能给出项目级状态与异常

### 4. Skill 编排要求

- skill 固定且可解释
- skill 顺序必须稳定
- 异常时要有回退 skill 或明确阻塞项
- 不允许无限开放工具池

### 5. 测试重点

- 项目筛选搜索
- 项目默认空态
- 联调步骤持续轮询
- 联调截图可见性
- 报表真实 tool 返回
- mock 数据不回流
- 监控意图识别准确
- 排查必须先选项目

## 七、推荐优先级

### P0

- 项目筛选搜索与过滤
- 报表去 mock
- 联调结果结构化
- 监控 / 排查 / 报表意图分流

### P1

- 项目包列表与可用状态
- 知识库结构化
- agent skill workflow 编排

### P2

- 结果回看增强
- 历史证据归档
- 更细的 UI 微交互优化

## 八、回退要求

所有改动必须满足：

- 前端展示层可单独回退
- 接口层可单独回退
- MCP 接入可单独回退
- 提示词可单独回退

如果一个改动会影响：

- 联调任务创建
- 结果摘要
- 项目默认值
- 轮询行为
- 报表真实返回

就必须先记录，再改。

## 结论

v0.2 的核心不是再做一个聊天页面，而是把智投 chat 变成一个可承接任务、可解释过程、可交付结果、可回看证据的工作台。

这份清单可以直接作为后续拆分需求、分工、排期和验收的基础文档。
