今天不是一次普通页面修复，而是一次“AI 产品交付链路是否可靠”的集中暴露。

表面上完成了智投chat 首页、我的资产、思维链、自动联调、消息操作、资产预览、欢迎区、端口、发布归档等一组功能收口；更深层的问题是：AI 前端开发在没有强约束时，会自然回退到熟悉的后台模板、静态卡片、文本说明和局部补丁。用户不断指出的问题，本质上不是单个 UI bug，而是执行系统缺少稳定的产品约束、组件契约、验收门禁和每日蒸馏闭环。

今日最终状态：

- 本地可迁移备份已生成：`imported/zhitou-chat-product-20260513.bundle`

## 2. 原始目标与实际结果

### 原始目标

用户今天的核心目标并不是“多改几个页面细节”，而是：

1. 让智投chat 不再像传统后台，而像真正的 AI 会话工作台。
2. 让我的资产、消息区、思维链、自动联调成为产品能力，而不是静态展示。
3. 让项目可留档、可回滚、可继续迭代。
4. 建立机制，避免后续 Codex 继续回到 Ant Design admin dashboard 默认形态。

### 实际结果

已解决：

- 智投chat 项目完成独立 Git 提交并推送到 PMOS 归档分支。
- UI Guardrail 从“口头要求”升级为仓库规则和可执行检查。
- 首页和我的资产从传统后台/弹窗方向转向 chat-first 工作台方向。
- 自动联调流程被提升为右侧面板和会话流程中的产品能力。
- 消息操作、来源、思维链、Markdown、会话标题、欢迎区等关键体验已进入可继续验收状态。

部分解决：

- 思维链已经从“没有展示”推进到“消息内展示入口 + 流式事件基础”，但还没有完成行业级过程可视化。
- 自动联调前端已有工作台形态，但真实后端状态流、结果归档、人工接管还需要继续接入。
- 我的资产有可用前端形态，但真实文件存储、预览服务、缩略图生成、会话锚点仍未闭环。
- 每日蒸馏有事件记录，但没有自动脚本和固定产物机制。

未解决：

- iconfont 指定图标未形成稳定资源导入流程。
- 真实模型调用链路下的会话标题、缩略图、思维链细粒度披露还未完全落地。
- 远端发布权限检查没有前置，直到发布阶段才暴露。

## 3. 今日关键时间线

### 阶段一：页面形态偏差暴露

用户指出“我的资产页面没有按新设计规范和参考截图来画”，随后进一步指出“AI 一直在生成传统后台”。这是今天最关键的质量信号。

这说明当时的执行系统只把需求理解成“实现一个资产页”，没有先约束页面范式，也没有把参考截图转成组件契约。

结果：

- 页面虽然有功能，但产品语言和视觉结构不对。
- 用户要反复纠偏，开发才回到正确方向。
- 问题被定位为机制缺失，而不是单个页面缺陷。

### 阶段二：UI Guardrail 建立

后续补了 `AGENTS.md`、UI Guardrail、review rules、design constraints、component contract 的方向，并接入 `validate:ad-ui`。

这是今天最有价值的机制性进展：从“凭感觉做页面”改为“先有规则，再落页面”。

但它仍是初版：

- 对页面形态有拦截能力。
- 对文案工程黑话有一定拦截能力。
- 对具体视觉细节、动效、图标一致性、截图对齐仍不够自动化。

### 阶段三：多轮验收问题集中收口

用户密集提出了侧边栏、消息操作栏、来源、思维链、资产页、自动联调、端口、logo、新会话、移动端、欢迎语、滚动条、按钮 hover 等问题。

这些问题看似分散，实质分成五类：

- 视觉语言不一致：圆角、hover、图标、logo、滚动条、按钮状态。
- 会话产品能力不足：思维链、来源、消息操作、会话标题、长消息折叠。
- 工作台能力不足：自动联调、排查记录、需求跟踪、右侧面板。
- 资产能力不足：引用、预览、下载、删除、缩略图、定位会话。
- 工程交付约束不足：端口漂移、发布归档、真实后端接入边界。

结果是前端逐步从“页面集合”转成“会话主线 + 右侧能力面板 + 资产服务”的产品结构。

这个过程说明发布路径没有在任务开始时前置验证。虽然最终解决，但中间浪费了时间，也造成“是不是没推上去”的信息不确定。

### 阶段五：日报与每日蒸馏检查

今天的日报可以基于 `mcp-context events` 手工蒸馏出来，说明事件采集层是有效的。

但没有发现：

- 固定日报命令

- 自动读取事件并生成复盘的脚本
- 每日结束时自动 checkpoint + 文档产出

所以每日蒸馏机制只能判定为“记录层生效，产出层未生效”。

## 4. 根因分析

### 根因一：需求被当成页面清单，而不是产品系统

用户一直在表达的是产品形态：

- 像 ChatGPT / DeepSeek Chat 的会话体验
- 资产要像服务页面，不是海报
- 自动联调要作为流程，不是文本建议
- 思维链要展示真实过程，不是一个标题

执行过程一开始更容易把它降级成页面功能清单：

- 加一个按钮
- 加一个弹窗
- 加一个列表
- 加一个卡片

这导致页面有功能但不像产品。

修正方向：

- 每个用户需求必须映射到产品能力，而不是直接映射到组件。
- 例如“点击来源没有披露知识库地址”不是“加来源按钮”，而是“可信回答需要证据披露协议”。
- “自动联调不是文本建议”不是“加一个面板”，而是“需求路由 -> 表单 -> 校验 -> 联调 -> 观测 -> 结果”的流程契约。

### 根因二：参考截图没有转成组件契约

用户多次给截图、对标产品、图标链接和视觉要求。问题在于截图如果只作为灵感，AI 会抽象得过度松散。

需要把截图转成明确契约：

- 布局：左右结构、顶部区域、消息区宽度、资产列表密度。
- 状态：hover、active、collapsed、drawer、empty、selected。
- 行为：点击后是否跳转、是否展开、是否收起、是否保留会话。
- 组件：使用 Ant Design X 的哪些组件，哪些自研组件承接。
- 禁止项：不要卡片墙、不要传统后台、不要海报页。

今天后半段建立了 Guardrail，但还需要把“截图到 contract”的步骤前置成默认流程。

### 根因三：思维链被误判为前端展示问题

用户反复提“思维链没有展示”“没有动态效果”。这不是单纯前端组件没放出来。

完整思维链至少包含四层：

1. 后端事件协议：模型推理、知识库检索、工具调用、浏览器动作、MCP 调用、参数、返回、耗时。
2. 前端状态机：pending、running、success、failed、collapsed、expanded、redacted。
3. 展示规范：什么给用户看，什么给管理员看，什么只进审计日志。
4. 安全边界：不能暴露模型私密推理，但可以展示可审计的工作过程。

今天只做到了初级形态：

- 有思考步骤。
- 有工具调用事件。
- 有消息内承载。
- 有默认收起。

还没做到成熟形态：

- 过程分层不够。
- 参数和返回没有分权限展示。
- 时间轴和状态动效不完整。
- 管理后台没有策略配置。

### 根因四：自动联调缺少端到端状态协议

自动联调的用户预期是“我能看到联调全过程并接管”，不是“页面上有自动联调模块”。

今天前端根据文档接入了工作台和观测面板，但仍主要是前端承载层。

真正需要的是：

- 任务对象：任务 ID、媒体、终端、账号、环境、目标。
- 状态机：待确认、待执行、执行中、需人工接管、已暂停、已失败、已完成、已归档。
- 步骤流：每一步的截图、日志、参数、结果、耗时。
- 事件流：后端通过 SSE/WebSocket 或轮询让前端增量展示。
- 结果对象：是否通过、失败原因、证据、可复用 Case、下一步建议。

没有这些协议，前端只能做“像在运行”的界面，而不是可靠的联调产品。

### 根因五：发布动作没有前置权限探测

今天发布时才发现：

- 当前工具不能创建 GitHub/GitLab 新仓库。
- 本机没有 `gh` / `gitlab` CLI。
- 有些远端 URL 不可访问。
- SSH 未完成校验。

正确做法应该在任务开始时就探测：

- 当前项目是否已有独立 Git 仓库。
- 远端有哪些。
- 哪个远端可 push。
- 如果不可 push，是否先生成 bundle。
- 目标是新库、现有仓库分支、还是压缩归档。

今天最后成功推到 PMOS 归档分支，但这个成功依赖临场排查，不是流程保障。

### 根因六：每日蒸馏只有数据源，没有产品化闭环

`mcp-context events` 是有效的数据源，但每日蒸馏不是“能查到事件”就算完成。

每日蒸馏需要完整闭环：

- 收集：读取当天事件、提交、验证、用户反馈。
- 归类：完成、未完成、风险、经验、明日计划。
- 判断：哪些是用户问题已解决，哪些只是产品机制部分完成。
- 输出：写到固定目录。
- 回写：生成 checkpoint，必要时生成待办。
- 可追踪：次日可以从待办继续推进。

今天缺的是后五步自动化。

## 6. 当前风险

### 高风险

- 思维链如果继续只做 UI，会无法满足可信 AI 工作台的核心要求。
- 自动联调如果没有真实状态协议，会变成演示面板，难以进入实际使用。
- 每日蒸馏不自动化，后续长链路工作仍会靠人工记忆，容易丢失上下文。

### 中风险

- 图标、圆角、hover、滚动条等视觉规范没有形成 token 级约束，仍可能反复偏。
- 我的资产如果不接真实文件链路，会停留在前端样板。
- 管理后台如果不承接思维链策略、提示词、模型和工具披露，会影响上线后的可控性。

### 低风险

- 项目已归档，短期丢失风险降低。
- 端口和构建验证已多次通过，演示可用性风险下降。

## 7. 机制改进方案

### 7.1 UI Guardrail 继续升级

现状：

- 已能拦截部分传统后台和工程黑话。

下一步：

- 增加“页面形态检查”：是否存在营销 hero、卡片墙、后台 dashboard 套壳。
- 增加“组件契约检查”：新增业务页必须有 screenType、componentBindings、evidence/source、action contract。
- 增加“截图转契约模板”：用户上传参考图后，先形成布局/状态/行为契约，再改代码。
- 增加“视觉 token 检查”：圆角、hover 色、滚动条、按钮尺寸、侧栏宽度。

### 7.2 思维链协议化

建议新增 `AgentProcessEvent` 标准事件：

```ts
type AgentProcessEvent =
  | { type: "route"; label: string; targetAgent: string; durationMs?: number }
  | { type: "knowledge_search"; query: string; sources: SourceRef[]; durationMs?: number }
  | { type: "tool_call"; toolName: string; params: unknown; durationMs?: number }
  | { type: "tool_result"; toolName: string; resultSummary: string; rawRef?: string; durationMs?: number }
  | { type: "browser_action"; action: string; target: string; resultSummary?: string; durationMs?: number }
  | { type: "model_step"; label: string; summary: string; durationMs?: number }
  | { type: "handoff"; from: string; to: string; reason: string; durationMs?: number };
```

展示分层：

- 普通用户：过程摘要、来源、工具名称、耗时、结论。
- 管理员：参数摘要、返回摘要、失败信息、重试记录。
- 审计日志：原始参数和完整返回，按权限查看。

### 7.3 自动联调状态流

建议把自动联调前端从“页面组件”升级为“任务运行时”：

- `DebugTask`
- `DebugStep`
- `DebugEvidence`
- `DebugResult`
- `ManualTakeover`
- `CaseArchive`

前端只订阅状态流，不自行假造流程。

### 7.4 每日蒸馏自动化

建议新增命令：

```bash
npm run daily:distill -- --date 2026-05-13
```

输入：

- `mcp-context events`
- 当日 git commit / branch / diff
- 验证命令结果
- 用户反馈
- 未完成任务列表

输出：

- `docs/review/daily/2026-05-13.md`
- `docs/review/daily/2026-05-13.action-items.json`
- mcp-context checkpoint

自动分类：

- `solved`
- `partial`
- `unsolved`
- `risk`
- `follow_up`
- `mechanism_gap`

## 9. 高质量需求提炼

下面不是用户原话搬运，而是把今天多轮反馈提炼成可进入需求池、设计、开发和验收的需求。

### R-CHAT-001 会话工作台体验一致性

- 优先级：P0
- 用户问题：当前会话界面容易退回传统后台或局部拼装，交互语言不稳定。
- 产品目标：智投chat 应作为 chat-first 工作台，主界面围绕会话、过程、证据、动作组织，而不是后台 dashboard。
- 核心范围：
  - 侧边栏、顶部栏、消息区、输入框、右侧面板形成统一布局。
  - 宽屏使用页面内侧栏；窄屏使用浮层抽屉。
  - 新会话、搜索、分享、更多、用户菜单行为统一。
  - 所有 hover、active、展开收起动效遵循同一 token。
- 验收标准：
  - 首屏不出现后台统计卡片墙、营销 hero、海报风资产页。
  - 宽屏点击新会话不主动折叠侧栏。
  - 窄屏抽屉打开后显示 logo，关闭按钮使用浮层语义。
  - 端口固定为 `8002`，浏览器访问不转圈。
- 当前状态：partial

### R-CHAT-002 消息操作与长消息阅读体验

- 优先级：P0
- 用户问题：消息操作按钮过多、顺序不对，用户消息和系统消息操作不应相同；长消息影响多轮阅读。
- 产品目标：不同角色消息有不同操作集合，保证常用动作直接可见，低频动作收纳。
- 核心范围：
  - 用户消息：复制、编辑。
  - 助手消息默认展示：复制、重新生成、喜欢、不喜欢、语音播报。
  - 更多中展示：来源、保存到个人知识库、分享到小闪、收起。
  - 长消息默认折叠，支持展开和返回底部。
  - 来源、摘要、思维链都属于消息载体内部，不应散落到消息外。
- 验收标准：
  - 用户消息不会出现重新生成、来源、喜欢等助手动作。
  - 助手消息按钮顺序和命名稳定。
  - 超长内容不会阻断后续多轮对话浏览。

## 11. 自我进化与实质效果

这一节专门回答：今天有没有从“被动改 bug”进化到“下次少犯同类错误”。

### 11.1 已发生的自我进化

#### 从口头偏好进化为执行规则

原始问题：

- 用户指出页面一直像传统后台。
- 用户指出“不要 Ant Design admin dashboard 默认形态”。

进化动作：

- 把“不要传统后台”写入 `AGENTS.md` 和 UI Guardrail。
- 明确 PMOS / 智投chat 的产品身份是工作流驱动的 AI 会话工作台。
- 将禁止项从审美描述变成可执行约束：不要 hero、poster card、feature grid、dashboard 套壳、工程黑话。

实质效果：

- 后续新增或修改页面时，不再只靠记忆用户偏好。
- Codex 进入仓库后会优先读取这些规则，降低回退到默认后台模板的概率。
- `validate:ad-ui` 已开始承担第一层自动拦截。

#### 从截图参考进化为组件契约意识

原始问题：

- 用户提供参考截图后，页面仍然偏成弹窗、卡片或海报风。

进化动作：

- 复盘中明确“截图不能只作为灵感，必须转成布局、状态、行为、组件、禁止项契约”。
- 我的资产页被重新定位为资产服务页，而不是展示页。
- 自动联调被重新定位为过程工作台，而不是文案回复。

实质效果：

- 后续遇到截图时，第一步应产出 screen contract，而不是直接写 React。
- 资产页、联调页、会话页可以复用同一套“会话主线 + 右侧能力面板 + 证据/动作”的结构判断。

#### 从功能补丁进化为产品能力分层

原始问题：

- 来源、思维链、自动联调、资产引用都被当成局部按钮或局部模块处理。

进化动作：

- 把来源提升为“可信回答证据披露”。
- 把思维链提升为“可审计过程流”。
- 把自动联调提升为“任务状态机 + 过程观测 + 人工接管”。
- 把我的资产提升为“会话产物管理与再利用服务”。

实质效果：

- 待办不再是零散 UI 修改，而是可以进入需求池的高质量需求。
- 今天新增的 `R-CHAT-003`、`R-CHAT-004`、`R-ASSET-001`、`R-REQ-001` 能直接指导后续设计和开发。

#### 从人工记忆进化为事件回放

原始问题：

- 长时间多轮迭代后，容易忘记为什么改、改到哪、还有什么没闭环。

进化动作：

- 使用 `mcp-context events` 回放今天关键 checkpoint。
- 生成日报和深度复盘文档。

实质效果：

- 今天的复盘不是凭聊天记忆写的，而是基于事件记录、提交归档和用户反馈。

### 11.4 判断标准：明天如何证明真的进化了

明天不能只看“又改了几个 UI 点”，要用下面标准判断自我进化是否真的生效：

1. 新需求是否先进入需求提炼和组件契约，再进入代码。
2. 修改页面前是否自动经过 UI Guardrail。
3. 思维链是否按统一事件协议返回，而不是临时拼字段。
4. 自动联调是否由真实任务状态驱动，而不是静态模拟。
5. 每日结束是否能一条命令生成日报、复盘和待办。
6. 发布前是否自动检查远端权限和归档路径。

如果这六条有三条以上落地，说明今天的经验真正变成了能力；如果只是继续手工修 UI，说明复盘没有产生系统收益。

## 13. 新增高质量需求：连弩评测接口

### R-EVAL-001 智投chat 对外评测适配接口

- 用户需求：连弩测试平台需要智投项目提供接口，用于自动化评测。
- 产品需求：智投chat 需要提供稳定的评测适配层，让外部平台可以拉取用例、发起评测、拿到回答、过程事件、来源证据和基础评分。
- 当前状态：partial。
- 已落地：
  - `/api/xiaoqiao/evaluation/health`：评测适配层健康检查和能力声明。
  - `/api/xiaoqiao/evaluation/cases`：返回基础评测用例。
  - `/api/xiaoqiao/evaluation/run`：执行单条评测，返回答案、过程事件、来源和评分。
  - `src/lib/evaluation-adapter.ts`：把评测契约从 route 中抽离，便于后续接真实模型、知识库和自动联调任务。

## 12. 对今天执行方式的判断

今天最大的进步不是页面多做了多少，而是暴露并开始修复了执行系统的问题：

- 从“直接实现用户点名功能”转向“先抽象产品能力”。
- 从“AI 自由设计页面”转向“UI Guardrail + component contract”。
- 从“完成后口头总结”转向“事件记录 + 日报 + 深度复盘”。
- 从“本地可跑”转向“归档分支 + bundle 备份”。

但今天也说明，机制还没有足够硬：

- Guardrail 还不够早。
- 蒸馏还不够自动。
- 发布探测还不够前置。
- 思维链和自动联调还不够协议化。

如果明天只继续改 UI 细节，这个项目会继续反复。如果明天先把思维链协议、自动联调状态流和每日蒸馏命令补上，后面的页面迭代会稳定很多。
