# 智投chat v0.2 模块分工与策略版

## 先说结论

v0.2 不是一个“单体功能升级”，而是一个多角色协作项目。  
不同模块必须由不同开发人员负责，文档也必须按“模块负责人 + 依赖关系 + 业务结果”来写，而不是按技术清单堆在一起。

这版文档的目标是让负责人看完后能直接拆人、拆期、拆验收。

## 一、产品策略

### 1. 智投 chat 要解决的核心问题

智投 chat 不是纯问答工具，而是一个面向业务交付的工作台。它要解决的是：

- 项目筛选和上下文切换成本高
- 报表问数、联调、排查、监控之间切换成本高
- 过程信息散在思维链、日志、结果、截图里，难以承接
- 业务结果和技术过程混在一起，用户看不到明确结论
- 真实能力、mock、提示词、规则路由混在一起，容易误伤链路

### 2. v0.2 的产品方向

v0.2 的方向不是“把 chat 做得更像聊天”，而是：

- 让用户先进入正确的任务入口
- 让系统把任务拆成可解释的过程
- 让每个过程有证据、有状态、有结果
- 让失败能定位，成功能交付，历史能回看

### 3. 业务上优先级最高的结果

从业务上看，v0.2 需要优先保证四件事：

1. 能准确进入正确场景
2. 能持续拿到真实进度
3. 能把过程结构化展示
4. 能把结果和证据完整交付

如果这四件事没有闭环，其他体验优化都只是表面改善。

## 二、模块拆分原则

### 1. 不同模块由不同开发人员负责

这不是一个“混合改”的项目，必须按模块分工。  
每个模块都有自己的输入、输出、回退点和验收标准。

### 2. 模块之间只通过清晰的契约连接

每个模块都要回答三件事：

- 输入是什么
- 输出是什么
- 失败时怎么退回

### 3. 先做能闭环的，不做只是“看起来完整”的

优先做业务闭环，不优先做装饰型功能。  
例如：

- 项目筛选先解决“选对项目”
- 联调先解决“看见真实步骤”
- 报表先解决“看见真实请求和真实结果”
- 排查先解决“看见过程和结论分层”

### 4. 人力分配方式不固定

这份方案不绑定固定人数，也不要求某个模块必须由某一个人独占。  
实际执行时按以下方式分配：

- 能独立闭环的模块，可以单人负责
- 需要接口和前端协同的模块，按契约拆给不同人
- 需要真实数据和 UI 同步推进的模块，允许并行
- 高风险模块必须先定边界，再开工

文档里只保留：
- 模块职责
- 输入输出
- 依赖关系
- 优先级
- 验收标准

不写死“第几个人做什么”，避免和实际资源调度冲突。

## 三、模块负责人拆分

## A. 后端负责人

### 职责范围

- Chat 会话基础能力
- 项目筛选服务
- 联调任务状态机
- 报表查询与结果摘要
- 证据与历史回看
- 结果最终收敛

### 业务目标

- 让 chat 有真实可用的数据源
- 让任务、结果、证据能够稳定串起来
- 让后端成为“结果可信”的来源

### 具体需求

#### 1. Chat 基础链路

- 会话创建与继续
- 流式消息返回
- 思维链步骤流
- 结果摘要
- 附件解析与引用

#### 2. 项目筛选

- 支持项目 ID / 名称搜索
- 默认项目为空
- 过滤烽火工作室
- 过滤非游戏项目
- 不回填假项目

#### 3. 联调链路

- 创建任务
- 查询任务详情
- 查询步骤
- 查询结果
- 暂停、恢复、接管
- 步骤必须支持结构化输出

#### 4. 报表链路

- 真实报表查询
- tool 请求与返回可展开
- 移除 mock 数据回流

#### 5. 结果收敛

- 成功只给结论和证据
- 失败只给最后失败步骤和原因
- 不把过程产物灌进正文

### 验收重点

- 项目列表真实可筛
- 联调任务可持续观测
- 报表返回真实结果
- 失败定位准确

## B. MCP 负责人

### 职责范围

- 联调 MCP
- 报表 MCP
- 监控 MCP
- 数据排查相关 MCP
- 项目包 / 可用状态相关 MCP

### 业务目标

- 让 chat 能拿到真实业务事实
- 让联调、报表、监控、排查不再依赖伪造数据
- 让返回结构适配前端的过程展示

### 具体需求

#### 1. 联调 MCP

- 任务创建、详情、步骤、结果摘要
- 步骤结构化输出
- 支持持续轮询
- 支持终态识别

#### 2. 报表 MCP

- 生产环境真实返回
- tool 请求和返回完整
- 不允许 mock 混入用户结果

#### 3. 监控 MCP

- 支持系统监控
- 支持项目监控
- 支持报警识别
- 支持定时任务区分

#### 4. 排查 MCP

- 支持固定 skill 调用
- 支持证据返回
- 支持项目约束

#### 5. 包与状态 MCP

- 包列表
- 包可用状态
- 包联调记录

### 验收重点

- 返回结构稳定
- 关键字段可解析
- 不靠前端猜字段
- 能持续提供真实状态

## C. 知识库负责人

### 职责范围

- 统一业务口径
- 固化规则、案例、失败模式
- 让 agent 和前端展示都能引用同一套知识

### 业务目标

- 让系统回答得一致
- 让团队对同一问题有统一解释
- 让排查和联调不依赖个人经验

### 需要建设的知识

- 项目、媒体、终端、包名关系
- 联调步骤与常见阻塞
- 报表指标口径
- 排查路径与典型案例
- 监控报警分类
- UI 视觉和流程规范

### 验收重点

- 知识能被 agent 使用
- 知识能被解释层引用
- 知识不替代真实数据

### 知识库的应用方式

知识库不是单纯“存资料”，而是要进入实际链路。建议按四种方式使用：

#### 1. 预判与路由

- 用户输入后，先用知识库判断这是报表、联调、排查、监控，还是项目筛选
- 用知识库里的场景定义辅助 agent 选对 skill
- 用知识库里的约束项判断是否必须先选项目、先选媒体、先选终端

#### 2. 过程解释

- 当 agent 进入某一步时，用知识库解释“为什么要做这一步”
- 思维链展示的，不是原始规则，而是知识驱动下的业务解释
- 比如“为什么要先查默认账号”“为什么要先看包状态”“为什么要先拉项目上下文”

#### 3. 结果校验

- skill 或 MCP 返回后，用知识库校验这次返回是否符合业务口径
- 如果结果不符合知识库中的常见模式，系统应提示阻塞项或异常项
- 例如报表口径冲突、联调步骤缺失、监控报警误识别，都可以通过知识库做二次约束

#### 4. 失败归因与建议

- 当任务失败时，知识库负责给出“这类问题通常卡在哪里”
- 不是替代真实失败原因，而是补充排查方向
- 让最终结果可以给出下一步建议，而不是只报失败

### 知识库在系统中的位置

推荐把知识库放在四个位置：

- Agent 前：用于意图识别和路由
- Skill 中：用于流程约束和条件判断
- 思维链侧：用于业务解释
- 结果层：用于失败建议和下一步动作

### 不建议的用法

- 不要用知识库直接伪造结果
- 不要让知识库替代 MCP 的真实返回
- 不要把知识库写成纯文档仓库，必须让它服务 workflow
- 不要让知识库内容和 UI 文案完全绑定，避免维护困难

### 需要补齐的知识字段

每条知识建议至少包含：

- 场景类型
- 适用项目 / 媒体 / 终端
- 前置条件
- 可调用能力
- 常见阻塞点
- 成功判定
- 失败判定
- 下一步建议
- 证据样例

## D. UI / 产品设计负责人

### 职责范围

- 项目筛选
- 思维链展示
- 联调组件
- 日志与证据
- 报表展示
- 结果摘要

### 业务目标

- 让用户一眼看懂当前在做什么
- 让过程和结果分层
- 让失败和成功都能快速理解

### 具体需求

#### 1. 项目筛选

- 支持搜索
- 默认空选
- 列表过滤明确

#### 2. 思维链

- 只保留有业务意义的步骤
- 不展示工程黑话
- 不展示无意义参数和时间戳

#### 3. 联调组件

- 步骤 / 子步骤 / 状态 / 截图 / 耗时
- 有图才展示图
- 失败定位清晰

#### 4. 日志与证据

- 日志与主步骤分层
- 代码块式可读
- 截图可预览

#### 5. 结果摘要

- 成功：结论 + 证据
- 失败：最后失败步骤 + 原因

### 验收重点

- 真实用户能看懂
- 不像研发说明页
- 能在移动端和桌面端正常工作

## E. 测试负责人

### 职责范围

- 数据排查 agent skill workflow
- 系统监控 workflow
- 项目监控 workflow
- 联调流程
- 报表流程
- 项目筛选流程

### 业务目标

- 验证流程不是跑通一次，而是稳定可复现
- 验证意图路由不会串线
- 验证结果交付是可信的

### 测试重点

#### 1. 数据排查

- 必须先选项目
- 思维链展示过程
- 组件展示分析
- 正文只出结论

#### 2. 系统监控 / 项目监控

- 不会误识别成定时任务
- 不会误识别成排查

#### 3. 联调

- 步骤持续轮询
- 截图正常
- 失败步骤准确
- 最终结果准确

#### 4. 报表

- tool 请求与返回可见
- 不出现 mock 假数据

#### 5. 项目筛选

- 搜索可用
- 默认空选
- 过滤规则正确

### 验收重点

- 每个 workflow 都有输入、过程、输出和回退
- 不只看 build 通过，要看业务行为对不对

## 四、团队协作顺序

### 第一阶段：先补入口和真实数据

- 项目筛选
- 报表真实返回
- 联调真实步骤
- 监控 / 排查路由分流

### 第二阶段：再补编排和分层展示

- agent skill workflow
- 思维链分层
- 结果摘要层
- 日志层

### 第三阶段：最后补知识和复用

- 知识库
- 失败样例
- 证据样例
- 归档和回看

## 五、回退原则

所有模块都必须能单独回退。

建议回退顺序：

1. UI 展示层
2. 结果摘要层
3. 路由和状态机
4. MCP 接入
5. 知识库和提示词

如果一个改动会影响联调结果、项目默认值、轮询行为、报表真实值，就必须先登记再改。

## 六、给管理层的简版结论

v0.2 的本质不是“再加几个功能”，而是把智投 chat 从单一问答提升为多角色协作的任务工作台。

要先把模块分开：

- 后端负责真实数据和状态机
- MCP 负责真实能力输出
- 知识库负责统一口径
- UI/设计负责过程和结果分层
- 测试负责 workflow 稳定性

只有模块拆开，团队才能并行推进，产品也才不会在一次修改里被误伤。

## 七、产品成功定义

### 1. 什么叫成功

智投 chat 的成功，不是“功能变多了”，也不是“页面更完整了”，而是用户在关键业务场景里能更快、更稳、更少沟通成本地拿到可执行结果。

成功至少要满足四个层面：

#### 场景成功

- 用户能在正确的入口进入正确的任务
- 项目、媒体、终端、包、监控、排查、联调这些入口不会混
- 用户不用先理解系统内部结构，直接就能开始做事

#### 过程成功

- 思维链能解释为什么走这条路径
- 联调 / 排查 / 报表过程能持续更新
- 过程里每一步都有状态、有证据、有回退可能
- 不会因为一次重构或一次误改，把整条链路打断

#### 结果成功

- 成功时能给出明确结论
- 失败时能给出最后失败步骤和原因
- 结果正文不堆过程、不堆技术字段
- 用户看完可以直接决定下一步

#### 组织成功

- 后端、MCP、知识库、前端、测试都知道自己的边界
- 不再靠口头同步去拼接一个完整结果
- 每个模块可以单独推进，也可以单独回退

### 2. 成功不是单点指标

这类产品不能只看一次请求是否成功，要看一整条链：

- 意图是否识别正确
- 入口是否选对
- 过程是否持续拿到真实数据
- 展示是否让用户看懂
- 结论是否足够可执行
- 失败时是否还能给出方向

只要这条链上任何一环错位，产品就不能算成功。

## 八、成功策略

### 1. 策略一：先保证“选对场景”

最优先不是展示有多好看，而是“系统先别走错路”。

具体做法：

- 项目筛选默认空选，避免误入错误上下文
- 报表、联调、排查、监控、定时任务的意图路由分开
- 数据排查必须绑定项目
- 不让用户一开始就陷入内部概念

### 2. 策略二：先保证“拿到真实事实”

产品是否可信，取决于真实数据是否能稳定回来。

具体做法：

- 报表必须去掉 mock 回流
- 联调必须持续轮询真实步骤
- 监控和排查必须接真实状态
- 项目包列表必须来自真实项目域

### 3. 策略三：把过程和结果分层

用户需要知道过程，但不应该被过程淹没。

分层原则：

- 思维链负责解释
- 联调组件负责步骤和证据
- 日志负责技术细节
- 正文只负责结果

### 4. 策略四：让知识库成为“解释和约束层”

知识库不是仓库，而是行为约束。

它要参与：

- 意图识别
- skill 选择
- 过程解释
- 失败归因
- 下一步建议

### 5. 策略五：优先做闭环，不优先做炫技

如果一个功能不能帮助用户完成任务、理解任务或复盘任务，就不应该优先。

所以 v0.2 里的优先级应该是：

1. 选对场景
2. 拿到真实事实
3. 讲清过程
4. 给出结论
5. 再做复盘和沉淀

### 6. 策略六：所有模块都要支持回退

因为这类产品的错误往往不是单点 bug，而是链路误伤。

所以每个模块都要有：

- 独立回退点
- 独立验收标准
- 独立负责人
- 独立对外输出契约

### 7. 策略七：用业务指标衡量，而不是只看完成度

建议后续评估时不要只看“做了多少页面、接了多少接口”，而是看：

- 用户能否更快选对项目
- 联调是否更快定位到最后失败步骤
- 报表是否减少来回追问
- 排查是否减少无效沟通
- 监控是否减少误路由
- 结果是否更容易直接交付

### 8. 策略八：产品最终目标

智投 chat 的最终目标是：

- 让用户少做判断
- 让系统多做编排
- 让过程有证据
- 让结论可交付
- 让失败可定位
- 让成功可复用

## 九、倒推法：从成功结果反推需求

### 1. 先定义终点

倒推时，不要先问“要加什么功能”，而先问：

- 用户最后要拿到什么
- 哪些结果是可交付的
- 哪些证据能支撑这个结果
- 哪些失败情况必须明确暴露

对智投 chat 来说，终点通常是四类：

- 成功结论
- 失败定位
- 可复用证据
- 下一步动作

### 2. 再定义中间层

终点确定后，再往回拆中间层：

#### 结果层

- 正文只保留结论
- 成功 / 失败各有明确输出格式

#### 证据层

- 截图
- 日志
- tool 请求 / 返回
- 步骤状态

#### 过程层

- 思维链
- 联调步骤
- 排查分析
- 报表查询过程

#### 决策层

- Agent 判断意图
- Skill 组织动作
- MCP 提供真实事实
- 知识库提供约束和解释

### 3. 再往回倒到模块

如果终点是“结论 + 证据 + 下一步动作”，那模块就必须这样拆：

- 后端：提供真实数据和状态机
- MCP：提供事实和结构化返回
- 知识库：提供规则和解释
- UI：把过程、证据、结果分层展示
- 测试：验证链路有没有稳定闭环

### 4. 再往回倒到接口

如果模块要成立，接口必须满足：

- 项目可筛选
- 任务可创建
- 步骤可轮询
- 结果可摘要
- 证据可引用
- 失败可定位

### 5. 再往回倒到 workflow

如果接口具备了，workflow 还要满足：

- 先识别意图
- 再选项目
- 再选 skill
- 再调 MCP
- 再拿步骤和结果
- 再展示给用户
- 最后把结论收口到正文

### 6. 倒推后得到的产品原则

倒推以后会得到几个很明确的原则：

- 不能先做页面，再补逻辑
- 不能先做提示词，再补数据契约
- 不能先做 mock，再想办法替换真实结果
- 不能先把结果写进正文，再想怎么分层
- 不能把一个模块的错误放到别的模块里兜底

### 7. 倒推的价值

这个方法的价值在于：

- 能避免做成“看起来功能很多，但无法交付”的系统
- 能避免把 UI、提示词、接口、知识库混在一起改
- 能让每个模块都围绕最终结果倒着对齐
