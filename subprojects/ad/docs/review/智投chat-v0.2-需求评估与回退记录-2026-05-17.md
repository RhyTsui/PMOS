# 智投chat v0.2 需求评估与回退记录

日期：2026-05-17

## 背景

当前 v0.1 已打通会话、思维链、联调、报表、排查等主要交互，但从数据层看，仍缺少几条真正可闭环的主流程：

- 项目筛选与项目上下文
- 报表问数与报表编排
- 联调结果与步骤摘要
- 项目包列表、包可用状态、包联调记录
- 监控、定时任务、排查、联调之间的统一入口
- agent / skill 的稳定编排

这份记录只做评估，不把未确认范围提前写成定案。

## 我认同的方向

### 1. 项目筛选
- 支持按项目 ID / 名称搜索。
- 项目列表默认不选中任何项目。
- 项目列表需要屏蔽不该进入当前工作台的项目类型。
- 这条属于入口层的基础能力，应该尽早补齐。

### 2. 报表
- 报表问数时，思维链需要展示具体的 tool 请求与返回。
- 报表结果里不应该再出现样例姓名、样例数据或其他示例结果。
- 报表编排不是单次问答，应该能形成可复用流程。

### 3. 联调
- 联调应该有结果摘要层。
- 成功时输出结论和可复用证据。
- 失败时直接落最后一个失败步骤和原因。
- 步骤展示应以明确的中文步骤为主，英文辅助字段不进入主视图。

### 4. 质量保障
- 项目下需要能看到包列表、包联调状态、包联调记录、可用状态。
- 这部分属于“数据闭环”，不是装饰性入口。

### 5. agent
- 数据排查 agent 需要固定几个核心 skill，而不是无限开放工具池。
- 数据排查必须先选项目，再进入后续流程。
- 监控、定时任务、数据排查要做明确分流，避免误判。

## 我部分认同、但要等明早确认的点

### 1. 入口层补全
- 我认同要补监控、定时任务、报表编排、项目包列表、可用状态这些入口。
- 但这些入口应该挂在统一工作台里，而不是拆成传统后台。
- 明早需要再确认每个入口的优先级，以及它们各自依赖哪条真实数据源。

### 2. agent / skill 编排
- 我认同需要一层明确的编排设计。
- 但这层不是先做成很重的可视化编排器，而是先把“路由规则、固定 skill 组合、结果回写”跑稳。
- 真正的编排面是否开放给用户，明早再定。

### 3. 项目包列表
- 我认同这块要补，但现在还缺少最终的数据归属定义。
- 需要确认包列表、包状态、包记录分别由哪个服务提供，以及是否先做只读视图。

## 我不认同现在直接推进的点

- 不把这些能力拆成多个孤立页面。
- 不把工作台做回传统后台模板。
- 不再回填样例数据、样例模板、样例项目作为默认结果。
- 不在没有数据归属确认前，先做一套重编排的前端壳子。

## 已先处理的小优化项

为了避免继续污染真实数据，我先做了两个低风险修正：

1. 项目列表接口不再返回假兜底项目。
2. 报表模板读取不再回样例模板，空文件时只返回空列表。

这两项都只改了数据兜底，不影响当前页面结构。
