# 智投chat 职责边界拆分与调整建议

## 背景

当前智投 chat 是在多轮对话、临时代码、提示词约束、联调数据和少量 mock 逻辑叠加的基础上逐步长出来的。  
这类实现能快速跑通，但边界不清时，容易出现“改一个展示点，连带改了流程”“改一个兜底，误伤了联调链路”的问题。

这份文档的目标是把现有实现拆成四类，明确哪些该保留，哪些该收敛，哪些适合在 v0.2 调整。

## 四类职责

### 1. 写死的内容

这类内容是产品约定或默认文案，适合短期保留，但要控制比例。

典型例子：
- 默认标题、空态文案、按钮文案
- 思维链中的固定承接语
- 项目选择器中的默认空值“请选择一个项目”
- 联调/排查/报表结果里的固定结论模板

建议：
- 保留用户可理解的默认文案
- 任何带工程味的固定句子尽量改成用户语言
- 固定步骤要少，用于承接流程，不要替代真实结果

### 2. Mock 逻辑

Mock 只应该存在于开发、测试或临时兜底，不应作为用户结果输出。

当前需要重点收敛的地方：
- 报表 mock 数据
- 联调流程里的假结果分支
- 项目列表兜底假项目

建议：
- 用户结果中不再输出 mock 内容
- mock 只保留在隔离测试层
- 一旦真实 MCP 或真实数据可用，就把 mock 入口收掉

### 3. 提示词逻辑

提示词负责“如何说、先说什么、结果怎么组织”，不负责真实数据生成。

适合放在提示词里的内容：
- 思维链展示顺序
- 结果摘要结构
- 失败时只输出最后失败步骤
- 成功时只输出结论与证据
- 排查、联调、报表的过程组织方式

不适合放在提示词里的内容：
- 真实项目列表
- 真实联调步骤
- 报表数值
- 任何会被用户当成事实的数据

建议：
- 提示词只管编排，不管造数
- 任何数据源变化都应在代码或服务层解决

### 4. 代码逻辑

代码逻辑负责真实数据接入、过滤、状态流转、轮询和回退。

当前应保留并增强的部分：
- 项目列表接口：支持关键字搜索、过滤烽火工作室、过滤非游戏项目
- 项目选择器：默认空选、可搜索、可回退
- 联调任务：持续轮询、步骤解析、截图展示
- 报表：真实 MCP 请求与返回
- 结果摘要：成功/失败分流

建议：
- 能在代码里解决的，不要堆进提示词
- 能在服务层解决的，不要写死到前端
- 任何影响流程的改动必须能回退

## 当前建议调整项

### 项目选择器

已确认要做的调整：
- 支持按项目 ID / 名称搜索
- 项目列表展示全部项目，但过滤烽火工作室与非游戏项目
- 默认项目为空，不自动兜底一个假项目

建议继续保留的实现方式：
- 选择器前端加关键字输入
- 后端接口按 keyword 做查询与过滤
- 搜索无结果时显示空态，不回填假数据

### 报表

建议：
- 删除 mock 数据服务输出
- 思维链里保留报表 MCP tool 请求与返回的可展开内容
- 正文只保留最终结论

### 联调

建议：
- 只保留真实步骤树、子步骤、状态、截图
- 去掉时间戳、英文辅助描述和传参
- 成功/失败结果单独摘要，不污染正文

### Agent / Skill

建议：
- 先绑定项目，再进入排查
- 固定几个核心 skill，不做无限工具池
- 监控、定时任务、报表、联调、排查按意图区分，不互相串线

## 回退建议

这类改动以后尽量按下面顺序回退：

1. 先回退前端展示层
2. 再回退数据过滤逻辑
3. 最后才回退提示词编排

如果一个改动会影响：
- 联调步骤加载
- 报表真实值
- 项目默认值
- 轮询行为

就必须先做记录，再改。

## 结论

当前 chat 的问题不是“能力不够”，而是“职责混在一起”。  
v0.2 的关键不是继续堆功能，而是把：
- 写死的默认值
- mock
- 提示词编排
- 代码真实逻辑

四层拆开，分别治理。
