## 0. 设计定位

这不是一份“画几个页面”的说明，而是小乔智投交付级前端页面的正式设计真源。

它回答四件事：

1. 用户如何进入和使用页面
2. 页面如何组织业务对象和交互
3. 页面依赖哪些状态、数据和结果结构
4. 前端实现必须遵守哪些视觉与组件规范

---

## 1. 产品界面类型

小乔智投属于：

`数据 / 广告 / 企业系统类 AI 工作台`

因此页面必须同时具备：

1. 企业系统清晰度
2. 高信息密度承载能力
3. 会话式入口的自然性
4. 结果与证据并列展示能力

不允许退化成：

1. 纯聊天页面
2. 营销官网式页面
3. 平铺文档页
4. 卡片拼盘式假工作台

## 2. 默认组件与样式基线

### 2.1 组件基线

1. 企业系统基础组件：`Ant Design`
2. 会话组件：`Ant Design X`
3. 前端分层与开源基座边界：见 `docs/小乔智投-前端分层与开源基座方案-2026-05-09.md`
4. Ant Design X 官方设计与实现体系作为默认设计规范：见 `docs/小乔智投-Ant-Design-X默认规范-2026-05-09.md`

### 2.2 字体规范

默认字体栈：

```css
font-family: "Inter", "PingFang SC", "Microsoft YaHei", "Noto Sans SC", "Segoe UI", sans-serif;
```

建议层级：

1. 页面主标题：24px / 32px / 600
2. 区块标题：18px / 28px / 600
3. 正文主字号：14px / 22px / 400
4. 对话和关键说明：16px / 24px / 400
5. 辅助信息：12px / 20px / 400

### 2.3 圆角规范

1. 按钮/输入框：8px
2. 常规卡片：12px
3. 抽屉/弹窗：16px
4. 标签/状态 pill：999px

### 2.4 间距规范

1. 行内元素：8px
2. 组件内间距：12px
3. 卡片内边距：16px 或 20px
4. 模块间距：24px
5. 页面区块间距：32px

### 2.5 颜色方向

方向：

1. 冷静蓝作为主色
2. 青绿作辅助
3. 浅灰蓝作为页面底色
4. 风险与错误使用标准语义色

## 3. 页面信息架构

第一阶段保留五个一级入口：

1. 主工作台
2. 监控大屏
3. 素材分析
4. 广告预测
5. 应用支持

其中：

1. 主工作台为默认首页
2. 其余四项为专项页
3. 核心业务链路先在主工作台承接

### 3.1 页面层级治理

1. 默认首屏只有一个主操作：发起并推进当前任务
2. `主工作台` 的默认结构优先服务当前任务，不承担“完整系统导览”的职责
3. `历史任务` 默认以抽屉、侧板或二级列表承接，不与主操作区平级常驻
4. `持续执行验证`、运行证据、治理记录默认作为二级视图或专项页承接，不直接占用业务首屏主区
5. `结果区` 是当前任务的检查器，负责结论、证据和下一步动作，不负责长段说明

## 3.2 用户文案规则

本文件可以定义页面结构、模块边界、状态、交互和结果承接方式，但不能把工程实现语言直接作为用户可见文案输出。

面向用户的页面标题、分区名、按钮文案、空态文案必须使用产品语言，不得直接复用以下工程词汇：

1. 子项目
2. 聚合
3. 主链
4. 全局状态
5. 任务回看
6. 接口
7. contract
8. schema
9. mock
10. workspace（当其含义是内部接口名时）

页面文案必须优先回答三件事：

1. 这是什么产品或能力
2. 当前用户可以做什么
3. 用户下一步应该怎么继续

## 4. 主工作台布局

主工作台采用“三层主视图 + 两个次级承接面”：

1. 顶部身份与系统状态区
2. 中部当前任务工作区
3. 右侧结果与证据检查器
4. 历史任务抽屉或侧板
5. 运行验证与恢复入口

### 4.1 顶部身份与系统状态区

必须包含：

1. 小乔名称与角色说明
2. 当前系统状态
3. 当前会话状态
4. 新对话入口
5. 管理页入口

### 4.2 中部当前任务工作区

必须包含：

1. Welcome/Prompts 区
2. 消息流
3. Sender 输入区
4. 附件上传与解析入口

### 4.3 右侧结果与证据区

必须包含：

1. 当前任务摘要
2. 结构化结果卡组
3. 证据与来源卡组
4. 下一步动作区

### 4.4 历史任务与上下文辅助区

必须包含：

1. 历史任务列表
2. 当前上下文摘要
3. 缺失字段提示
4. 快捷跳转与恢复入口

补充约束：

1. 该区域默认不作为首页平铺常驻大区块
2. 允许用抽屉、侧板、折叠区承接
3. 展开后服务的是“继续当前任务”而不是“浏览系统说明”

### 4.5 运行验证与恢复入口

必须满足：

1. 可以查看当前运行状态、恢复动作和关键证据
2. 默认不替代业务主工作区
3. 不把运行治理信息写成用户说明文案

## 5. 关键交互流程

### 5.1 使用帮助

流程：

1. 用户提出问题
2. 系统判断是否为帮助场景
3. 页面展示答案、来源、可信度
4. 如信息不足，追问或引导升级

### 5.2 需求沟通

流程：

1. 用户描述需求
2. 系统抽取字段并识别缺失项
3. 页面展示结构化需求单与缺失字段
4. 用户补录后推进下一步

### 5.3 问题排查

流程：

1. 用户描述异常
2. 系统形成排查任务
3. 页面展示结论、证据链、风险和建议动作
4. 用户可继续追问或转交

### 5.4 广告联调

流程：

1. 用户发起联调
2. 系统进入联调状态机
3. 页面展示阶段、日志、截图、失败节点
4. 输出验收结论或人工接管动作

---

## 6. 页面状态与异常态

每个核心页面必须显式设计：

1. 空态
2. 加载态
3. 处理中
4. 待补录
5. 成功态
6. 错误态
7. 降级态

错误态必须同时给出：

1. 错误原因概述
2. 是否可重试
3. 下一步动作

---

## 7. 前端页面交付 gate

页面只有同时满足以下条件才算通过：

1. 布局结构正确
2. 功能模块责任清楚
3. 用户流程能走通
4. 动态交互真实成立
5. 状态/异常态齐全
6. 结果区与证据区不是假数据说明板

不通过时不能进入联调与验收。

---

## 8. 与后端 contract 的绑定要求

前端页面必须绑定正式对象，不允许重新发明一套页面假结构。

页面至少要与以下对象对齐：

1. `WorkspaceResponse`
2. `Conversation`
3. `Task`
4. `WorkflowResult`
5. 四类业务流结果 contract
