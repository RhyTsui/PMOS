# 小乔智投-前置文档评审委员会清单-2026-05-08

- 版本：v0.4
- 状态：active
- 日期：2026-05-09
- 适用范围：小乔智投在 `ad` 子项目中的交付级评审

---

## 0. 评审目的

这轮评审不再只检查文档有没有写全，而是检查：

`基于当前系统现状，这套方案是不是当前阶段更优解，能不能被拆成可执行开发 task，并最终落成面向用户的真实前端页面。`

---

## 1. 默认评审链

每次关键产物完成后，默认按下面顺序评审：

`stage output -> self-check -> multi-role review committee -> Hermes governance -> human final approval`

其中人只做最终裁决，不做人肉 lint。

---

## 2. 委员会角色

### 2.1 Solution-Optimality Review

职责：

1. 判断当前方案是不是当前阶段更优解
2. 判断有没有更轻、更稳、更低成本的替代解
3. 判断是否存在过度设计

必看维度：

1. 问题-方案匹配
2. 替代方案比较
3. 版本阶段适配
4. 收益/成本比
5. 系统协同影响
6. 可演进性

### 2.2 Development Review

职责：

1. 结合当前系统现状评审这次开发任务是否被正确理解
2. 判断任务拆解是否足够支撑前后端开发
3. 判断改动路径、风险和影响面

必看维度：

1. 当前系统相关现状
2. 本次开发任务清单
3. 涉及文件/模块
4. 新增/修改/兼容点
5. 前端任务
6. 后端任务
7. 数据任务
8. 测试任务
9. 风险与回归面
10. 是否可直接进入开发

### 2.3 Design Review

职责：

1. 判断页面是否面向真实用户使用
2. 判断布局、模块、状态、交互是否成立
3. 判断页面是不是交付级页面，而不是示意页

强制检查：

1. 布局正确性
2. 模块责任边界
3. 用户主流程
4. 动态交互成立
5. 状态与异常态
6. 字体、圆角、间距、组件规范是否显式

### 2.4 Research Review

职责：

1. 检查系统现状 grounding 是否充分
2. 检查知识来源、业务来源、历史约束是否被正确引用
3. 检查是否与 Dataki / 版本知识库相冲突

### 2.5 Delivery Review

职责：

1. 判断是否能进入联调与验收
2. 判断前后端是否已经拿到同一套 truth source
3. 判断是否还存在多文档冲突、补丁文档漂移、缺字段阻塞

### 2.6 Hermes Governance

职责：

1. 汇总 committee 发现的问题
2. 输出 `keep / revise / block / promote`
3. 决定哪些问题要变成整改任务、哪些要回写默认规则

---

## 3. 评审对象必须拆解的粒度

委员会不能只看整份文档，必须拆到以下单元：

1. 业务对象
2. 核心流程
3. 关键状态
4. 页面块
5. 接口组
6. 异常流
7. 权限点
8. 指标口径
9. 开发 task

---

## 4. 当前小乔智投业务评审重点

当前必须重点审四类业务流：

1. 使用帮助
2. 需求沟通
3. 问题排查
4. 广告联调

每类业务流都要回答：

1. 用户如何进入
2. 页面如何承接
3. 系统要追问哪些字段
4. 结果结构长什么样
5. 哪些动作要进入任务化
6. 哪些动作要进入联调与验收

---

## 5. 当前前端页面专门 gate

`前端页面` 阶段必须额外过以下 gate：

1. 不是文档页
2. 不是示意页
3. 不是平铺模块页
4. 不是 Claude 风格工作台壳
5. 页面能驱动真实用户流程
6. 页面能落到现有或明确计划中的 contract

不通过时默认回退到：

- `设计文档`
- `功能文档`
- `后端接口`

以问题根因所在阶段为准。

---

## 6. 当前结论

当前这份委员会机制相对旧版已经解决了：

- 只看结构不看业务：`solved`
- 只看文档不看 task：`solved`
- 没有专业角色分工：`solved`
- Hermes 不进评审：`solved`

还没完全解决的是：

- 跨项目默认继承：`partial`
- 专业 agent 自动强制激活全覆盖：`partial`
