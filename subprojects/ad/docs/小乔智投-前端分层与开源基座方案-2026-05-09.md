## 0. 文档目标

这份文档只解决一件事：

把小乔智投前端的开源基座组合、职责边界、默认落地方式定成一套可执行方案，避免 coding 阶段反复摇摆“到底用哪个框架、谁负责哪层”。

---

## 1. 结论

小乔智投前端采用以下组合：

1. `React`
2. `Ant Design`
3. `Ant Design X`
4. `CopilotKit`
5. `Zustand`
6. `React Query`

这不是六套平级框架共同主导页面，而是分层组合：

1. `Ant Design + Ant Design X` 负责工作台 UI 与会话表现层
2. `CopilotKit` 负责 agent 驱动交互编排层
3. `Zustand` 负责前端本地交互状态
4. `React Query` 负责服务端数据拉取、缓存、失效与同步

---

## 2. 总体原则

### 2.1 前端主体不是聊天框，而是业务工作台

小乔智投不是通用 AI chat 产品，而是广告问题排查、需求沟通、联调协同工作台。

因此：

1. 会话区只是入口，不是产品终点
2. `Task`
3. `WorkflowResult`
4. 结构化结果区
5. 历史任务区

才是产品真正的主承接层。

补充治理边界：

1. 这里定义的是主承接层，不是要求它们在首页同层平铺常驻
2. 首页默认仍然只能围绕一个主操作推进当前任务
3. 历史任务区默认用抽屉、侧板或二级视图承接

### 2.2 不做双核心会话状态

前端不能同时让：

1. `Ant Design X`
2. `CopilotKit`
3. 自定义组件

各自维护一套独立消息/任务状态。

必须始终由本项目自己的业务对象作为事实源：

1. `Conversation`
2. `Message`
3. `RoutingDecision`
4. `Task`
5. `TaskContext`
6. `WorkflowResult`

### 2.3 优先沉淀结构化结果，而不是继续美化聊天壳

当前阶段前端最重要的差异化能力不是“更像 ChatGPT”，而是：

1. `Conversation + Task + Result` 跑通
2. 把结果区做成插件式 `Structured Result Renderer`
3. 根据 `result_type` 稳定渲染不同业务结果

## 3. 各层职责边界

### 3.1 Ant Design

负责：

1. 页面骨架
2. 表单
3. 卡片
4. 抽屉
5. Table / List / Tabs / Tag / Descriptions
6. 状态反馈

适用区域：

1. 主工作台框架
2. 结果区卡片
3. 任务区
4. 管理页
5. 补录表单

### 3.2 Ant Design X

负责：

1. `Bubble`
2. `Conversations`
3. `Sender`
4. `Prompts`
5. `Attachments`
6. `Sources`

适用区域：

1. 对话消息流
2. 输入区
3. 对话列表
4. 会话提示词入口

明确要求：

1. `Ant Design X` 是会话 UI 基座
2. 它不负责业务任务状态定义
3. 它不负责结果区产品语义

### 3.3 CopilotKit

负责：

1. agent 驱动步骤推进
2. human-in-the-loop 交互
3. AI 触发表单补录
4. AI 驱动下一步操作区
5. AI 与前端共享临时任务推进态

适用场景：

1. 缺失字段补录
2. agent 决定进入哪个任务步骤
3. 诊断/联调过程中的步骤式推进
4. 用户确认后继续执行

明确限制：

1. `CopilotKit` 不是主视觉框架
2. `CopilotKit` 不接管整页布局
3. `CopilotKit` 不单独定义产品级消息模型
4. `CopilotKit` 不替代本项目 `Task / WorkflowResult`

### 3.4 Zustand

负责本地 UI 状态：

1. 当前选中的会话
2. 当前选中的任务
3. 右侧结果区展开/折叠
4. 抽屉开关
5. 草稿输入
6. 当前临时过滤条件
7. 本地 UI 偏好

不负责：

1. 服务端数据缓存
2. 持久化任务真源
3. 业务对象长期存储

### 3.5 React Query

负责服务端数据同步：

1. `conversations`
2. `conversation detail`
3. `tasks`
4. `task detail`
5. `task results`
6. `workspace/home aggregate`
7. prompt 管理数据

职责包括：

1. 请求生命周期
2. loading/error/success
3. 缓存
4. query invalidation
5. mutation 后刷新

明确要求：

1. 所有后端聚合视图、列表、详情默认走 `React Query`
2. 不再把这些请求散落进页面组件里各自手写

## 4. 主工作台固定分工

### 4.1 会话区

默认组合：

1. `Ant Design X Conversations`
2. `Ant Design X Bubble`
3. `Ant Design X Sender`
4. `React Query` 拉取/提交数据
5. `Zustand` 维护当前选中会话与草稿态

### 4.2 结果区

默认组合：

1. `Ant Design` 做结构化布局
2. `Structured Result Renderer` 做业务结果渲染
3. `React Query` 拉取 `TaskDetail + Results`
4. `CopilotKit` 只在需要 agent 推进时接管局部交互

补充约束：

1. 结果区是当前任务检查器，不是系统说明面板
2. 不承接运行治理总览，不承接首页导览文案

### 4.3 历史任务区

默认组合：

1. `Ant Design List / Table / Drawer`
2. `React Query` 拉取任务列表
3. `Zustand` 维护当前筛选与选中项

补充约束：

1. 默认以 `Drawer` 或二级面板承接
2. 不作为首页常驻平铺主模块

### 4.4 补录与推进区

默认组合：

1. `Ant Design Form`
2. `CopilotKit`
3. `Zustand`

适用点：

1. 缺失字段补录
2. 用户确认后继续执行
3. agent 给出下一步操作建议

## 5. Structured Result Renderer 作为当前前端最高优先线

当前最优先的前端产品化方向，不是继续更换聊天框架，而是做：

`Structured Result Renderer`

建议第一批最少包含：

1. `HelpResultRenderer`
2. `DemandResultRenderer`
3. `DiagnosisResultRenderer`
4. `DebugResultRenderer`

建议固定分发方式：

1. `help_answer` -> `HelpResultRenderer`
2. `demand_form` -> `DemandResultRenderer`
3. `diagnosis_report` -> `DiagnosisResultRenderer`
4. `debugging_report` -> `DebugResultRenderer`

建议固定接口：

1. `result_type`
2. `summary`
3. `structured_payload_json`
4. `confidence`
5. `next_action`

---

## 6. 当前阶段不建议做的事

1. 不为了“聊天体验更像通用 AI 产品”去重做整页
2. 不把 `CopilotKit` 当成整页 UI 框架
3. 不同时维护两套消息状态容器
4. 不把所有前端状态都塞进 `Zustand`
5. 不继续让结果区只展示通用 JSON
6. 不把开源基座讨论拖成开放性长期议题

---

## 7. 建议落地顺序

1. 用 `Ant Design X` 收口会话区正式组件
2. 引入 `React Query` 承接 conversations/tasks/results 请求
3. 用 `Zustand` 收口当前选中会话/任务/面板状态
4. 做 `Structured Result Renderer`
5. 在缺失字段补录和步骤推进场景引入 `CopilotKit`
6. 再扩管理页、专项页和更强 agent 化交互
