# 产品结构草图

## 1. 当前结构判断

连弩不适合继续按“文档中心 / 规则中心 / 执行中心”这种偏后台系统的方式组织产品。

如果要做成接近 `Coze Studio` 的形态，产品结构必须先从“模块堆叠”改成“工作台 + 资产 + 运行态”的结构。

原因很明确：

1. 用户进入产品时，首先感知的不是技术分层，而是“我现在在管理什么工作对象”
2. AI 测试平台的核心不是静态配置，而是围绕 `样本 -> suite -> run -> trace -> report -> compare` 的连续工作流
3. 如果页面继续按底层能力拆，界面会天然变成后台感，而不是工作台感

## 2. 当前版产品骨架

当前版建议固定为 `1 个 Studio 外壳 + 6 个核心工作台 + 1 个横向治理面`。

### 2.1 Studio 外壳

外壳负责统一承接：

1. 工作区切换
2. 环境切换
3. 全局搜索
4. 当前对象上下文
5. 最近运行与最近报告入口

它更像 `Coze Studio` 的 workspace shell，而不是传统后台首页。

### 2.2 六个核心工作台

#### 1. Case Studio

负责测试样本资产。

承接：

1. Case 列表
2. Case 详情
3. 对象类型筛选
4. 风险标签
5. 预期结果
6. 评估器绑定

#### 2. Suite Studio

负责评测集资产。

承接：

1. Baseline suites
2. Regression suites
3. Scenario suites
4. Suite 成员管理
5. Suite 与版本绑定

#### 3. Run Studio

负责运行态。

承接：

1. 新建 run
2. 运行队列
3. 运行详情
4. 多 trial
5. 重跑失败样本

#### 4. Trace Studio

负责结果下钻与问题定位。

承接：

1. Case result 列表
2. Failure stage
3. Trace timeline
4. Review hook
5. 问题回跳

#### 5. Report Studio

负责结论输出。

承接：

1. Run report
2. 聚合报告
3. 版本对比
4. 恶化样本
5. 放行建议

#### 6. Eval Studio

负责判定逻辑与规则资产。

承接：

1. Evaluator 列表
2. 阈值配置
3. 红线规则
4. 对象绑定
5. Suite 绑定

### 2.3 横向治理面

治理面不单独做成“首页”，而作为横向能力嵌入各工作台：

1. Baseline 管理
2. Compare 结果
3. Release gate
4. Review 状态
5. 历史版本复用

也就是说，治理是横向覆盖能力，不是独立业务岛。

## 3. 信息架构原则

### 原则 1：主导航按工作台，不按对象类型

不能做成：

1. MCP 区
2. Chat 区
3. Agent 区
4. Retrieval 区

这样会直接退化成对象分类目录。

正确方式是：

1. 在工作台里按对象筛选
2. 在工作台里共享统一操作模型

### 原则 2：资产与运行分离

必须清楚区分：

1. `Case / Suite / Evaluator`
   这是资产
2. `Run / Trace / Report / Compare`
   这是运行结果

这两类对象在界面上不能混成一个层次。

### 原则 3：报告不是首页，报告是结果出口

首页只承担：

1. 当前工作区状态
2. 最近运行
3. 关键阻塞项

正式结论必须进入 `Report Studio`。

### 原则 4：规则不做成“配置孤岛”

`Eval Studio` 必须跟：

1. Case
2. Suite
3. Run
4. Report

形成明确引用关系，否则规则页面一定会空。

## 4. 与 Coze Studio 的相似点

建议学习的不是扣子的业务内容，而是它的产品组织方式：

1. 用 workspace shell 承接全局状态
2. 用左侧细导航承接一级工作台
3. 用中间大画布承接当前主要工作
4. 用右侧上下文面板承接当前对象信息
5. 让页面看起来像“在工作”，而不是“在看后台”

## 5. 与 Coze Studio 的差异点

连弩不是 Agent 搭建平台，所以不能照抄：

1. 不做 Prompt Playground 主中心
2. 不做 Workflow Builder 主中心
3. 不做 Plugin / Knowledge / Database 资源主中心

连弩的真正核心对象仍然是：

1. Case
2. Suite
3. Run
4. Trace
5. Report
6. Evaluator

所以更准确地说，是“用 Coze Studio 的工作台骨架，承载 AI 测试平台自己的对象模型”。

## 6. 一句话结论

当前版连弩应该被重构成一个 `AI Testing Studio`，而不是一个“多模块后台系统”。产品结构要围绕 `Case / Suite / Run / Trace / Report / Evaluator` 六类核心工作台展开，再由 compare 和 gate 横向贯穿。
