# 连弩-AI自动化质量控制平台 PRD

- 项目名称：`连弩-AI自动化质量控制平台`
- 英文标识：`ChoKoNu`
- 访问域名：`chokonu.dobest.com`
- 文档版本：`v1.0`
- 文档日期：`2026-04-28`
- 文档定位：作为当前首版产品真源，统一产品、设计、前端、后端、测试口径

## 1. 文档目标

当前需要先把本项目的产出重新收口成四层：

1. `PRD`
2. `设计图`
3. `前端`
4. `后端`

其中第一优先级是先冻结 `PRD`，避免出现：

1. 多份规划文档并存
2. 三个系统整合口径漂移
3. 设计图、前端、后端各自按不同理解推进

本 PRD 的目标不是描述一个长期理想平台，而是明确：

1. `v1.0` 到底做什么
2. 三个系统的能力如何收口
3. 产品结构、页面、对象和链路如何定义
4. 前端与后端接下来各自应该落什么

## 2. 产品背景

当前团队已经具备三类可复用能力来源，但它们分散在不同系统中：

1. `AI应用评测 demo`
   对应仓库：`chat_mcp_qa_platform`

2. `扣子罗盘 AI 评测模块能力`
   对应仓库：`coze-studio`

3. `前端 Web 自动化测试`
   对应仓库：`ai-auto-test`

当前问题不在于能力不存在，而在于：

1. 三套能力没有统一入口
2. 项目接入、需求、用例、执行、Trace、报告之间没有统一主链
3. Web UI 自动化和 AI 应用评测没有沉到同一个平台
4. 现有结果难以形成统一复核和上线门禁

因此当前要做的不是再做一个并列系统，而是：

`把三个系统中首版必须落地的能力，按统一设计图整合进同一个连弩工作台。`

## 3. 产品定位

一句话定义：

`用 AI 驱动需求理解、用例生成、自动执行、结果判定、问题归因和上线门禁，覆盖 Web 自动化与 AI 应用评测的统一质量控制平台。`

当前首版不追求做成“全能力测试平台”，而是先把最有价值、最能落地的两条主线站稳：

1. `Web UI 自动化`
2. `MCP / Agent 的 AI 应用评测`

## 4. 当前版本目标

### 4.1 版本信息

- 版本：`v1.0`
- 上线日期：`2026-05-10`

### 4.2 版本承诺

`v1.0` 必须持续可用于：

1. `Web UI 自动化`
2. `MCP / Agent 的 AI 应用评测`

### 4.3 当前版本不承诺

以下内容不进入 `v1.0` 的主承诺：

1. App 自动化测试完整闭环
2. Skill 评测完整闭环
3. 知识库 / RAG 评测完整闭环
4. 接口评测完整闭环
5. 大规模真实编排调度能力
6. 复杂生产级权限体系

这些内容进入后续 `v1.1 / v1.2` 规划，但不反向侵入 `v1.0`。

## 5. 产品目标

`v1.0` 要证明五件事：

1. 三个能力来源可以收口到同一个平台
2. 用户可以通过统一登录进入统一工作台
3. 用户可以按项目组织质量工作
4. Web UI 自动化和 AI 应用评测可以走同一条质量主链
5. 执行结果可以沉到 Trace、报告、复核、门禁，而不是只“跑过一次”

## 6. 用户与角色

### 6.1 主要用户

1. 产品经理
2. 测试 / 测开
3. 平台工程 / 研发
4. 质量负责人
5. 版本评审 / 发布决策角色

### 6.2 角色关注点

#### 产品经理

关心：

1. 项目接入是否完成
2. 需求是否被转成质量资产
3. 当前版本是否具备上线判断依据

#### 测试 / 测开

关心：

1. 用例是否可组织
2. 执行是否可发起
3. Trace 是否可定位
4. 报告是否可交付

#### 平台工程 / 研发

关心：

1. 项目如何接入
2. 执行失败如何回传
3. Trace 和报告如何帮助定位与修复

#### 质量负责人 / 发布决策人

关心：

1. 当前版本风险等级
2. 是否应放行
3. 是否需要人工复核

### 7.2 `chat_mcp_qa_platform`

首版主要吸收：

1. MCP / Agent AI 应用评测对象建模
2. AI 评测执行链
3. 评测结果分析
4. AI Judge / 复核逻辑

### 7.3 `coze-studio`

1. AI 评测模块的产品表达
2. 评测对象呈现方式
3. 评测结果与结论表达
4. 与 AI 应用评测相关的工作台组织思路

## 8. 产品结构

平台固定围绕这条质量主链组织：

`项目接入 -> 需求 -> 用例 -> 执行 -> Trace -> 报告 -> 复核 -> 门禁`

### 8.2 统一对象模型

`v1.0` 至少统一以下对象：

1. `Project`
2. `Requirement`
3. `Case`
4. `Run`
5. `Trace`
6. `Report`
7. `Review`
8. `Gate`

### 8.3 统一工作台

一级工作台固定为：

1. `总览`
2. `项目台`
3. `需求台`
4. `用例台`
5. `执行台`
6. `Trace 台`
7. `报告台`
8. `复核台`
9. `门禁台`

## 10. 页面需求

### 10.1 登录页

目标：

1. 承接统一登录入口
2. 建立“连弩”品牌心智
3. 进入工作台

必须具备：

1. 左侧品牌区
2. 右侧登录卡
3. 小闪扫码登录结构
4. 权限申请 / 帮助中心

当前约束：

1. 扫码区不需要额外登录按钮
2. 点击二维码区域进入工作台

### 10.2 总览

目标：

1. 快速说明当前平台质量状态
2. 展示多项目健康情况
3. 展示 AI Coding 衔接状态
4. 暴露高风险项和最近执行

必须具备：

1. 摘要指标卡
2. 质量主链状态
3. 多项目健康概览
4. AI Coding 衔接
5. 高风险关注
6. 最近执行

### 10.3 项目台

目标：

1. 承接项目接入
2. 把三系统能力以“项目上下文”形式挂进平台

必须具备：

1. 项目列表
2. 仓库 / 分支 / 环境信息
3. 项目接入抽屉或表单
4. 能力映射
5. AI Coding 衔接
6. 接入校验

### 10.4 需求台

目标：

1. 承接 PRD、Diff、AI Coding、人工补录的需求输入
2. 把需求沉成后续用例资产的上游

必须具备：

1. 需求列表
2. 来源标识
3. 优先级
4. 业务背景
5. AI 理解摘要
6. 关联用例
7. 审计信息

### 10.5 用例台

目标：

1. 统一承接测试资产
2. 同时服务 Web UI 自动化和 AI 应用评测

`v1.0` 只强承诺三类 `case_type`：

1. `ui`
2. `agent`
3. `mcp`

必须具备：

1. 用例列表
2. case_type 标签
3. 执行器信息
4. 断言 / Judge
5. Trace 引用
6. AI 建议

### 10.6 执行台

目标：

1. 成为统一执行入口
2. 统一承接 Web UI 自动化和 AI 应用评测

必须具备：

1. 执行任务列表
2. 环境 / 版本 / 执行器
3. 触发来源
4. 任务状态
5. 任务步骤
6. 失败定位

### 10.7 Trace 台

目标：

1. 成为统一证据中心
2. 承接两条主线的证据链

必须具备：

1. 步骤轨迹
2. 日志
3. 截图
4. 工具调用
5. AI 分析
6. 人工备注

### 10.8 报告台

目标：

1. 输出质量结论
2. 让测试结果变成可审阅的判断材料

必须具备：

1. 测试范围
2. 通过率
3. 风险等级
4. 结果分布
5. 证据引用
6. AI 分析
7. 修复建议

### 10.9 复核台

目标：

1. 承接 AI Judge 与人工结论的收束

必须具备：

1. AI 结论
2. 规则判定
3. 人工复核
4. 争议记录

### 10.10 门禁台

目标：

1. 输出版本放行判断

必须具备：

1. 门禁概览
2. 阻断项
3. 复核结论
4. 放行建议
5. 决策记录

## 11. 核心用户路径

### 11.1 Web UI 自动化路径

`项目接入 -> 需求 -> ui 用例 -> UI 执行 -> 截图/日志 Trace -> Web 回归报告 -> 复核 -> 门禁`

### 11.2 MCP / Agent AI 应用评测路径

`项目接入 -> 需求 -> agent / mcp 用例 -> AI 评测执行 -> 对话/工具调用 Trace -> AI 评测报告 -> 复核 -> 门禁`

## 12. 前端需求

当前前端应完成三件事：

1. 按统一工作台完成页面骨架
2. 把三系统能力映射到统一页面表达
3. 提供 `v1.0` 演示链路：
   `项目 -> 需求 -> 用例 -> 执行 -> Trace -> 报告 -> 复核 -> 门禁`

前端当前不强求：

1. 真实生产登录
2. 完整真实后端接口
3. 全量真实数据接入

## 13. 后端需求

当前后端应优先承接：

1. 项目接入对象
2. 需求、用例、执行、Trace、报告、复核、门禁的基础对象接口
3. Web UI 自动化与 AI 评测的执行记录落库
4. 统一报告与门禁结论对象

`v1.0` 后端优先级顺序：

1. 先保证对象与链路存在
2. 再保证三系统能力可挂载
3. 最后再补高级调度与权限

## 14. 非目标

当前 PRD 明确不承诺以下内容：

1. 生产级真实登录体系
2. 生产级复杂权限体系
3. 全量真实后端联调
4. App 自动化完整主线
5. Skill / 知识库 / 接口评测完整主线
6. 大规模编排与调度系统

## 15. 上线标准

### 15.1 可上线

满足以下条件即可判定 `v1.0` 可上线：

1. 用户可从登录页进入统一工作台
2. 用户可在工作台中切换项目上下文
3. 用户可完整访问 9 个核心页面
4. Web UI 自动化主线可演示
5. MCP / Agent AI 应用评测主线可演示
6. Trace、报告、复核、门禁链路可演示

### 15.2 不可上线

出现以下情况，不应按 `v1.0` 上线：

1. 登录后无法进入工作台
2. 核心页面无法访问
3. `ui / agent / mcp` 三类核心对象无法成立
4. 两条主线无法讲通
5. 因提前引入 `v1.1 / v1.2` 内容而破坏首版稳定性
