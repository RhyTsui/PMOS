# 连弩 2026-05 版本计划
- 项目：`连弩-AI自动化质量控制平台 / ChoKoNu`
- 日期：`2026-04-28`
- 作用：`冻结 2026-05 版本节奏、边界与交付口径`

## 1. 版本目标收口

5 月的目标不是一次性做成“全能力 AI 质量平台”，而是拆成三段递进版本：

1. `v1.0`
   上线日期：`2026-05-10`
   目标：`确保平台持续可用于 Web UI 自动化与 MCP / Agent 的 AI 应用评测`

2. `v1.1`
   目标：`补齐 App 自动化测试能力，形成新的执行主线`

3. `v1.2`
   目标：`补齐 Skill、知识库与接口评测能力，形成更完整的 AI 应用评测域`

核心原则：

1. `v1.0` 必须先站稳，不能为了追 `v1.1 / v1.2` 把 `v1.0` 冲散。
2. `v1.1 / v1.2` 必须建立在统一项目接入、统一执行、统一 Trace、统一报告主链之上。
3. 后续版本是“能力扩容”，不是重新推翻 `v1.0` 的骨架。

## 2. 版本规划判断

当前产品问题要被拆成两层：

### 用户需求层

用户真正需要的是：

1. 先有一个可用的平台，能稳定承接项目。
2. 首先把 `Web UI 自动化` 和 `MCP / Agent AI 应用评测` 跑通。
3. 然后再逐步扩到 `App 自动化`、`Skill`、`知识库`、`接口评测`。

### 产品需求层

产品上必须保证：

1. 版本节奏有清晰边界，不混做。
2. 每个版本都明确：
   - 承诺什么
   - 不承诺什么
   - 上线判断是什么
   - 为下一版本预留什么

当前结论：

- `v1.0` 是 `平台站稳版`
- `v1.1` 是 `App 自动化扩展版`
- `v1.2` 是 `AI 评测域补齐版`

## 3. v1.0

### 3.1 上线日期

`2026-05-10`

### 3.2 版本定位

`统一质量平台首个可持续使用版本`

重点不是页面更多，而是：

1. 平台统一入口可用
2. 项目接入主链可用
3. Web UI 自动化可持续使用
4. MCP / Agent AI 应用评测可持续使用
5. Trace、报告、复核、门禁具备最小闭环

### 3.3 v1.0 必达能力

#### A. 平台底座

1. 统一登录入口
2. 项目选择器
3. 项目接入页面
4. 统一工作台导航
5. 统一桌面端详情栏 / 移动端抽屉交互

#### B. Web UI 自动化主线

1. 项目可配置 Web 自动化执行信息
2. 用例台可承接 `ui` 类型用例
3. 执行台可发起 Web UI 执行任务
4. Trace 台可查看截图、日志、失败定位
5. 报告台可输出 Web 回归结论

#### C. MCP / Agent AI 应用评测主线

1. 项目可配置 AI 应用评测对象
2. 用例台可承接 `agent / mcp` 类型用例
3. 执行台可发起 AI 应用评测任务
4. Trace 台可查看对话轨迹、工具调用、AI Judge 结果
5. 报告台可输出 AI 应用评测结论

#### D. 质量主链闭环

1. 需求台
2. 用例台
3. 执行台
4. Trace 台
5. 报告台
6. 复核台
7. 门禁台

### 3.4 v1.0 非目标

以下内容不进入 `2026-05-10` 的主承诺：

1. App 自动化执行器正式可用
2. Skill 专项评测闭环
3. 知识库 / RAG 专项评测闭环
4. 接口评测完整产品化闭环
5. 大规模真实调度编排系统
6. 复杂生产级权限体系
7. 全量真实后端与真实数据打通

### 3.5 v1.0 上线判断

`v1.0` 上线不是“所有能力都做完”，而是满足以下标准：

1. 用户可以登录并进入统一工作台
2. 用户可以选项目并切换上下文
3. 用户可以完成 Web UI 自动化主线演示与操作
4. 用户可以完成 MCP / Agent AI 应用评测主线演示与操作
5. 用户可以从执行进入 Trace，再进入报告、复核、门禁
6. 平台在结构上具备持续扩展到 `v1.1 / v1.2` 的稳定骨架

## 4. v1.1

### 4.1 版本定位

`App 自动化测试应用版本`

### 4.2 版本目标

在不推翻 `v1.0` 骨架的前提下，把 `App 自动化测试` 纳入统一平台。

### 4.3 v1.1 必达能力

1. 项目台支持 App 测试接入信息
2. 用例台支持 `device / app_ui` 类用例
3. 执行台支持 App 自动化执行入口
4. Trace 台支持 App 截图、步骤日志、失败证据
5. 报告台支持 App 回归报告

### 4.4 v1.1 关键约束

1. 不能新长一套“App 测试独立系统”
2. 必须继续复用：
   - 项目台
   - 用例台
   - 执行台
   - Trace 台
   - 报告台
3. App 自动化只是新增执行能力，不是重做平台

## 5. v1.2

### 5.1 版本定位

`AI 评测域补齐版本`

### 5.2 版本目标

在 `v1.0` 已具备的 `MCP / Agent` 基础上，继续补齐：

1. `Skill`
2. `知识库`
3. `接口评测`

### 5.3 v1.2 必达能力

#### A. Skill

1. Skill 作为评测对象可接入项目
2. 用例台可承接 `skill` 类型用例
3. 执行台可发起 Skill 评测

#### B. 知识库

1. 知识库 / RAG 可作为评测对象接入
2. 用例台可承接知识检索与回答评测用例
3. Trace 台可承接检索证据、引用、回答输出
4. 报告台可输出知识库评测结论

#### C. 接口评测

1. 项目台可配置接口测试对象
2. 用例台可承接 `api` 类型用例
3. 执行台可发起接口评测
4. Trace 台可查看请求、响应、断言结果
5. 报告台可输出接口质量结论

### 5.4 v1.2 关键约束

1. 仍然不新增独立一级产品壳
2. 只在统一骨架上增加对象类型与执行器

## 6. 三个版本的关系

### 6.1 版本递进逻辑

1. `v1.0`
   先把平台站稳

2. `v1.1`
   扩到 App 自动化

3. `v1.2`
   补齐 Skill、知识库、接口评测

### 6.2 能力演进图

`v1.0`
- Web UI 自动化
- MCP / Agent AI 应用评测

`v1.1`
- v1.0 全保留
- + App 自动化

`v1.2`
- v1.0 / v1.1 全保留
- + Skill
- + 知识库
- + 接口评测

## 7. 5 月执行口径

### 7.1 当前最高优先级

当前必须按下面顺序推进：

1. 冻结 `v1.0` 范围
2. 以 `2026-05-10` 为上线日倒排
3. 把 `v1.1 / v1.2` 明确标成后续版本，不得反向侵入 `v1.0`

### 7.2 本周应完成的产品动作

1. 冻结 `v1.0 / v1.1 / v1.2` 版本边界
2. 冻结 `v1.0` 页面与模块清单
3. 冻结 `v1.0` 必达验收口径
4. 标记 `v1.1 / v1.2` 的预留位

### 7.3 本周应完成的研发动作

1. 优先保证 `v1.0` 页面稳定可用
2. 不提前把 `App / Skill / 知识库 / 接口评测` 做成半吊子页面
3. 先围绕 `Web UI 自动化` 与 `MCP / Agent 评测` 打通真实主线

### 7.4 5 月倒排

#### 2026-04-28 至 2026-04-30

目标：`冻结版本边界与 v1.0 页面/模块清单`

必须完成：

1. 冻结 `v1.0 / v1.1 / v1.2` 版本边界
2. 冻结 `v1.0` 工作台页面范围
3. 冻结 `v1.0` 用例类型范围：`ui / agent / mcp`
4. 冻结 `v1.0` 上线判定口径

#### 2026-05-01 至 2026-05-05

目标：`把 v1.0 前端和主链交互收稳`

必须完成：

1. 登录页与统一工作台稳定
2. 项目台、需求台、用例台、执行台、Trace 台、报告台、复核台、门禁台可走通
3. Web UI 自动化主线交互可演示
4. MCP / Agent 评测主线交互可演示

必须完成：

`连弩 v1.0 已可持续用于 Web UI 自动化与 MCP / Agent AI 应用评测。`

## 8. 最终版本结论

当前 5 月版本计划正式收口为：

1. `v1.0`
   `2026-05-10` 上线  
   核心承诺：`持续可用于 Web UI 自动化和 MCP / Agent 的 AI 应用评测`

2. `v1.1`
   核心承诺：`可做 App 自动化测试应用`

3. `v1.2`
   核心承诺：`可做 Skill、知识库、接口评测`

一句话收口：

`5 月不是把所有能力一起做完，而是先让 v1.0 在 5 月 10 日稳定上线，再按 v1.1、v1.2 逐步扩能力，不推翻统一平台骨架。`
