# 连弩-AI自动化质量控制平台 v1.0 产品规划文档

- 项目名称：`连弩-AI自动化质量控制平台`
- 英文标识：`ChoKoNu`
- 域名：`chokonu.dobest.com`
- 文档版本：`v1.0`
- 文档日期：`2026-04-28`
- 文档定位：面向业务、产品、设计、研发管理的产品规划真源

## 1. 规划目标

当前项目先按四层产出收口：

1. `产品规划文档`
2. `设计图`
3. `前端实现`
4. `后端实现`

本文件只回答产品规划层问题：

1. 为什么做
2. 做到什么边界
3. 三个系统如何整合
4. `v1.0` 为什么这样切
5. 后续 `v1.1 / v1.2` 如何演进

## 2. 项目背景

当前团队已经有三类现成能力来源，但分散在不同系统中：

1. `chat_mcp_qa_platform`
   主要沉淀 `MCP / Agent` 方向的 AI 应用评测能力

2. `coze-studio`
   当前项目里主要吸收 `扣子罗盘的 AI 评测模块内容`

3. `ai-auto-test`
   主要沉淀 `Web UI 自动化` 能力

当前问题不是缺能力，而是缺统一平台：

1. 多套系统并存，入口分散
2. 项目接入、需求、用例、执行、Trace、报告、复核、门禁没有统一主链
3. Web UI 自动化和 AI 应用评测没有统一资产承接
4. 登录、权限、后端服务能力没有形成生产级底座

因此本项目不是做一个新的第四套系统，而是：

`把三个系统中首版必须落地的能力，整合到同一个连弩工作台，并补齐生产级登录、基础权限和后端服务底座。`

## 3. 产品定位

一句话定义：

`以项目接入为起点，以质量门禁为终点，用 AI 驱动需求理解、用例生成、自动执行、结果判定、问题归因和上线决策，统一承接 Web 自动化与 AI 应用评测的质量控制平台。`

当前平台不是：

1. 单纯的 Web 自动化工具
2. 单纯的 AI 评测 Demo
3. 三套旧系统的导航聚合页

当前平台是：

1. 一个统一工作台
2. 一条统一质量主链
3. 一套统一对象模型
4. 多能力来源在同一平台内被吸收和表达

## 5. v1.0 范围修正

这次口径明确修正如下：

### 5.1 进入 v1.0 的内容

1. `生产级真实登录体系`
   必须接入 `小闪登录`

2. `生产级基础权限体系`
   不要求复杂 RBAC 全量完成，但必须有基础权限管理能力

3. `真实后端联调与服务建设`
   需要完成后端服务、基础对象接口和关键链路能力建设

4. `三系统现有能力整合`
   当前版本聚焦：
   - `Web UI 自动化`
   - `MCP / Agent AI 应用评测`
   - `扣子罗盘 AI 评测模块内容吸收`

### 5.2 暂不进入 v1.0 主范围

以下能力暂不进入 `v1.0` 主范围：

1. `App 自动化测试`
2. `Skill`
3. `知识库`
4. `接口测试`

说明：

不是这些方向不做，而是当前先不作为 `v1.0` 的首版交付目标。

## 6. 三系统整合策略

### 6.1 `ai-auto-test`

吸收方向：

1. Web UI 自动化执行链
2. 截图、日志、失败归因
3. Web 回归报告表达

### 6.2 `chat_mcp_qa_platform`

吸收方向：

1. MCP / Agent AI 应用评测对象
2. AI 评测执行链
3. Trace / Judge / 报告结构

### 6.3 `coze-studio`

吸收方向：

1. 扣子罗盘 AI 评测模块的产品表达
2. 评测对象组织方式
3. 评测结果与结论表达

补充说明：

当前项目中，`coze-studio` 不是以“编排主线”身份进入，而是以 `AI 评测模块内容来源` 身份进入。

## 7. 平台主链

`v1.0` 固定围绕这条主链组织：

`项目接入 -> 需求 -> 用例 -> 执行 -> Trace -> 报告 -> 复核 -> 门禁`

这是三个系统整合后的统一工作方式，不再按系统来源拆。

## 8. 平台结构

一级工作台固定为：

1. `总览`
2. `项目台`
3. `需求台`
4. `用例台`
5. `执行台`
6. `Trace 台`
7. `报告台`
8. `复核台`
9. `门禁台`

统一对象模型固定为：

1. `Project`
2. `Requirement`
3. `Case`
4. `Run`
5. `Trace`
6. `Report`
7. `Review`
8. `Gate`

## 9. v1.0 用户价值

对业务与产品的价值：

1. 三系统能力第一次收口到一个统一入口
2. 能从项目维度组织质量资产与执行链
3. 登录、权限、后端开始具备真实接入条件

对测试与研发的价值：

1. Web 自动化和 AI 评测可以在同一工作台操作
2. 执行结果有统一 Trace、报告、复核、门禁出口
3. 失败可以更快回传研发处理

对发布决策的价值：

1. 形成版本级质量判断
2. 支撑是否放行、是否阻断的统一决策

## 10. v1.0 规划交付物

`v1.0` 的产品层交付物包括：

1. 一份统一产品规划文档
2. 一套统一设计图
3. 一套统一工作台前端
4. 一套基础后端服务能力
5. 一套接入小闪登录与基础权限管理的真实能力

## 11. 上线判断

### 可判定为 v1.0 达标

1. 小闪登录已接入
2. 基础权限可用
3. 后端基础服务可支撑对象链路
4. 三系统能力已按统一工作台整合
5. Web UI 自动化主线可运行
6. MCP / Agent AI 应用评测主线可运行

### 不应判定为 v1.0 达标

1. 还停留在纯前端 Demo
2. 没有真实登录
3. 没有基础权限
4. 没有后端服务承接
5. 三系统仍只是并列展示，没有统一主链

## 12. 当前结论

`v1.0` 的重点已经收敛，不再扩散：

1. 聚焦三个系统现有能力整合
2. 聚焦 `Web UI 自动化` 与 `MCP / Agent AI 应用评测`
3. 同时把 `小闪登录`、`基础权限管理`、`真实后端服务` 纳入首版交付

后续开发与设计都应以这份规划文档为产品层真源。
