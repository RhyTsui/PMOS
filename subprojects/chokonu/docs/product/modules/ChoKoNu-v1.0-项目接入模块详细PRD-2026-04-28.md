# ChoKoNu v1.0 项目接入模块详细 PRD

- 项目：`连弩-AI自动化质量控制平台 / ChoKoNu`
- 模块：`项目接入`
- 版本：`v1.0`
- 日期：`2026-04-28`
- 状态：`active`
- 适用角色：`产品经理`、`测试/测开`、`研发/平台工程`、`前端`、`后端`

## 1. 模块定位

项目接入模块是 `v1.0` 质量主链的起点。

它不只是“录一个项目”，而是负责回答：

1. 这个被测对象是谁
2. 这个对象属于什么测试能力类型
3. 这个对象如何被平台连接、启动和验证
4. 后续需求、用例、执行、Trace、报告属于哪个项目上下文

如果没有统一项目接入，后续需求台、用例台、执行台就只能停留在脱离真实归属的演示态。

## 2. 用户问题与目标

### 2.1 用户问题

1. 当前多类测试能力来源分散，项目接入口径不统一。
2. 用户无法明确知道某个项目适合走 Web 自动化、AI 应用评测，还是两者都支持。
3. 后续质量资产缺少统一项目归属，无法形成可追溯主链。

### 2.2 模块目标

1. 提供统一项目列表与项目详情。
2. 提供统一项目接入表单。
3. 记录仓库、分支、环境、安装命令、启动命令等接入信息。
4. 记录项目的能力映射。
5. 记录接入校验结果。
6. 把项目作为后续需求、用例、执行的默认上下文来源。

### 2.3 非目标

1. 不在本模块内实现需求管理。
2. 不在本模块内实现执行编排。
3. 不在本模块内处理完整权限体系。
4. 不在本模块内实现复杂 CI/CD 编排。

## 3. 主链位置

项目接入位于主链最前：

`项目接入 -> 需求 -> 用例 -> 执行 -> Trace -> 报告 -> 复核 -> 门禁`

它的作用是把“被测对象”和“质量流程”绑定起来。

## 4. 模块边界

### 4.1 模块内负责

1. 项目列表
2. 项目详情
3. 新建/编辑项目接入表单
4. 仓库信息录入
5. 环境信息录入
6. 安装/启动命令录入
7. 能力映射配置
8. 接入校验结果展示

### 4.2 模块外负责

3. 需求对象创建
4. 用例对象创建
5. 执行器真实调度

## 5. 页面结构

### 5.1 页面目标

让用户在项目台完成三件事：

1. 找到已有项目
2. 接入新项目
3. 确认项目是否具备进入后续主链的条件

### 5.2 页面结构

1. 顶部摘要区
   - 项目总数
   - 已接入项目数
   - 待校验项目数
   - 校验失败项目数
2. 筛选与动作区
   - 项目状态
   - 能力类型
   - 负责人
   - 环境
   - `新建项目接入`
3. 项目列表区
   - 项目名称
   - 项目标识
   - 能力类型
   - 默认环境
   - 校验状态
   - 最近更新时间
4. 项目详情区
   - `基本信息`
   - `仓库与环境`
   - `能力映射`
   - `接入校验`
   - `关联需求`
   - `审计`

## 6. 接入表单需求

### 6.1 基本信息

必填字段建议：

1. `项目名称`
2. `项目标识`
3. `项目描述`
4. `负责人`
5. `项目类型`

### 6.2 仓库信息

必填字段建议：

1. `仓库地址`
2. `默认分支`
3. `代码来源类型`

可选字段：

1. `子目录`
2. `README 路径`
3. `依赖文件路径`

### 6.3 环境信息

必填字段建议：

1. `默认环境`
2. `可用环境列表`
3. `运行平台`

可选字段：

1. `Base URL`
2. `环境说明`
3. `环境变量备注`

### 6.4 安装与启动

必填字段建议：

1. `安装命令`
2. `启动命令`

可选字段：

1. `构建命令`
2. `测试前置命令`
3. `端口说明`

### 6.5 能力映射

首版必须支持以下能力映射标签：

1. `Web UI 自动化`
2. `MCP / Agent AI 应用评测`
3. `扣子罗盘评测能力吸收`

规则：

1. 一个项目可绑定多个能力类型。
2. 能力映射决定后续需求台和用例台默认展示哪些链路能力。

## 7. 接入校验需求

### 7.1 校验目标

接入校验不是执行正式测试，而是确认项目具备被平台继续承接的最小条件。

### 7.2 首版校验项

1. 仓库地址是否存在
2. 默认分支是否可识别
3. 安装命令是否已填写
4. 启动命令是否已填写
5. 默认环境是否已配置
6. 至少绑定一个能力类型

### 7.3 校验结果状态

1. `未校验`
2. `校验中`
3. `校验通过`
4. `校验失败`
5. `部分通过`

### 7.4 校验结果展示

详情区必须展示：

1. 最近一次校验时间
2. 校验状态
3. 校验项逐项结果
4. 失败原因摘要
5. 建议修复动作

## 8. 交互动作

必须支持：

1. 新建项目
2. 编辑项目
3. 触发接入校验
4. 查看校验结果
5. 复制项目接入模板
6. 从项目进入需求台

跳转要求：

1. 从项目台进入需求台时必须带上 `projectId`。
2. 若项目校验未通过，进入需求台前需要明确提示“可继续，但后续主链可能不完整”。

## 12. 权限要求

### 12.1 页面权限

1. 有项目查看权限的用户可进入项目台。
2. 无权限用户不可见项目台导航或进入后显示无权限页。

### 12.2 动作权限

至少区分：

1. `查看项目`
2. `新建项目`
3. `编辑项目`
4. `触发校验`

## 13. 边界与异常场景

1. 新建项目但仓库地址为空。
2. 项目存在但未配置环境。
3. 项目可查看但不可编辑。
4. 项目能力标签为空。
5. 校验接口失败但项目信息已保存。
6. 项目已删除但旧需求仍引用该项目。

## 14. 验收标准

### 14.1 产品验收

1. 能在项目台查看已接入项目。
2. 能新增一个真实项目并保存接入信息。
3. 能看到能力映射和校验结果。
4. 能从项目进入需求台，形成主链起点。

### 14.2 前端验收

1. 项目列表、详情、表单可用。
2. 校验状态展示完整。
3. 跳转需求台时上下文保留正确。

### 14.3 后端验收

1. 项目对象可增改查。
2. 校验结果可查询。
3. 能力映射和环境配置可稳定存储。

### 14.4 用户需求判定

- 产品要求层：`solved`
  已明确项目接入模块的页面、字段、校验和接口要求。
- 用户需求层：`partial`
  只有当真实项目可成功接入并进入后续需求台时，用户问题才算真正解决。

## 15. 优先级

### P0

1. 项目列表
2. 项目详情
3. 接入表单
4. 能力映射
5. 接入校验结果
6. 跳转需求台

### P1

1. 项目模板复制
2. 更细粒度的校验项
3. 多环境切换增强

## 16. 当前结论

项目接入模块是 `v1.0` 真正的主链前置层，而不是可有可无的配置页。

没有这一层：

1. 工作台没有真实项目上下文
2. 后续需求、用例、执行无法稳定归属
3. 三类能力整合只能停留在概念汇总

因此它应作为工作台之后的第二批详细 PRD 优先落地。
