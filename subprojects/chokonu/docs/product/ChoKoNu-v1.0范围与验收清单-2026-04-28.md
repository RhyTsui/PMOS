# 连弩 v1.0 范围与验收清单
- 项目：`连弩-AI自动化质量控制平台 / ChoKoNu`
- 版本：`v1.0`
- 上线日期：`2026-05-10`
- 作用：`把 v1.0 拆成需求清单、页面清单、验收清单`

## 1. v1.0 版本定义

`v1.0` 不是全能力完成版，而是首个稳定可上线版本。

唯一主承诺：

`持续可用于 Web UI 自动化和 MCP / Agent 的 AI 应用评测。`

这意味着 `v1.0` 必须同时满足两层要求：

### 用户需求层

1. 用户能登录并进入统一平台
2. 用户能按项目组织测试工作
3. 用户能跑通 Web UI 自动化主线
4. 用户能跑通 MCP / Agent 评测主线
5. 用户能看到 Trace、报告、复核、门禁结论

### 产品需求层

1. 平台骨架稳定
2. 页面范围可控
3. 用例类型边界明确
4. 后续 `v1.1 / v1.2` 有预留但不侵入

## 2. v1.0 需求清单

### P0 必做

#### A. 平台基础

1. 登录页可用
2. 统一工作台可访问
3. 顶部项目 / 环境 / 版本切换可用
4. 左侧导航可切页
5. 桌面端右侧详情栏可展开 / 收起
6. 移动端详情抽屉可打开

#### B. 项目接入

1. 项目台存在
2. 项目列表可展示
3. 项目详情可查看：
   - 仓库
   - 分支
   - 环境
   - 能力映射
   - AI Coding 衔接
   - 接入校验

#### C. 需求主线

1. 需求台存在
2. 需求可展示来源：
   - PRD
   - Diff
   - AI Coding
   - 人工补录
3. 需求详情可查看业务背景、AI 理解、关联用例、审计

#### D. 用例主线

1. 用例台存在
2. `v1.0` 只强承诺以下 `case_type`：
   - `ui`
   - `agent`
   - `mcp`
3. 用例详情可查看：
   - 用例内容
   - 执行配置
   - 断言 / Judge
   - Trace 引用
   - AI 建议

#### E. 执行主线

1. 执行台存在
2. 执行任务可展示：
   - 环境
   - 版本
   - 执行器
   - 触发来源
   - 状态
3. `v1.0` 至少支撑两条执行能力主线：
   - Web UI 自动化
   - MCP / Agent AI 应用评测

#### F. Trace 主线

1. Trace 台存在
2. Trace 可展示：
   - 步骤轨迹
   - 日志
   - 截图
   - 工具调用
   - AI 分析
   - 人工备注

#### G. 报告主线

1. 报告台存在
2. 报告可展示：
   - 范围
   - 通过率
   - 风险等级
   - 结果分布
   - 证据引用
   - AI 分析
   - 修复建议

#### H. 复核主线

1. 复核台存在
2. 复核详情可展示：
   - AI 结论
   - 规则判定
   - 人工复核
   - 争议记录

#### I. 门禁主线

1. 门禁台存在
2. 门禁详情可展示：
   - 门禁概览
   - 阻断项
   - 复核结论
   - 放行建议
   - 决策记录

### P1 预留但不作为 5/10 主承诺

1. `api` 类型用例完整闭环
2. `workflow` 类型用例完整闭环
3. App 自动化执行入口
4. Skill 评测对象
5. 知识库 / RAG 评测对象

## 5. v1.0 验收清单

### A. 登录验收

1. 可访问 `#/login`
2. 登录页布局正确
3. 扫码区域可进入工作台
4. 登录后默认进入 `#/workbench/overview`

### B. 工作台骨架验收

1. 所有工作台路由可访问
2. 左侧导航可切换
3. 顶部项目切换可用
4. 顶部环境切换可用
5. 顶部版本切换可用

### C. Web UI 自动化主线验收

1. 可在项目台看到 Web 自动化项目
2. 可在用例台看到 `ui` 用例
3. 可在执行台看到 UI 执行任务
4. 可在 Trace 台查看截图和日志
5. 可在报告台查看 Web 回归结论

### D. MCP / Agent 主线验收

1. 可在项目台看到 AI 应用评测项目
2. 可在用例台看到 `agent / mcp` 用例
3. 可在执行台看到 Agent / MCP 执行任务
4. 可在 Trace 台看到对话轨迹、工具调用和 AI Judge
5. 可在报告台查看 AI 应用评测结论

### E. 质量闭环验收

1. 能从需求进入用例
2. 能从用例进入执行
3. 能从执行进入 Trace
4. 能从 Trace 进入报告
5. 能从报告进入复核
6. 能从复核进入门禁

### F. 响应式验收

1. PC 下详情区可收起
2. 移动端下详情通过抽屉打开
3. 页面不出现严重横向溢出

## 6. v1.0 上线判断

### 可上线

满足以下条件即可判定 `v1.0` 可上线：

1. 统一平台可访问
2. 登录链路可用
3. Web UI 自动化主线可演示
4. MCP / Agent 评测主线可演示
5. Trace、报告、复核、门禁页面可完整浏览
6. `v1.1 / v1.2` 需求未反向侵入 `v1.0`

### 不可上线

出现以下任一情况，不应按 `v1.0` 上线：

1. 登录后无法进入工作台
2. 工作台核心页面无法访问
3. Web UI 自动化主线无法成立
4. MCP / Agent 评测主线无法成立
5. 为塞入 App / Skill / 知识库 / 接口评测而破坏首版稳定性

## 7. 当前结论

当前这份清单已经把 `v1.0` 从“版本方向”进一步收成了“执行范围”。

状态判断：

1. 用户需求层：`partial`
   已明确 `v1.0` 到底要交付什么，但还没完全映射到开发排期与页面删减。

2. 产品需求层：`solved`
   `v1.0` 的需求范围、页面范围、能力边界和验收清单已经具备直接执行条件。
