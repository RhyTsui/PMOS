# ChoKoNu 项目执行工作流
- 项目：`连弩-AI测试平台 / ChoKoNu`
- 子项目路径：`subprojects/chokonu/`
- 当前状态：`active`
- 文档层级：`internal-detailed`

## 1. 工作流定位

ChoKoNu 当前不是“继续发散方向”的研究型占位项目，而是进入了：

`规划收口 + 执行收口 + 下游交付收口`

的并行阶段。

当前执行目标是把下面 3 条链收实：

1. `规划链`
   `背景 -> 最佳实践 -> 首版目标 -> 架构 -> 动作链 -> skill`
2. `执行链`
   `任务单 -> 同步 -> 评审 -> 决策 -> 回写`
3. `交付链`
   `详细版 -> 人读版 -> 开发评审版 -> 版本评审`

## 2. 当前主链

ChoKoNu 首版默认沿这条主链推进：

`定义测试标准 -> 沉淀测试样本 -> 组织评测集 -> 配置执行任务 -> 运行评估器 -> 归集结果与 Trace -> 生成测试报告 -> 沉淀回归集`

所有新增页面、模块、需求、版本项，都必须能回挂到这条主链的某一环。

## 3. 当前阶段

目标：
- 统一项目名称、路径、主文档
- 明确哪些是活跃真源，哪些是 legacy
- 明确哪些历史分析属于别的子项目，不再留在 ChoKoNu 活跃目录

通过条件：
- `subprojects/chokonu/` 成为 `ChoKoNu` 唯一活跃路径
- `subprojects/tracking-acceptance/` 作为独立的 `AI 埋点验收` 子项目维护自己的活跃文档

### 阶段 B：首版规划收口

目标：
- 形成可直接引用的首版总规划
- 明确架构图、五层要素、动作链、skill 链

通过条件：
- 有一份正式 `首版总规划`
- 每个关键动作都有输入 / 输出 / 责任角色 / skill

### 阶段 C：执行文档收口

目标：
- 形成 PM 可直接使用的执行工作流和任务单
- 形成下游可读的精简交付文档

通过条件：
- 有 `项目执行工作流`
- 有 `产品经理任务单`
- 有 `需求清单`
- 有 `版本评审模板`

### 阶段 D：版本评审与下游交付

目标：
- 面向开发、老板、下游团队形成稳定交付格式

通过条件：
- 版本文档按模板
- 双产物齐全
- 需求清单表达简洁清晰

## 4. 角色分工

### 产品主管 / Chief

负责：
- 首版边界判断
- 关键架构收敛
- 规则、模板、评审口径
- 最终拍板

### 项目产品经理

负责：
- 同步信息
- 首稿整理
- 需求清单维护
- 版本评审准备
- 人读版交付材料整理

### 测试 / 测开 / 工程角色

负责：
- 规则执行可实现性判断
- 自动化测试链路
- 评估器与报告落地
- Trace / 回归链路实现

## 5. 固定同步格式

项目产品经理每次同步统一为：

1. `本轮目标`
2. `已完成`
3. `当前在写`
4. `卡点`
5. `需要主管决策`

未按这 5 行同步，视为未正式提交。

## 6. 固定问话环节

以下时点必须形成书面问题单：

1. 第一轮研究稿后
2. 首版边界收敛前
3. 架构图与动作链定稿前
4. 范围明显变化时

问题必须覆盖：

1. `目标确认`
2. `边界确认`
3. `优先级确认`
4. `非目标确认`
5. `待主管拍板`

## 7. 文档分层规则

### 过程文档

给内部团队使用，可详细。

用途：

- 研究
- 推理
- 对比
- 风险暴露
- 中间过程沉淀

### 交付文档

给开发、老板、下游使用，必须精简。

必须只保留：

- 目标
- 范围
- 核心结论
- 行动项
- 风险
- 待拍板事项

## 8. 版本文档规则

版本文档默认双产物：

1. `MD 详细版`
2. `DOCX 人读版源文档`

其中需求清单统一按以下格式书写：

- 一条需求一句话
- 先动作，后对象
- 不把背景叙述塞进需求句子
