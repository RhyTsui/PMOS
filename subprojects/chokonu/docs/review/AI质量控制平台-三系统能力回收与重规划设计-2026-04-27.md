# AI 质量控制平台三系统能力回收与重规划设计 2026-04-27

- 项目：`连弩-AI测试平台 / ChoKoNu`
- 状态：`draft`
- 作用：把三个现有系统的能力回收、统一抽象、页面收口和开发整合方向压成一份设计真源，作为后续设计图确认与开发实施依据

## 1. 问题定义

当前不是从零设计一个全新平台，也不是把三个系统并列挂到一个总入口。

当前要做的是：

`基于三个现有系统的既有实现，先完成能力回收，再做统一重规划和设计收口，最后进入二次开发整合。`

这里的“三个系统”是：

1. `AI应用评测demo`
   `http://gitlab.sh.com/xuqing/chat_mcp_qa_platform.git`
2. `coze部署仓库`
   `https://github.com/coze-dev/coze-studio`
3. `前端web自动化测试`
   `http://gitlab.sh.com/huangsenda/ai-auto-test/tree/cursor/requirements-refactor-staged`

## 2. 设计目标

本轮设计要同时满足四件事：

1. 不丢失三个系统里已经存在的关键能力
2. 不保留三个独立产品壳长期并存
3. 不按仓库来源做平台主结构
4. 能直接指导后续前后端整合开发

## 3. 统一产品定位

统一平台定位为：

`AI 自动化质量控制平台`

一句话定义：

`以统一测试资产、执行编排、Trace 观测、AI Judge 与质量门禁为底座，用 AI 驱动需求理解、用例生成、自动执行、结果判定、问题归因和上线决策，承接 Web 自动化、AI 应用评测、Agent / MCP / Workflow 等对象的统一质量控制平台。`

## 4. 为什么不能保留三个系统并列

如果继续按三个系统并列推进，会出现四个问题：

1. 三套对象模型并存
2. 三套执行入口并存
3. 三套报告和证据链并存
4. 用户必须先理解系统来源，再理解质量流程

这会把平台再次做成“能力集合页”，而不是“质量控制平台”。

因此，三个系统只能作为能力来源，不能继续作为最终产品结构。

## 5. 三系统能力回收结论

### 5.1 `chat_mcp_qa_platform`

这套系统更接近：

`AI 应用评测能力源`

优先回收的能力：

1. AI 应用作为测试对象的建模方式
2. AI 评测集 / 黄金集思路
3. 面向 Agent / MCP / RAG / Workflow 的执行主线
4. AI 结果分析、复核、结论输出

不应直接继承的部分：

1. 如果其现有前台壳与统一平台主骨架冲突，不保留整壳
2. 其对象命名和页面结构不应直接定义最终 IA

### 5.2 `coze-studio`

这套系统更接近：

`Agent / Workflow 编排参考底座`

优先回收的能力：

1. 节点化流程表达
2. 多步骤任务运行模型
3. Workflow / Agent 执行轨迹
4. 编排态与运行态的结构参考

不应直接继承的部分：

1. 不把 Coze 现有产品壳直接当成最终平台壳
2. 不把其完整产品边界直接抬升为当前平台边界

### 5.3 `ai-auto-test`

这套系统更接近：

`Web UI 自动化测试能力源`

优先回收的能力：

1. 需求理解到用例生成链
2. Web 自动化执行链
3. 截图、失败定位、证据链
4. 回归测试与执行报告

不应直接继承的部分：

1. 不把“前端自动化测试平台”单独保留成一级产品区
2. 不只围绕 UI 自动化去定义整个总平台

## 6. 回收后的统一平台主骨架

三个系统回收后，统一平台固定围绕这条主链组织：

`需求 → 用例 → 执行 → 结果 → 报告 → 分析 → 复核 → 门禁`

这是统一后的唯一主骨架。

说明：

1. Web 自动化只是执行对象之一
2. AI 应用评测只是执行对象之一
3. Agent / MCP / Workflow 是能力域与对象类型，不再单独长成产品壳

## 7. 统一对象模型

首版统一平台至少要有以下核心对象：

1. `Requirement`
2. `Case`
3. `Suite / Plan`
4. `Run`
5. `Trace`
6. `Report`
7. `Review`
8. `Gate`

这些对象分别承接：

1. 需求输入
2. 用例资产
3. 测试集与批次
4. 执行任务
5. 过程证据
6. 结果输出
7. 人工或 AI 复核
8. 最终放行判断

## 8. 统一能力底座

三个系统整合后，平台底座统一为五块：

1. `统一测试资产`
2. `统一执行编排`
3. `统一 Trace 观测`
4. `统一 AI Judge`
5. `统一质量门禁`

### 8.1 统一测试资产

承接：

1. 需求
2. 用例
3. 测试集
4. 对象类型
5. 断言方式

### 8.2 统一执行编排

承接：

1. DAG / 队列 / 重试
2. 版本 / 环境 / 分支 / 任务
3. 单步与多步执行

### 8.3 统一 Trace 观测

承接：

1. 日志
2. 截图
3. 工具调用轨迹
4. 节点运行轨迹
5. 结果证据链

### 8.4 统一 AI Judge

承接：

1. LLM Judge
2. 规则判定
3. AI 分析摘要
4. 问题归因与修复建议

### 8.5 统一质量门禁

承接：

1. 最终质量结论
2. 是否建议放行
3. 风险等级
4. 审批与审计

## 9. 统一信息架构收口

统一平台建议固定为 7 个工作台：

1. `需求台`
2. `用例台`
3. `执行台`
4. `Trace 台`
5. `报告台`
6. `复核台`
7. `门禁台`

### 9.1 为什么不是按能力域做一级导航

因为如果按 `UI / API / Agent / MCP / Workflow` 做一级导航，会把产品再次拆回多个测试平台。

这些能力域更适合作为：

1. 用例类型筛选
2. 执行器类型筛选
3. 结果分析维度

### 9.2 为什么不是按仓库来源做一级导航

因为用户不应该先理解“这个能力来自哪个系统”，再理解“我现在要做什么质量动作”。

平台应该先服务质量动作，再隐藏底层能力来源。

## 10. 页面级设计收口建议

### 10.1 首页 / 总览

首页不做成三个系统入口，而做成统一质量总览：

1. 当前需求数
2. 当前待执行任务数
3. 近 7 天运行结果概览
4. 高风险失败项
5. 待复核项
6. 待门禁决策版本

### 10.2 需求台

主要承接：

1. 需求列表
2. 版本和对象类型筛选
3. AI 理解摘要
4. 转用例动作

### 10.3 用例台

主要承接：

1. 用例列表
2. 测试集
3. `case_type`
4. 断言方式
5. 来源需求

`case_type` 建议首版支持：

1. `ui`
2. `api`
3. `data_chain`
4. `device`
5. `rag`
6. `agent`
7. `mcp`
8. `workflow`
9. `skill`

### 10.4 执行台

主要承接：

1. 环境选择
2. 版本选择
3. 测试集选择
4. 执行器选择
5. 运行状态
6. 重跑与重试

### 10.5 Trace 台

主要承接：

1. 截图
2. 日志
3. 工具调用轨迹
4. 节点轨迹
5. 错误定位

### 10.6 报告台

主要承接：

1. 质量概览
2. 问题分布
3. 通过率与失败率
4. AI 分析摘要
5. 修复建议

### 10.7 复核台

主要承接：

1. AI Judge 结果
2. 人工复核
3. 争议项处理
4. 审计记录

### 10.8 门禁台

主要承接：

1. 版本质量结论
2. 风险等级
3. 放行建议
4. 阻断项

## 11. 必须完整落地的旧能力

后续设计图和开发不能把下面这些能力做丢：

1. 从需求到用例生成的能力
2. Web UI 自动化执行能力
3. AI 应用评测能力
4. Agent / MCP / Workflow 对象测试能力
5. 截图、日志、调用轨迹的证据链
6. 结果分析、人工复核和质量门禁

## 12. 当前不建议承诺的内容

下面这些内容可以列入后续阶段，但不建议现在压成首版主承诺：

1. 真机全量成熟体系
2. 长期线上监控和告警平台
3. 自动发布阻断全量闭环
4. 多团队超复杂协同编排

原因不是不重要，而是会明显抬高整合成本，打乱当前三系统收口节奏。

## 13. 当前最合理的执行顺序

在“不先部署，先重规划”的口径下，当前推荐顺序是：

1. 回收三系统能力
2. 收口统一信息架构
3. 输出统一页面设计图
4. 列开发映射表
5. 进入二次整合开发
