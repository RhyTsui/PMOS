# AI 质量控制平台能力回收矩阵 2026-04-27

- 项目：`连弩-AI测试平台 / ChoKoNu`
- 状态：`draft`
- 作用：在不先部署三套仓库的前提下，基于既有仓库实现、现状截图和最新产品草稿，回收现有能力并约束后续统一设计图与开发整合

## 1. 本文定位

本文件不是新的功能愿望清单，而是后续重规划和开发整合的能力约束真源。

本轮执行口径为：

`先基于既有仓库实现和现状截图完成能力回收式重规划，先出统一设计图确认，再直接进入二次开发整合；但必须保证现有关键能力完整落地，不因重规划而丢失。`

因此，凡是已经存在于：

1. 既有仓库实现
2. 现状截图
3. 已确认产品草稿

中的关键能力，都必须在新平台中有明确落点。

## 3. 统一收口原则

1. 新平台按质量主链组织，不按仓库来源组织。
2. `UI / API / 数据链路 / 真机 / RAG / Agent / MCP / Workflow / Skill` 是能力域，不是并列产品壳。
3. 三套仓库是能力来源，不是长期并行保留的三套系统。
4. 设计图确认前，先完成能力回收，不允许脱离既有能力重画一套空平台。
5. 进入开发前，每一项关键旧能力都必须映射到新平台对象、流程或页面中。

## 4. 平台统一主链

后续统一平台固定围绕这条主链展开：

`需求 → 用例 → 执行 → 结果 → 报告 → 分析 → 复核 → 门禁`

能力回收、页面规划、对象建模、开发整合都必须映射回这条主链。

## 5. 平台统一对象模型

首版至少应统一以下对象：

1. `Requirement`
2. `Case`
3. `Suite / Plan`
4. `Run`
5. `Trace`
6. `Report`
7. `Gate`

如果某个既有系统的关键能力无法映射到这些对象之一，就说明统一模型仍不完整。

## 6. 三套仓库能力回收矩阵

### 6.1 AI 应用评测 demo

来源：

- `chat_mcp_qa_platform`

预期回收的关键能力：

1. AI 应用评测对象
   - `Agent`
   - `MCP`
   - `RAG / 知识库`
   - `Workflow`
2. 评测集 / 黄金集思路
3. AI 应用执行与结果输出链
4. AI 场景下的分析、复核、评测结论表达

在统一平台中的落点：

1. `Case / Suite` 资产层
2. `Run` 执行层
3. `Trace / Report` 结果层
4. `Analysis / Review / Gate` 结论层

不允许丢失的能力：

1. AI 应用不是只作为普通接口对象出现
2. MCP / Agent / RAG 相关对象必须能进入统一评测链
3. 评测结果不能只停留在运行成功/失败，要保留分析和复核空间

### 6.2 Coze 部署仓库

来源：

- `coze-studio`

预期回收的关键能力：

1. Agent / Workflow 编排思路
2. 节点化运行对象
3. 多步骤任务执行主线
4. 运行态过程可视化参考

在统一平台中的落点：

1. `Run` 的编排与执行计划
2. `Trace` 的过程轨迹表达
3. `Workflow / Agent` 类型对象的专项用例与执行器

不允许丢失的能力：

1. 多步骤流程对象不能被压扁成单次请求
2. Agent / Workflow 测试不能只做结果断言，要保留过程轨迹观察
3. 运行态编排思想要进入统一调度层，但不等于整壳继承 Coze 产品结构

### 6.3 前端 Web 自动化测试

来源：

- `ai-auto-test`

预期回收的关键能力：

1. 基于需求理解生成测试用例
2. Web UI 自动化执行
3. 截图证据链
4. 失败定位
5. 回归执行与结果报告

在统一平台中的落点：

1. `Requirement → Case` 生成链
2. `Run` 执行链
3. `Trace / Screenshot / Report` 证据链

不允许丢失的能力：

1. 从需求到用例的生成链必须保留
2. UI 自动化截图和失败证据不能丢
3. 回归语义必须保留，不能只剩一次性执行

## 7. 现状截图能力回收矩阵

来源：

- `系统v0.1现状截图`

当前已能确认的页面和模块语义：

1. 事件管理 / 事件字典
2. 上报管理
3. 日志查询
4. 数据质量
5. 详情页签化结构
6. 操作审计
7. 数据概览
8. 优先级配置

这些截图说明当前系统已经具备的不是抽象概念，而是一套较强的工作台骨架。

建议回收方式：

1. 保留“列表 + 详情 + 页签 + 审计 + 配置”的工作台结构优势
2. 不直接保留旧业务语义词，但保留其治理型产品壳能力

在统一平台中的落点建议：

1. `Requirement / Case` 采用列表和详情双栏结构
2. `Run / Report / Trace` 采用详情页签结构
3. `Review / Gate` 保留审计、配置和优先级治理表达

不允许丢失的能力：

1. 详情页签承载复杂对象信息的方式
2. 审计与治理信息的展示能力
3. 数据概览与状态分层展示能力
