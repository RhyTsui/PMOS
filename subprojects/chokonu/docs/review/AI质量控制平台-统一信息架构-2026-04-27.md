# AI 质量控制平台统一信息架构 2026-04-27

- 项目：`连弩-AI测试平台 / ChoKoNu`
- 状态：`draft`
- 作用：基于能力回收矩阵，收口统一平台的信息架构、工作台骨架和对象入口，作为后续设计图与开发整合依据

## 1. 本文目标

本文不讨论三套仓库如何并列展示，而是回答一个更关键的问题：

`统一后的 AI 自动化质量控制平台，应该按什么结构组织，才能同时接住既有能力，又不把平台再次拆成多个系统。`

结论是：

平台不按仓库来源组织，不按测试类型做一级导航，而按统一质量主链和统一对象模型组织。

## 2. 信息架构总原则

1. 一级结构按质量主链组织。
2. 能力域只作为对象筛选维度、执行器维度和结果解释维度出现。
3. 不保留“AI应用评测平台 / Web自动化平台 / Coze工作流平台”三套并列壳。
4. 列表、详情、页签、审计、配置的工作台结构继续保留。
5. 设计上优先保证对象流转清晰，而不是菜单数量完整。

## 3. 一级工作台结构

统一平台建议收口为 7 个一级工作台：

1. `需求台`
2. `用例台`
3. `执行台`
4. `Trace 台`
5. `报告台`
6. `复核台`
7. `门禁台`

这 7 个工作台分别对应统一主链：

`需求 → 用例 → 执行 → 结果 / Trace → 报告 → 复核 → 门禁`

## 4. 一级工作台说明

### 4.1 需求台

定位：

- 承接业务需求、测试需求、版本需求、专项需求
- 作为后续用例生成和执行范围定义入口

核心内容：

1. 需求列表
2. 需求详情
3. 版本关联
4. 需求拆解结果
5. AI 理解摘要
6. 来源文档
7. 转用例动作

必须承接的旧能力：

1. `ai-auto-test` 的需求理解输入
2. 现有系统工作台的列表 + 详情结构

### 4.2 用例台

定位：

- 承接统一测试资产
- 不再按“平台”拆，而按 `case_type` 管理

核心内容：

1. 用例列表
2. 测试集 / 套件列表
3. 用例详情
4. 来源需求
5. 执行器类型
6. 断言方式
7. 评测规则

`case_type` 建议固定支持：

1. `ui`
2. `api`
3. `data_chain`
4. `device`
5. `rag`
6. `agent`
7. `mcp`
8. `workflow`
9. `skill`

必须承接的旧能力：

1. `gpt问答.md` 中“不分两个系统，只按 case_type 区分”的判断
2. AI 应用评测 demo 的黄金集 / 测试对象语义
3. Web 自动化的自然语言用例与执行脚本链

### 4.3 执行台

定位：

- 承接统一执行入口
- 统一发起回归、专项运行和批量执行

核心内容：

1. 执行任务列表
2. 执行配置
3. 环境选择
4. 版本选择
5. 用例集选择
6. 执行器选择
7. 任务状态
8. 重试 / 中断 / 重跑

必须承接的旧能力：

1. Web 自动化测试的运行入口
2. AI 应用评测的自动执行链
3. Coze / Workflow 相关的多步骤任务运行语义

### 4.4 Trace 台

定位：

- 承接统一证据链和过程观测
- 这是平台区别于普通测试后台的关键层

核心内容：

1. 执行轨迹
2. 工具调用轨迹
3. 截图证据
4. 日志明细
5. 数据链路证据
6. 节点级状态
7. 异常定位

必须承接的旧能力：

1. `ai-auto-test` 的截图和失败证据链
2. Coze / Agent / Workflow 的过程轨迹要求
3. 现状截图中的日志查询、日志明细、展开收起结构

### 4.5 报告台

定位：

- 承接统一结果输出
- 不只展示执行结果，而是输出质量结论

核心内容：

1. 报告列表
2. 质量概览
3. 测试范围
4. 执行结果分布
5. 关键问题清单
6. 证据引用
7. AI 分析摘要
8. 修复建议

必须承接的旧能力：

1. Web 自动化的测试报告
2. AI 应用评测的评测结果输出
3. 会议纪要中“通过率、问题分布、截图、分析”的报告要求

### 4.6 复核台

定位：

- 承接人工复核、AI Judge 复判、争议结果处理

核心内容：

1. 待复核列表
2. AI Judge 结论
3. 规则断言结果
4. 人工复核记录
5. 误判修正
6. 审计记录

必须承接的旧能力：

1. AI Judge 不直接替代人工
2. 会议纪要里“必须保留人工审核”的判断
3. 现状截图中的操作审计能力

### 4.7 门禁台

定位：

- 承接上线前最终决策
- 平台最终价值不是“跑了测试”，而是“支撑是否放行”

核心内容：

1. 门禁规则
2. 版本质量结论
3. 风险等级
4. 放行建议
5. 阻断原因
6. 变更记录
7. 审批链

必须承接的旧能力：

1. 规划草稿中的“上线门禁”
2. 质量控制平台的最终定位

## 5. 二级对象维度

为了避免一级导航膨胀，能力域不升格为一级产品区，而作为统一筛选和标识体系存在。

建议保留以下二级维度：

1. `对象类型`
   - UI
   - API
   - 数据链路
   - 真机
   - RAG
   - Agent
   - MCP
   - Workflow
   - Skill

2. `执行器类型`
   - 浏览器执行器
   - 接口执行器
   - Agent 执行器
   - MCP 执行器
   - 真机执行器
   - 数据链路执行器

3. `判定方式`
   - 规则断言
   - AI Judge
   - 人工复核

## 6. 页面结构建议

统一页面结构建议沿用现状工作台优势，采用以下通用骨架：

1. 顶部：
   - 页面标题
   - 版本 / 环境 / 类型筛选
   - 关键动作按钮

2. 左侧或中部：
   - 列表区
   - 支持搜索、筛选、状态分组

3. 右侧或下层：
   - 详情区
   - 采用页签结构承载复杂对象

4. 详情页签建议：
   - 基本信息
   - 资产关系
   - 执行记录
   - Trace / 证据
   - 分析 / 复核
   - 审计

## 7. 明确不采用的结构

以下结构当前不建议采用：

1. 按仓库来源做一级导航
   - `AI应用评测`
   - `Coze`
   - `Web自动化`

2. 按能力域做一级导航
   - `UI测试`
   - `API测试`
   - `Agent测试`
   - `MCP测试`

3. 三套系统并列保留后用统一首页跳转

原因不是这些能力不重要，而是会把平台再次拆回多系统结构。

主要映射到：

1. 需求台

重点吸收：

1. 需求到用例生成链

3. 截图和失败证据链

## 9. 首版 IA 判断

首版统一信息架构已基本可以收口，不需要再保留“两个系统整合”或“三个系统并列”的表达。

当前更准确的判断是：

`一个平台，七个工作台，一套对象模型，多执行器接入。`
