# ChoKoNu v1.0 候选范围 2026-04-27

- 项目：`连弩-AI测试平台 / ChoKoNu`
- 状态：`draft`
- 作用：基于 `04-27` 会议收口当前 `v1.0` 候选版本范围
- 说明：两个待整合系统的 GitHub 地址暂未补充，当前先定义版本目标、边界和依赖

## 1. v1.0 目标定义

v1.0 不追求做成大而全平台，而是完成：

`两个已有系统整合后的一版统一 AI质量控制平台`

这版要证明三件事：

1. 两套已有能力可以进入同一套质量主链。
2. 小闪登录、测试报告、开放接入这些平台级底座可以统一。
3. UI 自动化、接口自动化、Agent 自动化不再各自漂浮，而能被同一平台承接。

## 2. v1.0 候选范围

按你刚刚给出的口径，当前 `v1.0` 候选范围收口为六项：

1. `2个系统整合`
2. `小闪登录`
3. `测试报告完善`
4. `AI 开放接入能力`
5. `接口自动化`
6. `Agent 自动化`

## 3. 六项范围的产品化解释

### 3.1 两个系统整合

不是简单把两个页面拼在一起，而是统一：

1. 登录入口
2. 项目 / 需求接入入口
3. 样本 / 用例资产模型
4. 执行入口
5. Trace 归集
6. 报告出口

交付标准应是：

`同一平台壳下能承接两类能力，不再需要两套独立产品叙事`

### 3.2 小闪登录

小闪登录在 v1.0 里不是边缘功能，而是平台统一入口。

它解决的是：

1. 统一身份入口
2. 统一访问链路
3. 后续项目、报告、执行权限挂接的基础

### 3.3 测试报告完善

这项必须从“执行结果展示”升级为“质量结论输出”。

v1.0 的报告至少应覆盖：

1. 测试对象
2. 测试范围
3. 执行环境
4. 通过率与失败分布
5. 关键问题明细
6. Trace 证据
7. AI 归因判断
8. 修复建议
9. 是否建议放行

### 3.4 AI 开放接入能力

这项建议理解为平台接入层，而不是一句泛能力。

v1.0 更适合先做：

1. AI 应用对象接入规范
2. MCP / Agent / 知识检索类对象的最小接入机制
3. 后续扩展 UI、API、Workflow、Skill 的统一接入面

这项的核心价值是：

`让平台不是只测自己人内部一套东西，而是能持续接入新的 AI 对象`

### 3.5 接口自动化

这项建议作为 v1.0 第二条执行能力主线。

原因很直接：

1. UI 自动化只能覆盖表层行为。
2. 接口自动化更容易形成稳定回归能力。
3. 很多 AI 系统能力实际通过接口、工具调用、服务编排暴露。

v1.0 先收口到：

1. 传统 API
2. MCP 风格接口
3. 鉴权、参数、结构化返回校验

### 3.6 Agent 自动化

这项是平台差异化能力，不建议降级成普通测试分支。

v1.0 可先承接：

1. 任务完成率
2. 工具调用正确率
3. 参数生成正确率
4. 多步路径命中率
5. 基础 Trace 可解释性

## 4. 当前不建议塞进 v1.0 主承诺的内容

这些内容重要，但不建议成为 v1.0 必达项：

1. 真机联调全量成熟
2. 全链路数据归因平台化
3. 自动发布阻断
4. 长期线上监控告警
5. 复杂多 Agent 协同编排

原因不是它们不重要，而是会明显抬高交付复杂度，冲掉 v1.0 收口。

## 5. 建议的 v1.0 主叙事

如果要对内统一一句话，我建议先用：

`ChoKoNu v1.0 是把现有前端自动化测试能力与 AI 应用评测能力收口到同一平台中的第一版 AI质量控制平台，先完成统一登录、统一接入、统一执行、统一 Trace 与统一报告。`
