# ChoKoNu v1.0 工作台模块分析

- 项目：`连弩-AI自动化质量控制平台 / ChoKoNu`
- 模块：`工作台`
- 日期：`2026-04-28`
- 状态：`active`
- 文档类型：`模块分析文档`
- 作用：在进入模块 PRD 之前，先收敛工作台模块的产品问题、边界冲突、方案判断和后续执行前提

## 1. 本文要解决什么

当前总规划、页面框架、工作台架构稿已经写出了很多结论，但这些结论还混在：

1. 产品定义
2. 页面骨架
3. 竞品判断
4. 开发规格

之间。

这会导致一个问题：

`工作台模块看起来已经被定义过很多次，但真正进入模块级产出时，仍然没有一个先于 PRD 的分析收口。`

所以本文先回答四个问题：

1. 为什么工作台必须先做成独立模块
2. 现有文档之间有哪些冲突和漂移
3. 工作台模块真正的产品职责是什么
4. 后续 PRD、设计图、前端应该继承什么结论，不能再重新发明什么

## 2. 背景判断

### 2.1 当前不是缺页面，而是缺统一承载层

ChoKoNu 当前已经明确要承接：

1. `Web UI 自动化`
2. `MCP / Agent AI 应用评测`
3. `扣子罗盘 AI 评测能力吸收`

问题不在于“有没有页面名”，而在于这些能力如果没有统一工作台承载，就会自然滑向三种坏结果：

1. 每类能力各做一组页面，产品重新裂开
2. 每个页面各自实现导航、筛选、详情、上下文，研发重复造壳
3. 用户在需求、执行、Trace、报告之间跳转时持续丢上下文

因此，工作台模块本质上不是视觉层，而是平台承载层。

### 2.2 当前真正的用户问题

工作台模块要解决的用户问题不是“首页不够好看”，而是：

1. 登录后无法直接进入一个连续可工作的质量空间
2. 从需求到执行到 Trace 到报告的路径不连续
3. 项目、版本、环境、风险、待办等上下文散落在不同页面里
4. 用户容易被传统后台式导航拉去“看统计”，而不是“做动作”

## 3. 现有文档冲突与漂移

这是工作台模块在进入 PRD 前必须先解决的核心问题。

### 3.1 冲突一：一级导航骨架不一致

现有文档中至少存在两套一级导航口径：

#### 口径 A：6 个 Studio 骨架

来源：`docs/product/工作台架构与导航方案.md`

1. `Case Studio`
2. `Suite Studio`
3. `Run Studio`
4. `Trace Studio`
5. `Report Studio`
6. `Eval Studio`

这套口径的优点是：

1. 更像 `Coze Studio / LangSmith` 一类工作台产品
2. 结构更简洁
3. 更接近质量资产与运行态分层

但问题是：

1. 它没有显式承接 `项目接入`
2. 它没有显式承接 `需求`
3. `复核 / 门禁` 被折叠到了报告或评估语义里，不利于 `v1.0` 版本治理表达

#### 口径 B：9 个 v1.0 页面骨架

来源：`docs/review/AI质量控制平台-页面级原型框架-2026-04-27.md` 与总开发 PRD

1. `总览`
2. `项目台`
3. `需求台`
4. `用例台`
5. `执行台`
6. `Trace 台`
7. `报告台`
8. `复核台`
9. `门禁台`

这套口径的优点是：

1. 与当前 `v1.0` 主链更一致
2. 对产品、设计、研发、测试更容易拆任务
3. 显式保留了 `项目接入`、`复核`、`门禁`

但问题是：

1. 更容易滑回传统后台模块导航
2. 如果没有统一壳层约束，页面会看起来像 9 个独立系统

### 3.2 对冲突一的判断

当前阶段不能简单二选一。

更合理的收敛方式是：

1. `产品形态层` 继承 6 个 Studio 的工作台思维
2. `v1.0 交付层` 采用 9 个页面入口的主链表达

也就是说：

`6 个 Studio 是形态原则，9 个页面是首版交付分解。`

这是后续 PRD 必须明确写死的判断，否则后续设计和前端会反复漂移。

### 3.3 冲突二：总览是否应该成为大盘首页

现有文档已经多次强调：

1. 不做传统首页
2. 不做大而全总览
3. 不做企业后台式统计中心

但页面级原型里又保留了 `总览`。

这不是矛盾，前提是把 `总览` 定义清楚：

1. `总览` 只能是轻量工作入口页
2. 它的任务是恢复上下文、暴露风险和待办
3. 它不是全局分析中心

如果不先在分析阶段钉住这点，后续设计图最容易重新长成 KPI 大盘。

### 3.4 冲突三：工作台模块与业务模块的边界容易混写

现有文档里，工作台分析经常会滑进：

1. 需求详情页签
2. 用例字段
3. 执行状态
4. 报告结构

这些内容虽然相关，但不属于工作台模块本体。

工作台模块应该只负责：

1. 壳层
2. 导航
3. 上下文
4. 路由与状态
5. 跨模块跳转

不负责具体业务对象字段本体。

## 4. 行业与竞品判断

基于已有竞品研究，工作台模块应该吸收的是“产品组织方式”，而不是“业务对象命名”。

### 4.1 应吸收的部分

来自 `Coze Studio / LangSmith / Braintrust / Coze Loop` 的正确启发是：

1. `workspace shell + 左侧工作区导航 + 中央画布 + 右侧上下文`
2. 以持续工作流承接复杂任务，而不是以后台菜单树承接
3. 把 `trace / report / compare / evaluator` 组织成可跳转工作链，而不是离散后台模块

### 4.2 不应照搬的部分

不能照搬：

1. `Agent / Workflow / Plugin / Knowledge` 作为导航骨架
2. 开发平台心智替代质量平台心智
3. 把平台做成资源门户，而不是质量工作系统

### 4.3 行业最佳实践结论

工作台模块最合理的方向是：

`形态学 Coze Studio，内核学 Coze Loop / LangSmith / Braintrust，但主链表达必须服务 ChoKoNu 的质量治理。`

## 5. 工作台模块的真实职责

如果把工作台模块说成“统一前端壳层”，这还不够。

它的真实职责至少有五个：

### 5.1 承接统一上下文

统一承接：

1. 当前用户
2. 当前项目
3. 当前环境
4. 当前版本范围
5. 当前风险状态
6. 当前待办摘要

### 5.2 把主链表达成可连续操作的空间

要把：

`项目接入 -> 需求 -> 用例 -> 执行 -> Trace -> 报告 -> 复核 -> 门禁`

表达成用户能连续推进的工作空间，而不是多个独立页面。

### 5.3 约束业务模块气质

工作台必须从产品层约束后续所有页面：

1. 少做统计拼盘
2. 多做当前对象与下一步动作
3. 详情与上下文分层承载
4. 同一平台保持一致导航和状态语义

### 5.4 统一状态和异常表达

没有工作台模块，所有业务台都会各自定义：

1. `loading`
2. `error`
3. `empty`
4. `forbidden`
5. `partial data`

这会直接导致交互碎裂。

### 5.5 作为后续模块的先决约束

后续每个业务模块 PRD 都应该默认引用工作台分析结论，而不是再讨论：

1. 导航长什么样
2. 右侧面板要不要有
3. 总览要不要做成大盘
4. 详情是并排还是抽屉

## 6. 边界切割

### 6.1 本模块应收住的内容

1. 壳层结构
2. 一级导航原则
3. 顶部全局区原则
4. 右侧上下文面板原则
5. 页面骨架原则
6. 全局状态原则
7. 跨模块跳转原则
8. 响应式退化原则

### 6.2 本模块不应提前展开的内容

1. 各业务台字段级设计
2. 具体接口字段明细
3. 具体图表组件选型
4. 具体视觉样式细节
5. 具体前端组件拆分代码

## 7. 风险判断

### 7.1 最大风险不是做不出来，而是重新发散

当前工作台模块最需要防的是：

1. 后续又回到 6 页面和 9 页面二选一争论
2. 总览被做成 KPI 大盘
3. 业务模块各自造壳
4. 右侧上下文面板被取消，页面重新退回后台信息堆叠

### 7.2 对用户需求的风险

如果工作台模块只解决“页面长什么样”，没有解决“上下文连续”，那用户原问题仍然没解。

也就是说：

工作台模块的完成标准不是“有一个框架页面”，而是“用户在主链不同阶段切换时仍然知道自己在哪、接下来干什么”。

## 8. 收敛结论

### 8.1 精炼后的问题定义

工作台模块不是导航设计问题，而是：

`ChoKoNu 是否能把多源质量能力收敛成一个连续质量工作空间的问题。`

### 8.2 模块级判断

1. 工作台必须独立成模块先做。
2. 它必须先于业务台 PRD 产出。
3. 它必须同时约束产品形态和研发壳层。

### 8.3 对后续 PRD 的前置要求

后续工作台模块 PRD 必须明确写清：

1. `6 Studio 思维` 与 `9 页面交付` 的关系
2. `总览不是大盘` 的约束
3. 工作台模块与业务模块的边界
4. 全局上下文与跨模块跳转的强约束

## 9. 用户需求映射

原始用户要求：`每个模块单独做，先按规划逐条输出分析文档，再出 PRD，再出设计图，再出前端，再出前后端。先只搞一个模块。`

当前结果：

- 当前模块：`工作台`
- 当前阶段：`分析文档`
- 产品要求层：`solved`
  工作台模块在进入 PRD 前需要先解决的问题、冲突和边界已经收敛。
- 用户需求层：`partial`
  已按你要求开始单模块顺序产出，但当前只完成了第一步分析文档，尚未进入该模块 PRD、设计图和实现阶段。

## 10. 下一步

如果这份分析文档通过，下一步只做一件事：

`把工作台模块 PRD 重写/收敛为严格继承本文结论的模块 PRD。`
