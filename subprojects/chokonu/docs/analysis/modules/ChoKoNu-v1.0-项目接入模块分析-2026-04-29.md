# ChoKoNu v1.0 项目接入模块分析

- 项目：`连弩-AI自动化质量控制平台 / ChoKoNu`
- 模块：`项目接入`
- 日期：`2026-04-29`
- 状态：`active`
- 文档类型：`模块分析文档`
- 作用：在进入项目接入模块 PRD 之前，先收敛该模块的真实问题、主链职责、边界切割和首版约束

## 1. 本文要解决什么

项目接入模块看起来好像已经很明确：

1. 有项目台
2. 有接入表单
3. 有能力映射
4. 有接入校验

但真正进入模块化产出时，仍然存在一个核心风险：

`项目接入很容易被做成一个“填配置”的后台页，而不是质量主链的真实起点。`

所以本文先回答四个问题：

1. 项目接入为什么必须独立成模块
2. 它在主链里到底负责什么
3. 现有口径里有哪些容易做偏的地方
4. 后续 PRD、设计图、前端应该继承什么，不该再发散什么

## 2. 背景判断

### 2.1 当前缺的不是“项目列表”，而是“项目归属真源”

ChoKoNu 首版要统一承接：

1. `Web UI 自动化`
2. `MCP / Agent AI 应用评测`
3. `扣子罗盘评测能力吸收`

这三类能力如果没有统一项目接入，就会出现三个问题：

1. 同一个平台里不同测试对象没有统一归属
2. 后续需求、用例、执行、Trace、报告会漂成“孤立对象”
3. 工作台壳层虽然存在，但没有真实项目上下文可以承接

因此项目接入模块不是项目管理页，而是：

`质量主链的归属起点`

### 2.2 当前真正的用户问题

用户不是想“录一个仓库”这么简单。

他们真正要解决的是：

1. 某个被测对象如何被平台正式接纳
2. 这个对象属于哪类能力域
3. 这个对象是否具备进入后续质量主链的最小条件
4. 后续产出的需求、用例、执行和报告到底归谁

## 3. 当前容易做偏的地方

### 3.1 风险一：把项目接入做成静态资料卡

如果项目台只展示：

1. 仓库名
2. 分支
3. 环境
4. 状态标签

那它只是资料卡，不是接入模块。

项目接入模块必须比“看项目”更进一步，至少要承接：

1. 接入动作
2. 能力判断
3. 校验判断
4. 主链起跳

### 3.2 风险二：把接入校验做成伪动作

现有文档里已经写了 `接入校验`，但这里最容易失真。

因为很多系统会把“校验”做成：

1. 一个按钮
2. 一个绿色标签
3. 一个“已通过”的静态说明

但对 ChoKoNu 来说，接入校验的价值不是形式化按钮，而是确认：

1. 项目是否可被平台定位
2. 项目是否可被后续主链承接
3. 该项目是否已经具备进入需求台的条件

也就是说：

`接入校验必须是项目能否进入质量主链的最小 gate，而不是 UI 装饰。`

### 3.3 风险三：能力映射可能被做成标签堆叠

当前文档里要求项目接入支持能力映射，这是对的。

但如果只把能力映射做成多个 tag：

1. `UI 自动化`
2. `Agent`
3. `MCP`
4. `AI Eval`

那它只是分类，不是映射。

真正有价值的能力映射应该回答：

1. 这个项目默认走哪条质量链
2. 这个项目能进入哪些需求台/用例台表达
3. 这个项目后续应绑定哪些执行器、Trace 表达和报告结论

因此能力映射不是展示性字段，而是：

`主链分流与后续工作台默认视图的前置规则`

### 3.4 风险四：项目接入与工作台壳层重复承接上下文

工作台模块已经负责：

1. 当前项目
2. 当前环境
3. 当前版本范围
4. 当前工作区上下文

项目接入模块如果又开始重复定义整套壳层逻辑，就会与工作台模块冲突。

因此项目接入模块只能负责：

1. 项目对象本身
2. 项目接入表单
3. 项目校验
4. 从项目进入需求台

而不能重新定义：

1. 顶部壳层
2. 左侧导航
3. 通用状态规则

## 4. 主链职责判断

项目接入位于主链最前：

`项目接入 -> 需求 -> 用例 -> 执行 -> Trace -> 报告 -> 复核 -> 门禁`

它至少承担四个职责：

### 4.1 定义“谁是被测对象”

平台后续所有质量资产都必须能回挂到这里。

### 4.2 定义“这个对象属于哪类能力域”

即：

1. UI 自动化
2. MCP / Agent AI 应用评测
3. AI Eval / 评测能力吸收

### 4.3 定义“这个对象能否进入主链”

也就是通过接入校验判定是否具备最小承接条件。

### 4.4 提供“主链起跳入口”

项目接入模块的完成标准不是“项目已保存”，而是：

`用户能从项目进入需求台，主链真正开始。`

## 5. 行业与产品判断

### 5.1 项目接入不是传统项目管理

不能把它做成 Jira / DevOps 那种项目配置页。

因为 ChoKoNu 里的“项目”不是组织单元，而是：

`一个被平台承接的质量对象`

### 5.2 项目接入更像“能力归属入口”

它更接近：

1. 被测对象注册
2. 能力映射绑定
3. 环境与执行入口识别
4. 进入后续质量流的前置确认

### 5.3 首版不应扩成 CI/CD 中心

项目接入里最容易发散的点是把它做成：

1. 仓库自动分析中心
2. 环境编排中心
3. CI/CD 编排台

这些都不是 `v1.0` 必须目标。

`v1.0` 只需要把“项目如何进入质量主链”收住，不需要把“项目如何被全生命周期运维”一起吞下。`

## 6. 当前前端实现暴露的问题

从现有 `frontend/src/pages/workbench/ProjectsPage.jsx` 看，已经有一个相对完整的演示壳：

1. 项目列表
2. 项目详情
3. 接入抽屉
4. 能力映射
5. 接入校验

这说明产品方向基本对了，但也暴露三个问题：

### 6.1 接入表单更像“演示编辑器”，不像“首版接入动作”

当前表单有：

1. 项目名称
2. 仓库地址
3. 分支
4. 环境
5. 安装命令
6. 启动命令
7. 能力映射

这是对的，但还缺少一个更强的产品判断：

`哪些字段是项目能否进入主链的硬门槛，哪些只是补充字段。`

### 6.2 校验结果表达偏“说明式”，还不够 gate 化

当前详情页里能看到校验说明和条目，但还没有真正把“是否可进入需求台”的判断语义做强。

### 6.3 “进入需求台”还需要明确前置条件

虽然已经有 `进入需求台` 按钮，但产品上还要先定义清楚：

1. 校验未通过能不能继续
2. 如果能继续，要给什么提示
3. 如果不能继续，在哪个层面阻断

## 7. 边界切割

### 7.1 本模块应收住的内容

1. 项目对象最小真源
2. 接入表单字段层次
3. 能力映射规则
4. 接入校验最小 gate
5. 从项目进入需求台的跳转规则

### 7.2 本模块不应提前展开的内容

1. 复杂环境编排
2. 真实执行器调度
3. 仓库深度扫描能力
4. 自动生成需求的细节
5. 用例资产结构

## 8. 收敛结论

### 8.1 精炼后的问题定义

项目接入模块不是“项目管理页”，而是：

`ChoKoNu 如何确认一个被测对象能被正式纳入质量主链的入口模块。`

### 8.2 模块级判断

1. 它必须独立成模块。
2. 它是工作台之后第二个必须收住的模块。
3. 它的核心不是展示项目，而是决定项目能否进入主链。

### 8.3 对后续 PRD 的前置要求

后续项目接入模块 PRD 必须明确写清：

1. 接入字段的硬门槛与补充字段分层
2. 接入校验的 gate 语义
3. 能力映射如何影响后续主链表达
4. 未通过校验时，从项目到需求台的放行规则

## 9. 用户需求映射

当前总要求是：`每个模块单独做，按 分析文档 -> PRD -> 设计图 -> 前端 -> 前后端 联调规格 逐条产出。`
