# 两个方向合并后的产品定义

## 1. 结论先行

当前不应继续把能力拆成两个并列产品：

1. 前端自动化测试平台
2. AI 应用评测平台

更合理的收口方式是：

`统一升级为 AI质量控制平台`

一句话定义：

`一个用 AI 驱动需求理解、测试资产生成、自动执行、Trace 观测、结果判定、风险报告和上线门禁的统一质量控制平台。`

## 2. 为什么要合并

这两个方向虽然入口不同，但主问题其实相同：

1. 都要先接收需求、规格、场景或能力说明。
2. 都要把输入转成可复用的测试资产。
3. 都要经过执行、采证、判定、报告和复盘。
4. 都不是一次性跑脚本，而是要沉淀回归资产和版本判断依据。

所以真正应该统一的，不是页面名字，而是质量主链、资产模型和治理机制。

## 3. 正确的合并方式

### 3.1 主架构按质量流程组织

主架构应按下面这条链组织：

`需求/标准 -> 样本/用例 -> 评测集 -> 执行 -> Trace -> 报告 -> 复核 -> 门禁`

这条链决定平台是不是平台，而不是一组零散工具。

### 3.2 能力域按插件方式挂接

下面这些能力不应升格成一级产品骨架，而应作为对象类型、执行器或评估器挂入：

1. `UI`
2. `API`
3. `数据链路`
4. `真机`
5. `RAG`
6. `Agent`
7. `MCP`
8. `Workflow`
9. `Skill`

也就是说：

`按流程设计，按能力扩展`

而不是：

`按测试类型裂产品`

## 4. 推荐的产品分层

### 4.1 流程层

负责把质量工作串成闭环：

1. 需求接入
2. AI 理解与测试目标提炼
3. 样本或用例生成
4. 评测集编排
5. 执行调度
6. Trace 与证据归集
7. 结果判定
8. 报告输出
9. 人工复核
10. 版本门禁

### 4.2 资产层

负责统一复用模型：

1. `Requirement`
2. `Standard`
3. `Case`
4. `Suite`
5. `Run`
6. `Trace`
7. `Report`
8. `Review`
9. `Baseline`
10. `Gate`

### 4.3 执行层

负责多执行器挂接：

1. `Playwright / Web UI Executor`
2. `API Executor`
3. `Data Chain Executor`
4. `Mobile Device Executor`
5. `Agent Executor`
6. `MCP Executor`
7. `RAG Eval Executor`

### 4.4 判定层

统一承接三类判断：

1. 规则断言
2. AI Judge
3. 人工复核

### 4.5 治理层

负责让测试结论变成版本决策：

1. 版本对比
2. 红线规则
3. 放行建议
4. 回归集沉淀
5. 风险样本回流

## 5. 对 ChoKoNu 的具体影响

这次合并判断对 ChoKoNu 的影响，不是“换项目”，而是“正式确认项目定义”。

当前应明确：

1. ChoKoNu 不是旧埋点验收工具的放大版。
2. ChoKoNu 也不是 Agent Builder 或大而全 AI 中台。
3. ChoKoNu 是统一 AI 质量控制面。
4. 原前端自动化测试能力是其中一个能力域。
5. 原 AI 应用评测能力是另一个能力域。
6. 两者共用一套资产、执行、Trace、报告和门禁底座。

## 6. 首版收口建议

### 6.1 首版必须收住的主链

首版先做：

1. `需求/标准` 入口
2. `Case / Suite` 资产入口
3. `Run` 执行入口
4. `Trace` 证据归集
5. `Report` 报告输出
6. `Review / Gate` 复核与放行建议

### 6.2 首版先接入的对象类型

优先级建议：

1. `UI`
2. `MCP`
3. `Agent`
4. `知识检索 / RAG`

这些对象最容易形成“统一质量控制面”的辨识度。

### 6.3 二阶段再扩的能力

放到二阶段更稳：

1. 真机联调
2. 数据全链路归因
3. 自动发布阻断
4. 长期线上监控告警
5. 复杂多 Agent 协同

## 7. 风险与边界

当前最需要防的不是技术做不出来，而是产品边界再次发散。

主要风险：

1. 按对象类型各做一套页面，主链被拆碎。
2. 讨论重新退回单点埋点工具心智。
3. 一期同时吞下 UI、API、真机、MCP、评测、监控，结果没有一个闭环完整。
4. 报告只停留在“跑过没跑过”，没有版本判断和放行价值。

## 8. 最终判断

本次合并的正确结论不是：

`把两个平台拼起来`

而是：

`把两个方向降为能力域，把质量主链升为产品主骨架`

对应到 ChoKoNu，当前结果判断为：

1. 用户需求层：`solved`
   用户想确认两个方向是否该合并，以及怎么合并，这个问题已经有清晰答案。
2. 产品要求层：`partial`
   合并方向已清楚，但还需要继续把命名、首版边界、页面骨架和版本口径统一到活跃真源里。
