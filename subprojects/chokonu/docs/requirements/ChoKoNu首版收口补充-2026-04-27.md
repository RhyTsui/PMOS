## 1. 收口结论

ChoKoNu 当前不再理解为两个并列平台：

1. 前端自动化测试平台
2. AI 应用评测平台

正式收口为：

`AI质量控制平台`

这个平台的主骨架不是测试类型，而是统一质量主链：

`需求/标准 -> 样本/用例 -> 评测集 -> 执行 -> Trace -> 报告 -> 复核 -> 门禁`

## 2. 架构原则

### 2.1 主骨架按流程设计

产品骨架固定围绕质量流程展开，不按对象类型拆成多套产品区。

### 2.2 能力按插件扩展

下面这些能力作为对象类型、执行器或评估器挂接：

1. `UI`
2. `API`
3. `数据链路`
4. `真机`
5. `RAG`
6. `Agent`
7. `MCP`
8. `Workflow`
9. `Skill`

正式判断：

`按流程设计，按能力扩展`

## 3. 首版边界

### 3.1 首版必须完成

1. 需求与标准入口
2. 样本/用例入口
3. 评测集入口
4. 执行任务入口
5. Trace 归集
6. 报告输出
7. 复核挂点
8. 门禁建议

### 3.2 首版优先接入对象

1. `UI`
2. `MCP`
3. `Agent`
4. `知识检索 / RAG`

原因是这四类对象最容易形成统一质量控制面的首版辨识度。

### 3.3 二阶段扩展

1. 真机联调
2. 复杂数据归因链路
3. 自动发布阻断
4. 长期线上监控告警
5. 复杂多 Agent 协同

## 4. 对旧输入的处理

历史埋点验收、前端自动化和 AI 评测相关输入继续保留价值，但角色改为：

1. 样本来源
2. 方法输入
3. 对象类型专项能力来源

它们不再单独定义 ChoKoNu 的产品边界。

## 5. 页面与模块收口建议

如果后续继续做页面或原型，建议优先围绕下面这些工作台展开：

1. `Case Studio`
2. `Suite Studio`
3. `Run Studio`
4. `Trace Studio`
5. `Report Studio`
6. `Eval / Gate Studio`

对象类型只作为筛选维度、方法维度和结果解释维度出现，不升格为一级导航。

## 6. 当前结果判断

1. 用户需求层：`solved`
   已经明确两个方向应该合并，且应如何合并。
2. 产品要求层：`partial`
   还需要继续把命名、首版页面骨架和版本评审口径同步到更多活跃真源文档。
