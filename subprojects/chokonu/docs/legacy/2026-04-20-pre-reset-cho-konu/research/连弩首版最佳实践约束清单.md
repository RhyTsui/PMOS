## 1. 这份清单解决什么问题

这份文档不是再做一轮行业研究，而是把已经确认的外部最佳实践压成 `连弩` 首版必须遵守的约束。

它回答的是：

`连弩首版哪些事情必须做，哪些可以延后，哪些不能做偏。`

## 2. 首版定位约束

### 2.1 连弩首版不是单点脚本工具

连弩首版不能只停留在：

1. 单次埋点验收
2. 临时脚本执行
3. 手工截图比对
4. 零散测试记录

连弩首版必须至少形成：

`规则/字段定义 -> 样本集 -> 自动执行 -> 测试报告 -> 异常 Trace`

### 2.2 连弩首版也不是大而全平台

首版不追求一次性做完：

1. 全量 AI 语义评测
2. 模型裁判/自动评分全套
3. 完整 CI/CD 质量门禁
4. 全链路线上观测治理平台

这些可以作为第二阶段能力。

## 3. 首版必须具备的 5 个约束

### 3.1 必须同时覆盖“字段契约”和“行为规则”

不能只验证字段名、类型、枚举、JSON 结构。

还必须覆盖：

1. 事件触发时机
2. 页面进入/退出/切后台/回退等行为链
3. 曝光与点击顺序
4. 数据是否正确落表

执行要求：

1. 样本要拆成 `字段契约类` 和 `行为规则类`
2. 报告里必须区分：
   - 漏埋
   - 错埋
   - 业务口径错误

### 3.2 必须有“样本资产”，不能靠临时 case

首版必须沉淀可复用资产，而不是每次需求来了临时写脚本。

最少要有：

1. 基础样本集
2. 回归执行集
3. 高频场景样本

执行要求：

1. 每个样本必须标明来源、场景、预期、校验方式
2. 样本要能复用到后续版本，不是一次性消耗品

### 3.3 自动化执行不是终点，报告必须是正式交付

首版不能只输出“跑过/没跑过”。

必须产出正式测试报告，至少包含：

1. 本轮执行范围
2. 通过/失败样本统计
3. 失败原因分类
4. 异常 Trace 入口
5. 待回归项

执行要求：

1. 报告是正式交付物，不是附属日志
2. 项目验收和版本判断必须看报告，不只看执行记录

### 3.4 异常必须可追溯，Trace 不是可选项

连弩首版必须能把失败样本和异常链路挂起来。

至少要能关联：

1. 样本 ID
2. 执行时间
3. 事件/页面上下文
4. 失败规则
5. 原始日志或关键信息片段

执行要求：

1. 报告里的失败项必须能跳到 Trace
2. 不能只有一句“失败了”，没有排查入口

### 3.5 必须为“线上预警”和“回归复用”预留结构

首版可以不把线上预警完全做完，但结构上必须留口。

至少要预留：

1. 失败类型标准化
2. 样本与指标映射
3. 执行结果可对比
4. 可挂接版本号/构建号

执行要求：

1. 不允许首版把数据结构做成一次性结果页
2. 必须考虑后续版本对比和回归复用

## 4. 首版建议优先级

### 4.1 P0：必须收口

1. `字段契约样本`
2. `行为规则样本`
3. `自动执行集`
4. `测试报告`
5. `异常 Trace`

### 4.2 P1：建议同时预埋

1. `线上预警结构`
2. `版本对比结构`
3. `回归集沉淀规则`

### 4.3 P2：第二阶段再补

1. `AI语义评测`
2. `模型裁判/评分器`
3. `CI/CD质量门禁`

## 6. 一句话结论

连弩首版的正确方向不是“先做一个大平台”，而是先把：

`字段契约 + 行为规则 + 样本资产 + 自动执行 + 正式报告 + 异常Trace`

这条最小闭环做实。
