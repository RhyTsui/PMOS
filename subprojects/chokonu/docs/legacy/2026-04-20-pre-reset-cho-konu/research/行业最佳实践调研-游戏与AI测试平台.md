# 行业最佳实践调研：游戏与 AI 测试平台

- 版本：v0.1
- 日期：2026-04-20
- 状态：Draft
- 项目：ChoKoNu / 连弩-AI测试平台

## 1. 研究目标

这份文档不是为了罗列公司，而是为了回答一个更实际的问题：

`连弩` 这条线到底该借鉴哪些已经被验证过的行业做法。

当前重点参考：

1. 三七互娱
2. 网易
3. 阿里系优酷
4. 京东

## 2. 核心结论

当前公开资料显示，行业里比较成熟的实践并不是“做一个大而全的平台”，而是先把下面四层打稳：

1. 埋点/测试方案标准化
2. 自动化执行集与批量回归
3. 报告、Trace、线上预警闭环
4. 把 AI 特有的不确定性测试纳入数据集、评测指标和 CI/CD 门禁

对 `连弩` 的直接启发是：

- 旧的“埋点验收”应该保留
- 但必须升级成 `规则校验 + 自动执行 + 报告 + 线上预警 + AI评测集`
- 平台首版不要只做页面，也不要只做脚本

## 3. 公司样本

### 3.1 三七互娱

#### 已确认事实

从三七互娱公开招聘信息看，他们对 AI 类测试的公开能力要求已经非常明确：

1. 功能、性能、可靠性、数据质量
2. AI 特有维度，例如输出一致性、幻觉识别、提示词鲁棒性、偏见与公平性
3. 自动化测试框架与工具链
4. 评测数据集、评测基准和效果度量体系
5. 将 AI 测试嵌入 CI/CD，形成端到端质量门禁

来源：

- 三七互娱官方招聘：资深测试开发工程师  
  https://zhaopin.37.com/index.php?a=post_info&c=society&id=11039708&m=Home

#### 对 ChoKoNu 的启发

这里虽然不是完整技术方案，但已经足够说明三七互娱的目标模型是：

`数据集 + 度量体系 + 自动化框架 + CI/CD门禁`

这意味着对我们来说，`连弩` 不应该只做“埋点验证工具”，而应该至少具备：

1. 样本集
2. 自动校验
3. 结果度量
4. 版本门禁

#### 判断

三七互娱更像是在走“AI 测试工程化平台”路线，而不是单点脚本路线。  
这是基于官方招聘职责做的推断，不是对其内部系统细节的直接确认。

### 3.2 网易

#### 已确认事实 A：Airtest / Poco / 云真机

网易公开的 Airtest Project 已经形成比较完整的自动化测试工具链：

1. 跨平台 UI 自动化 IDE
2. 支持 Android / iOS / Windows
3. 支持录制、回放、报告查看
4. 结合图像识别和 UI 层级树
5. 官方教程明确提到网易内部已在多个项目中使用，并配合移动云实验室做大规模测试

来源：

- Airtest 官方站  
  https://airtest.netease.com/
- Airtest 官方教程  
  https://airtest.netease.com/tutorial/Tutorial.html

#### 已确认事实 B：网易严选埋点质量保障

网易严选公开分享过埋点质量保障实践，核心点很清楚：

1. 埋点不是只靠一种方式，采用“自动化埋点 + 代码埋点”的平衡方案
2. 通过任务流和流程化管理贯穿埋点定义、开发、测试、上线
3. 埋点质量保障里明确包含：
   - 手动测试
   - 自动化测试
   - UI 测试
   - 数据校验规则
   - 线上预警
4. 自动化测试针对批量回归，建立埋点执行集
5. 线上预警用于发现“测试通过但线上场景仍异常”的问题

来源：

- InfoQ：网易严选流量体系建设实践  
  https://www.infoq.cn/article/VrJIq8xRjDPTQSHucgpK

#### 对 ChoKoNu 的启发

网易给出的最佳实践最完整，尤其适合 `AI埋点验收` 这条线：

1. 不能只有手工验收
2. 不能只有自动化执行
3. 不能没有线上预警
4. 执行集必须存在，回归必须批量化
5. UI 自动化、数据校验、线上监控必须是一条链

#### 判断

如果 `连弩` 只做“执行+报告”，还不够。  
按网易这条线，更合理的首版结构应是：

`样本集 -> 执行集 -> 自动校验 -> 报告 -> 线上预警`

### 3.3 阿里系优酷

#### 已确认事实

优酷公开分享的埋点质量保障，重点在“三步曲”：

1. 从流程、测试方案、业务深入度三个方面改造
2. 常见埋点问题包括：
   - 大问题偶尔有
   - 小问题不断
3. 通过系统化方案提升客户端埋点质量

来源：

- InfoQ：优酷客户端埋点质量保障三步曲  
  https://www.infoq.cn/article/pxkVQ5jB4q7efYV2Cw5C

#### 对 ChoKoNu 的启发

优酷的价值不在于工具细节，而在于一个组织经验：

`埋点质量问题不会自然消失，必须用流程 + 方案 + 业务理解三层一起压。`

这对 `连弩` 的含义是：

1. 不能只看日志字段
2. 也要看业务语义是否对
3. 报告里必须区分“漏埋 / 错埋 / 业务口径错”

### 3.4 京东

#### 已确认事实

京东公开的埋点质量保障实践更偏“验收和用例体系”：

1. 提测后要做上报规则验证和字段验证
2. 验证完成要出测试报告
3. 验收时不仅看测试记录，还要做落表验证
4. 字段验证重点包括：
   - 字段名和字段类型
   - 参数长度和枚举
   - 嵌套 JSON 结构
5. 上报规则用例要覆盖：
   - 页面进入
   - 回退
   - 快速离开
   - 下拉刷新
   - 切后台/锁屏
   - 点击
   - 曝光

来源：

- 京东云技术团队：浅谈埋点及其质量保障  
  https://www.cnblogs.com/jingdongkeji/p/17848555.html

#### 对 ChoKoNu 的启发

京东给我们的最直接启发是：

`验收不能只有字段校验，还要有行为规则校验。`

所以 `连弩` 里的历史埋点验收样本，最适合先拆成两类：

1. 字段契约类
2. 行为规则类

## 4. 行业共性做法

综合上面四类样本，可以抽出 6 个稳定模式：

### 4.1 测试对象不只是字段

除了字段名、类型、枚举，还要覆盖：

1. 事件顺序
2. 页面跳转链
3. 曝光与点击时机
4. 数据落表结果

### 4.2 必须有执行集 / 评测集

行业成熟做法都不靠“临时跑一下”，而是沉淀成：

1. 执行集
2. 回归集
3. 长期复用的样本资产

### 4.3 自动化只是中间层

成熟体系一般都不是“自动化脚本 = 全部平台”，而是：

1. 样本定义
2. 自动执行
3. 校验规则
4. 报告
5. 线上监控/预警

### 4.4 AI 场景必须加新维度

从三七互娱的官方岗位信息可以看到，AI 场景里除了传统测试，还要补：

1. 幻觉
2. 输出一致性
3. 提示词鲁棒性
4. 偏见与公平性
5. 上下文依赖异常

### 4.5 报告是正式交付，不是附属产物

京东和网易都强调：

1. 测完要出报告
2. 验收要看报告
3. 线上异常还要继续监控

### 4.6 质量门禁必须进发布流程

三七互娱公开岗位已明确提到：

`AI 测试能力嵌入 CI/CD，形成端到端质量门禁`

这说明 `连弩` 后续必须和版本、发布、工作流接起来。

## 5. 对连弩的直接建议

### 5.1 首版必须做

1. `字段契约样本`
2. `行为规则样本`
3. `自动执行集`
4. `测试报告`
5. `异常 Trace`

### 5.2 首版最好做

1. `线上预警`
2. `版本对比`
3. `回归集沉淀`

### 5.3 第二阶段补

1. `AI 语义评测`
2. `模型裁判/评分器`
3. `CI/CD 质量门禁`

## 6. 结论

对 `AI埋点验收 / 连弩` 这个项目来说，行业最佳实践给出的方向不是“再造一个大平台”，而是先把下面这条最小闭环打通：

`规则/字段定义 -> 样本集 -> 自动执行 -> 报告 -> 线上预警 -> 回归复用`

如果再加上 AI 场景要求，后续再往上长：

`规则校验 -> 行为校验 -> AI评测集 -> 自动评分 -> 发布门禁`
