# 连弩页面流转与关键状态

- 版本：v0.1
- 日期：2026-04-20
- 状态：Draft
- 项目：连弩-AI测试平台 / ChoKoNu

## 1. 核心结论

连弩首版页面流转必须围绕统一主链，而不是围绕对象分类跳转。

主链固定为：

`首页概览 -> 样本中心 -> 评测集中心 -> 执行中心 -> 结果与Trace -> 报告与版本对比`

`规则与评估器` 作为贯穿式配置工作台插入主链，不作为独立主业务流起点。

## 2. 主链流转

### 2.1 首页概览 -> 样本中心

触发条件：

1. 项目刚接入，尚无样本
2. 发现某类对象样本不足
3. 需要补充回归样本

到达结果：

1. 新建样本
2. 补充红线样本
3. 补充回归样本

### 2.2 样本中心 -> 评测集中心

触发条件：

1. 样本达到可编组条件
2. 已完成 expected 和 evaluator 绑定

到达结果：

1. 新建 suite
2. 生成 baseline suite
3. 生成 regression suite
4. 生成 scenario suite

### 2.3 评测集中心 -> 执行中心

触发条件：

1. 已选定 suite
2. 已明确目标环境
3. 已明确 evaluator 组合

到达结果：

1. 发起新 run
2. 保存运行配置
3. 准备重跑

### 2.4 执行中心 -> 结果与Trace

触发条件：

1. run 执行完成
2. run 失败
3. run 出现 review-needed 结果

到达结果：

1. 查看 case result
2. 下钻失败样本
3. 查看 trace
4. 发起人工复核

### 2.5 结果与Trace -> 报告与版本对比

触发条件：

1. 当前 run 已完成评估
2. 当前 run 有可读结果摘要

到达结果：

1. 生成 run 报告
2. 查看聚合结论
3. 对比 baseline
4. 查看放行建议

## 3. 旁路流转

### 3.1 样本详情 -> 绑定 suite -> 发起 run

适用场景：

1. 新增了关键高风险样本
2. 需要快速验证单类样本

路径：

`样本详情 -> 加入指定 suite -> 发起 run`

### 3.2 失败结果 -> Trace 详情 -> 回到样本或规则修订

适用场景：

1. 某个 case 连续失败
2. 需要判断是样本问题、规则问题还是对象能力问题

路径：

`结果列表 -> Trace 详情 -> 样本详情 / 规则与评估器`

### 3.3 版本对比 -> 放行建议 -> 回到 run / suite / threshold

适用场景：

1. 当前版本恶化
2. 红线命中
3. 基线不合理

路径：

`版本对比 -> 放行建议 -> run 详情 / suite 详情 / 规则与评估器`

## 4. 页面关键状态

## 4.1 通用关键状态

所有页面必须统一支持：

1. 空状态
2. 加载状态
3. 失败状态
4. 无结果状态

## 4.2 首页概览关键状态

1. 无运行状态：提示先建立样本和 suite
2. 红线命中状态：突出显示高风险问题卡片
3. 待复核状态：突出显示 review 入口

## 4.3 样本中心关键状态

1. 无样本状态：提示创建第一批样本
2. 无筛选结果状态：提示放宽筛选条件
3. 样本信息不完整状态：缺少 expected、risk flag 或 evaluator 绑定

## 4.4 评测集中心关键状态

1. 无 suite 状态：提示从样本创建
2. 无 baseline 状态：提示补建基线集
3. suite 成员缺失状态：提示补齐样本

## 4.5 执行中心关键状态

1. 未运行状态：暂无 run 记录
2. 运行中状态：显示进度和最近更新
3. 运行失败状态：给出失败样本入口
4. 多 trial 状态：展示 trial 维度摘要

## 4.6 结果与Trace 关键状态

1. 全通过状态：可快速跳转报告
2. 失败集中状态：按 failure stage 聚合
3. 待复核状态：高亮 review-needed
4. Trace 缺失状态：提示采集缺口

## 4.7 报告与版本对比关键状态

1. 无报告状态：提示先完成 run
2. 无 baseline 状态：提示建立可对比基线
3. 红线命中状态：标记不可直接放行
4. 版本恶化状态：展示恶化样本和核心指标下降

## 4.8 规则与评估器关键状态

1. 无规则配置状态：提示配置首批 evaluator
2. 阈值冲突状态：提示配置存在冲突
3. 红线变更影响状态：展示受影响 suite 和 run

## 5. 状态驱动的动作规则

### 5.1 遇到空状态时

必须给出下一步动作入口，而不是只显示空白提示。

### 5.2 遇到失败状态时

必须给出重试或回退入口，并说明失败发生在数据加载、执行还是判定阶段。

### 5.3 遇到 review-needed 状态时

必须给出明确人工复核入口，不允许只显示模糊状态标签。

### 5.4 遇到红线命中状态时

必须在概览页、结果页、报告页三处保持一致提示，避免结论冲突。

## 6. 一句话结论

连弩首版页面流转要让用户沿着“建样本、组 suite、发 run、查 trace、看报告、做版本判断”的路径前进，而关键状态设计的目标是让异常和高风险问题始终有明确入口和回路。
