# 评估器分类与执行草案

- 版本：v0.1
- 日期：2026-04-20
- 状态：Draft
- 项目：ChoKoNu / 连弩-AI测试平台

## 1. 核心结论

ChoKoNu 首版的评估器不能只做一个“大打分器”，而应拆成可组合执行的评估层。

首版建议固定四类评估器：

1. 规则评估器
2. 契约评估器
3. 语义评估器
4. 人工复核评估器

## 2. 事实

当前项目输入已经明确：

1. 平台要同时覆盖评测集、评估器、自动化测试和报告
2. 历史埋点验收场景更偏规则校验和数据一致性校验
3. Demo 规划说明系统不仅做执行，还要做结果解释与报告

## 3. 评估器分类建议

### 3.1 规则评估器

用途：

1. 字段存在性
2. 枚举合法性
3. 值域校验
4. 阈值判断

适合：

- 埋点验收
- 数据一致性
- 指标门禁

### 3.2 契约评估器

用途：

1. MCP/tool/schema 契约一致性
2. 参数必填项校验
3. 返回结构校验

适合：

- MCP 工具测试
- API 契约测试

### 3.3 语义评估器

用途：

1. 语义回答正确性
2. 工具路线是否合理
3. 结果解释是否满足业务期望

适合：

- 问数场景
- Agent 输出场景

### 3.4 人工复核评估器

用途：

1. 处理无法自动判定的高风险样本
2. 处理边界争议样本
3. 作为自动评估失效时的兜底

## 4. 执行建议

### 4.1 执行顺序

首版推荐执行顺序：

1. 先跑规则/契约评估
2. 再跑语义评估
3. 最后把高风险样本送人工复核

### 4.2 执行接口抽象

建议统一为：

| 字段 | 说明 |
|---|---|
| `evaluator_id` | 评估器唯一标识 |
| `evaluator_type` | 规则/契约/语义/人工 |
| `input_ref` | 评测样本引用 |
| `output_ref` | 执行结果引用 |
| `score` | 分值 |
| `decision` | pass/fail/review |
| `reason` | 判定依据 |

## 5. 首版边界

首版不追求把所有 AI 质量问题都自动判完。

首版重点是：

1. 让规则类和契约类先自动化
2. 让语义类先形成稳定接口
3. 让人工复核只处理少量必要样本

## 6. 不确定项

1. 语义评估首版是否直接依赖模型裁判
2. 人工复核工作台是否进入首版范围
3. 分值体系是百分制还是通过/失败制优先

## 7. 需要主管决策的问题

1. ChoKoNu 首版是否优先做规则/契约评估器
2. 语义评估器首版是否允许只输出 `review-needed`
3. 人工复核是否进入首版页面范围
