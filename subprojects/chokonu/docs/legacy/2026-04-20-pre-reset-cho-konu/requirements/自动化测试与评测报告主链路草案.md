# 自动化测试与评测报告主链路草案

- 版本：v0.1
- 日期：2026-04-20
- 状态：Draft
- 项目：ChoKoNu / 连弩-AI测试平台

## 1. 核心结论

ChoKoNu 首版最需要稳定的不是“更多页面”，而是自动化测试到报告产出的主链路。

首版主链路建议固定为：

`选择评测集 -> 发起执行 -> 记录 Trace -> 运行评估器 -> 汇总结果 -> 生成人类可读报告 -> 回流回归集`

## 2. 事实

当前已有输入已经明确：

1. Trace 是首版必选能力，不是附属日志
2. 执行结果需要支持版本对比和风险提示
3. 历史埋点验收价值在于成为自动化样本，而不是继续作为独立平台范围

## 3. 自动化测试主链路

### 3.1 发起执行

执行前需要固定：

1. 评测集
2. 运行配置
3. 模型或工具目标
4. 评估器组合

### 3.2 执行阶段

执行阶段至少保留：

1. `run_id`
2. `case_id`
3. 状态
4. 开始/结束时间
5. 原始输出
6. Trace 引用

### 3.3 失败定位

失败定位至少区分：

1. 调用失败
2. 契约失败
3. 规则失败
4. 语义失败
5. 人工复核未通过

## 4. 报告主链路

### 4.1 报告最小结构

| 模块 | 内容 |
|---|---|
| 概览 | 执行范围、通过率、失败率 |
| 结果分布 | 各类失败占比 |
| 重点失败样本 | Top 风险样本 |
| Trace 引用 | 可回放或追踪入口 |
| 版本对比 | 与上一版本差异 |
| 建议动作 | 是否允许发布、需补什么 |

### 4.2 报告用途

报告不是展示层装饰，而是：

1. 研发复盘输入
2. 质量门禁输入
3. 回归集回流入口

## 5. 首版判断

### 判断

首版报告先做“单次执行报告 + 基础版本对比”就够，不要一开始就做大而全治理看板。

### 理由

1. 先把单次执行链跑通，才有稳定资产
2. 没有稳定执行与 Trace，治理看板只是空壳

## 6. 不确定项

1. 首版报告导出优先 Markdown、HTML 还是两者都要
2. 首版是否支持按业务线生成不同模板
3. Trace 查看是内嵌还是跳转

## 7. 需要主管决策的问题

1. ChoKoNu 首版是否把版本对比列为必做
2. 报告导出格式首版优先哪一种
3. 发布门禁是否直接接入首版主链
