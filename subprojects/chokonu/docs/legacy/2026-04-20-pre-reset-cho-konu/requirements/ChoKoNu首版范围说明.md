# ChoKoNu 首版范围说明

- 版本：v0.1
- 日期：2026-04-20
- 状态：Draft
- 项目：连弩-AI测试平台
- 英文名：ChoKoNu
- 主要来源：`docs/sources/inbox/01-项目规划.md`、`02-架构规划.md`、`03-功能需求.md`、`05-数据库表说明.md`

## 1. 核心结论

ChoKoNu 首版应先聚焦一个可运行、可复盘、可沉淀资产的评测闭环，而不是直接做成大而全的 AI 中台。

首版主链路应固定为：

`维护用例 -> 组织评测集 -> 发起执行 -> 查看结果与 Trace -> 生成报告 -> 基于结果回流样本`

首版优先落地六个核心模块：

1. 用例
2. 评测集
3. 执行
4. Trace
5. 报告
6. 配置

## 2. 首版目标

首版目标不是覆盖所有 AI 研发活动，而是把评测平台最小闭环做稳：

1. 让评测样本成为长期资产，而不是散落在脚本或临时文档里
2. 让同一批样本可复用、可回归、可对比
3. 让执行结果具备 Trace 和失败定位能力
4. 让报告成为复盘和交付出口
5. 逐步提升用例生成与校验自动化比例，减少人工逐条处理

## 3. 首版范围

### 3.1 用例资产

支持两类用例并保持边界清晰：

1. 契约用例 `contract_cases`
2. 语义用例 `semantic_cases`

首版至少支持：

1. 新建、编辑、查询
2. 基础导入导出
3. 基于 schema 或现有规则生成基础用例骨架
4. 维护期望工具、参数、答案要点等核心字段

### 3.2 评测集

首版把 `test_suites` 作为可复用执行单元，至少支持：

1. 新建和编辑评测集
2. 维护 case 列表
3. 区分契约评测集和语义评测集
4. 从运行结果中回流失败样本或回归样本

### 3.3 执行

首版把 `eval_runs` 作为主轴对象，至少支持：

1. 发起执行
2. 跟踪进度和状态
3. 查看单样本结果
4. 重跑全部或失败样本
5. 基础人工复核

### 3.4 Trace

Trace 是首版必选能力，不是附属日志。首版至少保留：

1. Trace 主记录
2. 原始 Trace
3. 规范化 Trace
4. Span 明细
5. 失败阶段和错误信息

### 3.5 报告

首版至少支持：

1. 单次执行报告
2. 基础聚合报告
3. Markdown 或 HTML 产物导出

### 3.6 配置

首版至少支持运行时配置切换：

1. 模型
2. MCP Profile
3. 执行器类型
4. 并发与超时
5. 评分与系统提示配置

## 4. 非首版范围

以下内容暂不应作为首版主目标：

1. 大规模多人协作治理
2. 完整前端工程重构
3. 分布式任务调度体系
4. 多数据库和多租户治理
5. 泛化成覆盖所有 AI 中台能力的超级平台

## 5. 首版数据对象

根据 demo 系统规划，建议把以下对象作为首版领域事实：

| 对象 | 作用 |
|------|------|
| `contract_cases` | MCP 契约测试样本 |
| `semantic_cases` | 端到端语义测试样本 |
| `test_suites` | 评测集 |
| `eval_runs` | 执行任务 |
| `qa_run_case_results` | 单样本执行结果 |
| `traces` / `normalized_traces` / `normalized_trace_spans` | Trace 子系统 |
| `reports` | 报告产物 |

主关系链：

`cases -> test_suites -> eval_runs -> qa_run_case_results -> traces -> reports`

## 6. 自动化优先级

从 inbox 里的 demo 规划看，当前最重要的缺口不是页面数量，而是自动化比例不足。

首版后续演进优先级应固定为：

1. 用例生成自动化
2. 期望值补全自动化
3. 差异比对与自动判定
4. 仅把高风险样本留给人工复核

## 7. 依据来源

1. `01-项目规划.md` 明确了六模块和主链路
2. `02-架构规划.md` 明确了系统分层、执行链路和 Runner 模型
3. `03-功能需求.md` 明确了角色视角和模块需求
4. `05-数据库表说明.md` 明确了核心表、主关系链和自动化落点

## 8. 不确定项

1. 当前是否沿用 demo 系统现有前后端与数据库设计，还是仅继承领域模型
2. 首版优先支持 MCP 契约评测、语义问答评测，还是两者同时推进
3. 报告模板是否按业务线拆分，还是先统一模板
4. 人工复核工作台是否进入首版页面范围

## 9. 影响范围

这份范围定义会直接影响：

1. README 中的平台定位
2. 后续 `评测集设计草案`
3. `评估器分类与执行草案`
4. `自动化测试与评测报告主链路草案`
5. 是否继续复用 demo 系统的数据对象命名

## 10. 需要主管决策的问题

1. ChoKoNu 首版是否明确继承 demo 系统的“契约用例 + 语义用例”双轨设计
2. `tracking-acceptance` 目录下是否继续保留历史命名路径，还是后续再统一目录名
3. 首版要不要把人工复核工作台定义为必做模块
4. 首版数据库是否允许先按 SQLite 单机方案推进
