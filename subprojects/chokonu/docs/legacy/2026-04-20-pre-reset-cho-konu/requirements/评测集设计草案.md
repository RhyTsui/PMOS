# 评测集设计草案

- 版本：v0.1
- 日期：2026-04-20
- 状态：Draft
- 项目：ChoKoNu / 连弩-AI测试平台

## 1. 核心结论

ChoKoNu 首版的评测集不应该只是“把旧埋点验收脚本搬进来”，而应该作为长期可复用的测试资产层。

首版建议固定三层结构：

1. `case`：最小测试样本
2. `suite`：同类场景集合
3. `baseline/regression pack`：可复盘、可对比的回归集

## 2. 事实

### 2.1 已有输入事实

来自现有 demo 规划和背景文档，当前已明确：

1. 系统会同时支持契约用例和语义用例
2. 评测集需要作为可复用执行单元
3. 历史埋点验收内容可以沉淀为样本，不再定义项目边界

### 2.2 当前首版主链路事实

首版主链路已经收敛为：

`用例 -> 评测集 -> 执行 -> Trace -> 报告`

## 3. 设计建议

### 3.1 样本类型

首版先支持两大类样本：

1. `contract_case`
用途：
校验 MCP/tool/schema/参数/返回结构是否符合契约

2. `semantic_case`
用途：
校验真实业务问题在语义层面的回答质量、路线选择和结果可用性

### 3.2 评测集类型

首版建议把评测集固定成三种用途：

1. `baseline_suite`
用途：
定义当前能力基线，作为版本对比参考

2. `regression_suite`
用途：
把历史出错样本、线上事故样本、埋点验收样本沉淀进回归集

3. `scenario_suite`
用途：
按业务场景组织样本，例如广告报表问数、MCP 契约校验、页面交互回归

### 3.3 样本最小字段

| 字段 | 说明 |
|---|---|
| `case_id` | 样本唯一标识 |
| `case_type` | 契约/语义 |
| `scenario` | 所属场景 |
| `input` | 输入内容 |
| `expected` | 预期输出或预期行为 |
| `executor_type` | 执行器类型 |
| `evaluator_refs` | 绑定评估器 |
| `trace_required` | 是否必须保留 Trace |
| `priority` | 优先级 |
| `source` | 来源，如历史埋点验收、线上事故、人工补录 |

## 4. 首版组织方式

建议首版按“领域 -> 场景 -> 用例”组织，而不是按人或脚本组织。

推荐目录与逻辑：

1. `ad-reporting`
2. `mcp-contract`
3. `knowledge-qa`
4. `tracking-validation`

其中 `tracking-validation` 用于吸收旧埋点验收资产，但不继续主导整个平台。

## 5. 历史埋点验收如何吸收

### 判断

旧埋点验收内容应被拆成：

1. 契约校验样本
2. 数据一致性样本
3. 回归事故样本

而不是整个保留成一条独立产品线。

### 影响

这会直接影响：

1. 首版回归集内容
2. 自动化测试优先级
3. 报告里的缺陷分类

## 6. 不确定项

1. 首版是否先只支持文本型 AI 应用样本
2. 埋点验收样本是否需要单独一套导入模板
3. 契约用例与语义用例是否共用一张底表

## 7. 需要主管决策的问题

1. ChoKoNu 首版是否明确采用“契约 + 语义”双轨评测集
2. 首版是否优先建设 `regression_suite`
3. 旧埋点验收内容是否要求首批完成结构化导入
