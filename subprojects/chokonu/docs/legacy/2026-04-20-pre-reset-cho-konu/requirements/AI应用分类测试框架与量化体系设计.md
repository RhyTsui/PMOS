# AI应用分类测试框架与量化体系设计

- 版本：v0.1
- 版本：v0.2
- 日期：2026-04-20
- 状态：Draft
- 项目：连弩-AI测试平台 / ChoKoNu

## 0. 主要新增依据

本版补充吸收了 `docs/sources/inbox/AI测试平台-openai.docx` 中较完整的方法论输入。

该输入的关键价值不在于再次解释“平台定位”，而在于补充了各类 AI 应用对象的专项测试方法、过程指标与发布门禁思路。

## 1. 核心结论

连弩-AI测试平台下一阶段不应继续停留在“通用评测平台”描述上，而应进入 AI 应用专项测试方法学建设。

平台必须围绕六类 AI 应用能力建立三层体系：

1. `专项测试对象框架`
2. `专项测试方法`
3. `量化评估体系`

六类核心测试对象：

1. `MCP`
2. `Chat`
3. `Skill`
4. `Agent`
5. `知识检索`
6. `Workflow`

## 2. 为什么现在必须转到方法学建设

传统功能测试关注的是：

1. 接口是否可调用
2. 数据是否返回
3. 功能链路是否跑通

AI 应用测试关注的则是：

1. 回答质量是否稳定
2. 工具选择是否合理
3. 检索内容是否相关且可支撑回答
4. Agent 是否完成任务而不是只输出文本
5. Workflow 是否在复杂条件下仍保持正确协同
6. 新版本是否真的比旧版本更好

这意味着平台如果只保留“通用评测、实验、报告”表达，无法指导真实产品建设。

## 3. 分类测试框架

### 3.1 MCP

测试目标：

1. 工具发现是否正确
2. schema / 参数契约是否一致
3. 请求映射是否正确
4. 返回结果是否可消费
5. 失败是否可定位到具体阶段
6. 鉴权、确认与权限边界是否正确

主要风险：

1. schema 漂移
2. 参数缺失
3. 请求视图错误
4. 返回结构不稳定
5. 调用成功但业务含义错误
6. 高风险工具误触发
7. 鉴权流程缺失或错误

适合的测试手段：

1. 契约测试
2. 参数映射测试
3. 请求/响应差异测试
4. Trace 分阶段校验
5. 鉴权与确认流程测试
6. 高风险工具误触发测试

核心量化指标：

1. 契约通过率
2. 参数命中率
3. schema 差异率
4. 失败阶段分布
5. 工具调用成功率
6. 鉴权通过率
7. 错误码正确率
8. 高风险工具误触发率

### 3.2 Chat

测试目标：

1. 回答是否正确
2. 回答是否完整
3. 回答是否符合业务口径
4. 是否出现幻觉、答非所问、跑题
5. 多轮上下文是否连贯
6. 是否真正解决用户问题而不只是生成一段可读文本

主要风险：

1. 表面可读但事实错误
2. 业务口径偏移
3. 多轮记忆错乱
4. 回答稳定性差
5. 多轮对话中上下文丢失
6. 该拒绝时不拒绝、该升级时不升级

适合的测试手段：

1. 语义问答测试
2. 多轮对话测试
3. 参考答案对比
4. LLM-as-judge 与规则混合评测
5. 多轮会话模拟测试
6. 会话级任务完成度测试

核心量化指标：

1. 回答正确率
2. 关键点覆盖率
3. 幻觉率
4. 多轮一致性
5. 用户意图满足率
6. 会话解决率
7. 上下文丢失率
8. 重复问答率

### 3.3 Skill

测试目标：

1. 单项能力是否在边界输入下稳定
2. 输入输出格式是否满足预期
3. 技能被调用时是否能完成明确任务
4. 技能降级和异常处理是否稳定
5. 技能是否在该触发时触发，不该触发时不触发

主要风险：

1. 技能能力边界不清
2. 结果格式波动
3. 在复杂输入下退化明显
4. 单项技能指标好，但组合使用效果差
5. skill 被误触发或漏触发

适合的测试手段：

1. 能力边界测试
2. 输入变体测试
3. 输出格式测试
4. 压力与稳定性抽样测试
5. 显式/隐式/上下文触发测试
6. 负样本触发测试

核心量化指标：

1. 任务完成率
2. 输出格式一致率
3. 异常恢复率
4. 稳定性波动系数
5. 触发准确率
6. 漏触发率
7. 误触发率

### 3.4 Agent

测试目标：

1. 是否完成任务目标
2. 是否选择了合理的工具路线
3. 是否在有限步数内完成任务
4. 是否在失败时正确中止或回退
5. 最终结果是否满足业务期望
6. 是否在真实或近真实环境中稳定完成任务

主要风险：

1. 会调用工具但完不成任务
2. 步数膨胀
3. 工具选择偏差
4. Trace 很长但产出价值低
5. 多 Agent 或复杂编排时责任不清
6. 随机性导致同任务结果波动过大

适合的测试手段：

1. 任务完成度测试
2. 工具选择路径测试
3. 计划-执行一致性测试
4. Trace 复盘测试
5. 多轮任务回归测试
6. 多 trial 稳定性测试
7. 沙箱环境任务执行测试

核心量化指标：

1. 任务完成率
2. 正确工具路径命中率
3. 平均完成步数
4. 无效步骤占比
5. 最终结果可用率
6. 多 trial 一致性
7. 审批与 SOP 合规率

### 3.5 知识检索

测试目标：

1. 检索结果是否相关
2. 检索结果是否足够支持最终回答
3. 召回是否稳定
4. 证据链是否可解释
5. 旧知识与新知识切换后是否退化
6. 回答是否真正被检索证据支撑

主要风险：

1. 召回不到
2. 召回过多噪声
3. 回答引用了错误证据
4. 知识更新后版本退化
5. 回答看似流畅但未被证据支撑

适合的测试手段：

1. 检索相关性测试
2. Top-K 命中测试
3. 证据支撑测试
4. 知识版本对比测试
5. 生成层与检索层拆分测试
6. 知识库外问题拒答测试

核心量化指标：

1. Recall@K
2. Precision@K
3. 证据命中率
4. 回答可支撑率
5. 知识版本回归率
6. groundedness / faithfulness
7. 拒答正确率

### 3.6 Workflow

测试目标：

1. 节点间协同是否正确
2. 条件分支是否按预期执行
3. 重试、回退、异常分支是否有效
4. 关键输入输出是否在全链路保持一致
5. 工作流版本变更后是否引入隐性退化
6. AI 节点质量与工程链路稳定性是否同时达标

主要风险：

1. 单节点正常但链路组合出错
2. 分支条件误判
3. 状态传递丢失
4. 局部修改破坏全局流程
5. AI 节点正确但整体业务未达成

适合的测试手段：

1. 链路回放测试
2. 分支覆盖测试
3. 状态一致性测试
4. 关键路径回归测试
5. 节点 I/O 契约测试
6. 异常补偿与回滚测试

核心量化指标：

1. 链路成功率
2. 分支覆盖率
3. 状态一致率
4. 回退成功率
5. 工作流版本回归失败率
6. 关键路径业务成功率
7. 节点契约通过率

## 4. 通用测试方法层

在六类对象之上，平台应沉淀统一方法层，至少包括：

1. `契约测试`
2. `语义测试`
3. `任务完成度测试`
4. `工具选择测试`
5. `检索质量测试`
6. `链路稳定性测试`
7. `回归对比测试`
8. `实验对照测试`
9. `多轮会话模拟测试`
10. `多 trial 稳定性测试`
11. `安全与越权风险测试`
12. `鉴权与人工确认测试`

其中前六类偏“单版本质量判断”，后两类偏“版本迭代判断”。

## 5. 通用量化体系层

平台不应只有一套总分，而应分成四层指标：

### 5.1 基础运行指标

1. 成功率
2. 延迟
3. Token 消耗
4. 错误率

### 5.2 质量指标

1. 正确率
2. 覆盖率
3. 幻觉率
4. 可支撑率
5. 任务完成率
6. 完整性
7. 相关性

### 5.3 行为指标

1. 工具路径命中率
2. 平均步数
3. 无效步骤占比
4. 分支正确率

### 5.4 迭代指标

1. 版本回归率
2. 实验胜率
3. 指标提升幅度
4. 高风险场景恶化率

### 5.5 风险指标

1. 权限越界率
2. 敏感数据泄露率
3. 高风险工具误触发率
4. Prompt injection 命中率
5. 红线违规率

## 6. 发布门禁建议

平台首版不应只输出一个总分，而应采用四段式门禁判断：

1. `红线项`
2. `主指标项`
3. `版本回归项`
4. `上线后观察项`

### 6.1 红线项

必须是 must-pass：

1. 权限越界
2. 敏感数据泄露
3. 高风险工具误调用
4. 关键业务规则违反

### 6.2 主指标项

按应用类型配置核心阈值，例如：

1. MCP 契约通过率
2. Chat 会话解决率
3. Agent 任务完成率
4. RAG 可支撑率
5. Workflow 关键路径成功率

### 6.3 版本回归项

不能只比较平均分，还要比较：

1. 高风险场景是否恶化
2. 关键 slice 是否退化
3. 多 trial 稳定性是否下降

### 6.4 上线后观察项

上线后仍应持续观察：

1. Trace
2. 告警
3. 线上失败样本
4. 用户反馈回流

## 7. 平台建设优先级建议

### 第一优先级

先做最能形成平台差异化的四类方法学：

1. MCP 契约测试
2. Chat / Agent 语义与任务测试
3. 知识检索质量测试
4. Workflow 关键路径回归测试

### 第二优先级

再补：

1. Skill 能力边界测试
2. 多轮一致性测试
3. 实验对照与量化报告

### 第三优先级

最后做：

1. 统一评分归一化
2. 门禁阈值体系
3. 自动发布阻断规则

## 8. 对产品文档的直接要求

从现在开始，AEP 后续文档不应只写：

1. 评测集
2. 评估器
3. 执行
4. Trace
5. 报告

还必须继续细化到：

1. 每类 AI 应用的专项测试方法
2. 每类 AI 应用的核心风险
3. 每类 AI 应用的量化指标
4. 版本对比时的判断口径

## 9. 建议的后续文档拆分

建议下一轮继续输出 6 份专项草案：

1. `MCP专项测试方法草案`
2. `Chat专项测试方法草案`
3. `Skill专项测试方法草案`
4. `Agent专项测试方法草案`
5. `知识检索专项测试方法草案`
6. `Workflow专项测试方法草案`

## 10. 不确定项

1. 首版是否六类对象全部进入一期
2. Chat 与 Agent 的边界是否在首版中强行拆开
3. Skill 是否作为独立对象，还是暂并入 Agent/Workflow
4. Workflow 首版是否只做关键路径回归，不做完整覆盖

## 11. 需要主管决策的问题

1. 首版专项测试方法是否先聚焦 `MCP + Chat/Agent + 知识检索 + Workflow`
2. 量化体系首版是否先以“通过/失败 + 核心指标”双层输出，而不是统一总分
3. 是否要求下一轮文档按六类对象分别展开
