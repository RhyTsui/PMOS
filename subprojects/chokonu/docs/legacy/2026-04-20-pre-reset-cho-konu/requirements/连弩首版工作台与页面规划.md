# 连弩首版工作台与页面规划

- 版本：v0.1
- 日期：2026-04-20
- 状态：Draft
- 项目：连弩-AI测试平台 / ChoKoNu

## 1. 核心结论

连弩首版页面不从“功能堆砌”出发，而从动作链出发。

首版固定为 6 个工作台：

1. 样本中心
2. 评测集中心
3. 执行中心
4. 结果与 Trace 中心
5. 报告与版本对比中心
6. 规则 / 评估器中心

## 2. 工作台总览

| 工作台 | 页面目标 | 服务对象 | 核心对象 |
|---|---|---|---|
| 样本中心 | 维护可复用测试资产 | 产品、测试、测开 | cases、labels、expected、risk flags |
| 评测集中心 | 组织 baseline / regression / scenario suites | 测试、测开 | suites、suite members、version tags |
| 执行中心 | 发起与管理测试运行 | 测试、测开、研发 | runs、runner config、targets |
| 结果与 Trace 中心 | 复盘失败、定位问题 | 测试、研发 | case results、failure stage、traces |
| 报告与版本对比中心 | 输出结论、支撑版本判断 | 主管、产品、测试 | run reports、aggregate reports、version compare |
| 规则 / 评估器中心 | 管理测试判定规则 | 测开、产品 | evaluators、rules、thresholds |

## 3. 工作台定义

### 3.1 样本中心

页面目标：

1. 维护六类对象的测试样本
2. 给样本打标签、场景、风险等级
3. 维护 expected / schema / judge 参考信息

核心对象：

1. contract cases
2. semantic cases
3. task cases
4. retrieval cases
5. skill cases
6. workflow cases

必备操作：

1. 新建
2. 编辑
3. 查询与筛选
4. 导入导出
5. 标记回归样本
6. 绑定红线项

关键结果：

1. 样本资产沉淀
2. 回归样本复用
3. 样本版本可追溯

不做项：

1. 复杂标注工作流
2. 大规模数据处理平台

### 3.2 评测集中心

页面目标：

1. 把分散样本组织成可执行 suite
2. 区分 baseline / regression / scenario 视图

核心对象：

1. test suites
2. regression suites
3. scenario grouping

必备操作：

1. 新建 suite
2. 添加/删除样本
3. 按对象和场景聚合
4. 标记基线版本

关键结果：

1. 可重复执行的 suite
2. 版本化回归集

不做项：

1. 复杂权限共享体系
2. 多租户 suite 治理

### 3.3 执行中心

页面目标：

1. 发起 run
2. 管理执行状态
3. 支持重跑与多 trial

核心对象：

1. runs
2. runner config
3. evaluator sets
4. targets / environments

必备操作：

1. 选择 suite
2. 选择对象类型和运行目标
3. 选择 evaluator 组合
4. 发起 run
5. 重跑全部/失败样本

关键结果：

1. 稳定 run 记录
2. 多 trial 运行结果

不做项：

1. 复杂任务调度编排平台
2. 自动发布流水线

### 3.4 结果与 Trace 中心

页面目标：

1. 查看结果分布
2. 下钻失败样本
3. 关联 trace 做问题定位

核心对象：

1. case results
2. failure stage
3. raw trace
4. normalized trace

必备操作：

1. 结果筛选
2. 只看失败/高风险/待复核
3. 查看 trace
4. 查看失败原因分类

关键结果：

1. 问题定位入口清晰
2. trace 与 case result 打通

不做项：

1. 全量 observability 平台
2. 复杂告警编排

### 3.5 报告与版本对比中心

页面目标：

1. 输出 run report
2. 做基础版本对比
3. 给出放行建议

核心对象：

1. run reports
2. aggregate reports
3. version compare

必备操作：

1. 查看单次报告
2. 查看版本对比
3. 导出 Markdown/HTML
4. 标记红线问题

关键结果：

1. 正式报告
2. 版本优劣判断
3. 基础门禁依据

不做项：

1. 复杂 BI 看板体系
2. 全量组织级报表中心

### 3.6 规则 / 评估器中心

页面目标：

1. 管理规则、阈值、评估器
2. 让不同对象的判定逻辑可配置

核心对象：

1. rule evaluators
2. contract evaluators
3. semantic evaluators
4. thresholds

必备操作：

1. 维护 evaluator 类型
2. 配置阈值
3. 绑定对象与 suite
4. 标记人工复核挂点

关键结果：

1. 判定逻辑统一
2. 红线与主指标有出处

不做项：

1. 大而全低代码规则引擎

## 4. 页面约束

首版页面规划必须遵守：

1. 不做通用 Agent Builder
2. 不做复杂 Workflow 设计器
3. 不做长期线上监控告警中心
4. 不做与对象-能力-指标矩阵无关的独立页面

## 5. 页面交付顺序

建议实现顺序：

1. 样本中心
2. 评测集中心
3. 执行中心
4. 结果与 Trace 中心
5. 报告与版本对比中心
6. 规则 / 评估器中心

## 6. 需要主管决策的问题

1. 报告与版本对比中心是否在首版作为独立工作台呈现
2. 规则 / 评估器中心是独立页面，还是先作为配置页轻接入
3. Skill / Workflow 是否只在样本中心与执行中心表现，不做对象专属页面
