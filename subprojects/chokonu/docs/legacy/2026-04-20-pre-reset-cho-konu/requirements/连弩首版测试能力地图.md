# 连弩首版测试能力地图

- 版本：v0.1
- 日期：2026-04-20
- 状态：Draft
- 项目：连弩-AI测试平台 / ChoKoNu

## 1. 核心结论

六类 AI 应用的专项测试方法已经补齐，首版下一步不该再扩写方法论，而应该把方法论收口成：

1. 一期先做哪些测试对象
2. 每类对象先落哪些测试能力
3. 哪些能力属于首版必须项
4. 哪些能力作为二期扩展位

首版建议采用 `4 + 2` 结构：

### 首版主战场

1. `MCP`
2. `Chat`
3. `Agent`
4. `知识检索`

### 首版预埋但不做深

1. `Skill`
2. `Workflow`

## 2. 为什么这样切

原因不是“后两类不重要”，而是：

1. `MCP` 代表 AI 应用接工具的基础能力
2. `Chat` 代表最常见的 AI 应用交互形态
3. `Agent` 代表自主任务执行能力
4. `知识检索` 代表企业内最常见的知识增强场景

这四类已经能覆盖当前最主要的 AI 应用测试需求，也最能形成连弩首版的平台差异。

而：

1. `Skill` 更适合在首版中以方法预埋、能力轻落地的方式进入
2. `Workflow` 需要工程链路、状态管理、回放与补偿体系更成熟后再做深

## 3. 首版能力地图

| 测试对象 | 首版目标 | 首版必须能力 | 二期扩展能力 |
|---|---|---|---|
| MCP | 工具接入正确、可测、可追溯 | 工具发现、schema 契约、参数映射、鉴权确认、失败阶段 | OAuth 全链路、复杂权限策略、跨 server 组合测试 |
| Chat | 会话真正解决问题 | 单轮语义质量、多轮上下文、会话解决率、拒答/升级边界 | 用户模拟器、复杂 persona、多会话长期稳定性 |
| Agent | 在环境中完成任务 | 任务完成率、工具路径、步数、异常恢复、多 trial 稳定性 | 多 Agent 协同、复杂审批链、生产沙箱深集成 |
| 知识检索 | 回答有证据支撑 | 检索层/生成层拆分、Recall@K、可支撑率、版本回归、拒答 | citation 片段级对齐、混合检索策略优化、长期知识演化分析 |
| Skill | 组件触发与结果稳定 | 触发准确率、漏触发率、误触发率、输出结构稳定性 | 复杂触发策略、组合 Skill 测试 |
| Workflow | 关键路径正确 | 节点契约、关键路径回归、状态一致性、补偿回退 | 全量分支覆盖、可视回放、复杂状态治理 |

## 4. 各对象首版最小能力集

### 4.1 MCP

首版最小能力集：

1. 工具发现测试
2. schema 契约测试
3. 参数映射测试
4. 错误处理测试
5. 鉴权与确认测试

首版最小指标集：

1. 契约通过率
2. 参数命中率
3. 工具调用成功率
4. 鉴权通过率
5. 高风险工具误触发率

### 4.2 Chat

首版最小能力集：

1. 单轮语义问答测试
2. 多轮上下文测试
3. 会话解决率测试
4. 拒答与升级测试
5. 业务口径稳定性测试

首版最小指标集：

1. 回答正确率
2. 关键点覆盖率
3. 会话解决率
4. 幻觉率
5. 拒答正确率

### 4.3 Agent

首版最小能力集：

1. 任务完成度测试
2. 工具路径测试
3. 计划-执行一致性测试
4. 多 trial 稳定性测试
5. 异常恢复测试

首版最小指标集：

1. 任务完成率
2. 工具路径命中率
3. 平均完成步数
4. 多 trial 一致性
5. 恢复成功率

### 4.4 知识检索

首版最小能力集：

1. 检索相关性测试
2. 证据支撑测试
3. 生成正确性测试
4. 知识版本回归测试
5. 知识库外拒答测试

首版最小指标集：

1. Recall@K
2. Precision@K
3. 回答可支撑率
4. groundedness
5. 知识版本回归率

### 4.5 Skill

首版最小能力集：

1. 触发精度测试
2. 能力边界测试
3. 输出结构测试

首版最小指标集：

1. 触发准确率
2. 漏触发率
3. 误触发率
4. 输出格式一致率

### 4.6 Workflow

首版最小能力集：

1. 节点契约测试
2. 关键路径回放测试
3. 状态一致性测试
4. 异常补偿与回滚测试

首版最小指标集：

1. 节点契约通过率
2. 链路成功率
3. 状态一致率
4. 回退成功率

## 5. 首版平台能力映射

要承接上述测试对象，连弩首版平台至少要具备以下共性能力：

1. `样本管理`
2. `评测集管理`
3. `执行编排`
4. `评估器执行`
5. `Trace 关联`
6. `结果归集`
7. `版本对比`
8. `报告输出`

进一步映射到对象层：

| 平台能力 | MCP | Chat | Agent | 知识检索 | Skill | Workflow |
|---|---|---|---|---|---|---|
| 样本管理 | 高 | 高 | 高 | 高 | 中 | 中 |
| 评测集管理 | 高 | 高 | 高 | 高 | 中 | 高 |
| Trace 关联 | 高 | 中 | 高 | 中 | 低 | 高 |
| 多 trial | 低 | 中 | 高 | 低 | 低 | 低 |
| 版本对比 | 高 | 高 | 高 | 高 | 中 | 高 |
| 门禁规则 | 高 | 高 | 高 | 高 | 中 | 高 |

## 6. 首版优先级建议

### P0

必须在首版打实：

1. MCP 契约与鉴权测试
2. Chat 会话质量测试
3. Agent 任务完成与路径测试
4. 知识检索证据支撑测试
5. 样本 -> suite -> run -> trace -> report 主链路

### P1

首版可轻量接入：

1. Skill 触发测试
2. Workflow 关键路径回归
3. 版本对比
4. 基础门禁规则

### P2

二期再做深：

1. 多 Agent 协同
2. Workflow 全量分支覆盖
3. 复杂 Skill 组合测试
4. 上线后长期观测与自动告警

## 7. 首版不该做什么

为了不失焦，首版不应做：

1. 通用 Agent Builder
2. 大而全低代码 Workflow 设计器
3. 全栈研发协同平台
4. 全行业统一评测大而全总分体系

首版应该守住的是：

`测试对象清晰 + 测试方法清晰 + 指标清晰 + 结果可回归`

## 9. 需要主管决策的问题

1. 首版是否确认采用 `4 + 2` 结构
2. Skill 与 Workflow 是否只做轻量接入，不做深度建设
3. 首版是否必须包含版本对比与基础门禁
