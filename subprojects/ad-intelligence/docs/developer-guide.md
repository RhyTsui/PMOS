# 开发者操作手册

这份文档回答的是“开发者接手后第一天怎么有效推进”，不是产品愿景说明。

## 1. 进入项目后先确认什么

先确认三件事：

1. 当前代码只实现了最小后端骨架。
2. 前端依赖已存在，但前端页面并未成型。
3. 历史 `docs/*.md` 中有大量方案演进稿，适合参考，不适合直接当成现状。

如果不先确认这三点，后续很容易把研究草稿误认为已落地能力。

## 2. 本地开发

### 安装依赖

```bash
npm install
```

### 环境变量

复制模板：

```bash
cp .env.example .env
```

关键字段：

- `PORT`

目前项目没有强制要求其他环境变量；`OPENAI_API_KEY` 等只是预留。

### 启动后端

```bash
npm run dev:backend
```

或：

```bash
npm run start
```

### 类型检查

```bash
npm run lint
```

### 当前已知问题

- `npm run build` 失败：缺少前端入口 `index.html`
- `tests/` 目录为空：还没有自动化测试保护
- `src/server.ts` 承担了全部后端逻辑：需要尽快拆层

## 3. 代码现状

当前后端只有两个接口：

- `GET /`
- `GET /health`

这意味着项目还没有真正进入“广告素材情报系统”的业务实现阶段。

所以后续开发不要从“优化体验”开始，而要从“建立业务主链”开始。

## 4. 推荐的第一批开发任务

### 任务 1：拆分服务骨架

把 `server.ts` 拆成：

- 路由层
- 服务层
- 数据模型层

目标是让后续采集和分析逻辑有明确落点。

### 任务 2：定义素材数据结构

先定义统一字段，再谈采集：

- 来源平台
- 素材类型
- 素材地址或文件引用
- 投放主体
- 抓取时间
- 分析状态
- 标签与摘要

如果这个结构不先稳定，后面无论是爬虫还是 AI 分析都会反复返工。

### 任务 3：跑通一个最小导入闭环

优先做：

- 导入一条素材数据
- 存储
- 查询返回

不要一开始就尝试覆盖多个广告平台。

### 任务 4：补测试

至少先补：

- 健康检查接口测试
- 素材导入接口测试
- 素材查询接口测试

## 5. 文档阅读顺序

推荐按这个顺序看：

1. `README.md`
2. `ARCHITECTURE.md`
3. `docs/memory/project-memory.md`
4. 历史方案稿，例如 `docs/实现规划 v2.0.md`

前 3 个是当前入口，后面的历史稿只用于补背景，不直接作为实施依据。

## 6. 怎样判断文档写得是否合格

一份对开发者真正有用的文档，至少要满足：

- 新人能在 5 分钟内判断项目现状
- 能区分“已实现”和“规划中”
- 能知道下一步最值得做什么
- 能知道哪些信息可信、哪些只是历史草稿

如果文档只讲愿景、不讲现状，那不是开发者文档，只是项目宣传稿。
