# 广告素材情报项目架构

## 1. 架构定位

这个子项目当前不是一个完整系统，而是一个“广告素材情报服务”的起步骨架。

现阶段目标不是一次性做成工业级平台，而是先跑通最小闭环：

`素材来源 -> 采集 -> 分析 -> 结构化输出 -> 查询接口`

## 2. 当前实现架构

当前代码只有一个最小 Express 服务：

```text
Client
  -> Express Server (src/server.ts)
      -> GET /
      -> GET /health
```

特点：

- 没有业务模块拆分
- 没有数据库
- 没有任务调度
- 没有采集器
- 没有分析器
- 没有真实前端

这意味着当前项目更接近“服务入口已建立”，而不是“业务系统已成型”。

## 3. 目标模块划分

建议后续按下面的边界拆分，而不是继续把逻辑堆在 `server.ts`：

```text
src/
├─ server.ts                 # 服务启动与中间件装配
├─ api/                      # HTTP 路由层
├─ collector/                # 广告素材采集器
├─ analyzer/                 # OCR / 标签 / 文案提取 / 初步分类
├─ services/                 # 业务编排
├─ repositories/             # 数据读写
├─ models/                   # 数据模型与类型
└─ shared/                   # 通用工具与常量
```

各层职责建议：

- `api/`
  负责请求参数校验、响应格式、错误码，不直接承载业务细节。
- `collector/`
  负责接入广告平台或素材来源，输出统一的原始素材结构。
- `analyzer/`
  负责把素材从原始输入转为可检索、可分析的结构化结果。
- `services/`
  负责把采集、分析、存储、查询串起来。
- `repositories/`
  负责持久化，不让上层直接依赖数据库细节。

## 4. 推荐的最小闭环

开发上最值得优先完成的不是“大而全”，而是下面这条主链：

1. 提供一个素材导入入口
2. 将素材元数据落到统一结构
3. 触发基础分析
4. 提供查询接口返回结果

建议的第一批 API：

- `POST /api/materials/import`
  导入单条或一批素材来源
- `GET /api/materials`
  查询素材列表
- `GET /api/materials/:id`
  查看素材详情
- `POST /api/materials/:id/analyze`
  触发单素材分析
- `GET /api/health`
  服务健康检查

## 5. 当前脚本与架构现实

### 后端

当前后端可运行，`lint` 可通过，说明 TypeScript 骨架是可维护的最小起点。

### 前端

依赖和脚本里已经出现了 Vite，但仓库内还没有真正的前端入口文件，因此：

- 开发服务器可启动
- 生产构建会因缺少 `index.html` 失败

这类状态如果不写清楚，会直接误导新开发者，以为前后端骨架都已经准备好了。

## 6. 演进路线

建议按三阶段推进。

### 阶段 A：最小后端闭环

- 拆出 `api/` 与 `services/`
- 引入素材数据模型
- 增加导入与查询接口
- 为分析结果预留结构

### 阶段 B：分析能力接入

- 接入 OCR 或文案提取
- 增加标签与分类结果
- 把“原始素材”和“分析结果”分层存储

### 阶段 C：工作台与趋势视图

- 增加基础前端入口
- 展示素材列表、详情、分析结果
- 再考虑趋势图、竞品监控等高级能力
