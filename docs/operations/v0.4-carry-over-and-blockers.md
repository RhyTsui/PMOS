# PMAIOS v0.4 Carry-Over 与阻塞清单

- version: v0.1
- date: 2026-04-21
- status: active
- purpose: 明确 `v0.4` 哪些项属于当前可继续收口的内容，哪些项应正式转入 carry-over，而不是继续悬空

## 分类规则

### 1. 当前版本继续处理

满足以下任一条件：

- 直接阻塞 `v0.4` 核心验收
- 本地可独立推进，不依赖新的外部凭证或系统
- 继续处理的成本明显低于挂起成本

### 2. 正式 carry-over

满足以下任一条件：

- 依赖新的外部账号、token、database、team 权限
- 需要额外真实业务系统输入
- 当前版本继续追会拖慢收口

## 当前清单

### A. Notion database target

- 状态：`carry-over`
- 原因：当前 token 可用，但提供的是 page，不是 database；要继续必须拿到真实 database id
- 归属：`v0.5 / external connectors hardening`

### B. Figma team/token

- 状态：`carry-over`
- 原因：`FIGMA_TEAM_ID` 已配置，但当前 token 远程检查是 `403 Invalid token`
- 归属：`v0.5 / external connectors hardening`

### C. DingTalk 自动桌面提取

- 状态：`carry-over`
- 原因：目前只有本地 UI automation scaffold，缺少稳定官方导出/API/固定 source path，不适合继续拖住 `v0.4`
- 归属：`v0.5 / automation hardening`

### D. 浏览器端全面 UI smoke

- 状态：`current-version optional`
- 原因：不阻塞 `v0.4` 主要验收，但仍可作为补充验证项
- 建议：若今天有余量再做，否则不阻塞 `v0.4` 收口

### E. 子项目 workflow rollout

- 状态：`current-version continue`
- 原因：这是平台当前样板扩展，仍属于当前工作方式收口的一部分
- 范围：
  - `knowledge-base`
  - `ad`
  - `chokonu`
  - `server`

## 结论

当前 `v0.4` 不应再被以下三项拖住：

1. Notion database target
2. Figma team/token
3. DingTalk 自动桌面提取

它们都应视为正式 carry-over。

`v0.4` 当前更应继续收的，是：

1. 当前版本快照与进度协议
2. 子项目 workflow rollout
3. 可选的 UI smoke
