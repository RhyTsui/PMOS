# PMOS v1.0 GitHub 仓切换执行顺序

- version: v0.1
- date: 2026-05-09
- status: active
- owner: product office

## 1. 目的

这份清单定义：

1. 旧 GitHub 仓怎么处理
2. 新 `PMOS product repo` 怎么发布
3. 两者切换时的最稳执行顺序

## 2. 总原则

不要先删旧仓，再现场拼新仓。

正确顺序是：

1. 先让新仓可发布
2. 再处理旧仓

## 3. 推荐切换方案

### 方案 A：新建干净产品仓

这是默认推荐方案。

顺序：

1. 在 GitHub 新建独立仓，例如 `pmos` 或 `pmos-v1`
2. 把本地 `product-repo/` 作为首版内容推上去
3. 验证 README、部署文档、truth docs、cloud-mirror 样例都可读
4. 再决定旧仓改名、归档、或保留只读

优点：

1. 最稳
2. 不会污染旧历史
3. 不会误删老仓里还要留档的内容

### 方案 B：复用旧仓名

只在“必须保留旧仓名”的情况下使用。

顺序：

1. 先在本地完成新 product-repo 的最终整理
2. 备份旧仓当前状态
3. 清空旧仓默认分支内容
4. 再把新 product-repo 推上去
5. 验证对外入口是否正常

风险：

1. 一旦顺序错，很容易删完再补不回来
2. 更容易把旧仓脏内容和新仓混在一起

## 4. 旧仓处理策略

### 4.1 推荐

把旧仓当作：

`historical mother-repo mirror`

可选动作：

1. 改仓库描述，注明不是独立产品仓
2. 设为 archive
3. 在 README 顶部加跳转到新 product repo

### 4.2 不推荐

1. 不加说明直接继续沿用旧仓对外展示
2. 边删边判断要保留什么
3. 让旧仓继续承担 PMOS 正式产品入口

## 5. 新 product-repo 发布顺序

1. 重新确认 `.gitignore`
2. 确认不含 `node_modules / dist / .env / logs`
3. 确认 README 已更新
4. 确认 first-run / second-machine / release-readiness 文档都在
5. 确认 `lint / build / start` 已通过
6. 初始化 Git 仓
7. 首次 commit
8. 推送到新 GitHub 仓
9. 在 GitHub 页面再做一次人工检查

## 6. 发布后第一轮验证

发布后立即检查：

1. GitHub 首页 README 是否清晰
2. docs 链接是否正常
3. 外部读者是否能看到 `v1.0 / v0.7 baseline` 口径
4. 是否仍残留母仓语义
5. 是否仍有私有信息

## 7. 当前建议

截至 `2026-05-09`，建议采用：

`方案 A：新建干净产品仓`

## 8. 下一步

1. 生成新 product-repo 的实际发布步骤
2. 如果你准备发仓，再进入 Git 初始化和推送动作
