# PMAIOS 分阶段专业 Agent 与样式记忆机制提案 v1

日期：2026-05-07

来源：

- `tracking-acceptance` 三天实施复盘
- 用户最新真实使用感受反馈

定位：

- 本文档不是项目文档
- 本文档是面向 PMAIOS 平台的能力提案
- 用于解决“同一套工作模式贯穿所有阶段导致效率低、反复纠正、风格漂移”的问题

---

## 1. 要解决的核心问题

当前 PMAIOS 暴露出 4 个效率瓶颈：

1. 不同阶段仍过度依赖同一套通用工作模式
2. 问答式澄清没有成为需求阶段默认能力
3. UI 规范存在但不自动生效
4. 同类纠正无法被自动吸收，导致用户反复重说

这些问题共同导致：

- 项目推进链变长
- 设计与实现反复返工
- 工具和 skill 看起来接了，但用户体感并未真正生效

---

## 2. 分阶段专业 Agent 编排提案

建议 PMAIOS 正式把工作流拆成以下阶段，并为每阶段指定默认主导 agent / 主导模式：

### 2.1 需求阶段

主目标：

- 澄清问题
- 抽取对象
- 收口边界
- 定义成功标准

默认工作模式：

- 问答式澄清
- 需求池录入
- 问题驱动收口

不建议：

- 过早进入页面实现
- 直接产出视觉稿

### 2.2 原型阶段

主目标：

- 定页面结构
- 定对象落位
- 定页面关系

默认工作模式：

- 结构化原型
- 页面职责分配
- 列表/详情/分组对象分层

### 2.3 交互阶段

主目标：

- 定行为链
- 定状态机
- 定切换规则

默认工作模式：

- 交互链说明
- 状态流确认
- 局部动作与全局动作边界确认

### 2.4 UI阶段

主目标：

- 定视觉语言
- 定组件使用
- 定样式规范

默认工作模式：

- 强制读取 UI 规范
- 组件语义对齐
- 差异诊断后整改

### 2.5 前端阶段

主目标：

- 产出最终交付前端页
- 保留真实交互
- 对齐组件和工程边界

默认工作模式：

- 真实组件优先
- mock 数据驱动
- 局部滚动与交付质量校验

---

## 3. 问答式澄清能力提案

建议将问答式工作模式从“偶发能力”升级为“需求阶段默认能力”。

默认触发场景：

- 用户表述模糊
- 对象边界不清
- 页面职责冲突
- 术语歧义
- 成功标准不明确

默认输出：

- 澄清问题清单
- 统一术语
- 对象与边界
- 候选方向
- 取舍说明

目标：

- 在需求阶段多问清
- 避免后面原型/UI/前端反复返工

---

## 4. UI规范自动激活提案

当前问题：

- 规范可能存在
- 但没有自动影响页面整改

建议新增：

`UI Spec Activation Gate`

行为：

1. 一旦项目存在正式 UI 规范文件
2. 后续页面整改默认必须先读取并应用该规范
3. 若未读取，应显式告警

最低生效范围：

- 字体与字号
- 圆角
- 色彩语义
- 标题层级
- 筛选区间距
- 表格滚动策略
- 组件优先级

目标：

- 减少用户重复补“圆角多少、滑动条不能溢出”等细节

---

## 5. 重复纠正吸收机制提案

建议新增平台机制：

`repeat-correction-memory`

规则：

- 当同一项目中，用户对同类问题重复纠正达到阈值
- 例如：
  - 同一组件重复指定
  - 同一风格规则重复指定
  - 同一局部布局重复指定
- 系统自动将其标记为：
  - 项目级默认规则候选
  - 平台级复用规则候选

吸收后：

- 项目内同类页面默认继承
- 相似项目中进入推荐复用清单

目标：

- 让“说一次不再说第二次”真正变成系统能力

---

## 6. skill 与设计工具生效检查提案

建议新增两类检查：

### 6.1 skill activation check

验证：

- skill 是否被正确读取
- 是否真的参与当前输出
- 若失效，是否显式提示

### 6.2 design tool effectiveness check

验证：

- Figma / Pixso / 设计规范输入是否真正影响当前输出
- 输出是否仍表现为默认模型风格
- 是否偏离项目约束

目标：

- 不再因为“工具接了”就误判为“能力已生效”

---

## 7. 平台默认目标修正

建议在 PMAIOS 平台目标中明确补充：

`一次确认，项目内默认继承；完成一个项目，其他项目可按同一方式复用。`

具体表述可收口为：

- 说一次，不再说第二次
- 用过某个组件，类似需求优先复用同一组件
- 定过某种页面结构，同类页面优先继承同一结构

这应成为平台效率目标，而不是个人使用期待。
