# Cloud Sync Layer Contract

- version: v0.1
- date: 2026-05-21
- status: active

## Purpose

This document defines the governed cloud knowledge sync layer for PMOS v1.0.

It answers:

1. what local state is allowed to become cloud knowledge
2. which layer owns latest runtime state
3. how GitHub, Notion, Dataki, and ChatGPT retrieval should relate
4. what is still manual before automated publish is enabled

## Layering

| Layer | Role | Current Source | Publish Shape |
| --- | --- | --- | --- |
| Local repo truth | authoritative product and runtime files | `docs/operations`, `docs/memory`, `cloud-mirror` | committed files and generated mirrors |
| Cloud mirror | latest retrieval-friendly status | `cloud-mirror/runtime-status.json`, `cloud-mirror/runtime-status.md` | compact status object plus Markdown summary |
| GitHub | durable external repo and release reference | pushed PMOS product repo | source, docs, release notes, mirror files |
| Notion | human-readable knowledge page layer | selected digest and status pages | page/database records for PM review and ChatGPT browsing |
| Dataki | downstream retrieval grounding | Notion / local knowledge base sync | indexed chunks for agent grounding |
| ChatGPT retrieval | external question answering context | GitHub and Notion links | latest-state lookup, not hidden local memory |

## Runtime Mirror Contract

The current local latest-state mirror is generated by:

```bash
npm run cli -- cloud-mirror runtime-status
```

The generated files are:

- `cloud-mirror/runtime-status.json`
- `cloud-mirror/runtime-status.md`

The mirror must include:

- current product version and runtime baseline
- current mode and highest-priority line
- in-progress tasks
- recent checkpoints
- blocker counts and a bounded blocker sample
- Task SSOT, Outbox, Scheduler, and Document Governance summaries
- latest review evidence summary

The mirror must not include:

- secrets
- local absolute credential paths
- unbounded historical event dumps
- private executor-only prompt internals

## Publish Cadence

| Trigger | Required Action |
| --- | --- |
| After a meaningful runtime change | regenerate `cloud-mirror/runtime-status.*` |
| Before GitHub push / release | regenerate mirror, run `npm run validate`, and review dirty files |
| Before Notion/Dataki refresh | publish `runtime-status.md` or an equivalent compact status page |
| After external publish | record an Outbox envelope or receipt when the adapter exists |

## Current Boundary

Landed:

- local latest-state mirror generation
- CLI and API entrypoints
- document governance registration path

Still partial:

- automated GitHub/Notion publish adapter for the runtime mirror
- Dataki refresh against the current Notion path is operator-verified; Dataki refresh against a future published Cloud mirror layer is outside the current active todo
- ChatGPT retrieval source list with stable public links

## Acceptance

This track is `partial-strong` when:

1. the local mirror exists and can be regenerated
2. the mirror is registered as a truth source
3. the mirror is compact enough for cloud retrieval

It is `solved` only when:

1. GitHub and Notion publish paths are automated or documented as a repeatable release step
2. Dataki refresh is verified against the published layer
3. ChatGPT can answer current-state questions from cloud-visible sources without hidden local context
