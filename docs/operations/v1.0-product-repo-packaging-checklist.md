# PMOS v1.0 产品仓打包清单

- version: v0.1
- date: 2026-05-09
- status: active
- owner: product office

## 1. 目的

本清单用于把当前研发母仓拆分为：

1. `研发母仓`
2. `PMOS product repo`

它回答的不是“现在仓库里有什么”，而是“首版可部署 PMOS 产品仓应该带什么，不应该带什么”。

## 2. 打包原则

1. 只打包平台级能力
2. 只打包平台级真源
3. 只打包可部署所需最小文件
4. 不混入业务子项目主体
5. 不携带真实密钥、真实本机路径、真实 session

## 3. 应保留到产品仓的内容

### 3.1 源码

1. 平台 backend 源码
2. 平台 frontend 源码
3. workflow / scheduler / review / Hermes / proof / context 等平台运行时代码
4. 平台 CLI / API / operator 入口

### 3.2 平台真源文档

1. `docs/operations/current-version-progress.md`
2. `docs/operations/pmaios-v1.0-direction.md`
3. `docs/operations/v1.0-product-version-program.md`
4. `docs/operations/v1.0-acceptance-standard.md`
5. `docs/operations/v1.0-gap-list.md`
6. `docs/operations/v1.0-product-repo-and-cloud-sync-plan.md`
7. `docs/operations/module-roadmap.md`
8. `docs/operations/platform-truth-source-index.md`
9. `docs/architecture/PMAIOS_product_management_agent_operating_model.md`

### 3.3 部署与运行文件

1. `README.md`
2. `.env.example`
3. `.gitignore`
4. `docker-compose.yml` 或对应样例
5. `docs/deployment/*`
6. 启动脚本 / 安装脚本 / 初始化脚本

### 3.4 样例与镜像

1. `cloud-mirror/` 下的样例镜像结构
2. `runtime-status.sample.json`
3. `runtime-status.sample.md`
4. 最小 demo / mock 数据

## 4. 默认不进入产品仓的内容

### 4.1 子项目主体

1. `subprojects/ad/` 业务主体
2. `subprojects/knowledge-base/` 业务主体
3. `subprojects/tracking-acceptance/` 业务主体
4. 其他业务项目完整真源包

这些项目可以保留为 benchmark 或独立仓，不应混入 PMOS 产品仓主体。

### 4.2 历史和实验资产

1. 历史版本试验文档
2. 一次性验证脚本
3. 临时草稿
4. 无法解释用途的旧资产

### 4.3 本地私有数据

1. `docs/memory/mcp-context/session-state.json` 真实内容
2. 私人 checkpoint / task / event 全量历史
3. 本机 cache
4. 本机日志
5. 真实浏览器截图证据

## 5. 配置 sample 化清单

产品仓必须提供这些样例：

1. `.env.example`
2. `config/connectors.example.json`
3. `config/runtime.example.json`
4. `docs/deployment/environment-variables.md`
5. `docs/deployment/first-run.md`
6. `config/providers.example.json`

产品仓绝不能提供这些真实值：

1. OpenAI key
2. Notion token
3. GitHub token
4. Dataki key
5. 私有数据库地址
6. 本机绝对路径

## 6. codex 规格迁移清单

### 6.1 必迁回产品仓

1. 启动与恢复规则
2. 任务 / checkpoint / review / rework 协议
3. 云端同步规则
4. 跨会话衔接规则
5. 部署和使用约束

### 6.2 保留本地

1. 个人模型偏好
2. skill 安装状态
3. 本机插件配置
4. 本机脚本快捷方式

## 7. 旧 GitHub 仓处理策略

推荐顺序：

1. 先新建干净的 `PMOS product repo`
2. 按本清单迁移首版可发布内容
3. 完成首版部署验证
4. 再决定旧仓是归档还是清空重推

不建议先删旧仓内容再现场判断保留范围。

## 8. 首版发布前检查

发布前必须回答：

1. 其他设备能否按文档启动
2. 其他协作者能否不依赖当前会话上下文理解产品
3. ChatGPT 绑定 GitHub 后能否读到 v1.0 主真源
4. 产品仓是否已经去除业务子项目污染
5. 配置是否全部 sample 化
6. 是否还存在只能从 codex 私有目录得知的关键规则

## 9. 打包顺序

1. 盘点保留目录
2. 盘点排除目录
3. 提取必须迁回 PMOS 的 codex 规格
4. 生成配置样例
5. 生成 cloud-mirror 样例结构
6. 组装产品仓
7. 执行跨设备部署验证
8. 再接 GitHub / Notion 云端知识层
