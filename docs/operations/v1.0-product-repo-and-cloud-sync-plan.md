# PMOS v1.0 产品仓与云端同步规划

- version: v0.1
- date: 2026-05-09
- status: active
- owner: product office

## 1. 这份规划解决什么问题

`v1.0` 不能只回答“本地仓库里已经有平台能力”，还必须回答：

1. `PMOS` 能不能被打包成一个干净、可部署、可分享的产品仓
2. 真实协作时的文档、状态、checkpoint、review 能不能持续同步到云端知识层
3. ChatGPT 在绑定 `GitHub + Notion` 后，能不能尽量不依赖复制粘贴上下文理解当前状态
4. 哪些规格属于 `PMOS` 产品真源，哪些只属于本机 `codex` 执行环境

一句话定义：

`v1.0` 必须同时建立 `deployment-ready PMOS product repo` 与 `cloud knowledge sync layer`。

## 2. 正式目标

本规划固定两个新增的产品化目标：

### 2.1 独立产品仓目标

建立一个独立于研发母仓的 `PMOS product repo`，用于：

1. 对外发布
2. 其他设备部署
3. GitHub 作为主知识源接入 ChatGPT
4. 保持平台真源与业务子项目隔离

### 2.2 云端连续性目标

建立一个 `cloud knowledge sync layer`，用于：

1. 把本地真源持续镜像到云端
2. 把运行态状态快照持续镜像到云端
3. 让 `GitHub + Notion` 成为 ChatGPT 的主要外部上下文来源
4. 减少跨会话、跨设备、跨模型时的人工复制粘贴

## 3. 仓库边界

### 3.1 研发母仓

当前本地仓继续作为：

1. 平台研发母仓
2. 子项目实战母仓
3. 历史资产和实验资产承载仓

它不等于对外发布的 PMOS 产品仓。

### 3.2 PMOS 产品仓

`PMOS product repo` 只承载：

1. 平台运行时源码
2. 平台级真源文档
3. 可部署配置样例
4. 启动、运行、发布说明
5. 最小 demo / mock / sample 数据
6. 云端同步规范

### 3.3 不进入产品仓的内容

以下内容默认不进入 `PMOS product repo`：

1. `subprojects/*` 的业务项目主体
2. 业务项目的完整需求真源包
3. 历史试验资产
4. 本地私有运行状态
5. 个人账号、token、cookie、session
6. 与平台产品无关的临时脚本和临时文档

## 4. GitHub 与 Notion 分层

### 4.1 GitHub 的定位

`GitHub = system of record`

GitHub 承载：

1. 平台源码
2. 平台级真源文档
3. 结构化状态快照
4. review / proof / acceptance 结构化产物
5. 部署与运行文档

适合放到 GitHub 的对象：

1. `docs/operations/*.md`
2. `docs/architecture/*.md`
3. `docs/review/*.json`
4. `docs/memory/document-governance/*.json`
5. `cloud-mirror/runtime-status.json`
6. `cloud-mirror/runtime-status.md`
7. `cloud-mirror/checkpoints/latest.json`
8. `cloud-mirror/review-scorecards/*.json`

### 4.2 Notion 的定位

`Notion = operator cockpit`

Notion 承载：

1. 当前版本总控页
2. 项目状态总览
3. checkpoint 摘要流
4. blocker / decision 页面
5. review 打分汇总页
6. active runs 汇总页

Notion 不作为唯一真源，不承载高频结构化原始运行态。

### 4.3 分层原则

1. 正式真源先落本地仓，再镜像 GitHub
2. 人类协作页默认镜像到 Notion
3. 高频原始状态不直接散落在 Notion 多页面里
4. ChatGPT 默认通过 `GitHub + Notion` 读取云端知识，不再依赖会话贴文

## 5. 实时同步策略

仅靠云端文档本身不足以支撑实时协作，因此 `v1.0` 需要统一的 `latest-state` 镜像层。

### 5.1 必须新增的镜像对象

1. `cloud-mirror/runtime-status.json`
2. `cloud-mirror/runtime-status.md`
3. `cloud-mirror/checkpoints/latest.json`
4. `cloud-mirror/review-scorecards/latest.json`

### 5.2 runtime-status 最小字段

`runtime-status.json` 至少包含：

1. current product version
2. current runtime baseline
3. current highest-priority line
4. in-progress tasks
5. latest checkpoints
6. blockers
7. active subprojects
8. each active task's `owner / state / artifact`
9. latest review summary
10. last updated time

`runtime-status.md` 必须与 JSON 对应，但面向人阅读。

### 5.3 更新频率

建议固定三档：

1. 高频状态：`1-2 分钟`
2. 工作产物：`5-10 分钟`
3. 版本规划：事件触发

### 5.4 实时原则

1. 先写本地真源
2. 再生成 `latest-state`
3. 再推送 GitHub / Notion 镜像
4. ChatGPT 通过已绑定的云端源读取，而不是依赖人工复述

## 6. 配置文件发布规则

### 6.1 可以进入产品仓的配置

只能发布：

1. `.env.example`
2. `config/*.example.json`
3. `docker-compose.example.yml`
4. 环境变量说明文档
5. 配置项用途说明

### 6.2 不能进入产品仓的配置

必须留在本地并加入忽略：

1. `.env`
2. `.env.local`
3. 真实 token / key / cookie
4. 私有数据库连接串
5. 私有 Notion / GitHub / Dataki 凭证
6. 本机路径
7. 真实 session / state 文件

### 6.3 sample 化原则

1. 发布结构，不发布真实值
2. 发布字段说明，不发布个人环境
3. 发布样例状态，不发布真实私人状态

## 7. codex 规格边界

### 7.1 必须迁回 PMOS 产品仓的内容

凡是影响 `PMOS` 产品行为的规格，不允许只存在于 `codex` 私有目录。

例如：

1. 启动规则
2. agent 协作协议
3. 任务状态协议
4. review / rework / checkpoint 规则
5. 云端同步规范
6. 部署和运行约束

这些内容必须迁入：

1. `docs/operations/`
2. `docs/architecture/`
3. `docs/deployment/`
4. `config/*.example.*`

### 7.2 留在 codex 本机环境的内容

以下内容默认只留本地：

1. 个人 skill 安装状态
2. 本机插件偏好
3. 模型偏好
4. 账号绑定
5. 本机路径和私有脚本

### 7.3 双份处理原则

对于“既有产品意义，又有本机执行细节”的内容：

1. `PMOS` 仓保留正式规范
2. `codex` 本机保留执行器适配副本

## 8. 首版产品仓结构建议

建议首版最小结构如下：

1. `src/`
2. `frontend/` 或 `web/`
3. `docs/operations/`
4. `docs/architecture/`
5. `docs/deployment/`
6. `scripts/`
7. `docker/`
8. `.env.example`
9. `docker-compose.yml`
10. `README.md`
11. `cloud-mirror/`

## 9. v1.0 验收映射

以下能力正式并入 `v1.0` 验收：

1. 存在独立 `PMOS product repo`
2. 产品仓不混入业务子项目主体
3. 新设备可按文档完成部署
4. 至少存在一条稳定的 `本地真源 -> GitHub/Notion 云端镜像 -> ChatGPT 检索` 路径
5. `runtime-status.json` 与 `runtime-status.md` 可持续更新
6. 产品行为规格不再隐含依赖 `codex` 私有目录
