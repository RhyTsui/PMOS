# PMAIOS v0.5 收尾计划

- version: v0.1
- date: 2026-04-24
- status: active closeout
- purpose: 让 `v0.5` 从开放式平台愿景收敛为可正式收尾的一版真源

## 1. 收尾后的问题定义

当前 `v0.5` 的主要问题不是方向不够，而是方向太多且一直开着。

所以这轮工作的真实目标不是继续扩，而是：

1. 把 `v0.5` 压缩成有限的收尾项。
2. 明确哪些已经成立。
3. 明确哪些只保留基线。
4. 明确哪些后移。

## 2. 本轮收尾范围

本轮统一按以下 `14` 项处理：

1. 长时连续执行能力
2. 思考与同步并行
3. 分母式进度默认化
4. 意图识别后再选解法
5. 批量授权与阻塞分离
6. 多会话 / 多模型上下文恢复强化
7. 人读层与 AI 归档层强关联
8. Hermes 进一步产品化
9. Office 兼容链路
10. Web UI 产品化
11. AI 产品研发协同闭环全链路
12. 每日对话蒸馏升级到稳定机制
13. 文档 wiki 分层：人版 / AI 版
14. image2 做 PMAIOS 全景图

## 3. 收尾判定口径

### 已验收

要求：

- 有稳定真源
- 有实际入口或运行路径
- 不再需要继续扩目标才能算成立

### 收尾中

要求：

- 已有明确样板或最小运行路径
- 允许以“基线已具备，但未完全产品化”结束本轮

### 后移

要求：

- 明显属于下一阶段工程化、产品化或生态兼容化事项
- 如果继续做，会明显拖住本轮收口

## 5. 用户需求回查

### 原始用户诉求

- 不要漏需求、漏计划
- 不要只报完成项
- 要有稳定恢复和协作方式
- 要能支撑真实项目推进

### 当前判断

- 不要漏需求、漏计划 -> `partial`
- 不要只报完成项 -> `partial`
- 稳定恢复和协作方式 -> `partial`
- 支撑真实项目推进 -> `solved`

原因：

- 机制和真源已建立，但还没有覆盖到所有默认工作路径
- 对 `knowledge-base / WeKnora` 二开准备和开工判断已经形成直接支撑

## 8. 收尾后的执行规则

1. `v0.5` 不再继续新增开放式平台愿景条目。
2. `收尾中` 项默认以当前基线收口，不再被表述成“马上全做完”。
3. `后移` 项不得继续伪装成当前主线阻塞。
4. 如果后续继续做平台升级，必须以新任务或新版本重新立项。
