# v1.0 产品版本计划

- version: v0.2
- date: 2026-05-09
- status: active

## 1. 版本目标

`v1.0` 的目标不是继续证明 PMAIOS 的概念成立，而是交付一个高质量的产品版本：

`一个能干活、能部署、能协作、能交付的产品经理 Agent 产品。`

## 2. 正式范围

### 1. 产品经理 Agent 身份与入口

要解决：

1. 对外不知道 PMOS 是什么
2. 对内不知道它给谁用、怎么用

产出：

1. 系统介绍
2. 角色地图
3. 核心场景图
4. 统一入口与导航

### 2. 多角色工作承接

要解决：

1. 测试、UI、算法、前后端、运营进入后拿不到适合自己的结果包

产出：

1. 角色化承接策略
2. 差异化追问规则
3. 按角色输出的结果包 contract

### 3. 深拆链路

要解决：

1. 需求拆解太粗
2. 功能拆解不到接口
3. 接口拆解不到开发任务

产出：

1. `requirement-to-function matrix`
2. `function-to-api mapping`
3. `api-to-task mapping`
4. 影响面与风险模板

### 4. 专业评审与 Hermes 治理

要解决：

1. review 走过场
2. 多角色 agent 只是名字，没有强制生效

产出：

1. 专业 reviewer 强制映射
2. review packet
3. Hermes 治理闭环
4. writeback / closure 机制

### 5. 交付级前端页面生产

要解决：

1. 页面总是退回示意页、平铺页、Claude 风格壳
2. 页面没有真实动态交互和用户流程

产出：

1. 默认组件基线
2. 页面交付标准
3. 浏览器级验证
4. 前端反模式审计

### 6. 文档生命周期治理

要解决：

1. 补丁文档泛滥
2. 原文档更新导致内容丢失
3. 新旧文档互相矛盾

产出：

1. 生命周期状态机
2. 原文档优先更新机制
3. 冲突检测与真源指认
4. 存档 / 淘汰 / 删除规则

### 7. 知识层与系统现状 grounding

要解决：

1. agent 不理解当前系统现状
2. 文档和版本知识没有默认进入 workflow

产出：

1. Dataki 默认 knowledge context
2. 子项目知识库映射
3. stage / review 自动 grounding
4. `inbox -> 提炼 -> 路由 -> 知识层` 主链

### 8. 部署、操作、验收、发布

要解决：

1. 只能在当前机器和当前会话里跑
2. 无法成为别人可部署使用的产品

产出：

1. deployment guide
2. operator guide
3. acceptance standard
4. changelog / 更新记录
5. GitHub 共享方案
6. 独立 `PMOS product repo` 打包方案
7. GitHub / Notion 云端同步分层规范
8. `latest-state` 实时镜像对象
9. `codex` 规格迁移清单

## 3. 推进方式

`v1.0` 不再按“单点 feature 完成就算推进”的方式推进，而按：

`workflow 主链 + reviewer gate + Hermes 治理 + acceptance standard`

推进原则：

1. 每项能力都必须落到 workflow 主链
2. 每项能力都必须有 review 与验收口径
3. 每项能力都必须回查原始用户问题是否真正被解决

## 4. 当前版本分母

`v1.0` 当前固定按 `10` 条主 track 追踪：

1. 产品经理 Agent 身份与入口
2. 多角色工作承接
3. 深拆链路
4. 专业评审与 Hermes 治理
5. 交付级前端页面生产
6. 文档生命周期治理
7. 知识层与系统现状 grounding
8. 部署、操作、验收、发布
9. 云端知识同步层
10. PMOS 与 codex 规格边界

## 5. 完成标准

只有以下事情同时成立，才算 `v1.0` 完成：

1. 平台可部署
2. 平台可配置
3. 平台可被多人真实使用
4. 多角色工作入口可用
5. 交付级 workflow 全链条可运行
6. 前端页面生产和验收达标
7. 文档治理不再靠补丁维持
8. Hermes 能持续驱动治理闭环
9. GitHub 共享与多设备部署路径成立
10. 独立产品仓、云端知识层、配置 sample 化、codex 规格迁移边界同时成立
