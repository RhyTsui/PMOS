## 1. 反馈结论

`tracking-acceptance` 已经验证出一条可复用的实施链：

`需求池 -> 主真源 -> 原型/交互/UI分层 -> 最终前端交付 -> 会议纪要转任务链 -> 页面整改清单`

这条链不是概念性的，而是在真实项目中跑通过了。

因此，建议 PMAIOS 不把这次项目经验视为局部成功，而应判断为：

`已出现足够证据，值得升级为平台内置流程与能力。`

## 2. 建议升级为 PMAIOS 默认流程的内容

### 2.1 Requirement Pool 必须成为所有项目的入口门槛

建议：

- 每个活跃项目必须维护常设需求池
- 所有会议纪要、用户反馈、需求方建议、评审意见必须先进入需求池
- 没有需求池的项目，不能判定为流程完整

原因：

- 本项目清楚暴露出，没有常设需求池时，反馈会散落在聊天、会议纪要和页面里
- 一旦页面推进较快，就会出现“做了很多，但需求治理失真”

建议内化对象：

- workflow gate
- project entry contract
- adoption checklist
- requirement-pool template

### 2.2 设计确认链必须标准分层

建议平台默认采用如下分层：

1. 原型设计稿
2. 交互设计稿
3. UI设计初稿
4. 高保真视觉稿
5. 最终前端交付页

原因：

- 本项目多次因为“设计稿”概念混杂而返工
- 分层后，确认责任、修改责任、交付责任都明显更清楚

建议内化对象：

- design workflow
- design template
- confirmation gate
- design confirmation checklist

### 2.3 最终交付前端页的定义必须明确

建议平台正式定义：

`最终交付前端页 = 带真实前端交互、未接后端的最终产品前端交付物`

它不是：

- 示意稿
- 原型附带 HTML
- 仅演示页

原因：

- 本项目中这件事没有先定义清楚，导致多轮概念漂移

建议内化对象：

- product workflow
- front-end handoff spec
- implementation handoff template

### 2.4 页面整改默认先做差异诊断

建议平台默认要求页面整改按：

`现状 -> 规范 -> 差异 -> 修改`

执行。

原因：

- 本项目证明，直接 patch 页面会导致越改越散
- 一旦建立整改清单，逐页推进明显更稳

建议内化对象：

- page rectification workflow
- design review checklist
- frontend refactor checklist

### 2.5 inbox 原始输入必须隔离

建议平台正式规则化：

- `docs/sources/inbox/` 只做原始输入区
- 任何转换、导入、模板使用、资源复制，都必须先生成副本

原因：

- 本项目里已经实际暴露出原始输入被误改的风险

建议内化对象：

- global rule
- file handling protocol
- source ingestion checklist

## 3. 建议升级为 PMAIOS 默认能力的内容

### 3.1 需求池对象与工具

建议能力：

- requirement-pool object
- requirement-pool excel import/export
- requirement source kinds
- requirement promotion traceability

这是因为本项目已证明：

- 需求池不是抽象概念，而是高频真实需求

### 3.2 页面整改清单对象

建议新增平台对象：

- `page-rectification-checklist`

能力包括：

- 按页面记录整改目标
- 必须保持
- 继续整改点
- 不建议继续改
- 当前状态

原因：

- 本项目已经证明这类清单比散落在对话里更可执行

### 3.3 设计反馈转需求工具

建议新增能力：

- 设计师反馈 -> 需求池
- UI 规范 -> 通用设计规范
- 设计图/示意图 -> 页面整改条目

原因：

- 本项目中 UI 设计反馈的价值很高
- 但如果没有正式转换机制，就只能停留在附件里

### 3.4 会议纪要转需求链

建议平台内置：

- 会议纪要产品分析模板
- 会议纪要 -> 迭代需求模板
- 会议纪要 -> 开发任务表模板
- 会议纪要 -> 研发任务单模板

原因：

- 本项目已完整跑通过该链路，证明值得平台化

## 9. 05-07 新增一轮真实使用感受反馈

除了前述结构化经验，本次项目还暴露出一批更接近“操作系统可用性”的问题。这些问题不只是项目局部体验，而是会直接影响 PMAIOS 是否能在其他项目中高效复用。

### 9.1 不同阶段必须由不同专业 agent 主导

建议虚拟产品主管正式确认：

- 需求阶段
- 原型阶段
- 交互阶段
- UI阶段
- 前端阶段

这 5 个阶段不应由同一套默认工作方式粗暴贯穿，而应由不同的专业 agent / 专业模式主导。

原因：

- 当前项目已证明，不同阶段需要的判断框架完全不同
- 需求阶段需要问答、澄清、抽象、对象化
- 原型阶段需要结构化和边界控制
- 交互阶段需要行为链和状态机思维
- UI阶段需要设计规范、组件语义、视觉密度和一致性
- 前端阶段需要组件实现、交付标准、可运行交互与工程边界

建议动作：

- PMAIOS 新增“阶段型专业 agent 默认编排”
- 不再默认由一套通用问答或通用实现模式贯穿所有阶段

### 9.2 问答式工作模式应从“偶发”升级为“阶段能力”

当前感受表明：

- 问答式推进在需求阶段、部分项目、部分场景中非常有效
- 但它现在更像偶发能力，没有被平台正式编排

建议虚拟产品主管确认：

- 问答模式应成为需求阶段默认能力之一
- 不仅用于补答问题，也用于：
  - 需求澄清
  - 歧义消解
  - 边界冻结
  - 术语收口

建议动作：

- 将问答式需求澄清纳入 PMAIOS 默认需求阶段 workflow
- 形成可重复使用的 `question-driven requirement refinement` 能力

### 9.3 UI规范不能只存在，应自动生效

本项目暴露出一个高频低效问题：

- 即使已导入 UI 规范
- 即使已提供设计师提示词
- 后续仍不断需要人工反复补充：
  - 圆角多少
  - 滑动条不能溢出
  - 列宽如何处理
  - 组件间距如何统一
  - 颜色和字重如何控制

这说明当前问题不是“有没有规范”，而是：

1. UI规范可能缺失
2. UI规范可能存在，但没有被稳定读取
3. UI规范即使被读取，也没有转成自动生效的执行约束

建议虚拟产品主管确认：

- UI规范不应停留在参考文档层
- 应升级为可执行约束层
- 一旦项目建立正式 UI 规范，后续页面整改默认继承，不应反复让用户口述第二次

建议动作：

- 新增 `UI spec activation gate`
- 新增 `页面整改前自动读取并应用有效UI规范` 的默认流程

### 9.4 Hermes 自进化应解决“同类纠正反复发生”

用户明确表达的核心痛点是：

- 同样的 UI 纠正反复说
- 同类组件用法反复说
- 同类页面风格反复说

这正是 Hermes / 平台自进化应解决的问题。

建议虚拟产品主管确认：

- Hermes 不应只记录高层决策
- 还应吸收高频重复纠正，提升成“默认记忆和默认约束”

应重点吸收的类型：

- 某类组件一旦选定，后续类似需求优先复用同类组件
- 某类页面一旦定下布局原则，后续同类页面默认继承
- 某套 UI 规范一旦生效，后续相同项目不再反复确认同样细节

建议动作：

- 新增 `repeat-correction absorption` 机制
- 将“用户重复纠正 >= N 次”的内容自动升格为候选平台规则或项目规则

### 9.5 设计工具 / UI skill / 设计提示词当前“有接入但未真正生效”

用户当前的真实体验是：

- 工具和 skill 数量很多
- 但结果仍然常常表现为：
  - Claude 风
  - 信息过载
  - 传统后台味太重
  - 偏离需求
  - 难以直接改动

这说明平台当前缺的不是“更多工具”，而是：

- 工具触发门槛
- 工具生效确认
- 工具输出诊断
- 工具失败后的回退逻辑

建议虚拟产品主管确认：

- skill 是否被真正读取、真正生效，应成为显式可见状态
- 设计工具是否真的参与了当前输出，也应可见

建议动作：

- 新增 `skill activation check`
- 新增 `design tool effectiveness check`
- 不再只因“工具装了”就视为能力成立

### 9.6 平台目标应升级为“说一次，不再说第二次”

用户给出的最终目标非常明确：

- 说一次，不再说第二次
- 做完一个项目，其他项目都按这种方式来
- 用过某个组件，后面类似需求都默认用这个组件

这是非常清楚的平台目标，应正式提升为 PMAIOS 的产品标准：

`一次确认 -> 项目内默认继承 -> 平台内可复用推广`

建议动作：

- 把“减少重复说明成本”正式写进平台目标
- 把“同类问题默认沿用已确认解法”写进运行时规则
