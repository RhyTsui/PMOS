# 子项目二开接手与开工工作流

- version: v0.1
- date: 2026-04-24
- status: active draft
- scope: 需要基于外部系统、开源项目、既有内部底座或已落库工程进行二次开发的子项目

## 1. 目的

这份文档把 `knowledge-base / WeKnora` 这次二开准备经验上抬为 PMAIOS 平台级工作流。

要解决的不是“怎么改某一个项目”，而是以下共性问题：

1. 子项目接手时，容易直接跳进页面和功能实现，缺少准备阶段门禁。
2. 共享上下文、项目入口、版本治理已经存在，但还没有稳定承接“二开项目开工”。
3. 外部底座一旦不能直接跑起来，团队容易把“环境阻塞”误判成“方案不可行”。
4. 没有统一分母时，二开准备常常只报做了什么，不知道是否真的达到可开工状态。

## 2. 适用条件

满足任一条件，就默认走这套工作流：

1. 需要在开源项目、第三方产品或既有内部系统上做二开。
2. 需要接手一个代码量较大、边界不清、不能直接从零重写的工程。
3. 需要先判断“沿用、接管、替换、隔离”的策略，再决定实现路线。
4. 环境启动、镜像、依赖、权限、代理、许可证等外部条件会显著影响开工节奏。

## 3. 用户需求与产品需求

### 3.1 用户需求层

用户真正要的是：

- 能接手一个现有系统
- 能判断是否值得二开
- 能快速知道卡点在哪
- 能形成可持续推进的工程包，而不是一次性分析稿

### 3.2 产品需求层

PMAIOS 需要提供的是：

- 标准化准备真源
- 开工条件检查清单
- 代码扫描与证据产物
- 环境验证结论
- 架构边界与冲突判断
- 子项目入口与共享上下文接续
- 能映射回“是否已真正可开工”的回查结论

## 4. 标准分母

每个二开项目默认按以下 `8` 项分母跟踪：

1. 准备真源文档
2. 开工条件检查清单
3. 可执行扫描脚本或等价扫描能力
4. 首批扫描证据产物
5. 子项目专用 skill / capability 入口
6. 项目入口资产：`project-board.svg`、`roadmap-board.svg`、`decision-board.svg`、`change-log.md`
7. 环境启动与健康检查结论
8. 架构边界、接管策略、冲突判断

默认状态分类：

- `solved`：该项已经形成可复用真源或可重复执行结论
- `partial`：已有初稿或样板，但未形成稳定闭环
- `blocked`：依赖真实外部条件，当前无法继续推进
- `missing`：还没有明确放置位置或没有开始

## 5. 标准阶段

### P0 准备收口

目标：

- 不直接改业务模块
- 先补齐准备真源、清单、项目入口和共享上下文

最低产物：

- 准备真源
- 开工条件检查清单
- `mcp-context` 任务与 checkpoint
- 子项目入口资产

### P1 扫描与证据

目标：

- 把“我以为系统是这样”变成“仓库证据说明系统是这样”

最低产物：

- 文件树摘要
- 模块摘要
- 前后端路由/API 摘要
- 扫描报告

### P2 环境与开工门禁

目标：

- 区分“仓库问题、配置问题、依赖问题、外部网络问题”

最低产物：

- 依赖是否可用
- 启动命令是否明确
- 健康检查是否通过
- 阻塞是否已定位到具体外部依赖

### P3 架构定位与接管策略

目标：

- 在改业务前明确“底座、临时壳、平台层、项目层”边界

最低产物：

- 外部系统在整体架构中的定位
- 哪些能力沿用，哪些接管，哪些不上抬为平台真源
- 关键冲突和降级策略

### P4 产品化二开

只有当 P0-P3 达到可开工标准后，才进入页面、流程、权限、接口等产品化改造。

## 6. 必过门禁

以下规则默认必过：

1. 没有准备真源，不进入具体功能二开。
2. 没有扫描证据，不直接改主业务模块。
3. 没有环境结论，不把“跑不起来”直接判成方案失败。
4. 没有架构边界，不把外部系统临时壳误升为 PMAIOS 平台真源。
5. 没有项目入口，不视为可持续接手状态。
6. 没有回查到用户需求，不把“产品机制已写好”视为任务闭环。

## 7. 标准输出

每个二开项目至少应沉淀以下输出：

1. `<project>/docs/<system>二开准备方案.md` 或等价准备真源
2. `<project>/docs/<system>开工条件检查清单.md`
3. `<project>/docs/<system>环境或依赖清单.md`
4. `<project>/docs/<system>接管方案.md`
5. `<project>/docs/scan/*` 证据产物
6. `<project>/config/skill-registry.json` 或等价 capability 入口
7. `<project>/project-board.svg`、`roadmap-board.svg`、`decision-board.svg`、`change-log.md`

如需统一起稿，使用：

- `docs/templates/subproject_secondary_development_prep_template.md`

## 8. 回查规则

二开准备做完后，必须同时回答两层问题：

### 8.1 用户需求回查

- 现在是否已经可接手
- 现在是否知道主要阻塞点
- 现在是否知道下一步该做什么
- 现在是否已经进入可持续推进状态

### 8.2 产品需求回查

- workflow 是否可复用到下一个子项目
- 真源、模板、检查清单是否已经固化
- 共享上下文和项目入口是否已经接住这个流程

结果只能标为：

- `solved`
- `partial`
- `unsolved`

## 9. 当前样板

当前第一个正式样板是：

- `subprojects/knowledge-base/docs/WeKnora二开准备与PMAIOS-v0.5收敛方案.md`

当前样板已证明：

- 二开准备包可以被 PMAIOS 承接
- 子项目入口、共享上下文、扫描证据、环境阻塞定位可以形成闭环

当前样板仍未完全闭环：

- 还没有完全抽成平台模板前，容易继续停留在 `knowledge-base` 特例
- 环境阻塞诊断仍偏人工，尚未完全产品化为平台执行能力

## 10. 下一步平台化方向

1. 把二开准备分母接入子项目采用清单，而不是只在单项目文档里存在。
2. 把环境检查、扫描、启动验证逐步接入 CLI/API，而不是只靠人工命令。
3. 把“二开项目是否已达到可开工状态”做成可读、可回查、可同步的标准状态。
