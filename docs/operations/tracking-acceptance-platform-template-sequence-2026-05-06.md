# 埋点项目剩余平台化项模板顺序

- version: v0.1
- date: 2026-05-06
- status: active
- source project: `subprojects/tracking-acceptance`
- purpose: 把埋点项目剩余 4 类平台化缺口压成统一模板顺序，明确哪些先抽成 schema，哪些先抽成 product truth，哪些再进入 gate/runtime

## 1. 这份文档解决什么

`tracking-acceptance` 当前还剩 4 类高频缺口：

1. 标准底座平台化
2. 历史导入与版本继承
3. 查询方案与成熟看板
4. AI 回写对象

问题不在于没发现，而在于如果 4 条线同时散开推进，会再次回到：

1. 术语不统一
2. 对象和流程混写
3. 先做页面，后补真源
4. 局部实现有了，但平台承接关系没收口

所以这一轮先不做 4 条并行实现，而是先明确统一模板顺序。

## 2. 总体顺序

推荐顺序固定为：

1. 标准底座平台化
2. 历史导入与版本继承
3. 查询方案与成熟看板
4. AI 回写对象

原因：

1. 没有标准底座，后面三个对象口径都会漂。
2. 没有导入/继承对象，标准无法平滑接住历史数据和版本差异。
3. 没有查询对象，证据层和成熟度表达只能靠临时筛选。
4. AI 回写对象必须最后接，因为它依赖前面三类真源和确认链。

## 3. 四类模板的对象化次序

### 3.1 第一类：标准底座平台化

先抽什么：

1. `standard-baseline`
2. `standard-field-profile`
3. `standard-event-catalog`

先定义成什么：

1. product truth
2. schema
3. 项目裁剪关系

先不要急着做什么：

1. 不要先做前端配置页
2. 不要先做临时导入脚本
3. 不要先让 AI 直接生成标准结果

最低模板问题：

1. 这个项目的标准底座由哪些对象组成
2. 哪些是平台通用字段，哪些是项目裁剪字段
3. 标准事件、标准属性、标准校验分别归谁

### 3.2 第二类：历史导入与版本继承

先抽什么：

1. `import-task`
2. `inheritance-link`
3. `diff-confirmation-record`

先定义成什么：

1. 导入任务对象
2. 版本继承关系对象
3. 差异确认对象

先不要急着做什么：

1. 不要把 Excel/旧表直接视作真源
2. 不要只做“导入成功”状态而没有差异确认
3. 不要跳过人工确认直接合并继承结果

最低模板问题：

1. 历史导入来源是谁
2. 当前版本继承自哪里
3. 哪些差异是自动承接，哪些必须人工确认

### 3.3 第三类：查询方案与成熟看板

先抽什么：

1. `query-plan`
2. `evidence-query`
3. `maturity-board`

先定义成什么：

1. 查询方案对象
2. 可复用证据查询对象
3. 成熟看板对象

先不要急着做什么：

1. 不要直接做临时筛选 UI
2. 不要让查询只存在在日志工具里
3. 不要让成熟度判断只存在在报告描述里

最低模板问题：

1. 哪些查询是跨版本可复用的
2. 哪些查询能直接支撑验收证据
3. 成熟度看板是看对象状态、证据完备度，还是问题收敛度

### 3.4 第四类：AI 回写对象

先抽什么：

1. `ai-context-pack`
2. `ai-writeback-task`
3. `ai-writeback-confirmation`

先定义成什么：

1. AI 输入上下文包
2. AI 结果回写任务对象
3. 人工确认对象

先不要急着做什么：

1. 不要先做一个更强的聊天入口
2. 不要把 AI 回复直接当结果
3. 不要跳过确认链直接改正式真源

最低模板问题：

1. AI 这次被给了哪些上下文
2. AI 结果准备回写到哪个正式对象
3. 哪一步需要人工确认后才能生效

## 4. 每类模板统一回答的三件事

后续每补一类模板，都必须统一回答：

1. 哪些是 `schema`
2. 哪些是 `product truth`
3. 哪些是 `gate / confirmation`

默认判断方式：

- `schema`
  - 机器可读结构
- `product truth`
  - 人和流程共同引用的正式定义
- `gate / confirmation`
  - 决定能否继续推进的状态控制

## 5. 与前两步成果的关系

### 5.1 与确认链对象化

这四类模板里至少有两类直接依赖确认链：

1. 历史导入与版本继承
2. AI 回写对象

原因：

1. 差异没有确认就不能算继承完成
2. AI 没有人审确认就不能算回写完成

### 5.2 与 `DESIGN.md`

这四类里与 `DESIGN.md` 直接关联不强，但方法是相同的：

1. 先补正式对象
2. 再补 skill / 模板
3. 最后接 gate/runtime

## 6. 实际推进建议

如果继续往下做，最稳的推进顺序是：

1. 先补 `standard-baseline` 平台模板说明
2. 再补 `import-task / inheritance-link / diff-confirmation-record`
3. 然后补 `query-plan / maturity-board`
4. 最后补 `ai-context-pack / ai-writeback-task`
