# PMAIOS v0.5 研究与执行 Backlog

- version: v0.4
- date: 2026-04-22
- status: active planning backlog

## Purpose

这份文档是统一后的 `v0.5` 研究与执行 backlog。

它把战略方向转成可跟进的研究队列和实施队列，避免 `v0.5` 继续停留在散点想法层。

## 顶层规则

1. `v0.4` 要尽可能快地收口。
2. 外部阻塞项不能继续拖住 `v0.4`。
3. 被阻塞但仍有价值的项，应标成 carry-over 进入 `v0.5`。
4. `v0.5` 不是孤立连接器版本，而是下一次完整的操作系统级收敛。
5. 用户明确提出的重要需求，不能只留在对话或候选草案里，必须被晋升、挂起或拒绝，并留下原因。

## v0.5 主题

`协作模式 + 2 小时连续执行目标 + 优先级同步 + 意图识别后再选解法 + 更强 Hermes + 可复用产品技能 + 更深自动化 + 人阅读层 + AI 归档层 + 产品级 Web UI`

## Backlog

### 1. 协作与授权

- 研究 Claude Projects 协作模式
- 对比当前 PMAIOS 的文档驱动异步协作方式
- 优化长时自主执行下的授权、确认和协作协议
- 明确哪些动作在用户明确授权后可以安全自动继续

### 2. 自主编码效率

- 分析当前编码路径上的打断点
- 区分哪些确认是必须的，哪些只是 routine
- 提升自主编码路径的连续执行效率
- 让非阻塞工作在等待权限时继续推进
- 显式解决“阶段汇报 / 中间总结 -> 伪停工”的问题

### 3. 意图识别后再选解法

- 不把用户原始 idea 直接当实施指令
- 先做：
  - 领域驱动拆解
  - 行业最佳实践扫描
  - 相关性与影响评估
  - 边界与依赖澄清
- 先同步强化后的提案，再进入实施

### 4. Intent Amplifier / Delivery Ladder / Cognition Mirror

- 向上升维
- 向下降维
- 向外关联
- 向深研究
- 降维不能只拆成“今天做什么”，还要按项目交付链展开：
  - 研究
  - 规划
  - 设计
  - Figma
  - 前端 demo
  - 后端架构
  - 后端 demo
  - 全栈 demo
  - 可部署版本
  - 下游交付件
- 认知镜像要支持多维提炼，而不是只记几条 note

### 5. 每日对话整理与方法论蒸馏

- 每天晚上自动整理当天对话记录
- 识别其中的高价值方法点、规则点、skill 候选和 deep research 候选
- 支持白天实时识别时轻量同步：
  - “这条我会保留成方法论”
  - “这条更像一个规则候选”
  - “这条适合升级成 skill”
- 最后接入知识库与认知地图

### 6. 人阅读访问面

- 用固定 `IP + 端口 + 路径` 统一访问：
  - SVG
  - Markdown
  - DOCX
  - 版本文档
  - 项目入口
- 支持本机、公司内网其他电脑、远程本机、Pad 访问同一入口
- 优先暴露 chief-facing 和人读产物，不把 AI-only 归档默认展示给人

### 7. 项目实时叙事面

- 每个活跃项目默认维护：
  - `project-board.svg`
  - `roadmap-board.svg`
  - `decision-board.svg`
  - `change-log.md`
- 图板不是临时汇报材料，而是由更新日志持续驱动的实时阅读面
- 别人随时来问时，应能在项目目录或统一入口里直接打开

### 8. 人读层 / AI 归档层 / 强关系 / 知识图谱

- 人读层与 AI 归档层分开存放
- 两层之间建立双向强关系
- 支持知识库优先消费 AI 层建立图谱
- 人在知识库里能通过 AI 调到人读层产物
- 支持从 board 追到 session / decision / source / task

### 9. Extend Thinking / 强制思考插入

- 研究 Claude `extend thinking` 概念及相近设计模式
- 找出 PMAIOS 中哪些场景必须前置更强思考
- 重点候选场景：
  - Product Chief 分析
  - 需求收敛
  - 架构判断
  - review / gate 决策
  - 高风险代码改动

### 10. 持久化全局规则

- 强化 PMAIOS 的持久化全局规则层，不再只依赖单会话记忆
- 明确：
  - 平台级规则
  - 子项目级规则
  - 用户偏好规则
  - startup 恢复规则

### 11. 文档与 Office 格式硬化

- 修正文档编码和格式问题
- 提升 Word 兼容输出链路
- 增强 Excel 读写与构建循环
- 让版本文档、需求导出和人审文档能在 Markdown 与 Office 格式之间稳定流转

### 12. Hermes 与产品技能演化

- 强化 Hermes 在合适边界内向 orchestrator 演进的能力
- 明确 Hermes 责任拆分：
  - 规则执行
  - workflow selection
  - skill reuse
  - evidence collection
  - escalation
- 把高重复产品工作流重构成可复用技能

### 13. Intelligence Source Harvesting

- 定义对 PMAIOS 真有价值的信息源
- 建立轻量持续采集例程
- 把采集到的信号送入 governed input / requirement pool
- 对需要进入真实工作流的项，要求 chief 主动确认

### 14. 多模态与设计提升

- 增强多模态输入支持
- 补 UI / 设计短板，不把它当成边角问题
- 接入更强的设计技能、前端框架和组件系统
- 提升子项目基础 UI / UX 质量

### 15. AI 产研协同闭环

- 定义标准闭环：
  - AI 辅助产品 demo 验证
  - 演示后需求包
  - 现有系统改动说明
  - 相关 SVG 资产
  - 完整版本需求包
  - `aicoding` 全栈重实现
  - AI 测试验证
  - 上线后全流程回归
- 人读产物与 AI 产物分层，但强关联
- 需求变更应自动传播到 demo、文档、版本记录和测试资产
- demo 与正式实现必须显式分开

### 16. Web UI 产品化

- 把 Web UI 当成独立子产品处理
- 围绕真实 chief 工作流重构，不继续停留在 admin console 拼盘
- 收敛 outputs、monitoring、assets、action 四条主面
- 持续把共享上下文、agent 状态、阅读入口和交付链拉到同一个产品面
