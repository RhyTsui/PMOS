# PMAIOS v0.5 Checklist

- version: v0.4
- date: 2026-04-22
- status: planned
- purpose: 把 `v0.5` 收敛成可执行、可演示、可对齐的统一清单

## 核心目标

`v0.5` 不是继续堆更多零散功能，而是把 PMAIOS 推向更稳定的持续工作状态：

- 更长时间连续执行
- 更少无效确认和等待
- 更强的人机优先级同步
- 更清晰的人阅读层与 AI 归档层
- 更可用的知识导航与 Web UI

## Checklist

### 1. 连续工作能力

- [ ] 把 `2 小时连续执行` 作为明确目标，而不是模糊优化方向
- [ ] 区分真正阻塞工作与非阻塞工作，减少小确认带来的中断
- [ ] 在可安全前提下批量执行 routine 操作，不把每一步都变成确认点
- [ ] 形成“长任务期间持续推进，只有关键分叉才停”的默认执行模式

### 2. 思考与同步并行

- [ ] 默认要求“要思考，也要同步”，不能只埋头做，也不能只回复不推进
- [ ] 长任务执行中持续同步当前优先级、主线、支线、风险和偏差
- [ ] 用户没有反馈时，按当前同步过的优先级继续执行
- [ ] 用户反馈优先级不对时，立刻切换
- [ ] 阶段性汇报默认不是暂停点
- [ ] 中间总结默认不是收尾信号

### 3. 优先级外显与分母进度

- [ ] 把“当前优先级判断”显式同步给用户，而不是留在隐含流程里
- [ ] 长任务开始后，先同步目标、先做什么、为什么这个顺序最优
- [ ] 主线深入时保留其他线的可见性，避免单线钻深后忘掉其他高优先工作
- [ ] 进度汇报默认基于长目标分母，而不是只汇报“刚做完了什么”
- [ ] 长目标分母至少区分：总项数、已解决、已晋升进 plan/backlog、已挂起、仍未妥善落位

### 4. 意图识别后再选解法

- [ ] 不把用户原始 idea 直接等同于最终实施方案
- [ ] 进入实现前，先识别背后目的、约束、风险和更优替代路径
- [ ] 先做领域驱动拆解、行业实践扫描、相关性与影响评估，再决定是否实施
- [ ] 优先复用成熟开源能力或成熟方法，不默认从零搭建
- [ ] 这条规则出现在关键人机协同入口中，而不是只留在单次对话里
- [ ] 明确区分 `用户需求` 与 `产品需求`
- [ ] 产品需求完成后，必须回到原始用户需求场景走查，并标记 `solved / partial / unsolved`

### 5. 批量权限与阻塞分离

- [ ] 把“需要权限”的动作尽量成批整理，不零散逐个打断
- [ ] 把“必须等权限”的部分与“可以继续推进”的部分拆开
- [ ] 有阻塞时先并行做不受阻部分，避免整段时间空等
- [ ] 明确哪些工具和命令值得预授权，降低后续反复申请成本

### 6. 协作与上下文连续性

- [ ] 强化多模型、多会话、多项目切换后的上下文恢复
- [ ] 减少每次切换都要重新讲项目背景、工作方式和长期规则
- [ ] 强化 startup 读取、全局规则和共享上下文的可靠性

### 7. 人阅读访问面

- [ ] 用固定 `IP + 端口 + 路径` 统一访问 SVG、Markdown、DOCX、版本文档和项目入口
- [~] 每个活跃项目默认具备 `project-board.svg`、`roadmap-board.svg`、`decision-board.svg`、`change-log.md`
- [~] 保证项目目录和统一入口都可直接查阅

### 8. 人读层与 AI 归档层强关系

- [ ] 分离 human-facing 与 AI-facing 两层
- [~] 不让两层变成孤岛，先建立项目入口到 README / memory / workflow / PRD / 架构文档的最小回钻关系
- [ ] 支持从人读图板追到任务、决策、会话、来源文档
- [ ] 支持知识库同时消费 AI 层，并向人返回人读层内容

### 9. Hermes 与可复用产品技能

- [ ] 强化 Hermes 在 workflow selection、evidence collection、skill reuse、escalation 上的作用
- [ ] 把高重复 PM 流程做成可复用技能
- [ ] 建立“哪些技能真的产生复利”的判断标准

### 10. 规则与强制思考插入

- [ ] 在架构、需求收敛、评审、高风险改动等场景前置更强思考
- [ ] 把平台规则、子项目规则、用户偏好从会话记忆升级为持久真源
- [ ] 把“默认中文、中文交流、SVG 中文优先”前移到启动入口

### 11. 知识导航、Office 兼容与 Web UI 产品化

- [ ] 引入 Obsidian 风格的人类知识导航层，但不替代 repo 真源
- [ ] 强化 Word / Excel / DOCX 兼容与版本文档流转
- [ ] 继续把 Web UI 从后台拼盘收敛成真实产品面

### 12. 协作模式架构

- [ ] 不强行用一套默认行为覆盖所有场景
- [~] 建立模式路由层，至少支持 `default`、`plan`、`deep`、`do`
- [~] 允许通过显式指令进入 `{do}` 快速实施路线
- [~] 把模式状态接入共享上下文运行态，而不只停留在文档约定中
- [~] 让 CLI / API / Web UI 都能读取当前模式与模式历史
- [ ] 让不同模式影响汇报节奏、停顿条件、新需求插入策略和权限处理策略
- [ ] 先以实验机制运行，再决定是否更深接入执行器

### 13. AI 产研协同闭环

- [ ] 形成面向项目协作的标准 AI 产研流程，而不是产品、开发、测试各自零散调 AI
- [ ] 产品侧默认用 AI 产出并维护 demo、演示后需求文档、现有系统改动说明、相关 SVG、按版本模板维护的完整版本需求文档
- [ ] 人看的与 AI 看的产物分层管理，但保持强关系和可追溯
- [ ] 开发侧默认用 `aicoding` 基于产品产物重新实现一遍全栈工程，而不是直接把 demo 当生产系统
- [ ] 测试侧默认调用 AI 测试平台和 AI 测试流程做验证
- [ ] 发布上线后，由 AI 测试平台承担全流程回归评测
- [ ] 需求修改、增加、补全尽量自动同步到 demo、文档、改动说明、版本文档和测试资产

### 14. 每日对话整理与方法论蒸馏

- [~] 每天晚上默认整理当天对话记录，产出“今日高价值概念 / 方法 / 规则 / skill 候选 / deep research 候选”
- [~] 白天对话实时识别时，判断哪些内容应归档、哪些值得同步提示、哪些应直接升级进 PMAIOS 真源
- [ ] 对识别出的高价值方法点，允许轻量同步：“这条我会保留成方法论 / 规则候选 / skill 候选”
- [ ] 建立从日对话蒸馏结果升级到 `skill / rule / method asset / deep research` 的明确标准
- [ ] 后续将这条机制接入知识库与认知地图，而不只停留在会话归档

## 围绕 AI 产研协同闭环需要优化的点

### 1. 产物契约

- [ ] 明确产品交给开发的标准产物包，而不只是描述或 demo
- [ ] 明确开发交给测试的标准产物包，而不只是一套代码
- [ ] 明确测试回写给产品与版本系统的标准结果包

### 2. 需求变更自动同步

- [ ] 需求变更后，自动识别受影响的 demo、文档、SVG、版本项、测试用例和回归集
- [ ] 让变更同步不再依赖人工全量回忆和手工补丁

### 3. Demo 与正式实现边界

- [ ] 明确 demo 验证版与正式工程版的边界，避免 demo 直接滑成生产实现
- [ ] 定义从 demo 到 `aicoding` 重实现之间的输入契约和验收标准

### 4. 测试接入点

- [ ] 把 AI 测试平台前移，不只上线后用，也用于实现前后的验证和阶段性回归
- [ ] 让测试结果能回写到需求、版本、改动说明和项目图板

### 5. 版本化治理

- [ ] 每一轮产品、开发、测试闭环都应落到版本模板和版本记录里
- [ ] 让“当前版本改了什么、为什么改、验证是否通过”始终可追溯
