# PMAIOS 产品定义与方法论拆解稿

- version: v0.1
- date: 2026-04-27
- status: draft
- owner: product office
- purpose: 在任何页面、图谱、PPT、image2 设计稿之前，先收敛 PMAIOS 的产品定义、价值主张、方法论、优先级与关联关系

## 0. 为什么先写这份文档

当前 `PMAIOS` 的问题，不是材料不够，也不是页面入口不够，而是产品叙事还没有完成真正收敛。

现状偏差主要有三类：

1. 先做展示，再做产品定义。
2. 先堆能力名词，再讲系统为何成立。
3. 先改页面结构，再回答 `PMAIOS 到底是什么`。

所以这份文档的目标不是产出一个新页面，而是先建立一个可以经得起推敲的产品论证底稿。后续的 Wiki、Graph、SVG、image2、PPT，都应从这份文档往外长，而不是反过来。

---

## 1. 一句话定义

`PMAIOS = 一个由 Virtual Product Chief 驱动、以仓库真源为核心、面向复杂产品协作与持续交付的 AI Product Management Operating System。`

这句话里每个短语都不是修饰词，而是边界定义：

- `Virtual Product Chief 驱动`：说明它不是单个聊天助手，而是以“虚拟产品主管”作为治理核心。
- `仓库真源为核心`：说明它不是会话记忆系统，而是文档、规则、流程、版本和资产驱动系统。
- `复杂产品协作与持续交付`：说明它面对的是多角色、多轮次、多资产的产品工作，而不是单次回答。
- `AI Product Management Operating System`：说明它不是普通项目管理工具，也不是纯 Agent 展示台。

## 2. 5W2H 拆解

## 2.1 What

`PMAIOS` 是什么：

- 它是一个产品管理操作系统。
- 它的核心角色不是“万能 AI”，而是 `Virtual Product Chief`。
- 它的目标不是只产出文档，而是把需求、研究、方案、评审、交付、复盘串成一条受治理主链。

它不是什么：

- 不是单纯聊天工作台
- 不是知识库换壳
- 不是一堆 prompt 的集合
- 不是只做前端展示的 AI demo
- 不是“多 Agent 越多越强”的堆叠系统

## 2.2 Why

为什么必须有 `PMAIOS`：

现有 AI 协作方式的根问题，不是“模型不够强”，而是：

1. 需求与决策散落在聊天、文档、会议纪要和临时口头结论里。
2. AI 输出无法自动进入受治理资产，容易一次生成、一次丢失。
3. 多角色协作时，谁在决策、谁在研究、谁在交付、谁在复盘，边界并不清楚。
4. 产品工作经常能得到“答案”，却得不到“可追踪、可执行、可复用”的结果。
5. 展示层容易跑在产品定义前面，最终只剩信息拼盘。

所以 `PMAIOS` 要解决的不是“让 AI 更聪明”，而是“让 AI 协作变成一个可持续运转的产品系统”。

## 2.3 Who

核心用户不是一个群体，而是三层：

1. `真实产品主管 / 产品负责人`
作用：
- 给方向
- 定原则
- 做最终取舍

2. `产品与交付执行层`
包括：
- 产品经理
- 交付负责人
- 研发协同负责人
- 评审与运营角色

3. `未来的人读对象 / 演示对象`
包括：
- 管理层
- 合作方
- 新加入的执行成员

这三类用户不能混着讲。

如果是给产品负责人讲，重点是：
- 为什么它能替代碎片化产品协作方式

如果是给执行层讲，重点是：
- 它如何减少信息漂移和重复劳动

如果是给管理层讲，重点是：
- 它如何把 AI 协作变成治理化资产

## 2.4 When

`PMAIOS` 最有价值的场景，不是“问一个小问题”，而是：

1. 需求来源很多，口径容易散的时候
2. 方案需要多轮研究和收敛的时候
3. 交付物必须和版本、评审、风险关联的时候
4. 重要产品判断需要留下依据和变更记录的时候
5. 要把一次次临时协作沉淀成长期方法论的时候

## 2.5 Where

它嵌在产品工作流中间，而不是外面：

`信息输入 -> 研究 -> 方案 -> 决策 -> 输出 -> 评审 -> 交付 -> 复盘 -> 方法沉淀`

`PMAIOS` 不是某个环节上的外挂工具，而是这条链的治理壳。

## 2.6 How

它通过五类机制工作：

1. `角色治理`
由 `Virtual Product Chief` 统筹不同产品管理角色与 specialist。

2. `工作流治理`
把产品工作拆成明确阶段，而不是任意跳步。

3. `资产治理`
把输出写回 requirements、versions、reports、prompt packs、reviews、digests。

4. `规则治理`
让 prompt、template、workflow、review gate 共同约束行为。

5. `共享记忆治理`
通过 `mcp-context`、版本记录和文档真源，降低跨会话、跨工具、跨角色的信息丢失。

## 2.7 How Much

当前虽然还没有完整量化数据，但至少可以明确它在改善三类成本：

1. `沟通成本`
减少需求、决策、结论在聊天里反复解释和重复确认。

2. `治理成本`
把临时判断沉淀为可回溯资产，而不是每次从头理解。

3. `交付成本`
让产品输出和版本、评审、设计、交付链对齐，减少“写了很多但接不下去”的浪费。

同时也要承认它增加了两类成本：

1. 规则建立成本
2. 初期维护真源和结构化资产的成本

这不是缺点，而是系统化的必要代价。

## 3. 五层要素拆解

## 3.1 问题层

这是 `PMAIOS` 存在的原点。

核心问题不是页面长什么样，而是：

1. 输入散
2. 决策丢
3. 责任模糊
4. 输出不可复用
5. 复盘无法回流

如果问题层不成立，后面的架构、页面、图、PPT 都是空转。

## 3.2 价值层

`PMAIOS` 提供的价值，不是“多一个 AI”，而是让产品工作形成闭环：

1. 把原始输入变成可治理信息
2. 把研究过程变成可比较判断
3. 把判断结果变成可追踪输出
4. 把输出结果变成可交付资产
5. 把交付结果变成下一轮方法升级输入

所以它的价值核心是：

`把 AI 从会话能力，提升为产品协作闭环能力。`

## 3.3 机制层

价值不是自动成立的，必须靠机制支撑。

当前最关键的机制应收敛为五组：

1. `Virtual Product Chief`
负责顶层产品治理、派工、收敛和判断。

2. `Specialist / Manager Agent 结构`
把需求、版本、评审、交付、复盘等职责分开，而不是一个角色全做。

3. `Governed Outputs`
所有重要输出都不是临时文本，而是受治理产物。

4. `Workflow + Review Gates`
产品工作必须经过阶段和检查点，而不是只靠语言连续性。

5. `Shared Context + Repository Truth`
跨工具、跨会话协作，必须依赖共享状态和仓库真源。

## 3.4 资产层

机制的最终结果必须沉淀为资产，否则系统不会持续进化。

关键资产包括：

- requirement cards
- version records
- review records
- delivery outputs
- prompt packs
- shared docs
- daily digests
- workflow definitions
- templates
- checkpoints

`PMAIOS` 的成熟度，不只看它能不能回答问题，而是看这些资产是否持续增长、彼此关联、可回放。

## 3.5 呈现层

呈现层永远最后讨论。

呈现层当前应被视为以下对象，而不是产品本体：

- `Wiki`：状态型介绍页
- `Prompts`：独立提示词清单页
- `Graph`：结构关系图入口
- `Boards`：图板与图解入口
- `Workspace`：运行控制台

如果前四层没收敛，呈现层越做越散。

## 4. 优先级排序

## 4.1 P0：必须先收敛

这些东西不清楚，任何视觉设计、PPT、Wiki 重构都不应继续深入。

1. `PMAIOS` 的一句话定位
2. `Virtual Product Chief` 的角色定义
3. 产品主问题与主价值
4. 产品主工作流
5. “它不是什么”的边界定义

## 4.2 P1：让系统成立

这部分决定它是不是一个真正的产品系统，而不是概念堆积。

1. 能力分层
2. 角色分层
3. 资产分层
4. 规则治理边界
5. 方法论到工作流的映射

## 4.3 P2：让系统被理解

这部分才进入表达层：

1. Wiki 结构
2. Graph 导航结构
3. SVG / image2 / PPT 图示
4. 对外叙事顺序
5. 演示路径

也就是说，`P2` 不能跑到 `P0` 前面。

---

## 5. 关联关系图

应按下面的因果关系理解，而不是平铺能力：

`问题层 -> 价值层 -> 机制层 -> 资产层 -> 呈现层`

进一步展开：

`产品问题`
-> 决定 `产品定位`
-> 决定 `价值主张`
-> 决定 `Virtual Product Chief` 为什么成立
-> 决定 `workflow / review / outputs / memory` 应该如何分工
-> 决定 `requirements / versions / prompts / docs / digests` 需要如何沉淀
-> 最后才决定 `Wiki / Graph / Boards / PPT / image2` 如何呈现

这个顺序一旦反过来，就会出现：

- 页面看起来很多
- 名词看起来很多
- 图看起来很多
- 但用户依然不明白 `PMAIOS` 到底是什么

## 6. 对外叙事建议

如果后续要做“几页 PPT 级”的表达，建议先按以下页序组织，而不是先设计页面组件：

1. `封面页`
一句话定义 PMAIOS。

2. `问题页`
为什么现有 AI 协作方式不够。

3. `定位页`
PMAIOS 是什么，不是什么。

4. `角色页`
为什么核心是 `Virtual Product Chief`，而不是普通 Agent。

5. `方法论页`
PMAIOS 如何把输入、研究、决策、输出、复盘串成主链。

6. `架构页`
问题层、价值层、机制层、资产层、呈现层的关系。

7. `价值页`
对产品负责人、执行层、管理层分别提供什么价值。

8. `状态页`
最近动作、当前阶段、下一步重点。

9. `附录页`
prompt、模板、图板、版本、路径索引。

这意味着：

`prompt` 不是主叙事中心。
`页面导航` 也不是主叙事中心。
主叙事中心是 `PMAIOS 作为 Virtual Product Chief 驱动的 Product OS 为何成立`。

## 7. 当前判断

围绕“如何把这件事做好”，当前结论如下：

- 继续直接改页面：`unsolved`
- 继续先做图再反推产品：`unsolved`
- 先收敛产品定义、价值、方法论、优先级和关联关系：`correct`
- 以 `5W2H + 五层要素拆解 + P0/P1/P2 优先级` 作为下一阶段工作底盘：`solved`

---

## 8. 下一步建议

基于这份文档，建议后续按这个顺序推进：

1. 产出 `PMAIOS 产品定位与价值主张`
2. 产出 `PMAIOS Virtual Product Chief 角色定义`
3. 产出 `PMAIOS 五层产品架构图`
4. 产出 `PMAIOS 对外叙事大纲`
5. 最后才进入 Wiki / Graph / image2 / PPT 呈现重构

如果这 5 步不按顺序走，展示层大概率还会再次跑偏。
