# PMAIOS HTML-First Document Output Policy

- version: v0.1
- date: 2026-05-18
- status: active
- source input: `docs/sources/inbox/PMAIOS_HTML_First_Architecture_Decision.docx`
- scope: platform document output, human-reading surfaces, runtime-rendered product artifacts

## 1. Decision

PMAIOS adopts an HTML-first policy for runtime-facing and human-reading delivery surfaces.

This is not a decision to let agents freely generate arbitrary HTML. The actual architecture decision is:

`source material -> structured schema / PMOS DSL -> registered renderer -> HTML runtime surface`

HTML is the rendered product surface. Schema, registry bindings, source references, and lifecycle metadata remain the governed contract.

## 2. Why This Change Exists

The original document system separated:

- AI-reading artifacts
- human-reading artifacts

That split remains useful, but the human-reading layer now needs to carry more than prose:

- PRD pages
- dashboards
- workflow views
- review pages
- explanation pages
- trace / DAG / timeline views
- agent status and event streams
- runtime workbench surfaces

Markdown remains good for source text and durable Git review, but it is weak as the final surface for complex layouts, live state, component semantics, tool calls, and runtime patching.

## 3. Format Roles

### Markdown

Use Markdown for:

- source documents
- PRD source text
- meeting notes
- long-form reasoning
- RAG-friendly knowledge
- Git diff review
- appendices

Markdown is still valid. It is no longer the default final surface when the artifact is expected to behave like a product page.

### JSON / UISchema / PMOS DSL

Use structured formats for:

- page contracts
- component bindings
- workflow definitions
- runtime state shape
- evidence and source references
- approval / audit semantics

These are the durable contracts for AI and engineering.

### HTML

Use HTML as the default rendered surface for:

- human-reading product pages
- runtime-visible reports
- review and approval workspaces
- dashboards
- workflow and trace exploration
- delivery-quality UI direction comparisons

HTML should normally be generated by a renderer from governed schema. Raw standalone HTML may be used only as a temporary direction artifact or prototype, and must be converted back into governed schema / registry bindings before it becomes active product truth.

### SVG

SVG remains the default chief-facing artifact for:

- research summaries
- planning boards
- architecture maps
- workflow diagrams
- decision and confirmation boards

SVG-first is therefore still active for visual reasoning and decision communication. HTML-first overrides it only when the target is an interactive human-reading page or runtime product surface.

## 4. Hard Rules

1. Do not let AI generate arbitrary production HTML as the source of truth.
2. Production HTML must come from approved schema, PMOS DSL, or registered component bindings.
3. Any HTML page that represents a governed decision must carry source references, uncertainty, owner / reviewer state, and lifecycle status.
4. Any page with risky actions must carry approval and audit semantics.
5. Markdown and SVG companions must link back to the active HTML / schema truth when the HTML runtime surface is the current review object.
6. Static HTML confirmation is not equal to final implementation unless its schema, component registry, and acceptance path are explicit.

## 5. Recommended Output Chain

For product and design delivery:

`source input -> requirement baseline -> functional spec -> html-direction-pack -> ui-schema-spec -> registered renderer -> HTML runtime surface -> implementation handoff -> verification`

For planning and architecture:

`source input -> reasoning / research -> SVG decision board -> architecture decision record -> schema / implementation contract when needed`

For operational reports:

`runtime data -> report schema -> registered renderer -> HTML report page -> Markdown summary only when needed`

## 6. User Requirement Backcheck

Original user requirement:

Some future documents should be allowed to default to HTML output. This is close to the previous document architecture that separated AI-reading artifacts from human-reading artifacts.

Product requirement:

- Preserve the AI-reading / human-reading distinction.
- Upgrade the human-reading layer from prose documents to runtime-rendered HTML where the artifact needs layout, interaction, or live state.
- Keep Markdown / JSON / schema as AI-readable and governance-friendly contracts.
- Prevent HTML-first from becoming uncontrolled raw HTML generation.

Result classification: `solved`

Reason:

This policy accepts HTML as the default for suitable final human-facing/runtime surfaces, while keeping the existing two-layer document architecture and adding schema / registry constraints so the change does not weaken governance.

## 7. Current Adoption Boundary

Adopt immediately for:

- new PMOS runtime reports
- review / approval pages
- dashboard-like product artifacts
- HTML direction packs
- human-reading pages that need layout or interaction

Do not automatically migrate:

- existing Markdown truth sources
- historical documents
- meeting notes
- RAG knowledge bases
- immutable archive artifacts

Migration should happen only when an active workflow needs an HTML runtime surface.
