# v0.7 最后核心收尾方案

- version: v0.1
- date: 2026-05-09
- status: active
- scope: `v0.7` final closure only

## 1. 目标边界

这份文档只定义 `v0.7` 最后几件核心事情。

不在这份文档里扩写：

- `v0.8` 产品化发布叙事
- PMOS 对外介绍包装
- GitHub 共享部署包装

`v0.7` 的目标不是再开新故事，而是把已经成立的平台主干真正收口成：

- 可跨项目复用
- 可治理文档漂移
- 可交付前端页面
- 可强制多 agent 生效
- 可让 Hermes 进入更强自治闭环

## 2. 分母

最后核心收尾项固定为 `6` 项：

1. 跨项目 rollout
2. 旧文档更新治理机制
3. Hermes 跨项目默认写回 / 更广 writeback target / 更强 unattended autonomy
4. delivery-grade workflow
5. 前端页面产出质量与效率的手术级治理
6. 多 agent 提示词优化与强制生效机制

## 3. 当前判断

### 1. 跨项目 rollout

- current state: `partial`
- 现状：
  - 平台本体已经形成主链、Hermes、proof、review、Dataki grounding
  - 但更多子项目还没有默认继承同一套链路
- 真正缺的不是“再写一份方法文档”，而是：
  - subproject 默认配置映射
  - shared workflow contract rollout
  - rollout adoption dashboard
  - rollout blocker 分类

### 2. 旧文档更新治理机制

- current state: `unsolved`
- 这是目前最严重的平台级治理缺口之一

根因不是“AI 会多写几个文档”这么简单，而是 6 层失控：

1. 缺少单域单真源机制
2. 缺少文档生命周期状态机
3. 缺少“更新原文档优先”而不是“新增补充文档优先”的硬规则
4. 缺少长文重写与收敛机制，AI 倾向生成短片段补丁
5. 缺少淘汰、归档、废弃标记与引用迁移
6. 缺少“防丢失”机制，导致旧内容在整理中被覆盖掉

### 3. Hermes 跨项目默认写回与更强自治

- current state: `partial`
- 已有：
  - research
  - committee grounding
  - watch
  - promote
  - writeback
  - closure
- 还缺：
  - 跨项目默认 writeback mapping
  - 业务对象 / 指标口径 / 方案优解 / 开发 task 评审知识
  - unattended autonomy 的安全执行边界

### 4. delivery-grade workflow

- current state: `partial`
- 已有：
  - 交付主链 contract
  - review committee
  - proof-of-work acceptance package
- 还缺：
  - 真正按交付标准约束前端页面
  - 文档到工程任务的强 handoff
  - rollout 后的一致交付验证

### 5. 前端页面产出质量与效率问题

- current state: `unsolved`
- 这是业务阻塞级问题

根因要拆透，不然永远只会一页页返工：

1. 错误的产物链
   - 之前把前端阶段拆成“简单版 + UI/UX版”
   - 这天然鼓励产出示意页、说明页、平铺页

2. 缺少“页面必须面向用户”的强制检查门
   - 没有把：
     - 布局是否正确
     - 功能模块是否合理
     - 用户体验流程是否成立
     - 动态交互是否闭环
     写成 gate

3. 组件系统没有前置
   - 先写页面，再补组件映射
   - 导致总回落到通用 AI 审美和 Claude 遗留壳

4. 旧前端遗留文件持续污染
   - Claude 风格 hero/card/workbench 旧页面仍在 repo 里
   - 它们持续作为“可复用参考”污染新页面

5. 没有系统级验证
   - 缺少 page audit
   - 缺少真实交互 smoke
   - 缺少 screenshot / browser-level 复核

### 6. 多 agent 提示词优化与强制生效

- current state: `unsolved`
- 不是“提示词不够长”，而是“提示词没有进入强制运行时”

根因是 7 层：

1. 多角色 agent 是方法定义，不是强制运行时实例
2. 角色选择还是软约束，不是 gate
3. 缺少阶段到专业 agent 的硬映射
4. 缺少各 agent 必交付的结构化 review packet
5. 缺少 prompt activation trace
6. 缺少“没经过专业 agent 不许过 gate”的失败条件
7. 缺少回写到 requirement / workflow / template / prompt 的长期吸收机制

## 4. 系统性解决方案

### A. 跨项目 rollout

必须落 4 个机制：

1. `platform -> subproject` 默认能力映射
2. rollout adoption dashboard
3. rollout blocker taxonomy
4. rollout completion gate

完成标准：

- 至少一组重点子项目默认继承同一条交付主链
- rollout 不再靠人工解释，而是配置可见、状态可见、阻塞可见

### B. 文档生命周期治理

必须建立正式状态机：

`draft -> active -> superseded -> archived -> deleted`

并配套 8 条硬规则：

1. 单域单真源
2. 更新原文档优先
3. 补充文档只能临时存在，必须回并
4. 长文重写优先，禁止只打碎片 patch
5. superseded 文档必须写 successor
6. archived 文档不得继续作为当前引用入口
7. delete 必须先完成引用迁移
8. AI 改文档必须保留“旧内容去向说明”

### C. Hermes 跨项目自治深化

必须落 5 层：

1. 项目类型 -> writeback target family 映射
2. 业务对象 / 指标口径 / 方案优解 / 开发 task 评审知识注入
3. writeback target 扩展到 workflow / prompt / template / rules / subproject defaults
4. unattended autonomy 安全边界
5. 跨项目 watch memory 与 rollout evidence

### D. delivery-grade workflow

必须变成：

`文档 contract + review gate + engineering handoff + proof closure`

重点不是“再补阶段名”，而是让每一阶段都能真实支持下游交付。

### E. 前端页面手术级治理

这块必须作为专项治理，不再按页面逐个返工。

必须一起做 6 件事：

1. 删掉旧前端产物链
   - 不再保留“前端简单版”
   - 前端页面阶段直接是交付级页面

2. 建立 `page delivery gate`
   - 必查：
     - layout correctness
     - module responsibility
     - user flow
     - dynamic interaction
     - UI spec alignment

3. 建立 `component-first mapping`
   - 先定组件映射，再写页面
   - chat 区、企业系统壳、数据表、筛选、表单、通知分别有默认基线

4. 清理 Claude 遗留文件
   - 不只是 ban className
   - 要做：
     - legacy file inventory
     - replace / archive / delete policy
     - migration target mapping

5. 引入行业最佳实践与工具
   - design-system baseline
   - component mapping registry
   - browser-level check
   - frontend style audit
   - 可视化 smoke / screenshot review

6. 建立 reference shells
   - PMOS control plane shell
   - chat workspace shell
   - enterprise data workbench shell
   - config / form / table shell

### F. 多 agent 强制生效

必须从“写 prompt”升级到“强制运行时编排”。

需要 6 个机制：

1. `stage -> required specialist agents` 硬映射
2. 每个 agent 的必交付 packet 结构
3. agent activation trace
4. committee 必须聚合各 agent 结果
5. 缺角色即 block
6. 结果默认进入 Hermes writeback / watch / promote

## 5. v0.7 完成判定

这 6 项不是都要做到“终局版”，但都必须从方法论变成平台真实机制。

`v0.7` 最后收尾成立的标准：

1. 跨项目 rollout 有真实 adoption 面，不再只在平台本体成立
2. 文档生命周期治理有正式状态机、规则和执行入口
3. Hermes 能跨项目默认写回并扩展到更广 target
4. delivery-grade workflow 不再停在文档 contract
5. 前端页面质量问题从局部返工变成系统级治理
6. 多 agent 不再是口头设定，而是 runtime 强制生效

## 6. 与 v1.0 的边界

以下内容属于 `v1.0` 产品版本，不放进本轮 `v0.7` runtime 收尾定义：

- PMOS 系统介绍重写
- PMOS 更新记录体系
- 操作系统化文档重构
- 工程剥离与框架梳理
- 产品化验收标准
- GitHub 对外共享与多设备部署
