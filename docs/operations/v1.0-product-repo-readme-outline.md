# PMOS v1.0 Product Repo README Outline

- version: v0.1
- date: 2026-05-09
- status: active
- owner: product office

## 1. Purpose

This outline defines how the standalone `PMOS product repo` README should be rewritten.

The goal is not to describe every historical asset in the mother repo.
The goal is to help a new external operator understand:

1. what PMOS is
2. what this repo contains
3. how to configure it
4. how to boot it
5. how to connect cloud knowledge

## 2. README Must Answer

1. What is PMOS
2. What version is active now
3. What is the runtime baseline
4. What is included in this product repo
5. What is intentionally excluded
6. How to install and run
7. How to configure providers and connectors
8. How to verify the first-run loop
9. How to connect GitHub and Notion for cloud knowledge continuity

## 3. Recommended README Structure

### 3.1 Title and One-Line Definition

Example intent:

`PMOS is a deployable product-manager Agent platform for governed product workflow, review, delivery, and cloud knowledge continuity.`

### 3.2 Current Version Statement

Must clearly state:

1. active product version is `v1.0`
2. landed runtime baseline is `v0.7`
3. this repo is the standalone product repo, not the full mother repo

### 3.3 What This Repo Includes

Should mention:

1. platform runtime
2. platform truth docs
3. workflow definitions
4. prompts
5. deployment docs
6. cloud mirror samples

### 3.4 What This Repo Excludes

Should mention:

1. business subproject bodies
2. private raw inbox materials
3. local chat history
4. private tokens and local-only state

### 3.5 Quick Start

Should include:

1. prerequisites
2. install
3. `.env` setup
4. dev run
5. build run

### 3.6 Verification

Should include:

1. lint
2. test
3. build
4. first-run checks

### 3.7 Cloud Knowledge

Should explain:

1. GitHub as system of record
2. Notion as operator cockpit
3. `cloud-mirror` as latest-state layer

### 3.8 Truth Docs

Should point to:

1. platform truth index
2. current version progress
3. v1.0 direction
4. v1.0 program
5. v1.0 acceptance

## 4. Tone Rules

The standalone product repo README should:

1. be product-facing
2. be deployment-facing
3. avoid internal-only shorthand
4. avoid explaining unrelated historical experiments

## 5. Non-Goals For README

It should not:

1. list every legacy version document
2. describe every subproject in detail
3. depend on the reader knowing the mother repo history
4. depend on current conversation context

## 6. Next Step

After this outline, the next direct action is to rewrite the product repo `README.md` itself against this structure.
