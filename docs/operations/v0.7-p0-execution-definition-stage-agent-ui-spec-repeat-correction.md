# PMAIOS v0.7 P0执行定义：Stage-Agent / UI Spec Activation / Repeat-Correction Memory

- version: v0.1
- date: 2026-05-07
- status: active
- owner: product office

## Purpose

这份文档把 `tracking-acceptance` 三天实施经验中已经被证明必须平台化的 3 个 P0 能力，压成可执行定义。

目标不是再解释“为什么重要”，而是明确：

- 要做什么
- 不做什么
- 最小运行形态是什么
- 如何判断已经生效

## Scope

当前 P0 只覆盖：

1. `stage-agent orchestration`
2. `ui-spec-activation-gate`
3. `repeat-correction-memory`

不在本轮覆盖：

- 完整自动外部研究执行
- 完整跨项目 UI 设计系统
- 自动代码修改推荐引擎
- 任意设计工具深度双向同步

## 1. Stage-Agent Orchestration

### Definition

平台不再默认用同一套泛化工作模式贯穿所有阶段。

必须显式区分以下阶段：

1. `需求阶段`
2. `原型阶段`
3. `交互阶段`
4. `UI阶段`
5. `前端阶段`

### Required Behavior

#### 需求阶段
- 默认启用 question-driven clarification
- 输出对象、边界、术语、成功标准
- 先入需求池，再进入方案层

#### 原型阶段
- 优先做结构、对象归位、页面职责
- 不默认进入视觉表达

#### 交互阶段
- 优先做状态流、切换、动作、异常态
- 不默认进入视觉风格判断

#### UI阶段
- 必须读取 active UI spec
- 必须优先对齐组件语义、布局密度、色彩语义
- 页面整改默认先做差异诊断

#### 前端阶段
- 必须优先使用真实组件语义
- 以 delivery-grade 页面为目标
- mock 数据可用，但不再把示意壳误当交付物

### Not Allowed

- 同一套 generic 模式直接贯穿 5 个阶段
- 在需求未收口时直接滑向前端实现
- UI 阶段跳过规范读取

### Minimum Runtime Form

最小运行态要求：
- workflow 已写入 stage-specific rule
- active project checklist 能检查是否采用
- 至少一个真实项目已经验证该链路（当前为 tracking-acceptance）

### Acceptance

- 至少一个项目中，这 5 个阶段不再由同一套 generic mode 处理
- 需求阶段出现问答式澄清证据
- UI 阶段出现规范驱动整改证据
- 前端阶段出现真实组件交付证据
- 运行时工作台能直接展示这 5 个阶段的证据快照

## 2. UI Spec Activation Gate

### Definition

只要项目已有 active UI 规范或 design-system baseline，后续 UI-stage 输出和页面整改必须默认读取并生效。

### Required Behavior

- 项目存在 UI 规范时，页面整改前必须先加载
- 未加载时应被视为 workflow warning
- 不允许 UI 规范只停留在参考文档层

### Effective Spec Scope

默认应覆盖：
- 字体与字号层级
- 圆角
- 色彩语义
- 标题层级
- 间距与密度
- 表格滚动与列宽策略
- 组件优先级

### Not Allowed

- 用户已提供 UI 规范，但后续页面仍不断重复确认同类细节
- skill / agent 明知有规范仍按默认模型风格输出

### Minimum Runtime Form

- workflow 已有 UI spec activation gate
- adoption checklist 能判断项目是否存在 UI spec activation evidence
- 至少一个项目验证过（当前为 tracking-acceptance）

### Acceptance

- 页面整改前自动读取活动 UI 规范
- 后续同类细节不再反复由用户手动补充
- 若未加载规范，系统能显式暴露
- 运行时工作台能直接展示 active spec、effective rules 与缺口

## 3. Repeat-Correction Memory

### Definition

项目内重复纠正不能只停留在聊天记录里，必须被吸收成项目默认规则，必要时升级为平台候选规则。

### Required Behavior

- 识别同类重复纠正
- 判断是项目级还是平台级
- 项目内同类页面默认继承已确认纠正
- 对高频重复纠正形成候选平台规则

### Typical Absorption Targets

- 组件选型
- 标题层级
- 颜色语义
- 滚动条与局部滚动策略
- 表格列宽与固定列习惯
- 页面信息密度
- 不允许的说明性文案

### Not Allowed

- 同一项目里用户对同类问题重复说多次仍无吸收
- 用过一次的组件或结构，后续类似页面仍从零猜

### Minimum Runtime Form

- review/feedback 文档里能显式识别重复纠正模式
- project-level rule 可承接重复纠正
- roadmap 已把该能力列为 P0

### Acceptance

- 同一项目里同类纠正出现第 2 次后，能被识别为重复纠正候选
- 候选可直接提升进入需求池
- 用户不需要再靠聊天上下文手工回忆重复纠正内容

## Back-Check Against User Goal

原始目标是：
- 说一次不再说第二次
- 做完一个项目，其他项目也能按同样方式来
- 用了某个组件，后面类似需求都用这个组件

当前判断：

- `stage-agent orchestration`：`solved`
- `ui-spec-activation-gate`：`solved`
- `repeat-correction-memory`：`solved`

因此这 3 项已完成 `v0.7 P0` 的 first runtime landing，后续应从“是否存在”转入“覆盖范围和自动化深度”。

## 2026-05-08 Runtime Adoption Extension

The runtime landing has now been extended from behavior-only governance into the built-in delivery chain itself.

Default product delivery is now explicitly ordered as:

1. `调研文档`
2. `规划文档`
3. `需求文档`
4. `功能文档`
5. `前端页面（简单版）`
6. `前端页面（UI/UX版）`
7. `数据表`
8. `后端接口`
9. `前后端联调`

Implication:

- `stage-agent orchestration` is no longer only a recommendation over abstract stages
- it is now attached to a concrete built-in sequence that operators, runtime, tests, CLI, and backend API all share
- `前后端联调` carries the final workflow review gate instead of requiring a detached generic review-stage label
