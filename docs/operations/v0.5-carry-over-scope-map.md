# PMAIOS v0.5 Carry-Over Scope Map

- version: v0.1
- date: 2026-04-21
- status: active
- purpose: 把 `v0.4` 已确认的 carry-over 阻塞项，明确映射到 `v0.5` 的能力范围与后续处理线

## Scope Rule

以下内容不再继续拖住 `v0.4` 收口：

- 依赖新的外部账号、token、database、team 权限
- 依赖真实业务系统输入
- 继续推进会明显稀释当前版本收口效率

这类项应：

1. 在 `v0.4` 里标记为正式 carry-over
2. 在 `v0.5` 里明确挂到具体能力块
3. 不允许再作为“悬空未决项”存在

## Carry-Over Mapping

### 1. Notion database target

- source status: `carry-over`
- original blocker:
  - 当前 token 可用
  - 但提供的是 page，不是 database
  - 要继续必须拿到真实 database id
- mapped v0.5 strand:
  - `external connectors hardening`
  - `AI product-development collaboration loop`
- expected next action:
  - 明确数据库目标
  - 补齐真实同步目标
  - 再接入版本化产品/需求流转

### 2. Figma team/token

- source status: `carry-over`
- original blocker:
  - `FIGMA_TEAM_ID` 已配置
  - 但当前 token 远程检查返回 `403 Invalid token`
- mapped v0.5 strand:
  - `external connectors hardening`
  - `human reading surface`
  - `project standard entry contract`
- expected next action:
  - 修正 token / team 权限
  - 明确 Figma 项目空间映射
  - 再接项目级 board / design asset 流转

### 3. DingTalk 自动桌面提取

- source status: `carry-over`
- original blocker:
  - 目前只有本地 UI automation scaffold
  - 缺少稳定官方导出、API 或固定 source path
- mapped v0.5 strand:
  - `automation hardening`
  - `daily conversation digestion`
  - `AI-facing archive ingestion`
- expected next action:
  - 判断是否走官方导出 / API / 本地固定源路径
  - 若无法稳定落地，不再挂在主线自动化承诺上

## Version Boundary

`v0.4` 继续负责：

- 当前版本快照与分母式进度协议
- 子项目 workflow rollout
- 本地可直接推进的运行态能力

`v0.5` 接手负责：

- 外部连接器硬化
- 自动化摄入能力
- 更完整的人阅读 / AI 归档双层体系
- 产研协同闭环中的真实系统接入

这份映射创建后，以下 checklist 项应视为已完成：

- `把 carry-over 与 v0.5 范围显式对齐`
