# 需求晋升与防漏机制

- version: v0.3
- date: 2026-04-22
- status: active draft
- purpose: 防止用户在多轮收敛中明确说过的需求，最后既没有进入 plan，也没有被显式挂起或拒绝

## 问题定义

当前最容易发生的不是“完全没听到需求”，而是：

- 需求先在对话里出现
- 被写进候选清单、研究草案或临时总结
- 后续版本收敛时没有晋升到 canonical plan
- 再经过几轮重组或压缩，被自然丢失

这会导致：

- 用户感觉同一需求反复重提
- 计划越来越像模型自己整理后的版本，而不是对话真实共识
- 候选层和正式层之间形成黑洞

## 两层需求模型

### 1. 用户需求

用户需求是用户直接表达的目标、约束、不满、偏好和期望结果。

例子：

- 不要总漏需求
- 进度不要只说做了什么，要对长目标分母汇报
- 人看的和 AI 看的要分开，但要强关联
- 固定 `IP + 端口 + 路径` 访问

### 2. 产品需求

产品需求是系统为了满足用户需求而形成的产品能力、机制、流程、界面和产物契约。

例子：

- 需求晋升与防漏机制
- 长目标分母 / 已解决分子进度协议
- 人阅读层 / AI 归档层 / 强关系模型
- 统一 human-reading surface

### 3. 规则

任何重要项都不能只停留在其中一层。

至少要建立：

- `用户需求 -> 产品需求`
- `产品需求 -> 回查原始用户需求`

如果只写产品需求，不回查用户需求，就会出现“方案写得很多，但你最初的问题并没有真正解决”。

## 规则

### 1. 所有重要需求必须落到四选一

用户明确提出的重要需求，在一次工作周期结束前，必须进入以下之一：

1. `version plan` 正式范围
2. `backlog` 明确条目
3. `candidate` 列表并附原因
4. `parked / rejected` 记录并附原因

不允许长期停留在“对话里提过，但主计划里没有”。

### 2. 收敛不是删除

当需求被“重组”“收敛”“并项”时，必须保留映射：

- 原始用户需求说法
- 收敛后的产品能力块
- 是被保留、弱化、拆散，还是明确排除

如果只是换表达，也要能追溯。

### 3. 大收敛后必须做一次防漏审计

出现这些情况时，必须做一次 dropped-requirement audit：

- 版本清单大改
- 主题级重组
- 从散点讨论收敛成 capability list
- 用户明确表示“好像有遗漏”

审计结果至少分成：

- 已正式进入
- 被弱化但仍在
- 明确遗漏
- 明确排除

### 4. 研究草案不等于正式进入计划

仅出现在这些位置，不算已进入 canonical plan：

- research doc
- implementation draft
- temporary summary
- checklist candidate note

只有进入下列之一，才算真正进入主计划：

- `docs/operations/pmaios-version-plan.md`
- `docs/operations/v0.5-research-and-execution-backlog.md`
- 当前版本正式 snapshot

### 5. 缺失项要回填，不要只口头承认

发现遗漏后，必须至少做一项真实回填：

- 回填 version plan
- 回填 backlog
- 新增审计文档
- 新增 parked/rejected 记录

不能只说“我记住了”。

### 6. 收敛方案必须回查原始需求

当一个需求被收敛成方案、机制、模板或能力块后，必须回查：

- 原始用户需求是什么
- 当前方案解决了它的哪一部分
- 哪一部分仍未解决
- 是否只是“换了一个漂亮说法”，但问题本身还在

如果没有这一步，就不允许把“方案存在”当成“需求已解决”。

### 7. 产品需求完成后必须回到用户需求场景

产品需求完成后，不能停在：

- “文档写完了”
- “机制补上了”
- “能力已经进入计划了”

还必须继续做两步：

1. 回答用户：用用户听得懂的语言说明这次到底补了什么
2. 走查用户需求场景：检查用户当时真实遇到的问题，现在是否真的被解决

最后要明确标记：

- `solved`
- `partial`
- `unsolved`

如果只是产品层完成，但用户场景没有闭环，状态只能算 `partial`。

## 建议执行动作

每次版本收敛后，至少做这四步：

1. 从最近 1 到 3 天对话里抽出用户明确提出的能力点
2. 区分 `用户需求` 和 `产品需求`
3. 对照 version plan / backlog / checklist / research
4. 输出一份防漏审计结果，并回查原始用户需求是否真的被解决

## 当前已识别的高风险漏项类型

- 人阅读层与 AI 归档层的强关系
- 固定 `IP + 端口 + 路径` 访问面
- 项目实时叙事面与标准 board 契约
- 每日对话整理与方法论蒸馏
- 意图升降维与交付阶梯
- 知识库图谱接入
- 长目标清单式进度协议

## 与版本真源的关系

这份机制文档本身不是 plan。

它的作用是保证需求不会在从对话进入 plan 的过程中被自然蒸发，并且保证收敛后的方案要回查原始用户需求。
