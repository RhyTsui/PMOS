# v1.0 Runtime Capability Closure 2026-05-11

- date: 2026-05-11
- status: active
- scope: `Kernel Context` / `Gate Runtime` / `Task SSOT` / `L0-L3` / `Pipeline Launcher` / `Asset Backwrite Contract` / `External Sync Outbox` / `Autonomous Scheduler`

## 这轮做了什么

这轮不是再写 8 条能力债的愿景，而是把它们推进成统一 runtime 主线里可读取、可触发、可验收的一组机制切面。

已落地点：

1. `ContextInjectionBundle`
   - 新增 `ContextInjectionBundleSchema`
   - 新增 `ContextInjectionService`
   - 暴露 `GET /api/task-ssot/tasks/:taskId/context-bundle`
   - 默认输出：主线标签、阶段、active/blocked gates、truth refs、review evidence、sync targets、resume anchor、next safe step、redlines

2. `L0-L3` 协作等级不再只是枚举
   - `TaskSsotService` 现在会按 shared-context / workflow / reviewer / activated specialist / rework 真实状态推导协作等级
   - 当前规则：
     - `L0`: 纯共享上下文任务
     - `L1`: 轻量 workflow 主线
     - `L2`: review-required 或已进入 reviewer 协作
     - `L3`: 多 specialist 激活或进入 rework / 多方协同闭环

3. `Pipeline Launcher`
   - 新增 `PipelineLauncherPlanSchema`
   - 新增 `PipelineLauncherService`
   - 暴露 `GET /api/task-ssot/tasks/:taskId/pipeline-launcher`
   - 当前可显式给出：
     - `Requirements -> Functional`
     - `Functional -> Design`
     - `Design -> Delivery`
   - 每个 plan 都会返回 `ready / needs-input / blocked`、缺失输入、证据引用和下一动作

4. `Asset Backwrite Contract` 变硬
   - `TaskSsotService` 的 `asset-backwrite-gate` 不再因为“有任意 artifact”就放行
   - 现在规则是：
     - 任务运行中：有 `working-output` 可 `pass`
     - `ready_for_delivery / completed`：必须有 `final-delivery` backwrite，否则直接 `block`
   - `FinalStateValidationService` 同步改成同一口径

5. `External Sync Outbox` 从壳层走向 runtime
   - `TaskSsotSyncEnvelopeSchema` 新增：
     - `targetCategory`
     - `topicKey`
   - `OutboxService.enqueue` 默认推断目标类别
   - 新增 `buildRuntimeSummary`
   - 暴露 `GET /api/outbox/runtime`
   - 当前可直接看 pending/completed/failed/dropped 以及按 `targetSystem` / `targetCategory` 分桶

6. `Autonomous Scheduler` 增加统一恢复语义
   - `SchedulerRunSchema` 新增 `recoveryPolicy`
   - `SchedulerRunService` 现在会显式输出：
     - `manual-only`
     - `auto-retry`
     - `auto-rework`
     - `scheduled`
   - 新增 `buildRuntimeSummary`
   - 暴露 `GET /api/scheduler/runtime`

## 还没完成的部分

这轮没有假装把 8 条债全关掉，剩余缺口明确是：

1. `ContextInjectionBundle` 还没有默认喂给所有 workflow stage / specialist prompt / review packet
2. `Pipeline Launcher` 目前是 plan 生成器，不是 run 触发器
3. `L0-L3` 还没有正式 operator-facing 协议页和 UI 展示
4. `Outbox` 还没有 Figma / test-system / release-system 的标准 adapter
5. `Scheduler` 还没有 quota、session budget、跨日自治恢复策略

## 当前判断

- 用户需求层：`partial`
  - 原因：8 条债已不再停留在文字规划，已有一轮统一 runtime 收口，但还没有全部闭环

- 产品需求层：`partial-strong`
  - 原因：已经形成可读取、可验收、可继续扩展的统一机制切面，下一步可以直接做默认注入和一键触发
