# Tracking Acceptance 问题复盘与平台化提升

- version: v0.1
- date: 2026-05-06
- status: active
- source project: `subprojects/tracking-acceptance`
- purpose: 把埋点验收平台这轮落地过程中暴露出的共性问题、已验证解法和平台级升级动作，沉淀为 PMAIOS 可复用规则

## 1. 这次真正暴露的问题

这轮工作表面上是在补前端、补真源、补设计和补协作入口，本质上暴露的是一组反复出现的平台级问题：

1. 同一主题存在多份“看起来都像真源”的文档，导致实现口径漂移。
2. 设计链路缺少严格分层，概念稿、高保真稿、静态 HTML 和实现承接稿边界混用。
3. AI 输出容易停在聊天和侧栏，不稳定回写正式对象。
4. 子项目恢复与共享协作入口不完整，导致接手成本高、CLI/运行时易崩。
5. 证据层、版本继承层、标准底座层一开始没有被显式建模，结果前端先跑、产品后补。
6. 历史文档、跨项目残留和旧设计稿会继续污染默认检索与判断。
7. 确认环节没有被当成独立治理对象，导致“已产出”和“已确认”容易被混说。

这说明埋点验收平台遇到的不是孤立项目问题，而是 PMAIOS 在“子项目产品化承接”上的一组高频失败模式。

## 2. 问题模式、解决方案与平台升级

## 3. 对 PMAIOS 的平台判断

### 3.1 已经可以直接推广的规则

以下内容已经不是项目内尝试，而是经过这轮工作验证后可以直接推广到 PMAIOS 的稳定规则：

1. 重要子项目先建唯一产品主真源，再进入设计和实现主链。
2. 设计交付必须分层，概念图不再直接充当前端合同。
3. AI 结果必须回写正式对象链，聊天本身不算交付。
4. 日志/证据层必须独立存在，不能被概览或 AI 摘要替代。
5. 活跃子项目必须有最小运行合同：恢复链、共享状态、manifest、prep checklist。
6. 历史文档治理默认优先“降级标记 + 迁移指针”，而不是继续并行活跃。
7. 确认环节必须显式建模，不能把“稿件已生成”“文件已落盘”“AI 已给建议”混同为“已确认”。

### 3.2 还没有完全平台化的部分

以下能力在埋点验收项目里已经形成产品定义，但在 PMAIOS 平台层还没有变成强约束模板或通用对象：

1. 标准底座平台化
2. 历史导入与版本继承
3. 查询方案与成熟看板
4. AI 助手上下文包与回写任务对象
5. 组件库异常时的统一降级策略
6. 确认对象与确认状态的统一 schema / gate

这些部分当前更适合作为“可复用模式库”，下一步再视跨项目复用频率升级为平台模板、schema 或 workflow。

## 5. 下一安全步

如果继续往前推进，优先顺序应是：

1. 把“标准底座 / 版本继承 / 查询方案 / AI 回写”抽成可复用平台模板或 schema。
2. 把“确认对象 / 确认状态 / 确认责任人 / 放行条件”抽成统一 gate 或 schema。
3. 把活跃子项目最小运行合同做成统一脚手架或校验规则。
4. 把组件语义对齐、HTML 承接稿、build 验证补进通用前端交付检查链。
