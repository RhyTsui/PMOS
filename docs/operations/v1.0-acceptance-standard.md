# v1.0 验收标准

- version: v0.2
- date: 2026-05-09
- status: active

## 1. 验收原则

`v1.0` 按产品版本验收，不按“某个功能已经存在”验收。

每一项核心能力都必须同时满足：

1. 能被真实用户使用
2. 能被多角色协作使用
3. 能被 workflow 承接
4. 能被 review 与 Hermes 检查
5. 能被部署、复现、交接

## 2. 一级验收项

### A. 产品定位成立

必须满足：

1. 系统介绍清晰
2. 目标用户明确
3. 角色场景明确
4. 非目标范围明确
5. 外部读者能在首次接触时理解 PMAIOS 是什么、给谁用、解决什么问题

### B. 多角色使用成立

必须满足：

1. 产品同学可用
2. 测试同学可用
3. UI/UX 同学可用
4. 算法同学可用
5. 前端同学可用
6. 后端同学可用
7. 运营同学可用
8. 不同角色进入后都能拿到与自身工作相匹配的输入、输出和下一步动作

### C. Workflow 主链成立

必须满足：

1. `调研 -> 规划 -> 需求 -> 功能 -> 设计 -> 前端页面 -> 数据表 -> 后端接口 -> 联调与验收` 全链路可运行
2. `需求文档` 必须能拆到功能层
3. `功能文档` 必须能拆到接口层
4. `接口定义` 必须能继续落到开发 task 层
5. 每个阶段都能给出真实 artifact，而不是只有口头说明

### D. Review 与 Hermes 成立

必须满足：

1. 多专业 reviewer 被强制激活，而不是可选启用
2. review packet 完整，包含业务目标、系统现状、拆解结果、风险、证据
3. Hermes 能输出 `keep / revise / block / promote`
4. Hermes 能生成整改任务、回写目标和 closure evidence
5. 人的角色是最终裁决，而不是替代前置专业评审

### E. 前端页面交付质量成立

必须满足：

1. 页面面向真实用户和真实业务流程
2. 页面不是示意页、解释页、静态展示页
3. 页面具备真实动态交互
4. 页面通过布局、模块、流程、状态、交互评审
5. 页面通过浏览器级验证
6. 页面默认遵循既定组件系统和反漂移规则

### F. 文档治理成立

必须满足：

1. 每个稳定主题只有一个 active truth source
2. 补丁文档必须回并原真源
3. 文档生命周期状态明确：`draft / active / superseded / archived / deleted`
4. 多文档冲突能够被识别
5. 更新原文档时不会在收敛过程中丢失关键内容

### G. 知识层 grounding 成立

必须满足：

1. Dataki 默认 knowledge context 可用
2. 子项目可映射自己的知识库
3. `inbox` 输入能被自动提炼为背景知识
4. review 与 workflow 默认可消费系统现状 grounding
5. Notion / Dataki / inbox 的更新链至少有一条稳定可用主路径

### H. 部署与发布成立

必须满足：

1. 可安装
2. 可启动
3. 可配置
4. 可在多设备复现
5. 有操作手册
6. 有 changelog / 更新记录
7. 有 GitHub 共享路径
8. 外部协作方在不依赖当前会话上下文时仍能按文档完成部署和使用
9. 存在独立 `PMOS product repo`，且不混入业务子项目主体
10. GitHub 与 Notion 的分层明确，且至少有一条稳定的 `latest-state` 云端镜像主路径

## 3. 二级验收口径

### 3.1 用户层

回查结论只允许标记：

1. `solved`
2. `partial`
3. `unsolved`

### 3.2 产品层

每个关键模块都必须有：

1. owner
2. truth source
3. review gate
4. acceptance evidence

### 3.3 工程层

每个交付阶段都必须能给出：

1. artifact path
2. review result
3. Hermes result
4. proof-of-work summary

## 4. 不通过判定

以下任一情况成立，都不能算 `v1.0` 完成：

1. 只能在当前人、当前机器、当前会话里使用
2. 页面仍以示意页、静态页、卡片工作台壳子为主
3. review 仍然走形式
4. 文档仍然靠补丁维持
5. 多角色协作拿不到一致 truth source
6. 部署路径、操作路径、交接路径不成立
7. 知识层无法为 workflow 和 review 稳定提供背景信息
8. PMOS 产品行为规格仍隐含依赖 `codex` 本机私有目录

## 5. 当前验收判断

截至 `2026-05-09`：

1. `v1.0` 方向成立：`solved`
2. runtime baseline 已落地：`solved`
3. 产品级完成度：`partial`
4. 多角色真实使用：`partial`
5. 部署与发布可交接：`partial`

## 6. 当前最高优先级

当前最优先补齐的不是新能力扩张，而是以下四件事：

1. 统一入口与 control plane 成熟
2. 多专业 agent 强制生效
3. 文档生命周期治理落到运行机制
4. 前端页面生产与浏览器级验收成为默认能力
