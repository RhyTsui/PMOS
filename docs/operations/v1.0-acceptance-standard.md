# PMOS v1.0 Acceptance Standard

- version: `v1.0`
- date: `2026-05-09`
- status: `active`

## Acceptance Principle

`v1.0` 要按产品版本验收，而不是按“某个功能存在”验收。

独立产品仓里的每一条关键能力，至少都要满足：

1. 能被真实读者理解
2. 能被其它设备部署和启动
3. 能被多角色协作使用
4. 能被真源文档、评审和证据链追踪
5. 未闭环项必须明确进入后续补丁，而不是被隐藏

## Product-Repo Acceptance Slice

独立 `PMOS product repo` 至少应满足：

1. 可以被 `clone`、安装、构建、启动
2. 包含当前活跃 `v1.0` 的公开真源文档
3. 不混入业务子项目主体
4. 公开的是 sample/example 配置，而不是真实 secrets
5. 清楚解释 `GitHub / Notion / cloud-mirror` 的分层
6. 外部读者第一次进入时，能看懂 PMOS 是什么、给谁用、解决什么问题
7. README、产品介绍、首跑说明、版本进度、验收标准之间口径一致
8. 代码层不再隐式依赖母仓专属业务样板路径；保留 `subprojects/` 结构时，明确其属于产品多项目能力且默认不捆绑真实业务子项目主体

## Not Accepted

以下任意一项成立，都不能算 `v1.0` 产品仓完成：

1. 仍依赖母仓私有上下文才能理解或启动
2. 仍依赖隐藏的本地 Codex 规则才能正常使用
3. 仍带真实 secrets、Token、私有记忆或机器路径
4. 对外读者只能看到方向，看不到当前缺口与未完成项
5. 文档把未完成项伪装成已完成，而没有明确补丁策略
6. 产品仓代码仍依赖母仓专属业务样板路径，却没有在边界文档里说明
