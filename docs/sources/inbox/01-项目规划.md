# MCP QA Platform 项目规划

## 1. 文档目标

本目录文档的目标，是让新的接手人能够在不了解历史背景的情况下，快速理解这个系统当前做了什么、页面主流程是什么、核心对象如何落库，以及后续如果要继续设计，应该沿着什么边界推进。

建议阅读顺序：

1. `01-项目规划.md`
2. `02-架构规划.md`
3. `03-功能需求.md`
4. `04-工程级项目文档.md`
5. `05-数据库表说明.md`

## 2. 系统定位

这是一个面向 LLM/MCP 场景的质量保障平台，重点不在“聊天页面”本身，而在于把测试资产沉淀下来，并形成可重复执行、可查看 Trace、可输出报告、可继续优化的闭环。

当前主线围绕六个模块展开：

1. 用例
2. 评测集
3. 执行
4. Trace
5. 报告
6. 配置

可以把系统主路径理解为：

`维护用例 -> 组织评测集 -> 发起执行 -> 查看结果与 Trace -> 生成报告 -> 基于结果继续补充和修正用例`

## 3. 当前产品边界

### 3.1 页面主流程

当前前端 `web/index.html` 已体现出以下主流程能力：

1. 维护契约用例
2. 维护语义用例
3. 维护评测集
4. 发起执行并查看执行结果
5. 查看人工复核队列
6. 查看 Trace
7. 生成和查看报告
8. 修改运行时配置
9. 查看统计概览

### 3.2 核心对象

| 对象 | 作用 |
|------|------|
| 契约用例 `contract_cases` | 面向 MCP 工具接口的结构化样本 |
| 语义用例 `semantic_cases` | 面向问答效果和工具选择的端到端样本 |
| 评测集 `test_suites` | 对一组用例做组织和复用 |
| 执行任务 `eval_runs` | 一次完整运行实例 |
| 单条结果 `qa_run_case_results` | 一次 run 中每条样本的详细执行结果 |
| Trace `traces` 等 | 执行链路观察与定位 |
| 报告 `reports` | 对执行结果进行汇总和交付 |
| 运行配置 | 模型、MCP、并发、内部对话等参数 |

## 4. 为什么这样设计

从代码结构可以看出，这个系统的设计出发点是：

1. 测试样本要长期沉淀，不能只存在脚本里。
2. 同一个平台要同时支持“接口契约测试”和“语义端到端测试”。
3. 执行过程必须可回看、可解释、可定位失败阶段。
4. 报告不是附属产物，而是复盘和交付的重要出口。
5. 模型和联调环境要能通过配置切换，而不是每次改代码。

## 5. 推荐的后续演进方向

在进入分阶段演进之前，需要先明确一个当前最重要的缺口：

**自动化比例还不够高，人力仍然大量消耗在用例编写、期望补录、结果校验和人工复核上。**

这个缺口主要体现在两条线上：

1. 用例生成自动化不足
2. 校验自动化不足

因此，后续规划的第一优先级，不是继续堆页面，而是持续提升自动化比例，减少人工投入，让人工主要承担审核和例外处理工作。

如果基于理解继续延续这个项目，建议按下面三步走。

### 5.1 第一阶段：稳定主干

目标是把当前主流程做得更稳、更清晰：

1. 统一页面概念和数据库对象命名。
2. 明确用例、评测集、执行、Trace、报告之间的边界。
3. 补全文档和注释，降低接手门槛。
4. 把页面里分散的逻辑逐步沉淀到服务层。

### 5.2 第二阶段：增强执行平台

目标是让执行系统更可扩展：

1. 让执行策略、校验级别、Trace 级别形成统一模型。
2. 让执行任务具备更稳定的调度方式。
3. 让报告生成支持更明确的模板和指标体系。
4. 让配置支持多环境、多 MCP Profile、多模型组合。
5. 把自动用例生成、自动期望补全、自动校验和人工复核策略串成完整闭环。

### 5.3 第三阶段：升级为平台化产品

目标是支持多人协作和长期演进：

1. 将前端从单文件页面升级为正式前端工程。
2. 将 SQLite 升级为更适合团队协作的数据库。
3. 引入更标准的迁移、审计和版本治理机制。

## 6. 面向接手人的三条结论

1. 这个系统的核心不是模型调用本身，而是“测试资产沉淀 + 执行复盘闭环”。
2. 当前最重要的六个模块是：用例、评测集、执行、Trace、报告、配置。
3. 如果要重设计，应该先保住这六个模块的领域边界，并把“提升自动化比例、减少人力投入”作为优先目标，再考虑技术栈升级。
