## 一、项目概述

### 1.1 项目目标

实现巨量广告平台游戏应用联调的全自动化流程，从参数配置到二维码生成、通知发送的端到端自动化，大幅提升联调效率。

### 1.2 核心功能

| 功能模块 | 说明 |
|---------|------|
| Web参数配置 | 可视化表单，支持所有联调参数配置 |
| 浏览器自动化 | 自动登录、导航、操作巨量平台 |
| 资产管理 | 自动检查/创建应用资产和事件资产 |
| 联调执行 | 自动发起联调并生成二维码 |
| 通知推送 | 自动发送飞书消息通知 |

### 1.3 技术栈

```
┌─────────────────────────────────────────────────────────────┐
│                      技术架构                                │
├─────────────────────────────────────────────────────────────┤
│  前端界面    │  Gradio 4.x (Python Web框架)                  │
│  浏览器自动化 │  Playwright (异步API)                        │
│  消息通知    │  飞书开放平台API / Webhook                    │
│  配置管理    │  JSON配置文件                                 │
│  日志系统    │  Python logging + colorlog                    │
```

## 六、元素定位策略

### 6.1 定位优先级

| 优先级 | 策略 | 示例 | 稳定性 |
|-------|------|------|-------|
| 1 | data-testid | `[data-testid='submit']` | ⭐⭐⭐⭐⭐ |
| 2 | XPath (contains) | `//button[contains(text(), '确认')]` | ⭐⭐⭐⭐ |
| 3 | CSS Selector | `.btn-primary` | ⭐⭐⭐ |
| 4 | 文本定位 | `text='提交'` | ⭐⭐⭐ |
| 5 | Playwright定位器 | `page.get_by_role('button')` | ⭐⭐⭐⭐ |

### 6.2 定位器设计

每个元素定义主选择器和备用选择器：

```python
@dataclass
class Locator:
    primary: str           # 主定位器
    backups: List[str]     # 备用定位器
    description: str       # 描述
```

### 6.3 动态等待策略

```python
# 等待元素可见
await page.wait_for_selector(selector, state='visible', timeout=30000)

# 等待网络空闲
await page.wait_for_load_state('networkidle')

# 等待特定文本
```

# 广告联调自动化 - 技术方案设计

## 1. 项目背景

### 1.1 业务需求
广告投放联调测试需要人工扫码、授权、刷视频找广告，流程繁琐耗时。需要自动化替代人工操作。

### 1.2 核心流程
```
生成二维码 → 扫码授权 → 刷新首页 → 刷视频找广告 → 点击广告 → 验证结果
```

### 1.3 边界划分
- **模块A（另一位测试负责）**：生成二维码
- **模块B（本项目负责）**：扫码授权 + 刷广告点击

---

## 2. 整体架构

### 2.1 模块划分

```
┌─────────────────────────────────────────────────────────┐
│                    测试执行流程                           │
├──────────────────┬──────────────────────────────────────┤
│     模块A        │              模块B                    │
│   (二维码生成)    │         (自动化执行)                  │
├──────────────────┼──────────────────────────────────────┤
│ 1. 生成二维码     │ 1. 接收二维码                         │
│ 2. 保存二维码     │ 2. 打开抖音扫一扫                     │
│ 3. 调用模块B      │ 3. 从相册选择二维码                   │
│                  │ 4. 点击授权测试                       │
│                  │ 5. 返回首页刷新                       │
│                  │ 6. 刷视频找广告                       │
│                  │ 7. 点击广告                           │
│                  │ 8. 返回执行结果                       │
└──────────────────┴──────────────────────────────────────┘
```

### 2.2 技术选型

| 模块 | 技术栈 | 说明 |
|------|--------|------|
| 模块A | 待定 | 另一位测试负责 |
| 模块B | Python + Appium | 本项目 |
| 手机控制 | UiAutomator2 (Android) / XCUITest (iOS) | Appium驱动 |
| 封装形式 | CLI工具 | 本项目 |

---

## 3. 模块间协作方案

### 3.1 方案对比

| 方案 | 实现方式 | 优点 | 缺点 | 推荐度 |
|------|----------|------|------|--------|
| **A. CLI调用** | 打包成exe，命令行传参 | 简单，无依赖 | 需要物理传递文件 | ⭐⭐⭐ |
| **B. 文件共享** | 约定固定路径，轮询等待 | 无需网络 | 需要共享存储 | ⭐⭐⭐⭐ |
| **C. HTTP接口** | 模块B暴露Web服务 | 灵活，可扩展 | 需要搭服务 | ⭐⭐⭐⭐⭐ |

### 3.2 推荐方案：CLI调用

## 5. 执行流程详细设计

### 5.1 状态机

```
初始化 → 等待二维码 → 扫码 → 授权 → 刷新 → 刷广告 → 点击 → 完成
   ↓                    ↓       ↓               ↓
 失败                  失败    失败           失败
```

### 5.2 异常处理

| 异常场景 | 处理方式 |
|----------|----------|
| 二维码文件不存在 | 等待超时后返回错误 |
| 扫码失败 | 重试3次 |
| 授权按钮未找到 | 截图保存，返回错误 |
| 刷N条未找到广告 | 返回错误，提供刷了多少条 |
| 网络异常 | 重试机制 |
