## 5. 给 Chat 接入方的排查建议

### 5.1 确认测试连通和真实问答是否同一链路

请确认以下两条链路是否使用同一服务、同一 SDK client、同一配置来源：

```text
项目配置页测试连通 -> SDK 上报逻辑
真实 Chat 问答请求 -> SDK 上报逻辑
```

如果二者不是同一段代码，则“测试连通成功”不能证明真实 Chat 问答已完成埋点。

### 5.2 在真实问答入口打印运行时配置

真实问答请求进入后，请打印以下运行时配置：

```text

sampleRate
apiUrl
workspaceId
projectId
appId
env
serviceName
token 后 6 位
```

当前期望值：

```text

workspaceId = 7640023864084267009

```

如果真实问答服务仍使用旧地址、旧 workspace 或旧 Token，则不会进入当前新环境链路。

### 5.3 在真实问答链路打印 SDK 执行过程

请在真实问答链路中增加临时日志：

```text
是否进入 SDK 埋点逻辑
是否执行 start_span
生成的 trace_id
生成的 root_span_id
是否执行 set_input
是否执行 set_output
是否执行 span.finish
是否执行 client.flush
flush event 是否 success
```

Python SDK 可临时增加：

```python
def on_finish_event(info):
    print(info.event_type, info.is_event_fail, info.detail_msg)

client = cozeloop.new_client(
    api_base_url="http://liannu-test.dc.yokagames.com:1118",
    workspace_id="7640023864084267009",
    api_token="当前新环境 token",
    trace_finish_event_processor=on_finish_event,
)
```

成功时应看到：

```text
exporter.span_flush.rate False
```

失败时重点关注：

```text
602000101 no access permission
Network error occurred
export spans fail
retry second time failed
```

### 5.4 流式 Chat 场景重点确认 finish/flush 时机

如果真实 Chat 是流式返回，推荐埋点流程：

```text
用户提问开始
-> start root span
-> 记录 input
-> Agent / LLM / MCP / 工具调用分别记录子 span
-> 流式响应全部完成
-> 记录 output
-> root span.finish()
-> client.flush()
```

常见问题：

```text
流式响应开始后提前 return，导致 finish/flush 未执行；
异常分支未 finish；
只在测试连通接口里 flush，真实问答接口没有 flush；
服务进程结束或请求上下文释放前，SDK 异步队列未发送；
sampleRate 实际运行值不是 1。
```

### 5.5 检查真实服务是否加载了新配置

如果配置来自配置中心，请确认真实 Chat 服务是否已重新加载配置或重启。

建议在真实 Chat 服务容器中检查：

```bash
printenv | grep -Ei "COZELOOP|LIANLU|SDK|TOKEN|WORKSPACE|PROJECT"
```

如配置不是环境变量，请打印配置中心实际读取值。

### 5.6 检查代理和网络

真实 Chat 服务容器中检查：

```bash
printenv | grep -Ei "HTTP_PROXY|HTTPS_PROXY|ALL_PROXY|NO_PROXY"
```

如果存在代理，请确认：

```text
liannu-test.dc.yokagames.com
```

不会被错误代理拦截。

## 6. 连弩侧联调观察方式

联调时，建议接入方发起一个带唯一标记的真实问题：

```text
trace_probe_newenv_real_chat_20260515_001 昨日消耗是多少
```

连弩侧同步监听：

```bash
docker logs -f --since 10m coze-loop-nginx 2>&1 | grep -Ei "traces/ingest|opentelemetry|connectivity-test|10100283|xiaoqiao|probe|POST"
```

```bash
docker logs -f --since 10m coze-loop-app 2>&1 | grep -Ei "IngestTraces|traces/ingest|ingest|trace|span|600900101|602000101|no access|permission|10100283|xiaoqiao|probe"
```

判断方式：

```text
测试连通有 IngestTraces，真实问答无 IngestTraces：
真实问答链路未触发 SDK 上报。

真实问答有 IngestTraces 且 ClickHouse insert success：
平台侧已接收并落库，若页面未显示需检查页面筛选条件。

真实问答出现 602000101：
真实问答仍使用旧 Token、旧 workspace，或 Token 权限不匹配。

真实问答出现 Network error：
检查真实 Chat 服务网络、代理、DNS。

真实问答有 start_span 日志但无 flush success：
检查 finish/flush 调用时机，尤其是流式返回场景。
```
