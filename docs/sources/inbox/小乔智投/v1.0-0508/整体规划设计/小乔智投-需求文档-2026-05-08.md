## 0. 本版需求文档解决什么问题

本版不再只把需求写成“页面模块”或“抽象场景”，而是把第一阶段需求压到更接近真实交付对象的层级：

1. 明确每条需求对应哪条真实业务流。
2. 明确这条业务流最少要收什么字段。
3. 明确系统要交付什么结果物。
4. 明确需求继续拆成哪些功能。

目标是让后续功能文档能直接继续压到接口层级。

---

## 1. 需求拆解原则

### 1.1 拆解标准

每条需求至少回答：

1. 谁触发。
2. 在什么业务上下文触发。
3. 系统最低要拿到哪些输入。
4. 系统必须输出什么正式结果物。
5. 后续要拆成哪些功能。

### 1.2 本轮比上一版更落地的地方

1. 不再只说“帮助 / 需求 / 排查 / 联调”四类场景。
2. 增加了每条业务流的最小字段包。
3. 增加了正式结果物定义。
4. 增加了依赖系统能力域与验收方式。
5. 明确当前阶段以正式 mock contract 先承接真实业务对象，不等待本地直连预发库。

## 2. 第一阶段需求总览

第一阶段正式需求共 9 条：

1. R-01 统一对话入口
2. R-02 工作流路由与追问
3. R-03 使用帮助结果化
4. R-04 需求沟通结构化
5. R-05 问题排查证据化
6. R-06 广告联调任务化
7. R-07 会话与任务沉淀
8. R-08 单对话工作台视图
9. R-09 管理页面与提示词管理

其中：

1. `R-01 ~ R-02` 是入口需求。
2. `R-03 ~ R-06` 是四条业务流需求。
3. `R-07 ~ R-08` 是系统化承接需求。
4. `R-09` 是管理与治理需求。

补充实现约束：
1. 当前阶段后端数据源以本地 mock 为主。
2. mock 的职责是承接真实需求、功能、页面和接口，不是演示占位。
3. 后续用户补充 mock schema / mock content 后，应直接替换到现有 contract，不重做需求层定义。

## 3. 正式需求清单

## 3.1 R-01 统一对话入口

- 需求编号：`R-01`
- 类型：产品需求
- 优先级：P0
- 主链路：是

### 触发角色

1. 广告技术支持
2. 投放 / 运营同学
3. 数据分析 / 排查同学
4. 联调协作角色

### 业务背景

现行广告业务入口分散，用户通常先判断去哪个系统、找谁、提什么形式的问题，导致问题在进入系统前已经损失了效率。

### 最小输入

1. 用户自然语言问题
2. 可选附件
3. 可选上下文补充

### 系统必须输出

1. 可直接承接首条输入的对话入口
2. 首轮系统反馈
3. 当需要进入业务流时，进入后续任务链路

### 对应功能

1. F-01 首屏输入区
2. F-02 推荐起手问题
3. F-03 首条消息创建与发送
4. F-04 首轮系统反馈

### 验收标准

1. 用户进入页面无需先选模块即可发起问题。
2. 首条问题发出后，系统必须给出有效反馈。
3. 不要求用户理解内部路由和系统边界。

---

## 3.2 R-02 工作流路由与追问

- 需求编号：`R-02`
- 类型：产品需求
- 优先级：P0
- 主链路：是

### 业务背景

统一入口成立的前提，不是“任何话都只当聊天处理”，而是系统要判断是不是业务问题、属于哪条业务流、还缺哪些最小上下文。

### 最小输入

1. 用户问题文本
2. 当前会话上下文
3. 可选附件

### 系统必须输出

1. 是否属于业务工作
2. 属于哪条业务流
3. 是否需要追问
4. 待补字段方向

### 对应功能

1. F-05 路由判断
2. F-06 业务相关性识别
3. F-07 意图类型识别
4. F-08 追问判断

### 验收标准

1. 非业务闲聊不进入任务化流程。
2. 四条业务流都能被正确承接。
3. 信息不足时优先追问，而不是假装已理解。

---

## 3.3 R-03 使用帮助结果化

- 需求编号：`R-03`
- 类型：业务需求
- 优先级：P0
- 主链路：是

### 业务背景

广告业务里大量问题并不是“查异常”，而是“解释口径、告诉我去哪看、引用哪个规则”。这类问题若没有正式结果物，回答一次就会再次重复。

### 典型问题

1. 指标是什么意思。
2. 某个结果在哪个系统看。
3. 某条规则怎么理解。
4. 常见广告技术问题如何处理。

### 最小字段包

1. 问题类型
2. 涉及对象
3. 是否需要系统路径
4. 是否需要引用来源

### 系统必须输出

1. 指标 / 概念解释
2. 对应系统路径
3. 引用来源或知识依据
4. 可继续追问方向

### 对应功能

1. F-09 帮助任务创建
2. F-10 帮助结果生成
3. F-11 帮助结果面板展示
4. F-12 帮助追问承接

### 验收标准

1. 帮助类问题不只返回一段话，还要有结构化帮助结果。
2. 结果可继续追问，不丢上下文。
3. 若无法直接回答，系统至少能说明缺什么信息。

---

## 3.4 R-04 需求沟通结构化

- 需求编号：`R-04`
- 类型：业务需求
- 优先级：P0
- 主链路：是

### 业务背景

现行广告接入和配置类需求的主要痛点不是没人处理，而是需求长期口头化、字段缺失、依赖项遗漏，最终拖慢开发与联调。

### 第一阶段覆盖的需求方向

1. 媒体回传接入
2. 事件映射 / 埋点相关需求
3. 归因 / 配置 / 白名单类需求
4. 其他需要跨角色推动的信息补齐型需求

### 最小字段包

1. 需求类型
2. 媒体
3. 应用 / 包体
4. 目标事件或配置对象
5. 目标时间
6. 验收方式
7. 依赖系统或依赖方

### 系统必须输出

1. 结构化需求单
2. 缺失字段清单
3. 依赖项清单
4. 下一步协作动作

### 对应功能

1. F-13 需求任务创建
2. F-14 缺失字段识别
3. F-15 结构化需求单生成
4. F-16 需求结果展示

### 验收标准

1. 模糊需求能被追问成结构化对象。
2. 系统明确提示还缺哪些字段。
3. 最终结果可供后续数据表 / 接口 / 实现继续承接。

---

## 3.5 R-05 问题排查证据化

- 需求编号：`R-05`
- 类型：业务需求
- 优先级：P0
- 主链路：是

### 业务背景

广告异常排查的难点不只是找答案，而是要在多个系统之间组织证据链。第一阶段必须把排查结果沉淀成“结论 + 证据 + 置信度 + 下一步”。

### 第一阶段重点异常方向

1. 激活异常
2. 付费异常
3. 回传异常
4. 归因异常
5. BI / 数仓结果不一致

### 最小字段包

1. 异常类型
2. 媒体
3. 应用
4. 计划 / 广告对象
5. 时间窗
6. 指标或异常现象
7. 可选截图 / 日志

### 系统必须输出

1. 一句话结论
2. 证据链
3. 置信度
4. 下一步建议

### 对应功能

1. F-17 排查任务创建
2. F-18 排查上下文记录
3. F-19 排查结果生成
4. F-20 排查结果面板展示

### 验收标准

1. 异常问题能生成正式排查任务。
2. 结果区可读到证据链，而不只是聊天回复。
3. 信息不足时系统能明确指出缺口。

## 3.6 R-06 广告联调任务化

- 需求编号：`R-06`
- 类型：业务需求
- 优先级：P0
- 主链路：是

### 业务背景

广告联调问题天然跨账号、设备、环境、配置、日志与执行流程，若没有任务化承接，只会继续依赖人工协调和局部截图。

### 最小字段包

1. 联调类型
2. 账号
3. 媒体
4. 应用 / 包体
5. 测试设备
6. 入口方式
7. 环境
8. 当前卡点

### 系统必须输出

1. 联调准备项
2. 联调执行状态
3. 执行日志或日志摘要
4. 联调结果报告

### 对应功能

1. F-21 联调任务创建
2. F-22 联调准备项与状态展示
3. F-23 联调日志展示
4. F-24 联调结果展示

### 验收标准

1. 联调问题可形成正式联调任务。
2. 页面能看到联调当前处于哪个阶段。
3. 联调结束后能回看结果和日志摘要。

## 3.7 R-07 会话与任务沉淀

- 需求编号：`R-07`
- 类型：技术需求
- 优先级：P0
- 主链路：是

### 业务背景

如果系统只会回答问题，却没有会话、路由、任务、结果对象，就无法支撑企业系统级协同与回看。

### 系统必须沉淀的对象

1. Conversation
2. Message
3. RoutingDecision
4. Task
5. TaskContext
6. WorkflowResult

### 对应功能

1. F-25 会话创建与读取
2. F-26 消息记录
3. F-27 路由记录
4. F-28 任务记录
5. F-29 结果记录
6. F-30 历史任务回看

### 验收标准

1. 一条业务消息至少形成消息、路由与任务中的必要对象。
2. 历史任务可回看。
3. 后续追问可延续既有上下文。

## 3.9 R-09 管理页面与提示词管理

- 需求编号：`R-09`
- 类型：管理需求
- 优先级：P1
- 主链路：否，但属于正式范围

### 业务背景

小乔是 AI 驱动系统，如果没有管理页面，尤其没有提示词管理，后续路由、回复风格、业务模板和 agent 行为就只能散落在代码和临时文档里，不适合持续运营。

### 系统形态要求

1. 管理页面与主工作台属于同一套系统。
2. 通过页面跳转进入系统管理。
3. 不做第二套登录系统。

### 当前必须覆盖

1. 提示词管理
2. 路由规则管理入口
3. MCP / skill / tool 绑定查看位
4. 结果模板管理入口
5. 通知与开关配置入口

### 明确不自建

1. 知识库管理后台
2. 评测平台管理后台

### 提示词管理最小字段包

1. prompt 名称
2. prompt 分类
3. 适用业务流
4. 适用 agent / 模型
5. 版本号
6. 生效状态
7. 变量说明
8. 更新时间

### 系统必须输出

1. Prompt 列表与详情
2. Prompt 新建 / 编辑 / 版本记录
3. Prompt 启停与生效状态
4. Prompt 适用范围说明
5. 操作记录

### 对应功能

1. F-36 管理页面导航
2. F-37 提示词列表与详情
3. F-38 提示词编辑与版本管理
4. F-39 Prompt 适用范围与开关配置
5. F-40 外部平台接入位展示

### 验收标准

1. 管理页面可在同一登录态下独立进入。
2. 提示词不再只能通过改代码维护。
3. 知识库和评测平台只体现接入关系，不在小乔内重做后台。

## 3.10 当前阶段 mock contract 约束

虽然第一阶段暂不直连智投预发布数据库，但这不改变需求层的正式交付标准。

### 背景

1. 运维已确认当前预发布数据库禁止本地开发环境直接连接。
2. 因此本阶段页面、接口、联调都不能等待真实库权限打通后再推进。

### 需求约束

1. 四条业务流和管理页必须继续按正式交付物推进。
2. 前后端必须使用稳定 mock contract 对齐。
3. mock 结构必须贴近后续真实业务对象。
4. mock 内容必须能覆盖四条业务流与提示词管理的验收场景。

### 对后续功能拆解的要求

1. 每个功能都要明确当前 mock 输入输出。
2. 每个功能都要预留后续真实数据映射位。
3. 后续切换测试 / 生产数据时，不允许重写需求定义，只允许替换数据来源和 payload 细节。
