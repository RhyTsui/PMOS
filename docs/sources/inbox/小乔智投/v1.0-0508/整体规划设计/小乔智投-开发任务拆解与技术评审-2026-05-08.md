## 1. 技术评审结论

### 1.1 可以直接做的

1. 前端页面（简单版）结构重做
2. 顶部身份与状态区接 `GET /workspace`
3. 历史任务区接 `GET /tasks`
4. 对话发送主链路接 `POST /conversations` + `POST /conversations/{conversation_id}/messages`

### 1.2 可以做，但要接受后续返工风险的

1. 结果区直接消费当前 `WorkflowResult.structured_payload_json`
2. 用当前 `TaskDetailResponse.latest_result` 承接四类业务流结果

风险原因：

1. 当前 payload 结构太泛，不足以支撑帮助 / 需求 / 排查 / 联调四类结果卡片的稳定渲染
2. 如果前端先按现在的 payload 硬写，后面一旦 schema 细化就会返工

### 1.3 不能只靠前端解决的

1. 四类业务流结果 schema
2. 缺失字段包 schema
3. 排查证据链 schema
4. 联调状态机与执行日志 schema

这些都属于后端 contract 问题，不是页面样式问题。

## 4. 当前主判断

### 产品视角

当前路线仍然是更优解：

1. 先把单对话工作台做活
2. 再细化四类业务流结果 contract
3. 再把结果区做实
4. 当前不等待预发布数据库直连，而是先把 mock contract 做成可替换真实数据的稳定层

这比“先画完整 UI”或者“先补四大专项页”更优。

### 开发视角

当前最值得做的不是一次性把整页做满，而是拆成两层：

1. 低返工层：
   - workspace
   - conversation
   - tasks
2. 高返工层：
   - result schemas
   - result rendering

所以正确策略是：

1. 先做低返工层
2. 再补后端 contract
3. 再做高返工层
