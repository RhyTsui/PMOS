## 0. 文档目标

这份文档聚焦小乔中的“使用帮助”专项，回答：

1. 它到底解决哪类问题
2. 它需要沉淀哪些正式结果物
3. 它要拆成哪些功能、对象、接口与页面承接
4. 它和知识库、规则库、系统路径之间是什么关系

---

## 1. 专项范围

### 1.1 当前解决的问题

使用帮助不是泛聊天，而是聚焦广告业务中的高频说明类问题：

1. 指标是什么意思
2. 某个结果在哪个系统看
3. 某个规则、口径、归因逻辑怎么理解
4. 某类常见技术问题通常如何处理

### 1.2 当前不做的事情

1. 不自建知识库平台
2. 不做开放问答社区
3. 不承诺所有帮助问题都可自动给出最终答案

---

## 2. 专项需求

### 2.1 目标用户

1. 广告技术支持
2. 投放运营
3. 数据分析
4. 联调协作角色

### 2.2 核心需求

#### R-HELP-01 帮助问题统一承接

输入：

1. 自然语言问题
2. 可选指标名 / 系统名 / 规则名
3. 可选截图或上下文

输出：

1. 问题已识别为帮助类任务
2. 如果信息不够，先追问补齐

#### R-HELP-02 正式帮助结果物

使用帮助不能只返回一段聊天文本，必须沉淀为正式结果物：

1. 指标解释
2. 系统路径
3. 引用来源
4. 适用边界
5. 下一步建议

#### R-HELP-03 不确定性表达

当系统没有足够依据时，必须明确：

1. 当前回答基于什么来源
2. 哪些地方不确定
3. 需要补什么上下文

---

## 3. 专项功能

### 3.1 功能分组

#### F-HELP-01 帮助任务创建

1. 识别帮助类问题
2. 记录问题上下文
3. 创建帮助任务

#### F-HELP-02 问题追问与字段补齐

1. 识别缺失信息
2. 发起追问
3. 更新任务上下文

#### F-HELP-03 帮助结果生成

1. 汇总定义说明
2. 汇总系统路径
3. 汇总引用来源
4. 输出下一步建议

#### F-HELP-04 帮助结果展示

1. 结果标题
2. 解释说明区
3. 来源区
4. 路径区
5. 下一步动作区

### 3.2 后端承接对象

1. `HelpTask`
2. `HelpContext`
3. `HelpResult`
4. `HelpSourceRef`

### 3.3 结果结构建议

`HelpResult` 建议字段：

1. `question_type`
2. `subject`
3. `definition_text`
4. `system_path`
5. `source_refs`
6. `confidence_level`
7. `next_actions`

---

## 4. 页面与设计承接

### 4.1 主工作台结果区

应展示：

1. 当前问题主题
2. 指标 / 规则解释
3. 系统入口路径
4. 引用来源
5. 继续追问入口

### 4.2 历史任务区

帮助类任务摘要至少显示：

1. 主题
2. 类型
3. 最近更新时间
4. 是否有来源引用

### 4.3 管理页承接

管理页不做知识库管理，但需要有：

1. 帮助模板入口
2. 来源规则入口
3. 结果模板入口

---

## 5. 实施顺序

1. 先定义帮助结果 mock schema
2. 先把帮助结果区做成正式结构
3. 再补来源引用与不确定性表达
4. 最后再细化模板与规则管理
