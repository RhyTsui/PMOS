## 0. 文档目的

本文档不是单纯的 UI 布局说明，而是小乔智投第一阶段的正式设计文档。

它同时覆盖：

1. 信息架构
2. 页面结构
3. 关键交互承接
4. 视觉规范
5. 组件规范
6. 状态规范
7. 响应式规范

本文档直接服务于：

1. 前端页面（简单版）
2. 前端页面（UI/UX 版）
3. 后续组件拆分与视觉验收

## 1. 设计定位

### 1.1 产品界面类型

小乔智投属于：

`数据 / 广告 / 企业系统类数字员工工作台`

这意味着界面设计不能走以下方向：

1. 消费级娱乐化聊天产品
2. 过度营销化官网视觉
3. 花哨但低信息密度的概念稿
4. 只强调视觉包装、不强调状态和结果结构的展示页

### 1.2 设计目标

界面设计必须同时满足四件事：

1. 看起来是企业系统，而不是 Demo 页面
2. 以任务推进和结果沉淀为主，不以装饰性卡片为主
3. 保持足够的数据承载能力
4. 对广告技术、数据分析、联调排查等专业场景友好

### 1.3 设计气质

建议气质：

1. 冷静
2. 清晰
3. 可信
4. 可操作
5. 不花哨

关键词：

`企业级` `数据工作台` `任务驱动` `结构化结果` `专业但不压迫`

### 1.4 默认组件基线

针对会话 / Chat 类企业系统，默认组件基线约定为：

1. 企业系统基础组件：`Ant Design`
2. 会话式交互组件：`Ant Design X`

分工原则：

1. `Ant Design`
   - 承接 Layout、Menu、Tabs、Drawer、Table、Form、Modal、Tag、Badge、Button、Input 等企业系统基础组件
2. `Ant Design X`
   - 承接 Bubble、Sender、Welcome、Prompts、Conversations、Suggestion、Attachments、Actions、Notification 等会话式组件

说明：

1. 会话类产品不再默认自造聊天基础组件
2. 优先在 `Ant Design + Ant Design X` 语义范围内做组合和扩展
3. `Notification` 仅用于弱打扰应用级通知，不替代站内主反馈

## 2. 信息架构

第一阶段保留五个一级入口：

1. 主工作台
2. 监控大屏
3. 素材分析
4. 广告预测
5. 应用支撑

其中：

1. 主工作台是默认首页
2. 其余四项为辅助专项页
3. 所有业务主链路先由主工作台承接

## 3. 页面结构

## 3.1 主工作台结构

主工作台采用三段式结构：

1. 顶部身份与状态区
2. 中部双栏主内容区
3. 历史任务区

桌面端：

1. 左侧是主对话工作区
2. 右侧是结构化结果区
3. 历史任务区可作为下方列表或右侧抽屉

移动端：

1. 对话区优先全宽
2. 结果区变为下方折叠区
3. 历史任务改为全屏抽屉

## 3.2 顶部身份与状态区

必须包含：

1. 小乔名称
2. 身份说明
   - `广告技术服务数字员工`
3. 系统状态
4. 当前会话状态
5. 新对话入口
6. 历史任务入口

## 3.3 主对话工作区

必须包含：

1. 空态欢迎区
2. 推荐起手问题
3. 消息流
4. 输入区
5. 附件上传入口

## 3.4 结构化结果区

必须包含：

1. 当前结果标题
2. 当前任务标签
3. 指标卡片区
4. 结构化内容区
5. 下一步动作区

## 3.5 历史任务区

每条任务至少展示：

1. 标题
2. 类型
3. 状态
4. 更新时间
5. 一句话摘要

## 4. 设计系统基础规范

### 3.5 当前阶段数据承接约束

本设计文档约束的不是纯静态页面，而是正式交付期页面，因此补充两条实现约束：

1. 当前阶段由于预发布数据库禁止本地连接，前端先消费本地仓库后端返回的 mock contract。
2. 这里的 mock contract 必须被视为正式页面数据结构，不是临时截图级演示数据。

这意味着：

1. 简单版页面要先把真实组件区、真实结果区、真实任务区接到稳定接口结构上。
2. UI/UX 版页面只能在同一套 contract 上提升视觉与交互，不能另起一套静态数据结构。
3. 结果区、历史任务区、管理页列表区都要按可替换真实数据的方式设计。

## 4.1 字体规范

### 4.1.1 字体家族

针对数据 / 广告 / 企业系统，推荐使用：

1. 中文优先字体：
   - `PingFang SC`
   - `Microsoft YaHei`
   - `Noto Sans SC`
2. 英文与数字辅助字体：
   - `Inter`
   - `Segoe UI`
   - `SF Pro Display`

建议最终前端字体栈：

```css
font-family: "Inter", "PingFang SC", "Microsoft YaHei", "Noto Sans SC", "Segoe UI", sans-serif;
```

原则：

1. 企业系统优先无衬线
2. 中英文统一气质
3. 数字与英文要清晰，适合数据密集界面

### 4.1.2 字号层级

建议字号体系：

| 层级 | 字号 | 行高 | 用途 |
| --- | --- | --- | --- |
| Display | 32px | 40px | 页面主标题 |
| H1 | 24px | 32px | 模块主标题 |
| H2 | 20px | 28px | 区块标题 |
| H3 | 16px | 24px | 卡片标题 / 表单组标题 |
| Body-L | 16px | 24px | 重点正文 |
| Body-M | 14px | 22px | 常规正文 |
| Body-S | 12px | 20px | 辅助信息 |
| Caption | 11px | 16px | 标签、说明、状态补充 |

原则：

1. 企业系统正文主字号建议以 `14px` 为主
2. 关键说明与对话文本可用 `16px`
3. 不建议出现过多离散字号

### 4.1.3 字重规范

建议统一使用：

1. 400：常规正文
2. 500：次级强调
3. 600：模块标题
4. 700：关键标题 / 状态值 / KPI

## 4.2 颜色规范

### 4.2.1 色彩方向

小乔智投属于企业级数据工作台，不适合高饱和娱乐色盘。

建议方向：

1. 主色偏冷静蓝
2. 辅助色偏青绿
3. 背景为浅灰蓝或浅中性色
4. 风险与告警使用标准语义色

### 4.2.2 基础色板

建议色板：

| 角色 | 建议色值 | 用途 |
| --- | --- | --- |
| Primary | `#175CD3` | 主按钮、主选中态、关键链接 |
| Primary Soft | `#EAF2FF` | 主色浅底、轻提示底色 |
| Accent | `#0F766E` | 成功、辅助高亮 |
| Text Strong | `#0F172A` | 主文字 |
| Text Normal | `#334155` | 正文 |
| Text Muted | `#64748B` | 辅助文字 |
| Border | `#D9E2EC` | 常规边框 |
| Surface | `#FFFFFF` | 卡片背景 |
| Surface Soft | `#F8FAFC` | 弱背景 |
| Page BG | `#F4F7FB` | 页面背景 |

### 4.2.3 语义色

| 语义 | 建议色值 | 用途 |
| --- | --- | --- |
| Success | `#12B76A` | 成功状态 |
| Success Soft | `#ECFDF3` | 成功浅底 |
| Warning | `#F79009` | 风险提醒 |
| Warning Soft | `#FFFAEB` | 风险浅底 |
| Error | `#D92D20` | 错误 / 高风险 |
| Error Soft | `#FEF3F2` | 错误浅底 |
| Info | `#1570EF` | 信息提示 |
| Info Soft | `#EFF8FF` | 信息浅底 |

原则：

1. 状态必须靠颜色 + 文案双重表达
2. 不能只靠颜色区分状态

## 4.3 圆角规范

企业系统建议使用克制、统一的圆角体系：

| 层级 | 圆角 |
| --- | --- |
| XS | 6px |
| S | 8px |
| M | 12px |
| L | 16px |
| XL | 20px |
| Pill | 999px |

使用建议：

1. 输入框、按钮：`8px` 或 `12px`
2. 卡片：`12px` 或 `16px`
3. 浮层 / 抽屉：`16px`
4. 标签 / 状态 pill：`999px`

原则：

1. 不要同一页面混用太多圆角值
2. 企业系统不建议使用过大、过软的圆角

---

## 4.4 间距规范

### 4.4.1 间距体系

建议采用 4pt 或 8pt 体系，这里以 4pt 递进为准：

| Token | 数值 |
| --- | --- |
| Space-1 | 4px |
| Space-2 | 8px |
| Space-3 | 12px |
| Space-4 | 16px |
| Space-5 | 20px |
| Space-6 | 24px |
| Space-8 | 32px |
| Space-10 | 40px |
| Space-12 | 48px |

### 4.4.2 使用建议

1. 行内元素间距：`8px`
2. 表单项内间距：`12px`
3. 卡片内边距：`16px` 或 `20px`
4. 模块之间间距：`24px`
5. 页面级区块之间间距：`32px`

原则：

1. 信息密集型企业系统要稳，不要忽大忽小
2. 卡片内部留白不能太松，避免“营销页化”

---

## 4.5 边框与阴影规范

### 4.5.1 边框

建议统一使用：

1. 常规边框：`1px solid #D9E2EC`
2. 强调边框：`1px solid #175CD3`
3. 风险边框：`1px solid #F79009` / `#D92D20`

### 4.5.2 阴影

企业系统阴影应轻量，不能过重。

建议：

1. Card Shadow：
   - `0 1px 2px rgba(16, 24, 40, 0.06)`
2. Popover Shadow：
   - `0 8px 24px rgba(16, 24, 40, 0.12)`

原则：

1. 用边框和浅背景建立层级
2. 阴影只作为辅助，不作为主要层级工具

---

## 4.6 专项结果区与管理区承接说明

主工作台与管理页的设计承接不再只停留在通用结构，后续前端实现时应直接对齐以下专项视图：

1. 使用帮助：解释、路径、来源、不确定性、继续追问
2. 需求沟通：结构化需求单、缺失字段、依赖项、协作动作
3. 问题排查：结论、证据链、置信度、缺失证据、下一步动作
4. 自动联调：状态机、当前步骤、截图、日志、成功判断、人工接管
5. 提示词管理：列表、详情、版本、绑定、开关、发布状态

## 5. 组件规范

## 5.1 按钮

按钮建议分为：

1. 主按钮
2. 次按钮
3. 文字按钮
4. 危险按钮

主按钮：

1. 用主色填充
2. 主要用于“发送并推进”“新建任务”等核心动作

次按钮：

1. 浅底 + 边框
2. 用于上传、查看、返回等辅助动作

按钮状态必须包含：

1. 默认
2. Hover
3. Active
4. Disabled
5. Loading

## 5.2 输入框

输入框必须包含：

1. 默认态
2. Focus 态
3. Disabled 态
4. Error 态
5. Placeholder 规范

企业系统输入框要求：

1. 边界清晰
2. 支持多行输入
3. 错误提示不只靠红边，要有文字说明

## 5.3 卡片

卡片是小乔智投最核心组件之一。

卡片至少分为：

1. 状态卡片
2. 指标卡片
3. 结果卡片
4. 历史任务卡片
5. 专项页信息卡片

卡片通用规则：

1. 结构明确
2. 标题、值、说明分层清楚
3. 不做过度装饰

## 5.4 标签与状态 Chip

标签分两类：

1. 分类标签
   - 帮助 / 排查 / 联调 / 需求沟通
2. 状态标签
   - 可用 / 进行中 / 待补充 / 已完成 / 风险

标签规范：

1. 高度统一
2. 文字精炼
3. 使用语义色浅底 + 深色文字

## 5.5 消息气泡

消息气泡应区分：

1. 用户消息
2. 小乔普通回复
3. 小乔追问
4. 工作流摘要
5. 系统通知

工作流摘要消息必须能一眼看出“这是任务推进，不是普通闲聊”。

## 6. 状态规范

## 6.1 系统状态

系统状态只允许三类：

1. `可用`
2. `繁忙`
3. `降级`

## 6.2 会话状态

会话状态建议统一为：

1. `普通对话`
2. `帮助中`
3. `排查中`
4. `需求沟通中`
5. `联调中`

## 6.3 任务状态

任务状态前端统一映射为：

1. `待澄清`
2. `进行中`
3. `等待中`
4. `已完成`
5. `已归档`

## 6.4 页面级状态

页面至少定义以下状态：

1. 空状态
2. 加载状态
3. 处理中状态
4. 待补充状态
5. 成功状态
6. 错误状态

### 空状态

用于首次进入主工作台时。

必须包含：

1. 小乔身份说明
2. 输入提示
3. 推荐起手问题

### 加载状态

不能空白，至少要给出：

1. 正在分析
2. 正在联调
3. 正在生成结果

### 错误状态

必须包含：

1. 出错原因概述
2. 可执行下一步动作
3. 重试或返回建议

---

## 7. 页面级规范

## 7.1 主工作台

主工作台是第一优先级页面。

它必须满足：

1. 首页即输入
2. 不要求先选模块
3. 对话和结果同时存在
4. 历史任务可回看

## 7.2 专项页

专项页必须保持企业系统风格统一，不单独“另起一套视觉语言”。

专项页统一骨架：

1. 页面标题区
2. 指标卡片区
3. 模块明细区
4. 返回主工作台入口

---

## 8. 响应式规范

### 8.1 桌面端

建议主内容宽度在信息密集和阅读舒适之间平衡。

要求：

1. 左侧主对话区占主宽度
2. 右侧结果区保持固定阅读宽度
3. 顶部区避免挤成多行

### 8.2 移动端

要求：

1. 对话区优先
2. 结果区可折叠
3. 历史任务全屏抽屉
4. 主要按钮保持可点性

---

## 9. 文案规范

### 9.1 文案原则

企业系统文案应做到：

1. 直接
2. 清晰
3. 不营销
4. 不炫技

### 9.2 文案层级

1. 一级文案：说明当前在做什么
2. 二级文案：说明当前结果或原因
3. 三级文案：说明下一步动作

避免：

1. 过度使用 Agent、MCP 等内部技术词作为前台主文案
2. 把普通状态写得像广告 Slogan

1. 页面结构清楚

3. 组件规范齐全
4. 状态规范齐全

## 补充：AI 企业工作台交互范式

小乔不是传统表单系统，也不是普通聊天产品。

第一阶段设计文档需要显式承认并约束一种新的 AI 企业工作台范式：

`对话承接 + 卡片组织 + 表单补录 + 抽屉编辑 + 结果工作板 + 证据回显`

### 1. 六层模型

1. 对话承接层
   负责接住自然语言输入、发出追问、维持多轮上下文。
2. 卡片组织层
   负责展示任务摘要、缺失字段、附件摘要、步骤状态。
3. 表单补录层
   负责补录媒体、应用、设备、时间范围、需求字段。
4. 抽屉编辑层
   负责承接复杂对象编辑，避免把复杂配置塞进聊天主流。
5. 结果工作板层
   负责把聊天输出沉淀为正式结果对象，而不是一段回复。
6. 证据回显层
   负责展示截图、日志、附件、引用来源，建立结果可信度。

### 2. 为什么必须采用这套范式

1. 只有对话，复杂业务字段和多材料输入无法稳定承接。
2. 只有表单，统一入口和模糊问题承接能力会消失。
3. 四条业务流都需要“自然输入 + 结构化沉淀”并存。

### 3. 对设计输出物的直接要求

后续每一页、每一张卡片、每一个弹窗或抽屉，都要能回答：

1. 它属于六层模型里的哪一层。
2. 它服务哪条业务流。
3. 它接的是对话、对象、还是证据。
4. 它是轻反馈，还是正式交付对象。

### 4. 对主工作台的强约束

主工作台不允许退回到单一聊天壳。

至少必须同时存在：

1. 对话区
2. 上下文摘要区
3. 缺失字段卡区
4. 附件区
5. 结果区
6. 证据区
