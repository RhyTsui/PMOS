## 1. 专项定位

本专项不是某一条业务流的附属细节，而是小乔智投主工作台的公共支撑能力。

如果这层没有单独设计清楚，前端会出现两个问题：

1. 只能做出“能发消息”的对话壳子，无法承接真实广告业务材料。
2. 结果区、任务区、证据区无法和对话输入形成闭环，页面只能停留在示意状态。

因此，会话支撑能力必须作为正式专项沉淀，而不是分散在帮助、需求、排查、联调文档里的零散说明。

---

## 2. 范围定义

本专项覆盖以下 12 项支撑能力：

1. 会话级上下文维护
2. 任务级上下文维护
3. 缺失字段追问与补录
4. 文件上传
5. 图片上传
6. 粘贴上传与拖拽上传
7. 附件预览、删除、失败重试
8. 附件解析与摘要生成
9. 图片识别与截图理解
10. 对话消息引用附件与证据回显
11. 结果区附件摘要、证据卡、截图卡
12. 上下文继承、切换、重置与人工补充

不纳入本专项的内容：

1. 知识库平台本体
2. 评测平台本体
3. 自动联调执行器本体
4. 独立素材分析产品能力

---

## 3. 业务价值

### 3.1 对用户需求的回映

这层能力直接服务以下原始用户场景：

1. 用户带截图、报表、日志、文档来提问，而不是只输一句话。
2. 小乔需要接住这些材料，并把它们写入当前会话和任务，不让用户重复上传和重复说明。
3. 结果区需要把“结论来自哪些材料、哪些截图、哪些补充上下文”展示清楚。
4. 同一问题跨多轮推进时，系统要记得之前补过什么材料，还缺什么材料。

### 3.2 对产品能力的作用

这层能力是四条业务流共同底座：

1. 使用帮助要引用文档、截图、规则页。
2. 需求沟通要接附件、表格、截图、口头说明并整理成需求对象。
3. 问题排查要承接日志、报表、截图和证据链。
4. 自动联调要承接执行截图、执行日志和回写结果。

## 4. 需求拆解

### R-S01 会话上下文持续维护

- 目标：用户在同一会话内不需要反复重复媒体、应用、账号、异常类型、时间范围等基础背景。
- 最低输入：
  - 历史消息
  - 当前消息
  - 当前任务上下文
- 输出：
  - 会话摘要
  - 当前任务上下文字段
  - 缺失字段清单

### R-S02 任务上下文补录

- 目标：当四条业务流任意一条缺少必要字段时，小乔要显式追问并写回任务。
- 最低输入：
  - 当前任务类型
  - 缺失字段列表
  - 用户后续补充
- 输出：
  - 补录后的上下文对象
  - 已补齐字段
  - 仍缺字段

### R-S03 文件与图片材料接入

- 目标：支持用户直接上传截图、文档、表格、日志文件进入当前会话。
- 第一阶段支持：
  - 图片
  - 文档
  - 表格
  - 文本日志
- 输出：
  - 附件元数据
  - 上传状态
  - 解析状态

### R-S04 附件解析与引用

- 目标：上传后的附件不能只挂一个文件名，系统要生成可引用摘要并进入任务上下文。
- 输出：
  - 附件摘要
  - 附件类型
  - 解析结果
  - 可引用片段

### R-S05 结果区证据回显

- 目标：结果区要明确展示哪些截图、哪些附件、哪些字段补充支撑了当前结论。
- 输出：
  - 证据摘要卡
  - 附件引用卡
  - 图片卡
  - 失败或待补证据状态

### R-S06 上下文切换与重置

- 目标：当用户换问题、换业务流、换应用或主动重置时，系统不能把旧上下文污染到新任务。
- 输出：
  - 继承提示
  - 切换提示
  - 重置动作
  - 新旧上下文边界

## 5. 功能拆解

### F-S01 会话上下文摘要生成

- 前端承接：主工作台顶部上下文摘要条、结果区上下文区
- 后端对象：Conversation / TaskContext
- 说明：从历史消息和任务字段生成当前会话摘要

### F-S02 缺失字段识别与追问

- 前端承接：对话区追问消息、结果区缺失字段卡
- 后端对象：TaskContext.missing_fields_json
- 说明：缺少媒体、应用、账号、时间范围、异常现象时触发追问

### F-S03 上传入口管理

- 前端承接：Sender 附件入口、拖拽热区、粘贴上传
- 后端对象：AttachmentInput / TaskContext.attachments_json
- 说明：同一套入口服务四条业务流

### F-S04 上传状态管理

- 前端承接：上传队列、进度态、失败态、重试态
- 后端对象：附件临时态对象
- 说明：第一阶段即使是 mock，也要保留正式状态位

### F-S05 附件解析任务

- 前端承接：附件摘要卡、解析中状态
- 后端对象：AttachmentSummary / EvidenceItem
- 说明：图片、文档、表格、日志解析结果统一进入摘要层

### F-S06 图片识别结果引用

- 前端承接：图片预览弹层、识别摘要、截图引用卡
- 后端对象：EvidenceItem
- 说明：服务联调截图、报错截图、规则页截图等场景

### F-S07 对话消息与附件绑定

- 前端承接：消息气泡下方附件列表、引用标签
- 后端对象：Message -> attachments
- 说明：用户在哪条消息上传，就能在该消息和后续任务中看到材料来源

### F-S08 结果区证据聚合展示

- 前端承接：结果区证据卡、附件卡、截图卡、待补证据卡
- 后端对象：WorkflowResult / EvidenceItem
- 说明：把结论和材料打通，而不是分散在聊天记录里

### F-S09 上下文继承与重置

- 前端承接：切换提示、重置按钮、继承标识
- 后端对象：Conversation / TaskContext
- 说明：同会话追问默认继承，跨问题切换允许用户清空或部分保留

### F-S10 人工补录与编辑

- 前端承接：上下文编辑抽屉、缺失字段补录抽屉
- 后端对象：TaskContext
- 说明：允许用户直接修正媒体、应用、设备、时间范围等关键字段

## 8. 交互设计要求

### 8.1 上传入口

必须同时支持：

1. 点击上传
2. 拖拽上传
3. 粘贴上传

### 8.2 上传状态

每个附件至少有：

1. 上传中
2. 上传成功
3. 解析中
4. 解析成功
5. 上传失败
6. 解析失败

### 8.3 上下文补录

当系统判断缺失关键字段时：

1. 对话区发出追问
2. 结果区出现缺失字段卡
3. 用户可直接在抽屉中补录
4. 补录后结果区即时刷新

### 8.4 结果引用

结果区每条结论支持展开查看：

1. 关联附件
2. 关联截图
3. 关联消息
4. 关联补录字段

## 9. UI 设计要求

### 9.1 主工作台新增区域

1. 输入区附件条
2. 拖拽上传热区
3. 上下文摘要条
4. 缺失字段卡
5. 证据卡组
6. 图片预览弹窗
7. 附件详情抽屉

### 9.2 组件建议

1. 对话区：`Ant Design X Sender + Attachments + Bubble`
2. 列表与卡片：`Ant Design Card / List / Tag / Badge`
3. 预览与补录：`Ant Design Drawer / Modal / Descriptions / Form`
4. 上传：`Ant Design Upload`

### 9.3 页面风格要求

1. 强调任务与证据，不强调装饰性对话壳
2. 图片、文档、日志都以业务材料卡呈现
3. 缺失字段卡必须显式可见，不能埋进说明文案
4. 上传失败与解析失败必须给出强反馈

---

## 10. 前端落地影响

前端不能再只实现“消息列表 + 输入框 + 结果侧栏”。

至少要同步承接：

1. Sender 附件入口
2. 上传队列
3. 上下文摘要条
4. 缺失字段补录卡
5. 附件预览
6. 证据卡
7. 图片卡
8. 结果区引用块

---

## 11. 与四条业务流的映射

1. 使用帮助：截图、规则页、文档引用
2. 需求沟通：需求附件、表格、配置截图
3. 问题排查：报表截图、日志、证据链
4. 自动联调：执行截图、执行日志、阶段产物

---

## 12. 当前结论

“上下文、上传、图片、附件”不是四个零散功能点，而是一组正式的会话支撑能力域。

后续主需求、主功能、主设计和前端实现，都要以这份专项为正式依据。
## 补充：多层交互范式要求

本专项不允许把所有复杂动作都塞进对话气泡。

至少要形成下面的多层交互分工：

1. 对话层：接收问题、追问、过程性反馈
2. 卡片层：展示任务、缺失字段、附件摘要、证据摘要
3. 表单层：补录媒体、应用、设备、时间范围、需求字段
4. 抽屉层：承接复杂编辑和修正动作
5. 结果层：沉淀正式输出对象

这套分工是小乔主工作台后续所有页面设计的共同约束。
