## 0. 本轮纠偏

上一轮评审解决了文档链路漂移和接口占位问题，但问题也很明显：

1. 更偏“文档结构检查”，不够“业务正确性评审”
2. 更偏“是否写全”，不够“是否真能解决广告业务问题”
3. 没有把不同专业 agent 的评审职责拆开
4. 没有把四条核心业务流拆成独立审查对象

所以本轮改为：

1. 不再只按“规划 / 需求 / 功能 / 设计”四份文档横向扫
2. 增加按业务流纵向拆解的评审
3. 把评审席改成有专业分工的 agent 视角
4. 结论必须落到“哪条业务流哪里成立、哪里还虚”

---

## 1. 评审方式

本轮采用双层评审：

1. 文档层评审
   - 看链路、规格、映射是否成立
2. 业务层评审
   - 看四条核心业务流是否真的能承接广告企业系统场景

只有两层都过，才算前置文档真正可用。

---

## 2. 专业评审席

### 2.1 产品策略 Agent

关注：

1. 小乔到底是不是“广告支持数字员工工作台”
2. 第一阶段范围是否克制且闭环
3. 用户问题是否被映射到真实产品能力
4. 当前解决方案是不是该问题在现阶段的更优解，而不是“先拍一个能讲通的方案”

评审方式：

1. 结合用户原始问题，看当前方案是否真的缩短支持、排查、联调链路
2. 对比替代方案，例如：
   - 继续做多菜单后台
   - 继续做纯 FAQ/知识库
   - 继续靠人工串行支持
3. 判断为什么当前“单对话主工作台 + 任务沉淀 + 结构化结果”比替代方案更优
4. 明确是不是存在“看起来完整，但不是最优路径”的设计

### 2.2 广告支持 Agent

关注：

1. 使用帮助场景是否覆盖广告支持高频问答
2. 需求沟通场景是否覆盖媒体回传 / 接入 / 配置需求
3. 联调场景是否符合广告技术支持真实推进方式

### 2.3 数据诊断 Agent

关注：

1. 问题排查场景是否覆盖 gap、归因异常、回传异常、BI 不一致
2. 结论是否要求证据链，而不是空口判断
3. 输入字段是否足够支撑排查

### 2.4 运行时与后端 Agent

关注：

1. 需求是否拆到功能
2. 功能是否拆到接口
3. Conversation / Message / Routing / Task / Result 是否真能承接四类业务流
4. 接口与页面承接是否存在假映射

评审方式：

1. 不只看文档定义，而是结合当前后端现状评审：
   - 已有接口
   - 已有数据对象
   - 已有返回结构
   - 已有缺口
2. 把“下一步开发”拆成具体 task 再判断：
   - 哪些 task 是直接可做
   - 哪些 task 缺 schema / contract
   - 哪些 task 会引发前后端返工
3. 判断当前文档里的功能拆解，是否已经细到可以直接进入开发排期
4. 判断有没有“文档可讲通，但实现上会卡住”的点

### 2.5 设计与前端 Agent

关注：

1. 工作台结构是否真的适合会话式企业系统
2. 结果区、历史任务区、专项页入口是否符合业务使用方式
3. 设计约束是否足够支撑后续 React 落地

### 2.6 主控席

输出统一结论：

1. `通过`
2. `退回补充`
3. `部分通过，但必须带问题进入下一阶段`

## 3. 四条核心业务流拆解

本轮不再只看文档章节，而是强制按四条核心业务流审。

### 3.1 使用帮助流

要回答：

1. 用户是谁
2. 会问什么
3. 小乔要给什么结果
4. 是否需要任务化
5. 是否需要结果结构化

当前判断：

- 成立部分：
  - 已定义帮助类高频问题范围
  - 已拆出 `R-03 / F-09 ~ F-12`
  - 已定义帮助结果区承接
- 仍然偏虚：
  - 还没明确帮助知识来源的优先级和可信度规则
  - 还没定义“只是回答”与“升级为排查/需求”的分流阈值
- 当前结论：`partial`

### 3.2 需求沟通流

要回答：

1. 模糊需求如何追问
2. 缺哪些字段
3. 输出的结构化需求单长什么样
4. 是否能进入后续需求池或交付链

当前判断：

- 成立部分：
  - 已定义 `R-05 / F-17 ~ F-20`
  - 已要求缺失字段识别与结构化结果输出
- 仍然偏虚：
  - 缺少广告接入类需求的标准字段模板分层
  - 缺少“媒体回传 / SDK接入 / 配置调整 / 权限申请”这几类需求的差异化规则
- 当前结论：`partial`

### 3.3 问题排查流

要回答：

1. 排查输入需要哪些字段
2. 小乔能给出什么级别的结论
3. 如何证明不是胡猜
4. 什么时候需要继续追问或转人工

当前判断：

- 成立部分：
  - 已定义 `R-04 / F-13 ~ F-16`
  - 已要求结论、证据链、置信度、下一步建议
  - 已把排查与普通帮助区分
- 仍然偏虚：
  - 还没有把排查证据源拆成媒体数据、客户端日志、归因平台、BI 报表等具体来源
  - 还没有定义不同异常类型的最小排查字段包
- 当前结论：`partial`

### 3.4 广告联调流

要回答：

1. 联调前要准备什么
2. 联调中要看哪些状态
3. 联调结果要沉淀什么
4. 小乔是“看板”还是“执行协同者”

当前判断：

- 成立部分：
  - 已定义 `R-06 / F-21 ~ F-24`
  - 已要求任务、状态、日志、结果四层承接
- 仍然偏虚：
  - 还没拆清账号、设备、环境、媒体、入口方式这些准备项的最小集合
  - 还没拆清自动化执行失败时的回退策略
- 当前结论：`partial`

## 4. 核心专业判断

### 4.1 产品策略判断

结论：`通过`

理由：

1. 产品定位已经收紧到广告支持数字员工工作台，没有继续发散成泛知识库或泛聊天产品
2. 第一阶段四条核心业务流是成立的
3. 以当前系统现状看，单对话主工作台比继续堆菜单页、FAQ 页或纯概念稿更优
4. 范围仍然偏大，但方向正确

### 4.2 广告支持判断

结论：`部分通过`

理由：

1. 已覆盖帮助、需求沟通、联调三类典型广告支持问题
2. 但还没有把广告业务中的“媒体差异、接入差异、配置差异”拆成规则层

### 4.3 数据诊断判断

结论：`部分通过`

理由：

1. 已认识到排查必须给证据链，不只是给回答
2. 但还没有形成不同异常类型的最小排查字段包和证据源清单

### 4.4 运行时与后端判断

结论：`部分通过`

理由：

1. 文档链已经做到 `需求 -> 功能 -> 接口`
2. 但这还不等于“开发 task 已经拆实”
3. 四条业务流共用的返回结构 contract 还没有写成更硬的 schema 约束
4. 还缺少一层“基于现状代码的开发任务拆解”，否则前端简单版做完后仍可能回头改接口

### 4.5 设计与前端判断

结论：`通过`

理由：

1. 单对话主工作台方向是对的
2. 设计文档已具备企业系统可落地基础
3. 现在缺的不是概念，而是业务态页面实现

## 6. 下一轮必须补的业务拆解

进入页面前或与页面并行，必须继续补这四件事：

1. 帮助流：知识来源优先级、可信度、升级阈值
2. 需求流：广告需求类型模板和字段包
3. 排查流：异常类型 -> 最小排查字段包 -> 证据源映射
4. 联调流：准备项清单、执行状态机、失败回退策略

同时必须补一层开发侧拆解：

5. 基于当前系统现状，把下一步开发拆成明确 task：
   - 哪些只改前端页面
   - 哪些需要补接口返回字段
   - 哪些需要补任务结果 schema
   - 哪些需要补 workspace / task / result 的真实 contract
