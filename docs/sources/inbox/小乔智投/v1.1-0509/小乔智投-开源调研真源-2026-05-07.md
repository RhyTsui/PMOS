- 版本：v0.1

- 负责人：Codex + 用户

---

## 0. 文档目的

本文档用于沉淀小乔智投在需求、原型、交互、前端之前的开源调研与复用判断。

该步骤应视为项目正式流程的一部分，后续各项目默认复用这一做法。

---

## 1. 调研结论总览

小乔智投当前不适合直接切换到一个通用开源 Agent 平台整体重做。

更合理的策略是：

1. 保留现有内部平台分工
2. 优先复用通用运行底座与 AI 工程底座
3. 业务工作台保持自有任务模型与上下文模型
4. 对适合的开源产品采用 `部署 + 二开`，而不是盲目自研或整体替换

---

## 2. 已有内部平台

### 2.1 Dataki

当前已承担：

1. 知识维护
2. API / key 对接
3. 知识资产管理
4. 后续提示词资产管理与归档

结论：

小乔智投不应重做知识库平台。

### 2.2 连弩测试平台

当前已承担：

1. AI 应用评测
2. 调试验证
3. 效果对比

结论：

小乔智投不应重做通用 AI 应用评测平台。

---

## 3. 推荐重点候选

### 3.1 Langfuse

类型：

开源 LLM 工程平台，支持 prompt management、observability、evals、自托管。

适配判断：

适合用于补小乔智投的：

1. 应用级 trace
2. prompt 版本可视化
3. 运行观测
4. 数据集与线上行为关联

建议：

优先列为 `可部署后二开` 候选。

来源：

- https://github.com/langfuse/langfuse
- https://langfuse.com/self-hosting
- https://langfuse.com/docs/prompt-management/overview

### 3.2 Opik

类型：

开源 AI observability / evaluation / optimization 平台，支持自托管。

适配判断：

适合用于补小乔智投的：

1. 评测结果视图
2. trace 与调试链路
3. prompt / agent 优化实验

建议：

优先列为 `可部署后二开` 候选。

来源：

- https://github.com/comet-ml/opik

### 3.3 n8n

类型：

开源 workflow automation 平台。

适配判断：

适合承接：

1. 定时任务
2. 通知编排
3. 轻量跨系统自动化

建议：

不作为小乔智投主流程核心编排平台。

如后续需要，可作为外围自动化工具候选保留。

来源：

- https://github.com/n8n-io/n8n

### 3.4 Playwright + Appium

类型：

开源浏览器 / 移动自动化框架。

适配判断：

适合继续承接：

1. 广告联调浏览器自动化
2. App 自动化控制
3. 联调验证链路

建议：

继续保留，不为统一平台感而强行替换。

来源：

- https://github.com/appium/appium

## 4. 研究但不建议作为主底座替换

### 4.1 Dify

类型：

开源 LLM 应用开发平台，支持 workflow、RAG、agent、observability。

判断：

适合参考其：

1. 应用控制台结构
2. agent / workflow 编排方式
3. RAG 与 workflow 的组合方式

但不适合直接替掉当前：

1. Dataki
2. 连弩
3. 小乔智投自身业务工作台

原因：

容易与现有内部平台分工重叠，反而造成平台重复建设。

来源：

- https://github.com/langgenius/dify

### 4.2 Langflow

类型：

开源可视化 AI agent / workflow builder。

判断：

适合参考其：

1. 可视化 flow 表达
2. 组件化 agent 搭建方式

但不适合直接作为小乔智投主底座。

原因：

小乔智投长期核心仍然会回到自己的业务上下文、任务模型和结果模型，不宜把核心控制权交给通用画布平台。

来源：

- https://github.com/langflow-ai/langflow

---

## 5. 当前不建议优先引入

### 5.1 Novu

类型：

开源通知基础设施。

判断：

如果后续通知体系复杂到超出小闪与现有工作流编排能力，可以再评估。

当前阶段不建议优先接入。

来源：

- https://github.com/novuhq/novu

### 5.2 Temporal

类型：

开源 durable execution / workflow orchestration 平台。

判断：

如果 8 月后广告执行 Agent 出现大量长事务、补偿、恢复、重试场景，可再作为中后期候选。

当前阶段不建议优先接入。

来源：

- https://github.com/temporalio

## 7. 边界结论

小乔智投当前更适合定义为：

建立在 Dataki、连弩、自有业务运行时与联调自动化能力之上的广告业务 Agent 应用层。

其主流程编排方向为：

1. 开发自写流程
2. 建设小乔智投自己的业务运行时
3. 不以 n8n 为核心编排中枢

PMAIOS 只承担：

1. 产品侧治理
2. 方法论支撑
3. 真源与流程管理

不承担小乔智投业务运行时职责。

它应保留自己的：

1. 业务会话上下文
2. 任务上下文
3. 历史任务
4. 结构化结果
5. Case 沉淀

而不应重做：

1. 知识库平台
2. 通用评测平台
3. 通用低代码 Agent 平台
