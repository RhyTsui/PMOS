# MCP QA Platform 架构规划

## 1. 总体架构

项目采用单体应用架构，后端是 FastAPI，前端是单页静态页面，数据库采用 SQLite。虽然实现比较轻量，但业务层次已经分得比较清楚。

## 2. 分层设计

| 层次 | 职责 | 代码位置 |
|------|------|----------|
| 页面层 | 单页控制台，承载用例管理、执行发起、结果查看、配置维护 | `web/index.html` |
| API 层 | 对外提供 REST 接口 | `app/api/router.py`、`app/api/v1/*.py` |
| 服务层 | 业务编排、校验、执行控制、报告生成 | `app/service/**` |
| Runner 层 | 契约执行、语义执行、内部对话执行等 | `app/service/runner/**` |
| 存储层 | 建表、迁移、SQL 读写 | `app/storage/**` |
| 配置层 | 静态配置和运行时配置读取 | `configs/**`、`app/core/config.py`、`app/service/runtime_config_service.py` |

## 3. 六个核心模块

### 3.1 用例

系统把用例分成两类：

1. 契约用例 `contract_cases`
2. 语义用例 `semantic_cases`

它们共用同一套执行平台，但执行逻辑不同。

### 3.2 评测集

评测集 `test_suites` 负责把同类用例组织起来，便于复用和回归执行。

### 3.3 执行

执行任务 `eval_runs` 是主轴对象。页面发起一次执行后，后台线程会根据请求参数加载样本、选择 Runner、并发执行，再把进度和结果写回数据库。

### 3.4 Trace

Trace 不是普通日志，而是执行系统的关键组成。当前会保存：

1. 总体 Trace
2. 原始 Trace
3. 规范化 Trace
4. Span 明细

### 3.5 报告

报告模块根据执行结果输出 Markdown 或 HTML 产物，承担总结、复盘和交付功能。

### 3.6 配置

配置分成两层：

1. 静态配置：应用名、数据库路径、默认产物目录等
2. 运行时配置：模型、MCP、并发、超时、内部对话接口等

## 4. 主链路架构

### 4.1 页面主链路

```text
维护用例
  -> 组织评测集
  -> 发起执行
  -> 查看结果
  -> 查看 Trace
  -> 生成报告
  -> 基于结果继续优化用例
```

### 4.2 后端执行链路

```text
API 收到执行请求
  -> QaRunService 创建 eval_run
  -> QaRunTaskService 后台线程启动
  -> 选择 ContractRunner / SemanticRunner / InternalChatRunner
  -> 写入 qa_run_case_results
  -> 写入 traces / normalized_traces / spans
  -> 回写 eval_runs 汇总
  -> 生成 reports
```

## 5. 页面模块与后端模块映射

| 页面模块 | API 前缀 | 核心服务/存储 |
|----------|----------|---------------|
| 仪表盘 | `/api/v1/dashboard`、`/api/v1/qa/stats` | `DashboardService`、`EvalStore` |
| 契约用例 | `/api/v1/qa/contract-cases` | `ContractCaseService`、`ContractCaseStore` |
| 语义用例 | `/api/v1/qa/semantic-cases` | `SemanticCaseStore`、`SemanticCaseImportService` |
| 评测集 | `/api/v1/qa/test-suites` | `TestSuiteStore` |
| 执行 | `/api/v1/qa/runs` | `QaRunService`、`QaRunTaskService` |
| Trace | `/api/v1/traces` | `TraceService` |
| 报告 | `/api/v1/reports` | `ReportService` |
| 配置 | `/api/v1/runtime-config`、`/api/v1/models` | `RuntimeConfigService`、`YamlLoader` |

## 6. 两种核心执行模式

### 6.1 契约执行

契约执行对应 `ContractRunner`，主要流程是：

1. 读取注册表和映射规则
2. 发现 MCP 工具
3. 调用工具
4. 解析请求视图
5. 比较 schema / request / result
6. 记录失败阶段和阶段结果

它强调的是“结构正确”和“链路正确”。

### 6.2 语义执行

语义执行对应 `SemanticRunner` 或 `InternalChatRunner`，主要流程是：

1. 用自然语言问题驱动 Agent 或内部对话
2. 记录工具调用和最终回答
3. 调用评分逻辑判断答案质量
4. 在需要时补充参考结果

它强调的是“工具是否选对、回答是否合理、答案是否符合预期”。

## 7. 当前架构优点

1. 用例、执行、Trace、报告已经拆成独立对象。
2. 契约测试和语义测试都能统一在一个平台里。
3. 运行时配置已和代码解耦，联调成本较低。
4. Store 层隔离了数据库读写，为后续替换数据库留出了空间。

## 8. 当前架构限制

### 8.1 前端工程化不足

前端集中在一个 `web/index.html` 中，页面功能很多，但可维护性会随着迭代迅速下降。

### 8.2 长任务依赖后台线程

现在通过后台线程执行长任务，适合单机和联调，但不适合更复杂的调度场景。

### 8.3 SQLite 更适合单机

SQLite 配合 WAL 足够支撑当前单机使用，但不适合未来多人并发协作。

## 9. 后续架构建议

在继续演进这套架构时，最重要的建设方向是把现有链路升级成“自动生成 + 自动校验 + 少量人工复核”的架构。

从现状看，系统已经有一些自动化基础：

1. 契约用例可基于 MCP schema 自动生成
2. 注册表和映射规则可参与期望请求推导
3. 语义用例在部分场景下可自动补全 `expected_result`
4. 契约执行支持阶段化校验
5. 语义执行支持自动评分

下一步架构重点建议放在两条线上：

1. 用例生成自动化
   让 `schema -> 用例骨架 -> 期望参数 -> 期望请求 -> 参考结果` 的链路更完整。
2. 校验自动化
   让 `执行结果 -> 差异比对 -> 自动判定 -> 高风险复核` 的链路更稳定。

如果要延续这个系统，建议优先做三件事：

1. 先稳定领域模型，不要先推倒页面。
2. 先把“用例-评测集-执行-Trace-报告”五段主链路抽清楚。
3. 再逐步把页面和执行系统做成真正的工程化模块，同时持续提高用例生成和校验自动化比例。
