# MCP QA Platform 功能需求文档

## 1. 文档说明

本文件描述“当前系统已经实现或明显支持的功能需求”，重点围绕页面主流程展开，尤其关注：

1. 用例
2. 评测集
3. 执行
4. Trace
5. 报告
6. 配置

## 2. 角色视角

| 角色 | 主要诉求 |
|------|----------|
| QA / 评测负责人 | 维护用例、组织评测集、发起执行、复核结果 |
| 研发 / 联调人员 | 查看失败样本、查看 Trace、定位问题 |
| 项目负责人 | 查看统计和报告，判断当前质量状态 |

## 3. 功能主线

系统需要支持一条完整闭环：

1. 把测试样本沉淀为资产
2. 把资产组织成可复用评测集
3. 按不同方式执行
4. 留下结构化结果和 Trace
5. 生成可阅读报告
6. 通过配置切换运行环境

## 4. 功能模块需求

## 4.1 用例模块

### FR-CASE-01 契约用例管理

系统应支持维护面向 MCP 接口的契约用例，包括：

1. 新建
2. 编辑
3. 查询和分页
4. 批量删除
5. 导入 Excel
6. 导出 Excel
7. 根据 MCP tool schema 自动生成基础用例

契约用例应至少覆盖以下信息：

1. 工具名
2. 来源类型 `MF / DW / UNKNOWN`
3. 用例类型
4. 期望 MCP 参数
5. 期望 API 请求体
6. 请求校验规则
7. 断言模式
8. 风险等级

### FR-CASE-02 语义用例管理

系统应支持维护面向问答能力的语义用例，包括：

1. 新建
2. 编辑
3. 查询和分页
4. 批量删除
5. 批量修改部分字段
6. 导入 Excel
7. 维护期望答案
8. 在具备条件时自动补充期望答案

语义用例应至少覆盖以下信息：

1. 问题文本
2. 业务域
3. 类别
4. 期望工具
5. 期望参数
6. 期望答案
7. 允许/禁止工具集合
8. 关键答案要点
9. 是否 golden

### FR-CASE-03 用例支撑能力

系统应支持维护工具注册表和映射规则，为契约用例生成、请求对比和执行校验提供基础信息。

### FR-CASE-04 用例生成自动化

系统应持续提升用例生成自动化比例，减少人工编写样本的工作量，至少覆盖以下方向：

1. 基于 MCP schema 自动生成契约用例骨架
2. 基于注册表和映射规则自动补全期望参数和期望请求
3. 基于参考接口或已有配置自动补全部分语义用例的 `expected_result`
4. 基于历史失败样本和回归样本辅助生成新样本

## 4.2 评测集模块

### FR-SUITE-01 评测集维护

系统应支持：

1. 新建评测集
2. 编辑评测集名称和描述
3. 维护评测集中的 case_id 列表
4. 删除评测集
5. 区分契约评测集和语义评测集

### FR-SUITE-02 从执行结果回流评测集

系统应支持从一次运行中提取样本，再加入新的或已有的评测集，用于沉淀失败样本和回归样本。

## 4.3 执行模块

### FR-RUN-01 发起执行

系统应支持从页面发起执行，并允许指定：

1. 用例类别
2. 评测集或手工 case_id 列表
3. 执行模式
4. 模型
5. 执行器类型
6. MCP Profile
7. 校验级别
8. Trace 级别

### FR-RUN-02 执行状态与进度

系统应支持查看执行生命周期状态，包括：

1. queued
2. running
3. cancelling
4. cancelled
5. completed
6. failed

同时需要显示：

1. 已完成样本数
2. 总样本数
3. 平均分
4. 自动通过/自动失败/待复核数量
5. 失败阶段汇总

### FR-RUN-03 执行结果查看

系统应支持查看某次执行的单样本结果列表，并支持：

1. 只看失败
2. 按复核状态筛选
3. 查看单条结果详情
4. 导出 CSV
5. 重跑全部样本
6. 重跑失败样本

### FR-RUN-04 人工复核

系统应支持人工对单条结果进行复核，包括：

1. 标记通过或失败
2. 填写复核意见
3. 对语义用例补录或回填期望答案
4. 将复核结果回写到 run 级统计

### FR-RUN-05 校验自动化比例提升

系统应持续提升校验自动化比例，减少人工逐条检查结果的工作量，至少包括：

1. 契约类自动进行 schema、请求参数、请求视图和失败阶段校验
2. 语义类自动进行答案评分、参考结果比对和工具命中校验
3. 对低风险、规则明确的样本尽量自动判定
4. 将人工复核集中到高风险、低置信度或争议样本

## 4.4 Trace 模块

### FR-TRACE-01 Trace 查询

系统应支持：

1. 查看 Trace 列表
2. 查看单条 Trace 详情

### FR-TRACE-02 Trace 记录粒度

系统应支持保存不同粒度的信息，至少包括：

1. 原始执行信息
2. 标准化 Trace
3. Span 列表
4. 工具调用信息
5. 最终回答
6. 错误类型和错误消息

### FR-TRACE-03 契约链路可观察性

对于契约执行，系统应支持观察至少以下阶段：

1. register
2. discover
3. schema_diff
4. call
5. request_view
6. response_summary
7. assert

## 4.5 报告模块

### FR-REPORT-01 单次执行报告

系统应支持根据一次运行生成 QA 报告，内容至少包括：

1. 执行基本信息
2. 汇总指标
3. 样本结果清单

### FR-REPORT-02 聚合报告

系统应支持把多个运行结果汇总成一个组合报告，用于阶段性质量观察。

### FR-REPORT-03 报告查看与下载

系统应支持：

1. 列表查看报告
2. 查看单个报告内容
3. 下载 HTML 或 Markdown

### FR-REPORT-04 报告修订

系统代码支持对报告发起对话式修订请求，可作为后续增强能力继续扩展。

## 4.6 配置模块

### FR-CONFIG-01 运行时配置管理

系统应支持通过页面维护运行时配置，包括：

1. 聊天模型地址、Key、模型名
2. 评分模型地址、Key、模型名
3. MCP 地址与认证信息
4. MCP Profile 列表
5. 内部对话接口地址与认证信息
6. 默认执行器
7. 系统提示词模板
8. Agent 最大步数
9. 超时时间
10. 并发度

### FR-CONFIG-02 模型列表查询

系统应提供可选模型列表，供页面配置和执行时选择。

### FR-CONFIG-03 自动化策略配置

系统后续应支持对自动化相关策略进行配置，例如：

1. 用例自动生成策略
2. 自动补全期望结果的开关和来源
3. 自动校验阈值
4. 进入人工复核的触发条件

## 4.7 统计模块

### FR-STAT-01 汇总统计

系统应支持查看整体统计，包括：

1. 契约用例数量
2. 语义用例数量
3. 最近执行列表
4. 自动通过率
5. 复核队列规模
6. MCP 接口测试指标
7. 语义端到端测试指标

## 5. 页面未完全体现但代码中存在的能力

为避免误解，这里集中说明一次：

1. 代码里保留了 legacy eval 数据集执行能力，接口前缀是 `/api/v1/evals`。
2. 报告支持对话式修订，不一定在每个页面入口里都被强调。
3. 契约用例自动生成依赖注册表和映射规则，不只是单纯读取 MCP schema。

除这一节外，本文其余部分都以页面主流程为中心，不再分散讨论页面外能力。

## 6. 当前最重要的功能缺口

当前最明确的功能缺口是：

**自动化比例还不够高，尤其是用例生成自动化和校验自动化仍不足，导致人工投入偏大。**

因此后续功能建设应优先围绕：

1. 减少手工编写用例
2. 减少手工补录期望值
3. 减少手工逐条核对执行结果
4. 让人工更多集中在高风险样本审核上
