# MCP QA Platform 数据库表说明

## 1. 说明

本说明以 `app/storage/db.py` 中的 `init_db()` 为准。当前数据库是 SQLite，表结构由代码维护，不是通过独立迁移框架维护。

本文优先说明“表的业务作用”和“表之间的关系”，让接手者能快速建立整体认识。

## 2. 表分组总览

| 分组 | 表名 |
|------|------|
| 配置与版本 | `prompts`、`versions` |
| 执行主表 | `eval_runs`、`eval_runtime_states` |
| 执行结果与复核 | `qa_run_case_results`、`qa_run_reviews`、`score_results` |
| Trace | `traces`、`raw_traces`、`normalized_traces`、`normalized_trace_spans` |
| 用例与评测集 | `contract_cases`、`semantic_cases`、`test_suites`、`canonical_cases` |
| 注册表与映射 | `tool_adapter_registry`、`adapter_mapping_rules` |
| 报告 | `reports` |
| 页面未主打但代码保留 | `release_decisions` |

## 3. 关键关系

主关系链可以理解为：

```text
contract_cases / semantic_cases
  -> test_suites
  -> eval_runs
  -> qa_run_case_results
  -> traces / normalized_traces / spans
  -> reports
```

需要注意：

1. 当前库里不是强外键设计，关系主要靠业务字段维护。
2. 大量复杂结构通过 JSON 文本字段存储。

## 4. 重点表说明

## 4.1 `contract_cases`

契约用例表，对应页面中的“契约用例”。

### 主要作用

1. 存储面向 MCP 接口的结构化样本
2. 支持导入、导出、自动生成和编辑
3. 为契约执行提供期望参数和校验规则

### 关键字段

| 字段 | 含义 |
|------|------|
| `case_id` | 主键 |
| `source_tool` | MCP 工具名 |
| `mcp_case_type` | 用例类型，如 `REPORT_TOOL`、`DW_TOOL` |
| `source_type` | 来源类型，如 `MF`、`DW` |
| `case_type` | 场景类型，如 happy path、missing required |
| `expected_tool_name` | 期望工具名 |
| `register_mode` | 执行模式 |
| `expected_mcp_args_json` | 期望 MCP 参数 |
| `expected_api_body_json` | 期望 API Body |
| `expected_api_request_json` | 期望请求视图 |
| `expected_validation_json` | 校验规则 |
| `assert_mode` | 断言模式 |
| `risk_level` | 风险等级 |
| `review_status` | 用例审核状态 |
| `generated_from` | 用例来源 |

这张表已经为“用例生成自动化”预留了基础字段，尤其是：

1. `generated_from`
2. `expected_mcp_args_json`
3. `expected_api_body_json`
4. `expected_api_request_json`
5. `expected_validation_json`

## 4.2 `semantic_cases`

语义用例表，对应页面中的“语义用例”。

### 主要作用

1. 存储自然语言问题样本
2. 维护期望工具、期望参数、期望答案
3. 支持 golden 样本维护

### 关键字段

| 字段 | 含义 |
|------|------|
| `case_id` | 主键 |
| `question` | 问题文本 |
| `domain` | 业务域 |
| `category` | 类别 |
| `expected_result` | 期望答案 |
| `expected_param_json` | 期望参数 |
| `report_api_params_json` | 报表接口参数 |
| `expected_tool` | 期望工具 |
| `allowed_tools_json` | 允许工具集合 |
| `forbidden_tools_json` | 禁止工具集合 |
| `report_refs_json` | 参考报表关联 |
| `answer_keypoints_json` | 关键答案点 |
| `is_golden` | 是否黄金样本 |

这张表是后续“语义用例自动化”的关键基础，尤其是：

1. `expected_result`
2. `expected_param_json`
3. `report_api_params_json`
4. `expected_tool`
5. `report_refs_json`

## 4.3 `test_suites`

评测集表，对应页面“评测集”模块。

### 主要作用

1. 组织一组 case_id
2. 区分契约评测集和语义评测集
3. 为重复执行提供稳定样本集合

### 关键字段

| 字段 | 含义 |
|------|------|
| `suite_id` | 主键 |
| `suite_name` | 名称 |
| `description` | 描述 |
| `case_category` | `contract` 或 `semantic` |
| `case_ids_json` | 样本 ID 列表 |
| `status` | 状态 |

## 4.4 `eval_runs`

执行任务主表，是整个执行系统的中轴。

### 主要作用

1. 表示一次完整运行
2. 记录运行参数和进度
3. 记录最终汇总统计

### 关键字段

| 字段 | 含义 |
|------|------|
| `eval_run_id` | 主键 |
| `run_type` | `contract`、`semantic` 或历史类型 |
| `status` | 任务状态 |
| `executor_type` | 执行器类型 |
| `model_name` | 模型名 |
| `test_category` | 测试大类 |
| `execution_mode` | 执行模式 |
| `source_type_filter` | 来源过滤条件 |
| `validation_level` | 校验级别 |
| `trace_level` | Trace 级别 |
| `total_cases` | 总样本数 |
| `completed_cases` | 已完成数 |
| `passed_cases` | 最终通过数 |
| `failed_cases` | 最终失败数 |
| `review_required_cases` | 待复核数 |
| `avg_score` | 平均分 |
| `version_snapshot_json` | 版本快照 |
| `metrics_json` | 运行请求和统计信息 |
| `failed_reason_summary_json` | 失败原因分布 |
| `failure_stage_summary_json` | 失败阶段分布 |

## 4.5 `qa_run_case_results`

单样本执行结果表，是执行复盘最重要的一张表。

### 主要作用

1. 存储某次 run 中每条样本的结果
2. 承载页面执行详情和人工复核数据
3. 承载 Trace、参数对比、失败阶段和评分结论

### 关键字段

| 字段 | 含义 |
|------|------|
| `result_id` | 主键 |
| `eval_run_id` | 所属 run |
| `case_id` | 所属用例 |
| `case_category` | 契约或语义 |
| `execution_mode` | 执行模式 |
| `trace_id` | 关联 Trace |
| `question` | 执行时的问题文本 |
| `expected_tool` / `actual_tool` | 期望与实际工具 |
| `expected_mcp_args_json` / `actual_mcp_args_json` | MCP 参数对比 |
| `expected_api_request_json` / `actual_api_request_view_json` | 请求视图对比 |
| `expected_api_body_json` / `actual_api_body_json` | 请求体对比 |
| `request_diff_json` | 请求差异 |
| `schema_diff_summary_json` | schema 差异摘要 |
| `mcp_raw_response_json` | 原始执行结果 |
| `final_answer` | 最终回答 |
| `score_detail_json` | 打分详情 |
| `fail_reasons_json` | 失败原因 |
| `auto_decision` | 自动判定 |
| `failure_stage` | 失败阶段 |
| `review_status` | 复核状态 |
| `review_comment` | 复核备注 |
| `review_score` | 复核分数 |

这张表是“校验自动化”最关键的落点。后续如果要减少人工投入，核心不是新建另一张表，而是继续围绕这里增强：

1. 自动判定能力
2. 失败阶段定位能力
3. 参数和请求差异自动比对能力
4. 人工复核触发条件

## 4.6 `traces`

Trace 主表，记录一次样本执行的整体 Trace 信息。

核心字段包括：

1. `trace_id`
2. `eval_run_id`
3. `case_id`
4. `query`
5. `executor_type`
6. `status`
7. `parsed_intent`
8. `parsed_params_json`
9. `final_answer`
10. `error_type`
11. `error_message`
12. `artifact_path`

## 4.7 `raw_traces`

保存原始 Trace 载荷。

## 4.8 `normalized_traces`

保存规范化后的 Trace 主体。

## 4.9 `normalized_trace_spans`

保存规范化 Span 列表。

这三张表共同组成 Trace 子系统。

## 4.10 `reports`

报告表，对应页面“报告”模块。

### 主要作用

1. 存储报告元信息
2. 关联执行任务
3. 指向实际报告产物

### 关键字段

| 字段 | 含义 |
|------|------|
| `report_id` | 主键 |
| `eval_run_id` | 对应 run |
| `version_id` | 关联版本 |
| `title` | 标题 |
| `format` | `markdown` 或 `html` |
| `summary_json` | 报告摘要 |
| `artifact_path` | 报告文件路径 |

## 4.11 `eval_runtime_states`

运行时状态表，用于展示后台任务当前进度和当前样本。

它与 `eval_runs` 的区别是：

1. `eval_runs` 更偏最终事实
2. `eval_runtime_states` 更偏运行中状态

## 4.12 `qa_run_reviews`

复核记录表，用于沉淀人工复核历史，而不是只保留当前状态。

## 4.13 `tool_adapter_registry`

注册表，描述 MCP 工具如何映射到目标 API。

### 关键字段

1. `tool_name`
2. `target_api`
3. `adapter_name`
4. `source_type`
5. `domain`
6. `report_family`
7. `config_json`

## 4.14 `adapter_mapping_rules`

映射规则表，描述 canonical 字段如何映射到目标参数。

### 关键字段

1. `adapter_name`
2. `canonical_field`
3. `target_param`
4. `transform_rule`
5. `required`
6. `default_rule`
7. `enum_mapping_json`

## 4.15 `prompts` 与 `versions`

这两张表属于配置和版本管理层。

### `prompts`

用于存储 Prompt 模板及其描述、变量、作者和更新时间。

### `versions`

用于保存执行版本快照，包括 prompt、rule、tool、executor、model 及其他快照信息。

## 4.16 `canonical_cases`

这是 legacy eval 用例表，主要服务于旧的数据集执行链路，不是当前页面主流程重点。

## 4.17 `score_results`

这是 legacy 评分结果表，和旧的评测流程关系更紧密。

## 4.18 `release_decisions`

这张表目前不是页面主流程重点，但代码中仍保留。接手时知道它存在即可，不建议把它作为本期理解主线。

## 5. 索引说明

`db.py` 中手工创建了一批索引，重点围绕以下查询场景优化：

1. 按 run 状态和时间查询
2. 按 run_id 查询结果和 Trace
3. 按 review_status 查询复核队列
4. 按 source_type、case_type 查询契约用例
5. 按 domain、category 查询语义用例

## 6. 数据库设计特点

### 6.1 大量 JSON 文本字段

为了灵活性，很多复杂结构以 JSON 文本形式存储，开发很快，但查询分析能力有限。

### 6.2 关系主要由业务字段维护

当前不是严格外键驱动模型，因此要特别注意数据清理和孤儿记录。

### 6.3 建表和迁移都在代码里

修改数据库时，不能只改表，还要同步更新 `init_db()` 和补列逻辑。

## 7. 从数据库视角看当前缺口

从数据模型角度看，当前最明确的缺口同样是自动化比例不足。

现有库已经具备一批自动化基础字段，但还可以继续围绕下面两件事加强：

1. 用例生成自动化
   需要更好利用 `generated_from`、期望参数、期望请求、参考结果等字段。
2. 校验自动化
   需要更好利用 `auto_decision`、`failure_stage`、`request_diff_json`、`schema_diff_summary_json`、`score_detail_json` 等字段。

这意味着后续数据库演进应优先服务于：

1. 降低手工录入用例成本
2. 降低手工核对结果成本
3. 提高自动判定可解释性

## 8. 重设计建议

如果未来需要继续演进数据库，建议优先考虑：

1. 保留 `contract_cases` 和 `semantic_cases` 的边界，不要轻易合并。
2. 继续把 `qa_run_case_results` 作为主结果事实表。
3. Trace 子系统应继续独立保留，因为它已经形成单独价值。
4. 若迁移到 PostgreSQL，可优先把 JSON 文本字段升级为 JSONB。
5. 后续所有字段演进，都优先服务“提升自动化比例、减少人工投入”这一目标。
