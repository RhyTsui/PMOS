一、这张图怎么讲

一句话版本：

终端负责承接，Runtime 负责组装和执行，安全中心负责判权，Prompt 决定“怎么说”，知识库决定“说什么”，Agent 负责“谁来干”，模型网关负责“找谁生成”，评测观测负责“跑得好不好”。

这套分法和 MCP 的“协议接入层”定位是一致的：
MCP 负责标准化接入 tools / resources / prompts，但企业自己的 Runtime Control Plane 仍然要负责编排、状态、权限、安全边界和治理。

二、安全中心怎么放

这里我建议你把安全中心独立成横切治理中枢，不要再只挂在 Runtime 里。

安全中心的终版职责
1. IAM
用户
部门
组织
角色
Agent 身份
服务身份
Bot 实例身份
2. RBAC

用于粗粒度授权：

谁能创建 Agent
谁能编辑 Prompt
谁能发布知识库
谁能看评测结果
谁能配置模型策略
3. ABAC

用于细粒度动态授权。NIST 的 ABAC 核心就是：
基于主体、客体、操作、环境属性去做策略计算。

所以你这边可以直接落成：

主体 Subject attributes
用户角色、部门、职级、Agent 类型、实例来源、租户
客体 Object attributes
知识库等级、文档密级、工具风险级别、Prompt 基线级别
操作 Action attributes
read / retrieve / invoke / publish / export / approve
环境 Environment attributes
渠道、时间、地域、网络、会话风险、任务模式
4. Agent 权限

这是你这版最关键的新增点。
Agent 不应该天然继承用户全部权限，而应该走：

用户权限 ∩ Agent 策略上限 ∩ 工具策略 ∩ 数据策略 ∩ 当前环境约束

这样才能防止“用户能看一点数据，但 Agent 被放出去后把整库都检出来”。这本质上就是 ABAC 在 agent runtime 里的落地。

三、多 Agent 分层终版

我建议你固定成这三层，不要再继续加概念：

1）Planner Agent

负责：

理解目标
拆任务
判断走强控还是弱控
选择 Worker
决定是否 handoff

OpenAI 现在对 agent orchestration 里 handoffs 和 guardrails 的强调，跟这层很匹配。

红线：

不直接操作高风险工具
不直接读取高敏知识
不拥有广泛业务执行权限
只做决策和路由
2）Worker Agents

负责：

检索
汇总
分析
写作
诊断
任务执行

Worker 是“任务处理层”，拿的是任务级临时权限，不是长期全局权限。

红线：

不能随意换工具域
不能突破 Planner 下发的任务边界
不能越过安全中心直接取高密级知识
3）Tool / Skill Agents

负责：

SQL 查询
搜索
API 调用
文件处理
浏览器操作
系统执行

这一层本质是“受控能力封装”。
要把高风险能力关在这层，不让上层 Agent 直接碰底层系统。

红线：

只暴露受控接口
默认无业务语义决策权
默认无跨域数据合并权
默认不持有长期主账号权限
四、知识库权限 vs Agent 权限

这是最容易做乱的地方，我直接给你终版规则。

原则

知识库权限是“数据能不能被看见”，Agent 权限是“它能不能替你去动用某种能力”。

两者不是一个东西。

1）知识库权限

控制的是：

哪个用户能搜哪个库
哪个 Agent 能检哪个库
哪个文档分片能出现在上下文
哪些内容需要脱敏 / 摘要化 / 不可原文返回

所以知识库权限要落在：

库级
文档级
分片级
字段级
返回级
2）Agent 权限

控制的是：

哪个 Agent 能调用什么工具
能否使用互联网搜索
能否调 SQL
能否写回系统
能否发消息、审批、创建工单、执行动作
3）两者组合关系

最终生效不是二选一，而是双重约束：

知识召回时

最终可召回范围 = 用户知识权限 ∩ Agent知识权限 ∩ 当前任务范围 ∩ 环境策略

工具调用时

最终可调用能力 = 用户操作权限 ∩ Agent工具权限 ∩ 工具风险策略 ∩ 当前模式约束

4）最重要的一条

Agent 不能因为“帮用户做事”就自动拿到比用户更高的数据权限。
最多只能在受控场景下拿到“代理执行权限”，而且必须被审计、可追溯、可回收。

五、Runtime 执行策略：强控 vs 弱控

这个我建议你直接做成产品里的一级配置，不要只停留在概念。

强控模式

适用于：

广告诊断
数据归因排查
财务分析
风险问答
合规审批
对外回复
特征
固定链路
固定 Prompt 骨架
固定工具白名单
固定输出模板
指定审批点
结果可回放
适合的 Agent 结构

Planner 只做分流，Worker 按预设流程执行，Tool Agent 严格白名单。

优点
稳
好审计
好评测
易控风险
代价
灵活性弱
探索能力差
泛化不足
弱控模式

适用于：

通用 Copilot
内部知识助理
研究探索
多轮写作
方案头脑风暴
特征
目标驱动
Agent 可自主拆解任务
可动态选择工具
可多次 handoff
受 guardrails 限制

OpenAI 当前的 agent 指南也把 guardrails、handoffs、tooling 作为这类模式的关键运行机制。

优点
灵活
适应复杂开放任务
用户体验更自然
代价
更难测
更难审计
成本更高
风险更大
终版建议

企业里不要二选一，而是：

默认策略
高风险业务域：强控
低风险知识助手：弱控
中间地带：弱控外壳 + 强控内核

也就是：
用户可以自然对话，但一旦进入高风险动作，Runtime 自动切到强控子流程。

六、你可以直接拿去讲的“边界口诀”
1

身份归安全中心，执行归 Runtime，决策归 Planner，能力归 Tool Agent。

2

知识库决定能看什么，Agent 权限决定能做什么。

3

Prompt 管怎么说，Knowledge 管说什么，Runtime 管怎么跑，Eval 管跑得好不好。

4

高风险走强控，开放任务走弱控，所有越权都必须可审计。

七、最终决策建议

我建议你就按下面这个口径定版：

组织层
安全中心单独立项
Runtime 作为 AI 中台核心
Prompt / Eval / Observability 可逻辑分域、工程合台
Knowledge 单独成域
Model Gateway 独立基础设施
产品层
Agent 体系固定三层：Planner / Worker / Tool
所有 Agent 权限统一由安全中心发策略，不在 Agent 内硬编码
知识权限与 Agent 权限双轨治理
Runtime 内置强控 / 弱控执行模式切换
架构层
Runtime 兼容 MCP，但不等于 MCP
全链路 Trace 采用 OTel 思路做 span 级打点与回放
评测观测独立做裁判，不参与实时业务决策。
