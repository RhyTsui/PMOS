# 小乔智投需求场景 PRD V2.1 多角色二次评审报告

**评审对象**：小乔智投需求场景 PRD V2.1 完整版  
**评审视角**：游戏发行人员、公司高管、数据部负责人、产品专家、技术架构负责人  
**评审日期**：2026-05-25  
**评审目的**：基于战略评审、需求 V2 评审反馈及后续对话确认，判断 PRD V2.1 是否具备进入系统设计、前端设计和研发拆解的条件，并从多角色视角补充建设性意见。

---

## 1. 总体二次评审结论

基于当前 PRD V2.1，我的总体判断是：

> **V2.1 已经从“场景说明文档”明显升级为“可进入系统设计的上游约束文档”，但还需要在商业价值闭环、数据治理责任、工程边界、运营归属和验收指标上继续补强。**

这版最明显的进步是：它明确了一期不是展示 Agent，也不是自动投放，而是把投放支持问题转成可路由、可追问、可执行、可证据化、可沉淀、可回放、可继续处理的标准任务闭环，并把一期聚焦到三条业务闭环和四个公共底座。

一期核心闭环：

1. 数据与报表闭环。
2. 包交付与流程状态闭环。
3. 异常 / 需求 / 逻辑协同轻闭环。

公共底座：

1. 我的资产。
2. 自动化任务中心。
3. 知识库 / 口径库。
4. trace / evidence / case。

结论上，**V2.1 已经具备进入系统设计的资格，但不建议直接进入研发排期**。在进入研发排期前，建议补充一版 **PRD V2.2 轻量增强版**，不扩新场景，只补多角色背书和工程 contract。

---

## 2. 这版 PRD 的主要进步

### 2.1 MVP 分层更清楚

V2.1 将一期拆成三个层次：

| 层级 | 核心目标 | 主要能力 |
|---|---|---|
| MVP0 | 证明小乔不是聊天壳 | 统一会话、意图路由、slot 补齐、能力预检、真实查数、报表、定时报表、包状态、资产写入、evidence、trace、无工具假完成拦截 |
| MVP1 | 证明小乔能沉淀资产、追责、复用 | 包批次、新旧包替代、联调记录、指标口径卡、定时报表运行记录、资产详情、case fallback、异常初诊、从资产继续任务 |
| MVP2 | 证明小乔具备智能协同能力 | 需求单结构化、类似需求匹配、问逻辑可转任务、推荐动作、跨任务转接、自动化扩展、小闪通知 |

这个分层比 V2 版本“一期大而全”的表达更稳，也更适合资源排期。

### 2.2 反虚原则已经具备工程约束力

V2.1 明确了以下全局红线：

1. 未调用工具，不得输出“已完成”。
2. 无证据，不得输出强结论。
3. 工具失败，必须进入失败闭环或 case。
4. 内部事实必须来自内部工具、数据库、workflow、MCP 或内部知识库。
5. 外部联网只能补充，必须标注来源和时效。
6. 大模型只能做解释、总结、推理和表达，不得伪造事实。
7. 一期不自动执行高风险投放动作。
8. mock 必须是可迁移 mock contract，禁止一次性静态假链路。

这部分应该被提升为研发验收红线，而不是只作为产品理念。

### 2.3 项目上下文和权限规则更加成熟

V2.1 明确采用：

> **自由对话 + 项目上下文感知 + 权限动态过滤。**

这个判断是合理的。小乔是服务台，不应该强迫用户每次先选择项目、再进入模块、再提问。但系统必须记录 `project_refs[]`，并在展示历史会话、资产、自动化任务和运行结果时按当前权限动态过滤。

这解决了一个关键矛盾：

- 用户侧需要自由提问。
- 公司侧必须保证数据权限安全。
- 系统侧需要保留完整 trace，不因前台权限过滤破坏审计链路。

### 2.4 包交付从“联调工具”升级为“可投放结论”

V2.1 将包交付的主语定义为：

> 当前包是否可用于投放交付。

并区分：

- 媒体后台包；
- 官包；
- 智投分包；
- 流程阶段；
- 业务结论。

这比“联调页面”或“联调工具”更符合发行和投放人员的真实心智。用户真正关心的是：

- 能不能投；
- 卡在哪里；
- 当前版本是不是最新；
- 失败原因是什么；
- 有哪些证据；
- 下一步谁处理。

### 2.5 页面和验收用例开始落地

V2.1 已经明确会话页必须展示：

- 当前任务；
- 当前步骤；
- 缺失字段；
- 工具调用状态；
- 证据来源；
- 完成态；
- 下一步建议；
- 资产写入状态。

并补充了查数、查询结果保存、报表草稿、定时报表、权限变化、历史会话项目切换、媒体后台包、官包、联调失败、可交付投放等验收用例。

这说明文档已经从“需求方向”推进到了“可验收场景”。

# 4. 游戏发行人员视角评审

## 4.1 发行人员最关心什么

游戏发行 / 投放人员并不关心内部有多少 Agent，也不关心系统使用了什么架构。他们最关心的是：

- 今天数据有没有异常；
- ROI、消耗、新增、付费是否达标；
- 包能不能投；
- 新包是否已经替代旧包；
- 联调失败原因是什么；
- 报表能不能自动出；
- 出问题时下一步找谁；
- 结果是否能直接发给项目组或广告支持。

从这个视角看，V2.1 的方向是正确的，因为它把“查数—报表—定时报表—资产”和“包状态—上报—联调—可交付结论”都做成了明确流程。

## 4.2 发行人员会认可的点

### 4.2.1 一次性查询结果默认不写入资产

这符合实际使用。不是所有查数都值得归档。只有用户保存、转报表、转模板或转定时任务时，才生成资产，能够减少资产中心垃圾数据。

### 4.2.2 定时报表结果默认进入我的资产

日报、周报不是一次性问答，而是持续产物。每次运行结果不覆盖历史、生成独立记录，有利于复盘和追溯。

### 4.2.3 包交付采用业务结论，而不是只展示技术状态

V2.1 将包状态拆成“流程阶段”和“业务结论”：

| 流程阶段 | 业务结论 |
|---|---|
| 上报不通过 | 不建议交付 |
| 等待提审 | 待处理 |
| 等待媒体后台更新 | 待处理 |
| 联调不通过 | 风险待关注 |
| 联调成功 | 可交付投放 |
| 已被新包替代 | 历史包 |

这比单纯展示“联调成功 / 失败”更符合发行人员理解。

## 4.3 发行人员会担心的点

### 担心一：自由对话入口会不会太自由，导致结果不可控

一线用户经常会问得很模糊，例如：

- “昨天怎么样？”
- “这个包能上吗？”
- “ROI 不对，看看。”
- “日报给我一下。”

系统必须非常强地做上下文推断和追问，否则自由对话会变成“系统总是在问你指哪个项目”。

建议补充：

- 最近活跃项目优先推断；
- 当前会话项目优先；
- 单项目权限用户默认填充项目；
- 多项目权限用户需要确认；
- 用户连续追问时继承上一次 `project/media/platform/date_range`；
- 系统推断字段必须可见并可一键修改。

### 担心二：包交付状态对一线是否足够直观

“流程阶段 + 业务结论”是正确的，但发行人员更需要一句话结果：

- 能投；
- 暂时不能投；
- 等提审；
- 等后台更新；
- 联调失败，建议关注；
- 已被新包替代；
- 历史包，仅供追溯。

建议页面展示顺序调整为：

> 一句话结论 → 可操作按钮 → 风险说明 → 证据 → 技术流程节点。

不要让用户先看一堆流程状态。

### 担心三：异常初诊对投放人员仍偏弱

V2.1 把异常分为 A 类可自动初诊、B 类给候选原因、C 类转 case，这个边界正确。但对发行人员来说，异常初诊最需要的是“下一步怎么做”。

建议每个异常输出强制包含：

- 当前结论；
- 置信等级；
- 影响范围；
- 已检查项；
- 未检查项；
- 建议动作；
- 是否需要人工；
- 负责人或处理队列；
- 可一键创建 case / 查看数据 / 查包 / 查回传。

## 4.4 发行人员视角修改建议

建议在 PRD V2.2 中新增一章：**发行人员日常工作剧本**。

至少补 5 个剧本：

1. 早上看项目日报；
2. ROI 异常下钻；
3. 新包可投放确认；
4. 联调失败跟进；
5. 多项目周报复盘。

每个剧本写清楚：

- 用户自然语言；
- 系统推断上下文；
- 缺失字段；
- 工具调用；
- 输出结果；
- 一键下一步；
- 是否写入资产。

# 5. 公司高管视角评审

## 5.1 高管最关心什么

高管不会只看功能完整度，而会看：

- 是否值得投入；
- 能节省多少人力；
- 能降低多少跨部门沟通成本；
- 能提升多少投放响应效率；
- 能否沉淀组织资产；
- 是否可控、可审计、可追责；
- 是否能支撑未来数字优化师自动化。

从这个角度看，V2.1 方向是正确的。尤其文档反复强调一期不自动执行高风险投放动作、不替代业务系统做强制拦截、不做无证据强结论，而是通过证据、状态、case 和资产让任务可追溯、可复用、可继续。

## 5.2 高管会认可的点

### 5.2.1 边界清楚

一期明确不自动投放、不执行高风险动作，而是先做可信业务闭环。这个边界能避免预算、账户、投放策略等高风险域过早进入自动化。

### 5.2.2 资产化方向正确

“我的资产”不是聊天历史，而是业务资产中心，包括：

- 报表结果；
- 报表模板；
- 定时任务；
- 包交付资产；
- 排查 case；
- 需求单；
- 指标口径卡；
- 工具证据；
- 任务运行记录。

这意味着系统不是一次性问答工具，而是在沉淀组织知识和工作成果。

### 5.2.3 可追责机制初步具备

V2.1 要求 trace 在连弩平台完整回放，小乔侧记录：

- `trace_id`；
- `project_refs[]`；
- `tool_calls`；
- `evidence_refs`；
- `asset_write_status`。

这对高管是安全感来源：AI 做了什么、基于什么证据、写入了什么资产，都能回放。

## 5.3 高管会质疑的点

### 质疑一：成功标准还不够经营化

现在验收用例主要是功能验收，例如查数成功、保存资产、定时报表、权限变化、包状态展示等。高管还需要经营指标：

- 人均报表时间减少多少；
- 投放支持响应时间缩短多少；
- 查数 / 包状态 / 联调咨询减少多少人工沟通；
- 定时报表覆盖多少项目；
- 包交付风险提前发现多少次；
- case 平均处理周期是否下降；
- 资产复用率是否提升；
- 无证据结论是否被拦截。

### 质疑二：MVP0 范围仍不小

MVP0 已包含会话、路由、slot、能力预检、查数、报表、模板、定时报表、包状态、包地址、上报/联调状态、资产、evidence、trace 和无工具拦截。

高管可能会问：

> 如果只给 4-6 周，先交付哪三个闭环？如果资源砍半，哪些能力必须保留？

建议把 MVP0 再拆成：

| 子阶段 | 核心交付 |
|---|---|
| MVP0-A | 查数 + 报表草稿 + 证据 + trace |
| MVP0-B | 定时报表 + 资产 |
| MVP0-C | 包状态 + 包地址 + 上报/联调状态 |

### 质疑三：ROI 还没有表达

建议补充经营价值表：

| 收益项 | 当前成本 | 小乔目标 | 衡量方式 |
|---|---|---|---|
| 日报生成 | 每人每天 X 分钟 | 降低 50%-70% | 报表任务运行次数、人均节省时间 |
| 包状态沟通 | 多群反复确认 | 一处查看 | 包状态查询次数、人工咨询下降 |
| 联调失败追踪 | 截图 / 聊天记录分散 | case + 证据 | case 完整率、回溯时间 |
| 需求沟通 | 口头反复确认 | 结构化需求单 | 需求补充轮次下降 |
| 数据口径解释 | 反复问数据部 | 口径卡复用 | 口径卡访问 / 复用率 |

## 5.4 高管视角修改建议

建议在 PRD 前面新增一页：**一期业务收益与经营验收**。

核心指标建议设为：

- 北极星指标：**可证据化完成的投放支持任务数**。
- 经营指标：
  - 报表人工耗时下降；
  - 包状态人工咨询下降；
  - 异常问题首次有效响应时间下降；
  - 需求单一次结构化完整率；
  - 资产复用率；
  - trace 可回放率；
  - 无工具假完成率 = 0；
  - 无证据强结论率 = 0。

# 6. 数据部负责人视角评审

## 6.1 数据负责人最关心什么

数据部负责人会非常关注：

- 指标口径是否正确；
- 数据来源是否可追溯；
- 数据延迟是否可解释；
- 多数据源冲突如何处理；
- 权限是否严谨；
- 小乔是否会错误解释数据；
- 定时报表是否会把未刷新数据当成正式结果；
- AI 是否会绕过数据治理直接生成结论。

从这个角度看，V2.1 已经有较好的基础。数据与报表系统流程包括术语归一化、意图识别、slot 抽取、权限预检、能力预检、工具调用、结果校验、口径补充、异常提示、结构化结果、用户选择保存 / 转报表 / 转定时、资产写入、trace 上报和下一步推荐。

## 6.2 数据负责人会认可的点

### 6.2.1 内部口径优先

知识库策略明确：

- 指标口径规则是内部口径库优先；
- 外部联网只做补充；
- 内部口径缺失必须说明；
- 外部信息必须标注来源和时效。

这对数据治理非常重要。

### 6.2.2 定时报表必须确认数据延迟策略

创建定时报表时必须确认数据延迟策略，每次运行结果默认进入资产且不覆盖历史。这能防止“每天 9 点自动发，但数据 10 点才稳定”的口径事故。

### 6.2.3 权限动态过滤被纳入核心规则

会话、资产、自动化任务可涉及单项目或多项目，所有业务内容记录 `project_refs[]`，失去权限后相关内容前台不可见，不做物理删除而是动态不可见 / 脱敏隐藏。这符合数据权限治理方向。

## 6.3 数据负责人会担心的点

### 担心一：指标体系 contract 还没独立成章

当前文档强调口径，但还没有列出一期指标白名单、维度白名单和口径优先级。

建议补：

- 一期指标白名单：消耗、激活、新增、付费、ROI、ROAS、回本、留存、首充、注册、下载、转化率等；
- 一期维度白名单：项目、媒体、平台、终端、日期、账户、计划、包、渠道、素材；
- 时间窗口规则：昨天、今天、近 7 天、自然周、投放周、T+1、T+N；
- 数据源优先级：内部 BI、MMP、媒体、智投系统分别用于什么；
- 数据刷新频率；
- 数据延迟提示规则；
- 数据源冲突展示规则。

### 担心二：结果校验规则不够明确

系统流程中有“结果校验”，但没有展开校验标准。

建议补结果校验规则：

- 空结果校验；
- 异常值校验；
- 权限过滤后结果缺失校验；
- 指标 / 维度不兼容校验；
- 时间窗口超出数据范围校验；
- 多数据源不一致校验；
- 数据未刷新校验；
- 查询结果部分成功校验。

### 担心三：AI 解释和数据结论的边界还要更硬

V2.1 的证据等级设计很好，但建议进一步规定：

- 数据结论必须来自 L1；
- 口径结论必须来自 L2；
- 异常原因强结论必须有 L1 + L2；
- 只有 L3 / L4 时不能输出“原因是”，只能输出“可能是”；
- 生成式总结必须附带原始查询条件和 `evidence_refs`。

## 6.4 数据负责人视角修改建议

建议新增一章：**数据口径与质量治理要求**。

至少包含：

1. 指标白名单；
2. 维度白名单；
3. 数据源优先级；
4. 数据刷新和延迟策略；
5. 口径冲突处理；
6. 数据异常判断规则；
7. 部分成功定义；
8. 数据质量 case 生成规则；
9. 数据资产版本规则；
10. 报表运行结果的版本与归档规则。

没有这一章，数据部很可能不愿意背书查数结果。

# 7. 产品专家视角评审

## 7.1 产品专家最关心什么

产品专家会重点看：

- 用户价值是否清晰；
- 产品心智是否统一；
- 入口和闭环是否混乱；
- 任务状态是否闭环；
- 前端交互是否能承载复杂任务；
- 是否过度工程化；
- 是否能从一期演进到后续数字优化师；
- 是否能避免“普通 Chat + 几个按钮”的伪产品。

从这个角度看，V2.1 的产品框架已经比较正确：前台保留自由会话和任务入口，后台按三条闭环组织；知识库不是帮助中心，而是术语、口径、流程、失败原因和建议的支撑，不替代工具执行；资产不是聊天历史，而是可复用结果。

## 7.2 产品专家会认可的点

### 7.2.1 产品心智从“Agent 展示”转向“服务台闭环”

一期目标不是展示 Agent，也不是自动投放，而是把高频问题转成标准任务闭环。这是正确的产品心智。

### 7.2.2 六入口和三闭环关系更清楚

看数据、拼报表、查异常、验流程、提需求、问逻辑只是用户理解方式；后台才是数据与报表、包交付、异常 / 需求 / 逻辑协同。这样能避免前端做成六个割裂模块。

### 7.2.3 资产写入规则变得更产品化

V2.1 明确：

- 一次性查询用户保存后写入；
- 报表草稿默认不写入；
- 报表模板用户保存后写入；
- 定时报表运行结果默认写入；
- 包交付结果写入；
- case 和需求单生成后写入。

这个规则能避免资产中心变成垃圾桶。

## 7.3 产品专家会担心的点

### 担心一：自由会话和任务入口的关系还需要明确

前端需要解决：

- 用户是先选项目再提问，还是先提问再识别项目；
- 六类入口是快捷问题、任务模板，还是导航；
- 当用户从资产继续任务时，是否回到会话；
- 当任务执行中缺字段时，是聊天追问还是表单补齐；
- 多项目结果如何展示，不让用户混乱；
- 任务是否有右侧工作区。

建议采用：

> 左侧历史 / 资产入口 + 中央会话 + 右侧任务工作区。

中央会话负责自然语言和追问；右侧工作区负责表格、报表、包详情、证据、任务状态、资产写入。

### 担心二：问逻辑容易再次变成知识问答

V2.1 已经定义问逻辑是解释层 + 转任务层，方向正确。但建议把“可一键转任务”做成强产品规范：

- 问 ROI → 可转查 ROI 趋势；
- 问回传 → 可转检查回传；
- 问包状态 → 可转查包；
- 问联调失败 → 可转创建 case；
- 问日报 → 可转报表模板 / 定时报表；
- 问需求 → 可转需求单。

每个知识解释后至少给 1-3 个任务按钮，否则还是容易变成解释型 Chat。

### 担心三：异常 / 需求 / 逻辑轻闭环缺少用户感知优先级

用户不一定知道自己是在“查异常”还是“问逻辑”。例如：

> ROI 为什么掉了？

这既可能是问逻辑，也可能是查数据，也可能是异常初诊。

建议产品上统一成：

- 系统先给“我理解你是在排查 ROI 异常”；
- 展示已识别字段；
- 展示可执行动作；
- 让用户一键确认或修改。

不要让用户自己选择复杂入口。

## 7.4 产品专家视角修改建议

建议新增一章：**前台产品形态与任务交互范式**。

包含：

1. 首页首屏；
2. 六类入口的展示方式；
3. 会话与任务工作区关系；
4. 缺字段追问模式；
5. 工具调用状态展示；
6. 证据展示方式；
7. 下一步推荐按钮规范；
8. 资产写入提示；
9. 从资产继续任务；
10. 多项目结果展示；
11. 部分成功和失败 case 的 UI 表达。

# 8. 技术架构负责人视角评审

## 8.1 技术架构负责人最关心什么

技术架构负责人会重点看：

- 是否能统一架构承载三条闭环；
- 是否会出现多个业务链路重复造轮子；
- 权限动态过滤是否会破坏 trace 完整性；
- 工具调用、证据、资产、case 是否有统一 contract；
- mock contract 是否可迁移；
- 能力预检如何做；
- 多项目、多权限、多资产如何建模；
- 系统是否能观测、可回放、可追责；
- 后续是否能扩展到更复杂的 Agent 工作流。

从这个视角看，V2.1 已经给出了很多上游约束，但还需要继续 schema 化。

## 8.2 技术架构负责人会认可的点

### 8.2.1 权限过滤和 trace 完整性被分离

V2.1 明确 trace 在连弩平台完整回放，小乔前台权限过滤不破坏 trace 完整性。这是关键设计。否则前台动态隐藏数据后，后端审计链路可能断裂。

### 8.2.2 roleProfile 问题被识别

文档指出当前已有 designer、optimizer、observer 角色基础，optimizer 允许 report、diagnosis、debugging、delivery 等意图，observer 偏只读，但主聊天执行链路尚未完整消费 roleProfile。这个问题识别得很实际。

### 8.2.3 包交付派生规则适合工程实现

“可交付投放”默认由系统派生，条件包括：

- 包地址有效；
- 上报 / 数据检查通过；
- 联调检查通过；
- 当前包有效；
- 未被新包替代；
- 无阻断型失败 case；
- 用户有查看权限。

这比人工维护状态更可靠。

## 8.3 技术架构负责人会担心的点

### 担心一：统一任务状态机还不够 schema 化

V2.1 有完成态定义，但还需要映射到所有任务流。

建议定义统一 `ServiceTask`：

```text
task_id
conversation_id
user_id
role_profile
intent_type
task_type
project_refs[]
slots
missing_slots[]
permission_check_result
capability_preflight_result
tool_calls[]
evidence_refs[]
asset_refs[]
case_refs[]
status
completion_state
recommended_actions[]
trace_id
created_at
updated_at
```

所有查数、报表、包、联调、异常、需求、问逻辑都挂在这个任务模型下。

### 担心二：动态权限过滤复杂度很高

用户失去项目权限后，历史会话内容、资产内容、自动化结果前台不可见，但 trace 完整性不破坏。这个设计正确，但实现复杂。

需要明确：

- 前台隐藏是字段级、消息级还是资产级；
- 多项目报表中部分项目无权限时，是隐藏列、隐藏行，还是提示部分不可见；
- 资产标题是否包含项目敏感信息；
- trace 回放是否只有管理员可见；
- evidence 是否需要按项目切片；
- 定时报表运行结果跨项目时如何过滤。

### 担心三：包交付状态跨多个系统，状态一致性风险高

包流程涉及智投收到包、MCP 上报检查、媒体后台更新、智投后台更新、联调检查、case、资产写入。这不是单一系统状态，很容易出现不同步。

建议架构上采用事件化设计：

- PackageReceived；
- ReportCheckStarted；
- ReportCheckPassed；
- ReportCheckFailed；
- MediaBackendUpdated；
- ZhitouBackendUpdated；
- DebuggingStarted；
- DebuggingPassed；
- DebuggingFailed；
- PackageSuperseded；
- CaseCreated；
- AssetWritten。

然后由状态派生器生成“流程阶段 + 业务结论”。

### 担心四：能力预检没有明确 contract

MVP0 和系统流程多次提到能力预检，但架构需要知道预检检查什么：

- 用户权限；
- 项目权限；
- 工具是否可用；
- 数据源是否可用；
- slot 是否足够；
- 是否支持该指标 / 维度；
- 是否支持该媒体 / 平台；
- 是否满足联调前置条件；
- 是否存在高风险动作；
- 是否可以 fallback 到 case。

建议定义 `CapabilityPreflightResult`：

```text
can_execute
blocking_reasons[]
missing_slots[]
permission_gaps[]
unsupported_capabilities[]
risk_level
fallback_strategy
suggested_questions[]
```

建议在系统设计前补一份：**PRD V2.1 技术 contract 附录**。

包括：

2. Intent Routing Schema；
3. Slot Schema；
4. Permission Check Schema；
5. Capability Preflight Schema；
6. ToolCallResult Schema；
7. EvidenceRef Schema；

这份附录不是技术设计本身，但它是技术设计的入口。

# 10. PRD V2.2 修订建议

## 10.1 新增模块一：业务收益与北极星指标

建议将北极星指标定义为：

> **可证据化完成的投放支持任务数。**

一个任务只有同时满足以下条件才计入：

- 意图识别正确；
- 必要 slot 已补齐或明确缺失；
- 权限校验完成；

- 资产写入或明确无需写入；
- 有下一步建议或完成态；
- trace 可回放。

|---|---|

# 11. 按角色给出的最终评审意见

## 11.1 游戏发行人员意见

**可以接受，但需要更贴近日常工作。**

V2.1 已经能覆盖查数、日报、定时报表、包状态、联调失败、可投结论等高频工作。最大补强点是把三条闭环翻译成发行人员日常剧本，并在前端优先展示“一句话业务结论 + 下一步动作”，而不是先展示流程细节。

## 11.2 公司高管意见

**方向值得投入，但需要经营指标。**

V2.1 已经有风险边界、证据链、资产沉淀和 trace，能避免一上来做高风险自动投放。高管视角下仍需补充 ROI、人效、响应时效、事故减少、资产复用等经营指标，否则难以判断优先级和投入规模。

## 11.3 数据部负责人意见

**原则正确，但需要数据治理 contract。**

内部口径优先、外部补充标注来源、数据延迟策略、权限动态过滤都很好。下一步必须明确指标白名单、维度白名单、数据源优先级、刷新频率、口径冲突规则和结果校验标准，否则数据部难以对 AI 输出背书。

## 11.4 产品专家意见

**产品框架已成立，但要定义前台体验范式。**

三闭环、公共底座、资产中心、任务转接都正确。最大风险是前端做成“聊天框 + 几个按钮”。建议明确中央会话 + 右侧任务工作区 + 资产 / 历史入口的交互框架，并规范每类任务的下一步动作按钮。

## 11.5 技术架构负责人意见

**可以进入系统设计，但必须先补 contract 附录。**

V2.1 已经提供了足够多的系统约束，包括权限动态过滤、trace / evidence / case、包双层状态、完成态、证据等级和验收用例。但研发前必须 schema 化，否则三条闭环容易分裂实现。重点补 Unified Task、Permission、Preflight、ToolCallResult、Evidence、Asset、Case、Package Event、Automation Run 等 contract。

# 12. 最终结论

这版 PRD V2.1 **已经具备进入系统设计的资格**，但还不建议直接进入研发排期。

更准确地说：

> V2.1 已经解决了“需求是否正确”的问题；  
> 接下来要解决“多角色是否愿意背书”和“工程是否能一致实现”的问题。

建议下一步进入 **PRD V2.2 轻量增强版**，不再扩展新场景，只补六个附件：

1. 业务收益与北极星指标；
2. 发行人员日常工作剧本；
3. 数据口径与质量治理；
4. 前台交互范式；
5. 技术 contract 附录；
6. 上线验收矩阵。

一句话评价：

> **V2.1 已经从“AI 功能清单”升级成“可信投放服务台需求文档”；V2.2 要把它进一步升级成“高管能投、业务能用、数据能背书、产品能设计、技术能落地”的跨角色共识文档。**
