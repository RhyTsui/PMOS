智投 Chat 完整版服务重构文档
从混乱 Agent 路由升级为“后台可配置的任务路由、Skill、Workflow 与观测上报服务”

版本：v1.0
适用范围：智投 Chat / 广告归因 / 数据统计分析 / 联调排查 / 投放包检查 / 需求协同
执行对象：Codex / 开发团队
目标形态：完整可扩展服务，不是 demo

1. 本次重构的核心结论

当前系统的问题不是“某个 Agent 识别错了”，而是服务架构混乱：

关键词 / 正则
  -> Agent
    -> MCP / Workflow / Prompt / UI

这会导致：

路由漂移；
双路由冲突；
Agent 职责膨胀；
Skill、Prompt、Workflow 无法后台调整；
MCP 调用分散；
缺少统一任务模型；
缺少质量检查；
缺少观测上报规范；
自动化测试容易被缺 token、缺权限、缺数据源阻塞。

本次重构目标是升级为：

User Request
  -> Task Intent
  -> Slot Extract
  -> Skill Contract
  -> Workflow / Tool Plan
  -> MCP / Data Service / Model
  -> Quality Check
  -> Result Renderer
  -> Trace Reporter 上报连弩观测

核心原则：

Agent 不是第一路由对象
Skill 不是写死在代码里的关键词逻辑
Prompt 不是固定在前端或 API route 里的字符串
Workflow 既支持固定流程，也支持灵活 Agentic Plan
Trace 不是 Chat 自建，而是按连弩观测规格上报

3. 核心概念重新定义
3.1 Domain：业务领域

Domain 表示业务系统所属领域，不等于用户意图。

P0 建议覆盖：

type Domain =
  | "ads_delivery"
  | "attribution"
  | "report"
  | "monitoring"
  | "integration"
  | "package_check"
  | "game_behavior"
  | "data_pipeline"
  | "metric_dictionary"
  | "demand_collaboration";

示例：

ads_delivery       广告投放
attribution        广告归因
report             数据报表
monitoring         监测与回传
integration        媒体联调
package_check      包检查
game_behavior      游戏行为数据
data_pipeline      上报、预处理、指标计算链路
3.2 Task Intent：用户目标动作

Task Intent 表示用户想做什么。

P0 支持：

type TaskIntent =
  | "ask_data"              // 查数
  | "analyze_anomaly"       // 查异常 / 解释波动
  | "debug_integration"     // 联调 / 回传 / 监测排查
  | "delivery_package"      // 投放包 / 包检查 / 媒体配置
  | "metric_explain"        // 指标解释 / 口径说明
  | "create_demand";        // 需求沟通协同 / 需求记录

P1 扩展：

type P1TaskIntent =
  | "scheduled_report"
  | "forecast"
  | "creative_analysis"
  | "data_pipeline_check"
  | "attribution_diagnosis"
  | "media_callback_check"
  | "game_event_check";
3.3 Skill：后台可配置的业务能力

Skill 是系统的可复用业务能力，不应该只写死在代码里。

例如：

report_query_skill
roi_anomaly_analysis_skill
attribution_diagnosis_skill
media_callback_debug_skill
package_check_skill
game_event_check_skill
metric_explain_skill
demand_capture_skill

Skill 应该支持后台配置：

名称；
适用 Task Intent；
适用 Domain；
必填 Slot；
可选 Slot；
关联 Prompt；
关联 Workflow；
关联 MCP / Tool；
输出 Schema；
质量检查规则；
是否启用；
优先级；
版本；
灰度策略；
回滚策略。
3.4 Workflow：固定流程 + 灵活流程

广告归因和数据统计分析不是一个简单函数调用，必须支持复杂排查过程。

Workflow 分三类：

固定 Workflow

适合强业务流程。

例如：

查数
  -> 时间补全
  -> 项目补全
  -> 指标口径确认
  -> 报表查询
  -> 质量检查
  -> 异常检测
  -> 输出结果
灵活 Agentic Workflow

适合复杂排查。

例如：

ROI 异常
  -> 先查 ROI 趋势
  -> 如果 ROI 掉，查消耗、激活、付费、回收
  -> 如果付费掉，查游戏行为事件
  -> 如果激活异常，查点击、曝光、归因、回传
  -> 如果媒体数据异常，查监测回传和媒体接口
混合 Workflow

建议主要使用混合模式：

固定主流程 + 每个阶段允许 Agent 动态选择工具

这适合你当前业务。

3.5 Agent：交互人格与协调器

Agent 不再作为第一层路由对象。

Agent 的职责是：

协调多轮上下文；
根据 Workflow 阶段生成解释；
决定是否追问；
选择结果展示方式；
在复杂排查时做动态计划；
作为 UI 上的“助手角色”出现。

Agent 不负责：

直接根据关键词选 MCP；
直接吞掉所有业务逻辑；
作为所有能力的唯一承载体。
3.6 MCP：工具接口

MCP 是标准化工具接口。

例如：

report_mcp
attribution_mcp
monitoring_mcp
media_callback_mcp
package_check_mcp
game_event_mcp
demand_mcp
feishu_mcp

MCP 不应该直接由关键词选择。

错误：

if (query.includes("ROI")) {
  selectMcpTool("forecast");
}

正确：

Task Intent -> Skill -> Workflow / Tool Plan -> MCP
3.7 Trace Reporter：观测上报适配器

Chat 不自建 Trace 服务。

Chat 只做：

把运行过程转成连弩观测 SDK / OTLP 需要的 Root Span、Agent Span、Model Span、Tool Span，并上报

连弩观测规格要求 root span 使用 cozeloop.span_type=custom，model span 使用 model，tool span 使用 tool，有 Agent 编排链路时还需要上报 agent span；只上报 root span 无法支撑模型、工具、token、TTFT、错误分布等统计。

5.3 低置信度策略

触发任一条件时不硬路由：

1. top1 score < 0.55
2. top1 与 top2 分差 < 0.08
3. 缺少必填 slot
4. 当前输入和历史上下文冲突
5. 同时命中多个任务
6. 工具权限未知
7. 媒体词、指标词、动作词产生交叉歧义

示例：

用户：ROI 怎么样？

不要直接路由 forecast。

应该返回：
你想看的是：
1. 查看 ROI 数据
2. 分析 ROI 是否异常
3. 查看 ROI 后续预测
5.4 典型误路由修复目标
“巨量应用列表”

禁止：

命中“巨量” -> debug_integration

应该：

候选：
1. ask_data
2. metric_explain
3. debug_integration

如无上下文，弹确认：
你是想查看巨量平台应用列表，还是排查巨量联调？
“ROI 怎么样”

禁止：

ROI -> forecast/prediction

应该：

候选：
1. ask_data
2. analyze_anomaly
3. forecast，P1 或确认选项
“异常波动”

禁止：

在 monitor / diagnosis / report query 之间随机漂移

应该：

Task Intent: analyze_anomaly
如果缺指标、项目、时间，追问或继承上下文。
6. 复杂广告归因与数据分析 Workflow 设计

你这套业务不能只做“查报表工具 + 总结”这种 demo。
必须把广告链路拆成可排查的业务阶段。

6.1 广告数据链路
点击数据上报
  -> 数据预处理
  -> 去重 / 清洗 / 有效性校验
  -> 归因
  -> 指标计算
  -> 报表调度
  -> 页面展示
  -> 筛选 / 汇总 / 细分
6.2 游戏行为数据链路
游戏行为数据上报
  -> SDK 接收
  -> 数据预处理
  -> 事件标准化
  -> 自动化检查
  -> 指标计算
  -> 归因关联
  -> 报表展示
6.3 媒体监测回传链路
监测链接生成
  -> 点击 / 展示 / 激活 / 付费事件采集
  -> 监测回传
  -> 媒体接口回调
  -> 回传结果校验
  -> 媒体侧数据对账
6.4 包检查与联调链路
投放包上传
  -> 包参数检查
  -> SDK 配置检查
  -> 事件埋点检查
  -> 媒体参数检查
  -> 联调请求检查
  -> 回传检查
  -> 问题归因
  -> 修复建议
7. P0 Workflow 清单
7.1 查数 Workflow：report_query_workflow
用户问数
  -> 抽取 project/media/dateRange/metrics/dimensions
  -> 缺 slot 时追问或继承上下文
  -> 指标口径确认
  -> 调 report_mcp
  -> 标准化 ReportQueryResult
  -> 执行 QualityCheck
  -> 如果发现异常，生成 anomaly hints
  -> 返回数据 + 解释 + 下一步动作

P0 QualityCheck：

1. 空返回
2. 必填指标缺失
3. 日期不连续
4. 今日数据延迟
5. 环比异常
6. 同比异常
7. 口径不确定
8. tool 报错
9. 权限失效
10. 数据源缺失
7.2 异常分析 Workflow：anomaly_analysis_workflow
用户问异常
  -> 明确异常对象：指标 / 项目 / 媒体 / 时间
  -> 查询基准数据
  -> 查询当前数据
  -> 做环比 / 同比 / 基线比较
  -> 分析消耗、点击、激活、付费、回收、ROI
  -> 如归因异常，查 attribution_mcp
  -> 如回传异常，查 monitoring_mcp
  -> 如游戏行为异常，查 game_event_mcp
  -> 如媒体侧异常，查 media_mcp
  -> 输出异常定位、证据、置信度、下一步
7.3 联调排查 Workflow：integration_debug_workflow
用户说联调 / 回传 / 监测异常
  -> 抽取 media/app/package/event/dateRange
  -> 检查包参数
  -> 检查监测链接
  -> 检查事件上报
  -> 检查归因结果
  -> 检查媒体回传状态
  -> 检查错误码
  -> 输出问题阶段、证据、修复建议
7.4 投放包检查 Workflow：delivery_package_workflow
用户问投放包 / 包检查 / 应用配置
  -> 抽取 app/package/media/channel
  -> 检查包信息
  -> 检查 SDK 配置
  -> 检查媒体参数
  -> 检查事件埋点
  -> 输出检查项、风险项、修复动作
7.5 指标解释 Workflow：metric_explain_workflow
用户问指标含义
  -> 抽取 metric
  -> 查 metric_dictionary
  -> 查业务口径
  -> 如果指标和媒体/项目有关，补充差异说明
  -> 输出定义、计算公式、数据来源、注意事项
7.6 需求沟通协同 Workflow：demand_capture_workflow
用户提出需求
  -> 抽取需求目标
  -> 抽取涉及业务域
  -> 抽取期望动作
  -> 抽取优先级和时间
  -> 生成需求记录
  -> 可选同步飞书 / 工单

P0 只做需求记录和协同，不自动创建定时报表。

9. Trace 上报设计：只上报，不自建
9.1 设计原则

本系统不做完整 Trace 服务。

只新增：

Trace Reporter / Observability Adapter

职责：

1. 接收 Chat Runtime 的运行事件
2. 转换为连弩观测 SDK / OTLP 需要的 span
3. 上报 root/model/tool/agent span
4. 不替代连弩观测平台
5. 不在 Chat 内建设完整 Trace 查询、统计、跳转能力

9.2 Trace 结构
Root Span                         cozeloop.span_type=custom
  ├─ Agent Span                    cozeloop.span_type=agent
  │   ├─ Model Span                cozeloop.span_type=model
  │   ├─ Tool / MCP Span           cozeloop.span_type=tool
  │   └─ Retriever Span            cozeloop.span_type=retriever，可选

说明：

1. 一次用户请求一个 Root Span。
2. 路由、计划生成、Agent 编排可以放 Agent Span。
3. 每次真实模型调用必须上报 Model Span。
4. 每次真实 MCP / API / Tool 调用必须上报 Tool Span。
5. 不要把 root 或 agent 调度伪装成 tool。

成功：cozeloop.status_code = 0
失败：cozeloop.status_code != 0

失败 output：

  "error_code": "TOOL_TIMEOUT",

连弩观测统计错误率只看数值型 cozeloop.status_code，output.error_code、error_stage、error_message 只用于 Trace 详情展示和人工排查。

18. 最终判断

这版的架构目标应该改成：

后台可配置的 AI 业务执行服务

复杂广告归因 / 数据统计 / 联调排查 Workflow

标准化 MCP Tool Registry

+
