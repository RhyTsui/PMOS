# 小乔智投需求场景 PRD V2 评审报告

**评审对象**：文档二《小乔智投需求场景 PRD V2》  
**评审视角**：B 端 AI 产品战略、需求可实施性、工程落地、业务闭环与反虚能力  
**评审结论**：方向正确，可以进入系统设计讨论；但不建议直接进入研发排期。下一步应补齐对象模型、状态机、权限矩阵、证据等级、验收用例和页面级交互 contract。  
**建议版本目标**：PRD V2.1 = 对象模型 + 状态机 + 验收用例版。

---

## 1. 总体判断

这版 PRD 相比旧方案有明显进步。它没有继续沿用“十大需求 / 五大主链路”的旧骨架，而是按“三条业务闭环 + 两个公共底座”组织一期需求，并保留六类前台任务入口：看数据、拼报表、查异常、验流程、提需求、问逻辑。

这说明需求层已经基本承接了前一阶段战略判断：小乔智投一期不是 Agent 展示平台，也不是自动投放系统，而是面向投放、发行、运营、数据和广告支持团队的 AI 投放服务台与任务运行工作台。

但是，从上层视角看，这版 PRD 仍然存在一个核心风险：

> 战略上已经反虚了，但需求层还没有完全防止实施阶段再次变虚。

文档已经写了“工具调用、证据、资产、trace、case、状态、失败闭环”，但很多内容仍停留在需求描述层，没有转成足够硬的产品 contract。研发如果按当前文档直接排期，仍可能做出一个“入口完整、页面完整、但细节大量依赖 fallback 和自然语言兜底”的版本。

因此，本评审的核心建议是：下一步不要继续扩场景，而要把现有三条闭环压实为可实现、可测试、可追责的对象、状态、权限、证据和验收体系。

## 3. 最值得肯定的改进

### 3.1 “入口不是链路”的判断非常关键

文档明确六类前台入口只是用户理解系统的方式，不是六条独立工程链路。这是非常重要的判断。

更合理的产品逻辑应该是：

> 前台是用户语言，后台是业务对象，系统中台是任务状态机。

例如：

- 用户说“看数据”，后台不应该只是进入一个“看数据 Agent”，而应该抽取项目、时间、媒体、指标、维度、权限，再执行查询、解释口径、沉淀资产。
- 用户说“ROI 掉了”，后台可能同时触发数据查询、异常初诊、指标解释、case 创建。
- 用户说“这个包能不能投”，后台要围绕包资产、包批次、上报检查、联调证据和可投结论组织闭环。

这版 PRD 已经开始按这个方向走，方向正确。

### 3.2 知识库使用策略明显成熟

文档把知识库拆成三层使用方式：

1. 路由前术语归一化；
2. 意图候选后定向检索；
3. 任务内证据检索。

尤其是“路由前只查受控词典和轻量索引，不允许大范围 RAG 直接影响路由”这一点非常关键。

很多 B 端 AI 产品失败在 RAG 前置污染意图：用户问一个业务问题，系统先召回一堆相似知识文本，然后被文本带偏，最后既没有正确路由，也没有真实执行工具。这版文档的策略更接近企业级 AI 服务台的正确做法：

> 先识别用户要完成什么任务，再在任务范围内查知识，而不是让知识库决定用户意图。

### 3.3 数据与报表闭环已经有比较完整的任务结构

自然语言查数场景已经包含用户问题、入口表达、补齐字段、能力调用、成功结果、失败闭环、资产沉淀、前后端影响和验收指标。

这说明需求已经从“AI 帮我查一下”进入到“查数任务 contract”的思路。尤其“无工具假完成比例趋近 0”这个指标非常好，建议提升为所有闭环的通用红线。

### 3.4 包交付方向纠偏非常正确

文档把包交付的主语从“联调”改成“包是否可投放”，并明确可交付投放包不是单纯下载地址，而是包含项目、包 ID、包版本、包批次、平台、媒体、下载地址、上报检查状态、联调状态、可交付状态、失败原因和证据记录的业务资产。

这个纠偏非常重要。旧逻辑如果围绕“联调工具”展开，用户会觉得系统只是一个技术工具；现在围绕“可交付投放包资产”展开，用户看到的是业务结果：

- 这个包现在能不能投？
- 如果不能，卡在哪里？
- 证据是什么？
- 下一步谁处理？
- 新包出来后旧包是否还有效？

这才是投放、发行和广告支持团队真正关心的事情。

### 3.5 “我的资产”定位基本正确

文档明确“我的资产是业务资产中心，不是聊天历史中心”，并把报表资产、定时任务资产、包交付资产、排查 case、需求单、指标口径卡、工具调用证据和任务运行记录都纳入一期资产范围。

如果小乔只保留聊天记录，它就是 Chat；如果它沉淀报表、包状态、排查结论、需求单、证据和运行记录，它才是业务服务台。

## 4. 当前 PRD 最大的问题

### 4.1 这是场景 PRD，还不是可实施 PRD

当前每个场景都按类似模板写了用户问题、入口表达、补齐字段、能力调用、成功结果、失败闭环、资产沉淀、前后端影响和验收指标。这个结构很好，但它仍然是场景层说明。

真正能交给设计和研发的 PRD，还必须补四类内容。

#### 第一，业务对象模型

至少要定义以下对象：

- ReportQuery；
- ReportDraft；
- ReportTemplate；
- ScheduledReportTask；
- ScheduledReportRun；
- DeliveryPackage；
- PackageBatch；
- PackageCheckRun；
- DebuggingRun；
- EvidenceRef；
- Asset；
- DiagnosisCase；
- RequirementTicket；
- KnowledgeCard；
- RecommendedAction；
- TraceEvent。

当前文档中提到了很多对象名，但大多只有名称，没有字段、状态、主外键关系、生命周期、创建时机、更新规则、归档规则和权限规则。

#### 第二，状态机

以包交付为例，文档写了等待出包、上报检查中、上报不通过、联调检查中、联调不通过、联调成功、可交付投放、已被新包替代等状态。但还缺少：

- 谁触发状态变化；
- 什么条件允许进入下一个状态；
- 是否允许回退；
- 是否允许跳过；
- 状态冲突怎么办；
- 多个检查批次怎么处理；
- 新包生成后旧包何时失效；
- 手工确认和系统自动判断哪个优先；
- 状态更新时间来自哪里；
- 状态是否有 SLA。

没有这些，研发实现时很容易各做各的。

#### 第三，页面级交互

文档写了前端需要支持六类任务入口、当前任务进度、缺失字段追问、工具调用状态、证据和来源、推荐下一步、资产入口、定时报表入口、包交付状态、联调记录、需求单预览等能力。

但还没有回答：

- 用户首次进入看到什么？
- 六类任务入口是卡片、快捷问题还是命令入口？
- 任务过程中进度如何展示？
- 字段追问是在聊天内追问，还是右侧表单补齐？
- 报表结果在聊天气泡里展示，还是右侧结果面板展示？
- 资产写入成功后如何提示？
- 失败 case 是否可点击进入详情？
- 包详情页是否独立存在？
- 我的资产是左下角入口，还是结果中也有资产抽屉？

这些不补，设计阶段会产生大量分歧。

#### 第四，验收用例

当前验收指标偏方向性，例如查询成功率、缺字段识别准确率、报表生成成功率、可投结论有证据占比、无证据强结论拦截率等。这些指标方向正确，但还不够验收。

建议每个场景至少补 3 条 Given / When / Then 用例。例如：

> Given 用户有项目 A 的数据权限；  
> When 用户输入“昨天 A 项目安卓消耗和 ROI 怎么样”；  
> Then 系统应识别为 report_query，补齐时间为昨日，终端为 Android，调用 query_metrics 工具，返回消耗、ROI、查询条件、数据来源、口径说明，并写入查询资产。

这类用例才是研发、测试和评测都能使用的 PRD 内容。

### 4.2 一期范围仍然偏大

当前一期范围包括自然语言查数、指标口径解释、拼报表、定时报表、获取可用包、上报检查、联调检查、单独触发联调、异常初诊、结构化需求单、问逻辑、我的资产、低风险自动化任务中心、trace、evidence、case fallback、术语归一化、候选意图定向检索和任务内知识检索。

这些都重要，但如果一期全部并行，很容易“每条都做一点，最后每条都不够真实”。

建议重新切成三个层级。

#### MVP0：必须真实可用的反虚闭环

只做：

- 自然语言查数；
- 报表草稿；
- 报表模板；
- 定时报表；
- 包状态查询；
- 包下载地址；
- 上报/联调状态读取；
- 我的资产写入；
- 工具调用 trace；
- 无工具假完成拦截。

目标是证明：小乔能真实执行，不是聊天壳。

#### MVP1：完整业务闭环

增加：

- 指标口径卡；
- 报表模板编辑；
- 定时报表运行记录；
- 包批次管理；
- 包详情证据；
- 联调记录、截图、日志；
- 异常初诊；
- 失败 case；
- 从资产继续任务。

目标是证明：小乔能沉淀资产，能追责，能复用。

#### MVP2：智能协同闭环

增加：

- 问逻辑可转任务；
- 需求单结构化；
- 类似需求匹配；
- 推荐动作；
- 跨闭环任务转接；
- 多轮澄清策略优化；
- 知识库定向检索增强。

目标是证明：小乔不只是执行工具，而是业务协同助手。

### 4.3 数据与报表闭环缺少指标体系 contract

数据与报表是一期开局主战场，但现在还缺少一个关键 contract：指标、维度、口径、权限、数据源和刷新频率的统一定义。

建议补充以下内容：

- 一期支持哪些指标：消耗、激活、新增、付费、ROI、ROAS、留存、首充、回本等；
- 一期支持哪些维度：项目、媒体、终端、日期、包、渠道、账户、计划、素材等；
- “昨天”“近 7 天”“本周”“自然周”“投放日”如何解析；
- 内部 BI 口径、媒体口径、MMP 口径冲突时谁优先；
- 每个数据源的刷新频率和数据延迟提示；
- 项目权限、媒体权限、角色权限、跨项目查询限制；
- 无权限、无数据、数据源异常、字段冲突分别如何提示；
- 异常判断是环比、同比、阈值，还是模型检测；
- 报表模板能否跨项目复用，字段是否允许用户改；
- 定时任务的数据延迟如何处理。

B 端数据产品的核心不是“查到了”，而是：

> 查得准、口径对、来源清、权限对、延迟可解释。

### 4.4 包交付闭环写得好，但状态机还不够严谨

包交付是当前 PRD 里业务对象定义最好的部分。尤其“包批次：首包 / 新包 / 修复包”和“旧包标记为当前有效、已被新包替代或历史可追溯”的设计很有必要。

但需要补一个完整状态流转表。

| 当前状态 | 触发事件 | 前置条件 | 下一个状态 | 是否自动 | 证据要求 | 失败处理 |
|---|---|---|---|---|---|---|
| 等待出包 | 包生成完成 | 项目存在、平台明确 | 上报检查中 | 自动/人工 | 包地址 | 无包 case |
| 上报检查中 | 上报检查完成 | 有检查结果 | 上报不通过 / 联调检查中 | 自动 | 检查日志 | 记录失败原因 |
| 上报不通过 | 修复包上传 | 修复包生成 | 上报检查中 | 自动 | 修复包地址 | 旧包保留 |
| 联调检查中 | 联调完成 | 上报通过 | 联调不通过 / 可交付投放 | 自动/人工 | 联调记录 | 生成 case |
| 联调不通过 | 重新联调 | 参数补齐 | 联调检查中 | 人工触发 | 补充信息 | 失败记录 |
| 可交付投放 | 新包生成 | 新批次创建 | 已被新包替代 / 历史可追溯 | 自动 | 新包记录 | 展示旧包状态 |

同时建议增加两条硬规则。

#### 可投结论必须是派生状态

“可交付投放”最好不是独立随便写入的状态，而是由以下条件派生：

- 包地址有效；
- 上报检查通过；
- 联调检查通过；
- 当前批次有效；
- 未被新包替代；
- 无阻断型失败 case；
- 用户有查看权限。

这样可以防止状态不一致。

#### 新包与旧包关系要有明确主键

建议模型上区分：

- package_id：包实体；
- package_batch_id：包批次；
- project_id：项目；
- version_code / version_name：版本；
- platform：Android / iOS；
- media：媒体；
- delivery_status：交付状态；
- superseded_by_batch_id：被哪个新批次替代；
- effective_status：当前有效 / 已替代 / 历史可追溯。

否则后续版本更新时，前台会乱。

### 4.5 异常 / 需求 / 逻辑闭环需要重构

文档第 6 章是最需要修的部分。异常初诊写得较完整，但“需求沟通与结构化需求单”和“问逻辑”部分存在结构断裂：场景 J 的成功结果不完整，资产沉淀下面混入验收指标，后续出现疑似标题丢失，问逻辑缺少完整标题和前后端影响描述。

这不是小问题。异常、需求、逻辑是最容易变成“AI 泛答”的区域。如果 PRD 本身结构不硬，实施时一定会虚。

建议第 6 章拆成三个清晰场景。

#### 场景 I：异常初诊

目标：把模糊异常转成诊断记录或 case。

输出物：

- DiagnosisCase；
- DiagnosisStep；
- EvidenceRef；
- NextAction；
- CaseStatus。

#### 场景 J：需求结构化

目标：把模糊诉求转成需求单。

输出物：

- RequirementTicket；
- RequirementSlot；
- AcceptanceCriteria；
- Dependency；
- Priority；
- RiskBoundary。

#### 场景 K：问逻辑与可执行推荐

目标：解释指标、流程、规则，并判断是否转任务。

输出物：

- KnowledgeCard；
- SourceRef；
- Uncertainty；
- RecommendedAction；
- TaskTransferRecord。

同时明确：问逻辑不是一个独立万能问答模块，而是解释层 + 转任务层。

## 5. 逐模块建设性建议

### 5.1 数据与报表闭环：补查询任务生命周期

建议查数任务统一生命周期如下：

1. 用户输入；
2. 术语归一化；
3. 意图识别；
4. slot 抽取；
5. slot 缺失判断；
6. 权限预检；
7. 能力预检；
8. 查询执行；
9. 结果校验；
10. 口径补充；
11. 异常提示；
12. 结果展示；
13. 资产写入；
14. 推荐下一步；
15. 可转报表 / 可转定时。

尤其要把“查询执行”和“结果校验”分开。工具返回结果不代表可以直接展示，系统还要判断是否为空、是否异常、是否口径冲突、是否维度不完整、是否查询条件被默认补齐、是否需要提示数据延迟。

建议 ReportQueryResult 至少包含以下字段：

```text
query_id
intent_type
project_id
project_name
media
platform
date_range
metrics[]
dimensions[]
filters[]
data_source
data_freshness
permission_status
result_table
metric_definitions[]
evidence_refs[]
warnings[]
recommended_actions[]
asset_write_status
trace_id
```

### 5.2 拼报表：明确报表草稿、模板和资产的区别

建议明确以下对象关系：

| 对象 | 含义 |
|---|---|
| ReportDraft | 一次性生成的临时报表结果，可编辑 |
| ReportTemplate | 可复用的报表配置，不一定有当期数据 |
| ReportAsset | 已归档的报表结果，代表某次实际产出 |
| ScheduledReportTask | 按模板周期执行的任务 |
| ScheduledReportRun | 某次定时执行记录 |

这样用户说“以后每天生成这个日报”时，系统才知道是基于当前结果、当前查询条件，还是当前报表模板创建任务。

### 5.3 定时报表：补时间窗口和数据延迟规则

定时报表是 P0，但它最容易出问题的不是创建任务，而是每天生成的数据是否可信。

例如用户说“每天 9 点生成昨天日报”，系统需要知道：

- “昨天”是自然日，还是投放日；
- 媒体数据 9 点是否已完整；
- 如果数据源 10 点才稳定，9 点生成是否标记“数据未完全刷新”；
- 如果任务失败，是否自动重试；
- 重试几次；
- 失败后是否通知；
- 历史结果是否覆盖还是新建版本；
- 用户修改模板后，旧运行结果是否保留。

建议 ScheduledReportTask 状态包括：draft、active、paused、running、failed、expired、archived。  
建议 ScheduledReportRun 状态包括：pending、running、success、partial_success、failed、skipped。

### 5.4 包交付：把单独触发联调归为中风险服务动作

单独触发联调不是高风险投放执行，但也不是纯查询。它会触发 workflow，可能影响广告支持、包状态、联调资源和后续资产记录。

建议动作分级：

| 动作 | 风险级别 | 一期是否允许 | 权限要求 |
|---|---|---|---|
| 查看包状态 | 低 | 允许 | 项目权限 |
| 查看下载地址 | 低/中 | 允许 | 包权限 |
| 查看联调记录 | 低 | 允许 | 项目权限 |
| 触发重新联调 | 中 | 允许但需权限 | 联调触发权限 |
| 修改包状态 | 中/高 | 谨慎 | 管理权限 |
| 跳过检查直接可投 | 高 | 不允许 | 不允许 |
| 自动投放该包 | 高 | 一期不允许 | 不允许 |

这能防止“低风险自动化中心不做高风险执行”的原则被联调触发类动作悄悄突破。

### 5.5 异常初诊：分级，不要承诺统一诊断所有异常

建议把异常分成三类。

**A 类：系统可自动初诊**

- 数据源无数据；
- 权限不足；
- 查询条件错误；
- 包状态不通过；
- 联调失败；
- 定时报表任务失败；
- 指标口径不匹配。

这类可以给出明确原因。

**B 类：系统可给候选原因**

- ROI 下降；
- 消耗异常；
- 新增异常；
- 付费异常；
- 回传延迟；
- 平台数据与内部数据不一致。

这类应输出候选原因 + 证据 + 待补证据。

**C 类：系统只能转 case**

- 平台模型波动；
- 媒体侧异常；
- 研发埋点疑似错误；
- 复杂归因争议；
- 跨团队业务策略问题。

这类不要硬诊断，应该创建 case。

### 5.6 需求单：做成业务分析能力，而不是表单生成

“提需求”场景不能只是把用户说的话整理成表单。它必须先判断：

- 是否已有能力可直接满足；
- 是否已有模板可复用；
- 是否是配置问题；
- 是否是权限问题；
- 是否是真正新需求；
- 是否应该转 case。

需求单还必须生成验收标准：

- 输入是什么；
- 输出是什么；
- 成功条件是什么；
- 数据来源是什么；
- 权限边界是什么；
- 不做什么；
- 异常怎么处理。

否则需求单只是换一种形式的口头需求。

### 5.7 问逻辑：做成解释卡 + 可执行动作

问逻辑是高价值入口，但也是最容易变虚的入口。建议所有问逻辑输出统一结构：

```text
结论
适用范围
内部口径 / 流程依据
外部补充信息，若有
不确定性
你可以继续做什么
可一键转任务
```

例如用户问“回传延迟为什么影响模型？”，系统不要只解释概念，还应该推荐：

- 查看当前项目回传延迟；
- 查询过去 7 天回传完整度；
- 检查媒体侧回传状态；
- 生成回传异常 case；
- 查看相关指标口径卡。

这样问逻辑才是任务转接层，而不是知识库问答。

## 6. 需要新增的上层产品约束

### 6.1 增加完成态定义

每个任务必须明确什么时候算完成。

| 完成态 | 含义 | 是否算成功 |
|---|---|---|
| completed | 工具执行成功，结果有证据，资产已写入或明确无需写入 | 是 |
| partial_completed | 部分结果成功，有缺口但已说明 | 部分成功 |
| pending_user_input | 缺字段，需要用户补充 | 否 |
| failed_case_created | 无法完成，但已生成 case 并记录原因 | 失败闭环成功 |

这样系统不会用自然语言假装完成。

### 6.2 增加证据等级

建议把证据分成四级：

| 等级 | 含义 |
|---|---|
| L1 工具结果证据 | 来自内部工具、数据库、workflow 的执行结果 |
| L2 内部知识证据 | 来自内部口径库、流程文档、规则库 |
| L3 用户提供证据 | 用户上传截图、聊天记录、手工说明 |
| L4 外部公开证据 | 外部平台公告、行业资料、公开网页 |

系统输出强结论时，至少需要 L1 或 L2；只有 L3/L4 时，应标注不确定性；没有证据时，只能输出建议或待确认态。

### 6.3 增加权限矩阵

一期至少需要角色权限矩阵。

| 角色 | 可查数 | 可看包地址 | 可触发联调 | 可建定时报表 | 可看资产 | 可创建 case |
|---|---|---|---|---|---|---|
| 投放 | 是 | 是 | 视权限 | 是 | 自己/项目 | 是 |
| 发行/运营 | 是 | 视项目 | 否/视权限 | 是 | 项目 | 是 |
| 数据 | 是 | 否/视权限 | 否 | 是 | 数据类 | 是 |
| 广告支持 | 是 | 是 | 是 | 是 | 项目/全量 | 是 |
| 管理者 | 是 | 是 | 审批/查看 | 是 | 团队 | 是 |

无权限不是失败，而是一种明确状态。

### 6.4 增加 AI 不确定性表达规范

所有结果建议分成四类：

- 确定结论；
- 候选判断；
- 待确认；
- 无法判断。

尤其异常初诊和问逻辑场景，不允许默认输出强结论。

## 9. 需要立即修正文档的具体点

### 9.1 修复第 6 章结构问题

当前“需求沟通与结构化需求单”后面格式明显断裂，需要补全“问逻辑”标题和完整结构。

建议改为：

- 6.2 场景 I：异常初诊；
- 6.3 场景 J：需求结构化；
- 6.4 场景 K：问逻辑与任务转接。

### 9.2 把验收指标升级为“验收指标 + 验收用例”

每个场景至少补 3 条 Given / When / Then。

### 9.3 把对象名从“前后端影响”中拿出来

DeliveryPackage、PackageBatch、PackageStatus、PackageDeliveryAsset 等对象不应该只出现在前后端影响里，而应该有独立字段定义和状态规则。

### 9.4 把低风险自动化中心进一步硬化边界

文档已经明确一期自动化中心不包含任意 skill 可视化编排、高风险投放执行、自动修改账户/预算/出价/定向、无审批执行动作。这个边界很好。建议补动作分级，避免后续业务把中高风险动作塞进自动化中心。

### 9.5 把反虚原则放到 PRD 前三页

建议作为全局原则写入：

> 任何需要内部事实或执行结果的回答，必须绑定工具调用、证据来源或明确待确认状态。  
> 未调用工具，不得输出“已完成”。  
> 无证据，不得输出强结论。  
> 工具失败，必须进入失败闭环或 case。

这是防止系统再次变虚的关键。

## 10. 最终结论

这版 PRD 已经解决了“需求方向虚”的问题，但还没有完全解决“实施可落地”的问题。

它最重要的价值在于：

1. 不再以 Agent 架构组织需求；
2. 不再沿用旧十大需求；
3. 明确三条业务闭环；
4. 明确数据/报表和包交付是一完整闭环；
5. 明确异常/需求/逻辑是一轻闭环；
6. 明确知识库不能粗暴前置 RAG；
7. 明确包交付主语是“包是否可投放”；
8. 明确定时报表是 P0；
9. 明确我的资产是业务资产中心；
10. 明确不能无工具、无证据假装完成。

但进入下一阶段前，必须补强：

1. 业务对象模型；
2. 状态机；
3. 权限矩阵；
4. 证据等级；
5. 页面级交互；
6. 验收用例；
7. 分期边界；
8. 第 6 章结构修复；
9. 数据指标 contract；
10. 包交付状态流转 contract。

**一句话总结**：下一步不要继续扩场景，而要把三条闭环压实为对象、状态、权限、证据、验收和页面 contract。只有这样，小乔智投才能从“战略上反虚”进入“实施上反虚”。
