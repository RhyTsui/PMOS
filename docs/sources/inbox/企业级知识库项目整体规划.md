## 一、项目定位

### 1.1 核心定位

**企业级知识库**——服务数据部和其他业务部门的统一知识管理与问答平台。

核心业务支撑：

1.  **数据服务业务**（数据部核心）- 数据口径、报表逻辑、指标定义、API 文档
    
2.  **广告服务业务**（数据部）- 媒体知识、投放规则、归因逻辑、API 对接
    
3.  **其他业务部门** - 数据字典、运营 SOP、产品文档、业务流程
    

### 1.2 价值主张

|  **问题**  |  **解决方案**  |
| --- | --- |
|  知识分散在多个系统和文档中  |  企业级统一知识库  |
|  部门间知识孤岛  |  跨部门知识共享  |
|  新人上手慢、培训成本高  |  Chat 问答快速获取知识  |
|  不同人理解不一致  |  统一语义层确保一致性  |
|  Agent 无法自动获取知识  |  Agent 注册机制和 API  |
|  数据血缘不清晰  |  知识图谱溯源  |

## 二、应用场景与价值目标

## 五、技术架构与演进

### 5.1 技术选型原则

1.  **本地优先**：核心业务数据本地存储，确保安全和可控
    
2.  **混合检索**：向量检索 + 精确查询 + 图谱查询，按知识类型路由
    
3.  **演进式 RAG**：从基础 RAG → 混合检索 → Graph RAG + Agentic
    
4.  **长 Context 兼容**：支持长文档直接输入和 RAG 混合使用

### 5.4 为什么这样设计？

|  **设计决策**  |  **理由**  |
| --- | --- |
|  本地知识图谱  |  数据血缘、指标溯源需要结构化关系，外部工具无法满足  |
|  混合检索  |  不同类型知识需要不同检索方式（指标用精确查询，案例用语义搜索）  |
|  Rerank 优化  |  提升检索结果相关性，解决简单 RAG 召回不精准问题  |
|  长 Context 兼容  |  短对话用长 Context 直接处理，知识库用 RAG，兼顾成本和效果  |
|  演进式架构  |  v1.0 快速上线，后续逐步引入高级能力  |

## 九、风险与依赖

|  **风险**  |  **影响**  |  **缓解措施**  |
| --- | --- | --- |
|  小闪平台接入方式不明确  |  可能影响上线时间  |  提前与小闪团队沟通  |
|  知识源整理工作量大  |  可能影响 v1 进度  |  优先核心知识源，分批次接入  |
|  数仓血缘数据不完整  |  影响血缘追溯功能  |  先手动梳理核心链路，后续自动同步  |
|  LLM API 成本  |  运营成本增加  |  优化 prompt、缓存答案、本地模型备选  |
|  部门协作配合度  |  知识源导入延迟  |  明确负责人、定期跟进  |
