# 一、业务需求（战略层）-需求分析Agent

## 1、对接联调

### （1）媒体首次联调（对接需求，联调测试）

1.  会话（对接）-识别意图-无法识别-追问选择-确认意图-对接
    
2.  提供对接清单-请输入要对接的范围：需求类型（监测回传）、应用类型、媒体、测试项目、预期测试时间、文档地址、测试账户
    
3.  检查列表是否支持-不支持-检索知识库-获取回传页面-文档地址和需求类型一致-判断是否需要密钥-由用户提交密钥才能继续
    
4.  如果需要授权-则打开开发者注册页面-如果没有问用户要-暂时无法提交需求-下次再来调取进度。
    
5.  如果有注册页，则写入需求文档，并在回复时间上增加N天。
    
6.  中间还可确认-是否需要分支需求-比如采集数据-完成后回到主流程-采集数据需要提供页面。
    
7.  下次会话-提交密钥-生成需求文档-调用配置工具-完成配置更新-需求回执可以建链接了且邮件通知开发点击接收处理服务和预期时间。
    
8.  文档地址和需求类型不一致-提示确认文档-无法提交需求。
    
9.  联调环境是否OK判断-没有账户共享无法联调-没有应用共享无法联调
    
10.  齐备-发起联调-展示联调信息-展示自动化介绍-展示数据流日志-日志解读
    
11.  知识库中的需求工单更新-随时检查需求进度和记录。
    

### （2）游戏首次联调

1.会话（联调）-意图识别-无法识别-追问选择-确认意图-自动联调

2.提供联调要求-请共享账号和应用-检测已共享-浏览器工具登录巨量-自动创建资产/复用已有资产-浏览器登录智投-创建短链-切换巨量后台固定清除设备id-确认智投白名单和配置-创建联调-生成联调二维码-保存到本地-调用飞书桌面端-发送二维码消息-调用手机飞书客户端-保存二维码到手机本地-调用抖音app-扫码本机相册的二维码-确认并前往首页刷广告-点击转化追踪-点击落地页-启动app-真实登录-真实充值-回到浏览器截图2边的联调结果-比对结果-生成联调成功报告文档。

3.过程如果存在问题，直接打印问题日志和解读。

### （3）日常联调（版本更新）

1.启动app-虚拟注册-虚拟充值-回到浏览器截图2边的联调结果-比对结果-生成联调成功报告文档。

2.过程如果存在问题，直接打印问题日志和解读。

### （4）自定义回推联调（无付费、虚拟付费）

1.启动app-真实登录-虚拟充值-回到浏览器截图2边的联调结果-比对结果-生成联调成功报告文档。

3.过程如果存在问题，直接打印问题日志和解读。

    ## 一、项目概述
    
    ### 1.1 项目目标
    
    实现巨量广告平台游戏应用联调的全自动化流程，从参数配置到二维码生成、通知发送的端到端自动化，大幅提升联调效率。
    
    ### 1.2 核心功能
    
    | 功能模块 | 说明 |
    |---------|------|
    | Web参数配置 | 可视化表单，支持所有联调参数配置 |
    | 浏览器自动化 | 自动登录、导航、操作巨量平台 |
    | 资产管理 | 自动检查/创建应用资产和事件资产 |
    | 联调执行 | 自动发起联调并生成二维码 |
    | 通知推送 | 自动发送飞书消息通知 |
    
    ### 1.3 技术栈

    ## 五、异常处理
    
    ### 5.1 异常分类

    |---------|---------|---------|---------|
    | NetworkError | 网络连接失败 | 重试 | 3 |
    | ElementNotFoundError | 元素定位失败 | 备用定位器 | 2 |

    | PageLoadTimeoutError | 页面加载超时 | 刷新重试 | 2 |
    | DownloadFailedError | 下载失败 | 截图替代 | 2 |

    ## 六、元素定位策略
    
    ### 6.1 定位优先级
    
    | 优先级 | 策略 | 示例 | 稳定性 |
    |-------|------|------|-------|
    | 1 | data-testid | `[data-testid='submit']` | ⭐⭐⭐⭐⭐ |
    | 2 | XPath (contains) | `//button[contains(text(), '确认')]` | ⭐⭐⭐⭐ |
    | 3 | CSS Selector | `.btn-primary` | ⭐⭐⭐ |
    | 4 | 文本定位 | `text='提交'` | ⭐⭐⭐ |
    | 5 | Playwright定位器 | `page.get_by_role('button')` | ⭐⭐⭐⭐ |
    
    ### 6.2 定位器设计
    
    每个元素定义主选择器和备用选择器：
    
    ```python
    @dataclass
    class Locator:
        primary: str           # 主定位器
        backups: List[str]     # 备用定位器
        description: str       # 描述
    ```
    
    ### 6.3 动态等待策略
    
    ```python
    # 等待元素可见
    await page.wait_for_selector(selector, state='visible', timeout=30000)
    
    # 等待网络空闲
    await page.wait_for_load_state('networkidle')
    
    # 等待特定文本
    await page.wait_for_function("document.body.innerText.includes('成功')")
    ```

    # 广告联调自动化 - 技术方案设计
    
    ## 1. 项目背景
    
    ### 1.1 业务需求
    广告投放联调测试需要人工扫码、授权、刷视频找广告，流程繁琐耗时。需要自动化替代人工操作。
    
    ### 1.2 核心流程
    ```
    生成二维码 → 扫码授权 → 刷新首页 → 刷视频找广告 → 点击广告 → 验证结果
    ```
    
    ### 1.3 边界划分
    - **模块A（另一位测试负责）**：生成二维码
    - **模块B（本项目负责）**：扫码授权 + 刷广告点击

    ### 2.1 模块划分
    
    ```
    ┌─────────────────────────────────────────────────────────┐

    ├──────────────────┬──────────────────────────────────────┤
    ```

    本说明文档拆解了游戏买量链路中最核心的两个数据异常场景（激活 Gap 与 付费 Gap），为【诊断 Agent】提供底层逻辑判定的决策树基准。
    
    场景一：激活数据 Gap（媒体侧数据 < 内部 BI/归因数据）
    
    业务痛点： 优化师在巨量引擎后台看到 100 个激活，但在我们内部 BI 看板或归因平台上却只有 70 个，或者反过来。
    
    Agent 诊断逻辑（自底向上排查）：
    
    graph TD
        A[收到激活 Gap 排查指令] --> B{第一步：校验 SDK 原始日志}
        B -- 无数据 --> C[诊断：客户端 SDK 未成功上报 / 网络拦截]
        B -- 有数据 --> D{第二步：校验设备号合法性}
        
        D -- IDFA/OAID 缺失或全0 --> E[诊断：隐私授权弹窗未同意，导致设备号获取失败]
        D -- 设备号正常 --> F{第三步：校验媒体点击日志匹配度}
        
        F -- 未匹配到对应的 Click 日志 --> G[诊断：非该渠道买量用户，或超出了归因回溯期 通常为7-30天]
        F -- 匹配成功 --> H{第四步：校验防作弊/风控系统拦截}
        
        H -- 命中黑名单/IP异常 --> I[诊断：被内部反作弊系统标记为假量并过滤]
        H -- 正常通过 --> J{第五步：校验向媒体的回传 HTTP 状态}
        
        J -- HTTP 4xx/5xx --> K[诊断：媒体侧接口异常，或回传 Token/宏参数配置错误]
        J -- HTTP 200 成功 --> L[诊断：平台数据处理存在延迟，或媒体侧报表更新延迟]
    
    
    Agent 具体执行 Action 建议：
    
    查 ClickHouse（SDK日志表）： SELECT count(*) FROM sdk_log WHERE event='activation' AND date='今日'
    
    查归因明细表： 检查对应设备的 attributed_media 字段。
    
    查回调日志表： 检查 response_code 和 response_body。
    
    场景二：付费金额/后端事件不一致（媒体侧金额 ≠ 内部 BI 真实收入）
    
    业务痛点： 优化师看媒体后台 ROAS 是 120%，但实际业务线核算只有 80%，导致超成本拿量却浑然不知；或者媒体后台显示没充值，模型没法继续跑。
    
    Agent 诊断逻辑（基于订单与货币对账）：
    
    graph TD
        A[收到付费金额 Gap 排查指令] --> B{第一步：比对双方总订单数}
        B -- 订单数不同 --> C{排查订单漏斗}

    Tool 1: query_device_status(device_id)
    
    功能： 一键查询某个设备在全链路节点（点击、激活、归因、回传）的时间戳。

    国内系统代际割裂：广告系统核心链路已从2.0升级至3.0，但素材管理模块仍停留在2.0时代。新旧系统之间数据不互通，素材ID体系断裂，导致无法在全链路视角下评估素材价值。
    
    海外素材归因瘫痪：海外投放时，同一视频素材上传至不同媒体（Meta、Google、TikTok）后，MD5哈希值会发生改变。同时，运营人员会手动修改素材文件名以作区分。这导致基于文件名或MD5的传统主键匹配彻底失效，素材表现数据无法聚合，形成了“数据孤岛”。
    
    解决方案的核心突破：
    我们提出的 Creative Agent（素材结构发现引擎） 并非为了生成视频，而是为了建立一套内容本质不变的主键系统。
    
    其技术逻辑为：
    
    text
    同一视频内容 → 大模型生成结构化脚本描述 → 语义向量化 → 向量数据库聚类
    MD5变了，文件名改了，但视频的“故事内核”没变。
    
    通过将视频转译为结构化脚本（Hook/冲突/反转/结尾），并进行Embedding，我们就能在语义空间里把全球不同媒体、不同命名、不同哈希值的同一条广告素材重新关联起来。
    
    产品价值闭环：
    
    打破数据孤岛：海外多媒体素材表现数据首次实现以“创意”为维度的聚合分析。
    
    打通升级断点：为国内3.0系统补齐素材认知能力，实现从“广告计划→素材创意”的全链路归因下钻。
    
    沉淀创意资产：自动识别并归类爆款脚本结构，将“这个素材跑得好”的经验，沉淀为“这类反转结构跑得好”的可复用知识。
