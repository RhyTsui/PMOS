# 智投 Chat 思维链视觉交互体验 10 分设计说明文档（Codex CLI 实施版）

> 适用对象：Codex CLI / 前端工程 / 交互实现 / AI Runtime UI 优化  
> 文档版本：v1.0  
> 设计目标：在不改动现有业务逻辑、Agent 流程、Skill 编排和 MCP 调用逻辑的前提下，优化当前“思维链 / 执行过程”的视觉交互体验。  
> 核心原则：**只改表现层，不改执行层。**

---

## 0. 给 Codex CLI 的执行指令摘要

请基于本文档优化当前智投 Chat 的“思维链 / 执行过程”展示体验。

### 必须遵守

1. 不修改现有 Agent 逻辑。
2. 不修改现有 Workflow 流程。
3. 不修改现有 Skill 编排。
4. 不修改现有 MCP 调用顺序、调用参数、调用条件。
5. 不修改业务判断逻辑。
6. 不新增复杂页面，不改成三栏 Dashboard。
7. 只优化 Chat 会话中的过程展示、状态展示、信息压缩、打字机效果、渐进式披露和可视化呈现。

### 本次要实现的核心效果

将当前可能偏长、偏散、偏日志化的“思维链展示”，升级为：

```text
会话卡片式 AI Runtime 过程展示
```

最终体验应该是：

```text
用户看到：AI 正在做什么、做到哪一步、是否卡住、结果是否可信。
用户不看到：大段内部推理、原始 JSON、冗长日志、重复工具返回。
```

# 1. 背景与问题

当前智投 Chat 已具备 Agent、Skill、MCP、流程调度等基础能力。用户通过快捷入口触发任务后，系统会在会话中展示执行过程。

现阶段问题集中在视觉交互体验，而不是业务逻辑：

| 当前问题 | 用户感受 | 需要优化的方向 |
|---|---|---|
| 思维链文本过长 | 看不懂重点 | 信息压缩 |
| 过程和结果混在一起 | 不知道最终结论在哪里 | 结果与过程分离 |
| 工具调用细节直接暴露 | 像系统日志，不像 AI 助手 | Trace 分层展示 |
| 长时间运行缺少节奏 | 不知道 AI 是否还在工作 | 运行态反馈 |
| 异常信息不聚焦 | 不知道卡在哪 | 阻塞点显式化 |
| 完成后过程仍占据大篇幅 | 影响结果阅读 | 完成后自动折叠 |
| 打字机效果滥用或缺失 | 要么慢，要么没有生命感 | 精准使用打字机 |
| 所有信息同权展示 | 信息密度过高 | 渐进式披露 |

本次设计只解决这些交互体验问题。

---

# 2. 产品定位：不要展示“真实思维链”，而是展示“AI 执行过程”

产品上可以继续沿用“思维链”这个内部叫法，但前端展示建议使用以下文案：

- 执行过程
- 分析过程
- 处理过程
- 任务进度
- 工作流进度
- 调用记录
- 运行日志

不建议面向用户展示：

```text
思维链全文
模型内心推理
详细推理过程
```

原因：

1. 用户需要的是“AI 做了什么”，不是“AI 脑子里想了什么”。
2. 真实模型推理不应直接暴露。
3. 业务过程、工具调用、状态推进比长推理文本更可信。
4. 企业用户更关注可追溯、可审计、可复现。

因此，本次设计的“思维链视觉交互”本质是：

```text
AI Runtime Execution Visualization
```

即：

> 用卡片、状态、步骤、摘要、Trace、异常提示，把 AI 执行过程可视化。

# 3. 设计目标

## 3.1 总目标

做到 10 分的 AI Runtime 体验：

> 过程可见，但不喧宾夺主；结果突出，但不是黑盒；细节可追溯，但默认不打扰。

## 3.2 用户体验目标

用户在任务执行过程中，应能清楚知道：

1. AI 当前是否正在工作。
2. AI 当前进行到哪一步。
3. AI 已经完成了哪些检查。
4. AI 是否调用了工具。
5. AI 是否遇到阻塞。
6. 阻塞原因是什么。
7. 最终结果在哪里。
8. 结果是否可信。
9. 是否可以展开查看详细过程。
10. 是否需要自己补充信息。

## 3.3 工程目标

前端实现应满足：

1. 不侵入现有 Agent / Workflow / MCP 逻辑。
2. 兼容现有流式输出。
3. 兼容现有 Tool Call / MCP Trace 数据。
4. 支持运行中增量更新。
5. 支持完成后自动压缩。
6. 支持异常态强提示。
7. 支持侧边栏日志继续保留。
8. 支持正文卡片化展示。

# 5. 核心设计原则

## Principle 1：结果优先

正文优先展示最终结果。

过程区只服务于：

- 建立信任
- 表达进度
- 暴露阻塞
- 支持追溯

不要让过程区抢占结果区。

---

## Principle 2：默认压缩

默认只展示一行状态摘要，例如：

```text
正在检查投放包状态 · 已完成 3/5
```

用户需要时再展开详情。

---

## Principle 3：渐进披露

信息分三层：

| 层级 | 展示内容 | 默认状态 |
|---|---|---|
| L1 | 当前状态摘要 | 默认展示 |
| L2 | 业务步骤过程 | 可展开 |
| L3 | MCP / Tool Trace | 高级详情 / 侧边栏 |

不要一次性把三层信息全部展示在正文里。

---

## Principle 4：卡片化

所有过程信息使用卡片展示。

禁止将过程写成长段自然语言。

推荐结构：

```text
[AI 正在处理]
✓ 已识别项目
✓ 已查询包列表
⏳ 正在检查联调状态
○ 准备生成结果
```

---

## Principle 5：异常优先显式化

一旦出现阻塞，应优先展示：

```text
当前阻塞：
缺少默认账户授权
```

而不是展示：

```text
工具调用失败，返回 code=xxx
```

技术细节可以放在展开详情或侧边栏。

---

## Principle 6：打字机只用于短状态，不用于结构化结果

允许打字机：

```text
正在检查项目上下文...
已找到 3 个可用包
发现 1 个阻塞项
```

不允许打字机：

- 包列表
- 表格
- JSON
- 代码
- 结果卡片
- 长段解释
- MCP 返回详情

---

## Principle 7：完成后自动折叠

任务完成后，过程卡自动变为摘要态：

```text
处理完成 · 已检查 5 项 · 调用 3 个工具 · 无阻塞
[查看过程]
```

最终结果保留展开。

---

## Principle 8：正文不展示原始 Trace

正文不直接展示：

- 原始 JSON
- 完整 SQL
- 原始接口返回
- 全量日志
- 大段工具返回

这些内容放到：

- 侧边栏日志
- 高级详情
- Trace 抽屉
- 可复制详情区

---

## Principle 9：运行中不能长时间静默

如果任务耗时超过 2 秒，必须出现运行态反馈。

如果任务耗时超过 5 秒，必须更新阶段性状态。

例如：

```text
正在等待媒体配置返回...
```

---

## Principle 10：思维链不是主内容，是信任层

最终页面关系：

```text
结果卡片 = 主内容
过程卡片 = 信任层
侧边栏日志 = 审计层
```

---

# 6. 信息架构

## 6.1 会话消息结构

推荐每次 AI 任务消息由以下模块组成：

```text
AI Message
├── Runtime Summary Card
├── Process Card
├── Result Card
├── Blocker Card
└── Footer Actions
```

其中：

- Runtime Summary Card：始终存在
- Process Card：运行中展示，完成后折叠
- Result Card：有结果时展示
- Blocker Card：有阻塞时展示
- Footer Actions：复制、查看过程、查看调用、重新执行等

---

## 6.2 展示优先级

展示优先级从高到低：

1. 最终结果
2. 当前阻塞
3. 当前运行状态
4. 已完成步骤
5. 工具调用摘要
6. 原始 Trace

# 7. 组件设计

---

## 7.1 Runtime Summary Card

### 作用

展示当前任务总体状态。

### 展示位置

AI 回复顶部。

### 状态枚举

```ts
type RuntimeStatus =
  | 'idle'
  | 'running'
  | 'completed'
  | 'blocked'
  | 'failed'
  | 'cancelled';
```

### 推荐视觉

#### Running

```text
AI 正在处理
正在检查投放包状态 · 已完成 3/5
```

#### Completed

```text
处理完成
已检查 5 项 · 调用 3 个工具 · 无阻塞
```

#### Blocked

```text
当前阻塞
缺少默认账户授权，无法继续自动处理
```

#### Failed

```text
处理失败
智投配置 MCP 暂不可用
```

---

## 7.2 Process Step Card

### 作用

展示 AI 执行到哪一步。

### Step 状态

```ts
type StepStatus =
  | 'pending'
  | 'running'
  | 'success'
  | 'blocked'
  | 'failed'
  | 'skipped';
```

### 推荐结构

```text
执行过程

✓ 识别当前项目
✓ 查询投放包列表
✓ 检查验收状态
⏳ 检查媒体审核状态
○ 生成结果
```

### 图标规则

| 状态 | 图标 | 文案风格 |
|---|---|---|
| success | ✓ | 已完成 |
| running | ⏳ / spinner | 正在 |
| pending | ○ | 等待 |
| blocked | ⚠ | 当前阻塞 |
| failed | ✕ | 失败 |
| skipped | — | 已跳过 |

### 交互规则

1. 默认最多展示 3 行。
2. 超过 3 行折叠。
3. 点击“展开过程”展示全部。
4. 完成后自动收起为摘要。
5. 阻塞时自动展开阻塞步骤。

---

## 7.3 Tool / MCP Badge

### 作用

轻量展示调用过哪些系统。

### 推荐样式

```text
已调用：智投配置 MCP · 媒体配置 MCP · 联调 MCP
```

或：

```text
智投配置 MCP  成功
媒体配置 MCP  成功
联调 MCP      未调用
```

### 展示规则

1. 默认只展示 MCP 名称和结果。
2. 不展示参数和原始返回。
3. 点击“查看调用”进入高级详情。
4. 失败工具显示失败原因摘要。

---

## 7.4 Trace Detail Drawer / Side Panel

### 作用

展示详细调用记录。

如果当前产品右侧已有日志 / Trace 侧边栏，本次不新增新页面，只复用侧边栏。

### 展示内容

```text
时间
MCP 名称
Tool 名称
请求摘要
返回摘要
耗时
状态
错误码
```

### 禁止内容

除非用户点击高级详情，否则正文禁止展示完整：

- request body
- response body
- stack trace
- raw log

---

## 7.5 Result Card

### 作用

展示最终业务结果。

### 规则

1. 结果卡片不使用打字机。
2. 结果卡片稳定渲染。
3. 结构化信息优先。
4. 多个结果使用列表或多卡片。
5. 结果字段不与过程混杂。

### 示例

```text
已找到 3 个可投放包

包 1
巨量 Android 广告包
验收通过 · 审核通过 · 联调通过
状态：可投放

包 2
腾讯 Android 广告包
验收通过 · 审核通过 · 联调通过
状态：可投放
```

---

## 7.6 Blocker Card

### 作用

展示当前不能继续推进的原因。

### 推荐结构

```text
当前阻塞

缺少默认账户授权

影响：
无法检查媒体后台应用状态，也无法继续自动共享。

下一步：
请补充默认账户授权。
```

### 规则

1. 阻塞卡优先于普通过程展示。
2. 阻塞信息必须短。
3. 不展示底层日志。
4. 必须指出缺口。
5. 如有下一步，给明确动作。

---

## 7.7 Confidence / Evidence Summary

### 作用

增强可信度。

### 推荐展示

```text
结果可信度：高
依据：已从 3 个系统获取一致状态
```

或：

```text
可信度：中
原因：媒体审核状态未返回，仅基于智投配置判断
```

### 规则

1. 不需要复杂评分。
2. 可用“高 / 中 / 低”。
3. 必须给依据。
4. 只在结果态展示。

# 8. 动效设计

## 8.1 打字机效果

### 允许范围

只用于短状态句：

```text
正在获取项目上下文...
正在查询包列表...
正在检查联调状态...
已找到 3 个可投放包
```

### 禁止范围

禁止用于：

- 结果卡片
- 包列表
- 指标表格
- 报告正文
- JSON
- SQL
- 工具返回
- 长段解释

### 速度建议

```text
中文：30-45ms / 字
英文：15-25ms / char
最大持续时间：1200ms
```

超过 1200ms 的文本不使用打字机，直接展示。

---

## 8.2 Step 渐入

步骤出现采用逐条渐入：

```text
opacity: 0 -> 1
translateY: 4px -> 0
duration: 160ms - 240ms
```

不要使用夸张动画。

---

## 8.3 Running 状态

运行中步骤使用轻量 spinner 或 shimmer。

不建议使用大面积骨架屏。

推荐：

```text
⏳ 正在检查联调状态
```

---

## 8.4 完成态转换

完成后：

```text
Process Card 展开态
↓
300ms 后
↓
Summary 折叠态
```

如果用户手动展开过，则不要强制折叠。

# 10. 信息压缩规则

## 10.1 一句话摘要

每个任务必须生成一句过程摘要：

```text
已完成包状态检查，发现 3 个可投放包。
```

或：

```text
检查已阻塞，缺少默认账户授权。
```

## 10.2 压缩优先级

保留：

1. 当前状态
2. 关键步骤
3. 阻塞原因
4. 最终结果
5. 调用来源

隐藏：

1. 原始参数
2. 原始返回
3. 重复日志
4. 大段分析
5. 中间态噪音

## 10.3 长内容截断

如果过程步骤超过 6 个：

默认只展示：

```text
前 2 个成功步骤
当前运行步骤
最后 1 个待执行步骤
```

并提供：

```text
展开全部
```

# 12. UI 状态机

本次不修改业务状态机，只新增前端展示状态机。

```ts
type RuntimeDisplayState =
  | 'init'
  | 'typing'
  | 'running'
  | 'expanded'
  | 'collapsed'
  | 'completed'
  | 'blocked'
  | 'failed';
```

## 状态转换

```text
init
↓
typing
↓
running
↓
completed → collapsed
```

异常路径：

```text
running
↓
blocked
```

失败路径：

```text
running
↓
failed
```

用户交互路径：

```text
collapsed
↓ click 展开
expanded
↓ click 收起
collapsed
```

# 13. 数据映射建议

不要求后端改字段。如果现有事件里已有类似字段，前端做映射即可。

## 推荐前端中间结构

```ts
interface RuntimeStep {
  id: string;
  title: string;
  status: 'pending' | 'running' | 'success' | 'blocked' | 'failed' | 'skipped';
  description?: string;
  startedAt?: string;
  finishedAt?: string;
  toolName?: string;
  mcpName?: string;
  errorSummary?: string;
}

interface RuntimeTrace {
  id: string;
  mcpName: string;
  toolName: string;
  status: 'success' | 'failed' | 'running';
  latencyMs?: number;
  requestSummary?: string;
  responseSummary?: string;
  errorCode?: string;
  errorSummary?: string;
}

interface RuntimeCardModel {
  taskId: string;
  title: string;
  status: 'running' | 'completed' | 'blocked' | 'failed';
  summary: string;
  progress?: {
    current: number;
    total: number;
  };
  steps: RuntimeStep[];
  traces?: RuntimeTrace[];
  blocker?: {
    title: string;
    reason: string;
    impact?: string;
    nextAction?: string;
  };
}
```

# 17. 视觉层级建议

## 17.1 色彩语义

| 状态 | 语义 |
|---|---|
| Success | 已完成 |
| Running | 正在处理 |
| Warning | 阻塞 / 缺口 |
| Error | 调用失败 |
| Neutral | 等待 / 跳过 |

具体颜色跟随现有 Design System，不强行新增色板。

## 17.2 字重

| 内容 | 字重 |
|---|---|
| 卡片标题 | Semibold |
| 当前状态 | Medium |
| 步骤描述 | Regular |
| Trace 元信息 | Regular / muted |
| 阻塞原因 | Semibold |

## 17.3 空间

过程卡不要过大。

推荐：

```text
padding: 12-16px
step gap: 8px
card radius: 跟随现有系统
```

# 18. 可访问性

1. 动画必须可关闭或跟随 prefers-reduced-motion。
2. 状态不能只依赖颜色。
3. 图标必须配文字。
4. 阻塞信息必须文本可读。
5. 过程卡展开 / 收起支持键盘操作。
6. Trace 详情不要只通过 hover 访问。

---

# 19. 性能要求

1. 不因打字机导致整页频繁 re-render。
2. 长列表步骤虚拟化或折叠。
3. Trace 明细默认不渲染完整大对象。
4. 流式更新做节流。
5. 单条消息中的过程卡更新频率建议不超过 5 次 / 秒。
6. 打字机文本长度超过阈值直接展示。

# 24. 10 分体验的判断标准

当前优化完成后，体验应达到：

```text
像 ChatGPT 的会话流
像 Deep Research 的过程透明
像企业系统的可审计
像 AI Runtime 的自动推进
```

而不是：

```text
像系统日志
像后端 Debug 面板
像普通 FAQ
像流程说明文档
```

最终用户感受应该是：

> AI 在稳定、透明、可追溯地推进任务；  
> 我不用看复杂过程，也能相信结果；  
> 需要排查时，我可以展开看到依据。

# 25. 最终设计结论

本次优化的核心不是增加更多内容，而是减少噪音、提升秩序、强化状态感。

最终形态：

```text
正文负责交付结果
过程卡负责建立信任
侧边栏负责审计追踪
```

落地原则：

```text
只改视觉交互
不改业务逻辑
只压缩展示
不隐藏关键阻塞
只展示执行过程
不暴露模型内心推理
```

这是当前智投 Chat 思维链视觉体验升级到 10 分的最小改造路径。
