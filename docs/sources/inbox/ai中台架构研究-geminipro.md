企业私有 AI 中台架构决策与前瞻性评估报告

评估专家： 小乔（前腾讯/字节/Google 资深产品架构专家）
当前阶段： 架构定型与技术可行性验证期

一、 整体架构评价

该 6 层架构设计达到了一线大厂准商业化标准。核心亮点在于**“极端的解耦”**：

资产与逻辑解耦：Prompt（逻辑指令）与 Knowledge（事实素材）物理隔离，这解决了传统 Agent 搭建中“系统提示词臃肿导致上下文溢出”的死穴。

调度与观测解耦：MCP 中枢专注于“跑通”，扣子罗盘专注于“跑好”，职责互不干扰，便于未来独立扩容。

安全红线前置：明确规定终端不固化资产、完全依赖 ID 绑定，这为企业数据资产的安全管控、审计溯源打下了完美基础。

二、 专家决策建议与潜在风险预判（实战避坑）

基于过去搭建高并发广告系统与复杂 Workflow（如 N8N 级编排）的经验，我对各模块提出以下前瞻性建议与风险预判：

1. MCP 统一调度运行中台（高危瓶颈点）

潜在风险：作为“心脏总线”，所有请求均经过 MCP。当企业内部出现高频并发（类似广告竞价 RTB 场景），MCP 将成为单点故障（SPOF）和延迟瓶颈。

决策建议：

架构选型：建议 MCP 核心调度层采用 Golang 或 Rust 重写，而非传统的 Python，以支撑高并发。

状态管理：MCP 需要引入 Redis 等高速缓存，用于维护“会话上下文（Session Memory）”和“配置热更新快照”。当 Prompt 中台修改发布新版本时，MCP 应对进行中的会话保持老版本快照，新会话采用新版本，避免运行时上下文割裂。

2. 企业私有知识库中台（召回率陷阱）

潜在风险：仅依赖单纯的“向量检索（Vector Search）”在企业严肃场景下（如规章制度、财务报表）准确率极差。

决策建议：

必须在 MVP 阶段就引入 混合检索（Hybrid Search：向量 + BM25 倒排索引）。

增加 重排（Rerank）节点 过滤低相关性片段。如同广告推荐系统中的“粗排+精排”漏斗，确保喂给 MCP 的上下文是最高质量的。

针对复杂实体关系（如“查询某业务线的组织架构及所有产品线”），需预留 GraphRAG（知识图谱+RAG） 的升级接口。

3. 评测观测平台 (扣子罗盘)（成本与 ROI 考量）

潜在风险：全量线上对话如果都由 LLM 作为评估器（LLM-as-a-judge）进行打分，Token 消耗成本会呈指数级上升，ROI 极低。

决策建议：

采样策略：类似广告数据归因中的采样统计，线上观测应采用“规则拦截打分（快/便宜） + 关键用户随机抽样大模型打分（慢/准） + 用户显式反馈（点踩/点赞）”的三维评价体系。

链路埋点追踪（Trace）：必须设计类似 SDK 埋点的数据规范，MCP 在调用时生成全局唯一的 trace_id，打通模型网关、知识召回、Prompt 版本的全链路耗时与结果记录。

4. 终端应用层（用户体验与冷启动）

决策建议：业务 Chat 不要只是一个简单的输入框。前端需要具备基于当前配置 ID 动态渲染“欢迎语”、“开场预设问题（Quick Replies）”的能力，降低用户的对话冷启动门槛。

三、 结论

架构方案强烈建议通过，并可作为最终蓝图进入研发排期。在实施路径上，建议优先打通“模型网关 -> MCP -> Prompt中台 -> 终端”的最短链路（MVP1），再接入“知识库”（MVP2），最后完善“罗盘评测”（MVP3）。
