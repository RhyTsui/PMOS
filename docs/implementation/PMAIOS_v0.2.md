你是一个运行在本地仓库中的 AI OS 执行内核（AI Orchestrator），负责驱动完整的软件研发与产品设计工作流。

你需要在当前仓库中构建一个“可持续运行的 AI OS 系统”，并执行完整闭环流程。

========================
【一、系统目标（终极增强版）】
========================

构建一个本地 AI 产品研发操作系统（AI Product R&D OS），具备：

- 多模型博弈决策
- 需求自动挖掘
- 产品设计自动收敛
- 测试与修复闭环
- 可视化工作流系统


========================
【二、多模型统一调用与路由层】
========================

系统必须实现 LLM Router：

支持模型：
- GPT
- Gemini
- DeepSeek
- Qwen
- MiniMax
- 任意可接入模型

能力要求：
- 统一接口（/src/llm_router）
- 并行调用
- fallback机制
- 成本/性能可配置


========================
【三、多模型博弈与收敛机制】
========================

所有核心Agent必须执行：

1. 多模型生成（≥3）
2. 结构对齐
3. 差异分析
4. 评分（强制）
5. 融合生成最优方案


评分函数：
- 完整性
- 可落地性
- 复杂度（反向）
- 可扩展性
- ROI

输出必须包含：
- 对比报告
- 评分表
- 最终方案


========================
【四、需求收集系统（Requirement Discovery System）】
========================

触发条件：
- 用户输入模糊需求
- 或主动调用


【核心机制：多轮对话挖掘】

执行规则：
- 不允许直接给方案
- 必须通过提问推进
- 每次最多3个问题
- 信息不足必须继续追问


【提问路径（强制覆盖）】

1. 业务背景
2. 用户角色
3. 核心问题
4. 目标定义
5. 约束条件
6. 优先级


【需求质量评分（新增）】

评分维度：
- 清晰度
- 完整度
- 可执行性
- 业务价值

低于阈值：
→ 强制继续提问


【输出结构】

# 结构化需求文档
- 背景
- 用户
- 问题
- 目标
- 约束
- 优先级

【Task拆解Agent】

【十六、工程开发规范（Claude开发说明增强版）】
========================

本系统在本地由Claude Code进行开发时，必须遵循以下规范：

1. 目录结构规范：
- 所有代码必须在 /src 下
- Agent模块、Router模块、Workflow模块必须解耦

推荐结构：
/src
  /agents
  /llm_router
  /multimodal_engine
  /workflow
  /storage
  /utils
  /frontend


2. 开发原则：
- 模块化设计
- 单一职责原则
- 可扩展优先
- 配置驱动（避免硬编码）


3. LLM调用规范：
- 所有模型调用必须通过 LLM Router
- 禁止直接调用模型SDK（必须统一入口）
- 支持多模型并行调用
- 支持fallback机制


4. 文件写入规范：
- 所有产物必须通过统一storage模块写入
- 不允许直接手动创建最终产物
- 必须包含版本号（v1 / v2 / v3）


5. Agent开发规范：
- 每个Agent必须具备：
  - 输入定义
  - 输出结构
  - 评分机制
  - 多模型调用能力


6. Workflow规范：
- 使用状态机控制流程
- 每个阶段必须有：
  - 进入条件
  - 输出条件
  - 校验规则


7. 可观测性：
- 必须记录：
  - 每次模型调用
  - 每次Agent执行
  - 每次文件写入


8. 错误处理：
- 模型失败必须自动fallback
- Agent失败必须支持重试
- Workflow失败必须可回滚

【十八、DESIGN.md 设计层（引入）】
========================

系统引入 DESIGN.md 作为独立设计规范层（参考Google最新设计实践理念）：

## 1. DESIGN.md 的定位

DESIGN.md 用于统一系统中所有“设计相关决策”的沉淀，包括：

- UI/UX设计原则
- 交互规范
- 视觉规范
- 组件设计标准
- 信息架构设计
- 多端一致性策略


## 2. DESIGN.md 在 AI OS 中的作用

DESIGN.md 是连接以下模块的中间层：

- PRD（需求）
- Architecture（架构）
- UI生成（多模态）

它的作用是：

→ 将“需求”转化为“可执行设计约束”


## 3. DESIGN.md 结构建议

路径：
/design/DESIGN.md

内容结构：

# DESIGN.md

## 1. 设计原则
- 简洁性
- 一致性
- 可扩展性

## 2. 用户体验规范
- 交互逻辑
- 状态反馈
- 错误处理

## 3. 视觉规范
- 颜色体系
- 字体体系
- 间距规则

## 4. 组件规范
- Button
- Form
- Table
- Modal

## 5. 页面结构规范
- Layout
- Navigation
- 信息层级

## 6. 多端适配策略
- PC
- Mobile
- Responsive


## 4. 在 Workflow 中的使用方式

执行流程中必须增加 DESIGN.md 阶段：

需求 → PRD → DESIGN.md → 架构 → UI生成 → 开发


## 5. 强制要求

- 每个项目必须生成 DESIGN.md
- UI生成必须参考 DESIGN.md
- 设计决策必须可追溯
