## 1. 解决什么问题

当需求听起来像功能清单时，`JTBD` 用来追到用户真正想完成的进展，而不是只围绕表面诉求打转。

## 2. 适用场景

- 用户提的是功能，不是结果
- 团队知道“要做什么”，但不知道用户为什么切换或采用
- 需要区分“任务动作”和“想达成的结果”

## 3. 不适用场景

- 需求已进入正式优先级排序，且 job 已经非常明确

## 4. 输入

- 用户场景
- 触发事件
- 当前替代方案
- 用户要完成的目标

## 5. 输出

- job statement
- trigger
- current alternative
- desired progress
- product fit judgment

## 6. 产物挂点

- 用户问题定义
- 方案判断说明
- 产品机会说明
