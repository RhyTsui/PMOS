## 1. 解决什么问题

当一组需求都看起来“很重要”时，`Kano` 用来区分哪些是基础要求，哪些是性能项，哪些是惊喜项。

## 2. 适用场景

- 候选需求很多
- 团队无法区分“必须有”和“有了更好”
- release scope 需要保底

## 3. 不适用场景

- 问题还停留在产品定义阶段
- 候选需求太少，不需要系统分类

## 4. 输入

- 候选需求列表
- 用户预期信息
- 使用场景
- 版本目标

## 5. 输出

- basic / performance / delighter classification
- must-have baseline
- optional delight candidates
- scope risk note

## 6. 产物挂点

- 需求分层判断
- 版本范围说明
- 交付保底清单
