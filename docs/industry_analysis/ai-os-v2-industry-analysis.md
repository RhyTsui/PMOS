# AI OS v2 行业分析

- 版本：v0.1
- 日期：2026-04-07
- 目标：为本地 AI OS 产品工作流构建提供行业基准、竞品对标、可借鉴模式与设计约束

---

## 1. 研究结论摘要

本地 AI OS 的最佳实践并不是做一个“万能对话框”，而是构建一个**文件驱动、流程可追踪、能力可替换、评审可审计**的系统。成熟方案的共同点包括：

1. 把 AI 能力嵌入开发工作流，而不是只做聊天入口
2. 把工具调用与模型调用分离
3. 强调可追踪的任务、产物和评审
4. 用显式规范和工作流减少 agent 漫游
5. 用本地或仓库内资产作为长期记忆与团队协作基础

---

## 2. 对标对象

### 2.1 Claude Code / Claude 驱动开发流

**模式：**
- 强调在本地仓库中执行
- 擅长文件驱动与工具协同
- 适合“读代码 → 规划 → 修改 → 测试 → review”闭环

**优点：**
- 本地上下文强
- 适合工程变更与多文件协作
- 工具能力强，适合 workflow 执行内核

**局限：**
- 默认不是完整产品平台
- 需要仓库显式定义 prompts、workflow、memory 才能长期稳定运行

**可借鉴点：**
- 把仓库当作操作系统状态面
- 用文件作为工作流输入输出
- 用工具执行代替纯对话说明

---

### 2.2 GitHub Copilot / Workspace 风格产品

**模式：**
- 从需求到任务再到代码生成
- 强调与代码仓库、PR、Issue 的深度结合

**优点：**
- 开发任务入口自然
- 适合任务驱动与代码生成

**局限：**
- 产品设计、行业分析、评审委员会这类上游治理能力相对弱

**可借鉴点：**
- 任务切分和代码生成必须联动
- 代码产出需要与 review / CI 紧密耦合

---

### 2.3 Cursor / IDE 原生 AI 方案

**模式：**
- 以开发体验为中心
- 强调上下文理解、文件修改、agent 式实现

**优点：**
- 开发效率高
- 上下文感知强

**局限：**
- 不是完整的产品研发操作系统
- 往往缺少文件化流程资产与组织级治理结构

**可借鉴点：**
- 保持执行路径短
- 对用户请求做任务化拆分
- 快速反馈运行状态

---

### 2.4 Devin / 自主软件工程 agent 模式

**模式：**
- 长链路任务执行
- 更强调自主规划与持续运行

**优点：**
- 适合复杂研发流程
- 适合作为多阶段执行参考

**局限：**
- 如果缺少明确边界，容易不可控
- 需要更强任务治理、审计、回滚能力

**可借鉴点：**
- 需要显式工作流与状态机
- 需要 review gate 与人工接管点

---

### 2.5 MCP 生态

**模式：**
- 模型通过统一协议访问外部工具与上下文
- 将工具发现、调用、权限边界标准化

**优点：**
- 扩展性强
- 工具整合成本低
- 适合做 AI OS 的 integration layer

**局限：**
- 如果没有策略层，工具权限容易失控
- 需要与 workflow、provider abstraction 解耦

**可借鉴点：**
- MCP 只负责工具协议，不负责业务流程编排
- 应单独设计 registry、policy、adapter

---

## 3. 行业最佳实践

### 3.1 文件驱动优先
- 所有关键产物都应落盘
- PRD、架构、任务、review、决策、memory 需要在仓库中可追踪
- 避免对话成为唯一系统状态

### 3.2 工作流显式化
- 阶段清晰：行业分析 → PRD → 架构 → 任务 → 实现 → 评审 → 迭代
- 每阶段都有输入、输出、验收标准
- 支持暂停、回滚、返工、人工审批

### 3.3 角色与能力解耦
- 角色负责视角和责任
- 能力模块负责执行逻辑
- 不把所有逻辑塞进单一 agent

### 3.4 评审机制前置
- review 不能只做“代码对不对”
- 应覆盖产品、架构、技术、数据、风险
- 每个评审结论必须有问题、影响、建议、标准答案、通过状态

### 3.5 Memory 显式建模
- Memory 不是 LangChain 的内置聊天记录替代物
- 需要文件化长期记忆、运行时短期记忆和知识库分层

### 3.6 Provider 可替换
- 不绑定单一模型厂商
- 统一文本、代码、多模态入口
- 密钥只通过环境变量注入

---

## 4. 对本项目的设计启示

### 应保留
- Claude Code 作为执行内核
- MCP 作为工具扩展协议
- Prompt + Workflow + File Artifact 组合模式
- Review Committee 作为质量门禁

### 应避免
- 再造 Claude Code / OpenSpec / Superpowers 的完整替代物
- 把所有状态塞在内存或聊天中
- 把 workflow 和 prompt 重复定义在多个地方
- 未经治理直接让 agent 执行高风险动作

### 应新增
- 行业分析文档层
- 可视化 workflow viewer
- review committee 聚合机制
- 多模态产物模板（Mermaid、UI说明、视频脚本）

---

## 5. 建议产品定位

AI OS v2 应定位为：

> 一个本地优先、文件驱动、工作流化的 AI 产品研发操作系统。

它的核心竞争力不在于“比通用聊天更能说”，而在于：
- 能组织复杂研发流程
- 能持续积累项目知识
- 能标准化交付产物
- 能把 AI 行为纳入工程治理

---

## 6. 结论

本项目应采用以下路线：

1. 本地优先
2. 文件驱动
3. 分层架构
4. 多角色 workflow
5. review committee 审核门禁
6. provider / MCP 可替换集成
7. 最小可运行前后端 dashboard
