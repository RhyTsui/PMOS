## 0. 文档目的

本文档不替代调研、规划、需求、功能、设计、前后端真源文档。

它只做三件事：

1. 把小乔智投当前轮需要交付的正式产物收成一条链
2. 明确每个交付物的上游依赖、当前状态、目标产物与验收口径
3. 规定后续本会话的推进顺序，避免文档、前端、后端各自漂移

---

## 1. 本轮总目标

本轮不是继续发散想法，而是把小乔智投从“真源已初步收口”推进到“交付物齐套且最小可演示闭环成立”。

补充约束：

1. 之前产出的 HTML / SVG 方案不再作为有效交付物
2. 旧 demo 型静态稿不再作为本轮页面输入
3. 当前轮只以前置文档、当前代码骨架和后续新页面输出为准

本轮总目标拆成两层：

1. 用户需求层
   - 让用户能看懂小乔第一阶段到底解决什么问题
   - 让用户能看到一套从对话入口到结果回看的可演示闭环
2. 产品与工程层
   - 把调研、规划、需求、功能、设计、两轮前端页面、数据表、接口、联调串成同一口径

当前闭环判断：`partial`

原因：

- 文档真源已经比较齐
- 最小前端与最小后端骨架已存在
- 但正式交付包还没有按当前 10 类交付物完成统一映射
- 功能文档、设计文档、两轮前端页面与前后端联调口径还没有完全压到实现上

## 2. 正式交付链

本轮按以下顺序推进，不建议跳步：

1. 调研文档
2. 规划文档
3. 需求文档
4. 功能文档
5. 设计文档
6. 前端页面（简单版）
7. 前端页面（UI/UX 版）
8. 数据表
9. 后端接口
10. 前后端联调
补充范围约束：
1. 管理页面属于正式交付范围。
2. 提示词管理属于第一优先管理模块。
3. 管理页与主工作台共用同一登录系统，只做页面跳转，不做第二套登录。
4. 知识库与评测平台只作为外部依赖接入，不在小乔内重做平台。
5. 当前不以本地直连预发布数据库作为第一阶段前提。
6. 当前阶段正式采用本地 mock contract 驱动前后端联调。
7. mock 数据不是 demo 占位数据，而是后续切换测试 / 生产数据前的稳定承接层。

顺序判断原则：

- 1 到 5 负责把“做什么、做到什么程度、如何承接”说清
- 6 到 10 负责把“系统是否真的可演示、可验证”做实
- 前端页面分两轮推进：先做简单版确认结构，再做 UI/UX 版收口体验
- 数据表、接口、服务必须回指前面的需求与功能，不单独生长

## 4. 当前事实判断

### 4.1 已经具备的基础

1. 小乔产品定义已经从旧 demo 思路中重启
2. 第一阶段需求范围已经从散点想法压成需求对象与需求池入表稿
3. 单对话主工作台方向已经明确
4. 后端已经有 `conversation / message / routing / task / result / workspace` 最小骨架
5. 前端已经有 `XiaoQiaoPage.jsx` 演示页，可作为重构底稿

### 4.2 当前最明显的缺口

1. 缺“正式交付件目录”视角
2. 缺两轮前端页面的明确分层口径
3. 数据表、接口、代码模型三者还没有做统一回指校验
4. 前后端联调还缺一版完整联调与验收口径
5. 旧 HTML / SVG 输出仍在仓库中残留引用，容易污染当前交付判断

### 4.3 优先级判断

当前最高优先级不是继续扩新功能，而是把正式交付包补齐并压实到当前代码。

原因：

1. 如果不先明确“简单版”和“UI/UX 版”的页面边界，前端会持续混做
2. 如果不先做文档口径收敛，后面页面重构和接口扩展会继续返工
3. 当前已经有最小代码骨架，适合先打简单版页面，再收 UI/UX 版
4. 旧 HTML / SVG 已被判定为无效输入，必须从当前交付链中移出

## 5. 本轮推进策略

本轮采用“前置文档确认后直接压到实现”的策略。

当前数据策略补充：
1. 运维已确认智投预发布数据库当前禁止本地连接。
2. 因此第一阶段不再等待真实库连通后再推进页面、接口和联调。
3. 当前前后端以正式 mock contract 为主线推进，后续收到真实 mock schema / content 后再逐步替换。
4. `.env.pre.local` 和数据库配置入口仅作为后续切换环境的预留位，不作为当前主交付阻塞项。

具体顺序：

1. 先建立交付总控
2. 以前置文档为实现基线
   - 调研文档
   - 规划文档
   - 需求文档
   - 功能文档
   - 设计文档
3. 先输出前端页面（简单版）
4. 再升级前端页面（UI/UX 版）
5. 再回查数据表、接口、代码模型是否一致
6. 最后做前后端联调说明
