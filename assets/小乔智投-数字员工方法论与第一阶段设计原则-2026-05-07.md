## 0. 文档目的

本文档用于在需求、原型、交互、前端、数据表、接口、后端之前，先明确小乔智投作为“数字员工”产品的设计方法论与第一阶段设计原则。

它不是页面文档，而是后续所有产品判断的上位约束。

---

## 1. 当前产品定义

小乔第一阶段不是知识库平台，不是评测平台，也不是一组散落的广告工具页。

当前产品定义为：

`小乔第一阶段是广告技术服务的统一对外数字员工，以自由问答方式接住请求，并在背后调度多 Agent 体系完成支持、判断、分发和处理。`

其中：

1. 对外角色是 `小乔`
2. 对外形态是 `统一对话窗口`
3. 对内结构是 `多 Agent 体系`
4. 对内目标是 `把广告技术服务工作数字化`

## 2. 方法论总原则

结合 OpenAI、Anthropic、Microsoft 当前关于 Agent 系统的公开最佳实践，小乔智投第一阶段遵循以下原则。

### 2.1 前台单人格，后台多 Agent

用户优先感知的是一个统一数字人 `小乔`，而不是一组并列功能 Agent。

后台可以有：

1. 使用帮助 Agent
2. 问题排查 Agent
3. 需求沟通 Agent
4. 广告联调 Agent
5. 运维 Agent
6. 素材分析 Agent
7. 广告预测 Agent

但用户原则上不需要先理解这些内部结构。

### 2.2 先做最简单可行系统，再增加复杂度

第一阶段不追求一开始就把所有 Agent 显式化、流程可视化、权限平台化。

先做：

1. 自由问答入口
2. 隐式判断
3. 轻重工作流分层
4. 必要结构化结果

再逐步增加：

1. 更复杂 Agent 协同
2. 更复杂自动化执行
3. 更复杂治理与评测平台化

### 2.3 交互像通用大模型 Chat，能力强于通用大模型 Chat

小乔前台交互应对齐 Codex / DeepSeek 这类大模型 Chat：

1. 自由输入
2. 连续追问
3. 允许改口
4. 不强制菜单起手

但它背后比通用 Chat 多两层增强：

1. 内部知识增强
2. 广告域定向确定性工作流增强

### 2.4 所有有效广告请求都属于工作，只区分轻重

不要把“指标解释”“逻辑说明”“系统去哪看”“一般知识问答”误判为非工作。

正确分层是：

1. 非工作：无广告业务处理目标的问题
2. 轻工作流：解释、引导、导航、知识问答
3. 重工作流：排查、需求、联调、监控处置等

### 2.5 先判断，再分发，再处理

入口层不能直接等同于排查页、联调页或需求页。

正确入口链路为：

`自由问答 -> 业务 / 非业务判断 -> 业务域判断 -> 广告域诉求判断 -> 追问补齐 -> 工作流分发 -> Agent 承接`

### 2.6 判断不清就追问，不强猜，不乱拒答

前一个 Chat 的核心问题之一是：

1. 人为限制过多
2. 容易拒答
3. 经常不回应
4. 上下文容易失真

小乔第一阶段应尽量避免：

1. 过早拒答
2. 过度分类暴露
3. 在不确定时强行给结论

正确做法是：

1. 追问补上下文
2. 不清楚就降级回答
3. 能自然回答就先自然回答
4. 能进入工作流就进入工作流

### 2.7 结构化结果必须落地，不只留在对话里

如果一个系统只能对话、不能留下结果物，就不能称为数字员工。

所以重工作流至少要沉淀：

1. 需求单
2. 排查结论
3. 联调报告
4. 监控事件
5. Case

### 2.8 从第一天就考虑评测、观测和升级

数字员工不是一次性做完的功能，而是持续调优的服务能力。

第一阶段就要考虑：

1. prompt 版本
2. 知识引用
3. MCP / 工具调用链
4. bad case 回流
5. 评测结果引用
6. 真源升级机制

## 3. 小乔数字员工的岗位抽象

小乔第一阶段要模拟的，不是“一个 AI 工具”，而是广告技术服务窗口的工作方式。

当前岗位抽象为：

1. 接待
2. 判断
3. 解释
4. 排查
5. 需求收口
6. 联调推进
7. 记录沉淀

### 3.1 接待

接住所有进入广告技术服务窗口的请求。

### 3.2 判断

判断是否是业务问题、是否属于广告域、是否要进入工作流、属于哪一类工作流。

### 3.3 解释

对指标、逻辑、路径、系统能力做说明和引导。

### 3.4 排查

对广告异常、数据 gap、归因与回推问题进入结构化排查。

### 3.5 需求收口

把模糊需求变成结构化需求。

### 3.6 联调推进

把广告联调从人工口头推进变成可跟踪流程。

### 3.7 记录沉淀

保留会话、任务、结果、历史与 Case。

## 4. 第一阶段工作流分层

## 4.1 非工作

定义：

不属于广告业务处理目标的问题。

例如：

- 闲聊
- 无关提问
- 与游戏广告业务支持无关的输入

处理原则：

1. 像通用大模型 Chat 一样自然回答
2. 不进入业务工作流
3. 不生成业务结果物

## 4.2 轻工作流

定义：

广告域内的解释与引导型工作。

当前至少包括：

1. 指标解释
2. 逻辑说明
3. 系统去哪看
4. 一般知识问答
5. 使用指引

处理原则：

1. 保留业务上下文
2. 依赖内部知识与示例
3. 输出解释、引用、路径、建议
4. 默认不进入复杂状态推进
5. 识别出真实任务时可升级到重工作流

## 4.3 重工作流

定义：

需要明确推进、结构化结果和后续动作的广告业务任务。

当前至少包括：

1. 问题排查
2. 需求沟通
3. 广告联调

后续可扩：

1. 监控处置
2. 素材分析任务
3. 广告预测任务

处理原则：

1. 建立任务上下文
2. 保留状态
3. 允许追问
4. 产出结构化结果
5. 沉淀历史记录与 Case

## 5. 第一阶段设计原则

### 5.1 对话优先，不是页面优先

前端第一优先不是菜单和模块堆叠，而是对话入口和对话体验。

### 5.2 隐式分流，不强制暴露分类过程

用户优先看到的是结果和状态，而不是“现在进入哪个 Agent”。

### 5.3 结果优先，不是回答优先

只会说，不会推进事情，不算合格数字员工。

### 5.4 会话与任务上下文必须属于小乔自己

Dataki 和连弩可以是外部支撑平台，但业务运行时上下文必须留在小乔内部。

### 5.5 不把外部平台重做进小乔

当前明确边界：

1. Dataki 负责知识与知识资产
2. 连弩负责评测
3. 小乔负责业务工作台与业务运行时

### 5.6 不以 n8n 为核心编排

与开发确认后：

1. 小乔主流程不以 n8n 为核心编排
2. 主流程由开发自写
3. 小乔需要自己的业务运行时

---

## 6. 当前第一阶段重点

当前第一阶段的重点不是：

1. 把所有能力做完
2. 先把 UI 做华丽
3. 先把多 Agent 显式展示完整

而是：

1. 先把统一对话入口做对
2. 先把轻 / 重工作流判断做对
3. 先把问题排查这条主价值链打透
4. 先把任务记录和结果沉淀做出来
