## 0. 文档目标

这份文档聚焦“需求沟通”专项，目标不是记录需求背景，而是把广告支持中的配置 / 接入 / 协作类需求收成正式结果物。

---

## 1. 专项范围

### 1.1 当前专项解决的问题

1. 广告接入需求长期口头化
2. 字段缺失导致反复追问
3. 依赖项不清导致推进缓慢
4. 最终无法沉淀为正式需求对象

### 1.2 第一批覆盖的需求类型

1. 媒体回传接入
2. 事件映射 / 埋点相关需求
3. 归因 / 配置 / 白名单类需求
4. 跨角色协同的信息补齐型需求

---

## 2. 专项需求

### 2.1 目标用户

1. 广告技术支持
2. 对接联调角色
3. 产品 / 运营协作方
4. 数据 / 埋点协作方

### 2.2 核心需求

#### R-DEMAND-01 模糊需求结构化

输入：

1. 自然语言需求
2. 媒体
3. 应用 / 包体
4. 目标配置对象

输出：

1. 识别需求类型
2. 收成结构化需求单

#### R-DEMAND-02 缺失字段识别

必须识别：

1. 哪些字段缺失
2. 缺失字段优先级
3. 由谁补

#### R-DEMAND-03 依赖项清单

除了需求单本身，还要输出：

1. 依赖系统
2. 依赖角色
3. 依赖动作
4. 下一步协作建议

---

## 3. 专项功能

### 3.1 功能分组

#### F-DEMAND-01 需求类型识别

1. 识别媒体接入
2. 识别回传配置
3. 识别事件 / 埋点需求
4. 识别归因 / 白名单需求

#### F-DEMAND-02 缺失字段追问

1. 缺字段识别
2. 字段分组
3. 追问补齐

#### F-DEMAND-03 需求单生成

输出正式结果：

1. 需求单主体
2. 缺失字段清单
3. 依赖项清单
4. 下一步动作

#### F-DEMAND-04 结果展示与沉淀

1. 页面结果区展示
2. 任务记录
3. 可继续追问

### 3.2 后端承接对象

1. `DemandTask`
2. `DemandForm`
3. `MissingFieldItem`
4. `DependencyItem`

### 3.3 结果结构建议

`DemandForm` 建议字段：

1. `demand_type`
2. `media`
3. `app_name`
4. `package_name`
5. `target_object`
6. `target_timeline`
7. `acceptance_method`
8. `owner_roles`

`MissingFieldItem` 建议字段：

1. `field_name`
2. `priority`
3. `owner_role`
4. `reason`

---

## 4. 页面与设计承接

### 4.1 主工作台结果区

需求结果区应至少包含：

1. 结构化需求单
2. 缺失字段区
3. 依赖项区
4. 下一步协作区

### 4.2 历史任务区

应显示：

1. 需求类型
2. 媒体
3. 应用
4. 当前完整度

### 4.3 管理页承接

管理页后续应支持：

1. 需求类型模板
2. 字段模板
3. 结果模板

---

## 5. 实施顺序

1. 先定义需求单 mock schema
2. 先做缺失字段与依赖项结构
3. 再做结果区正式渲染
4. 最后再接模板管理
