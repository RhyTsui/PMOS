## 0. 本版功能文档解决什么问题

本版不再只说明“有哪些功能”，而是把关键功能压到可开发、可接接口、可继续接外部能力的层级。

每个功能至少说明：

1. 来源需求。
2. 前端承接点。
3. 当前后端 contract。
4. 需要的结果结构。
5. 后续可接入的外部能力位。
6. 异常与回退方式。

## 2. 功能分组总览

第一阶段功能当前按 F-01 ~ F-40 管理，但本版把重点压在真正承接业务和系统治理的部分：

1. F-01 ~ F-08：统一入口与路由
2. F-09 ~ F-24：四条业务流核心功能
3. F-25 ~ F-30：运行时沉淀
4. F-31 ~ F-35：前端工作台承接
5. F-36 ~ F-40：管理页面与提示词治理

其中最关键的是 `F-09 ~ F-24`，因为它们决定小乔是不是在做真实业务，而不是浅层对话。

## 3. 功能清单

## 3.1 统一入口与路由

### F-01 首屏输入区

- 来源需求：`R-01`
- 前端承接：主工作台底部输入区
- 后端依赖：无直接接口
- 关键行为：允许首条消息直接进入系统
- 异常回退：输入为空时禁止发送

### F-02 推荐起手问题

- 来源需求：`R-01`
- 前端承接：会话空态区
- 后端依赖：当前可静态配置，后续可升级为服务端配置
- 关键行为：帮助用户快速进入四条业务流

### F-03 首条消息创建与发送

- 来源需求：`R-01`
- 前端承接：发送动作
- 数据对象：Conversation / Message
- 接口：
  - `POST /api/v1/xiaoqiao/conversations`
  - `POST /api/v1/xiaoqiao/conversations/{conversation_id}/messages`
- 开发说明：
  - 无激活会话时先建会话
  - 再发首条消息
  - 返回 message / routing / task / assistant_reply
- 异常回退：
  - 建会话失败：阻断发送
  - 发消息失败：提示重试，不丢输入内容

### F-04 首轮系统反馈

- 来源需求：`R-01`
- 前端承接：消息列表首轮 assistant 消息
- 数据对象：Message / RoutingDecision / Task
- 接口：
  - 依赖 `POST /messages` 返回值
- 开发说明：
  - 非业务输入给普通回复
  - 业务输入给任务型回复
  - 缺信息时给 clarification

### F-05 路由判断

- 来源需求：`R-02`
- 后端承接：`route_message`
- 数据对象：RoutingDecision
- 当前 contract：
  - `is_business_related`
  - `intent_type`
  - `workflow_level`
  - `clarification_needed`
  - `decision_reason`

### F-06 业务相关性识别

- 来源需求：`R-02`
- 前端承接：可选顶部状态或回复文案提示
- 后端依赖：`routing.is_business_related`

### F-07 意图类型识别

- 来源需求：`R-02`
- 前端承接：结果区类型切换、任务类型展示
- 后端依赖：`routing.intent_type`
- 当前支持：
  - `help`
  - `demand`
  - `diagnosis`
  - `debugging`

### F-08 追问判断

- 来源需求：`R-02`
- 前端承接：提示用户补充信息
- 后端依赖：`routing.clarification_needed`
- 当前缺口：
  - 需要进一步把“缺什么字段”显式挂到任务上下文

### 3.9 当前阶段 mock contract 承接策略

当前技术现实已经明确：

1. 智投预发布数据库当前禁止本地连接。
2. 因此小乔第一阶段继续使用本地仓库后端作为承接层。
3. 但本地后端返回的数据不能停留在“随便写点 mock”。

本阶段的正式要求是：

1. 所有页面消费的数据，都必须来自稳定接口 contract。
2. 所有接口返回的 mock payload，都必须对应真实业务对象和结果物。
3. 四条业务流、管理页、提示词治理都要有各自明确的 mock schema。

当前 contract 分层如下：

1. workspace 级 contract
   - 承接首页状态、入口摘要、系统导航状态
2. conversation 级 contract
   - 承接会话、消息、追问与路由判断
3. task 级 contract
   - 承接帮助 / 需求 / 排查 / 联调 / 管理任务对象
4. result 级 contract
   - 承接四条业务流正式结果结构与管理页结果结构

后续当用户补充 mock schema / content 后，应优先落在这些 contract 上，而不是让页面各自维护一套静态本地数据。
